## PART 2: WHAT DIDN'T GO WELL

### 2.1 Initial Quick Reference Used Markdown Table (Wrong Format)

**What Happened:**
First version of Quick Reference used a markdown table:

```
| # | Section | Key Components |
|---|---------|----------------|
| 1 | Sticky Header/Announcement | 🔶 Sale banner, 🔶 Discount %, Phone, Trust badges |
...
```

**Human Feedback:**
> "fix the formatting of the summary to be this exactly: SECTION #: SECTION NAME / component 1 name / component 2 name / component 3 name"

**Root Cause Analysis:**

| Factor | Description |
|--------|-------------|
| **Format assumption** | Assumed markdown tables were the "professional" format |
| **Readability oversight** | Didn't consider that plain text lists are easier to scan |
| **Component cramming** | Table cells forced multiple components onto one line with commas |

**Impact:**
Required one revision cycle. Human had to specify exact format.

**What Changed:**
Replaced table with plain text list where each section header is on its own line and each component is listed individually underneath:

```
SECTION 1: STICKY HEADER / ANNOUNCEMENT BAR [🔶 PG-UNIQUE]
🔶 Sale announcement banner ("63% OFF")
🔶 Discount percentage callout
Phone number
🔶 Bonus count callout ("4 FREE Bonuses")
```

**Lesson Learned:**
Plain text formats are often more readable than tables, especially for lists of components. Tables are better for data with multiple columns of different types (counts, percentages, etc.). Component lists are better as plain text.

### 2.2 Section Headers Lacked OEM Indicators

**What Happened:**
First version of Quick Reference had section headers like:

```
SECTION 3: HERO / ABOVE THE FOLD
```

Human couldn't tell at a glance which sections were OEM baseline vs PG-UNIQUE.

**Human Feedback:**
> "indicate which sections are from OEMs as well in the section # / section name lines"

**Root Cause Analysis:**

| Factor | Description |
|--------|-------------|
| **Information hiding** | Section type info was only in detailed outline, not Quick Reference |
| **Assumed reader would scroll** | Expected readers to check detailed outline for section type |
| **Overlooked reference value** | Quick Reference should be COMPLETE overview, not partial |

**Impact:**
Required revision to add section type indicators.

**What Changed:**
Added indicators to EVERY section header:
- `[🔶 PG-UNIQUE]` — Section only exists on PG pages (0% OEM)
- `[OEM BASELINE + PG ADDITIONS]` — OEM section with PG conversion components
- `[DTC BASELINE + PG ADDITIONS]` — DTC section with PG additions
- `[DR BASELINE + PG ADDITIONS]` — DR section with PG additions
- `[HYBRID]` — Mixed sources

**Lesson Learned:**
Every piece of information that's in the detailed version should be SUMMARIZED in the quick reference. The Quick Reference isn't just a list — it's a complete overview that should answer basic questions without requiring scroll.

**What Happened:**
Initial component names were abstracted from source terminology:
- "Title" instead of "Product name headline"

**Root Cause Analysis:**

| Factor | Description |
|--------|-------------|
| **Over-abstraction** | Tried to "clean up" component names into generic terms |
| **Lost source traceability** | Abstract names broke connection to source files |
| **Assumed meaning was obvious** | "Title" means product title to me, but not clear to reader |
| **Missed critical component** | By calling it "Title" instead of "Product name headline," I obscured that there's also a "Value proposition subheadline" |

**Impact:**
Deliverable was initially confusing and missing obvious components (like HEADLINE). Required significant revision.

**What Changed:**
Rewrote ALL component names using EXACT terminology from source files:
- "Product name headline" (not "Title")
- "Value proposition subheadline" (was omitted)
- "Configuration selectors (Hand, Loft, Flex)" (not "Config")
- "Add to Cart button" (not "CTA")
- "Sale announcement banner ('63% OFF')" (not "Sale banner")

**Lesson Learned:**
**NEVER abstract component names.** Use the EXACT terminology from source files. The source names are:
1. More specific
2. More traceable
3. More understandable
4. Less ambiguous

If the source says "Product image gallery (5-9 images)" then the component name is "Product image gallery (5-9 images)" — not "Gallery" or "Images."

### Insight 4: Plain Text Lists Are Superior for Component Reference

**The Observation:**
After the human requested plain text format, the Quick Reference became significantly more usable:

**Table Format (Before):**
```
| 3 | Hero/Above the Fold | Gallery (GAP FIX), Title, Price, Config, CTA + 🔶 Countdown, 🔶 Triple strikethrough, 🔶 Urgency, ⚡ Star rating, ⚡ Review count |
```

**Plain Text Format (After):**
```
SECTION 3: HERO / ABOVE THE FOLD [OEM BASELINE + PG ADDITIONS]
Product image gallery (GAP FIX) [OEM 100%]
Product name headline [OEM 100%]
Value proposition subheadline [OEM 100%]
Configuration selectors (Hand, Loft, Flex) [OEM 83%]
Price display [OEM 75%]
Add to Cart button [OEM 75%]
🔶 Countdown timer
🔶 Triple strikethrough pricing ($399 → $249 → $187)
🔶 Savings callout ("Save $212")
🔶 Urgency message ("Hurry! claim your discount...")
🔶 Trust badge stack (365-day, Free Shipping, Secure Checkout)
🔶 Testimonial snippet
⚡ Star rating display [DTC 80%]
⚡ Review count link [DTC 80%]
⚡ Payment options (Affirm/Shop Pay) [DTC 80%]
```

**Why Plain Text Is Superior:**
1. **Each component on its own line** — easy to scan, count, reference
2. **No character cramming** — table cells force abbreviation
3. **Room for examples** — can include ("63% OFF") without breaking layout
4. **Room for percentages** — can include [OEM 100%] naturally
5. **Copy-paste friendly** — Phase II copywriters can copy individual components

**The Lesson:**
Tables are for comparing ACROSS categories (rows = items, columns = attributes). Lists are for enumerating items with their attributes inline. Component lists are better as plain text.

### Insight 5: Source Terminology Must Be Preserved

**The Observation:**
When I abstracted component names, I broke traceability and lost meaning:

| My Abstraction | Source Terminology | What Was Lost |
|----------------|-------------------|---------------|
| "Title" | "Product name headline" | Clarity that it's the H1 |
| (missing) | "Value proposition subheadline" | Entire component omitted |
| "Config" | "Configuration selectors (Hand, Loft, Flex)" | What's being configured |
| "CTA" | "Add to Cart button" | What the CTA actually says |
| "Sale banner" | "Sale announcement banner ('63% OFF')" | Example text |

**Why Abstraction Is Harmful:**
1. **Breaks traceability** — can't find "Title" in source files, but can find "Product name headline"
2. **Loses specificity** — "Config" could mean anything
3. **Hides components** — "Title" absorbed both headline AND subheadline
4. **Removes examples** — "Sale banner" doesn't show what the banner says

**The Rule:**
**NEVER abstract component names.** If the source says "Triple strikethrough pricing ($399 → $249 → $187)" then the component name is "Triple strikethrough pricing ($399 → $249 → $187)" — not "Triple strikethrough" or "Pricing" or "Price anchoring."
