# 2.2-design-note-completeness.md

> **Layer:** 2 (Revision)
> **Skill:** EC-06-ecomm-editorial
> **Purpose:** Ensure design notes are actionable for page builder
> **Outputs:** `design_notes_review` object with complete, actionable notes per section

---

## FUNCTION

Review and complete design notes for every section. Design notes bridge copy and page build -- they tell the page builder HOW to visually present each section. Vague notes ("nice layout") fail the page builder. Every section must have specific, actionable notes covering layout, mobile behavior, and visual elements.

---

## COMPLETENESS PROTOCOL

### Step 1: Audit Existing Design Notes
```
FOR each section:
  CHECK design_notes for three required components:
    [ ] LAYOUT: How is this section laid out? (grid, stack, card, table, accordion)
    [ ] MOBILE: How does this section behave on mobile? (stack, collapse, swipe, hide)
    [ ] VISUALS: What visual elements support the copy? (icons, images, badges, borders)

  CLASSIFY:
    COMPLETE: All 3 components present and specific
    PARTIAL: 1-2 components present or some vague
    MISSING: No design notes or placeholder only
```

### Step 2: Vague Note Detection
```
VAGUE NOTE PATTERNS (flag these):
  - "Nice layout" -> WHAT layout?
  - "Visual elements" -> WHICH visual elements?
  - "Mobile friendly" -> HOW is it mobile friendly?
  - "Clean design" -> WHAT makes it clean?
  - "Eye-catching" -> WHAT catches the eye?
  - "Modern look" -> WHAT specifically is modern?
  - "Good spacing" -> HOW MUCH spacing?
  - "Appropriate imagery" -> WHAT imagery?
```

### Step 3: Complete Missing Notes
```
FOR each section with PARTIAL or MISSING notes:
  GENERATE specific notes based on:
    section_type: [NLS PDP type determines default layout]
    copy_structure: [template used determines formatting]
    proof_type: [determines visual emphasis approach]
    priority: [higher priority = more visual investment]

  LAYOUT DEFAULTS BY SECTION TYPE:
    Feature Showcase (T3): "2-column grid on desktop; icon + name + description per card"
    Proof Block (T4): "Full-width pull-quote; large font size; source attribution below"
    Comparison (T7): "Responsive table; checkmark/X icons; product column highlighted"
    How-It-Works (T2): "3-5 numbered steps; icon per step; single column"
    Problem-Solution (T1): "Split layout: problem (dark bg) | solution (light bg)"
    Offer/Value (T5): "Price card centered; comparison struck-through; CTA button below"
    Informational (T6): "Accordion (FAQ) or numbered steps; collapse on mobile"
```

### Step 4: Mobile Behavior Specification
```
FOR each section:
  SPECIFY mobile behavior:
    stack: "Columns stack vertically"
    collapse: "Accordion sections collapse"
    swipe: "Cards become swipeable carousel"
    prioritize: "Show first N items, hide rest behind 'Show More'"
    maintain: "Layout unchanged on mobile"

  EVERY section must have explicit mobile behavior -- no assumptions
```

| Mode | Trigger | Remediation |
|------|---------|-------------|

| WARNING | Vague notes detected | Replace with specific, actionable notes |
| INFO | All notes complete | Verify mobile behavior specs are explicit |
