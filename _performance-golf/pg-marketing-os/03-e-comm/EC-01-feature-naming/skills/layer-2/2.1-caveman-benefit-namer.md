# 2.1-caveman-benefit-namer.md

> **Layer:** 2 (Feature Naming)
> **Skill:** EC-01-feature-naming
> **Purpose:** Name features in caveman-benefit format
> **Outputs:** `named_features` object with caveman-benefit names for all features

---

## FUNCTION

Generate feature names for every capability in the hierarchy using the caveman-benefit naming format. Each name must communicate WHAT the feature is AND WHY it matters -- in 2-4 words. This is the most critical creative step in the E-Commerce Copy Engine.

---

## CAVEMAN-BENEFIT FORMAT

### The Core Principle

A caveman should understand the feature name. Not the science, not the mechanism -- the NAME itself tells you what it does.

```
BAD:  "X-Factor Technology"     -- What does this do? Nobody knows.
BAD:  "Advanced Formula"        -- Advanced compared to what? Generic.
BAD:  "ProSystem 3.0"           -- Means nothing to anyone.

GOOD: "Dynamic Loft Control"    -- It controls loft, dynamically. Clear.
GOOD: "L-Theanine for Tranquility" -- The ingredient, for the benefit.
GOOD: "HyperSpeed Face"         -- The face is about speed. Fast.
GOOD: "Triple-Layer Cushioning" -- Three layers, for cushion. Simple.
```

---

## NAMING PROTOCOL

### Step 1: Prepare Naming Inputs per Feature

```
FOR each feature in hierarchy:
  GATHER:
    capability_id: [from mapper]
    what_it_is: [specific component/ingredient/technology]
    what_it_does: [functional description]
    why_it_matters: [customer benefit]
    tier: [hero/supporting/technical]
    specimen_patterns: [relevant patterns from calibrator]
    crossover_patterns: [relevant patterns from Skill 15]
```

### Step 2: Generate Name Candidates (3-5 per feature)

```
FOR each feature:
  GENERATE 3-5 name candidates using different patterns:

  Pattern A: [Branded Mechanism] + [Result Word]
    Take the mechanism/ingredient, coin a branded version, add result
    Example: "HyperSpeed Face" (speed + face type)

  Pattern B: [Descriptive Action] + [Component]
    Lead with what it does, follow with what it is
    Example: "Dynamic Loft Control" (what it does + what it controls)

  Pattern C: [Ingredient/Material] + "for" + [Benefit]
    Explicit ingredient-to-benefit linkage
    Example: "Marine Collagen for Skin Renewal"

  Pattern D: [Quantity] + [Component] + [Purpose]
    Quantify the structure, name the purpose
    Example: "Triple-Layer Cushioning System"

  Pattern E: [Benefit-Forward] + [Modifier]
    Lead with the benefit, modify for specificity
    Example: "Distance Lock Technology"

  CONSTRAINT: Hero features should use Pattern A or B (most memorable)
  CONSTRAINT: Supporting features should vary patterns (no monotony)
  CONSTRAINT: Technical features can use Pattern C or D (clarity over flair)
```

### Step 3: Apply Generic Kill List

```
FOR each name candidate:
  CHECK against generic kill list:
    REJECT if contains: Advanced, Premium, Superior, Professional,
    Ultra, Pro, Next-Gen, State-of-the-Art, Best-in-Class, Cutting-Edge

  CHECK against empty suffix list:
    REJECT if ends with: Technology (alone), System (alone),
    Solution, Platform, Complex, Formula, Blend (without specifics)

  IF rejected:
    REGENERATE with more specific language
    LOG: "Name [X] rejected -- generic pattern detected"
```

### Step 4: Select Best Name per Feature

```
FOR each feature:
  EVALUATE candidates against:
    caveman_clarity: [1-10] Does a non-expert understand?
    memorability: [1-10] Sticky after one exposure?
    benefit_presence: [1-10] Benefit communicated in name?
    word_economy: [1-10] Fewest words, maximum meaning?
    standalone_test: [PASS/FAIL] Works without description?

  SELECT highest-scoring name
  NOTE: Hero feature name must score >= 8.0 average
```

## FAILURE MODES

| Mode | Trigger | Remediation |
|------|---------|-------------|
| CRITICAL | Hero name scores < 8.0 | Regenerate hero names with different patterns |
| CRITICAL | Generic names pass kill list (false negative) | Manual review, tighten kill list |
| WARNING | All names use same pattern | Vary patterns -- minimum 3 different patterns used |
| WARNING | Standalone test failures | Revise failing names with benefit component |
| INFO | More than 5 strong candidates per feature | Good problem -- select best, keep alternatives |
