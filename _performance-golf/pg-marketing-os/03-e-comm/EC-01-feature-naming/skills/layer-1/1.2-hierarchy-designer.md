# 1.2-hierarchy-designer.md

> **Layer:** 1 (Capability Analysis)
> **Skill:** EC-01-feature-naming
> **Purpose:** Rank features into hero / supporting / technical tiers
> **Outputs:** `feature_hierarchy` object with tier assignments

---

## FUNCTION

Design the feature hierarchy by assigning every mappable capability to one of three tiers: hero (leads the product story), supporting (reinforces the hero), or technical (credibility-building specs). The hero feature is the MOST important naming decision -- it anchors the hero section, headlines, and product identity.

---

## HIERARCHY DESIGN PROTOCOL

### Step 1: Hero Feature Selection

```
FROM capability_map, EVALUATE HIGH potential capabilities for hero:

  HERO CRITERIA (must meet ALL):
    1. Most differentiated -- competitors don't have this or don't name it
    2. Most benefit-clear -- customer immediately understands the value
    3. Most memorable -- name potential is high (can be branded)
    4. Most relevant -- addresses primary customer desire or pain

  SCORING:
    FOR each hero candidate:
      differentiation: [1-10]
      benefit_clarity: [1-10]
      memorability_potential: [1-10]
      audience_relevance: [1-10]
      hero_score = weighted average (equal weights)

  SELECT top 1-2 as hero features
  IF tie: FLAG for human checkpoint
  IF no clear hero (all scores < 7): FLAG -- "Hero feature selection needs human input"
```

### Step 2: Supporting Feature Assignment

```
FROM remaining HIGH and MEDIUM potential capabilities:

  SUPPORTING CRITERIA:
    1. Meaningful benefit (not just a spec)
    2. Adds depth to the product story
    3. Not redundant with hero feature
    4. Worth naming (not just listing)

  ASSIGN 3-5 capabilities as supporting features
  ORDER by impact (highest impact supporting first)
```

### Step 3: Technical Feature Assignment

```
FROM remaining capabilities:

  TECHNICAL CRITERIA:
    1. Credibility-building (certifications, materials, specs)
    2. Expected by knowledgeable buyers
    3. Differentiates at technical comparison level

  ASSIGN 1-4 capabilities as technical features
  NOTE: Technical features get simpler naming (descriptive, not branded)
```

### Step 4: Validate Hierarchy Balance

```
CHECK hierarchy balance:
  hero: 1-2 features (REQUIRED)
  supporting: 3-5 features (REQUIRED minimum 2)
  technical: 1-4 features (OPTIONAL but recommended)

  IF hero == 0: FAIL -- "Must designate at least 1 hero"
  IF hero > 2: WARN -- "Too many heroes dilutes impact. Reduce to 1-2."
  IF supporting < 2: WARN -- "Insufficient depth. Add supporting features."
  IF total named < 4: FAIL -- "Minimum 4 named features required"
```

## FAILURE MODES

| Mode | Trigger | Remediation |
|------|---------|-------------|
| CRITICAL | No hero candidate scores >= 7 | FLAG for human hero selection |
| CRITICAL | Fewer than 4 total nameable features | Request more product data |
| WARNING | Hero tie (2+ candidates within 0.5 points) | Present options to human |
| WARNING | Supporting features < 2 | Promote technical features or request data |
| INFO | Strong hierarchy with clear hero | Best case -- proceed |
