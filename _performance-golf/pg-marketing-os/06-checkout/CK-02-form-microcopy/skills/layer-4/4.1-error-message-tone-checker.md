# 4.1-error-message-tone-checker.md

> **Layer:** 4 (Validation)
> **Skill:** CK-02-form-microcopy
> **Purpose:** Verify all error messages guide (not blame) the user
> **Outputs:** `tone_check_result` object with per-message pass/fail

---

## FUNCTION

Systematically check every error message in the micro-copy set for blaming language. This is the most critical validation step in CK-02 because blaming error messages are the number one source of fixable checkout abandonment. A single "Invalid card number" can cost a sale. Every error message must tell the user what went wrong, what the correct format is, and how to fix it -- without making them feel stupid or accused.

---

## TONE CHECK PROTOCOL

```
EXECUTE CK-02/4.1-error-message-tone-checker:
  INPUT: microcopy_set (from 2.1)
  OUTPUT: tone_check_result object

  STEP 1: EXTRACT ALL ERROR MESSAGES
    - Collect every error_message from field_copy array
    - Include any inline validation messages
    - Include form-level error messages (if any)
    - Count total messages to check

  STEP 2: CHECK EACH MESSAGE AGAINST BANNED PATTERNS
    FOR each error_message:
      SCAN for banned patterns:
        - "Invalid [anything]" -> FAIL
        - "Wrong [anything]" -> FAIL
        - "Incorrect [anything]" -> FAIL
        - "Bad [anything]" -> FAIL
        - "Error" (alone, without specific guidance) -> FAIL
        - "Failed" -> FAIL
        - "Rejected" -> FAIL
        - "Denied" -> FAIL
        - "You must [anything]" (commanding tone) -> FAIL
        - "Required" (alone, as error text) -> FAIL

      CHECK for positive patterns:
        - Starts with "Please" or action verb -> GOOD
        - Includes specific format guidance -> GOOD
        - Tells user what TO DO (not what they did wrong) -> GOOD
        - Mentions expected format (digits, characters, format) -> GOOD

  STEP 3: RATE EACH MESSAGE
    a. PASS: message guides without blaming
    b. FAIL: message blames, commands, or provides no guidance
    c. For each FAIL: provide specific rewrite recommendation

  STEP 4: CHECK OVERALL TONE
    - All messages should feel like they come from the same voice
    - Consistent use of "Please" or consistent alternative
    - No mixed register (formal + casual in same checkout)

  STEP 5: PRODUCE VALIDATION REPORT
    - Total messages checked
    - Pass count / fail count
    - For each fail: original message + recommended rewrite
    - Overall verdict: PASS (zero fails) or FAIL (any fail)

  IF any message FAILS:
    RETURN to Layer 2 for rewrite. Do not proceed to packaging.
```

---

## TONE CHECK EXAMPLES

| Original (FAIL) | Rewritten (PASS) | Why |
|-----------------|-----------------|-----|
| "Invalid email" | "Please enter a valid email address" | Guidance instead of blame |
| "Wrong card number" | "Please check your card number -- it should be 16 digits" | Specific format guidance |
| "Required" | "Please fill in this field" | Action instead of label |
| "Error" | "Please check the highlighted fields below" | Direction to fix |
| "Invalid zip code" | "Please enter your 5-digit zip code" | Shows exact expected format |
| "Card declined" | "This card could not be processed. Please try a different card or contact your bank." | Provides next steps |
| "You must enter your name" | "Please enter your first and last name" | Request instead of command |

---

## FAILURE MODES

| Mode | Trigger | Remediation |
|------|---------|-------------|
| CRITICAL | Any error message uses blaming language | RETURN to 2.1 -- rewrite failing messages |
| CRITICAL | Error message provides no guidance (just "Error") | REWRITE with specific format guidance |
| DEGRADED | Tone inconsistency across messages | Standardize voice (all "Please" or all action-oriented) |
| DEGRADED | Commanding tone ("You must") | Soften to request ("Please enter") |
