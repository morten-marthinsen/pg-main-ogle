# 1.1-trust-signal-mapper.md

> **Layer:** 1 (Mapping)
> **Skill:** CK-01-trust-security-copy
> **Purpose:** Map trust signals per category (security/payment/guarantee/social/contact)
> **Outputs:** `trust_signal_map` object with all signals, copy needs, and word budgets

---

## FUNCTION

Transform the trust architecture plan from CK-00 into a detailed signal map with specific copy needs for each signal. For every trust signal, define what copy is needed, what icon or visual element accompanies it, the word budget, and whether it requires client-provided data. This map becomes the generation specification for Layer 2.

---

## MAPPING PROTOCOL

```
EXECUTE CK-01/1.1-trust-signal-mapper:
  INPUT: upstream_context (from 0.1), validation_result (from 0.2)
  OUTPUT: trust_signal_map object

  STEP 1: MAP SECURITY SIGNALS
    - SSL/encryption badge: needs short text (2-5 words)
    - Lock icon: needs label (1-3 words)
    - Secure checkout header: needs header text (2-4 words)
    - For each: icon required Y/N, word budget, copy type (static text)

  STEP 2: MAP PAYMENT SIGNALS
    - Accepted payment logos: no copy needed (visual only) OR "We accept" label
    - Payment method icons: list which logos to display
    - Express checkout options: PayPal/Apple Pay button labels
    - For each: icon required Y/N, word budget, copy type

  STEP 3: MAP GUARANTEE SIGNALS
    - Main guarantee statement: needs headline (4-8 words)
    - Guarantee supporting text: needs body (8-15 words)
    - Guarantee icon/badge: needs visual treatment spec
    - Source: offer-package.json guarantee terms (EXACT match required)
    - For each: icon required Y/N, word budget, copy type

  STEP 4: MAP SOCIAL PROOF SIGNALS
    - Customer count: needs specific number from client OR placeholder
    - Review rating: needs star count + review count from client OR placeholder
    - Testimonial snippet: needs short quote (if available) OR skip
    - For each: icon required Y/N, word budget, copy type
    - FLAG if data needs to come from client

  STEP 5: MAP CONTACT SIGNALS
    - Phone number: needs real number from client OR "[CLIENT TO PROVIDE]"
    - Email support: needs real email from client OR "[CLIENT TO PROVIDE]"
    - Support hours: needs real hours from client OR standard placeholder
    - For each: word budget, copy type
    - NEVER fabricate contact information

  STEP 6: ASSIGN WORD BUDGETS
    a. Security signals: 2-8 words each
    b. Payment signals: 0-3 words each (mostly visual)
    c. Guarantee signals: 10-25 words total
    d. Social proof signals: 3-10 words each
    e. Contact signals: 5-12 words each

  IF client data needed:
    Mark signal as "requires_client_input" with specific data request.
```

## FAILURE MODES

| Mode | Trigger | Remediation |
|------|---------|-------------|
| CRITICAL | Any of the 5 categories has zero signals | ADD -- every category needs at least one signal |
| CRITICAL | Guarantee terms not sourced from offer package | HALT -- verify source |
| DEGRADED | Client data unavailable for social proof | Use placeholder structure "[X]+ customers" |
| DEGRADED | Contact information not provided | Use "[CLIENT TO PROVIDE]" placeholder |
