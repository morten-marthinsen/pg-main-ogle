# 1.2-trust-density-reviewer.md

> **Layer:** 1 (Audit Pass 1)
> **Skill:** CK-03-checkout-editorial
> **Purpose:** Are 3+ trust signals at every scroll position?
> **Outputs:** `trust_density_review` object with per-position verification

---

## FUNCTION

Independently verify that the trust density target (3+ trust signals visible at every scroll position) is achieved on both desktop and mobile viewports. This is a verification audit of CK-01's density work -- CK-03 confirms the claim, it does not generate new trust signals. If density fails, CK-03 escalates to CK-01 for remediation.

---

## DENSITY REVIEW PROTOCOL

```
EXECUTE CK-03/1.2-trust-density-reviewer:
  INPUT: checkout_copy_elements (from 0.1), strategy_reference (from 0.2)
  OUTPUT: trust_density_review object

  STEP 1: MAP VIEWPORT POSITIONS
    - Use flow map sections to estimate viewport boundaries
    - For desktop: 4-6 viewport segments (typical single-page checkout)
    - For mobile: 6-10 viewport segments (more scrolling, less per screen)
    - For multi-step: each step is its own viewport evaluation

  STEP 2: COUNT SIGNALS PER POSITION (DESKTOP)
    FOR each viewport segment:
      - Identify which trust signals are visible:
        * Header signals (if header is visible)
        * Inline signals within visible sections
        * Footer signals (if footer is visible)
        * Sticky signals (always visible)
      - Count total signals visible
      - Record: position, signals, count, pass/fail

  STEP 3: COUNT SIGNALS PER POSITION (MOBILE)
    FOR each mobile viewport segment:
      - Account for smaller screen = fewer elements visible
      - Account for stacking = different element visibility
      - Include sticky mobile elements (trust badge mini)
      - Count total signals visible
      - Record: position, signals, count, pass/fail

  STEP 4: IDENTIFY GAPS
    a. Any position with fewer than 3: FAILING
    b. Any position with exactly 3: AT MINIMUM (fragile)
    c. Document specific gaps with:
       - Which position
       - What signals are missing
       - What could be added (recommendation for CK-01)

  STEP 5: PRODUCE REVIEW VERDICT
    - PASS: all positions >= 3 on both desktop and mobile
    - FAIL: any position < 3
    - If FAIL: produce escalation recommendation for CK-01

  IF density FAILS at any position:
    DO NOT attempt to fix by generating new trust copy.
    ESCALATE to CK-01 with specific gap analysis.
    CK-03 does not generate -- it reviews.
```

---

## FAILURE MODES

| Mode | Trigger | Remediation |
|------|---------|-------------|
| CRITICAL | Any position below 3 signals | ESCALATE to CK-01 -- do not generate new trust copy |
| CRITICAL | Mobile density not evaluated | HALT -- mobile audit is mandatory |
| DEGRADED | Density depends entirely on sticky element | WARN -- if sticky element removed, density fails |
| DEGRADED | Position at exactly 3 (minimum) | FLAG as fragile -- recommend buffer signal |
