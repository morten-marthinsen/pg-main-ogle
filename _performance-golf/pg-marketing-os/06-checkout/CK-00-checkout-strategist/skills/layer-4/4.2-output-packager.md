# 4.2-output-packager.md

> **Layer:** 4 (Validation & Packaging)
> **Skill:** CK-00-checkout-strategist
> **Purpose:** Package checkout-strategy.yaml as final output
> **Outputs:** `checkout-strategy.yaml` file

---

## FUNCTION

Assemble the validated checkout strategy into the final `checkout-strategy.yaml` output file. This is the canonical handoff document that CK-01, CK-02, CK-03, and U1 consume. The package must be complete, schema-valid, and contain no placeholders or TODO items.

---

## PACKAGING PROTOCOL

```
EXECUTE CK-00/4.2-output-packager:
  INPUT: checkout_strategy (from 2.1), validation_report (from 4.1)
  OUTPUT: checkout-strategy.yaml file

  STEP 1: VERIFY VALIDATION PASSED
    - Confirm validation_report.status == PASS
    - If FAIL: HALT -- cannot package unvalidated strategy
    - If PASS with warnings: include warnings in package metadata

  STEP 2: ASSEMBLE YAML STRUCTURE
    - Header: version, timestamp, skill_id, project_id
    - Funnel classification block
    - Checkout pattern block
    - Trust architecture block (all 5 categories)
    - Friction mitigation block (all points with assignments)
    - Checkout flow map (all sections in order)
    - Order bump specification
    - Downstream handoff specification
    - Execution metadata

  STEP 3: VALIDATE SCHEMA
    - All required top-level keys present
    - No null values where values expected
    - No placeholder text ("TODO", "TBD", "[INSERT]")
    - Trust density target is numeric
    - Friction points have severity ratings
    - All sections have mobile behavior

  STEP 4: WRITE FILE
    - Write checkout-strategy.yaml to project outputs directory
    - Verify file written successfully
    - Log completion timestamp

  STEP 5: CREATE VERIFICATION OBJECT
    - Document file path, size, schema version
    - Confirm all sections present
    - Record any warnings carried from validation

  IF schema validation fails:
    HALT -- Fix specific schema violations before writing file.

  IF file write fails:
    RETRY once, then HALT with error details.
```

---

## OUTPUT STRUCTURE

The output IS the `checkout-strategy.yaml` file containing:

```yaml
checkout_strategy:
  version: "1.0"
  generated_at: "ISO timestamp"
  skill_id: "CK-00-checkout-strategist"
  project_id: "[from campaign brief]"

  funnel_classification:
    type: "direct|supplement|digital|free_plus_ship|trial"
    confidence: 0.92
    rationale: "Classification rationale"

  checkout_pattern:
    pattern: "single_page|multi_step"
    step_count: 1
    steps: [...]
    mobile_adaptation: {...}

  trust_architecture:
    signal_categories: {...}
    density_target: "3+ per viewport"
    mobile_density_plan: "..."

  friction_mitigation:
    total_points: 7
    mitigations: [...]

  flow_map:
    sections_in_order: [...]

  order_bump_spec:
    included: true
    placement: "..."
    word_budget: "50-150"
    format: "checkbox with copy block"

  downstream_handoff:
    ck_01: [...]
    ck_02: [...]
    u1: [...]
```

Additionally, create a verification object:

```json
{
  "package_assembly": {
    "status": "complete",
    "file_written": "checkout-strategy.yaml",
    "file_path": "/path/to/file",
    "schema_version": "1.0",
    "validation_passed": true,
    "warnings": [],
    "written_at": "ISO timestamp"
  }
}
```

## VALIDATION CHECKLIST

Before claiming package complete:

```
[ ] checkout-strategy.yaml file EXISTS in outputs folder
[ ] File is valid YAML (parseable)
[ ] All required sections present:
    [ ] funnel_classification block complete
    [ ] checkout_pattern block complete
    [ ] trust_architecture block complete (all 5 categories)
    [ ] friction_mitigation block complete
    [ ] flow_map block complete (no gaps)
    [ ] order_bump_spec block complete
    [ ] downstream_handoff block complete
[ ] No empty strings where values expected
[ ] No placeholder text ("TODO", "TBD", etc.)
[ ] Trust density target is 3+ per viewport
[ ] Every friction point has severity + mitigation + assignment
[ ] Every section has mobile behavior specified
```

## FAILURE MODES

| Mode | Trigger | Remediation |
|------|---------|-------------|
| CRITICAL | Validation report status = FAIL | HALT -- return to validation, fix gaps |
| CRITICAL | Schema validation fails | Fix specific violations, re-validate |
| CRITICAL | File write fails | Retry once, then HALT with error |
| DEGRADED | Warnings from validation | Include in metadata, proceed with package |
