# 1.2-checkout-pattern-selector.md

> **Layer:** 1 (Classification)
> **Skill:** CK-00-checkout-strategist
> **Purpose:** Select checkout pattern (single-page vs. multi-step)
> **Outputs:** `checkout_pattern` object with pattern, rationale, and step structure

---

## FUNCTION

Based on the funnel type classification from 1.1, select the optimal checkout pattern. Single-page checkouts show all fields at once and work best for simple purchases with few fields. Multi-step checkouts break the process into stages with progress indicators and work best for complex purchases or when progressive disclosure reduces perceived friction.

---

## PATTERN SELECTION PROTOCOL

```
EXECUTE CK-00/1.2-checkout-pattern-selector:
  INPUT: funnel_classification (from 1.1), upstream_context (from 0.1)
  OUTPUT: checkout_pattern object

  STEP 1: EVALUATE FIELD COMPLEXITY
    - Count total form fields required for this funnel type
    - Digital products: 4-5 fields (email, name, card, expiry, cvv)
    - Physical products: 7-8 fields (+ address, zip, state)
    - Subscription: +1-2 fields (subscription terms, frequency)
    - High-ticket: potentially more fields (phone, company)

  STEP 2: APPLY PATTERN DECISION MATRIX
    a. Single-page checkout:
       - Fewer than 8 form fields
       - Simple offer structure (one product, one price)
       - Low-to-mid price point
       - Direct sale, digital product, supplement
       - When speed matters more than guided experience

    b. Multi-step checkout:
       - 8+ form fields
       - Complex offer (multiple options, configuration needed)
       - High-ticket items ($200+)
       - Trial offers (need dedicated terms section)
       - When progressive disclosure reduces perceived complexity
       - When micro-commitments increase completion rate

  STEP 3: DEFINE STEP STRUCTURE (if multi-step)
    - Assign fields to steps (max 3-4 fields per step)
    - Name each step (1-2 words: "Contact", "Shipping", "Payment")
    - Define progress indicator format
    - Specify what trust signals appear at each step

  STEP 4: SPECIFY MOBILE ADAPTATION
    - Single-page: how sections stack on mobile
    - Multi-step: how step navigation works on mobile
    - Field sizing for mobile tap targets
    - Trust signal placement on mobile viewport

  IF field count and funnel type disagree:
    WARN -- "Field count suggests [pattern A] but funnel type suggests [pattern B].
    Recommending [chosen] because [rationale]."
```

---

## PATTERN DECISION MATRIX

| Factor | Single-Page | Multi-Step |
|--------|------------|------------|
| Field count | < 8 | 8+ |
| Price point | Low-mid | Mid-high |
| Product type | Digital, simple physical | Complex, high-ticket |
| Offer complexity | Single product | Multiple options/configs |
| Funnel type | Direct, digital, supplement | Trial, high-ticket, free+ship |
| Mobile priority | Scrollable form | Step-by-step navigation |
| Conversion priority | Speed | Guided completion |

---

## FAILURE MODES

| Mode | Trigger | Remediation |
|------|---------|-------------|
| CRITICAL | Funnel classification missing | HALT -- cannot select pattern without funnel type |
| DEGRADED | Field count and funnel type disagree | Document conflict, select based on stronger signal, flag for review |
| DEGRADED | Hybrid pattern needed | Design hybrid (single-page with expandable sections), document rationale |
