# 1.3 Cross-Platform Cascade Design

## Purpose
Design multi-platform posting sequences (cascades) that amplify reach without appearing as spam.

---

## Input Requirements

From Layer 1:
- `1.2-content-type-timing.yaml` (content schedule)

From Layer 0:
- `0.2-teachings-loaded.yaml` (cascade best practices)

---

## Execution Protocol

### Step 1: Identify Cascade-Worthy Content

Criteria for cascade sequences:
- Hero content (virality score ≥ 75)
- Campaign anchor content
- Cross-platform potential (works on 2+ platforms)

### Step 2: Design Cascade Sequences

For each cascade:

**Cascade Type Selection:**
- **Hero Cascade:** Major content, 4-6 platform sequence, 24-48hr timeline
- **Standard Cascade:** Regular content, 2-3 platforms, 4-24hr timeline
- **Rapid Cascade:** Trend response, all platforms within 4 hours

**Cascade Structure:**

```yaml
cascade_id: "[ID]"
content_id: "[ID]"
cascade_type: [Hero/Standard/Rapid]

sequence_steps:
  - step: "T0"
    platform: "[Primary platform]"
    time: "[Exact time]"
    content_type: "[Format]"
    actions:
      - "Post primary content"
      - "Begin engagement protocol (S16)"

  - step: "T1"
    delay: "[Time after T0]"
    platform: "[Platform 2]"
    time: "[Exact time]"
    content_type: "[Format adapted]"
    actions:
      - "Post adapted version"
      - "Cross-reference primary"

  # [Continue for all steps]
```

### Step 3: Validate Cascade Rules

- NO simultaneous posting across platforms
- Minimum 30min delay between platforms
- Primary platform always T0
- Each platform gets native adaptation
- Total cascade duration < 48 hours

## Quality Gates

- [ ] At least 1 cascade defined
- [ ] All cascades follow timing rules (no simultaneous, min 30min delay)
- [ ] Primary platform always T0
- [ ] All platforms in cascade have defined times
- [ ] Native adaptation noted for each step
