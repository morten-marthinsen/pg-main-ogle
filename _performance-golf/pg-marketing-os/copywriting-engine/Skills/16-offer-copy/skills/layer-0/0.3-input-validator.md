## PURPOSE

The Input Validator examines all loaded upstream data and source teaching indexes to determine whether the pipeline has everything it needs to generate offer copy. It performs three critical functions: (1) validates that all required offer data fields are present and populated with actionable values, (2) identifies specific gaps that must be filled by the human operator at the checkpoint, and (3) determines the appropriate human checkpoint default — CONFIRM if all data is complete, PARTIAL if gaps exist.

This microskill sits between the loading phase (0.1 and 0.2) and the human checkpoint (0.4). It is the quality gate that ensures the human sees an accurate picture of what the system knows and what it needs. Without rigorous validation here, the human checkpoint would either present false confidence (claiming data is complete when it has gaps) or false alarm (asking for data that is already present). Both waste the operator's time and compromise offer accuracy.

The validator operates with a "minimum viable offer" standard: the pipeline needs at minimum a product name, at least one deliverable description, a price, and a guarantee type. Anything below this minimum is a blocking failure. Everything above minimum but below complete is flagged for PARTIAL checkpoint.

---

## IDENTITY

**This microskill IS:**
- The completeness auditor for all offer data required to write offer copy
- A gap identifier that produces a specific, actionable list of missing fields
- The checkpoint default determiner — CONFIRM vs. PARTIAL based on data completeness
- A minimum viable offer enforcer — blocking the pipeline if core data is missing
- A field-by-field validator that checks not just presence but also quality of data
- A cross-reference validator ensuring data consistency across upstream packages
- The bridge between raw loaded data and the human-facing checkpoint presentation

**This microskill is NOT:**
- A data loader — loading is 0.1's job; this validates what was loaded
- A human-facing presenter — presenting to the human is 0.4's job
- A gap filler — it identifies gaps but does not fill them
- A draft generator — it does not write any offer copy
- A source teaching validator — SIN and TIER1 index quality is 0.2's responsibility

## PROCESS

### Step 1: Validate Product Identity Fields

1. Check `product_name`:
   - Present and non-empty: PASS
   - Contains placeholder text (`[INSERT]`, `TBD`, `TODO`, `PLACEHOLDER`): FAIL — log as critical gap
   - Empty or null: FAIL — log as critical gap
2. Check `product_format`:
   - Must be one of: supplement, digital, physical, information, hybrid, service
   - Unrecognized format: log as warning (not blocking — 1.1 will classify)
3. Check `positioning_statement`:
   - Present: PASS
   - Empty: log as non-critical gap (can be inferred from product data)
4. Check `components_established`:
   - Non-empty array: PASS
   - Empty array: log as warning (Skill 12 may not have established components)

### Step 2: Validate Deliverable Data

5. Check `deliverables` array:
   - Contains at least 1 deliverable: minimum PASS
   - Each deliverable has a `name` field: PASS per item
   - Each deliverable has a `description` or `features` field: PASS per item
   - Empty array: CRITICAL GAP — log "No deliverables available"
6. For each deliverable, assess D-F-W-B-P readiness:
   - Can a Feature be derived? (description or feature list present)
   - Can a Why be derived? (benefit context or problem-solution link present)
   - Can a Benefit be derived? (outcome or transformation link present)
   - Can a Proof element be attached? (proof inventory has relevant assets)
7. Score deliverable readiness: count of deliverables with full D-F-W-B-P potential vs. total
8. Log any deliverables that lack sufficient detail for D-F-W-B-P treatment

### Step 3: Validate Bonus Data

9. Check `bonuses` array:
   - Contains at least 1 bonus: PASS
   - Each bonus has a `name`: PASS per item
   - Each bonus has a `value` (dollar amount): PASS per item
   - Empty array: GAP — log "No bonuses available" (not critical — some offers have no bonuses)
10. For each bonus, assess D-F-W-B-P readiness (same criteria as deliverables)
11. Validate bonus values:
    - All values are positive numbers: PASS
    - Any value is zero, negative, or absent: log as gap

### Step 4: Validate Pricing Data

12. Check `retail_value`:
    - Present and > 0: PASS
    - Null or zero: CRITICAL GAP — log "No retail value for price anchoring"
13. Check `actual_price`:
    - Present and > 0: PASS
    - Null or zero: CRITICAL GAP — log "No actual price available"
14. Check price relationship:
    - retail_value > actual_price: PASS (normal — anchor is higher)
    - retail_value <= actual_price: WARNING — "Retail value not higher than actual price — price anchoring will not work"
    - retail_value >= 5x actual_price: GOOD — strong anchor ratio
    - retail_value < 3x actual_price: WARNING — "Anchor ratio below 3x — price contrast may be weak"
15. Check `terms`:
    - Present: PASS
    - Absent: GAP — log "No payment terms specified"
16. Check `anchor_points`:
    - Contains at least one middle anchor between retail and actual: GOOD
    - Empty or absent: OK — middle anchor is optional
17. Validate math consistency:
    - If bonuses have individual values: sum of all deliverable values + bonus values should approximate retail_value
    - Log any math discrepancy > 10% as WARNING

### Step 5: Validate Guarantee Data

18. Check guarantee object:
    - Has `type` or `duration`: PASS
    - Has `branded_name`: EXCELLENT — no human input needed for guarantee naming
    - Has `terms` or `specifics`: PASS
    - Entirely empty or null: CRITICAL GAP — log "No guarantee data available"
19. Check for generic guarantee language:
    - If guarantee description contains ONLY "money-back guarantee" or "satisfaction guaranteed": log WARNING — "Guarantee data is generic — needs branding at checkpoint"
20. Check guarantee strength:
    - Duration >= 60 days: strong
    - Duration 30-59 days: moderate
    - Duration < 30 days: log WARNING — "Short guarantee duration may weaken offer"

### Step 6: Validate Scarcity and Urgency Data

21. Check scarcity object:
    - Has `type` and `justification`: PASS
    - Has type but no justification: GAP — "Scarcity lacks justification"
    - Empty: GAP — "No scarcity data" (not critical — urgency section can be adjusted)
22. Check urgency object:
    - Has specific urgency reason: PASS
    - Generic urgency ("limited time"): GAP — "Urgency needs specific justification"
    - Empty: GAP — "No urgency data"

### Step 7: Validate Promise and Proof Availability

23. Check primary promise:
    - Present and non-empty: PASS
    - Has restatement bank with 3+ entries: EXCELLENT
    - Has restatement bank with 1-2 entries: PASS — Layer 3 will need to generate more
    - No restatement bank: GAP — Layer 3 will generate restatements from primary
24. Check proof inventory coverage:
    - Has proof assets relevant to deliverables: PASS
    - Has proof assets but not mapped to deliverables: PARTIAL — D-F-W-B-P proof elements will need creative mapping
    - No proof assets: WARNING — D-F-W-B-P will lack proof elements

### Step 8: Validate Handoff Continuity

25. Check emotional state at handoff:
    - Present and descriptive: PASS
    - Generic or vague ("good"): WARNING — "Emotional state too vague for offer entry calibration"
    - Missing: GAP — offer entry emotional tone will need to be inferred
26. Check continuity status:
    - Intact: PASS
    - Discontinuity flagged: WARNING — log specific discontinuity details

### Step 9: Determine Checkpoint Default and Compile Validation Report

27. Apply minimum viable offer check:
    - Product name present: YES/NO
    - At least 1 deliverable with description: YES/NO
    - Actual price present: YES/NO
    - Guarantee type present: YES/NO
    - If ALL four are YES: minimum viable = TRUE
    - If ANY is NO: minimum viable = FALSE — BLOCKING failure
28. Determine checkpoint default:
    - If completeness >= 95% and no critical gaps: default = CONFIRM
    - If completeness 60-94% with identified gaps: default = PARTIAL
    - If completeness < 60%: default = PARTIAL with prominent warning
    - If minimum viable = FALSE: default = PARTIAL with BLOCKING notice
29. Compile the full validation report with:
    - Field-by-field status (present, partial, absent, placeholder)
    - Critical gaps list (must be resolved at checkpoint)
    - Non-critical gaps list (can proceed without, quality may be lower)
    - Warnings list (data present but quality concerns)
    - Checkpoint default recommendation
    - Minimum viable offer status
    - D-F-W-B-P readiness score per deliverable and bonus

## QUALITY CRITERIA

| Criterion | Pass Condition |
|-----------|---------------|
| All fields checked | Every field in upstream_data has a validation status |
| Minimum viable offer assessed | Four minimum requirements explicitly checked |
| Critical gaps identified | All missing required fields listed in critical_gaps |
| Non-critical gaps identified | All missing optional fields listed in non_critical_gaps |
| Pricing math validated | Anchor ratio calculated, math consistency checked |
| Guarantee quality assessed | Generic vs. branded status determined |
| D-F-W-B-P readiness scored | Each deliverable and bonus scored for D-F-W-B-P potential |
| Checkpoint default determined | CONFIRM or PARTIAL with clear reasoning |
| Cross-reference consistency checked | Product name consistent across packages |
| Placeholder detection active | Template/placeholder values caught and flagged |
| No false confidence | Gaps are not minimized or hidden |
| No false alarm | Present data is not flagged as missing |

## FAILURE MODES

### Failure Mode 1: False Confidence — Missing Data Not Caught
- **Symptom:** Checkpoint default set to CONFIRM when critical data is missing
- **Cause:** Validator checks field existence but not field quality (empty string passes as "present")
- **Prevention:** Check for empty strings, null values, and placeholder text in addition to field existence
- **Recovery:** Add quality checks: `value != null && value != "" && !containsPlaceholder(value)`

### Failure Mode 2: False Alarm — Complete Data Flagged as Gaps
- **Symptom:** Checkpoint default set to PARTIAL when all data is actually present and valid
- **Cause:** Over-strict validation rules that flag acceptable data as insufficient
- **Prevention:** Define clear "pass" thresholds for each field — what constitutes "enough" data
- **Recovery:** Review validation rules against actual campaign data patterns; adjust thresholds

### Failure Mode 3: Pricing Math Discrepancy Not Caught
- **Symptom:** Retail value does not add up to sum of deliverable values + bonus values, causing downstream confusion
- **Cause:** Validator did not compare retail value against component values
- **Prevention:** Step 4, item 17 — validate math consistency with 10% tolerance
- **Recovery:** Flag discrepancy in warnings with specific numbers so human can resolve at checkpoint

### Failure Mode 4: Generic Guarantee Passes as Branded
- **Symptom:** Guarantee section written with "money-back guarantee" because validator did not catch generic language
- **Cause:** Validator checked for guarantee presence but not for branded naming
- **Prevention:** Step 5, item 19 — specifically check for generic guarantee phrases
- **Recovery:** Force guarantee through branding at checkpoint (0.4) if generic language detected

### Failure Mode 5: Minimum Viable Offer Check Too Lenient
- **Symptom:** Pipeline proceeds with insufficient data, producing incomplete or inaccurate offer copy
- **Cause:** Minimum viable check did not include guarantee, or accepted a deliverable with no description
- **Prevention:** Minimum viable requires: product name (non-placeholder), at least 1 deliverable WITH description, actual price (> 0), guarantee type (non-empty)
- **Recovery:** Tighten minimum viable criteria; add quality requirements to each minimum field
