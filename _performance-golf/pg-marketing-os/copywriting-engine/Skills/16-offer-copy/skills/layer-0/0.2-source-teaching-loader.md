## PURPOSE

The Source Teaching Loader ingests and indexes the two knowledge foundations that govern all offer copy generation: the SIN Offer Presentation template (from source-teachings/) and the TIER1 offer patterns extracted from 482+ elite controls in the vault. The SIN methodology provides the structural backbone — the exact sequence of chunks that compose an offer presentation. The TIER1 vault extractions provide the language layer — real patterns from controls that generated millions in revenue.

This loader creates a dual-indexed reference system. The SIN index maps the 18-chunk sequence with format requirements for each chunk. The TIER1 index catalogs offer_architecture patterns, guarantee patterns, CTA patterns, price presentation patterns, value demonstration patterns, and transition language patterns. Together, they give downstream microskills both the STRUCTURE (what to write and in what order) and the LANGUAGE (how elite copywriters actually wrote it).

Without this loader, downstream microskills would default to generic offer language. The SIN index prevents structural errors (wrong sequence, missing chunks). The TIER1 index prevents language errors (cliched phrasing, AI-default commercial language). Both are mandatory for offer copy that meets the CopywritingEngine quality standard.

## IDENTITY

**This microskill IS:**
- The SIN Offer Presentation methodology indexer — mapping the complete 18-chunk sequence
- The TIER1 vault offer pattern extractor — cataloging elite offer language patterns
- A dual-index builder providing both structure (SIN) and language (TIER1) references
- The D-F-W-B-P format codifier — indexing the sacred Deliverable-Feature-Why-Benefit-Proof format
- A pattern catalog builder for guarantee naming, CTA phrasing, price transitions, and value demonstration
- The quality baseline setter — establishing what "elite" offer copy looks like from real controls

**This microskill is NOT:**
- An upstream data loader — that is 0.1 (this loads teaching and pattern data, not campaign data)
- A draft writer — it indexes patterns for downstream use, not writes copy
- A creative generator — it catalogs existing elite patterns, not invents new ones
- A validator — it loads reference data for validators to use later
- A classifier — classification using these patterns is Layer 1's responsibility

## PROCESS

### Step 1: Load SIN Offer Presentation Template

1. Load the source teaching file: `E5AA_-M3E_-SIN-Offer-Presentation.md`
2. Parse the complete methodology document for structural directives
3. Extract the 18-chunk sequence in exact order:
   - Chunk 1: Product Introduction Line
   - Chunk 2: Deliverable Stack (D-F-W-B-P for each)
   - Chunk 3: Product Summary
   - Chunk 4: Bonus Transition
   - Chunk 5: Bonus Stack (D-F-W-B-P for each)
   - Chunk 6: Value Totaling
   - Chunk 7: Value Demonstration (3x "if all it did was...")
   - Chunk 8: Price Presentation (anchor cascade)
   - Chunk 9: Stack Review
   - Chunk 10: Savings Statement
   - Chunk 11: Risk Reversal Segue
   - Chunk 12: Guarantee Details
   - Chunk 13: CTA 1
   - Chunk 14: Consequence Amplification
   - Chunk 15: CTA 2
   - Chunk 16: Urgency Section
   - Chunk 17: CTA 3
   - Chunk 18: Optional Contrast Enhancement

### Step 2: Index SIN Chunk Sequence with Format Requirements

4. For each chunk, index:
   - `chunk_number`: position in sequence (1-18)
   - `chunk_name`: descriptive name
   - `required`: boolean (chunks 1-17 required, chunk 18 optional)
   - `format_requirements`: what the chunk must contain
   - `emotional_target`: what emotional state this chunk should evoke
   - `transition_to_next`: how this chunk connects to the next
   - `sin_template_language`: any specific phrasing from the template
5. Flag chunk dependencies: chunks that MUST reference output from previous chunks
   - Chunk 3 (Product Summary) references deliverables from Chunk 2
   - Chunk 6 (Value Totaling) references all deliverables and bonuses
   - Chunk 9 (Stack Review) re-lists everything from Chunks 2 and 5
   - Chunk 14 (Consequence Amplification) references pain from promise package

### Step 3: Index D-F-W-B-P Format Specification

6. Extract the D-F-W-B-P format definition:
   - **D (Deliverable):** "The first thing you get is [name]" — what the item is called
   - **F (Feature):** What the deliverable contains or does — the factual description
   - **W (Why):** Why this feature matters — the reason it is important to the prospect
   - **B (Benefit):** What the prospect gains — the personal impact on their life
   - **P (Proof):** Evidence that supports the deliverable — clinical, testimonial, authority
7. Index D-F-W-B-P transition language patterns from the SIN template:
   - First deliverable: "The first thing you get is..."
   - Subsequent deliverables: "The second thing you get is..." / "You also get..." / "Next..."
   - Bonus items: "And that's not all. When you grab [product], you also get..."
8. Index D-F-W-B-P depth variations:
   - Full depth: all 5 elements detailed (for primary deliverables)
   - Medium depth: D-F-B with brief W (for secondary deliverables)
   - Brief depth: D-F-B only (for minor bonuses)

### Step 4: Index Value Demonstration Pattern

9. Extract the "if all it did was..." value demonstration pattern:
   - Format: "If all [product] did was [benefit], would it be worth it? I would say yes, it would be worth it, because [compelling reason]."
   - Minimum iterations: 3
   - Each iteration must reference a different life area or benefit dimension
   - Each "why it would be worth it" reason must make the retail value feel like a bargain for that ONE benefit alone
10. Index benefit selection criteria:
    - Benefits should span different life domains (health + relationship + confidence; money + time + career)
    - Each benefit must be independently valuable enough to justify the retail price
    - Benefits must be traceable to the primary promise or supporting promises

### Step 5: Index Price Presentation Pattern

11. Extract the price anchor cascade pattern:
    - "But today, when you grab [product], it isn't gonna cost you $[retail]."
    - "It's not even gonna cost you $[middle anchor]." (optional)
    - "Instead, today, you get [product] for just $[actual price]."
12. Index supporting price presentation elements:
    - Stack review format: re-listing each deliverable + bonus with individual values
    - Savings statement: "That's [X]% off. That's a savings of over $[amount]."
    - Per-day breakdown: "That's less than [amount] per day" (optional)
    - Payment terms: "just [N] easy payments of $[amount]"

### Step 6: Load and Catalog TIER1 Offer Architecture Patterns

13. Scan TIER1 extraction reports for `offer_architecture` fields
14. Catalog the following pattern categories:

**Category A: Value Stack Patterns**
- How elite controls structure their deliverable presentations
- Common deliverable counts (3-7 for supplements, 4-8 for digital bundles)
- Value assignment approaches (individual pricing vs. bundle pricing)

**Category B: Price Psychology Patterns**
- Retail value anchor ratios (typical: 5x-12x actual price)
- Middle anchor usage frequency and positioning
- Price reveal transition language from elite controls
- Per-day breakdown language ("less than a cup of coffee per day")

**Category C: Guarantee Patterns**
- Branded guarantee names: "60-Day No-Questions-Asked Money-Back Guarantee" vs. "The Iron-Clad Triple Guarantee" vs. "The [Product] Promise"
- Guarantee strength hierarchy: extreme ("empty every bottle") > strong ("if you don't see results") > moderate ("if not completely satisfied") > standard ("money back")
- Guarantee confidence language: "I'm so confident..." / "That's how sure I am..."

**Category D: CTA Patterns**
- CTA 1 patterns: confidence-based ("Go ahead, click the button below, grab [product] right now")
- CTA 2 patterns: consequence-based ("There's no need to continue [pain]... you can change all of that today")
- CTA 3 patterns: urgency-based ("This is a limited time offer... if you want it, now is the time to act")
- Action language catalog: "click the button," "grab your copy," "secure your supply," "get started"

**Category E: Consequence Amplification Patterns**
- Pain reminder language between CTAs
- Current situation framing: "Think about where you'll be in 30 days if nothing changes..."
- Fear-of-missing language: "Every day you wait is another day of..."
- Contrast language: "You can keep doing what you've been doing... or..."

**Category F: Urgency Justification Patterns**
- Supply-based: "We can only produce [X] bottles per batch"
- Time-based: "This special pricing is only available during this launch"
- Demand-based: "We've already sold [X] units this week"
- Seasonal: "This offer ends [specific date/event]"
- Credibility-based: "Once our marketing partners see these results..."

### Step 7: Build Transition Language Catalog

15. From both SIN template and TIER1 extractions, build a transition language catalog:
    - Deliverable-to-deliverable transitions (minimum 5 variations)
    - Last-deliverable-to-summary transitions (minimum 3 variations)
    - Summary-to-bonus transitions (minimum 4 variations)
    - Bonus-to-bonus transitions (minimum 5 variations)
    - Last-bonus-to-value-totaling transitions (minimum 3 variations)
    - Value-totaling-to-value-demo transitions (minimum 3 variations)
    - Price-to-guarantee transitions (minimum 3 variations)
    - Guarantee-to-CTA transitions (minimum 3 variations)

### Step 8: Compile Dual Index Object

16. Assemble the complete `source_teaching_index` object with two sections:
    - `sin_index`: chunk sequence, D-F-W-B-P format, value demo pattern, price pattern
    - `tier1_index`: value stack patterns, price psychology, guarantee patterns, CTA patterns, consequence amplification, urgency justification, transition language catalog
17. Verify index completeness:
    - SIN: all 18 chunks mapped with format requirements
    - TIER1: minimum 3 patterns per category (A-F)
    - Transition catalog: minimum 3 variations per transition point
18. Mark index as ready for downstream consumption

## FAILURE MODES

### Failure Mode 1: SIN Template File Not Found
- **Symptom:** Source teaching file path does not resolve
- **Cause:** File not present in expected location or directory structure changed
- **Prevention:** Verify file path exists before attempting parse
- **Recovery:** BLOCKING failure. The SIN template is the structural foundation — without it, no offer copy can be generated. Log error and halt pipeline until template is located.

### Failure Mode 2: TIER1 Extractions Missing Offer Architecture Fields
- **Symptom:** Extraction reports loaded but `offer_architecture` fields are empty or absent
- **Cause:** Older extraction schema versions may not include offer_architecture
- **Prevention:** Filter extractions to v3 schema only, which includes offer_architecture
- **Recovery:** Use SIN template patterns as sole source. Flag TIER1 patterns as `limited` in index. Downstream microskills will rely more heavily on SIN template language, which may produce less varied output.

### Failure Mode 3: Insufficient Transition Language Variations
- **Symptom:** Fewer than 3 variations for one or more transition points
- **Cause:** TIER1 extractions did not contain enough transition examples for that specific point
- **Prevention:** Cast a wide net — scan ALL offer-related sections across ALL extractions, not just offer_architecture
- **Recovery:** Generate variations from the SIN template base language by combining with different connector phrases. Flag as `synthesized` rather than `extracted`. Quality will be lower but functional.

### Failure Mode 4: Guarantee Patterns All Generic
- **Symptom:** Extracted guarantee patterns are all "money-back guarantee" or "satisfaction guaranteed"
- **Cause:** Extraction parsing did not capture branded guarantee names, only generic terms
- **Prevention:** Search for guarantee branded names specifically, including surrounding context
- **Recovery:** Use the SIN template's guarantee presentation pattern as the primary reference. Generate branded name examples from product_name + guarantee_type combinations (e.g., "The [Product] Promise," "The [Duration]-Day [Adjective] Guarantee").

### Failure Mode 5: CTA Patterns Lack Emotional Diversity
- **Symptom:** All extracted CTA patterns use similar emotional appeals (e.g., all urgency-based)
- **Cause:** Extraction captured CTA text but not the emotional context surrounding each CTA
- **Prevention:** Catalog CTAs with their surrounding context (consequence amplification before, urgency after) to identify emotional positioning
- **Recovery:** Synthesize missing emotional CTA types from the SIN template patterns. Map: CTA 1 = confidence (from guarantee context), CTA 2 = consequence avoidance (from pain reminder context), CTA 3 = urgency (from scarcity context). Flag synthesized patterns.

      - "So you never have to wonder 'what do I say next?' again"

    description: "Evidence supporting the deliverable's value"
    template: "[Evidence type]: [specific proof point]"
    examples:

      - "These are the exact templates I used to build a 7-figure consulting practice"
