# Layer 2.2 — Proof-to-Claim Mapper

## Purpose

Map proof elements from the proof inventory to each engineered claim. Every claim must have at least one proof element supporting it. Prioritize "reason why" as the most versatile proof type, and ensure proof diversity across the argument.

## Execution Protocol

### Step 1: Build Claim-to-Proof Requirements Matrix

For each claim, identify:
- What the claim asserts (from claim_statement)
- What type of evidence would most powerfully support it
- Which proof_ids from the inventory are candidate matches

### Step 2: Apply Proof Type Prioritization

Per the E5 and Makepeace frameworks, proof types have a power hierarchy:

1. **Demonstration** — Show it working (most powerful)
2. **Reason Why / Because** — "The reason why is..." (most versatile)
3. **Strong Examples** — Specific examples showing results
4. **Before and Afters** — Transformation evidence
5. **Studies / Test Results** — Research backing
6. **Statistics** — Data support
7. **Authoritative Proof** — Expert/institutional backing
8. **Logic Flow** — "If X, then Y" reasoning

**Critical rule:** Every CPB chunk should include at least one "reason why" proof element. This is the most versatile type and the Makepeace framework identifies it as the single most important proof method.

### Step 3: Map Proof to Claims

For each claim, select proof elements:

```python
def map_proof_to_claim(claim, proof_inventory, proof_types):
    """Map the strongest available proof to each claim."""

    candidates = []
    for proof in proof_inventory["elements"]:
        relevance = assess_relevance(claim, proof)
        if relevance >= 0.6:  # minimum relevance threshold
            candidates.append({
                "proof_id": proof["proof_id"],
                "proof_type": proof["category"],
                "relevance": relevance,
                "strength": proof["five_dimension_scores"]["composite"],
                "content_summary": proof["content_summary"]
            })

    # Sort by: relevance * strength (combined score)
    candidates.sort(key=lambda x: x["relevance"] * x["strength"], reverse=True)

    # Select top candidates (1-3 per claim)
    selected = candidates[:3]

    # Ensure at least one "reason_why" if available
    has_reason_why = any(p["proof_type"] == "reason_why" for p in selected)
    if not has_reason_why:
        reason_why_candidates = [c for c in candidates if c["proof_type"] == "reason_why"]
        if reason_why_candidates:
            selected.append(reason_why_candidates[0])

    return selected
```

### Step 4: Validate Proof Traceability

Every selected proof element must have a valid `proof_id` that traces back to the proof-inventory-output.json. No fabricated proof.

```python
def validate_traceability(selected_proofs, lookup_index):
    """Every proof_id must exist in the inventory."""
    untraced = []
    for proof in selected_proofs:
        if proof["proof_id"] not in lookup_index:
            untraced.append(proof["proof_id"])
    return untraced  # must be empty
```

### Step 5: Assess Proof Distribution Across Argument

After mapping all claims, check the overall argument for:

| Check | Target | Issue if Violated |
|-------|--------|-------------------|
| Proof type diversity | ≥3 different proof types used | Argument feels one-dimensional |
| Reason why presence | In every CPB chunk | Weakest chunks lack explanation |
| Proof density balance | No chunk has 0 proof elements | Claim without proof = opinion |
| Knockout proof placement | Used in most impactful chunk | Wasted if buried in supporting chunk |
| No proof overuse | No single proof_id used >2 times | Repetition weakens argument |

### Step 6: Flag Proof Gaps

If any claim lacks a matching proof element in the inventory:
- Document the gap
- Specify what proof type is needed
- Trigger feedback bus request to 02-proof-inventory if gap is blocking

### Step 7: Generate "Reason Why" Supplements

For any chunk that lacks a "reason why" proof element from the inventory, generate a reason why bridge:

> "The reason why [claim] is because [mechanism explanation that logically supports the claim]"

This uses the mechanism explanation text and logical reasoning to create a "reason why" even when empirical proof is limited. Per Makepeace: "reason why" works even with brand-new, unproven mechanisms because it provides logical explanation.

## Quality Gates

- [ ] Every claim has at least 1 proof element mapped
- [ ] Every CPB chunk has at least 1 "reason why" proof element
- [ ] All proof_ids trace to valid inventory entries (untraced = 0)
- [ ] Minimum 3 different proof types used across the argument
- [ ] No single proof_id used more than 2 times
- [ ] Knockout proof placed in the most impactful chunk
- [ ] Proof gaps documented with severity assessment
- [ ] No chunks_without_proof (must be empty list)
- [ ] Distribution check shows adequate diversity

---

## Anti-Patterns

- **Proof stuffing:** Mapping 5+ proof elements to a single claim — diminishing returns, choose the strongest 2-3
- **Single-type argument:** All proof is "authority" or all "statistics" — arguments need variety
- **Fabricated proof:** Creating proof elements not in the inventory — violates traceability
- **Wasted knockout proof:** Using the strongest proof element on a "supporting" priority chunk
- **Missing reason why:** Any chunk without a "reason why" element lacks the most versatile explanation method
- **Proof-claim mismatch:** Mapping a study about weight loss to a claim about energy — relevance matters
