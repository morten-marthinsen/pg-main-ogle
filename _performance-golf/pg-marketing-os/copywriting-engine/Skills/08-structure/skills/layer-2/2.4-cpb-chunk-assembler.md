# Layer 2.4 — CPB Chunk Assembler

## Purpose

Combine engineered claims, mapped proof elements, and dimensionalized benefits into complete CPB chunk units. Each assembled chunk is a self-contained argument unit that advances the campaign thesis. This microskill integrates the outputs of 2.1, 2.2, and 2.3 into cohesive, scored chunk objects.

**Primary Persona:** The Legendary Copywriter — Master Copy Composite
**Secondary Persona:** Alex Rivera — Strategic Integration Director

---

## Execution Protocol

### Step 1: Merge Components by Chunk ID

For each chunk_id, pull together:
- The claim(s) from `engineered_claims`
- The proof elements from `proof_claim_mappings`
- The benefit triplet from `dimensionalized_benefits`
- The chunk assignment metadata from `chunk_assignments`

### Step 2: Validate Component Completeness

Every chunk MUST have:
- [ ] At least 1 claim (opinion_test = pass)
- [ ] At least 1 proof element (with valid proof_id)
- [ ] All three benefit dimensions (functional, dimensionalized, emotional)

If any component is missing, flag the chunk as incomplete and halt assembly for that chunk.

### Step 3: Assess Internal Chunk Coherence

Each assembled chunk must flow as a single argument unit. Score coherence:

> **Claim → Proof → Benefit must tell one story.**

Test: Read the claim, then the proof, then the benefit. Does it flow as a single thought? Or do the components feel disconnected?

**Coherence scoring dimensions:**

| Dimension | Weight | Scale | Description |
|-----------|--------|-------|-------------|
| Claim-proof alignment | 0.30 | 1-10 | Does the proof directly support the specific claim made? |
| Proof-benefit bridge | 0.25 | 1-10 | Does the benefit flow naturally from the proof? |
| Internal narrative | 0.20 | 1-10 | Does the chunk tell a single story from claim to benefit? |
| Thesis advancement | 0.15 | 1-10 | Does this chunk clearly advance the campaign thesis? |
| Conversational flow | 0.10 | 1-10 | Does it sound like spoken language when read aloud? |

**Minimum threshold:** Weighted average ≥ 6.0/10 (elevated to ≥ 7.0 at gate)

### Step 4: Structure Each Chunk

A complete CPB chunk follows this internal structure:

```
┌─────────────────────────────────┐
│  CLAIM                          │
│  [Declarative statement]        │
├─────────────────────────────────┤
│  PROOF                          │
│  [Evidence supporting claim]    │
│  • Reason why / Because...      │
│  • Study / Statistic / Example  │
│  • Authority / Demonstration    │
├─────────────────────────────────┤
│  BENEFIT                        │
│  [What this means for prospect] │
│  • Functional: So you can...    │
│  • Dimensionalized: Imagine...  │
│  • Emotional: Finally feel...   │
└─────────────────────────────────┘
```

The internal ordering is always: **Claim first → Proof second → Benefit third**

This is non-negotiable. The claim makes the assertion. The proof backs it up. The benefit shows why it matters.

### Step 5: Tag Chunk Metadata

For each assembled chunk, tag:
- `position`: integer (to be finalized by 3.1-chunk-sequencer, use chunk_assignment position as default)
- `gap_addressed`: the primary gap this chunk answers
- `thesis_advancement`: one sentence describing how this chunk moves the prospect closer to accepting the thesis
- `proof_density`: sparse (1 proof), moderate (2 proofs), heavy (3+ proofs)
- `dominant_proof_type`: the strongest proof type in this chunk
- `emotional_intensity`: low, moderate, high (based on benefit emotional resonance)

### Step 6: Compile Complete Chunk Array

Assemble all chunks into a single array, ordered by default position from chunk assignments. This ordering is preliminary — Layer 3 will finalize sequence.

## Quality Gates

- [ ] Every chunk has claim + proof + benefit (all three components)
- [ ] Zero incomplete chunks (all_chunks_complete = true)
- [ ] Every chunk coherence score ≥ 6.0/10
- [ ] Average coherence across all chunks ≥ 7.0/10
- [ ] Every claim passes opinion test
- [ ] Every proof_id traces to valid inventory entry
- [ ] Every benefit has all three dimensions
- [ ] Internal structure follows Claim → Proof → Benefit order
- [ ] Thesis advancement documented for each chunk
- [ ] No chunk has 0 proof elements

---

## Anti-Patterns

- **Disconnected components:** Claim about speed, proof about safety, benefit about ease — the three must align
- **Proof before claim:** Starting with evidence before making the assertion — the claim frames the evidence
- **Benefit before proof:** Jumping to "imagine this" before proving the claim is true — proof earns the benefit
- **Missing thesis link:** A chunk that doesn't clearly advance the campaign thesis — every chunk must contribute
- **Franken-chunk:** Combining components from different gap questions into one incoherent chunk
