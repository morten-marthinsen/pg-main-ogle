# 1.5 Mechanism Constraint Check

## Purpose
Validate that the derived root cause is SOLVABLE by the mechanism we have (or can develop). A root cause without a mechanism solution is useless for copy. This check ensures the derivation points to something the product can actually address.

---

## The Constraint Principle

**Root Cause must be MATCHED to Mechanism capability**

```
INVALID:
Root Cause: "Your genetics make you store fat"
Mechanism: "A diet plan"
Problem: Mechanism can't change genetics

VALID:
Root Cause: "Your hormones override your diet"
Mechanism: "Hormone-regulating protocol"
Match: Mechanism addresses root cause directly
```

---

## Input Requirements

From previous derivation microskills:
- Candidate root cause(s) from:
  - 1.2 Symptom Convergence
  - 1.3 False Belief Identification
  - 1.4 Hidden Layer Discovery

From project context:
- Product/mechanism being sold (known or in development)
- Mechanism capabilities and limitations

---

## Execution Protocol

### Step 1: State the Candidate Root Causes

List all root cause candidates discovered:

```
ROOT CAUSE CANDIDATES:
1. [From Convergence]: "[Root cause statement]"
2. [From False Belief]: "[Root cause statement]"
3. [From Hidden Layer]: "[Root cause statement]"
```

---

### Step 2: Define Mechanism Capabilities

Document what the mechanism CAN do:

```
MECHANISM CAPABILITY MAP:
Mechanism Name: "[Name]"
Mechanism Type: [product | method | technology | protocol | information]

Core Capabilities:
├── Can do: "[Capability 1]"
├── Can do: "[Capability 2]"
├── Can do: "[Capability 3]"

Limitations:
├── Cannot do: "[Limitation 1]"
├── Cannot do: "[Limitation 2]"
├── Cannot do: "[Limitation 3]"
```

**Vault Examples**:

| Swipe | Mechanism | Can Do | Cannot Do |
|-------|-----------|--------|-----------|
| pg_sf1_vsl_001 | Square Face Technology | Square clubface automatically via engineering | Fix swing mechanics |
| pg_ssts_vsl_001 | Simple Strike Sequence | Teach low point understanding + drill | Give more practice time |
| pg_fixd_vsl_001 | Swing Fix Sequence | Correct first move with specific drills | Require hours of practice |

---

### Step 3: Run the Match Test

For each root cause, test mechanism fit:

```
MATCH TEST:
Root Cause: "[Statement]"
Mechanism: "[Name]"

QUESTION 1: Does mechanism DIRECTLY address root cause?
├── YES → Proceed
└── NO → Root cause may need adjustment

QUESTION 2: Is mechanism the MOST DIRECT path to solving root cause?
├── YES → Strong match
└── NO → Consider if better mechanism exists

QUESTION 3: Can we credibly claim mechanism solves root cause?
├── YES → Match validated
└── NO → Risk of broken promise

MATCH SCORE: Strong | Moderate | Weak | No Match
```

---

### Step 4: Evaluate Match Quality

Use this matrix:

| Match Type | Description | Action |
|-----------|-------------|--------|
| **Direct Solve** | Mechanism directly eliminates root cause | Ideal—proceed |
| **Compensate** | Mechanism works around root cause | Valid if honest |
| **Enable** | Mechanism helps user address root cause | Valid for information products |
| **Partial** | Mechanism addresses part of root cause | Adjust root cause scope |
| **Mismatch** | Mechanism doesn't touch root cause | STOP—derivation invalid |

**Vault Examples**:

| Swipe | Root Cause | Mechanism | Match Type |
|-------|-----------|-----------|------------|
| pg_sf1_vsl_001 | "Technique breaks down under pressure" | Square Face Technology | Direct Solve (tech does it automatically) |
| pg_ssts_vsl_001 | "Misconception about low point" | Simple Strike Sequence | Enable (teaches understanding) |
| pg_one_vsl_001 | "Dual problem: mental + design" | ONE Wedge + PCT | Direct Solve (addresses both) |
| pg_fixd_vsl_001 | "Off-track first move" | Swing Fix Sequence | Enable (drills correct it) |

---

### Step 5: Adjust Root Cause if Needed

If match is weak, you have three options:

```
ADJUSTMENT OPTIONS:

OPTION A: Narrow the root cause
├── Before: "All your swing flaws"
└── After: "The ONE swing flaw that causes the others"
    └── (Now mechanism can credibly address ONE thing)

OPTION B: Reframe the root cause
├── Before: "Your clubs aren't good enough"
└── After: "Your clubs require perfect technique you don't have"
    └── (Now mechanism—forgiving clubs—solves the reframe)

OPTION C: Compound the root cause
├── Before: "Mental deceleration"
└── After: "Mental deceleration + wedge design requires perfection"
    └── (Now mechanism can address BOTH—more complete story)
```

---

### Step 6: Document Final Mechanism-Root Cause Pair

```
VALIDATED PAIR:
Root Cause: "[Final root cause statement]"
Mechanism: "[Mechanism name]"
Match Type: [Direct Solve | Compensate | Enable | Partial]

Match Logic:
"[Root cause] happens because [explanation].
[Mechanism] solves this by [how it addresses root cause].
This is a [match type] because [reasoning]."

Proof of Match:
- "[Evidence mechanism can do this]"
- "[Evidence from testing/results/credentials]"
```

---

## Quality Gates

Before passing to proof constraint check:

- [ ] At least one root cause has strong or moderate mechanism match
- [ ] Match logic is clearly articulated
- [ ] If adjustments made, they don't distort the research truth
- [ ] Mechanism capability claims are honest and defensible
- [ ] No "mismatch" root causes passed through

---

## Critical Principle

**Never distort root cause to fit mechanism.**

If research reveals a root cause the mechanism can't address, the honest options are:
1. Choose a different (valid) root cause from derivation
2. Develop a mechanism that CAN address it
3. Scope the promise appropriately

Forcing a false root cause → mechanism match creates broken promises and failed copy.

---

## Handoff

Pass Mechanism Constraint Report to:
- **1.6 Proof Constraint Check** (verify root cause can be proven)
- **1.7 Derivation Synthesis** (with validated root cause)
