# 1.2: Incompatibility Filter

## Purpose

Apply incompatibility rules to filter out invalid combinations from the raw Cartesian product. Remove combinations where the hook, body, CTA, visual, or platform create structural conflicts that cannot produce a coherent ad. Log every removed combination with the specific reason for removal.

## Inputs

- **RAW-COMBINATIONS.yaml:** From 1.1 -- full Cartesian product of all combinations
- **PLATFORM-SPECS.yaml:** From 0.3 -- platform-specific constraints
- **COPY-CATALOG.yaml:** From 0.1 -- copy element details (hook types, body word counts, CTA types)
- **ASSET-CATALOG.yaml:** From 0.2 -- asset format specs (duration, aspect ratio, file type)
- **0.0.1 Vertical Profile Output:** Compliance exclusions from vertical profile

## Process

```
STEP 1: LOAD ALL INPUTS
  - Read RAW-COMBINATIONS.yaml (all enumerated combinations)
  - Read PLATFORM-SPECS.yaml (platform constraints)
  - Read COPY-CATALOG.yaml (copy element details)
  - Read ASSET-CATALOG.yaml (asset format specs)
  - Read 0.0.1 output (compliance exclusions)

STEP 2: APPLY HARD INCOMPATIBILITY RULES
  For EVERY combination in RAW-COMBINATIONS.yaml, check each rule:

  INC-01: VIDEO HOOK + STATIC IMAGE VISUAL
    - IF hook requires video/motion AND visual is static image: REMOVE
    - Reason: "Cannot have moving hook on a still image"

  INC-02: SOUND-DEPENDENT HOOK + SOUND-OFF PLATFORM
    - IF hook relies on audio (voiceover, music cue, sound effect)
      AND platform/placement defaults to sound-off (Meta feed = 85% sound-off): REMOVE
    - Reason: "Sound-dependent hook on sound-off platform"

  INC-03: BODY LENGTH > PLATFORM DURATION LIMIT
    - IF body word count implies duration > platform max duration: REMOVE
    - Calculate: estimated_duration = word_count / 150 * 60 (approx. 150 words/min speaking rate)
    - Compare against platform duration limits per placement
    - Reason: "Body exceeds platform duration limit ([X]s body > [Y]s platform max)"

  INC-04: HORIZONTAL VISUAL + VERTICAL-ONLY PLATFORM
    - IF visual aspect ratio is 16:9 AND platform is 9:16-only (TikTok feed): REMOVE
    - UNLESS visual can be cropped (mark as NEEDS_ADAPTATION instead of REMOVE)
    - Reason: "Horizontal visual on vertical-only platform"

  INC-05: SWIPE-UP CTA + NON-STORIES FORMAT
    - IF CTA uses "Swipe Up" AND ad format is NOT Stories/Reels: REMOVE
    - Reason: "Swipe Up CTA only valid in Stories/Reels format"

  INC-06: CAROUSEL BODY + VIDEO VISUAL
    - IF body is structured as carousel AND visual is video: REMOVE
    - Reason: "Carousel format is image-based, not video"

  INC-07: SHORT BODY + COMPLEX MECHANISM
    - IF body duration < 10 seconds AND body type is detailed mechanism explanation: REMOVE
    - Reason: "Mechanism cannot be explained in [X] seconds"

STEP 3: APPLY COMPLIANCE EXCLUSIONS
  - Check each combination against vertical-specific compliance constraints
  - Remove combinations that violate health claims, financial disclaimers, etc.
  - Reason: "Compliance exclusion: [specific violation]"

STEP 4: APPLY SOFT INCOMPATIBILITY RULES (FLAG, not REMOVE)
  For surviving combinations, check soft incompatibility rules:

  SINC-01: UGC HOOK + POLISHED VISUAL
    - IF hook is UGC style AND visual is polished studio: FLAG
    - Override allowed if: "Visual is intentionally elevated UGC style"
    - Reason: "Tonal mismatch: UGC hook with polished visual"

  SINC-02: DATA HOOK + LIFESTYLE VISUAL
    - IF hook uses data/statistics AND visual is pure lifestyle: FLAG
    - Override allowed if: "Data is overlaid on lifestyle footage"
    - Reason: "Data hook with lifestyle-only visual"

  SINC-03: HUMOR HOOK + AUTHORITY BODY
    - IF hook uses humor AND body is authority-driven: FLAG
    - Override allowed if: "Humor-to-authority tonal shift is intentional"
    - Reason: "Tonal shift from humor hook to authority body"

  SINC-04: TESTIMONIAL HOOK + NO TESTIMONIAL IN BODY
    - IF hook references a testimonial AND body contains no testimonial: FLAG
    - Override allowed if: "Hook references testimonial that appears later in body"
    - Reason: "Testimonial hook without testimonial in body"

STEP 5: LOG ALL REMOVALS AND FLAGS
  For every removed combination:
    - Record variant ID
    - Record which rule triggered removal (INC-01 through INC-07 + compliance)
    - Record specific reason text
    - Record the conflicting elements (e.g., "Hook H03 (UGC) + Visual V02 (polished studio)")

  For every flagged combination:
    - Record variant ID
    - Record which soft rule triggered flag (SINC-01 through SINC-04)
    - Record specific concern
    - Record override conditions

STEP 6: COMPUTE FILTER STATISTICS
  - Total combinations before filtering
  - Total removed (hard incompatibility)
  - Total flagged (soft incompatibility)
  - Total surviving (for coherence validation)
  - Per-rule removal counts (which rules removed the most)
  - Per-concept surviving counts
  - Per-platform surviving counts

STEP 7: WRITE INCOMPATIBILITY-LOG.yaml
  - Write all removals and flags with full reasoning
  - Write filter statistics
  - Update combination status: REMOVED / FLAGGED / VALID
```
