# 1.3: Platform-Specific Mapper

## Purpose

For every surviving combination (after incompatibility filtering), verify platform-specific format requirements are met. Determine whether each combination is READY for the assigned platform, needs format ADAPTATION (e.g., aspect ratio crop), or is BLOCKED due to irreconcilable format issues. This mapper ensures every variant that proceeds to coherence validation can actually be deployed on its target platform.

## Inputs

- **INCOMPATIBILITY-LOG.yaml:** From 1.2 -- surviving combinations after filtering
- **RAW-COMBINATIONS.yaml:** From 1.1 -- full combination details
- **PLATFORM-SPECS.yaml:** From 0.3 -- format requirements per platform
- **ASSET-CATALOG.yaml:** From 0.2 -- asset format specs (resolution, aspect ratio, duration, file type, file size)

## Process

```
STEP 1: LOAD ALL INPUTS
  - Read surviving combinations from INCOMPATIBILITY-LOG.yaml
  - Read platform specs from PLATFORM-SPECS.yaml
  - Read asset details from ASSET-CATALOG.yaml

STEP 2: FOR EACH SURVIVING COMBINATION, CHECK PLATFORM FORMAT
  For each combination with status VALID or FLAGGED:

  CHECK A: ASPECT RATIO
    - Get visual's aspect ratio from ASSET-CATALOG.yaml
    - Get platform's required/supported aspect ratios from PLATFORM-SPECS.yaml
    - IF visual aspect ratio matches platform requirement: PASS
    - IF visual aspect ratio can be cropped to match: NEEDS_ADAPTATION
      (record: "Crop [source ratio] to [target ratio]")
    - IF visual aspect ratio cannot be adapted: BLOCKED
      (record: "Cannot adapt [source ratio] for [platform]")

  CHECK B: DURATION
    - Get visual/video duration from ASSET-CATALOG.yaml
    - Get platform's duration limit (per placement) from PLATFORM-SPECS.yaml
    - IF duration <= platform limit: PASS
    - IF duration can be trimmed (within 20% over limit): NEEDS_ADAPTATION
      (record: "Trim from [X]s to [Y]s max")
    - IF duration exceeds limit by >20%: BLOCKED
      (record: "Duration [X]s exceeds [platform] limit of [Y]s by [Z]%")

  CHECK C: FILE TYPE
    - Get asset file type from ASSET-CATALOG.yaml
    - Get platform's supported file types from PLATFORM-SPECS.yaml
    - IF file type is supported: PASS
    - IF file type can be converted: NEEDS_ADAPTATION
      (record: "Convert [source type] to [target type]")
    - IF file type is incompatible: BLOCKED

  CHECK D: FILE SIZE
    - Get asset file size from ASSET-CATALOG.yaml
    - Get platform's max file size from PLATFORM-SPECS.yaml
    - IF file size <= max: PASS
    - IF file size can be compressed: NEEDS_ADAPTATION
      (record: "Compress from [X]MB to under [Y]MB")
    - IF file size far exceeds limit: BLOCKED

  CHECK E: SOUND HANDLING
    - Get platform's sound default (sound-on/sound-off)
    - IF platform is sound-off (Meta feed):
      - Verify hook works without sound
      - Verify text overlay or captions are available
      - IF copy relies on sound and no text alternative: NEEDS_ADAPTATION
        (record: "Add text overlay/captions for sound-off playback")
      - IF hook is purely audio-dependent with no visual substitute: BLOCKED

  CHECK F: TEXT OVERLAY REQUIREMENTS
    - IF platform has text overlay rules (Meta 20% text rule):
      - Verify visual doesn't exceed text limit
      - IF over limit: NEEDS_ADAPTATION
        (record: "Reduce text overlay to meet [platform] [X]% text rule")

STEP 3: ASSIGN PLATFORM MAPPING STATUS
  For each combination:
    - IF all checks PASS: status = READY
    - IF any check is NEEDS_ADAPTATION (but none BLOCKED): status = NEEDS_ADAPTATION
      Record: list of required adaptations
    - IF any check is BLOCKED: status = BLOCKED
      Record: list of blocking issues

STEP 4: COMPUTE MAPPING STATISTICS
  - Total combinations mapped
  - Total READY (deploy without changes)
  - Total NEEDS_ADAPTATION (can deploy with modifications)
  - Total BLOCKED (cannot deploy without new assets)
  - Per-platform breakdown of READY/NEEDS_ADAPTATION/BLOCKED
  - Most common adaptation needed
  - Most common blocking reason

STEP 5: WRITE PLATFORM-MAPPING.yaml
  - Write per-combination platform mapping status
  - Write adaptation requirements where applicable
  - Write mapping statistics
```

## Quality Checks

- [ ] Every surviving combination from 1.2 has been mapped (100% coverage)
- [ ] All 6 platform format checks applied to every combination
- [ ] READY status only assigned when ALL checks pass
- [ ] NEEDS_ADAPTATION includes specific adaptation instructions (not vague)
- [ ] BLOCKED includes specific blocking reasons
- [ ] Per-platform breakdown computed correctly
- [ ] Common adaptations and blocking reasons identified
- [ ] No combination marked READY when a format check fails
- [ ] Sound handling checked for sound-off platforms (especially Meta feed)
- [ ] Text overlay rules checked where applicable

## Dependencies

- **Required before this runs:** 1.2 (Incompatibility Filter) must complete
- **Runs after 1.2, in parallel with 1.4 in some flows but ideally before 1.4**
- **This feeds:** 1.4 (Combination Validator), Layer 2 coherence validation (only READY + NEEDS_ADAPTATION proceed), 3.3 (File Manifest Builder), 3.4 (Platform Grouper)
