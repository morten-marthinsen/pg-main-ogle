# 3.2: Testing Priority Assigner

## Purpose

Assign a testing priority tier (TIER_1 / TIER_2 / TIER_3) to every variant in the matrix based on four weighted factors: Arena score for the parent concept, coherence score from Layer 2, hook type performance data from A01 intelligence, and platform priority from A03 format strategy. TIER_1 variants are tested first -- they represent the highest-probability combinations.

## Inputs

- **VARIANT-ID-MAP.yaml:** From 3.1 -- all variant IDs with concept assignments
- **ARENA-SCORES.yaml:** From 0.4 -- per-concept Arena scores
- **FULL-UNIT-COHERENCE.yaml:** From 2.4 -- per-combination coherence scores
- **PLATFORM-SPECS.yaml:** From 0.3 -- platform priority rankings
- **A01 Market Intelligence:** Hook type performance benchmarks (if available from upstream)

## Process

```
STEP 1: LOAD ALL INPUTS
  - Read VARIANT-ID-MAP.yaml (all variants with concept assignments)
  - Read ARENA-SCORES.yaml (per-concept Arena scores)
  - Read FULL-UNIT-COHERENCE.yaml (per-combination coherence scores)
  - Read PLATFORM-SPECS.yaml (platform priority rankings)
  - Read A01 hook performance data (if available)

STEP 2: COMPUTE PRIORITY SCORE FOR EACH VARIANT
  Apply the testing priority algorithm:

  PRIORITY_SCORE = (0.35 * arena_score_normalized) +
                   (0.25 * coherence_score_normalized) +
                   (0.25 * hook_performance_score) +
                   (0.15 * platform_priority_score)

  WHERE:
    arena_score_normalized = parent concept's Arena score / 10
      - C1 Arena score = 8.5 -> 0.85
      - C2 Arena score = 7.2 -> 0.72

    coherence_score_normalized = full-unit coherence score / 10
      - Coherence score 8.3 -> 0.83

    hook_performance_score = hook type's benchmark performance from A01 / 10
      - If A01 data unavailable: default to 0.5 (neutral)
      - Hook types with historical high CTR get higher scores
      - Hook types with no performance data get neutral score

    platform_priority_score:
      - Primary platform = 1.0
      - Secondary platform = 0.7
      - Tertiary platform = 0.4

STEP 3: RANK ALL VARIANTS BY PRIORITY SCORE
  - Sort all variants by PRIORITY_SCORE descending
  - Record rank position for each variant

STEP 4: ASSIGN TIERS
  Based on rank position:
    TIER_1: Top 20% of variants by PRIORITY_SCORE
      - These are tested FIRST
      - Represent highest-probability combinations
    TIER_2: Middle 50% of variants by PRIORITY_SCORE
      - These are tested SECOND
      - Represent solid but not top-priority combinations
    TIER_3: Bottom 30% of variants by PRIORITY_SCORE
      - These are tested THIRD (or held in reserve)
      - Represent lower-probability but still valid combinations

  Edge cases:
    - If total variants = 30, TIER_1 = 6, TIER_2 = 15, TIER_3 = 9
    - If total variants = 90, TIER_1 = 18, TIER_2 = 45, TIER_3 = 27
    - Round up for TIER_1 (never fewer than the percentage would indicate)

STEP 5: DOCUMENT TIER_1 VARIANTS
  For every TIER_1 variant, document:
    a. Why this variant is prioritized (cite specific scores):
       - Arena score: [X] (concept rank: [Y])
       - Coherence score: [X]
       - Hook performance: [X] (based on [data source])
       - Platform priority: [X] ([primary/secondary])
       - Composite PRIORITY_SCORE: [X]
    b. Recommended testing context:
       - Which ad set this should run in
       - Which audience segment to target
    c. What this variant tests:
       - Hook type being tested
       - Visual style being tested
       - CTA approach being tested
    d. Expected performance hypothesis:
       - Why this combination might outperform

STEP 6: WRITE TESTING-PRIORITY.yaml
```

- [ ] Priority score computed for every variant using the 4-factor algorithm
- [ ] Algorithm weights match specification (0.35/0.25/0.25/0.15)
- [ ] Arena scores correctly normalized (divided by 10)
- [ ] Coherence scores correctly normalized (divided by 10)
- [ ] Platform priority scores correctly assigned (1.0/0.7/0.4)
- [ ] Hook performance scores sourced from A01 or defaulted to 0.5
- [ ] Tier assignment follows percentage breakdown (20%/50%/30%)
- [ ] All variants ranked and assigned to a tier
- [ ] Every TIER_1 variant has full documentation (why prioritized, context, hypothesis)
- [ ] No variants without tier assignment
