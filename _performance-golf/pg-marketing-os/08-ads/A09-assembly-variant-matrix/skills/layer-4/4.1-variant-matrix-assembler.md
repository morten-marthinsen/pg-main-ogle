# 4.1: Variant Matrix Assembler

## Purpose

Assemble the primary structured output: VARIANT-MATRIX.yaml. This is the COMPLETE database of every testable variant with all metadata: variant ID, concept details, hook (text + type + category), body (summary + word count + framework), CTA (text + type), visual (description + type + format specs), platform, platform specs, testing priority, coherence scores, and file manifest paths. This file is what A10 (Pre-Launch Scoring) and A11 (Launch Package) consume.

**CRITICAL: For matrices with 30+ variants, chunked assembly is REQUIRED.** A single write operation cannot reliably output 90+ variant records. Use 15+ write operations, appending in structured chunks.

## Inputs

- **VARIANT-ID-MAP.yaml:** From 3.1 -- all variant IDs with component mappings
- **TESTING-PRIORITY.yaml:** From 3.2 -- tier assignments and priority scores
- **FILE-MANIFEST.yaml:** From 3.3 -- file paths and assembly status
- **FULL-UNIT-COHERENCE.yaml:** From 2.4 -- coherence scores and per-dimension verdicts
- **COPY-CATALOG.yaml:** From 0.1 -- hook text, body summary, CTA text
- **ASSET-CATALOG.yaml:** From 0.2 -- visual descriptions, format specs
- **ARENA-SCORES.yaml:** From 0.4 -- concept names and Arena scores
- **PLATFORM-SPECS.yaml:** From 0.3 -- platform-specific ad format details

## Process

```
STEP 1: LOAD ALL INPUTS
  - Read all Layer 3 outputs (3.1, 3.2, 3.3)
  - Read Layer 2 coherence results (2.4)
  - Read Layer 0 catalogs (0.1, 0.2, 0.3, 0.4)
  - Build complete variant record for each variant by cross-referencing

STEP 2: BUILD COMPLETE VARIANT RECORDS
  For each variant, assemble ALL fields from the output schema:

  variant_id: from VARIANT-ID-MAP.yaml
  concept:
    id: from VARIANT-ID-MAP
    name: from ARENA-SCORES.yaml
    arena_score: from ARENA-SCORES.yaml
  hook:
    id: from VARIANT-ID-MAP
    text: VERBATIM from COPY-CATALOG.yaml
    hook_type: from COPY-CATALOG.yaml (32-type taxonomy)
    hook_category: from COPY-CATALOG.yaml (A-J)
  body:
    id: from VARIANT-ID-MAP
    summary: from COPY-CATALOG.yaml (1-2 sentence summary)
    word_count: from COPY-CATALOG.yaml
    script_framework: from COPY-CATALOG.yaml (PAS/AIDA/BAB/etc.)
  cta:
    id: from VARIANT-ID-MAP
    text: VERBATIM from COPY-CATALOG.yaml
    cta_type: from COPY-CATALOG.yaml
  visual:
    id: from VARIANT-ID-MAP
    description: from ASSET-CATALOG.yaml
    visual_type: from ASSET-CATALOG.yaml
    format:
      aspect_ratio: from ASSET-CATALOG.yaml
      duration_seconds: from ASSET-CATALOG.yaml (or null for static)
      file_type: from ASSET-CATALOG.yaml
  platform: from VARIANT-ID-MAP
  platform_specs:
    ad_format: from PLATFORM-SPECS.yaml (feed/stories/reels/in_stream)
    sound_requirement: from PLATFORM-SPECS.yaml
    text_overlay_required: from PLATFORM-SPECS.yaml
    max_file_size_mb: from PLATFORM-SPECS.yaml
  testing_priority: from TESTING-PRIORITY.yaml (TIER_1/TIER_2/TIER_3)
  coherence:
    overall_score: from FULL-UNIT-COHERENCE.yaml
    hook_body: from 2.1
    body_cta: from 2.2
    hook_visual: from 2.3
    full_unit: from 2.4
  file_manifest:
    copy_file: from FILE-MANIFEST.yaml
    asset_files: from FILE-MANIFEST.yaml
    assembly_status: from FILE-MANIFEST.yaml

STEP 3: WRITE METADATA HEADER
  Write the metadata block:
    project, created timestamp, skill name,
    total_variants, concepts count, platforms list,
    tier_1_count, tier_2_count, tier_3_count

STEP 4: WRITE VARIANT RECORDS IN CHUNKS
  CRITICAL: Do NOT attempt to write all variants in a single operation.

  Chunked Assembly Protocol:
    - Chunk size: 5-10 variants per write operation
    - For 30 variants: minimum 4-6 write operations
    - For 90 variants: minimum 10-18 write operations
    - After each chunk: verify what was written
    - After all chunks: verify total variant count matches expected

  Write Order:
    1. TIER_1 variants first (highest priority)
    2. TIER_2 variants second
    3. TIER_3 variants third

STEP 5: VERIFY ASSEMBLED MATRIX
  After all chunks written:
    - Count total variants in VARIANT-MATRIX.yaml
    - Verify count matches total from VARIANT-ID-MAP.yaml
    - Verify all required fields are populated (not empty/placeholder)
    - Verify file size is reasonable (large matrices = large files)
    - Spot-check 5 random variants for data accuracy

STEP 6: VALIDATE DOWNSTREAM READINESS
  Verify the assembled matrix meets A10/A11 requirements:
    - Every variant has all fields populated
    - Every file path in file_manifest is a verified path
    - Testing priority is assigned to all variants
    - Coherence scores are present for all variants
    - Platform specs are present for all variants
```
