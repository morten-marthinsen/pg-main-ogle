# 0.5: Input Validator

**Version:** 1.0
**Layer:** 0 (Foundation)
**Type:** Validation
**Model:** haiku
**Dependencies:** 0.1-0.4 outputs (all must complete before this runs)
**Output:** layer-0-outputs/0.5-input-validator.md

---

## Purpose

Validate all inputs are present and above minimum thresholds. Confirm: performance data loaded with all required fields, variant matrix mapped (every data row maps to a variant), A10 predictions loaded, benchmarks loaded. Calculate: total variants with data, total days of data, total spend, spend per variant vs 3x AOV threshold. Flag variants with insufficient data. This is the GATE KEEPER for Layer 0 -- nothing proceeds to Layer 1 without this validation passing.

---

## Process

### Step 1: Verify All Loading Outputs Exist

```
CHECK file existence:
  [ ] 0.1-performance-data-loader.md EXISTS
  [ ] 0.2-variant-matrix-loader.md EXISTS
  [ ] 0.3-prediction-loader.md EXISTS
  [ ] 0.4-benchmark-loader.md EXISTS

IF ANY file missing:
  --> HALT: "Loading step [X] did not produce output. Cannot validate."
  --> Return to missing loading step

VERIFY each file meets minimum size (1KB):
IF ANY file below minimum:
  --> HALT: "Loading step [X] output is below minimum size. Likely incomplete."
```

### Step 2: Cross-Validate Data Integrity

```
COMPARE outputs from 0.1, 0.2, 0.3:

  VARIANT ID CROSS-REFERENCE:
    performance_variant_ids = set of variant_ids from 0.1 performance data
    matrix_variant_ids = set of variant_ids from 0.2 variant matrix
    prediction_variant_ids = set of variant_ids from 0.3 predictions

    matched = performance_variant_ids ∩ matrix_variant_ids
    unmapped_performance = performance_variant_ids - matrix_variant_ids
    untested_matrix = matrix_variant_ids - performance_variant_ids
    without_predictions = matched - prediction_variant_ids

  IF unmapped_performance > 0:
    --> CRITICAL: [X] performance data rows cannot be mapped to creative elements
    --> Document unmapped variant_ids
    --> These rows CANNOT be used for element attribution (Layer 2.5)
    --> They CAN be used for aggregate analysis if metrics are valid

  IF len(matched) == 0:
    --> HALT: "Zero variants match between performance data and variant matrix."
    --> Check for ID format mismatch (e.g., "V-001" vs "001" vs "variant_001")
```

### Step 3: Calculate Per-Variant Data Sufficiency

```
FOR EACH matched variant:
  CALCULATE:
    variant_total_spend = sum of spend across all dates
    variant_total_impressions = sum of impressions across all dates
    variant_days_of_data = count of unique dates with data
    variant_total_conversions = sum of conversions

  CLASSIFY data sufficiency:
    IF variant_total_spend >= 3 * AOV
       AND variant_total_impressions >= 5000
       AND variant_days_of_data >= 3:
      --> SUFFICIENT_DATA
    ELSE:
      --> INSUFFICIENT_DATA
      --> Reason: [which threshold not met]

COUNT:
  variants_sufficient = count where SUFFICIENT_DATA
  variants_insufficient = count where INSUFFICIENT_DATA
```

### Step 4: Determine Operational Mode

```
CALCULATE campaign_days = max(date) - min(date) from performance data

MODE DETERMINATION:
  IF campaign_days < 7 AND total_spend < $5000:
    --> HALT: "Insufficient data window. Need 7+ days or $5K+ spend."
    --> Cannot execute A12 at all

  IF campaign_days >= 7 AND campaign_days < 30 AND campaign is still running:
    --> MODE = MID_FLIGHT
    --> Propagation (Layer 3.5) will be SKIPPED
    --> Output = MID-FLIGHT-LEARNING.md (15KB+ minimum)

  IF campaign_days >= 30 OR campaign has ended:
    --> MODE = POST_CAMPAIGN
    --> Full pipeline including propagation
    --> Output = PERFORMANCE-LEARNING-REPORT.md (50KB+ minimum)

  IF campaign_days >= 5 AND total_spend >= $5000 AND campaign_days < 7:
    --> MODE = MID_FLIGHT (early exception)
    --> Log: "Early Mid-Flight analysis triggered by high spend ($[X] in [Y] days)"
```

### Step 5: Produce Validation Summary

```
COMPILE validation_summary:
  data_readiness:
    performance_data: LOADED / NOT_LOADED
    variant_matrix: LOADED / NOT_LOADED
    predictions: LOADED / NOT_LOADED / PARTIAL
    benchmarks: LOADED / PARTIAL / MINIMAL
  data_coverage:
    total_variants_with_data: integer
    variants_with_sufficient_data: integer
    variants_with_insufficient_data: integer
    total_spend: float
    total_days: integer
    total_conversions: integer
    total_impressions: integer
  cross_reference:
    matched_variants: integer
    unmapped_variants: integer
    prediction_coverage: float (% of variants with A10 predictions)
  mode: MID_FLIGHT / POST_CAMPAIGN
  gate_0_status: PASS / FAIL
  gate_0_failure_reasons: [string] (if FAIL)
```

## Quality Checks

1. All four loading outputs (0.1-0.4) exist and meet minimum size
2. Variant ID cross-reference completed with matched count > 0
3. Unmapped variants = 0 (or documented with attempted resolution)
4. Per-variant data sufficiency calculated for every matched variant
5. AOV and 3x AOV threshold defined (required for sufficiency calculation)
6. Operational mode determined (MID_FLIGHT or POST_CAMPAIGN)
7. Gate 0 status is binary PASS or FAIL -- no invented statuses
8. If FAIL, failure reasons are specific and actionable
