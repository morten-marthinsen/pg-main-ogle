# 0.1: Performance Data Loader

**Version:** 1.0
**Layer:** 0 (Foundation)
**Type:** Data Loading
**Model:** haiku
**Dependencies:** Client-provided performance data (CSV, XLSX, platform exports, or API data)
**Output:** layer-0-outputs/0.1-performance-data-loader.md

---

## Purpose

Load raw performance data provided by the client. Accept CSV, XLSX, platform exports, or API data. Normalize all platform-specific formats into a unified schema. Flag any missing required fields. This is the raw data ingestion step -- no analysis, no interpretation, just loading and validating the raw data.

---

## Inputs

```yaml
input:
  from_client:
    performance_data: "[raw data files -- CSV, XLSX, platform exports, or API data]"
    # Accept data from: Meta Ads Manager, Google Ads, TikTok Ads, YouTube Ads, or other platforms
    # Each row should represent a variant-day combination at minimum
```

---

## Process

### Step 1: Identify Data Source Format

```
INSPECT provided data files:
  - File type (CSV, XLSX, JSON, platform export format)
  - Platform origin (Meta, Google, TikTok, YouTube, other)
  - Data granularity (daily, weekly, campaign-level)
  - Column headers / field names

IF no data files provided:
  --> HALT: "No performance data provided. Cannot execute A12."
  --> Request data from client with format guidance

IF data is campaign-level aggregates ONLY (no per-variant breakdown):
  --> HALT: "Per-variant data required. Campaign-level aggregates are insufficient for A12 analysis."
  --> Request per-variant export from client
```

### Step 2: Map to Unified Schema

```
FOR EACH data file:
  MAP platform-specific columns to unified schema:

  UNIFIED SCHEMA:
    - variant_id: string (must map to A09 variant matrix)
    - platform: enum[meta, google, tiktok, youtube, other]
    - date: ISO date (YYYY-MM-DD)
    - impressions: integer
    - reach: integer (may be null for some platforms)
    - link_clicks: integer
    - conversions: integer
    - spend: float (USD)
    - revenue: float (USD, may be null if not tracked)
    - video_views_3s: integer (null for non-video)
    - video_completions: integer (null for non-video)
    - engagement_actions: integer (likes + comments + shares)

  PLATFORM-SPECIFIC MAPPING:
    Meta: "Link Clicks" --> link_clicks, "Results" --> conversions, "Amount Spent" --> spend
    Google: "Clicks" --> link_clicks, "Conversions" --> conversions, "Cost" --> spend
    TikTok: "Clicks" --> link_clicks, "Conversions" --> conversions, "Spend" --> spend
    YouTube: "Clicks" --> link_clicks, "Conversions" --> conversions, "Cost" --> spend

IF a required field cannot be mapped:
  --> FLAG as missing field
  --> Do NOT fabricate data
  --> Document which fields are missing and from which platform
```

### Step 3: Data Quality Checks

```
FOR EACH data row:
  VALIDATE:
    - variant_id is not null or empty
    - date is valid ISO date
    - impressions >= 0
    - spend >= 0
    - conversions >= 0
    - No negative values in count fields
    - No impossible values (conversions > clicks, clicks > impressions)

  FLAG ANOMALIES:
    - Sudden spend spikes (> 3x daily average)
    - Zero-impression periods (days with spend but no impressions)
    - Negative metrics (indicates data quality issue)
    - Duplicate rows (same variant + date combination)
    - Revenue without conversions or conversions without revenue
```

### Step 4: Calculate Data Coverage

```
CALCULATE:
  - total_data_rows: count of all valid data rows
  - date_range: earliest date to latest date
  - days_of_data: count of unique dates
  - platforms_present: list of unique platforms
  - variants_with_data: count of unique variant_ids
  - total_spend: sum of all spend
  - total_conversions: sum of all conversions
  - total_impressions: sum of all impressions

  - missing_fields_summary: {field_name: count of rows where missing}
  - anomalies_summary: {anomaly_type: count}
```

## Output Schema

```yaml
output:
  performance_data_load:
    load_status: enum[COMPLETE, PARTIAL, FAILED]

      file_count: integer
      file_types: [string]

      total_data_rows: integer

        start: ISO_date
        end: ISO_date

      variants_with_data: integer
      total_spend: float
      total_conversions: integer
      total_impressions: integer

      mapped_fields: [string]
      missing_fields: [string]
      platform_specific_gaps: [{platform: string, missing: [string]}]

      anomalies_found: integer
      anomaly_details: [{type: string, count: integer, severity: enum[LOW, MEDIUM, HIGH]}]
```

---

## Quality Checks

1. Raw data files exist and are readable
2. All data rows have variant_id populated
3. All data rows have date populated
4. Required numeric fields (impressions, spend, conversions) are present
5. No impossible values detected (or anomalies documented)
6. Date range spans at least 7 days (or exception for high-spend early analysis)
7. Platform-to-unified-schema mapping is documented for every field
8. Data coverage metrics calculated and recorded

## Dependencies

- Client-provided performance data files
- No upstream A12 dependencies (this is the first loading step)
