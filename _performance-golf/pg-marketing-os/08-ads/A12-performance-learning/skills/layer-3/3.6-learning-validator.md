# 3.6: Learning Validator

**Version:** 1.0
**Layer:** 3 (Learning Extraction)
**Type:** Validation
**Model:** opus
**Dependencies:** 3.1-3.5 outputs (all learning extraction microskills)
**Output:** layer-3-outputs/3.6-learning-validator.md

---

## Purpose

Validate all extracted learnings from 3.1-3.5. Each learning must have: (a) WHAT -- specific observation with data. (b) WHY -- causal hypothesis. (c) HOW -- actionable recommendation for next campaign. (d) CONFIDENCE -- statistical basis (HIGH if based on sufficient data, MEDIUM if directional, LOW if suggestive). (e) SCOPE -- vertical-specific or universal? Campaign-specific or generalizable? Reject learnings that are vague, lack data backing, or fail the actionability test. Minimum 10 validated learnings required.

## Process

### Step 1: Collect All Learnings

```
READ all 5 learning extraction outputs (3.1-3.5)
COMPILE master learning list:
  - H-001, H-002, ... (hook learnings)
  - V-001, V-002, ... (visual learnings)
  - P-001, P-002, ... (platform learnings)
  - F-001, F-002, ... (fatigue learnings)
  - PR-001, PR-002, ... (prediction learnings)

TOTAL learnings collected: [count]
IF total < 10:
  --> WARNING: "Only [X] learnings collected. Minimum is 10."
  --> Return to 3.1-3.5 for additional extraction before validation.
  --> Identify which domains (hook, visual, platform, fatigue, prediction)
      have fewer than expected learnings.
```

### Step 2: Validate Each Learning -- Format Compliance

```
FOR EACH learning:

  CHECK FORMAT COMPLIANCE (all 6 fields required):
    [ ] WHAT field populated (not empty, not placeholder)
    [ ] WHY field populated
    [ ] HOW field populated
    [ ] CONFIDENCE field populated (must be HIGH, MEDIUM, or LOW)
    [ ] SCOPE field populated (vertical + campaign scope)
    [ ] PROPAGATION TARGET field populated (specific engine file(s))

  IF ANY field missing:
    --> REJECT learning
    --> Document: "Learning [ID] rejected: missing [field_name]"
    --> Return to source microskill for completion
```

### Step 3: Validate Each Learning -- Data Backing

```
FOR EACH learning:

  CHECK DATA BACKING:
    WHAT field must contain SPECIFIC DATA:
      PASS: "UGC testimonial hooks achieved 34% hook rate vs 18% for curiosity hooks"
      FAIL: "Video hooks performed well" (no specific data)
      FAIL: "Some hook types outperformed others" (vague)

    WHY field must contain CAUSAL HYPOTHESIS (not just correlation):
      PASS: "UGC bypasses the ad filter because users process them as social content"
      FAIL: "Because it just worked better" (not a hypothesis)
      FAIL: "Correlation suggests relationship" (correlation != causation explanation)

    HOW field must be ACTIONABLE (specific enough to change a decision):
      PASS: "Allocate 40% of hooks to D3 type, up from 20%"
      FAIL: "Use more of what worked" (not specific)
      FAIL: "Consider UGC" (too vague)

    CONFIDENCE must match data basis:
      HIGH: requires >= 5 variants with sufficient data supporting the pattern
      MEDIUM: requires >= 3 variants OR directional data from sufficient variants
      LOW: fewer than 3 variants OR preliminary data only

  IF data backing insufficient:
    --> DOWNGRADE confidence (HIGH --> MEDIUM, MEDIUM --> LOW)
    --> If LOW with no supporting data: REJECT learning
```

### Step 4: Validate Each Learning -- Actionability Test

```
FOR EACH learning:

  ACTIONABILITY TEST:
    "Could someone reading only the HOW field make a different decision
    in the next campaign than they would have without this learning?"

    PASS EXAMPLES:
      - "Allocate 40% of Meta hooks to D3 type" (specific percentage, specific type)
      - "Refresh UGC variants every 14 days, polished every 8 days" (specific cadence)
      - "Increase A10 UGC bonus from +0.5 to +1.2" (specific model change)

    FAIL EXAMPLES:
      - "Test more hooks" (what kind? how many?)
      - "Improve creative quality" (how specifically?)
      - "Spend more on winners" (which threshold? what increment?)

  IF fails actionability test:
    --> Return to source microskill for specificity improvement
    --> Do NOT accept vague recommendations
```

### Step 5: Validate Learning Uniqueness

```
CHECK for duplicate or overlapping learnings:

  FOR EACH pair of learnings:
    IF learning_A and learning_B make essentially the same point:
      --> MERGE into one stronger learning
      --> Combine evidence from both
      --> Keep the more specific HOW recommendation

  FOR EACH learning:
    IF learning contradicts another learning:
      --> Flag conflict
      --> Document both with evidence
      --> Recommend resolution (usually the one with stronger data wins)
```

### Step 6: Produce Validation Report

```
COMPILE validation_report:

  SUMMARY:
    total_learnings_submitted: integer
    validated_and_accepted: integer
    rejected_format: integer (missing fields)
    rejected_data: integer (insufficient backing)
    rejected_actionability: integer (too vague)
    downgraded_confidence: integer
    merged_duplicates: integer
    conflicts_flagged: integer

  VALIDATED LEARNING MASTER LIST:
    # Re-numbered sequentially: L-001 through L-[N]
    For each validated learning:
      original_id: [H-001 / V-002 / etc.]
      validated_id: L-[sequential]
      domain: [hook / visual / platform / fatigue / prediction]
      title: [title]
      confidence: [HIGH / MEDIUM / LOW]
      propagation_target: [engine file(s)]

  DISTRIBUTION CHECK:
    Hook learnings: [count]
    Visual learnings: [count]
    Platform learnings: [count]
    Fatigue learnings: [count]
    Prediction learnings: [count]

    IF any domain has 0 validated learnings:
      --> WARNING: "No validated learnings in [domain]. Return to 3.X for additional extraction."

  PRIORITY RANKING:
    Rank all validated learnings by impact potential:
    1. [L-XXX] -- [title] -- [impact rationale]
    2. [L-XXX] -- [title] -- [impact rationale]
    ...

  GATE_3 VERDICT:
    IF validated_count >= 10: PASS
    IF validated_count < 10: FAIL -- "[10 - count] more learnings needed"
```

## Quality Checks

1. All learnings from 3.1-3.5 reviewed (none skipped)
2. Format compliance checked for all 6 required fields
3. Data backing validated (specific numbers, not vague claims)
4. Actionability test applied (recommendations specific enough to act on)
5. Duplicates merged, conflicts flagged
6. Minimum 10 validated learnings (Gate 3 requirement)
7. All 5 domains represented in validated learnings (or gaps documented)
8. Priority ranking reflects actual impact potential
9. Gate 3 verdict is binary PASS or FAIL
