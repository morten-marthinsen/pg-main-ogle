# 3.1: Testing Sequence Planner

**Version:** 1.0
**Layer:** 3 (Testing Strategy)
**Type:** Strategy / Planning
**Model:** opus
**Persona:** PERSONA_PERFORMANCE_STRATEGIST
**Dependencies:** 2.3 (Tier Assignment), 0.4 (Campaign Brief)
**Output:** testing_sequence.md

---

## Purpose

Design the testing sequence — which variants launch in which order, over what timeline. The sequence determines how quickly data arrives, how efficiently budget is spent, and how rapidly winners are identified and scaled. The sequence must account for platform-specific ad review timelines, minimum data accumulation periods, campaign timeline constraints, and Tier 2 activation triggers.

## Process

### Step 1: Establish Timeline Framework

```
LOAD campaign_timeline_weeks from 0.4

CALCULATE available testing cycles:
  Each testing cycle = ~1.5 weeks (3-5 days data accumulation + kill/scale decisions + Tier 2 activation)
  total_testing_cycles = FLOOR(campaign_timeline_weeks / 1.5)

MAP timeline to testing phases:
  PHASE 1 (Week 1): Launch ALL Tier 1 variants simultaneously
    - Goal: Maximum learning velocity
    - All Tier 1 variants compete head-to-head
    - Equal budget split (see 3.2)

  PHASE 2 (Week 2-3): Kill losers, activate Tier 2
    - Kill underperforming Tier 1 variants (per kill criteria from 3.3)
    - Activate Tier 2 variants as replacements
    - Scale winners (per scale criteria from 3.4)

  PHASE 3 (Week 3+): Iterate and optimize
    - Winning variants get hook swaps (new hooks, same concept)
    - Losing approaches are abandoned (entire concept retired)
    - Fresh concepts from revised Tier 3A variants may enter
    - Creative refresh cycle begins

IF campaign_timeline < 3 weeks:
  WARNING: "Short campaign timeline limits testing cycles. Only [X] testing
    cycles available. Consider reducing Tier 1 count for faster kill decisions."
```

### Step 2: Account for Platform Review Timelines

```
PLATFORM REVIEW DELAYS:
  Meta: 24-48 hours (sometimes up to 72 hours for first-time advertisers)
  TikTok: 24-72 hours (longer for health/finance verticals)
  Google: 24-48 hours (generally consistent)
  YouTube: 24-48 hours (through Google Ads)

FOR EACH Tier 1 variant:
  IDENTIFY target platform
  CALCULATE: earliest data collection start = launch_date + review_delay

  IF platform mix:
    NOTE: Meta variants will typically start collecting data 24 hours before
    TikTok variants. Account for this when comparing cross-platform performance.

ADJUST timeline:
  Week 1 Day 1: Submit all Tier 1 variants
  Week 1 Day 2-3: Variants begin going live (staggered by platform)
  Week 1 Day 3-7: Data accumulation (minimum 72 hours before any kill decision)
  Week 2 Day 1-3: First kill/scale decisions
  Week 2 Day 3-5: Tier 2 activation begins
```

### Step 3: Design Tier 2 Activation Sequence

```
DETERMINE Tier 2 activation triggers:

TRIGGER 1: Tier 1 variant killed
  When: A Tier 1 variant hits kill threshold
  Action: Activate highest-ranked Tier 2 variant for that platform
  Budget: Reallocate killed variant's daily spend to new activation

TRIGGER 2: Tier 1 saturation
  When: All Tier 1 variants tested for 7+ days
  Action: Activate next batch of Tier 2 variants
  Budget: From reserve (20-30% held back)

TRIGGER 3: Performance gap
  When: Tier 1 data reveals a platform/format gap
  Action: Activate Tier 2 variants that fill the gap
  Budget: Strategic reallocation

PRIORITY ORDER for Tier 2 activation:
  Activate in ranking order (rank #[first Tier 2] first)
  UNLESS a specific platform/format gap exists, then priority shifts

FOR EACH Tier 2 variant:
  ASSIGN: expected activation window (Week 2-3 typically)
  ASSIGN: activation trigger type
  ASSIGN: budget source (killed variant reallocation OR reserve)
```

### Step 4: Plan Iteration Strategy (Week 3+)

```
WINNING VARIANT ITERATION:
  For each winning Tier 1 variant (still performing at Week 3):
    - Plan: Hook swap iteration (same concept, new hook)
    - Timeline: Week 4+ for hook swaps to enter testing
    - Source: A02 Hook Strategy alternative hooks

LOSING APPROACH RETIREMENT:
  For each losing concept (all variants from that concept killed):
    - Action: Retire the concept entirely
    - Reallocate all budget to winning concepts + Tier 2

TIER 3A RE-ENTRY (if revised):
  Timeline: Week 4+ (earliest)
  Condition: Only if revised by A07 and re-scored
  Entry: As Tier 2 variants (not directly to Tier 1)

CREATIVE REFRESH FORECAST:
  Based on fatigue predictions from 1.5:
    - First refresh needed: [earliest predicted fatigue date]
    - Refresh priority: highest-spend variants first
    - Refresh type: Hook swap (quick) or concept refresh (requires A02-A09)
```

### Step 5: Build Visual Testing Timeline

```
CONSTRUCT visual timeline:

WEEK 1:
  Day 1: Submit [N] Tier 1 variants
  Day 2-3: Variants go live (platform review complete)
  Day 3-7: Data accumulation (NO kill decisions)
  Day 7: First performance review

WEEK 2:
  Day 8-10: Kill decisions on underperformers
  Day 10-12: Tier 2 variants activated
  Day 12-14: Scale decisions on winners

WEEK 3:
  Day 15-17: Second kill/scale cycle
  Day 17-21: Tier 2 performance review
  Day 21: Creative refresh planning begins

WEEK 4+:
  Ongoing optimization cycle
  Hook swaps enter testing
  Revised Tier 3A variants may enter as Tier 2

KEY MILESTONES:
  - First data: Day 3 (earliest)
  - First kill decision: Day 7 (earliest — minimum 72hr data window + review)
  - First Tier 2 activation: Day 8-10
  - First scale decision: Day 10-14
  - First creative refresh: Week [X] (based on fatigue predictions)
```

## Quality Checks

1. All Tier 1 variants included in Week 1 launch
2. Platform review timelines accounted for (24-72 hours)
3. Minimum 72-hour data window before any kill decision
4. Tier 2 activation triggers defined with specific conditions
5. Tier 2 activation order follows ranking priority
6. Timeline fits within campaign duration
7. Creative refresh forecast connected to fatigue predictions
8. Visual timeline is specific (dates, not just "soon")
9. Iteration strategy covers winning variant evolution AND losing concept retirement
