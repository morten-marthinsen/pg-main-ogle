# 3.1-email-validator.md

> **Layer:** 3 (Validation & Output)
> **Skill:** E1-email-writer
> **Purpose:** Score the Arena-selected email against 9-criterion email-quality-rubric.md. Calculate weighted score. Minimum 7.5.
> **Inputs:** Human-selected email from Arena (Layer 2.5), `routing_decision` from 1.1, `structural_template` from 1.2, email-quality-rubric.md
> **Outputs:** Validated email with quality scores, structural compliance report, and pass/fail determination

---

## FUNCTION

Validate the human-selected email against the 9-criterion quality rubric used in the Arena. This is the FINAL quality gate before the email is packaged for downstream skills (E2 subject lines, E3 assembly). The validator checks both qualitative criteria (scoring rubric) and structural compliance (body type, ratio, word count, paragraph brevity). Minimum weighted score is 7.5.

**Key Principle:** Validation is not a rubber stamp. The Arena-selected email has already been judged, but this validation confirms the selected version (which may include human edits) still meets all structural and quality requirements. A human might select a great-sounding email that technically violates a structural constraint. This skill catches that.

---

## THE 9 QUALITY CRITERIA

These are the SAME criteria used in ARENA-LAYER.md. Scoring follows the identical rubric.

| Criterion | Weight | Definition |
|-----------|--------|------------|
| **Opening Hook Strength** | 15% | First 3 lines create immediate engagement |
| **Content-to-Pitch Ratio** | 15% | 70-80% content / 20-30% pitch |
| **Bridge/Transition Smoothness** | 15% | Content-to-pitch transition quality |
| **Body Type Structural Compliance** | 10% | Follows assigned template |
| **Voice Consistency** | 10% | Persona voice maintained throughout |
| **Entertainment/Engagement Value** | 10% | Reader would enjoy even without clicking |
| **CTA Clarity and Strength** | 10% | Clear action, emotionally compelling |
| **Paragraph Brevity** | 5% | Max 3 sentences, usually 1-2 |
| **Niche Appropriateness** | 10% | Language and content fit the niche |

## VALIDATION PROTOCOL

```
EXECUTE 3.1-email-validator:

  STEP 1: Load Selected Email
    1. READ the human-selected email from Arena output
    2. READ routing_decision from 1.1 (body type and structural requirements)
    3. READ structural_template from 1.2 (word allocations and constraints)
    4. LOAD email-quality-rubric.md (if exists as separate file)

  STEP 2: Structural Compliance Check
    These are BINARY checks -- pass or fail:

    2A. BODY TYPE COMPLIANCE
      [ ] Email follows assigned body type structure
      [ ] All required structural elements present
      [ ] Elements in correct order
      IF FAIL: Flag specific missing/wrong elements

    2B. CONTENT-TO-PITCH RATIO
      1. CALCULATE content word count (opening + body)
      2. CALCULATE pitch word count (bridge + CTA)
      3. CALCULATE ratio
      [ ] Content is 70-80% of total
      [ ] Pitch is 20-30% of total
      IF FAIL: Flag exact percentages and violation

    2C. WORD COUNT
      1. COUNT total words
      [ ] Total within target range +/-20%
      IF FAIL: Flag actual vs. target

    2D. PARAGRAPH BREVITY
      1. SCAN all paragraphs
      [ ] No paragraph exceeds 3 sentences
      [ ] Most paragraphs are 1-2 sentences
      IF FAIL: Flag specific paragraphs that violate

    2E. OPENING LENGTH
      1. COUNT opening lines
      [ ] Opening is 3 lines or fewer
      IF FAIL: Flag opening line count

    2F. CTA LENGTH
      1. COUNT CTA sentences
      [ ] CTA is 3 sentences or fewer
      IF FAIL: Flag CTA sentence count

  STEP 3: Quality Scoring (9 Criteria)
    FOR each criterion:
      1. SCORE 1-10 using ARENA-LAYER.md rubric (same rubric)
      2. PROVIDE specific evidence for score
      3. FLAG if below 6.0 (critical gap)

    CALCULATE weighted_total:
      weighted_total = (
        opening_hook * 0.15 +
        content_ratio * 0.15 +
        bridge_smoothness * 0.15 +
        body_type_compliance * 0.10 +
        voice_consistency * 0.10 +
        entertainment_value * 0.10 +
        cta_strength * 0.10 +
        paragraph_brevity * 0.05 +
        niche_appropriateness * 0.10
      )

  STEP 4: Anti-Slop Scan
    1. SCAN full email text for banned words/phrases
    2. CHECK against ARENA-LAYER.md anti-slop list
    3. CHECK against email-specific banned phrases
    4. IF any slop detected: FLAG each instance with location

  STEP 5: Pass/Fail Determination

    IF weighted_total >= 7.5 AND all structural checks pass AND zero slop:
      RESULT: PASS
      CREATE LAYER_3_COMPLETE.yaml

    IF weighted_total >= 7.5 BUT structural violations exist:
      RESULT: CONDITIONAL -- structural issues must be fixed
      DO NOT create checkpoint
      REPORT: Specific fixes needed

    IF weighted_total < 7.5:
      RESULT: FAIL
      OPTIONS:
        A: Return to Arena for different selection
        B: Request human revision guidance
        C: Regenerate with new approach

    IF slop detected:
      RESULT: REVISION NEEDED
      REPORT: Each slop instance with suggested replacement

  STEP 6: Output
    WRITE full validation report to output file
```

## FAILURE MODES

| Mode | Trigger | Remediation |
|------|---------|-------------|
| FAIL | Weighted score below 7.5 | Return to Arena or request human guidance |
| FAIL | Structural violation (ratio, word count, body type) | Fix specific structural issues |
| REVISION | Slop words detected | Replace each instance and re-validate |
| REVISION | Critical gap (any criterion below 6.0) | Address specific weak criterion |
| PASS | All checks pass, score >= 7.5, zero slop | Proceed to 3.2 output packaging |

---

## ANTI-PATTERNS (FORBIDDEN)

1. **Score inflation** -- Do not inflate scores to pass the 7.5 threshold. If the email isn't good enough, it isn't good enough.
2. **Ignoring structural violations** -- A high quality score does not excuse structural non-compliance. Both must pass.
3. **Rubber-stamping Arena selection** -- Arena selection is a human preference signal. This validation is an independent quality check.
4. **Skipping anti-slop scan** -- Every word must be checked. Zero tolerance.
5. **Creating checkpoint despite violations** -- LAYER_3_COMPLETE.yaml can ONLY be created on full PASS.
