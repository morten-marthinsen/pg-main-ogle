# Layer 2.1 -- Issue Clustering

## Purpose

Aggregate all issues from the 4 Layer 1 audit dimensions (structural audit, voice consistency, pitch ratio, SL-body alignment) and cluster them by type and severity. Create a prioritized revision plan that routes P1/P2 clusters to Arena and P3/P4 clusters to direct fix. The clustering matters because related issues across multiple emails should be addressed together in a single Arena instance, not as separate runs.

### Step 2: Cluster Issues by Type

Group related issues into clusters for efficient resolution:

```
CLUSTERING RULES:

  CLUSTER BY ISSUE TYPE (not by email):
    All bridge/transition issues → CLUSTER-BRIDGE
    All voice drift issues → CLUSTER-VOICE
    All ratio violations → CLUSTER-RATIO
    All SL misalignments → CLUSTER-SL-ALIGNMENT
    All structural template violations → CLUSTER-STRUCTURE
    All opening hook issues → CLUSTER-HOOKS
    All CTA issues → CLUSTER-CTA

  WHY CLUSTER BY TYPE:
    - Arena competitors can focus on ONE problem type
    - Solutions from one email can inform fixes for others
    - Consistency of fix approach across the sequence
    - More efficient than running separate Arena per email

  CLUSTER SEVERITY:
    A cluster's severity = its HIGHEST severity issue.
    If CLUSTER-BRIDGE has 2 P2 issues and 3 P3 issues:
      → Cluster severity = P2 → Arena required

  CLUSTER SIZE:
    If a cluster has > 8 affected emails:
      → Split into 2 sub-clusters (max 8 emails per Arena run)
      → Ensures Arena competitors have manageable scope
```

### Step 3: Assign Resolution Method

```
FOR each cluster:

  IF cluster.severity IN [P1, P2]:
    resolution_method = "arena"
    arena_config:
      mode: "editorial_revision"
      rounds: 2  # 2 rounds + audience evaluation
      competitors: 7
      scope: "[list of affected email positions]"
      focus: "[the issue type being fixed]"
      criteria: "[email-specific Arena criteria from AGENT.md]"

  IF cluster.severity IN [P3, P4]:
    resolution_method = "direct_fix"
    fix_instructions:
      - email_position: integer
        fix_description: string
        target_criterion: string

  VALIDATION:
    IF any P1 or P2 issue is assigned "direct_fix":
      HALT -- P1/P2 issues MUST use Arena. No exceptions.
```

### Step 4: Create Prioritized Revision Plan

Order clusters for execution:

```
REVISION ORDER:
  1. P1 clusters (critical -- fix first)
  2. P2 clusters (major -- fix second)
  3. P3 clusters (minor -- fix third)
  4. P4 clusters (cosmetic -- fix last)

WITHIN same severity level:
  Order by number of affected emails (highest first -- broadest impact)

DEPENDENCY CHECK:
  If fixing one cluster might affect another:
    Note the dependency in the plan.
    Example: fixing voice issues may also resolve some bridge issues.
```

## Failure Modes

| Failure | Remediation |
|---------|-------------|
| Zero issues found across all 4 audits | Valid outcome. Proceed to Layer 3 with baseline = final. Log: "No issues found -- sequence passes editorial." |
| All issues are P4 cosmetic | Valid. Apply direct fixes, no Arena needed. |
| More than 50% of emails have P1 issues | Flag for human: "Sequence quality is below editorial scope. May need regeneration (return to E1) rather than editorial fixes." |
| Issue belongs to multiple categories | Assign to the PRIMARY category. Note secondary category. It will be fixed once, addressing both. |
| Cluster too large (>8 emails) | Split into sub-clusters of max 8 emails each. |
