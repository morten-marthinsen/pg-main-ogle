# Layer 1.3 — Asset Inventory

## Purpose

Inventory all available content assets that can be mapped to email body types. Categorize assets by body type fit, tag each with content richness (1-10), and calculate per-type counts. The asset inventory directly determines which body types are available for sequence design and which email positions get the richest content.

---


## Execution Protocol

### Step 1: Scan All Available Assets

Systematically scan the loaded context for content that maps to each of the 7 email body types:

| Body Type | Code | What to Look For |
|-----------|------|------------------|
| **Story** | ST | Discovery stories, origin stories, transformation narratives, mechanism story, case studies, personal anecdotes |
| **Testimonial** | TM | Customer testimonials, before/after stories, case studies with outcomes, social proof |
| **Quote-Opener** | QO | Expert quotes, authority endorsements, research quotes, contrarian expert statements |
| **List-Based** | LB | Data points, statistics, research findings, numbered benefits, comparison data, ingredient lists |
| **Q&A** | QA | Common objections, frequently asked questions, reader questions, skeptic concerns |
| **Contrarian** | CT | Villain angles, industry criticisms, myth-busting, conventional wisdom challenges, root cause reframes |
| **Negative Response** | NR | Hater feedback, negative reviews to address, critic objections, troll responses |

### Step 2: Tag Each Asset with Body Type Fit

For each identified asset, assign:

```yaml
asset_entry:
  id: string  # unique identifier
  content_summary: string  # 1-2 sentence description
  source: string  # where this came from (research, proof, brief, etc.)
  body_type_fit: enum[ST, TM, QO, LB, QA, CT, NR]
  secondary_fit: enum[ST, TM, QO, LB, QA, CT, NR]  # optional alternate use
  content_richness: integer  # 1-10
```

### Step 3: Score Content Richness (1-10)

Rate each asset's content richness based on how much email content it can generate:

| Score | Criteria |
|-------|----------|
| 9-10 | Complete narrative with specific details, names, numbers, emotional arc — can fuel an entire email |
| 7-8 | Strong content with good specifics — can fuel 70%+ of an email body |
| 5-6 | Moderate content — can serve as the core of an email with supplementation |
| 3-4 | Thin content — can be a supporting element but not the core |
| 1-2 | Minimal content — a single data point, quote, or reference only |

**Mode A richness calibration:**
- Verbatim testimonials with outcomes → 8-10
- Research quotes with context → 6-8
- CPB chunks from structure → 7-9
- Mechanism explanation text → 7-9
- Data points without narrative → 3-5
- Expert names without quotes → 1-3

**Mode B richness calibration:**
- Detailed testimonial text → 7-9
- Brief-mentioned proof ("we have clinical studies") → 3-4
- Brief-mentioned story ("founder discovered this...") → 4-6
- No detail provided → 1-2

### Step 4: Calculate Per-Type Counts and Weights

Aggregate assets by body type:

```yaml
type_summary:
  ST:
    count: integer
    avg_richness: float
    top_asset_id: string
    recommended_weight: float  # proportion of emails that should use this type
  TM:
    count: integer
    avg_richness: float
    top_asset_id: string
    recommended_weight: float
  # ... for all 7 types
```

**Weight calculation rules:**
- Types with more assets AND higher richness get higher weight
- Minimum weight for any available type: 0.05 (at least 1 email in a 20-email sequence)
- Maximum weight for any single type: 0.35 (never more than 35% of emails)
- Types with zero assets: weight = 0 (cannot assign what doesn't exist)

### Step 5: Identify Body Type Gaps

Flag body types with zero or very low asset counts:

```
FOR EACH body type:
  IF count == 0:
    → gap_severity = "critical" — this type CANNOT be used in the sequence
  IF count == 1 AND avg_richness < 5:
    → gap_severity = "moderate" — this type can be used once at most
  IF count >= 2 AND avg_richness >= 5:
    → gap_severity = "none" — this type is available for regular use
```

### Step 6: Generate Recommended Body Type Weights

Based on available assets and campaign type, calculate recommended distribution:

```
FOR campaign_type PL (Product Launch):
  Heavy ST (stories build anticipation) + Heavy TM (proof closes) + Moderate LB + CT for differentiation

FOR campaign_type AR (Autoresponder):
  Heavy ST (trust-building) + Moderate QO (authority) + Moderate QA (objection handling) + Light TM

FOR campaign_type CP (Content Series):
  Heavy LB (education) + Heavy QA (engagement) + Moderate CT (perspective shifts)

FOR campaign_type DR (Daily Relationship):
  Even distribution — maximum variety needed for daily sending
```

---

## Failure Modes

| Failure | Remediation |
|---------|-------------|
| Fewer than 5 usable assets total | gate_1_eligible = false. Recommend Content Series (CP) to build assets first, or request additional brief detail. |
| All assets map to a single body type | Flag: "Asset concentration in [type]. Sequence variety will be limited. Consider providing [missing types]." |
| Mode B brief provides no proof/testimonials | TM gap_severity = critical. Recommend gathering testimonials before campaign launch. |
| NR type has zero assets | Not a failure — NR is optional. Many campaigns use zero NR emails. |
| Asset richness universally low (avg < 4) | Flag: "Overall content richness is low. Campaign will rely on thin content. Consider enriching brief or running CopywritingEngine Skills 01-09 first." |
