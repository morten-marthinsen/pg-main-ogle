# 1.3 — Proof Inventory

**Version:** 1.0
**Skill:** U2 — 1-Click Upsell Writer
**Layer:** 1
**Microskill:** 1.3 — Proof Inventory
**Purpose:** Select exactly 1-2 proof elements from available inventory for placement in the CAIRO draft. This is NOT a proof cascade — more than 2 proof elements is a FAIL.

---

## INPUT

- `layer-0/0.1-input-context.md` (proof elements available, audience description)
- `layer-0/0.2-specimen-calibration.md` (proof patterns observed in specimens)
- `layer-1/1.1-congruence-map.md` (mechanism name, root cause — proof must support these)
- Proof inventory from Skill 02 (if Mode A)
- Research output from Skill 01 (if Mode A)

## PROCESS

1. **Inventory all available proof elements.** Scan all upstream packages and input context for proof:
   - Social proof numbers ("Over 14,000 golfers," "more than 50,000 customers," "used by 237 professionals")
   - Authority endorsements (named expert with credential, institutional affiliation, published research)
   - Specific result claims (percentage improvement, before/after data, timeframe-specific outcome)
   - Testimonials (individual story, named person, specific result)
   - Clinical/study references (published study, clinical trial, laboratory test)
   - Media mentions (featured in, as seen on, endorsed by publication)
   - Awards or recognitions (industry award, patent, certification)

2. **Rank proof elements by upsell effectiveness.** The priority order for post-purchase upsell pages is DIFFERENT from front-end sales pages:

   **Priority 1 — Social proof numbers:** "Over 14,000 golfers have already added this to their training." Social proof numbers work best in upsells because they leverage the buyer's existing trust and add herd validation. They require zero explanation and zero credibility-building. One sentence.

   **Priority 2 — Authority endorsement:** "Developed with Dr. [Name], [credential]." Authority endorsements work in upsells because the buyer already trusts the brand — a named expert extends that trust to the upsell product specifically. One sentence.

   **Priority 3 — Specific result claim:** "Users who added [upsell product] saw results 47% faster." Result claims work in upsells because they directly answer "why should I add this?" with a specific, quantified answer. One sentence to one short paragraph.

   **Deprioritized for upsells:**
   - Testimonials — too long for 500-2000 word format, slows momentum
   - Clinical studies — requires explanation, feels like a new pitch
   - Media mentions — irrelevant at point of purchase (already bought)
   - Awards — too indirect, doesn't answer "why add this?"

3. **Select exactly 1-2 proof elements.** Rules:
   - If only Priority 1 proof exists → select 1 social proof number. That is sufficient.
   - If Priority 1 + Priority 2 exist → select 1 of each (social proof + authority). This is the ideal combination.
   - If Priority 1 + Priority 3 exist → select 1 of each (social proof + result claim).
   - If only Priority 2 or 3 exist → select 1-2 from what is available.
   - If no proof exists → flag "NO PROOF AVAILABLE" — do not fabricate. The upsell can function without proof (post-purchase trust), but it is weakened.
   - **HARD CAP: 2 proof elements maximum.** Selecting 3+ = FAIL. Delete extras.

4. **Verify proof supports congruence.** Each selected proof element must connect to the mechanism name or root cause from the congruence map:
   - "Over 14,000 golfers use [mechanism name]" — congruent (references mechanism)
   - "Over 14,000 golfers have improved their game" — generic (does not reference mechanism or root cause)
   - If proof is generic, reframe it to connect to the mechanism or root cause. If it cannot be reframed, deprioritize it.

5. **Map proof placement in CAIRO structure.** Each proof element gets assigned to ONE CAIRO section:
   - Social proof numbers → best in Amplify (validates the future pace: "14,000 others got these results") or Reason (adds credibility to the "why": "we learned this from 14,000 users")
   - Authority endorsement → best in Reason (adds credibility to the story of why the upsell was created)
   - Specific result claim → best in Amplify (makes the future pace specific: "47% faster results") or Intrigue (creates curiosity: "something that makes results 47% faster")

6. **Draft the proof integration language.** Write the exact sentence(s) that will carry each proof element. Keep it SHORT:
   - Social proof: 1 sentence (e.g., "Over 14,000 golfers have already added this to their training — and here's why.")
   - Authority: 1 sentence (e.g., "Developed alongside Dr. Mark Reynolds, one of the top biomechanics researchers in the country.")
   - Result claim: 1-2 sentences (e.g., "In our internal testing, users who added the Accelerator saw measurable improvement 47% faster than those using the protocol alone.")

7. **Write output file.**

## CONSTRAINTS

1. **HARD CAP: 2 proof elements maximum.** This is the single most important constraint. An upsell page with 3+ proof elements has become a sales page. The buyer already trusts you. Excessive proof signals you don't trust their trust. More than 2 = FAIL — delete extras before proceeding.
2. Never fabricate proof. If the upstream packages contain no proof elements, write "NO PROOF AVAILABLE" and proceed. An upsell can function on post-purchase trust alone. A fabricated proof element is fraud.
3. Proof integration language must be SHORT. One sentence per proof element is ideal. Two sentences maximum for result claims that need a qualifying statement. Paragraphs of proof = front-end behavior, not upsell behavior.
4. Each proof element is placed in exactly ONE CAIRO section. Do not scatter the same proof across multiple sections — that is proof cascade behavior.
5. Priority ranking is specific to POST-PURCHASE upsell pages. Social proof > authority > result claims. This is DIFFERENT from front-end priority where result claims and testimonials often dominate.
6. Generic proof that does not connect to the mechanism name or root cause language is weak. Always prefer proof that explicitly references the mechanism, even if it is lower in the priority hierarchy.
7. This microskill can run in parallel with 1.2 (Position Analyzer). It does not depend on 1.2 output.
