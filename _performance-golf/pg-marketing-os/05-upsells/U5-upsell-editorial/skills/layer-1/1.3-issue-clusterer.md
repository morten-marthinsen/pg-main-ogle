# 1.3 — Issue Clusterer

**Version:** 1.0
**Skill:** U5 — Upsell Editorial
**Layer:** 1
**Microskill:** 1.3 — Issue Clusterer
**Model:** opus
**Purpose:** Group related issues into fix clusters (issues that should be resolved together), prioritize clusters by combined severity, route clusters to Arena (P1/P2) or Direct Fix (P3/P4), and create a sequenced revision plan for Layer 2 execution.

---

## INPUT

- `layer-1/1.2-issue-list.md` (all issues with severity, piece, lens, passage, problem, target fix)
- `layer-1/1.1-baseline-scores.md` (for context on relative weaknesses)

## EXECUTION STEPS

1. **Load the complete issue list.** Read every issue from `1.2-issue-list.md`. Count total issues and verify severity distribution against the summary statistics.

2. **Identify clustering criteria.** Issues should be clustered when:
   - They affect the SAME passage or overlapping passages in the same piece
   - They share a root cause (e.g., three issues all stem from the Amplify section being written in selling tone)
   - Fixing one issue would naturally fix or affect another (dependency relationship)
   - They span multiple lenses but target the same structural element (e.g., congruence AND tone issues both in the Congratulate section)
   - Do NOT cluster issues just because they share severity level. Clustering is about fix dependencies, not administrative grouping.

3. **Build fix clusters.** For each cluster:
   - **Cluster ID:** FC-001, FC-002, ...
   - **Cluster name:** Descriptive label (e.g., "Upsell Amplify Section Tone Overhaul," "Downsell CTA Guilt Removal," "Sequence-Wide Mechanism Name Restoration")
   - **Issues included:** List of issue IDs (I-001, I-003, I-007)
   - **Highest severity in cluster:** The cluster inherits the severity of its most critical issue. If a cluster contains one P1 and two P3 issues, the cluster is P1.
   - **Piece(s) affected:** Which pieces this cluster touches
   - **Passage scope:** The specific passage(s) that will be revised in this cluster
   - **Combined fix description:** What the fix must achieve across all issues in the cluster

4. **Route each cluster:**
   - **Cluster severity P1 or P2 → Arena.** 7 competitors, 2 rounds + audience evaluation, editorial_revision mode per U5-ARENA-LAYER.md. Human selection per cluster.
   - **Cluster severity P3 or P4 → Direct Fix.** No Arena overhead. Fix applied directly with before/after documented.
   - Mixed-severity clusters always route by the HIGHEST severity. A cluster with P1 + P3 issues goes to Arena.

5. **Prioritize clusters.** Execution order matters — some fixes affect the context for later fixes:
   - **Priority 1:** P1 clusters — fix structural/congruence failures first. These changes ripple through everything.
   - **Priority 2:** P2 clusters — fix major quality issues next. These affect scoring across multiple lenses.
   - **Priority 3:** P3 clusters — polish after structural issues are resolved.
   - **Priority 4:** P4 clusters — cosmetic cleanup last.
   - **Within same priority:** Order by piece position in sequence (Bump -> Upsell -> Downsell) so fixes flow in reading order.
   - **Cross-piece clusters before single-piece clusters** at same priority — systemic fixes first.

6. **Check for fix conflicts.** Scan for clusters that might conflict:
   - Two clusters targeting the same passage → merge into one cluster
   - A cluster whose fix could invalidate another cluster's target passage → note the dependency and ensure execution order handles it
   - A P3 cluster that might be automatically resolved by a P1 cluster's fix → flag as "may resolve with FC-XXX" (still keep it in the plan but mark for post-fix verification)

7. **Count Arena rounds needed.** Sum all P1/P2 clusters. Each cluster = one Arena cycle. This tells the human how many Arena selections they will need to make. If >5 Arena cycles are needed, flag this — the sequence may have deeper upstream issues.

8. **Create the revision plan.** Sequenced list of all clusters in execution order with routing, dependencies, and Arena cycle count.

9. **Write output file** with clusters, routing, and revision plan.

## CONSTRAINTS

1. **Clustering is about fix dependencies, not administrative convenience.** Two issues on the same piece but different passages with no fix dependency should be separate clusters.
2. **Cluster severity = highest severity issue in the cluster.** No averaging. A cluster with I-001 (P1) and I-004 (P3) is a P1 cluster routed to Arena.
3. **P1/P2 clusters MUST go to Arena.** No shortcuts. No "this P2 has an obvious fix." Arena generates alternatives. The "obvious fix" might break congruence or tone.
4. **P3/P4 clusters MUST NOT go to Arena.** No overhead for minor/cosmetic issues. Direct fix is the appropriate resolution method.
5. **Do not create singleton clusters for issues that share a passage.** If I-002 and I-005 both target the same paragraph in the upsell Amplify section, they MUST be one cluster. Fixing the passage twice is wasteful and can create inconsistency.
6. **Cross-piece clusters (sequence-wide issues) execute FIRST.** A fix that spans all pieces sets the context for per-piece fixes.
7. **Revision plan must be SEQUENCED.** Layer 2 executes clusters in the plan order. Dependencies must be respected.
8. **>5 Arena cycles = escalation flag.** This indicates the assembled sequence has systemic quality issues. Flag for human: "Consider whether U2/U3 re-execution is more efficient than 6+ Arena editorial rounds."

## FAILURE MODES

| Failure | Detection | Response |
|---------|-----------|----------|
| Issues not clustered | Every issue is its own cluster with no grouping | Re-examine for shared passages and root causes. Some standalone clusters are valid, but zero clustering across 10+ issues is suspicious. |
| P1 issue in Direct Fix cluster | Cluster tagged DIRECT FIX but contains a P1 issue | Re-route to Arena. P1 = Arena. Always. |
| P3 issue in Arena cluster (alone) | Cluster is pure P3 but routed to Arena | Re-route to Direct Fix. P3 alone does not justify Arena overhead. (If P3 is clustered WITH a P2, the cluster goes to Arena — that's correct.) |
| Conflicting clusters not detected | Two clusters target overlapping passages | Merge into one cluster. Overlapping passage scope = one fix cluster. |
| No revision plan | Clusters listed but not sequenced | Create the plan. Layer 2 needs execution order. Unsequenced clusters lead to fix conflicts and wasted work. |
| Dependency cycle | FC-001 depends on FC-003, FC-003 depends on FC-001 | Merge the cyclically dependent clusters into one. Circular dependencies indicate they are one fix. |
