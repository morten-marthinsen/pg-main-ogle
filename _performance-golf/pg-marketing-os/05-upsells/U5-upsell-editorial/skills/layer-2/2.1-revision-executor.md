# 2.1 — Revision Executor

**Version:** 1.0
**Skill:** U5 — Upsell Editorial
**Layer:** 2
**Microskill:** 2.1 — Revision Executor
**Model:** opus (Arena generation) / sonnet (Direct Fix)
**Purpose:** Execute the revision plan from 1.3 — run full Arena (7 competitors, 3 rounds) for each P1/P2 fix cluster, apply direct fixes for each P3/P4 cluster, track before/after for every change, and verify that no fix breaks congruence, tone, or speed compliance.

---

## INPUT

- `layer-1/1.3-issue-clusters.md` (fix clusters with routing, severity, passage scope, execution order)
- `layer-1/1.2-issue-list.md` (full issue details for each cluster's issues)
- `layer-1/1.1-baseline-scores.md` (baseline context — what scores need to improve)
- `layer-0/0.1-input-context.md` (congruence thread, pricing data, word count limits)
- The assembled upsell sequence (full text — the copy being revised)
- U5-ARENA-LAYER.md (Arena competitors, execution protocol, judging criteria)
- UPSELL-ENGINE.md (5 Laws, CAIRO/ARO, word count limits — for preservation checks)

## EXECUTION STEPS

1. **Load the revision plan.** Read the execution order from `1.3-issue-clusters.md`. Confirm total cluster count, Arena cluster count, Direct Fix cluster count.

2. **Load the congruence preservation anchors.** From `0.1-input-context.md`, extract:
   - Mechanism name (EXACT string)
   - Root cause language (VERBATIM phrasing)
   - Promise phrasing
   - These are the THREE things that must NEVER be broken by any revision. Every fix is checked against these anchors.

3. **Load word count limits.** From UPSELL-ENGINE.md:
   - Bump: 50-150 words
   - Upsell: 500-2000 words
   - Downsell: 250-1000 words
   - Total sequence: < 800 words per minute * 4 minutes = 800 words (at 200 wpm, 4 min = 800 words max)

4. **Execute clusters IN ORDER per the revision plan.** For each cluster:

   **IF ARENA CLUSTER (P1/P2):**

   a. **Prepare Arena input package.** Per U5-ARENA-LAYER.md, each competitor receives:
      - The specific passage(s) to revise (quoted from current copy)
      - The identified issues in this cluster (severity, lens, problem, target fix)
      - The full assembled sequence (for context — competitors need to see the whole sequence to preserve congruence)
      - Congruence thread data (mechanism name, root cause language, promise)
      - Word count constraints for the affected piece
      - The 5 Laws of Upsell

   b. **Run 3-round Arena.** Per ~system/protocols/ARENA-CORE-PROTOCOL.md:
      - Round 1: All 7 competitors generate revision candidates
      - Round 1 critique: Adversarial Critic scores all 7 on the 5 judging criteria (Issue Resolution 25%, Congruence Preservation 25%, Tone Preservation 20%, Speed Preservation 15%, CTA Clarity 15%)
      - Round 2: Competitors revise based on critique + learning briefs
      - Round 2 critique: Second scoring pass
      - Round 3: Final revisions
      - Round 3 critique: Final scoring

   c. **Present to human for selection.** BLOCKING gate. Human selects the winning revision for this fix cluster. Include: all 7 final candidates, scores, Critic commentary, synthesizer recommendation if applicable.

   d. **Record the selection.** Document: which candidate won, final score, key rationale.

   e. **Apply the selected revision to the sequence.** Replace the affected passage with the winning revision.

   f. **Run preservation checks** (see Step 5).

   **IF DIRECT FIX CLUSTER (P3/P4):**

   a. **Read the cluster's issues and target fixes.** The target fixes from 1.2 provide specific guidance.

   b. **Apply the fix directly.** Rewrite the affected passage to resolve all issues in the cluster. Use sonnet for generation — P3/P4 fixes are polish, not creative generation.

   c. **Document before/after.** Quote the original passage and the revised passage.

   d. **Run preservation checks** (see Step 5).

5. **Preservation checks (EVERY fix, Arena and Direct).** After EACH cluster fix is applied:

   **Congruence check:**
   - Is the mechanism name still present in the revised passage (if it was there before)?
   - Is the root cause language still verbatim (if it was in the affected passage)?
   - Does the promise thread still hold?
   - If ANY congruence anchor is weakened or removed: REVERT the fix. Apply an alternative that preserves congruence.

   **Tone check:**
   - Scan the revised passage for pre-purchase selling language (urgency, fear, guilt, PAS elements)
   - Verify the tone matches the piece type: celebration (bump/upsell) or acknowledgment (downsell)
   - If selling language is introduced: REVERT. Apply an alternative with correct post-purchase tone.

   **Speed check:**
   - Count words in the revised passage vs. the original
   - If word count increased: is the increase justified by quality gain? If >10% increase, flag.
   - Recount the piece's total word count. Does it still meet limits?
   - If piece exceeds word count limits: REVERT. Apply a fix that maintains or reduces word count.

   **CTA check (if fix touches CTA):**
   - YES option still has product name + price + action verb?
   - NO option still guilt-free?
   - If CTA degraded: REVERT. Apply a fix that preserves CTA quality.

6. **Handle dependent clusters.** If the revision plan has dependencies (FC-002 depends on FC-001), re-read the current state of the sequence AFTER the prerequisite cluster is applied before executing the dependent cluster. The passage scope may have shifted.

7. **Handle "may resolve with" clusters.** After applying the prerequisite cluster, check whether the flagged P3/P4 cluster's issues are actually resolved. If yes, mark the cluster as "RESOLVED BY FC-XXX" and skip it. If partially resolved, update the cluster's remaining issues and apply direct fix to what remains.

8. **Compile all changes.** After all clusters are executed, compile a comprehensive change log.

9. **Write output file** with complete revision record.

## CONSTRAINTS

1. **Arena for P1/P2. No exceptions.** Even if the fix seems obvious. Arena generates 7 alternatives. The "obvious" fix might break congruence or tone in ways you don't anticipate. The 7 competitors catch what a single perspective misses.
2. **Direct Fix for P3/P4. No Arena.** Arena overhead for minor issues wastes human attention. Polish is mechanical, not creative.
3. **Preservation checks after EVERY fix.** Not after all fixes. After EACH one. A fix that breaks congruence must be reverted BEFORE the next cluster is attempted.
4. **Human selection is BLOCKING per Arena cluster.** Do not batch Arena selections. Each P1/P2 cluster gets its own Arena, its own presentation, its own human selection.
5. **Before/after documentation is MANDATORY.** Every fix must show the original passage and the revised passage, quoted exactly. This is the evidence trail for rescoring in Layer 4.
6. **Reversions are not failures — they are the system working.** If a preservation check catches a problem, reverting is correct behavior. Document the reversion, apply the alternative, and move on.
7. **Do not modify passages outside the cluster's scope.** If you notice an issue in a passage that isn't part of the current cluster, note it for a future cluster or flag for 1.2 re-run. Do not "fix it while we're here."
8. **Cluster execution order from 1.3 is BINDING.** Dependencies exist for a reason. Do not reorder.

## FAILURE MODES

| Failure | Detection | Response |
|---------|-----------|----------|
| Arena skipped for P1/P2 cluster | Cluster tagged ARENA but no Arena output | Run Arena. No shortcuts for critical/major issues. |
| Arena run for P3/P4 cluster | Cluster tagged DIRECT FIX but Arena output exists | Unnecessary overhead. Accept the Arena result but flag the routing error. |
| Preservation check skipped | No congruence/tone/speed check documented for a cluster | Run preservation checks retroactively. If a break is found, revert and re-apply. |
| Fix breaks congruence | Mechanism name removed, root cause paraphrased, or promise replaced | REVERT immediately. Apply alternative fix preserving congruence anchors. |
| Fix introduces selling language | Pre-purchase urgency, fear, guilt, PAS detected in revised passage | REVERT immediately. Apply alternative fix maintaining post-purchase tone. |
| Fix exceeds word count | Piece word count exceeds limit after revision | REVERT. Apply fix that maintains or reduces word count. |
| Out-of-scope modification | Passage modified that isn't in the cluster's scope | Revert the out-of-scope change. Note the issue for potential additional cluster. |
| Dependency ignored | Dependent cluster executed before prerequisite | Revert dependent cluster. Execute prerequisite first, then re-execute dependent cluster on updated copy. |
