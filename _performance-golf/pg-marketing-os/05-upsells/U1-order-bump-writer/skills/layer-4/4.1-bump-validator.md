# 4.1 — Bump Validator

**Version:** 1.0
**Skill:** U1 — Order Bump Writer
**Layer:** 4
**Microskill:** 4.1 — Bump Validator
**Model:** haiku
**Depends on:** 1.2 (Bump Variants), GATE_1 PASS
**Produces:** `layer-4/4.1-validation-results.md`
**Purpose:** Validate every variant against all hard constraints. Score pass/fail per criterion per variant. Remove failing variants.

---

## INPUT

- `layer-1/1.2-bump-variants.md` — all 5-7 variants with word counts and labels
- `layer-0/0.1-input-context.md` — FE mechanism name (for congruence check), price data (for anchor check)
- GATE_1 must be PASS before executing this microskill

---

## PROCESS

1. **Confirm GATE_1 PASS:**
   - Read 1.2 output. If GATE_1 = FAIL, do NOT execute. Halt.

2. **For EACH variant, run ALL validation checks:**

   ### Check 1: Word Count (HARD FAIL)
   - Count every word in the variant body (exclude the checkbox header and choice lines — count the copy between them)
   - Words = space-delimited tokens. Hyphenated compound = 1 word. Dollar amounts ($9.95) = 1 word. Abbreviations (e.g., "2x") = 1 word.
   - **PASS:** 50-150 words inclusive
   - **FAIL:** <50 or >150
   - This is the highest-priority check. A variant at 151 words FAILS regardless of all other criteria.

   ### Check 2: Congruence — FE Mechanism Name (HARD FAIL)
   - Search variant text for the EXACT FE mechanism name string from 0.1
   - Case-insensitive match is acceptable
   - **PASS:** Mechanism name appears at least once in the variant body
   - **FAIL:** Mechanism name not found. Paraphrases do not count. Partial matches do not count ("Metabolic" alone does not satisfy "Metabolic Switch").

   ### Check 3: Price Format — Anchor + Discount (HARD FAIL)
   - Verify both prices are present:
     - Anchor price (retail value, regular price, "was" price, or FE-relative value reference)
     - Discount price (the actual bump price the buyer pays)
   - Acceptable formats: "$67 value — just $9.95" / "was $49, now $14.95" / "Add for just $9.95 (a $67 value)" / "Save $57 — just $9.95 today"
   - **PASS:** Both an anchor/value reference and the discount price are present
   - **FAIL:** Only one price shown, or no clear anchor/savings context

   ### Check 4: Binary Choice Language (HARD FAIL)
   - Verify the variant includes:
     - A positive choice: "Add to My Order" / "Yes! Add..." / "Add [product] to my order" or equivalent
     - A negative choice: "No thanks" / "No, I'll pass" / "I don't need this" or equivalent
   - **PASS:** Both positive and negative choice options present
   - **FAIL:** Missing either option, or choice language is ambiguous

   ### Check 5: No Story Structure (SOFT — flag, likely fail)
   - Scan for narrative indicators:
     - "Imagine..." / "Picture this..." / "Let me tell you..."
     - Before/after structure ("I used to..." / "Then I discovered...")
     - Chronological narrative arc (beginning, middle, end)
     - Character introduction or scene setting
   - **PASS:** No narrative indicators found
   - **FAIL:** Any narrative indicator present. Order bumps are not stories.

   ### Check 6: No Proof Cascade (SOFT — flag, likely fail)
   - Count distinct proof elements in the variant:
     - Statistics or data points ("340% improvement")
     - Authority claims ("Dr. X recommends" / "Harvard study")
     - Social proof numbers ("50,000 customers" / "93% of buyers")
     - Testimonial quotes
     - Named expert endorsements
   - **PASS:** 0 or 1 proof element
   - **FAIL:** 2+ proof elements. This is a proof cascade. One maximum.

   ### Check 7: No PAS Structure (HARD FAIL)
   - Scan for Problem-Agitation-Solution flow:
     - Problem identification: "Are you tired of..." / "Most people struggle with..." / "The problem is..."
     - Agitation: "It gets worse..." / "Without this, you'll..." / "Every day you wait..."
     - Solution introduction: "Now there's a solution..." / "Introducing..." / "Finally..."
   - **PASS:** No PAS indicators found
   - **FAIL:** PAS flow detected. The buyer already accepted the problem when they purchased the FE. Bumps use Reminder-Extension-Offer-Choice, not PAS.

   ### Check 8: Post-Purchase Tone (SOFT — flag, likely fail)
   - Scan for pre-purchase selling language:
     - Hard urgency: "Last chance!" / "Don't miss out!" / "Limited time only!" / "Act now!"
     - Fear/guilt: "You'll regret..." / "Don't make this mistake..."
     - Manipulation: Guilt in the "no" option ("good luck without this!")
     - Sales page phrases: "But wait, there's more!" / "This is the opportunity of a lifetime"
   - **PASS:** Tone is celebratory, logical, extension-oriented. Language like "You just secured..." / "Complete your system..." / "Add this to make [mechanism] work even better..."
   - **FAIL:** Pre-purchase selling tone detected. The buyer already converted. Post-purchase is the appropriate tone.

   ### Check 9: 3-Element Completeness
   - Verify all three elements are present:
     - WHAT: Product name AND at least a one-phrase value proposition
     - WHY NOW: At least one reason to add at checkout (momentum, pricing, completeness, timing)
     - PRICE: At least the discount price (anchor checked separately in Check 3)
   - **PASS:** All three elements identifiable
   - **FAIL:** Any element missing

3. **Compile results into per-variant scorecard:**
   - 9 checks per variant
   - HARD FAIL on any of: Check 1, 2, 3, 4, 7 = variant rejected
   - SOFT FAIL on Check 5, 6, 8 = variant flagged, likely rejected unless the violation is marginal
   - A variant must PASS all 9 checks to be included in final output

4. **Count passing variants:**
   - If 3+ pass: proceed to 4.2
   - If <3 pass: GATE_2 = FAIL. Return to Layer 1 for regeneration of failed variants.

5. **Write output file**

## CONSTRAINTS

- **Do NOT fix failing variants.** The validator identifies failures. It does not repair them. Failing variants are REJECTED and returned to Layer 1 if regeneration is needed.
- **Do NOT soften HARD FAIL criteria.** A variant at 152 words fails Check 1. Period. No "close enough."
- **Do NOT count checkbox header text or choice option text as part of the word count.** Only the body copy between the checkbox header and the choice options.
- **Do NOT auto-pass variants.** Every check must be explicitly evaluated. "This variant looks fine" is not a validation result.
- **The mechanism name check (Check 2) requires EXACT match.** "Metabolic" does not satisfy "Metabolic Switch." "The Switch" does not satisfy "Metabolic Switch." The full name must appear.
