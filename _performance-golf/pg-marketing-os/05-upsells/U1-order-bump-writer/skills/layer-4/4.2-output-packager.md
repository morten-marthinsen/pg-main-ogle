# 4.2 — Output Packager

**Version:** 1.0
**Skill:** U1 — Order Bump Writer
**Layer:** 4
**Microskill:** 4.2 — Output Packager
**Model:** haiku
**Depends on:** 4.1 (Validation Results), GATE_2 PASS
**Produces:** `order-bump-copy.md` (FINAL DELIVERABLE)
**Purpose:** Package all passing variants into the final deliverable with labels, split-test recommendations, and implementation notes.

---

## INPUT

- `layer-4/4.1-validation-results.md` — list of passing variants with scorecards
- `layer-1/1.2-bump-variants.md` — full text of all variants (to pull passing ones)
- `layer-0/0.1-input-context.md` — project metadata (product names, prices, mode)
- GATE_2 must be PASS before executing this microskill

---

## PROCESS

1. **Confirm GATE_2 PASS:**
   - Read 4.1 output. If GATE_2 = FAIL, do NOT execute. Halt.

2. **Collect all passing variants:**
   - Pull the full text of each passing variant from 1.2
   - Preserve exact copy — do NOT edit, rewrite, or "polish" any variant
   - Preserve the emphasis label from 1.2
   - Preserve the word count from 4.1

3. **Order variants for presentation:**
   - Lead with the variant most likely to be the control (best overall balance of congruence, value, and clarity)
   - Recommended ordering:
     1. Congruence variant (strongest strategic alignment with FE)
     2. Value variant (strongest savings/deal framing)
     3. Urgency variant (strongest time pressure)
     4. Social proof variant
     5. Exclusivity variant
     6-7. Any additional variants
   - This is a recommendation order, not a mandatory sequence. Adjust based on template type:
     - Completeness templates: lead with Congruence
     - Accelerator templates: lead with Value or Results
     - Insurance templates: lead with Urgency
     - Exclusive templates: lead with Exclusivity

4. **Generate split-test recommendations:**
   Based on the variants produced, recommend:
   - **First test:** Which 2 variants to test against each other and why (pick the two most different angles — maximum learning per test)
   - **What to watch:** Primary metric (bump take rate), secondary metric (refund rate on bump product), sample size guidance (minimum 200 orders per variant before declaring winner)
   - **Second test:** Once first test has a winner, which variant to test next against the winner
   - **When to stop:** After 3 rounds of testing or when take rate differences are <1%

5. **Generate implementation notes:**
   - **Placement:** Order bump checkbox should appear on the order form page, below the order summary and above the submit button
   - **Design:** Checkbox should be unchecked by default (pre-checked bumps erode trust). Yellow or light-colored highlight box to draw attention. Product image (if physical product) next to copy.
   - **Mobile:** Copy must be readable without horizontal scrolling. Binary choice buttons should be thumb-friendly (minimum 44px tap targets). Consider a collapsible "learn more" for the full copy on small screens.
   - **A/B testing setup:** Each variant gets a unique ID for tracking. Ensure analytics capture: impressions, checkbox checks, and completed orders with bump.

6. **Write GATE_2 checkpoint file:**
   Write `checkpoints/GATE_2_VERIFIED.yaml`:
   ```yaml
   gate: GATE_2
   status: PASS
   timestamp: [ISO 8601]
   variants_passed: [N]
   variants_failed: [N]
   output_file: order-bump-copy.md
   downstream_ready:
     U4: true
   ```

7. **Write final deliverable**

- Average order bump take rate (industry): 20-40%

- If ALL variants are below 15%, the issue is likely not copy -- check placement, design, and price point.

## Implementation Notes

### Placement
- Location: Order form page, below order summary, above submit/payment button
- Visibility: Must be visible without scrolling past the order summary

### Design
- Checkbox: Unchecked by default (pre-checked erodes trust)
- Container: Yellow or light-colored highlight box to distinguish from form fields
- Image: Include product image if physical product (thumbnail, left-aligned)
- Typography: Body copy in readable font (14-16px), price in bold, savings in color (red or green)

### Mobile
- Copy must be readable without horizontal scroll
- Binary choice: thumb-friendly tap targets (minimum 44px height)
- Consider collapsible "details" section if copy is at the longer end (120-150 words)
- Test: Verify bump renders correctly on iOS Safari and Chrome Android

### Analytics Setup
- Each variant needs a unique tracking ID
- Track: impressions (page loads), interactions (checkbox toggles), conversions (orders with bump)
- Calculate: take rate = conversions / impressions
- Segment by: traffic source, device type, time of day (for split-test integrity)

## CONSTRAINTS

- **Do NOT edit variant copy.** The packager assembles and formats — it does not rewrite. If a variant needs editing, it should have been caught in 4.1 and sent back to Layer 1.
- **Do NOT include failed variants in the final deliverable.** Failed variants were rejected for a reason. They do not appear in `order-bump-copy.md`.
- **Do NOT fabricate split-test data or predictions.** Recommendations are based on structural analysis of the variants (angle differentiation, emphasis diversity). Specific conversion predictions are not made — only testing methodology.
- **Do NOT skip the implementation notes.** The bump copy is useless without placement, design, and mobile guidance. The implementation team needs this context.
- **The final deliverable must be standalone.** Someone reading `order-bump-copy.md` with no other context should be able to implement the bump on a checkout page. All necessary metadata (product names, prices, mechanism name, congruence type) must be in the file.
