# 1.1 — Page Type Validator

> **Skill:** LP-17-conversion-audit
> **Layer:** 1
> **Position:** 1 of 3
> **Purpose:** Confirm the page type from LP-00's classification and check for mismatches between the brief's declared type and what the assembled page actually contains.
> **Effort Level:** sonnet
> **Dependencies:** `assembled-page-load.md`, `brief-load.md`
> **Output:** `page-type-validation.md`

---

## INPUT

```yaml
input:
  assembled_page_load: "assembled-page-load.md"
  brief_load: "brief-load.md"
```

---

## EXECUTION

1. Read the page type from `brief-load.md` (LP-00's classification).

2. Read the section inventory and above-fold elements from `assembled-page-load.md`.

3. Run a structural cross-check — does the assembled page match the declared type?

   **Type A structural signals:**
   - Story-driven lead section present
   - No price or ATC above fold
   - Narrative sections (root cause, mechanism, discovery story)
   - Word count >4,000
   - Proof blocks distributed through page (not concentrated)

   **Type B structural signals:**
   - Price display above fold
   - ATC/purchase button above fold
   - Rating strip present
   - Product image as hero
   - Shorter word count (1,500-7,000)
   - Proof concentrated in review cascade section

   **Hybrid structural signals:**
   - Type B above-fold (price, ATC, rating strip)
   - Type A below-fold sections (narrative, mechanism education)
   - Word count 3,000-10,000

4. Compare brief classification against assembled page structure. Flag any mismatches:
   - Brief says Type A but page has price above fold -> MISMATCH
   - Brief says Type B but page has 15,000 words of narrative -> MISMATCH
   - Brief says Hybrid but page has no Type B elements above fold -> MISMATCH

5. Determine the **operational page type** for this audit:
   - If brief and page align -> use brief classification
   - If mismatch -> flag the mismatch, recommend using the type that matches the actual assembled page structure, note the discrepancy for human review

6. Write validation results to `page-type-validation.md`.

---

## QUALITY CRITERIA

- Brief classification and assembled page structure both documented
- All structural signals checked
- Match/mismatch status clearly declared
- Operational page type for audit determined
- If mismatch, clear recommendation provided
- File exists after execution

---

## FAILURE MODES

- Brief says one type, page is clearly another -> Flag MISMATCH. Use the assembled page's actual structure as the operational type. Note that LP-00 classification may need to be re-run.
- Page is genuinely ambiguous (could be either type) -> Default to Hybrid and apply both Type A and Type B criteria where they differ. Flag for human review.
- Brief classification is missing -> Infer type from assembled page structure alone. Note the gap.
