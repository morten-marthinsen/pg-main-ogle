# 2.2 — Offer Summary Builder

**Skill:** LP-00-brief-classifier
**Position:** Layer 2, Microskill 2
**Purpose:** Extract and structure the full offer from the product brief. Map product name, pricing model (single/multi/subscription/trial), bonuses, guarantee, and urgency levers into a standardized schema. Use `not_available` for any genuinely missing data — never leave fields empty.
**Effort Level:** medium
**Dependencies:** `brief-load.md` (Layer 0)
**Output:** `offer-summary.md`

---

## Input Schema

```yaml
input:
  brief_content: "[from brief-load.md — full brief text]"
  operating_mode: "[downstream | standalone]"
  package_inventory: "[from package-inventory.md, if Downstream Mode — for Layer 3 enrichment flag]"
```

---

## Execution Logic

**Step 1: Product identification.**
Extract the product name, product category, and primary benefit claim. If the brief names multiple products, identify the PRIMARY product (the one being sold on this page). Flag any secondary products as potential bonuses or upsells.

**Step 2: Pricing extraction.**
Scan the brief for all pricing signals and populate:
- Single unit price
- Multi-pack options (Y/N + details)
- Subscription pricing (Y/N + details)
- Trial offer (Y/N + details — free trial, $1 trial, etc.)

If pricing is not stated in the brief, mark all pricing fields as `not_available`. Do NOT guess prices.

**Step 3: Bonus inventory.**
Extract all mentioned bonuses. For each bonus, record:
- Bonus name
- Perceived value (if stated)
- Bonus type (digital/physical/access)

If no bonuses mentioned, set count to 0 and list to empty.

**Step 4: Guarantee extraction.**
Extract guarantee details:
- Duration (30-day, 60-day, 90-day, lifetime)
- Branded name (if any — e.g., "The Complete Confidence Guarantee")
- Type (money-back, exchange, results-based)

If no guarantee mentioned, mark as `not_specified`.

**Step 5: Urgency lever scan.**
Check for:
- Stock/inventory scarcity (real or implied)
- Price deadline (sale ends, price increase)
- Batch/production limitation
- Seasonal/event tie-in

Mark each as Y/N with justification if Y.

**Step 6: Source tagging.**
Tag the entire offer summary as `brief_inferred` or `offer_package_validated`. In Downstream Mode with an offer package available, note that Layer 3 (microskill 3.3) will enrich/overwrite this summary with validated data.

## Output Schema

`offer-summary.md`:

```markdown
# Offer Summary — [Product Name]

## Product
- Product name: [full name]
- Product category: [e.g., dietary supplement, online course, coaching program]
- Primary benefit: [single core promise statement]
- Mechanism name: [from brief if mentioned, else: not_available]
- Root cause anchor: [from brief if mentioned, else: not_available]

## Pricing
- Single unit price: [$X.XX | not_available]
- Has multi-pack: [Y/N]
- Multi-pack options: [details | N/A]
- Has subscription: [Y/N]
- Subscription price: [details | N/A]
- Has trial: [Y/N]
- Trial details: [details | N/A]

## Bonuses
- Count: [number]
- List:
  - [Bonus 1 name] — Value: [$X | not stated] — Type: [digital/physical/access]
  - [Bonus 2 name] — Value: [$X | not stated] — Type: [digital/physical/access]
  [or: "No bonuses identified in brief"]

## Guarantee
- Duration: [X days | not_specified]
- Branded name: [name | not_specified]
- Type: [money-back | exchange | results-based | not_specified]

## Urgency Levers
- Stock scarcity: [Y — justification | N]
- Price deadline: [Y — details | N]
- Batch limitation: [Y — details | N]
- Seasonal/event: [Y — details | N]

## Source: [brief_inferred | offer_package_validated]
## Layer 3 Enrichment Pending: [Y — offer package available for 3.3 extraction | N]
```

---

## Quality Criteria
- Product name extracted accurately (not a placeholder)
- All pricing fields populated or explicitly marked `not_available`
- Zero empty fields — every field has a value or `not_available` / `not_specified`
- Bonus count matches actual list length
- Guarantee type is one of the valid options or `not_specified`
- Source tag present and accurate

## Failure Modes
- Brief mentions no pricing at all → Mark all pricing fields `not_available`, add validation flag
- Brief describes multiple products → Identify primary (the conversion target), list others as secondary/bonus
- Guarantee language is ambiguous → Extract what's clear, mark type as `not_specified`, flag for human review
- Brief uses vague offer language ("great bonuses included") → Mark specifics as `not_available`, note the vague reference
