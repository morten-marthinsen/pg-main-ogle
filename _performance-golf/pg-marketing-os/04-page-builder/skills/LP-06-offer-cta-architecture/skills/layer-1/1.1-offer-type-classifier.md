# 1.1 — Offer Type Classifier

**Skill:** LP-06-offer-cta-architecture
**Position:** Layer 1, Microskill 1
**Purpose:** Classify the offer from the 7-type taxonomy (Hard, Soft, Free+S&H, Limited, Limited-Time, OTO, Combination). The classification governs pricing display strategy, CTA emotional appeal selection, and urgency approach. A Combination offer identifies the primary + secondary types.
**Effort Level:** medium
**Dependencies:** `offer-data-loaded.md` (0.2), `specimen-offers-loaded.md` (0.3)
**Output:** `offer-type-classification.md`

---

## Input Schema

```yaml
input:
  pricing_structure: "[from 0.2 — single, multi-pack, subscription, trial details]"
  urgency_levers: "[from 0.2 — any available urgency information]"
  bonus_stack: "[from 0.2 — count and details]"
  guarantee_details: "[from 0.2 — duration and type]"
  specimen_offer_model: "[from 0.3 — dominant model in vertical]"
  page_type: "[type_a | type_b | hybrid]"
```

---

## The 7-Type Classification Matrix

| Type | Key Signal | Pricing Model | CTA Implication | Urgency Implication |
|------|-----------|---------------|-----------------|---------------------|
| **Hard** | Standard purchase, pay full price now | Fixed price, no trial | Direct benefit appeal | Optional |
| **Soft** | Try first, pay later, installments | Trial, installments, subscribe | Risk reversal appeal dominant | Built into offer structure |
| **Free + S&H** | Product free, shipping only | $0 + shipping fee | Belonging + specificity appeals | High — limited supply framing |
| **Limited** | Genuine quantity cap | Standard pricing with cap | Consequence appeal | Required — real scarcity |
| **Limited-Time** | Deadline-driven pricing | Discount with expiration | Consequence + specificity appeals | Required — real deadline |
| **OTO** | Seen once, never again | Special pricing | Consequence appeal dominant | Extreme — page-level urgency |
| **Combination** | Mix of 2+ types | Hybrid pricing | Multi-appeal strategy | Depends on components |

---

## Execution Logic

**Step 1:** Review the pricing structure from `offer-data-loaded.md`. Identify the PRIMARY transaction model:
- Is there a trial period? -> Soft Offer
- Is the product free with shipping only? -> Free + S&H
- Is there a genuine quantity limit? -> Limited Offer
- Is there a real deadline on pricing? -> Limited-Time
- Is this an upsell page shown once? -> OTO
- Is it a standard buy-now transaction? -> Hard Offer

**Step 2:** Check for Combination indicators:
- Hard + Limited-Time = standard product with promotional pricing window
- Soft + Limited = trial offer with genuine quantity cap
- Hard + Soft = one-time purchase with subscription option (dual mode)
- Any combination of 2+ types -> classify as Combination, list primary + secondary

**Step 3:** Cross-reference with specimen patterns from `specimen-offers-loaded.md`:
- Does the classification match the dominant offer model in this vertical?
- If different, document WHY this offer deviates from vertical norms

**Step 4:** Document the classification with rationale (1-3 sentences).

## Quality Criteria
- Exactly one primary type selected from 7-type taxonomy
- If Combination: secondary types listed
- Classification supported by specific evidence from offer data
- Specimen alignment noted — matches or deviates with reason
- Downstream implications documented for pricing, CTA, and urgency

## Failure Modes
- Offer data too incomplete to classify -> Classify as Hard Offer (default for incomplete data). Flag as `needs_validation`. Note: "Classification based on limited data. Re-classify when full offer details available."
- Offer appears to be multiple products -> Flag IMMEDIATELY as potential one-offer violation. This is a Layer 3 audit issue but detecting it early saves rework.
