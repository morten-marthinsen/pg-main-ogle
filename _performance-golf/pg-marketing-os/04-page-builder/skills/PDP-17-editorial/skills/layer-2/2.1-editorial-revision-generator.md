# 2.1-editorial-revision-generator.md

> **Layer:** 2 (Revision)
> **Skill:** PDP-17-ecomm-editorial
> **Purpose:** Generate revised sections addressing audit findings
> **Outputs:** `revised_sections` object with edited copy for all flagged sections

---

## FUNCTION

Generate editorial revisions for all sections flagged in the Layer 1 audit. This is the core editorial function: take the existing copy and EDIT it -- tighten sentences, improve scan formatting, strengthen proof elements, fix feature name inconsistencies, and remove slop words. This skill edits, it does NOT rewrite.

---

## REVISION PROTOCOL

### Step 1: Load Audit Findings
```
FROM 1.1-scan-optimization-auditor:
  flagged_sections: [sections scoring < 7.0]
  scan_directives: [specific formatting changes per section]

FROM 1.2-proof-density-reviewer:
  proof_gaps: [sections needing proof addition or strengthening]
  proof_directives: [specific proof changes per section]
```

### Step 2: Revision Pass Per Section
```
FOR each section (in editorial priority order from 0.3):

  EDITORIAL CHECKLIST:
    1. SCAN OPTIMIZATION:
       - Apply scan directives from 1.1
       - Shorten paragraphs to 2-3 sentences max
       - Bold key benefit phrases and proof numbers
       - Convert 3+ parallel items to bullet lists
       - Ensure header is benefit-oriented, 3-8 words

    2. PROOF:
       - Apply proof directives from 1.2
       - Strengthen weak proofs (add specifics, attribution)
       - Reposition proofs for scan visibility
       - NEVER remove proof elements

    3. SLOP REMOVAL:
       - Scan for banned lexicon words
       - Replace with specific, concrete alternatives
       - Remove filler phrases ("in order to" -> "to")
       - Tighten passive constructions

    4. FEATURE NAME CONSISTENCY:
       - Verify all feature references match PDP-02 registry
       - Fix any paraphrasing, shortening, or invention
       - Ensure bold formatting on feature names

    5. WORD BUDGET CHECK:
       - Count revised section words
       - MUST stay within original word budget
       - Revisions should maintain or reduce word count
```

### Step 3: Change Tracking
```
FOR each revised section:
  TRACK:
    original_word_count: [before revision]
    revised_word_count: [after revision]
    words_changed: [count of changed words]
    change_ratio: [words_changed / original_word_count]
    changes_made: [list of specific edits]

  IF change_ratio > 0.30:
    HALT -- "Revision too extensive. Max 30% word change. This is an edit, not a rewrite."
    REDUCE changes to stay under threshold
```

### Step 4: Before/After Documentation
```
FOR each revised section:
  DOCUMENT:
    section_number: [N]
    original_copy: [exact text before revision]
    revised_copy: [text after revision]
    changes_summary: [bulleted list of what changed and why]
    change_ratio: [percentage]
    scan_score_improvement: [estimated new score]
```

## FAILURE MODES

| Mode | Trigger | Remediation |
|------|---------|-------------|
| CRITICAL | Change ratio exceeds 30% for any section | Reduce edits, keep most impactful changes only |
| CRITICAL | Feature name changed during revision | Revert to PDP-02 name immediately |
| CRITICAL | Proof element removed during revision | Restore proof, reposition if needed |
| WARNING | Word count increased during revision | Tighten further to match or reduce |
| WARNING | Slop word introduced during revision | Replace with specific alternative |
