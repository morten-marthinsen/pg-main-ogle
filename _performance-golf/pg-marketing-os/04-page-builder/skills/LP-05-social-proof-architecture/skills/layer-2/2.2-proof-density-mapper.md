# 2.2 — Proof Density Mapper

**Skill:** LP-05-social-proof-architecture
**Position:** Layer 2, Microskill 2
**Purpose:** Assign specific proof element counts, types, and quality requirements per section. This transforms the placement plan (2.1) from "what goes where" into "exactly how many of what type with what quality standard." LP-10 executes against this document.
**Effort Level:** high
**Dependencies:** `proof-placement-plan.md` (2.1), `proof-classification.md` (1.1), `volume-requirements.md` (1.2)
**Output:** `proof-density-map.md`

## Density Level Definitions

| Level | Element Count | Use When |
|-------|-------------|----------|
| **None** | 0 elements | Section serves pure education/navigation function |
| **Light** | 1 element | Authority anchor, quick credibility mention |
| **Medium** | 2-3 elements | Mix of types (study + testimonial, or authority + volume) |
| **Heavy** | 4-8 elements | Dedicated proof sections, testimonial cascades |
| **Maximum** | 9+ elements | Review sections (Type B), full proof blocks |

---

## Quality Requirements by Proof Type

Every proof element MUST have a quality requirement. These are the standards:

| Proof Type | Minimum Quality Requirement |
|-----------|---------------------------|
| `clinical_study` | Institution name + specific finding + statistic. NOT "studies show..." |
| `expert_endorsement` | Full name + credentials (MD, PhD, etc.) + specific endorsement statement. NOT "doctors recommend..." |
| `before_after` | Named person + specific outcome (numbers) + timeframe + photo if available. NOT "amazing results!" |
| `photo_testimonial` | Customer photo + full name + location + specific result with numbers. NOT stock photo + generic praise |
| `volume_signal` | Exact number + source. "83,502 reviews" NOT "thousands of customers." If exact not available, round to nearest thousand with "+" |
| `text_testimonial` | Name + location + specific outcome with numbers. NOT "Great product! — Sarah" |
| `platform_review` | Star rating + verified buyer badge + result-specific text. NOT rating alone |
| `video_testimonial` | Named person + pull-quote + specific outcome. Video embed + text summary |
| `media_mention` | Publication name + context. "Featured in Forbes" NOT "As seen on TV" |
| `certification` | Full certification name + issuing body. "GMP Certified — NSF International" NOT "certified" |

## Execution Logic

**Step 1:** Load the placement plan from 2.1.

**Step 2:** For each section with proof elements assigned:

a) Confirm the density level based on element count.

b) For each element, specify:
   - Proof type code (from hierarchy)
   - Count (exact number)
   - Quality requirement (from table above — tailored to this product)
   - Source status: `available` (exists in inventory) or `requires_sourcing`
   - Hierarchy rank (1-6)

c) Verify that higher-ranked proof types are in higher-visibility slots:
   - Rank 1-2 proof should appear in mechanism, ingredient, lead sections (high-visibility educational slots)
   - Rank 3-4 proof should appear in dedicated proof blocks (high-impact cascade slots)
   - Rank 5-6 proof should fill remaining slots (supporting evidence)

**Step 3:** Tally all elements across all sections. Compare to volume-requirements.md thresholds.

**Step 4:** Distribute available proof assets across slots:
- Assign the best available assets to the highest-visibility slots
- Mark remaining slots as `requires_sourcing` with specific quality requirements
- Never assign the same proof asset to multiple sections

**Step 5:** Create the density map with complete specifications per section.

## Quality Criteria
- Every proof element has: type code, count, quality requirement, source status
- No element has a vague quality requirement ("add testimonials" is forbidden)
- Higher-ranked proof types in higher-visibility sections
- Volume totals match or exceed thresholds from 1.2
- No single proof asset assigned to multiple sections

## Failure Modes
- Volume total below threshold -> HALT. Increase planned elements in proof blocks or add sections.
- All elements are `requires_sourcing` -> Acceptable in Standalone Mode. Add note: "Architecture complete but all proof must be sourced before execution."
- Quality requirement is generic -> Rewrite with specific standard from quality table above
