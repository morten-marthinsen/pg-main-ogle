# 2.8 — PDP Vertical Layout Writer

**Skill:** LP-09-benefits-features
**Position:** Layer 2, Microskill 8
**Purpose:** Produce the structural layout specification for PDP benefit sections using vertical stacking (not horizontal grids). Based on Baymard Institute research showing 27% item oversight in horizontal layouts vs. 8% in vertical. This microskill writes the LAYOUT SPECIFICATION — not the copy. Copy comes from 2.1, 2.6, and 2.7. This structural spec is consumed by PDP-07 (BTF Section Copy) and LP-15 (Page Assembly) for proper benefit section rendering.
**Effort Level:** medium
**Dependencies:** `benefit-section-classification.md` (1.1), `packages-loaded.md` (0.1), `pdp-ingredient-cards.md` (2.6 — if running), `pdp-microscripts.md` (2.7 — if running)
**Output:** `pdp-vertical-layout-spec.md`

---

## Pre-Execution Gate

**This microskill runs ONLY when:**
- Page type is `type_b` or `hybrid`
- At least one benefit/feature/ingredient section exists in 1.1

**If page type is `type_a`:** Do NOT run. Type A pages use long-form section layouts governed by the section sequence planner (LP-04), not PDP-specific vertical stacking.

## Pre-Generation Protocol

**Step 1: Inventory all benefit sections.** Count the total items that will be displayed in the benefit section(s): ingredient cards, feature explanations, benefit bullets, how-it-works steps.

**Step 2: Determine vertical layout variant.** Based on content type and item count:
- Ingredient cards (supplement): Vertical stack, each card full-width
- Feature explanations (SaaS): Vertical stack, icon-left + text-right per row
- Benefit bullets: Vertical stack, icon-left + text-right per row
- How-it-works steps: Vertical numbered sequence

**Step 3: Check 2.6 and 2.7 outputs** (if available). The layout spec must accommodate the card structure from 2.6 and the microscript placements from 2.7.

**Step 4: Review hero section layout** for visual continuity. The benefit section layout must feel like the same page as the hero — not a jarring visual shift.

## Execution Logic

### Core Principle: Vertical Stacking

**NEVER use horizontal grid layouts for PDP benefit sections.**

Baymard Institute research: In horizontal card layouts, 27% of items are overlooked by users. In vertical stacked layouts, only 8% are overlooked. For benefit sections where every item must be seen and processed, vertical is the only acceptable pattern.

### Layout Variant A: Ingredient Card Stack (Supplement PDPs)

For pages where 2.6 (PDP Ingredient Card Writer) is producing ingredient cards:

```
SECTION LAYOUT SPEC:
  container: full-width, max-width 800px, centered
  section_heading: H2, centered, 24-32px
  section_subheading: body text, centered, 16-18px, muted color

  PER CARD:
    layout: vertical stack, one card per row
    card_structure:
      - icon/image: left-aligned or centered, 48-64px
      - "X for Y" headline: H3, bold, 18-22px, below icon
      - dose badge: inline, small text, muted, next to or below headline
      - mechanism line: body text, 14-16px, 1 line
      - benefit statement: body text, 14-16px, slightly bolder weight
      - expandable detail: accordion/reveal, "Learn More" trigger
    card_spacing: 24-32px between cards
    card_separator: subtle divider line OR alternating background
    card_padding: 16-24px all sides

  MOBILE:
    single_column: true
    icon_position: above text (not left of text)
    padding: 16px minimum all sides
    card_spacing: 20px between cards
    "X for Y" headline: 16-18px
    touch_target: expandable trigger >= 44px height
```

### Layout Variant B: Feature/Benefit Row Stack (SaaS/Consumer PDPs)

For pages with feature explanations or benefit bullets (non-supplement):

```
SECTION LAYOUT SPEC:
  container: full-width, max-width 800px, centered
  section_heading: H2, centered, 24-32px

  PER ROW:
    layout: horizontal within row — icon (left) + text (right)
    row_structure:
      - icon: left-aligned, 40-48px, vertically centered
      - text_block:
        - benefit_headline: bold, 16-18px, 1 line
        - supporting_text: regular weight, 14-16px, 1-2 sentences
    row_spacing: 20-28px between rows
    row_separator: alternating background color (optional) OR divider line
    row_padding: 12-20px all sides

  MOBILE:
    single_column: true
    icon_position: above text (centered)
    padding: 16px minimum all sides
    row_spacing: 16-20px between rows
    benefit_headline: 16px
    supporting_text: 14px
```

### Layout Variant C: How-It-Works Vertical Sequence

For how-it-works/process sections:

```
SECTION LAYOUT SPEC:
  container: full-width, max-width 800px, centered
  section_heading: H2, centered, 24-32px

  PER STEP:
    layout: vertical sequence with step numbers
    step_structure:
      - step_number: large numeral (32-48px) or circled number, left or above
      - step_title: H3, bold, 18-20px
      - step_description: body text, 14-16px, 2-3 sentences
      - step_benefit: slightly bolder weight or accent color, 14-16px
    step_spacing: 28-36px between steps
    connecting_element: vertical line or arrow between steps (optional)
    step_padding: 16-24px all sides

  MOBILE:
    single_column: true
    step_number: centered above text, 28-36px
    padding: 16px minimum
    step_spacing: 24px between steps
```

### Typography Hierarchy

Consistent across all variants:

```
TYPOGRAPHY:
  section_heading: H2 — 24-32px, bold, high contrast
  item_headline: H3 or bold body — 16-22px, bold
  body_text: 14-16px, regular weight, sufficient line-height (1.5-1.6)
  supporting_detail: 14px, muted color or lighter weight
  dose_badge: 12-14px, muted, uppercase optional
  microscript: 14-16px, bold or semi-bold, accent color optional
```

### Icon Style Direction

```
ICON GUIDELINES:
  style: [line icons | filled icons | custom illustrations | ingredient photos]
  size: 40-64px depending on variant
  color: [monochrome matching brand | accent color | multi-color]
  consistency: ALL icons in the section must use the same style
  do_not: mix icon styles, use generic stock icons, use icons that require explanation
```

### Mobile Behavior (Non-Negotiable)

```
MOBILE REQUIREMENTS:
  layout: single column, always
  icon_position: above text (never side-by-side on mobile — too cramped)
  minimum_padding: 16px all sides
  minimum_touch_target: 44px for any interactive element
  font_sizes: no smaller than 14px for body text
  card/row_spacing: minimum 16px between items
  expandable_content: must be accessible without horizontal scroll
```

## Quality Criteria
- Layout uses vertical stacking for ALL benefit items (no horizontal grids)
- Mobile spec is explicitly defined (not "responsive" hand-wave)
- Minimum padding of 16px on mobile
- Touch targets >= 44px for interactive elements
- Typography hierarchy is explicit with pixel sizes
- Icon style direction is specific enough for a designer to execute
- Spacing values are concrete (pixel values, not "generous" or "sufficient")
- Layout accommodates the content from 2.6 (ingredient cards) and 2.7 (microscripts) if running
- Desktop and mobile specs are provided separately (not combined)

## Failure Modes
- Horizontal grid layout specified -> CRITICAL. Vertical stacking only. Baymard research is definitive: 27% oversight in horizontal vs. 8% in vertical.
- "Responsive" without explicit mobile spec -> FAIL. Mobile must be explicitly specified with concrete values, not left to CSS interpretation.
- Spacing values are vague ("good spacing," "generous padding") -> Fail. Concrete pixel values required. "24px between cards" not "good spacing."
- Icon style direction is "use appropriate icons" -> Too vague. Specify: line vs. filled, size, color scheme, consistency rule.
- Typography sizes not specified -> Fail. Every text element needs a concrete size range (e.g., "18-22px" not "large").
- Mobile body text below 14px -> Fail. Readability threshold on mobile.
- Running this microskill for Type A pages -> BLOCKED. Type A uses long-form section layouts, not PDP vertical stacking.
