# 1.1 — Benefit Section Classifier

**Skill:** LP-09-benefits-features
**Position:** Layer 1, Microskill 1
**Purpose:** Map each benefit/feature section identified in 0.2 to a specific content type and assign it to the correct Layer 2 microskill. This is the routing decision that determines which generator runs for each section.
**Effort Level:** medium
**Dependencies:** `packages-loaded.md` (0.1), `benefit-sections-identified.md` (0.2), `specimens-loaded.md` (0.3)
**Output:** `benefit-section-classification.md`

---

## Input Schema

```yaml
input:
  page_type: "[from packages-loaded.md]"
  vertical: "[from packages-loaded.md]"
  product_data_type: "[ingredients | features | deliverables | mixed]"
  benefit_sections: "[from benefit-sections-identified.md -- list of sections with IDs]"
  specimen_patterns: "[from specimens-loaded.md -- pattern extraction summary]"
```

---

## Execution Logic

1. **For each benefit section** identified in 0.2, classify the content type based on:

   **Signal 1 — LP-04 classification** (strongest signal): If section-sequence.json already classified the section as a specific type (e.g., "ingredient_deep_dive"), respect that classification unless clear evidence contradicts it.

   **Signal 2 — Product data type:**
   - Ingredients list present -> ingredient deep-dive sections likely (Type A supplement)
   - Features list present -> feature explanation sections likely (SaaS/info)
   - Deliverables list present -> benefit bullet sections likely (universal)

   **Signal 3 — Page type:**
   - Type A -> deeper, longer-form benefit treatment (deep-dives, explanations, before/after)
   - Type B -> scan-optimized, condensed treatment (bullets, grids, compressed breakdowns)
   - Hybrid -> Type B bullets above fold + Type A deep-dives below

   **Signal 4 — Section position:**
   - Early sections (after hero) -> benefit bullets or how-it-works (orient the reader)
   - Mid sections -> ingredient deep-dives or feature explanations (educate)
   - Late sections (pre-offer) -> comparison benefits or before/after (close)

2. **Classification taxonomy** (6 content types):

   | Content Type | When to Use | Layer 2 Microskill |
   |-------------|------------|-------------------|
   | `benefit_bullets` | Universal — every page gets at least one bullet section | 2.1 |
   | `ingredient_deep_dive` | Supplement/health pages with ingredient data | 2.2 |
   | `feature_explanation` | SaaS/info/B2B pages with feature list | 2.3 |
   | `how_it_works` | Any page with a process, method, or step-by-step | 2.4 |
   | `comparison` | Pages with named competitors or known alternatives | 2.5 |
   | `before_after` | Type A pages — typically near transitions | 2.1 (embedded in benefit bullets) |

3. **Verify completeness:** Every identified section must have a classification. No section left unclassified.

4. **Verify coverage:** At minimum, microskill 2.1 (benefit bullets) must run for every project. If no section was classified as benefit_bullets, add one.

5. **Write `benefit-section-classification.md`** with the full classification map.

## Quality Criteria
- Every identified section has a classification with rationale
- At least one section classified as `benefit_bullets` (universal requirement)
- Classifications consistent with page type (Type B sections not classified as long-form deep-dives)
- Execution routing table complete with no gaps
- Estimated word counts reference specimen norms from 0.3

## Failure Modes
- Section type ambiguous between two classifications -> Choose the type that best serves the reader at that position in the page. Document the decision and the alternative considered.
- All sections classified as the same type -> Verify this is correct. Pages typically have variety. If truly all one type, document why.
- Ingredient deep-dive classified for non-supplement page -> Re-classify. Ingredient deep-dives are for health/supplement verticals only. SaaS/info pages use feature explanations.
