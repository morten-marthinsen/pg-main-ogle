# 0.2 — Section Sequence Loader

**Skill:** LP-09-benefits-features
**Position:** Layer 0, Microskill 2
**Purpose:** Load section-sequence.json (LP-04 output), identify which sections are benefit/feature type, extract their position, assigned content type, and any section-specific notes. This determines how many benefit sections LP-09 must write and what type of content each one requires.
**Effort Level:** low
**Dependencies:** LP-04 output (`section-sequence.json`), `packages-loaded.md` (0.1)
**Output:** `benefit-sections-identified.md`

---

## Input Schema

```yaml
input:
  project_directory: "[path to active project output directory]"
  section_sequence_file: "section-sequence.json"
  page_type: "[from packages-loaded.md]"
```

---

## Execution Logic

1. **Load `section-sequence.json`** from the project output directory.

2. **If section-sequence.json is MISSING:** HALT. LP-09 requires section sequence to know which sections to write. Cannot infer section structure.

3. **Scan all sections** for those classified as benefit/feature content types. Look for section types that include any of:
   - `benefit_bullets`
   - `ingredient_deep_dive` / `ingredient_breakdown`
   - `feature_explanation` / `feature_deep_dive`
   - `how_it_works` / `how_to_use`
   - `comparison` / `comparison_table`
   - `before_after`
   - `key_benefits`
   - Any section with "benefit," "feature," "ingredient," or "how it works" in the name or description

4. **For each identified section**, extract:
   - Section ID (position number in sequence)
   - Section name
   - Content type classification from LP-04
   - Section purpose/description (if provided)
   - Preceding section (what the reader just read — context for entry point)
   - Following section (what comes next — context for exit/transition)
   - Any proof assigned to this section from proof-architecture.json

5. **Count total benefit/feature sections** identified. If count = 0 -> FLAG as unusual (some pages may not have dedicated benefit sections) and check if benefits are embedded in other section types.

6. **Write `benefit-sections-identified.md`** with the section map.

## Quality Criteria
- section-sequence.json loaded and parsed completely
- Every benefit/feature section identified with all required fields
- Each section mapped to the correct Layer 2 microskill
- Execution plan clearly shows which microskills will run
- Context sections (preceding/following) documented for each benefit section

## Failure Modes
- section-sequence.json missing -> HALT. Cannot determine which sections to write.
- Zero benefit sections identified -> FLAG as unusual. Check if benefits are embedded in other section types (e.g., inside a mechanism section or product introduction). If truly absent, report to human.
- Section type ambiguous (could be benefit OR proof) -> Classify as benefit if it primarily presents deliverable/feature/benefit content. Classify as proof if it primarily presents evidence. Document the decision.
