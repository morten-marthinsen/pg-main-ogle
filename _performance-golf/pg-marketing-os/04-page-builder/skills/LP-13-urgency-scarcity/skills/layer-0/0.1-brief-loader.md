# 0.1 — Brief Loader

**Skill:** LP-13-urgency-scarcity
**Position:** Layer 0, Microskill 1
**Purpose:** Load and verify the three upstream package files that LP-13 depends on: `page-brief.json` (LP-00), `offer-cta-architecture.json` (LP-06), and `section-sequence.json` (LP-04). Extract the key fields that govern urgency type selection and placement decisions.
**Effort Level:** low
**Dependencies:** LP-00 output (`page-brief.json`), LP-06 output (`offer-cta-architecture.json`), LP-04 output (`section-sequence.json`)
**Output:** `packages-loaded.md`

---

## Input Schema

```yaml
input:
  project_directory: "[path to active project output directory]"
  expected_files:
    - "page-brief.json"
    - "offer-cta-architecture.json"
    - "section-sequence.json"
```

---

## Execution Logic

1. **Locate each upstream file** in the project output directory. Check for existence of all three files.

2. **If any file is MISSING:** HALT immediately. Report which file(s) are absent. LP-13 cannot proceed without all three upstream packages.

3. **Load `page-brief.json`** and extract:
   - `page_type` (type_a | type_b | hybrid)
   - `vertical` (health, finance, info_product, personal_dev, technology, etc.)
   - `audience_profile` (primary pain, demographics, awareness stage)
   - `traffic_temperature` (cold | warm | hot)
   - `voice_direction` (tone, register, anti-voice)
   - `offer_summary` (product name, price point, guarantee, bonuses)
   - `urgency_inputs` (any urgency-related data from the original brief: deadlines, batch sizes, enrollment limits, price schedules)

4. **Load `offer-cta-architecture.json`** and extract:
   - `urgency_strategy` (LP-06's recommended urgency approach)
   - `urgency_types_recommended` (which of the 6 types LP-06 suggested)
   - `urgency_justification_notes` (any justification data LP-06 documented)
   - `cta_urgency_integration` (how urgency should relate to CTAs)
   - `offer_value_established` (confirmation that LP-11 can/has run)

5. **Load `section-sequence.json`** and extract:
   - `urgency_section_id` (the section ID and number for the urgency/scarcity section)
   - `urgency_section_mandatory` (is the urgency section marked mandatory or optional?)
   - `offer_section_number` (to verify urgency comes AFTER offer)
   - `close_section_number` (for P.S./close callback placement)
   - `total_sections` (count)
   - `sections_available_for_callbacks` (root cause, mechanism, story section IDs if natural urgency callbacks are needed)

6. **Cross-validate urgency positioning** across files:
   - Verify `urgency_section_number` > `offer_section_number` in section-sequence.json (value before urgency)
   - If urgency section is absent from section-sequence.json and LP-06 recommended no urgency, note this as valid (no justified urgency)
   - If urgency section is present but LP-06 provided no urgency strategy, flag the inconsistency

7. **Write `packages-loaded.md`** with all extracted fields organized by source file.

## Quality Criteria
- All three files loaded and confirmed present
- Key fields extracted with no gaps (missing fields marked `not_available`, not silently omitted)
- Urgency inputs from brief explicitly listed (even if all are "None provided" -- this is critical data for Layer 1)
- Cross-validation confirms urgency placement after offer section
- Output is self-contained -- downstream microskills can read this file without loading the original JSONs

## Failure Modes
- One or more upstream files missing -> HALT with specific file names listed. Do not proceed with partial data.
- Urgency section in sequence but no urgency strategy in LP-06 -> Flag inconsistency but do not halt. Layer 1 will evaluate.
- LP-06 recommends urgency but section-sequence.json has no urgency section -> Flag inconsistency. Layer 1 may determine urgency is unjustified.
- Brief has zero urgency inputs (no deadlines, no batch data, no price schedules, no capacity limits) -> Extract as `urgency_data: none` and flag. Layer 1 may determine no justified urgency exists.
