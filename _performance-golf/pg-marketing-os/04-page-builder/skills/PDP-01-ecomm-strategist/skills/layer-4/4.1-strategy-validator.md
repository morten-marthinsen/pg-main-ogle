# 4.1-strategy-validator.md

> **Layer:** 4 (Validation)
> **Skill:** PDP-01-ecomm-strategist
> **Purpose:** Validate strategy completeness, consistency, and downstream readiness
> **Outputs:** `validation_report` object with PASS/FAIL per check

---

## FUNCTION

Perform comprehensive validation of the assembled strategy before it is packaged as ecomm-strategy.yaml. This is the final quality gate that ensures the strategy is complete, internally consistent, and ready for downstream skill execution.

---

## VALIDATION PROTOCOL

### Step 1: Completeness Checks

```
CHECK assembled_strategy for required components:

  [ ] page_type classified (not null, not "pending")
  [ ] active_sections count >= 8
  [ ] every active section has priority (1-N, no gaps, no duplicates)
  [ ] every active section has word_budget (50-500 range)
  [ ] every active section has proof_type requirement
  [ ] every applicable section has crossover_skill mapping
  [ ] feature_research_scope has >= 1 hero candidate
  [ ] feature_research_scope has >= 2 supporting candidates
  [ ] word_budget_summary totals match sum of section budgets
  [ ] downstream_handoffs all marked ready

  RESULT: completeness_score = passed_checks / total_checks
  IF completeness_score < 1.0: FAIL with specific missing items
```

### Step 2: Consistency Checks

```
VERIFY internal consistency:

  [ ] Priority order matches conversion impact assessment
  [ ] Word budgets proportional to priority (higher priority >= lower priority budget, generally)
  [ ] Hero section is Priority 1
  [ ] At least one CTA section in top 5 priority
  [ ] At least one proof section in top 8 priority
  [ ] Crossover mappings match crossover reference table
  [ ] Excluded section rationales are specific (not vague)
  [ ] Feature research scope features exist in upstream product data
  [ ] Total word budget within page type bounds

  IF any inconsistency found:
    FLAG with specific remedy
    FAIL if critical inconsistency (priority misordering, budget mismatch)
    WARN if minor inconsistency (design note sparse, rationale could be stronger)
```

### Step 3: Downstream Readiness Checks

```
FOR each downstream skill:

  PDP-02 (Feature Naming):
    [ ] feature_research_scope populated
    [ ] hero candidates identified
    [ ] feature data quality assessed

  PDP-03 (Hero & Value Prop):
    [ ] hero section in active map with budget
    [ ] hero feature candidates available
    [ ] headline word budget set (5-12 words)

  PDP-07 (Section Copy):
    [ ] all BTF sections mapped with budgets
    [ ] crossover skills identified per section
    [ ] proof requirements per section defined

  PDP-08 (Micro-Scripts):
    [ ] section-to-script mapping possible
    [ ] feature data available for script content

  LP-15 (Assembly):
    [ ] priority order defined for assembly sequence
    [ ] design notes seeded per section
    [ ] word budget summary available

  IF any downstream not ready: FLAG with specific gap
```

## FAILURE MODES

| Mode | Trigger | Remediation |
|------|---------|-------------|
| CRITICAL | Completeness < 1.0 | Fix missing items before packaging |
| CRITICAL | Priority ordering broken | Re-rank priorities |
| CRITICAL | Downstream skill not ready | Fill gaps in strategy |
| WARNING | Minor consistency issue | Note in validation report, proceed |
| WARNING | Design notes sparse | Acceptable -- page builder will elaborate |
