# 1.1 — Review System Architecture

**Skill:** PDP-05-social-proof-review-system
**Position:** Layer 1, Microskill 1
**Purpose:** Design the complete review system UX architecture — histogram, keyword filter chips, review card layout, loading mechanism, sort behavior, merchant response rules, and "Write a Review" treatment. This is the master structural document that all other Layer 1 and Layer 2 microskills reference.
**Effort Level:** high
**Dependencies:** `input-verification.md` (0.1), `research-load.md` (0.2), `pdp-reference-load.md` (0.3)

## Execution Logic

**Step 1:** Load all Layer 0 outputs. Confirm scope determination from 0.1. The scope governs which components are included:

| Scope | Histogram | Filters | UGC | Q&A | Featured Reviews | Full Card Template |
|-------|-----------|---------|-----|-----|------------------|--------------------|
| full | Y | Y (5-8 chips) | Y | Y (conditional) | Y (3-5) | Y |
| standard | Y | Y (5-6 chips) | Conditional | Conditional | Y (3-5) | Y |
| minimal | N | N | N | N | Y (3-5) | Y (simplified) |
| generation_only | N | N | N | N | Y (3-5 generated) | Y (generated) |

**Step 2:** Design the review section page hierarchy (top-to-bottom order):

```
1. Section Header: "Customer Reviews" + aggregate rating + total count
2. Histogram (star distribution with tappable bars)
3. UGC Photo Strip (if available — ABOVE text reviews)
4. Keyword Filter Chips (5-8 product-specific)
5. Sort Dropdown (default: "Most Relevant")
6. Review Cards (stacked list)
7. "Load More" Button (15-30 per batch)
8. "Write a Review" (de-emphasized small link)
```

**Step 3:** Design the histogram component:
- 5 horizontal bars (5-star at top, 1-star at bottom)
- Each bar: star label (left) + horizontal fill bar (middle) + count label (right)
- Each bar is TAPPABLE — tapping filters the review list to that star rating only
- Active state: filled bar with highlighted color. Inactive: gray outline
- All 5 bars display even if count is 0 (grayed bar, "0" label — never hidden)
- Tap again to deselect (returns to "All Reviews")

**Step 4:** Design the keyword filter chip system:
- Position: below histogram, above review cards
- Count: 5-8 chips (never fewer than 5 for full/standard scope)
- Content: product-specific keywords derived from research data (0.2)
- If research data sparse: derive from product category defaults
- Interaction: tap to toggle on/off. Multi-select allowed.
- Active state: filled background + checkmark icon. Inactive: outline only.
- Filter behavior: when chip active, only reviews mentioning that keyword display

**Step 5:** Design the review card layout (detailed in 2.5, but structural template here):

```
[Stars: 1-5] [Bold Headline: 1-line summary, max 80 chars]
[Reviewer Name] | [Verified Buyer badge] | [Time Owned: "Owned for X months"] | [Location]
[Reviewer Attributes: age range, skin type, body type, etc. — product-specific]
---
Pros: [bullet list, 1-3 items]
Cons: [bullet list, 1-3 items]
---
[Body: 2-5 sentences, specific experience with product]
---
[Helpful? Yes (count) / No (count)]
[Merchant Response: indented block, branded — if rating <= 3 stars]
```

**Step 6:** Design the loading mechanism:
- Initial display: 15 reviews visible on page load
- "Load More" button below last visible review
- Each tap loads 15-30 additional reviews (batch size matches initial)
- Button text: "Show More Reviews" or "Load More Reviews"
- FORBIDDEN: numbered pagination (disrupts mobile scroll)
- FORBIDDEN: infinite scroll (causes disorientation + performance degradation)
- After loading, scroll position maintained (user stays where they were)

**Step 7:** Design the sort system:
- Default sort: "Most Relevant" (algorithmic: helpfulness votes + recency + specificity)
- Available options: Most Relevant, Newest, Highest Rated, Lowest Rated
- Dropdown position: right-aligned, above review cards, below filter chips
- Sort change resets to first batch of results

**Step 8:** Design the merchant response system:
- Trigger: any review rated 3 stars or below
- Format: indented block below review body, visually distinct (light background)
- Branded: company name + logo/icon as response author
- Tone specification: empathetic, solution-oriented, NOT defensive
- Content template: Acknowledge concern + Offer specific resolution + Provide contact info
- Response appears collapsed by default, "View Merchant Response" link to expand

**Step 9:** Design the "Write a Review" element:
- Format: small text link, NOT a button
- Position: top-right of review section header OR below last loaded review
- Text: "Write a Review" (not "Leave a Review" or "Share Your Experience")
- De-emphasized: smaller font, muted color, no background fill
- Purpose: available for post-purchase users but does NOT compete with reading flow

**Step 10:** Document the complete architecture with all component specifications.

## Quality Criteria
- All Baymard requirements from 0.3 reference load incorporated
- Histogram specified with tappable bars (not static display)
- Loading mechanism is "Load More" button (not pagination, not infinite scroll)
- Keyword filter chips are product-specific (not generic labels)
- Review card template includes ALL required fields (stars, headline, name, verified buyer, time owned, attributes, pros/cons, body, helpful, merchant response)
- Merchant response specified for 3-star and below
- "Write a Review" is de-emphasized text link (not button)
- Sort default is "Most Relevant" (not newest)
- Scope-appropriate components included/excluded per scope table

## Failure Modes
- Scope = minimal but histogram included -> Remove histogram. Minimal scope = featured reviews + simplified cards only.
- Generic filter chip labels like "Quality" or "Great" -> HALT. Replace with product-specific keywords from research or category defaults.
- Loading mechanism = pagination or infinite scroll -> HALT. Change to "Load More" button.
- Missing merchant response specification -> HALT. Add merchant response template for 3-star and below.
- "Write a Review" as prominent button -> HALT. Change to de-emphasized text link.
