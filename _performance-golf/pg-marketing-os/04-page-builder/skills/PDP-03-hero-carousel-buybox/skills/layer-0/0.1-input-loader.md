# 0.1 — Input Loader

**Skill:** PDP-03-hero-carousel-buybox
**Position:** Layer 0, Microskill 1
**Purpose:** Load LP-00 page brief and LP-01 conversion intelligence. Validate page type is type_b or hybrid. Extract product data, audience profile, and offer details needed for carousel and buy box architecture.
**Effort Level:** low
**Dependencies:** LP-00 page-brief.json, LP-01 conversion-intelligence output (if available)
**Output:** `input-verification.md`

---

## Input Schema

```yaml
input:
  page_brief: "[path to page-brief.json from LP-00]"
  conversion_intelligence: "[path to LP-01 output, or 'not available']"
  project_directory: "[path to active project outputs directory]"
```

---

## Execution Logic

**Step 1:** Locate and read `page-brief.json` from LP-00 output directory. If file does not exist → HALT. LP-00 must run first.

**Step 2:** Validate page type classification:
- `type_b` → PROCEED (PDP route)
- `hybrid` → PROCEED (PDP route with enhanced BTF)
- `type_a` → HALT — use LP-03 instead. PDP-03 is not designed for long-form sales pages.

**Step 3:** Extract product data from brief:
- Product name (full, never truncated)
- Product category / vertical
- SKU count and variant types (size, flavor, color, etc.)
- Price points (single, multi-pack, subscription)
- Key ingredients or features (raw list)
- Guarantee / return policy
- Shipping information

**Step 4:** Extract audience profile from brief:
- Target demographic
- Awareness stage (most-aware, product-aware, solution-aware)
- Traffic temperature (cold, warm, hot)
- Primary pain points
- Purchase motivations

**Step 5:** Extract offer details from brief:
- Pricing tiers (OTP vs subscription)
- Subscription discount percentage
- Multi-pack options and savings
- Bonuses (if any)
- Guarantee type and duration

**Step 6:** If LP-01 conversion intelligence is available, extract:
- Audience verbatim quotes (for carousel copy direction grounding)
- Competitor PDP patterns identified
- Conversion benchmarks for this vertical
- Trust signal priorities

**Step 7:** Write `input-verification.md`.

## Quality Criteria
- Page type is exactly `type_b` or `hybrid` (not `type_a`)
- Product name extracted in full (never truncated)
- All pricing tiers identified with exact dollar amounts
- Variant types enumerated (not "various")
- Guarantee details specific (not "money-back" without duration)
- File exists after execution

## Failure Modes
- page-brief.json not found → HALT, run LP-00 first
- Page type = type_a → HALT, use LP-03 instead
- No pricing data in brief → HALT, cannot architect buy box without prices
- No product name → HALT, cannot proceed without product identity
- LP-01 not available → PROCEED with brief-only data, note gap in verification
