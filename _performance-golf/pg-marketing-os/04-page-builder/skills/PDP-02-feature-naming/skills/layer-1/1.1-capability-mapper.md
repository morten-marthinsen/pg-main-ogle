# 1.1-capability-mapper.md

> **Layer:** 1 (Capability Analysis)
> **Skill:** PDP-02-feature-naming
> **Purpose:** Map all product capabilities/ingredients/technologies
> **Outputs:** `capability_map` object with categorized capabilities

---

## FUNCTION

Systematically map every product capability, ingredient, technology, material, and component that could become a named feature. This is an exhaustive inventory -- nothing should be missed. The mapper categorizes each capability by type and assesses its naming potential.

---

## MAPPING PROTOCOL

### Step 1: Extract All Capabilities

```
FROM upstream_context, SCAN for capabilities across categories:

  INGREDIENTS:
    - Active ingredients (supplements, skincare)
    - Key compounds or extracts
    - Dosages and concentrations

  TECHNOLOGIES:
    - Proprietary technologies
    - Engineering innovations
    - Software/firmware features
    - Manufacturing processes

  MATERIALS:
    - Construction materials
    - Material grades/sources
    - Material combinations

  COMPONENTS:
    - Physical parts/components
    - Structural elements
    - Functional subsystems

  CERTIFICATIONS:
    - Testing certifications
    - Quality standards met
    - Regulatory approvals

  DESIGN ELEMENTS:
    - Form factor innovations
    - Ergonomic features
    - Aesthetic differentiators
```

### Step 2: Categorize Each Capability

```
FOR each identified capability:
  ASSIGN category: ingredient | technology | material | component | certification | design
  IDENTIFY:
    what_it_is: [specific component name]
    what_it_does: [functional description]
    why_it_matters: [customer benefit]
    differentiator_level: [unique | category-leading | table-stakes]
    data_completeness: [complete | partial | minimal]
```

### Step 3: Assess Naming Potential

```
FOR each capability:
  RATE naming_potential:
    HIGH: Clear component + clear benefit + differentiating
    MEDIUM: Clear component OR clear benefit + some differentiation
    LOW: Vague component + vague benefit OR table-stakes
    NONE: Not feature-worthy (standard spec, not differentiating)

  IF naming_potential == NONE:
    EXCLUDE from feature naming
    MOVE to technical specs list (not named features)
```

### Step 4: Build Comprehensive Map

```
ASSEMBLE capability_map:
  total_identified: [count]
  by_category: { ingredients: [], technologies: [], ... }
  by_naming_potential: { high: [], medium: [], low: [] }
  excluded: [capabilities not worth naming]
  research_gaps: [capabilities needing more data]
```

## OUTPUT STRUCTURE

```json
{
  "capability_map": {
    "total_identified": 12,
    "capabilities": [
      {
        "id": "CAP-001",
        "name_raw": "Titanium club face",
        "category": "material",
        "what_it_is": "Forged titanium face insert",
        "what_it_does": "Maximizes energy transfer at impact",
        "why_it_matters": "More ball speed = more distance",
        "differentiator_level": "category-leading",
        "naming_potential": "HIGH",
        "data_completeness": "complete"
      }
    ],
    "summary": {
      "high_potential": 4,
      "medium_potential": 5,
      "low_potential": 2,
      "excluded": 1,
      "research_gaps": 1
    }
  }
}
```

---

## FAILURE MODES

| Mode | Trigger | Remediation |
|------|---------|-------------|
| CRITICAL | Fewer than 4 capabilities identified | Request more product data |
| CRITICAL | Zero HIGH naming potential capabilities | Review assessment criteria, may be too strict |
| WARNING | Many capabilities at MINIMAL data completeness | Flag for product research |
| INFO | More than 15 capabilities | Good depth -- hierarchy design will prioritize |
