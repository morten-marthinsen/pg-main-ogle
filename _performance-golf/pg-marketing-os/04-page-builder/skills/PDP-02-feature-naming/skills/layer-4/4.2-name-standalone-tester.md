# 4.2-name-standalone-tester.md

> **Layer:** 4 (Validation)
> **Skill:** PDP-02-feature-naming
> **Purpose:** Does name alone communicate what + why?
> **Outputs:** `standalone_test_result` object with PASS/FAIL per feature name

---

## FUNCTION

Test every feature name for standalone clarity -- can a person understand the feature from the name alone, without reading any description or micro-script? This is the final quality gate for naming. Names that fail must be revised before packaging.

---

## TESTING PROTOCOL

### Step 1: Isolation Test

```
FOR each feature name:
  PRESENT name in complete isolation:
    - No product name
    - No product category
    - No description
    - No micro-script
    - No visual context
    - JUST the feature name

  ASK 3 QUESTIONS:
    Q1: "What category of thing is this?" (component, ingredient, technology)
         PASS: Answer is clear from name
         FAIL: Answer requires guessing

    Q2: "What does this feature DO?" (function/action)
         PASS: Function is communicated or strongly implied
         FAIL: Function is unknown from name alone

    Q3: "Why would I CARE about this feature?" (benefit/outcome)
         PASS: Benefit is communicated or strongly implied
         FAIL: Benefit is unclear from name alone
```

### Step 2: Score Standalone Clarity

```
FOR each feature name:
  SCORE:
    Q1 (what category): PASS = 1, FAIL = 0
    Q2 (what it does): PASS = 1, FAIL = 0
    Q3 (why I care): PASS = 1, FAIL = 0

    standalone_score = sum / 3

    3/3 = FULL STANDALONE -- Name communicates everything
    2/3 = PARTIAL STANDALONE -- Name communicates most, one gap
    1/3 = WEAK STANDALONE -- Name communicates little
    0/3 = NO STANDALONE -- Name is meaningless alone

  THRESHOLD:
    Hero features: Must achieve 3/3
    Supporting features: Must achieve >= 2/3
    Technical features: Must achieve >= 2/3
```

### Step 3: Remediation for Failures

```
FOR each name that fails threshold:
  DIAGNOSE which question(s) failed:

    IF Q1 fails (category unclear):
      ADD component type to name
      "ProFlex" -> "ProFlex Cushioning"

    IF Q2 fails (function unclear):
      ADD action/function word
      "Titanium Core" -> "Titanium Distance Core"

    IF Q3 fails (benefit unclear):
      ADD benefit component
      "Ceramic Coating" -> "Ceramic Coating for Durability"

  RETURN revised name for re-testing
```

## OUTPUT STRUCTURE

```json
        "status": "PASS"
      }
    ],
    "failures_requiring_revision": [],
    "all_hero_features_pass": true
  }

```

---

## FAILURE MODES

| Mode | Trigger | Remediation |
|------|---------|-------------|
| CRITICAL | Hero feature fails standalone test | MUST revise before packaging |
| CRITICAL | More than 2 features fail | Return to Layer 2 for naming revision |
| WARNING | Supporting feature at 2/3 | Acceptable but note the gap |
| INFO | All features at 3/3 | Best case -- proceed to packaging |
