# 0.2 — Architecture Loader

> **Skill:** LP-15-page-assembly
> **Layer:** 0
> **Position:** 2 of 4
> **Purpose:** Load all four architecture packages from Phase 2 skills (LP-03, LP-04, LP-05, LP-06) and extract the structural blueprint for page assembly.
> **Effort Level:** haiku
> **Dependencies:** `above-fold-blueprint.json` (LP-03), `section-sequence.json` (LP-04), `social-proof-architecture.json` (LP-05), `offer-cta-architecture.json` (LP-06)
> **Output:** `architecture-load.md`

---

## INPUT

```yaml
input:
  above_fold_path: "[path to above-fold-blueprint.json from LP-03]"
  section_sequence_path: "[path to section-sequence.json from LP-04]"
  social_proof_arch_path: "[path to social-proof-architecture.json from LP-05]"
  offer_cta_arch_path: "[path to offer-cta-architecture.json from LP-06]"
  project_folder: "[path to project output folder]"
```

---

## EXECUTION

1. Locate each architecture package in the project output folder. Search paths per package:
   - `./~outputs/[project-name]/landing-page/lp-0X-[skill-name]/[package-name].json`
   - `./~outputs/[project-name]/lp-0X-[skill-name]/[package-name].json`

2. For each package found, extract:

### From LP-03 (above-fold-blueprint.json):
- Above-fold element order and hierarchy
- Layout specification (hero image placement, CTA position, trust bar position)
- Desktop vs. mobile fold breakpoints
- Type-specific above-fold rules applied

### From LP-04 (section-sequence.json):
- **CRITICAL:** The ordered list of sections — this is the assembly blueprint
- Section names, positions, purposes
- Section dependencies (which sections reference which)
- Estimated word counts per section
- Proof block placement positions

### From LP-05 (social-proof-architecture.json):
- Proof wave pattern (where proof blocks go in the sequence)
- Proof density targets per section
- Proof type recommendations (testimonials, reviews, studies, endorsements)
- Before/after requirements

### From LP-06 (offer-cta-architecture.json):
- CTA placement plan (which sections get CTA buttons)
- Offer structure (single offer, bundles, subscription options)
- Price display architecture (anchoring, comparison, per-day)
- Guarantee architecture (branded name, duration, placement)
- Value stack structure

3. For each missing package, log as `NOT_FOUND` with the upstream skill that must be run.

4. Write the consolidated architecture data to `architecture-load.md`.

## QUALITY CRITERIA

- All four architecture packages searched for in correct paths

---

## FAILURE MODES

- LP-04 section-sequence.json not found -> This is the most critical architecture package. Without it, LP-15 has no assembly blueprint. Write status as NOT_FOUND. Triggers GATE_0 HALT: "LP-15 requires section-sequence.json from LP-04. Run LP-04 first."
- LP-03 not found -> Assembly can proceed with reduced above-fold specificity. Flag as NOT_FOUND. Warn that above-fold assembly will use default Type A/B patterns from MASTER-BLUEPRINT.md.
- LP-05 or LP-06 not found -> Assembly can proceed with reduced proof/CTA architecture. Flag as NOT_FOUND. Warn that proof placement and CTA placement will use defaults.
- Package found but malformed -> Extract what is parseable. Flag unparseable fields. Do not invent data.
