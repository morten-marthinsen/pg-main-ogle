## EXECUTION

**POSITIONAL REINFORCEMENT:** LP-04 is the source of truth for section order. You CONFIRM the order. You do not redesign it. Deviations require explicit justification.

### Step 1: Extract LP-04 Section Sequence

From `architecture-load.md`, extract the complete section sequence from LP-04:
- Section number
- Section name
- Section purpose
- Estimated word count
- Content source skill

### Step 2: Map Content Packages to Sections

For each section in the LP-04 sequence, identify which content package provides its content:

| Section | Content Source Package | Source Skill | Available? |
|---------|---------------------|-------------|------------|
| Above Fold | hero-section-package.json | LP-07 | [from inventory] |
| Lead/Hook | hero-section-package.json (extended) or benefits-features.json | LP-07/LP-09 | [from inventory] |
| Story | [varies by project] | [varies] | [from inventory] |

### Step 3: Handle Missing Sections (Partial Mode Only)

If assembly mode is PARTIAL:
- Identify which sections have no content package available
- Options for each missing section:
  a. **SKIP** — remove from the assembly order (document why it is acceptable to skip)
  b. **PLACEHOLDER** — include the section position with a "CONTENT NEEDED" placeholder (for later completion)
  c. **MERGE** — merge the section's purpose into an adjacent section (document the merge rationale)
- Every adaptation must be documented with explicit justification

### Step 4: Confirm or Adapt Section Order

**If FULL mode:** The section order is LP-04's sequence verbatim. No adaptation needed. Confirm it.

**If PARTIAL mode:** Adapt the sequence based on Step 3 decisions. Document every change:
- Which section was removed/merged
- Why
- What the adapted sequence looks like

### Step 5: Identify CTA Placement Points

From LP-06's offer-cta-architecture (via architecture-load.md), identify which sections get CTA placements. Cross-reference against the confirmed section order to determine exact CTA positions.

### Step 6: Identify Proof Block Positions

From LP-05's social-proof-architecture (via architecture-load.md), identify where proof blocks fall in the sequence. Confirm they align with the section order.

### Step 7: Write section-order-plan.md

## QUALITY CRITERIA

- LP-04 section sequence is the source, not a new invention
- Every section has an identified content source package
- Every section's package availability is confirmed against inventory
- CTA placements identified and positioned
- Proof block positions confirmed
- Every section boundary identified (N-1 boundaries for N sections)
- Deviations (if any) are explicitly justified
- File exists after execution

---

## FAILURE MODES

- LP-04 section sequence not available in architecture-load.md -> HALT at GATE_0 should have caught this. If reached here without LP-04, use MASTER-BLUEPRINT.md default section flow for the page type. Document this as a major deviation.
- Section mapped to a package that inventory shows as missing -> In FULL mode, this should have triggered HALT. In PARTIAL mode, handle per Step 3 (SKIP/PLACEHOLDER/MERGE).
- CTA placement plan from LP-06 conflicts with section order -> Use the section order from LP-04 as primary. Adapt CTA placements to fit the confirmed order. Document the adaptation.
