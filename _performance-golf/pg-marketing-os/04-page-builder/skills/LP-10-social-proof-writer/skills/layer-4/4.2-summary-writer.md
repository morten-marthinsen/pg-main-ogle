# 4.2 — Summary Writer

**Skill:** LP-10-social-proof-writer
**Position:** Layer 4, Microskill 2
**Purpose:** Write SOCIAL-PROOF-SUMMARY.md — the human-readable summary of what LP-10 produced. This file serves as a quick-reference for human review and as context for downstream skills that need to understand what proof copy exists without parsing the full JSON package.
**Effort Level:** low
**Dependencies:** `social-proof-copy-package.json` (4.1), All Layer 2 outputs, All Layer 3 validation outputs
**Output:** `SOCIAL-PROOF-SUMMARY.md`

---

## Input Schema

```yaml
input:
  package: "[social-proof-copy-package.json — the compiled package]"
  proof_copy_validation: "[proof-copy-validation.md — score and details]"
  specificity_audit: "[specificity-audit.md — results]"
  anti_slop_scan: "[anti-slop-scan.md — results]"
  proof_slots_loaded: "[proof-slots-loaded.md — for total slot context]"
```

---

## Execution Logic

1. **Extract key metrics from the compiled package:**
   - Total slots addressed
   - Slots by copy type
   - Total word count across all proof copy
   - Transition pairs count
   - Volume signals count
   - Slots requiring sourcing

2. **Extract validation scores.**

3. **Create a copy-type-by-section matrix** showing what proof copy was written for which sections.

4. **List the strongest and weakest elements** (by specificity score).

5. **Document sourcing gaps** — what still needs to be sourced before launch.

6. **Write downstream handoff notes** — what LP-15 and LP-17 need to know.

---

## Quality Criteria
- Summary accurately reflects the compiled package (no discrepancies in counts)
- Copy-by-section matrix provides quick visual of proof distribution
- Strongest/weakest elements identified with specificity scores
- Sourcing gaps documented with severity
- Downstream handoffs are clear and actionable
- Notes for human review capture any execution-level concerns

## Failure Modes
- Summary counts don't match package counts -> Cross-verify against social-proof-copy-package.json before writing.
- Missing the sourcing gaps section -> Even if zero gaps exist, state "No sourcing gaps" explicitly.
- No notes for human review -> Always include at least one observation, even if it's "execution was clean — no concerns."
