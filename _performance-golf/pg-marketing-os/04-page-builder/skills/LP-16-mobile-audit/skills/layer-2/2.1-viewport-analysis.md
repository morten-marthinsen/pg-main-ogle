# 2.1 — Viewport Analysis

> **Skill:** LP-16-mobile-audit
> **Layer:** 2 (Mobile Audit)
> **Model:** opus
> **Input:** `assembled-page-load.md`, `mobile-criteria.md`, `device-context.md`
> **Output:** `viewport-analysis.md`
> **Covers Dimensions:** Viewport Fit (primary), Typography (partial), Navigation (partial)

---

## PURPOSE

Analyze the above-fold content of the assembled page against mobile viewports. Determine what is visible on 375px, 390px, and 414px screen widths without scrolling. Identify truncation, overflow, and critical element loss. Evaluate typography sizing and navigation clarity as they relate to viewport behavior.

**POSITIONAL REINFORCEMENT:** You are auditing the mobile viewport. Start from 375px width. Do NOT evaluate from a desktop perspective. Every finding must reference a specific viewport width.

---

## PROCEDURE

### Step 1: Inventory Above-Fold Elements

From `assembled-page-load.md`, identify every element that appears in Section 1 (Hero / Above-Fold):
- Headline text (count words, estimate line count at mobile font size)
- Subheadline/deck copy (count words, estimate line count)
- Hero image/video (note dimensions if specified)
- CTA button (note text, position, size if specified)
- Trust signals (badges, ratings, partner logos)
- Navigation bar (if present)

### Step 2: Calculate Viewport Fit

For each priority viewport width (from `device-context.md`), estimate whether above-fold elements fit:

**Estimation Method:**
- Standard mobile viewport HEIGHT: 667px (iPhone SE), 844px (iPhone 14), 896px (iPhone Plus)
- Navigation bar: ~60px
- Headline (24-32px font): ~40-60px per line
- Subheadline (16-18px font): ~25-35px per line
- Hero image: estimate from assembled page specs or default 200-300px
- CTA button: ~50-60px (including padding)
- Trust signal bar: ~40-60px
- Padding/margins between elements: ~80-120px total

**Calculate:** Sum all element heights. Compare to viewport height. Is CTA visible without scrolling?

### Step 3: Check for Horizontal Overflow

Identify any elements that might exceed viewport width:
- Tables without responsive wrapping
- Images without max-width constraint
- Long unbroken strings (URLs, product names)
- Fixed-width elements

### Step 4: Evaluate Typography for Viewport

Check font size specifications from the assembled page:
- Body copy: minimum 16px (iOS auto-zoom threshold)
- Headlines: minimum 24px on mobile
- Line height: minimum 1.4x body font
- Characters per line: maximum ~55 at 375px width

### Step 5: Assess Navigation Clarity

For the above-fold area:
- Is the visual hierarchy clear? (headline > subheadline > CTA)
- Is the scroll direction obvious?
- Are there competing elements that confuse the eye path?

## CRITICAL RULES

1. **Start from 375px.** Not 768px. Not 1024px. The smallest common mobile viewport is the baseline.
2. **"CTA visible" means without scrolling.** If the user must scroll even once to see the CTA, it is NOT above the fold.
3. **Estimate conservatively.** If element heights are uncertain, estimate HIGH (assume more space is needed). Better to flag a potential issue than miss one.
4. **"Unspecified" is a finding.** If the assembled page does not specify font sizes, image dimensions, or button sizes, that is a mobile risk — implementation will default to unknown behavior. Score accordingly.
5. **Type A vs Type B CTA rules differ.** Type A: CTA may be in deck copy below hero but within first viewport. Type B: CTA + price MUST be above fold. Check `mobile-criteria.md` for the correct threshold.
