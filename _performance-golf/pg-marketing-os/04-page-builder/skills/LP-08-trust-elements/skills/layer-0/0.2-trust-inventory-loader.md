# 0.2 — Trust Inventory Loader

**Skill:** LP-08-trust-elements
**Position:** Layer 0, Microskill 2
**Purpose:** Extract and organize all available trust assets from the loaded upstream packages into a comprehensive trust inventory. This inventory is the ONLY source of truth for what trust elements can be generated — if it is not in the inventory, it cannot appear in the output.
**Effort Level:** low
**Dependencies:** `packages-loaded.md` (0.1)
**Output:** `trust-inventory.md`

---

## Input Schema

```yaml
input:
  packages_loaded: "[from packages-loaded.md — all extracted fields]"
  project_directory: "[path to project directory for any supplemental files]"
```

---

## Execution Logic

1. **Read `packages-loaded.md`** and extract every trust-relevant data point.

2. **Build the trust inventory by type:**

   **Guarantee Trust Signals:**
   - Guarantee type (money-back, performance, satisfaction, results-based)
   - Guarantee duration (30-day, 60-day, 90-day, 365-day, lifetime)
   - Guarantee conditions (no questions asked, partial refund, restocking fee, etc.)
   - Guarantee branded name (if any — e.g., "NQARP" for LMNT)

   **Certification Trust Signals:**
   - List each certification: name, issuing body, what it certifies
   - Mark whether explanation copy exists or needs to be written
   - Health-specific: GMP, FDA-registered facility, Prop 65, NSF, Informed Sport, COA (Certificate of Analysis)
   - General: BBB rating, ISO certification, SOC 2 compliance

   **Media Mention Trust Signals:**
   - List each publication: name, date of mention, key claim/quote (if available), URL (if available)
   - Mark whether the mention is verifiable

   **Authority Trust Signals:**
   - List each expert: full name, credentials, institution, endorsement statement (if available)
   - Mark credential relevance to the product category

   **Volume Trust Signals:**
   - Customer count (specific number + date/source)
   - Review count (specific number + platform + rating)
   - Units sold (specific number + timeframe)
   - Subscriber count (if subscription product)

   **Shipping/Logistics Trust Signals:**
   - Free shipping threshold or policy
   - Delivery timeframe
   - Return/exchange policy

   **Security Trust Signals:**
   - Payment processor (Stripe, PayPal, etc.)
   - SSL/encryption details
   - Data privacy certifications

3. **For each trust signal, mark the source:**
   - `from_brief` — extracted from page-brief.json
   - `from_proof_architecture` — identified in proof-architecture.json
   - `from_supplemental` — found in supplemental project files
   - `requires_sourcing` — planned but not yet available

4. **Generate availability summary:** Count available signals per type. Flag any type with zero signals as `not_available`.

5. **Write `trust-inventory.md`** with the complete organized inventory.

## Quality Criteria
- Every trust signal from all upstream packages captured — no signals silently dropped
- Each signal has a source attribution (traceable to upstream file)
- Specific numbers used where available (not rounded or estimated)
- Gaps explicitly documented — Layer 1 needs to know what is and is not available
- Inventory is the single source of truth for all downstream trust copy generation

## Failure Modes
- Inventing trust signals not present in upstream packages → Every signal must trace to a source. If not in any upstream file, it is not in the inventory.
- Rounding specific numbers during extraction → If upstream says "1,270,268" extract "1,270,268" not "over 1 million."
- Marking a type as available when the data is vague → "Doctors recommend" without a named doctor is NOT an available authority signal.
- Skipping security/shipping signals as "not important" → Load everything available. Layer 1 decides what to use.
