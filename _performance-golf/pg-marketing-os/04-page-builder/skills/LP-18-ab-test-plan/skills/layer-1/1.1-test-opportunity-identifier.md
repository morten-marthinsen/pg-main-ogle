# 1.1 — Test Opportunity Identifier

> **Skill:** LP-18-ab-test-plan
> **Layer:** 1
> **Position:** 1 of 2
> **Purpose:** Identify all testable elements from audit failures, competitive gaps, specimen patterns, and standard high-impact areas. Map each to a test category (headline, CTA, proof, offer, layout, mobile).
> **Effort Level:** sonnet
> **Dependencies:** `audit-results-load.md`, `benchmark-load.md`, `assembled-page-load.md`
> **Output:** `test-opportunities.md`

---

## INPUT

```yaml
input:
  audit_results: "audit-results-load.md"
  benchmarks: "benchmark-load.md"
  assembled_page: "assembled-page-load.md"
```

## EXECUTION

**POSITIONAL REINFORCEMENT:** You are identifying test OPPORTUNITIES — raw, unranked, comprehensive. Cast a wide net. Include everything worth testing. The Prioritizer (1.2) will rank them. Do not pre-filter.

### Step 1: Extract Audit-Driven Opportunities

From `audit-results-load.md`, identify every FAIL point as a test opportunity:

For each FAIL point:
- Map to a test category: headline | cta | proof | offer | layout | mobile
- Note the specific element that failed
- Note LP-17's recommended fix (this becomes the basis for the test variant)
- Note the expected lift from LP-17

**Mapping rules:**
| Audit Point | Test Category |
|------------|---------------|
| 01 Headline promise | headline |
| 02 Hero image | layout |
| 03 CTA above fold | cta |
| 04 Trust signals | proof |
| 05 Reading level | headline (if headline), layout (if body) |
| 06 AI telltales | headline or layout (depends on location) |
| 07 Single offer | offer |
| 08 Benefits first | layout |
| 09 Proof early | proof |
| 10 Proof volume | proof |
| 11 Proof specificity | proof |
| 12 Before/after | proof |
| 13 Value before price | offer |
| 14 Guarantee quality | offer |
| 15 CTA count | cta |
| 16 Price anchoring | offer |
| 17 Page load | mobile |
| 18 Sticky ATC | mobile |
| 19 Image optimization | mobile |
| 20 Touch targets | mobile |

### Step 2: Extract Benchmark-Driven Opportunities

From `benchmark-load.md`, scan the A/B Test Priority Data for standard high-impact test areas. For each P1 and P2 priority element, check if it represents a test opportunity for this specific page — even if the audit did not flag it.

**Standard high-impact areas to always consider:**
- Headline type/angle (P1 — very high variance)
- Hero image type (P1 — high variance)
- Above-fold layout (P1 — high variance)
- CTA personalization level (P2 — +202% documented)
- Price display format (P2 — medium-high variance)
- Guarantee placement/prominence (P2 — medium variance)
- Social proof type/format (P2 — medium variance)

For each: compare the current state (from `assembled-page-load.md`) against the benchmark best practice. If a gap exists, add as a test opportunity.

### Step 3: Extract Specimen-Driven Opportunities

From `benchmark-load.md` (specimen patterns section), identify competitive gaps:

- Is the page using a headline type that was underrepresented in the specimen dataset? Consider testing a type that was more common (result/outcome headlines = 4/11).
- Is proof density aligned with the bimodal pattern (either extreme or zero)? If mid-range, consider testing higher density.
- Is CTA text consistent throughout the page? 7/11 specimens used consistent text. If the page varies, consider testing unified CTA copy.
- Is there a competitive gap the page could exploit? (e.g., charity tie-in, performance guarantee, before/after)

### Step 4: Check for PASS Points with Optimization Potential

Even PASS points from LP-17 may have optimization potential. Check:
- Did the headline PASS but use a common/expected format? Could a less conventional format (e.g., curiosity-driven for a page using direct benefit) outperform?
- Did CTA PASS but use generic text ("Buy Now")? Personalized CTA (+202%) is the highest-documented single-element lift.
- Did proof PASS but with minimal volume? More proof may improve conversion even if the minimum threshold was met.

### Step 5: Compile Opportunity List

For each opportunity:
- Assign a temporary ID (OPP-1, OPP-2, etc.)
- Assign a test category
- Describe the current state (from assembled page)
- Describe the test concept (what would change)
- Note the source (audit failure / benchmark gap / specimen pattern / optimization)
- Note the expected impact level (from benchmark data)

### Step 6: Deduplicate

Check for overlapping opportunities. If two sources point to the same test (e.g., audit flags headline AND benchmark data says headline is P1), merge into a single opportunity with both sources noted.

### Step 7: Write Test Opportunities

Write the full list to `test-opportunities.md`.

## FAILURE MODES

- Zero audit failures (page scored 20/20) -> Shift focus to benchmark gaps, specimen patterns, and PASS point optimization. Should still find 3-5 opportunities. If genuinely zero opportunities, document reasoning and flag for human review.
- Overwhelming number of opportunities (15+) -> Include all in the list. The Prioritizer (1.2) will cut to the top 5-10. Do not self-censor at this stage.
- Missing assembled page data for current state -> Flag the opportunity as having INCOMPLETE control data. Note what is missing. Layer 2 generators will need to handle this gap.
