# 2.5 — Layout Test Generator

> **Skill:** LP-18-ab-test-plan
> **Layer:** 2
> **Position:** 5 of 5
> **Purpose:** Generate layout/structure A/B test hypotheses covering section order, content length, mobile layout variations, above-fold composition, and page structure changes.
> **Effort Level:** opus
> **Dependencies:** `prioritized-opportunities.md`, `assembled-page-load.md`, `benchmark-load.md`
> **Output:** `layout-tests.md`

---

## INPUT

```yaml
input:
  prioritized_opportunities: "prioritized-opportunities.md"
  assembled_page: "assembled-page-load.md"
  benchmarks: "benchmark-load.md"
  page_type: "[from assembled-page-load.md]"
```

---

## EXECUTION

**POSITIONAL REINFORCEMENT:** You are generating SPECIFIC layout test hypotheses. Every hypothesis must cite the exact current page structure and propose a specific, implementable change. "Restructure the page" is forbidden. Layout tests have higher implementation difficulty than copy tests — rate accordingly.

### Step 1: Check if Layout Tests Are Warranted

Read `prioritized-opportunities.md`. Only proceed if layout-category or mobile-category opportunities appear. If none, write `layout-tests.md` with: "No layout test opportunities prioritized for this page." and exit.

### Step 2: Extract Current Layout State

From `assembled-page-load.md`, extract:
- Section inventory (ordered list with names)
- Above-fold composition (headline, image, CTA, trust signals — exact elements)
- Total section count
- Estimated word count / page length classification
- Mobile specifications (sticky bar, touch targets, image format)
- Hero image/video format
- Section that contains the first CTA
- Section that contains the first proof element

### Step 3: Generate Layout Test Hypotheses

For each layout/mobile opportunity, generate 1-2 specific hypotheses from these test dimensions:

**Dimension A: Section Order**
From cross-page-pattern-analysis.md — section sequence varies by page type and vertical:
| Current Order | Test Against | Reasoning |
|-------------|-------------|-----------|
| Hero -> Features -> Proof -> Offer | Hero -> Proof -> Features -> Offer | Moving proof before features shortens time-to-trust |
| Proof deep in page (section 5+) | Proof in section 2-3 | CXL: early proof placement = significant lift |
| Guarantee at bottom | Guarantee immediately after pricing | Risk reversal nearest to price reduces friction at decision point |
| FAQ after offer | FAQ before offer | Objection-handling before the ask removes friction |
| Benefits section before mechanism | Mechanism before benefits | For problem-aware audiences, understanding the mechanism first increases benefit credibility |

**Dimension B: Above-Fold Composition**
| Current Composition | Test Against | Reasoning |
|--------------------|-------------|-----------|
| Text headline + image | VSL/video hero | +86% for video (VWO). Strongest for products needing demonstration |
| VSL hero | Text headline + proof badges | Some audiences prefer scanning to watching |
| No trust signals above fold | Add trust signals above fold | 5/11 specimens show proof or trust signals above fold |
| Image hero | Lifestyle/outcome image | Product-in-context vs aspirational outcome image |
| CTA below fold (Type A) | CTA above fold | DTC-specific pattern: 3/5 Type B specimens have purchase CTA above fold |

**Dimension C: Page Length / Content Density**
From conversion-data-reference.md — long-form generates +220% more leads:
| Current State | Test Against | Reasoning |
|-------------|-------------|-----------|
| Short page (<2,000 words) | Extended page with education sections | +220% leads for long-form vs short-form (KlientBoost) |
| Very long page (>10,000 words) | Condensed version cutting low-value sections | Diminishing returns beyond optimal length. Test removing weakest sections |
| Dense text-heavy sections | Visual-heavy (icons, images, charts) interspersed with text | Break up text walls for scan-reading visitors |

**Dimension D: Mobile-Specific Layout**
| Current State | Test Against | Reasoning |
|-------------|-------------|-----------|
| No sticky CTA bar on mobile | Add sticky ATC/CTA bar | Highest-impact single mobile change (conversion-data-reference.md) |
| Standard mobile layout | Accordion FAQ on mobile (collapsed by default) | Reduces scroll depth on mobile — shorter perceived page |
| Full-width images on mobile | Smaller/compressed images with text visible alongside | Faster load + more content visible without scrolling |
| Desktop section order on mobile | Reordered for mobile (proof higher, long text lower) | Mobile users scroll less — key elements must appear earlier |

**Dimension E: Content Format**
| Current Format | Test Against | Reasoning |
|---------------|-------------|-----------|
| No video on page | Add explainer/demo video | +86% for video on landing pages (VWO) |
| Long text testimonials | Video testimonials | Video proof more credible, 4/11 specimens use video |
| Static FAQ | Interactive/accordion FAQ | Reduces visual clutter, lets users self-select relevant questions |
| No comparison table | Add product comparison table | BIOptimizers pattern — especially effective for supplement pages |

### Step 4: Write Each Hypothesis Using the Schema

```yaml
test_id: "T-[N]"
category: "layout"  # or "mobile" for mobile-specific tests
hypothesis: "If we change [exact layout element] from [control] to [variant], conversion rate will increase by [X-Y%] because [reasoning]."
control:
  description: "[current layout state]"
  specific_text_or_element: "[exact current structure — section order, above-fold elements, mobile specs]"
variant:
  description: "[proposed change]"
  specific_text_or_element: "[exact proposed change — which sections move where, what elements add/remove]"
expected_lift: "[X-Y%]"
lift_source: "[citation]"
implementation_difficulty: "[medium | high]"
```

**NOTE:** Layout tests are almost always "medium" or "high" implementation difficulty. They require design/development work, not just copy swaps. Rate honestly.

### Step 5: Write Layout Tests

Compile all layout test hypotheses into `layout-tests.md`.

---

## QUALITY CRITERIA

- Only generates tests if layout or mobile opportunities were prioritized
- Current layout state fully documented from assembled page
- Each hypothesis specifies exact structural change (not "reorganize the page")
- Implementation difficulty honestly rated (layout changes are medium-high, not low)
- Mobile-specific tests labeled with category "mobile" in the schema
- Lift estimates cite benchmark data (video +86%, long-form +220%, mobile-responsive +25.2%)
- File exists after execution

---

## FAILURE MODES

- Section inventory incomplete from assembled page -> Note gaps. Propose tests based on available structural data.
- Proposing tests that are essentially page redesigns -> If a test requires changing 3+ sections simultaneously, it is a redesign, not an A/B test. Split into smaller, single-variable tests or flag as "redesign recommendation" rather than an A/B test.
- Mobile-specific tests when no mobile specs available -> Note that mobile behavior is assumed from standard responsive patterns. Flag for validation with actual mobile screenshot/testing.
- Layout tests that interact with copy tests -> If a section-order change moves the headline to a different position, note the interaction with any headline tests. Flag as potentially conflicting.
