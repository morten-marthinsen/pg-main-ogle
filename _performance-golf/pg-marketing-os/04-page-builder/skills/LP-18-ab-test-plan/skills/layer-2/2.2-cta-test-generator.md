# 2.2 — CTA Test Generator

> **Skill:** LP-18-ab-test-plan
> **Layer:** 2
> **Position:** 2 of 5
> **Purpose:** Generate CTA A/B test hypotheses covering button copy, button size/color, CTA count/placement, P.S. inclusion, risk-reversal attachment, and sticky CTA behavior.
> **Effort Level:** opus
> **Dependencies:** `prioritized-opportunities.md`, `assembled-page-load.md`, `benchmark-load.md`
> **Output:** `cta-tests.md`

---

## INPUT

```yaml
input:
  prioritized_opportunities: "prioritized-opportunities.md"
  assembled_page: "assembled-page-load.md"
  benchmarks: "benchmark-load.md"
  page_type: "[from assembled-page-load.md]"
```

---

## EXECUTION

**POSITIONAL REINFORCEMENT:** You are generating SPECIFIC CTA test hypotheses. Every hypothesis must cite the exact current CTA text, placement, or format as the control. "Test a better CTA" is forbidden.

### Step 1: Check if CTA Tests Are Warranted

Read `prioritized-opportunities.md`. Only proceed if CTA-category opportunities appear in the selected list. If none, write `cta-tests.md` with: "No CTA test opportunities prioritized for this page." and exit.

### Step 2: Extract Current CTA State

From `assembled-page-load.md`, extract:
- All CTA instances (text, position, format)
- CTA consistency (same text throughout or varying)
- Risk reversal attached to CTA (yes/no, text if yes)
- Sticky CTA bar (specified or not)
- CTA specificity level (generic "Buy Now" vs personalized "Get My 60-Day Supply")

### Step 3: Generate CTA Test Hypotheses

For each CTA opportunity, generate 1-2 specific hypotheses from these test dimensions:

**Dimension A: CTA Copy Personalization**
The highest-documented single-element lift in the benchmark data (+202%, HubSpot).
| Current CTA | Test Against | Expected Impact |
|------------|-------------|-----------------|
| "Buy Now" | "Get My [Specific Product] — Risk-Free" | Very High (+50-202%) |
| "Add to Cart" | "Start My [Specific Benefit] Today" | High (+20-100%) |
| "Learn More" | "[Action] + [Specific Outcome]" | High (+30-100%) |
| "Get Started" | "Get My [Specific Deliverable]" | Medium-High (+20-80%) |

**Dimension B: CTA Count/Placement**
- Current: 2 CTAs -> Test: 4+ CTAs at key decision points (after proof, after guarantee, after urgency)
- Current: CTA below fold only -> Test: CTA above fold + repeated
- Current: No CTA after proof section -> Test: Add CTA immediately after proof

Reference cross-page-pattern-analysis.md:
- MAMG: 4 CTAs (all "start my free trial")
- BIOptimizers: 8+ CTAs
- Richmond Dinh: 8+ CTAs
- LMNT: only 2 CTAs
Specimen data shows 4-8 CTAs is the high-converting range for long pages.

**Dimension C: Risk Reversal at CTA**
- Current: No risk language at CTA -> Test: Add "Cancel anytime" or guarantee reminder below button
- Current: Generic risk reversal -> Test: Specific guarantee callout ("Protected by our 365-Day Full Refund Policy")

Reference: 6/11 specimens attach risk reversal directly at CTA. The pages that do NOT (Hyros, LMNT) defer it to a separate section.

**Dimension D: CTA Format**
- Current: Standard button -> Test: Button + micro-copy below ("4,582 customers this month")
- Current: Text CTA -> Test: Button CTA with high-contrast color
- Current: No sticky CTA on mobile -> Test: Add sticky ATC/CTA bar

**Dimension E: P.S. Section**
- Current: No P.S. -> Test: Add P.S. section above final CTA (summary of offer + urgency + guarantee)
- Current: P.S. exists -> Test: P.S. with different emotional appeal

### Step 4: Write Each Hypothesis Using the Schema

```yaml
test_id: "T-[N]"
category: "cta"
hypothesis: "If we change [exact current CTA element] from [control] to [variant], conversion rate will increase by [X-Y%] because [reasoning]."
control:
  description: "[current CTA state]"
  specific_text_or_element: "[exact current CTA text/format/placement]"
variant:
  description: "[proposed change]"
  specific_text_or_element: "[exact proposed CTA text/format/placement]"
expected_lift: "[X-Y%]"
lift_source: "[citation]"
implementation_difficulty: "[low | medium]"
```

### Step 5: Write CTA Tests

Compile all CTA test hypotheses into `cta-tests.md`.

---

## QUALITY CRITERIA

- Only generates tests if CTA opportunities were prioritized
- All current CTA instances documented from assembled page
- Each hypothesis specifies exact control and variant
- CTA copy variants are specific (not "better CTA text")
- Lift estimates cite HubSpot +202% for personalization or other benchmark data
- File exists after execution

---

## FAILURE MODES

- CTA text not extractable from assembled page -> Use description of CTA format/position as control. Flag as INCOMPLETE CONTROL.
- Too many CTA test ideas -> Limit to 2-3 strongest hypotheses. Prioritize personalization (highest documented lift) first.
- Proposing CTA changes that conflict with single-offer rule -> All CTA variants must still point to the same conversion action. Do not test a CTA that introduces a second offer.
