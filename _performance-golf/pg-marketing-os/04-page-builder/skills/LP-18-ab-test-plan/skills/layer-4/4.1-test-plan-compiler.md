# 4.1 — Test Plan Compiler

> **Skill:** LP-18-ab-test-plan
> **Layer:** 4
> **Position:** 1 of 3
> **Purpose:** Compile all validated, sized hypotheses into the final `ab-test-plan.json` output file following the schema defined in LP-18-AGENT.md.
> **Effort Level:** haiku
> **Dependencies:** `validated-hypotheses.md`, `sample-sizes.md`, `assembled-page-load.md`, `audit-results-load.md`
> **Output:** `ab-test-plan.json`

---

## INPUT

```yaml
input:
  validated_hypotheses: "validated-hypotheses.md"
  sample_sizes: "sample-sizes.md"
  assembled_page: "assembled-page-load.md"
  audit_results: "audit-results-load.md"
  benchmark_load: "benchmark-load.md"
```

---

## EXECUTION

### Step 1: Read All Source Files

Read all 5 input files and extract the data needed for the JSON schema defined in LP-18-AGENT.md.

### Step 2: Assemble JSON Structure

Build `ab-test-plan.json` following the schema:

1. **Header fields:**
   - `schema_version`: "1.0"
   - `created`: ISO timestamp
   - `project_name`: from assembled page or audit results
   - `page_type`: from assembled page
   - `page_url`: from page brief (if available, else null)

2. **Baseline data:**
   - `baseline_conversion_rate`: from sample-sizes.md parameters
   - `baseline_source`: from sample-sizes.md
   - `estimated_monthly_traffic`: from sample-sizes.md (or "unknown")

3. **Audit context:**
   - `audit_score`: from audit-results-load.md
   - `audit_verdict`: from audit-results-load.md

4. **Tests array:**
   For each validated hypothesis (in priority-rank order):
   - `test_id`: from validated-hypotheses.md
   - `category`: from the hypothesis
   - `hypothesis`: full hypothesis statement
   - `control`: { description, specific_text_or_element }
   - `variant`: { description, specific_text_or_element }
   - `expected_lift`: from the hypothesis
   - `lift_source`: from the hypothesis
   - `implementation_difficulty`: from the hypothesis
   - `sample_size`: { per_variant, total, minimum_detectable_effect, confidence_level, power, estimated_duration } from sample-sizes.md
   - `priority_rank`: from validated-hypotheses.md
   - `conflicts_with`: from validated-hypotheses.md conflict map
   - `notes`: any additional context from validation or sizing

5. **Test sequence recommendation:**
   - `phase_1`: test IDs from sample-sizes.md Phase 1
   - `phase_2`: test IDs from Phase 2
   - `phase_3`: test IDs from Phase 3
   - `rationale`: from sample-sizes.md sequence rationale

6. **Metadata:**
   - All source file paths
   - Benchmark data version
   - Specimen data version

### Step 3: Validate JSON Completeness

Before writing, verify:
- `test_count` matches the actual number of tests in the array
- Every test has all required fields (no nulls on required fields)
- `priority_rank` values are unique and sequential (1, 2, 3...)
- `conflicts_with` arrays reference only valid test IDs
- Sample size fields are populated for every test
- Test sequence includes all test IDs across all phases

### Step 4: Write ab-test-plan.json

Write the complete JSON file to the project output folder.

---

## OUTPUT SCHEMA

`ab-test-plan.json` — follows the complete schema defined in LP-18-AGENT.md OUTPUT SCHEMA section.

Key validation rules:
- `test_count` must be an integer 5-10
- Each test `test_id` must be unique (T-1, T-2, etc.)
- Each test must have non-null: hypothesis, control, variant, expected_lift, lift_source, sample_size, priority_rank
- `confidence_level` must be "95%" for all tests
- `power` must be "80%" for all tests
- `conflicts_with` must be an array (empty array if no conflicts)
- `test_sequence_recommendation` must include all test IDs distributed across phases
- `baseline_conversion_rate` must be a percentage string (e.g., "3%")

---

## QUALITY CRITERIA

- JSON is well-formed and parseable
- All validated tests present with complete data
- Priority ranking is sequential and unique
- Sample sizes match calculations from sample-sizes.md
- Conflict map is consistent (if T-1 conflicts with T-3, then T-3 must conflict with T-1)
- Test sequence accounts for all tests
- Metadata includes all source file paths
- File exists after execution

---

## FAILURE MODES

- JSON malformed -> Re-check structure. Ensure all brackets, quotes, and commas are correct. Pay special attention to nested objects (control, variant, sample_size).
- Missing data from source files -> Include the field with a value of "DATA_MISSING — [source file] did not contain this field" rather than omitting it.
- Test count outside 5-10 range -> If fewer than 5 validated tests exist, include all available and add a `warnings` field: "Fewer than 5 tests validated. Consider re-running Layer 2 generators."
- Priority rank mismatch -> Re-number sequentially from 1 based on the order in validated-hypotheses.md.
