# 3.1 — Hypothesis Validator

> **Skill:** LP-18-ab-test-plan
> **Layer:** 3
> **Position:** 1 of 2
> **Purpose:** Validate each hypothesis generated in Layer 2: Is it testable? Is the expected lift realistic? Is the control clearly defined? Is the variant specific enough to implement? Flag conflicts between tests.
> **Effort Level:** sonnet
> **Dependencies:** All Layer 2 test files (`headline-tests.md`, `cta-tests.md`, `proof-tests.md`, `offer-tests.md`, `layout-tests.md`)
> **Output:** `validated-hypotheses.md`

---

## INPUT

```yaml
input:
  headline_tests: "headline-tests.md"
  cta_tests: "cta-tests.md"
  proof_tests: "proof-tests.md"
  offer_tests: "offer-tests.md"
  layout_tests: "layout-tests.md"
```

## EXECUTION

**POSITIONAL REINFORCEMENT:** You are VALIDATING hypotheses, not generating new ones. Your job is to catch vague, unrealistic, untestable, or conflicting hypotheses before they enter the final test plan. Be strict. A vague hypothesis wastes real traffic and money.

### Step 1: Collect All Hypotheses

Read all Layer 2 test files. Compile a master list of all generated hypotheses. Note: not all files may contain hypotheses (some categories may have been skipped). Only read files that exist.

### Step 2: Validate Each Hypothesis Against 8 Criteria

For each hypothesis, run the following validation checks:

**Check 1: Control Defined (PASS/FAIL)**
Does the control field contain the exact current text/element from the assembled page?
- FAIL if: control says "the current headline" without quoting it
- FAIL if: control field is empty or says "current state"
- PASS if: control contains verbatim text or specific element description

**Check 2: Variant Implementable (PASS/FAIL)**
Could a developer or copywriter implement the variant without asking clarifying questions?
- FAIL if: variant says "a different headline" or "better CTA"
- FAIL if: variant requires decisions the hypothesis does not make
- PASS if: variant specifies the type of change, the direction, and enough detail to execute

**Check 3: Lift Estimate Realistic (PASS/FAIL)**
Does the expected lift fall within the documented range for that element type?
- FAIL if: predicted lift exceeds the "Maximum Documented" range from ANTI-DEGRADATION.md without extraordinary justification
- FAIL if: lift is stated without any source or reasoning
- PASS if: lift cites benchmark data or is labeled "estimated" with reasoning

**Check 4: Single Variable (PASS/FAIL)**
Does the test change ONE variable, or is it a multivariate test disguised as A/B?
- FAIL if: the test changes the headline AND the hero image AND the CTA simultaneously
- PASS if: the test changes one element (or a tightly coupled pair, e.g., CTA text + CTA subtext)

**Check 5: Testable (PASS/FAIL)**
Can this be run as a split test on the existing page?
- FAIL if: the test requires a full page rebuild
- FAIL if: the test requires business model changes that cannot be variant-tested (e.g., "switch to subscription pricing" when the page has no subscription infrastructure)
- PASS if: the change can be implemented as a page variant

**Check 6: Measures Conversion (PASS/FAIL)**
Does the test's expected outcome relate to the page's conversion goal?
- FAIL if: the test optimizes for engagement, time-on-page, or aesthetic preference rather than conversion rate
- PASS if: the hypothesis explicitly states conversion rate as the metric

**Check 7: No Duplicate Element (FLAG)**
Does this test modify the same element as another test?
- FLAG if: two tests both modify the headline, or both modify the primary CTA
- Note: flagged tests are not rejected — they are marked as conflicting and must be sequenced

**Check 8: Adequate MDE (PASS/FAIL)**
Is the expected lift large enough to detect with reasonable traffic?
- FAIL if: expected lift is <2% absolute (requires enormous sample to detect)
- PASS if: expected lift is 2%+ absolute or 10%+ relative of baseline

### Step 3: Classify Each Hypothesis

Based on the 8 checks:
- **VALIDATED:** All 8 checks PASS (flags noted but not blocking)
- **NEEDS_REVISION:** 1-2 checks FAIL — fixable issues. Revise the hypothesis inline.
- **REJECTED:** 3+ checks FAIL — remove from test plan. Document rejection reason.

### Step 4: Revise NEEDS_REVISION Hypotheses

For hypotheses with 1-2 failures:
- If control is missing: extract from assembled-page-load.md
- If variant is vague: add specifics based on the test dimension and benchmark data
- If lift is unrealistic: adjust to the documented range for that element type
- After revision, re-validate. If still failing, reclassify as REJECTED.

### Step 5: Detect Conflicts

Build a conflict map:
- Group all validated tests by the element they modify
- If two tests modify the same element -> mark as conflicting (cannot run simultaneously)
- If two tests modify elements in the same page zone (e.g., both in above-fold) -> mark as potentially interacting
- Assign sequence recommendation: higher-priority test first

### Step 6: Assign Final Priority Ranks

Re-rank all VALIDATED hypotheses 1 through N, incorporating:
- Original priority from Layer 1
- Validation confidence (fully valid > revised)
- Conflict sequencing (non-conflicting tests can run in parallel)

### Step 7: Write Validated Hypotheses

Compile results into `validated-hypotheses.md`.

---

## QUALITY CRITERIA

- All Layer 2 hypotheses evaluated against all 8 checks
- Every VALIDATED hypothesis passes all non-flag checks
- Every REJECTED hypothesis has a documented reason
- Conflicts detected and mapped
- Priority ranks assigned to all validated tests
- Final validated count is 5-10 (if fewer, note reasoning)
- File exists after execution

---

## FAILURE MODES

- All hypotheses from a category fail validation -> Note the category gap. Do not generate replacement hypotheses (that is Layer 2's job). Flag for human review.
- More than 10 hypotheses pass validation -> Trim to 10 by removing the lowest-priority items. Include trimmed items in a "Backlog" section.
- Fewer than 5 hypotheses pass validation -> Flag as LOW_TEST_COUNT. Include any NEEDS_REVISION that could not be fixed. Recommend human review of rejected hypotheses to see if any can be salvaged.
- Conflict detection misses an interaction -> Conservative approach: if unsure whether two tests interact, flag them as potentially conflicting. Better to over-flag than to run interacting tests simultaneously.
