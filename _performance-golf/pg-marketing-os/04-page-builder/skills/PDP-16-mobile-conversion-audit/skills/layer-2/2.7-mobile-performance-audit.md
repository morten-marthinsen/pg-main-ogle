# 2.7 — Mobile Performance Audit

> **Skill:** PDP-16-mobile-conversion-audit
> **Layer:** 2 (Audit Execution)
> **Position:** 7 of 8
> **Model:** sonnet
> **Purpose:** Audit mobile performance specifications in the assembled PDP: touch targets, font sizes, image optimization, and viewport fit. These are foundation-level requirements — if touch targets are too small, users cannot interact with the page. If fonts trigger auto-zoom, the layout breaks. If images are not optimized, load time kills conversion.
> **Dependencies:** `assembled-page-load.md`, `pdp-reference-load.md`, `pdp-audit-criteria.md`, `benchmark-load.md`
> **Output:** `mobile-performance-audit.md`

## Execution Logic

**POSITIONAL REINFORCEMENT:** You are auditing mobile performance SPECIFICATIONS in the assembled page. You are not running a Lighthouse test. You are evaluating whether the specifications in the assembled page meet mobile performance standards. If a specification is absent, it is FAIL — unspecified behavior defaults to whatever the implementation team does, which may violate standards.

**Step 1:** From `assembled-page-load.md`, extract all size, spacing, and performance specifications:
- Button/link sizes and touch targets
- Font size specifications
- Image specifications (dimensions, responsive rules, lazy loading)
- Viewport/responsive behavior specifications

**Step 2:** Evaluate each criterion:

### Criterion 7.1: Touch Targets >= 44x44px
- **Source:** Apple Human Interface Guidelines + Google Material Design (minimum 44x44px) / Baymard 6.4: minimum 44x44px for variant chips
- **PASS:** All interactive elements (buttons, chips, links, steppers, accordion toggles) specify >= 44x44px touch targets. Key elements to check: ATC button, variant chips, quantity stepper (+/-), review histogram bars, keyword filter chips, accordion toggles, "Load More" button.
- **FAIL:** Any interactive element specifies < 44px, or touch target size not specified for interactive elements.
- **Evidence required:** List each interactive element and its specified size. Flag any below 44px or unspecified.

### Criterion 7.2: Font Sizes Readable
- **Mobile readability research:** Body text < 16px triggers iOS auto-zoom on input focus. Body text < 14px is difficult to read on mobile. Headlines < 20px lose visual hierarchy on small screens.
- **PASS:** Body text >= 14px (preferably >= 16px). Headlines >= 20px. No elements with unreadably small text (< 12px). Micro-trust signals and fine print >= 12px.
- **FAIL:** Any text specified below minimums, or font sizes not specified (implementation may default to desktop sizes).
- **Evidence required:** List font size specifications found in assembled page.

### Criterion 7.3: Image Optimization
- **Mobile performance principle:** Unoptimized images are the #1 cause of slow mobile page loads.
- **PASS:** Images specify responsive sizing (srcset or max-width: 100%), BTF images specify lazy loading, carousel images specify appropriate dimensions for mobile (not desktop-sized images served to mobile).
- **FAIL:** No responsive image specifications, no lazy loading for BTF content, or not specified.
- **Evidence required:** Note image specifications found or absent.

### Criterion 7.4: No Horizontal Scroll
- **Baymard 6.7:** Horizontal scroll on mobile PDP is a fundamental layout failure.
- **PASS:** No elements exceed 375px width (smallest common mobile viewport). Tables specify responsive wrapping. Images specify max-width: 100%. No fixed-width elements wider than viewport.
- **FAIL:** Any element specified with fixed width > 375px, tables without responsive handling, or viewport behavior not specified.
- **Evidence required:** Identify any elements that risk horizontal overflow.

**Step 3:** For each criterion, also check `benchmark-load.md` for relevant performance standards and thresholds.

**Step 4:** Write finding for each criterion.

**Step 5:** Write `mobile-performance-audit.md`.

## Quality Criteria

- All 4 criteria evaluated
- Touch target audit covers ALL interactive elements (not just ATC button)
- Font size audit covers multiple text types (body, headlines, fine print)
- UNSPECIFIED = FAIL (unspecified touch targets will likely be too small)
- File exists after execution

## Failure Modes

- No performance specifications in assembled page -> Score all 4 as FAIL. The assembled page should specify key performance parameters. Note this as a systemic gap.
- Some elements have specs, others don't -> Score per-criterion based on what's specified. List unspecified elements as part of FAIL evidence.
- Performance specs are in PDP-03/PDP-05/PDP-06 output but not in LP-15 assembly -> FAIL for the assembled page, but note the upstream source. LP-15 should carry specs through to assembly.
