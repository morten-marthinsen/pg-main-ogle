# 3.3 — Priority Ranking

> **Skill:** PDP-16-mobile-conversion-audit
> **Layer:** 3 (Scoring + Prioritization)
> **Position:** 3 of 3
> **Model:** opus
> **Purpose:** Rank ALL failed/partial items by (conversion impact x fix effort). This is the most actionable output of the entire audit — it tells the team exactly what to fix first for maximum conversion lift with minimum effort. Priority ranking is NOT optional, even at high composite scores.
> **Dependencies:** `per-section-scores.md`, `composite-score.md`, all 8 Layer 2 audit files, `pdp-reference-load.md`, `benchmark-load.md`

## Execution Logic

**POSITIONAL REINFORCEMENT:** This is the most important output of PDP-16. A composite score without a priority ranking is a grade without guidance. The team needs to know: "Fix THIS first, THEN this, THEN this." Every FAIL item from every section must appear in this ranking.

**Step 1:** Extract the complete FAIL list from `per-section-scores.md` (which aggregated all FAILs from the 8 audit files). For each FAIL item, pull:
- Criterion ID (e.g., 1.1, 2.5, 5.3)
- Section name
- Criterion name
- Evidence summary (from the Layer 2 audit file)
- Fix recommendation (from the Layer 2 audit file)

**Step 2:** Assign Conversion Impact Weight to each FAIL item:

| Impact Level | Weight | Criteria |
|-------------|--------|----------|
| Critical (5) | 5 | Documented Baymard statistics with specific % impact. Items in the buy box (25% weight section). Items where the stat shows 40%+ user impact. Examples: thumbnails (50-80% miss content), dropdown variants (60% more friction), sticky ATC (4-10x viewport). |
| High (4) | 4 | Documented Baymard findings without specific % but with clear conversion mechanism. Items in high-weight sections (social proof 15%, image gallery 12%). Examples: review histogram tappability, UGC positioning, back-button traps. |
| Medium (3) | 3 | Best practice violations with logical but not specifically quantified impact. Items in medium-weight sections (navigation 10%, mobile perf 10%, product info 10%). Examples: breadcrumbs, font sizes, error state handling. |
| Low (2) | 2 | Supporting criteria that contribute to overall quality but have indirect conversion impact. Examples: in-scale image, ingredient naming format, FAQ accordion behavior. |

**Impact assignment rules:**
- If a Baymard stat cites a specific percentage -> Critical (5) or High (4)
- If the criterion is in the Buy Box section -> minimum High (4)
- If the criterion is about the cart/checkout path -> minimum High (4)
- If the criterion prevents basic usability (touch targets, font sizes) -> minimum Medium (3)

**Step 3:** Assign Fix Effort Weight to each FAIL item:

| Effort Level | Weight | Criteria |
|-------------|--------|----------|
| Quick Fix (5) | 5 | Specification change only — add a line to the assembled page spec. No structural redesign. No upstream skill rerun. Examples: add back-button spec, add lazy loading spec, add breadcrumb spec. |
| Minor Revision (4) | 4 | Small section revision in one upstream skill's output. Examples: change dot nav to thumbnails in PDP-03, add micro-trust signals to PDP-06 output. |
| Moderate Revision (3) | 3 | Upstream skill needs partial rerun or significant section edit. Examples: redesign review card template (PDP-05), add keyword filters (PDP-05), redesign subscription tiles. |
| Major Revision (2) | 2 | Upstream skill needs full rerun or multiple skills affected. Examples: redesign entire buy box UX (PDP-03 + PDP-06), restructure BTF sections (PDP-04 + PDP-07). |
| Structural Rework (1) | 1 | Fundamental architecture change requiring multiple skill reruns. Examples: convert dropdown variants to chips across all specs, add sticky ATC system from scratch. |

**Step 4:** Calculate Priority Score:

```
Priority Score = Conversion Impact Weight * Fix Effort Weight
```

Maximum: 25 (Critical impact * Quick fix)
Minimum: 2 (Low impact * Structural rework)

**Step 5:** Sort all FAIL items by Priority Score, descending. Tie-breaking:
1. Higher conversion impact wins
2. Higher section weight wins
3. Earlier section number wins

**Step 6:** Categorize into fix tiers:
- **Tier 1 — Quick Wins (Priority 15-25):** High impact, easy fix. Do these first.
- **Tier 2 — High Value (Priority 8-14):** Significant impact, moderate effort. Do these second.
- **Tier 3 — Planned Work (Priority 4-7):** Lower impact or higher effort. Schedule these.
- **Tier 4 — Backlog (Priority 2-3):** Low impact AND high effort. Track but deprioritize.

**Step 7:** Write `priority-ranking.md`.

## Quality Criteria

- Every FAIL item from per-section-scores.md appears in the ranking
- Impact weights justified with Baymard stats or section weight logic
- Effort weights justified with specific scope assessment
- Priority scores calculated correctly (impact * effort)
- Ranking sorted correctly with tie-breaking applied
- Tiers clearly separated with estimated score improvement
- Upstream skill map identifies which skills need revision
- File exists after execution

## Failure Modes

- Missing FAIL items -> Cross-reference against per-section-scores.md FAIL summary. Every item must appear.
- All items assigned same impact -> Differentiate using Baymard stats. Buy box items are higher impact than navigation items. Items with specific % stats are higher than items without.
- Effort assessment too generous (everything is "Quick Fix") -> Be honest about effort. Changing dot navigation to thumbnails requires PDP-03 revision, not just a spec line.
- Priority ranking skipped ("score is high enough") -> Non-negotiable. Even at 9.5/10, there are FAIL items worth fixing. Marginal gains compound.
- No upstream skill mapping -> Add it. The team needs to know which skills to rerun.
