# 2.4 — What to Expect Writer

**Skill:** PDP-07-btf-section-writer
**Position:** Layer 2, Microskill 4
**Purpose:** Write the What to Expect (Benefits Timeline) section — a step-by-step timeline showing when and how the product's benefits unfold over time. This section future-paces the customer, sets realistic expectations, and prevents premature doubt by reframing subtle early signals as progress.
**Effort Level:** high
**Dependencies:** `per-section-briefs.md` (1.1 — what_to_expect brief), `data-validation.md` (1.2 — GREEN or YELLOW), `pdp-reference-loaded.md` (0.3 — Section 5.5 template)
**Output:** `section-what-to-expect.md`

---

## Input Schema

```yaml
input:
  section_brief: "[what_to_expect brief from per-section-briefs.md]"
  word_count_target: "[from PDP-04]"
  proof_density: "[from PDP-04 — light: timeline icons, mechanism references]"
  pdp_reference: "[Section 5.5 template from pdp-reference-loaded.md]"
  timeline_data: "[from brief — milestones, mechanisms, expected results per phase]"
```

## Execution Logic

**Step 1:** Read the What to Expect template from PDP-BEST-PRACTICES-REFERENCE.md Section 5.5. Key pattern: 3-5 milestones on a vertical timeline with big circles at each milestone, benefit title + mechanism description.

**Step 2:** Define 3-5 timeline milestones from brief data. Standard timeline framework:

```
STANDARD MILESTONE FRAMEWORK:
  Phase 1: Immediate (Days 1-3 or "First Week")
    → What happens biochemically
    → What the user may notice (subtle signals)
    → Set expectation: "subtle" is normal and good

  Phase 2: Short-term (Week 1-2 or "First Two Weeks")
    → Deeper biochemical adaptation
    → More noticeable felt benefits
    → Set expectation: building momentum

  Phase 3: Medium-term (Week 3-4 or "First Month")
    → Mechanism reaching steady state
    → Clear felt difference
    → Set expectation: real results visible

  Phase 4: Long-term (Month 2-3 or "90 Days")
    → Cumulative benefits
    → Full mechanism expression
    → Set expectation: sustained improvement

  Phase 5: Extended (6+ months — optional)
    → Long-term adaptation
    → Compounding returns
    → Set expectation: maintained with continued use
```

Adjust phases based on product type and available data. Minimum 3, maximum 5.

**Step 3:** For each milestone, write:
- **Milestone Time Label:** Clear time marker (e.g., "Days 1-3," "Week 2," "Month 1")
- **Benefit Title:** Powerful benefit microscript headline (e.g., "Subtle Calm Emerges," "Steady Energy Throughout the Day")
- **Description:** 2-3 sentences connecting:
  1. What's happening biologically/mechanistically at this stage
  2. What the user will notice/feel
  3. Why this timing is expected (the mechanism needs X time)

**Step 4:** Frame early milestones to prevent premature doubt:
- Phase 1 should NOT promise dramatic results
- Use language like "subtle shifts," "early signs," "your body adjusting"
- This prevents the Day 3 "this isn't working" abandonment

**Step 5:** Frame later milestones to build anticipation:
- Progressive language: "building" -> "noticeable" -> "clear" -> "sustained"
- Each phase feels like the next logical step, not a separate promise

**Step 6:** Ensure no cross-section references. Timeline must be self-contained — visitor should understand the progression without reading ingredients or mechanism sections.

**Step 7:** Verify word count against target (+/-10%).

**Step 8:** Write `section-what-to-expect.md`.

## Quality Criteria
- 3-5 milestones with clear time labels
- Each milestone has: time label, benefit title, 2-3 sentence description
- Progression arc: subtle -> noticeable -> clear -> sustained
- Phase 1 sets realistic expectations (no "dramatic results in 24 hours")
- Each milestone connects mechanism to felt benefit
- Self-contained (no "as explained in the ingredients section")
- Word count within +/-10% of PDP-04 target
- Zero generic AI language

## Failure Modes
- Fewer than 3 milestones -> HALT. Minimum 3 phases required for a meaningful timeline.
- Phase 1 promises dramatic results -> Rewrite with tempered language. Overselling early results causes abandonment.
- All milestones use identical language structure -> Vary sentence patterns. Repetitive structure kills scannability.
- Timeline data not in brief -> Use product type defaults (supplement: 7d/2wk/1mo/3mo), flag as data gap.
- Milestone description exceeds 3 sentences -> Condense. Timeline is visual-first, not paragraph-first.
- Timeline references ingredients by name without brief explanation -> Add 2-3 word mechanism context inline.
