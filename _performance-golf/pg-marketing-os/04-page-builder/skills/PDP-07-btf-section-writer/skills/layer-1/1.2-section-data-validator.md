# 1.2 — Section-Specific Data Validator

**Skill:** PDP-07-btf-section-writer
**Position:** Layer 1, Microskill 2
**Purpose:** Verify that each section in PDP-07's execution queue has sufficient data to be written. Each section type has specific data requirements — an ingredients section needs an ingredient list, a proof section needs study data, a comparison chart needs differentiators. This validator catches data gaps BEFORE Layer 2 writing begins, preventing partial or vague section output.
**Effort Level:** medium
**Dependencies:** `per-section-briefs.md` (1.1)
**Output:** `data-validation.md`

---

## Input Schema

```yaml
input:
  per_section_briefs: "[path to per-section-briefs.md]"
```

---

## Execution Logic

**Step 1:** Load `per-section-briefs.md`. Parse each section's brief.

**Step 2:** For each section, apply the section-type-specific data requirements:

```
SECTION DATA REQUIREMENTS:

product_highlights:
  REQUIRED: product name, 3+ KSPs or common questions
  NICE_TO_HAVE: guarantee details, shipping info
  CRITICAL_GAP_IF_MISSING: product name or fewer than 3 KSPs

the_expert:
  REQUIRED: expert name, at least 1 credential (degree, title, institution)
  NICE_TO_HAVE: expert photo/video, quote, publication list
  CRITICAL_GAP_IF_MISSING: expert name or any credential

whats_in_it:
  REQUIRED: ingredient list (3+ ingredients), per-ingredient dosage
  NICE_TO_HAVE: per-ingredient study references, mechanism descriptions
  CRITICAL_GAP_IF_MISSING: ingredient list or dosages

what_to_expect:
  REQUIRED: at least 3 timeline milestones, per-milestone benefit
  NICE_TO_HAVE: clinical timeline data, mechanism per phase
  CRITICAL_GAP_IF_MISSING: fewer than 3 milestones

the_problem:
  REQUIRED: root cause or problem description, audience pain points
  NICE_TO_HAVE: pain quotes, statistics
  CRITICAL_GAP_IF_MISSING: problem description

the_solution:
  REQUIRED: paradigm shift / new approach, product mechanism
  NICE_TO_HAVE: differentiator vs conventional approach
  CRITICAL_GAP_IF_MISSING: product mechanism

why_it_works:
  REQUIRED: mechanism of action (at least 2-step logical chain)
  NICE_TO_HAVE: biochemical pathway, study references
  CRITICAL_GAP_IF_MISSING: mechanism description

comparison_chart:
  REQUIRED: 5+ product differentiators, competitor category context
  NICE_TO_HAVE: specific competitive weaknesses to contrast
  CRITICAL_GAP_IF_MISSING: fewer than 5 differentiators

cost_savings:
  REQUIRED: product price, bundle pricing, individual component prices
  NICE_TO_HAVE: price-per-serving math, competitor pricing
  CRITICAL_GAP_IF_MISSING: product price or bundle pricing

how_its_made:
  REQUIRED: at least 2 quality claims (sourcing, manufacturing, certification)
  NICE_TO_HAVE: specific certifications (GMP, NSF), testing protocols
  CRITICAL_GAP_IF_MISSING: fewer than 2 quality claims

how_to_use:
  REQUIRED: dosage, timing, method
  NICE_TO_HAVE: special instructions, contraindications
  CRITICAL_GAP_IF_MISSING: dosage or method

identity_matching:
  REQUIRED: 3 distinct customer personas with key benefit per persona
  NICE_TO_HAVE: demographics, psychographics per persona
  CRITICAL_GAP_IF_MISSING: fewer than 3 personas

the_proof:
  REQUIRED: at least 1 specific study/survey with data point
  NICE_TO_HAVE: multiple studies, sample sizes, journal names
  CRITICAL_GAP_IF_MISSING: no study or survey data at all
```

**Step 3:** Classify each section:
- **GREEN:** All REQUIRED data present. Ready for Layer 2.
- **YELLOW:** REQUIRED data present but minimal (e.g., 3 differentiators instead of 5). Proceed with flag.
- **RED:** CRITICAL data gap. Cannot write this section without human input.

**Step 4:** For RED sections, determine if PDP-04 should have excluded them (e.g., The Proof with no study data). Flag as PDP-04 mismatch if so.

**Step 5:** Write `data-validation.md`.

---

## Quality Criteria
- Every section in the execution queue validated
- Validation uses section-type-specific criteria (not generic)
- Critical data gaps are specific ("missing ingredient dosages" not "incomplete data")
- RED sections clearly flagged with what data is needed
- PDP-04 mismatches identified (section included in sequence but lacks data to write)

## Failure Modes
- per-section-briefs.md not found -> HALT. Layer 1.1 must run first.
- All sections are RED -> HALT. Brief is insufficient for PDP BTF writing. Escalate to human.
- Majority of sections are YELLOW -> WARN. Output will be data-light. Flag for human review.
- Section in queue not in validation rules -> WARN. Validate against generic "product name + purpose" minimum.
