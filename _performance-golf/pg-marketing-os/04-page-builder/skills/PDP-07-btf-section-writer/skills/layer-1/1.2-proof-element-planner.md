# 1.2-proof-element-planner.md

> **Layer:** 1 (Section Architecture)
> **Skill:** PDP-07-section-copy
> **Purpose:** Plan proof element per section (minimum 1)
> **Outputs:** `proof_plan` object with proof assignment per section

---

## FUNCTION

Plan at least one proof element for every active BTF section. Proof converts more than persuasion on ecom pages. This planner assigns the right TYPE of proof to each section based on section purpose and available proof assets.

---

## PROOF PLANNING PROTOCOL

### Step 1: Inventory Available Proof

```
FROM upstream_context, INVENTORY:
  customer_reviews: [count, quality, specificity]
  ugc_references: [count, type -- photos, videos, social posts]
  statistics: [numbers, studies, research citations]
  expert_endorsements: [names, credentials, quotes]
  awards_press: [award names, press mentions, logos]
  before_after: [available transformations, data points]
  customer_count: [total customers, satisfaction rate]
  certifications: [testing, quality, regulatory]
```

### Step 2: Match Proof Type to Section

```
FOR each active BTF section:
  ASSIGN proof type based on section purpose:

  SECTION -> BEST PROOF TYPE:
    Problem/Solution (9) -> Customer story (before/after, pain quote)
    Why It Works (10) -> Study/patent/scientific backing
    Comparison (11) -> Objective data, testing results
    Cost Savings (12) -> Price comparison data, calculator
    How to Use (13) -> Demo reference, customer usage photo
    Reviews (14) -> Pull-quote reviews (this section IS proof)
    FAQ (15) -> Expert answer citation
    Offer/Pricing (16) -> Value anchor, customer count
    Guarantee (17) -> Policy specifics, return rate stat
    Use Cases (18) -> Segment-specific testimonial
    Awards/Press (19) -> Logos, citations (this section IS proof)
    Product Highlights (4) -> Star rating, award badge
    UGC (5) -> UGC content (this section IS proof)
    Expert (6) -> Credentials, endorsement (this section IS proof)
    Ingredients (7) -> Source quality proof, testing cert
    Timeline (8) -> Before/after timeline data
    CTA Box (3) -> Trust badge, customer count
```

### Step 3: Verify Coverage

```
CHECK:
  [ ] Every active section has at least 1 proof element assigned
  [ ] No proof type used more than 3 times (variety)
  [ ] Proof matches section purpose (not forced)

  IF proof inventory insufficient:
    FLAG sections where proof will be placeholder
    NOTE: "Client needs to provide [proof type] for Section [X]"
```

```json
{

      "section_name": "Problem/Solution",

      "proof_source": "available|placeholder_needed",
      "placement": "After pain bullets, before product answer"
    }
  ],
  "proof_summary": {
    "sections_with_proof": 13,
    "placeholders_needed": 2,
    "proof_types_used": ["customer_story", "statistic", "expert_endorsement", "review_quote"]
  }
}
```

---

## FAILURE MODES

| Mode | Trigger | Remediation |
|------|---------|-------------|
| CRITICAL | Section has zero proof options | Assign "customer_count" as universal fallback |
| WARNING | More than 3 sections need placeholder proof | Flag for client data request |
| INFO | All sections have real proof | Best case |
