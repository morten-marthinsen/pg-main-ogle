# Cross-Product Feature Naming Patterns

**Sources:** 357 Fairway Hybrid, ONE.1 Wedge, RS1 Putter
**Purpose:** Pattern extraction across all PG products for PDP-02 calibration
**Extracted:** 2026-03-13

---

## The PG Feature Naming System

Across 3 products and 17+ named features, PG uses a consistent (but not rigid) naming architecture. These patterns are the specimens that PDP-02's Arena should internalize.

---

## Pattern 1: The Umbrella Technology

Every product has ONE umbrella name that encompasses all features.

| Product | Umbrella | Format | What It Names |
|---------|----------|--------|---------------|
| 357 | Tri-Fusion Technology | Engineering Approach | How features are combined |
| ONE.1 | Auto-Correct (product name) | Benefit Descriptor | What the product does |
| RS1 | Roll Straight Technology | Desired Outcome | What the prospect gets |

**Evolution:** 357 → engineering-centric. ONE.1 → benefit-centric. RS1 → outcome-centric. The trend moves toward naming the RESULT, not the process.

**Rule:** Every product needs an umbrella name. It appears in headlines, CTAs, and repeat mentions. It should be 2-3 words and immediately communicable.

---

## Pattern 2: Hero Feature Gets the Most Accessible Name

| Product | Hero Feature | Accessibility |
|---------|-------------|---------------|
| 357 | Progressed Face "The Bulldozer" | Nickname carries it — pure caveman |
| ONE.1 | Auto-Glide Sole | Action-benefit compound — self-explanatory |
| RS1 | Forward Axis Balance | Physics language — requires context |

**Rule:** The hero feature name must be comprehensible in 3 seconds. If the technical name isn't accessible, add a nickname (357 pattern). If the name IS the benefit, no nickname needed (ONE.1 pattern). Physics language works only when the audience is educated enough (RS1 pattern — putter enthusiasts).

---

## Pattern 3: Compound Naming (Two Concepts Fused)

Every PG feature name is a compound — minimum two concepts joined:

| Pattern | Examples |
|---------|---------|
| Action + Component | Auto-Glide Sole, Spin-Fast Face |
| Benefit + Component | Power Launch Crown, Distance Control Weighting |
| Spatial + Concept | Forward Axis Balance, Face-Forward Shaft Position |
| Descriptive + Component | Sculpted Sole Relief, Progressed Face |
| Action + Action | Square-Shoot Alignment |

**Rule:** No single-word feature names. Every name must communicate both WHAT it is and WHAT it does. The compound format forces information density.

---

## Pattern 4: Named Sub-Mechanisms

Features often contain a named sub-phenomenon:

| Feature | Sub-Mechanism | Role |
|---------|--------------|------|
| C-Cup Construction | "Elastic Effect" | Names the physics phenomenon |
| Auto-Glide Sole | "Four-Way Forgiveness" | Names the benefit of the design |
| Forward Axis Balance | "The Table Test" | Names the proof demonstration |
| Progressed Face | "The Bulldozer" | Names the mental image |

**Rule:** Sub-mechanisms add a second layer of naming to the feature. They're often more memorable than the parent feature name. Use them strategically — one sub-mechanism per hero feature, optionally for key supporting features.

## Pattern 5: Problem Naming

| Product | Problem Named | Industry Term Replaced |
|---------|--------------|----------------------|
| RS1 | "Face Drift" | "Toe flow" |
| ONE.1 | "The Consistency Crisis" | Generic "short game problems" |
| 357 | "Senior Swing Sacrifice" | Generic "losing distance" |

**Rule:** Name the problem before naming the solution. A named problem creates a diagnosis that only your product treats. This is Hopkins preemptive claiming applied to problem framing.

---

## Pattern 6: Named Guarantees and Challenges

| Product | Named Guarantee | Named Challenge |
|---------|----------------|----------------|
| 357 | 365-Day Money Back | "Let It Fly From Any Lie" Challenge |
| ONE.1 | "No Dig, No Doubt" Demo Period | — |
| RS1 | "Love Your Game" Guarantee | "The Table Test" |

**Rule:** Guarantees and challenges get names too. A named guarantee ("No Dig, No Doubt") converts better than generic ("Money Back Guarantee") because it echoes the product's benefit language.

---

## Pattern 7: The "This Means..." Bridge

All 3 products use a consistent micro-script structure:

```
[Feature Name] does [mechanism].
This means you'll [felt benefit].
Even when [specific scenario].
```

**Example (357):**
"The C-Cup Construction we use to weld the clubface to the head gives the face more room to flex. This is called the 'Elastic Effect' because it increases compression and ball speed... without adding swing speed. What's that mean for you? You'll gain distance..."

**Example (ONE.1):**
"The 'Auto-Glide Sole' gives you Four-Way Forgiveness because instead of digging, its four-way cambered shape glides through the ground automatically. This means you make consistently clean contact..."

**Rule:** Every feature micro-script must bridge from mechanism to felt benefit using explicit connectors ("This means...", "What's that mean for you?", "So you..."). Never assume the prospect will make the connection themselves.

---

## Pattern 8: Micro-Script Volume Scaling

| Product | Named Features | Total Micro-Scripts | Ratio |
|---------|---------------|-------------------|-------|
| 357 | 5 | ~10 | 2:1 |
| ONE.1 | 5 | 30+ | 6:1 |
| RS1 | 7 | 55+ | 8:1 |

**Rule:** More sophisticated products (higher price, more features, more competitive market) need higher micro-script volume. The RS1 (most expensive, most competitive category) has 8x more micro-scripts than features. Arena should generate proportional volume.

## PDP-02 Calibration Summary

When calibrating PDP-02's Feature Naming Arena:

1. **Every product needs an umbrella technology name** (2-3 words, immediately communicable)
2. **Hero feature gets the most accessible name** (nickname if needed)
3. **All feature names are compounds** (minimum 2 concepts)
4. **Name the problem** before naming the solution
5. **Include sub-mechanisms** for hero and key supporting features
6. **Name the guarantee** to echo product language
7. **Use the "This means..." bridge** in every micro-script
8. **Generate 5-8x more micro-scripts** than you think you need
9. **Feature hierarchy must be clear** — hero (30% of copy weight), supporting (50%), technical (20%)
10. **Consistency is non-negotiable** — one name per feature, locked before copy begins
