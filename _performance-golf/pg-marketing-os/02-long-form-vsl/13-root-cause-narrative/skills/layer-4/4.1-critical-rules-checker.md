# Skill 13.4.1: Critical Rules Checker

**Version:** 1.0
**Skill:** 10 — Root Cause Narrative
**Layer:** 4 (Validation & Assembly)
**Position:** First validation microskill — checks all 10 critical rules derived from TIER1 analysis
**Type:** Validation
**Dependencies:** All Layer 2 and Layer 3 outputs, threading_guide.json (from 3.4), tier1_intelligence_index.json (from 0.2), checkpoint_response.json (from 0.4)
**Output:** critical_rules_report.json

---

## PURPOSE

Validate the completed root cause narrative against all 10 critical rules derived from TIER1 analysis. These rules are non-negotiable — they represent the universal patterns found across every successful elite control. ALL 10 must pass. Zero tolerance. A single rule violation means the narrative fails to meet the standard established by TIER1 controls, and the specific violation must be remediated before the narrative can be assembled into the final package.

---

## IDENTITY

**This microskill IS:**
- A rule enforcer that checks each of 10 critical rules with specific evidence for pass/fail
- A zero-tolerance gate — every rule must pass; no exceptions, no partial credit
- A remediation advisor that provides specific fix instructions for each violation
- A TIER1 standard bearer — these rules ARE the standard extracted from elite controls

**This microskill is NOT:**
- A quality scorer (scoring happens in 4.4) — this is pass/fail, not scored
- A rewriter — it identifies violations; remediation happens through re-execution of appropriate microskills
- A subjective judge — each rule has objective, verifiable check criteria
- A flexible framework — all 10 rules are mandatory; none can be waived

## PROCESS

### Step 1: Check Rule 1 — EXTERNAL

```
RULE 1: The root cause is NEVER the reader's fault.
  Blame must be externalized to villain, biology, misinformation, or system.

CHECK PROCESS:
  SCAN full_narrative text for any language that internalizes blame:

  INTERNALIZED BLAME MARKERS (fail triggers):
    - "if you had..." / "if only you..."
    - "you should have..."
    - "you failed to..." / "your failure..."
    - "you weren't disciplined enough"
    - "your lack of..." (willpower, effort, commitment)
    - "you didn't try hard enough"
    - "your poor choices"
    - Any language suggesting the reader caused their own problem

  EXTERNALIZED BLAME MARKERS (pass indicators):
    - "It's not your fault"
    - "You were set up to fail"
    - "The [villain] caused this"
    - "[Root cause] is an external agent"
    - "You were misled/deceived/kept ignorant"
    - "The system failed you"

  RESULT:
    IF any internalized blame found:
      STATUS: FAIL
      EVIDENCE: [exact text that internalizes blame]
      REMEDIATION: "Remove internalized blame. Replace with externalized cause.
                    Change '[internal blame]' to '[external attribution].'"
    ELSE:
      STATUS: PASS
      EVIDENCE: "[Count] externalized blame markers found. Zero internalized."
```

### Step 2: Check Rule 2 — EXPLAINS ALL FAILURES

```
RULE 2: Every failed attempt the reader has tried must be accounted for.

CHECK PROCESS:
  FROM phases_5_6_draft.failure_explanation:
    EXTRACT failures_explained list

  FROM niche knowledge:
    COMPILE expected_failures — common solutions readers in this niche have tried

  COMPARE:
    FOR each expected failure:
      IS it addressed in the narrative? YES/NO

  COVERAGE = addressed_count / expected_count * 100

  RESULT:
    IF COVERAGE < 80% (missing more than 1 in 5 common approaches):
      STATUS: FAIL
      EVIDENCE: "Missing failures: [list of unaddressed approaches]"
      REMEDIATION: "Add explanation for: [missing approaches].
                    Each must be traced to symptom-treatment, not root cause."
    ELSE:
      STATUS: PASS
      EVIDENCE: "[addressed_count] of [expected_count] common approaches explained.
                Coverage: [COVERAGE]%."
```

### Step 3: Check Rule 3 — MORE SPECIFIC

```
RULE 3: The root cause must be MORE SPECIFIC than the false belief.
  It zooms IN, not out.

CHECK PROCESS:
  COMPARE:
    false_belief (what_they_think): [text]
    root_cause (what_it_really_is): [text]

  SPECIFICITY ASSESSMENT:
    Is the root cause a SUBSET, DETAIL, or PRECISE MECHANISM of the false belief?

    PASS examples:
      "Bad diet" → "Lectins — plant proteins in wheat, tomatoes, beans" (MUCH more specific)
      "Poor swing" → "Clubface-to-path relationship at impact" (precise variable)
      "Bad investments" → "Federal Reserve monetary policy since 1971" (specific actor and date)

    FAIL examples:
      "Bad diet" → "The wrong foods" (equally vague)
      "Poor swing" → "Bad technique" (no specificity gain)
      "Bad investments" → "Financial mistakes" (less specific)

  RESULT:
    IF root cause is NOT more specific than false belief:
      STATUS: FAIL
      EVIDENCE: "False belief: '[what_they_think]'. Root cause: '[what_it_really_is]'.
                Root cause is not demonstrably more specific."
      REMEDIATION: "Increase specificity: name the exact enzyme, process, actor,
                    mechanism, or variable. 'Bad diet' must become 'Lectins' or
                    'Citrate Lyase enzyme' — not 'wrong foods.'"
    ELSE:
      STATUS: PASS
      EVIDENCE: "False belief: '[text]' → Root cause: '[text]'. Clear specificity increase."
```

### Step 4: Check Rule 4 — PATH TO SOLUTION

```
RULE 4: The root cause must be prescriptive, not just diagnostic.
  It must point toward a solution (the mechanism).

CHECK PROCESS:
  FROM mechanism_bridge (Phase 7):
    VERIFY: Does the narrative establish that a solution EXISTS?
    VERIFY: Does the root cause IMPLY how to fix it?

  PRESCRIPTIVE TEST:
    IF you know the root cause, can you logically deduce what the solution would address?
    "Lectins damage gut" → solution addresses lectins (prescriptive)
    "Life is hard" → no implied solution (diagnostic only)
    "Clubface-to-path wrong" → solution fixes clubface-to-path (prescriptive)

  MECHANISM BRIDGE TEST:
    Does Phase 7 connect root cause to mechanism entry point?
    Is there a clear "here's how to fix it" pivot?

  RESULT:
    IF root cause is purely diagnostic with no solution path:
      STATUS: FAIL
      EVIDENCE: "Root cause '[text]' does not imply a solvable mechanism"
      REMEDIATION: "Sharpen root cause to be actionable. The cause must
                    contain the seed of its own solution."
    IF Phase 7 doesn't connect to mechanism:
      STATUS: FAIL
      EVIDENCE: "Mechanism bridge missing or doesn't connect root cause to solution"
      REMEDIATION: "Rewrite Phase 7 to explicitly bridge from root cause to mechanism."
    ELSE:
      STATUS: PASS
      EVIDENCE: "Root cause implies solution path. Mechanism bridge present."
```

### Step 5: Check Rule 5 — VILLAIN PAIRED

```
RULE 5: Someone or something to blame must be identified with specificity.

CHECK PROCESS:
  FROM villain_deep_dive:
    VERIFY: villain_named = true
    VERIFY: motive_established = true
    VERIFY: evidence_presented is non-empty

  SPECIFICITY CHECK:
    Is the villain NAMED? (not just "the system" or "them")
    Is the villain TYPE clear?
    Is the villain ROLE defined?
    Is there a MOTIVE?
    Is there EVIDENCE of the villain's actions?

  RESULT:
    IF villain not named OR too generic:
      STATUS: FAIL
      EVIDENCE: "Villain '[name]' is too generic or unnamed"
      REMEDIATION: "Name the villain specifically — entity, organization, or
                    named process. Add motive and evidence of actions."
    IF motive not established:
      STATUS: FAIL
      EVIDENCE: "Villain named but no motive established"
      REMEDIATION: "Add villain motive — profit, power, ignorance, institutional bias."
    ELSE:
      STATUS: PASS
      EVIDENCE: "Villain '[name]' (type: [type], role: [role]) with motive and evidence."
```

### Step 6: Check Rule 6 — COUNTER-INTUITIVE

```
RULE 6: The root cause must surprise the reader.
  If the reader already believes it, it has no persuasive power.

CHECK PROCESS:
  ASSESS: Would the target reader be SURPRISED by this root cause?

  COUNTER-INTUITIVE MARKERS:
    - Root cause contradicts conventional wisdom
    - Root cause challenges what "experts" say
    - Root cause inverts a common assumption
    - Root cause names something the reader has never heard of
    - Root cause is the OPPOSITE of what the reader expects

  NOT COUNTER-INTUITIVE MARKERS:
    - Root cause is obvious ("you need to eat better")
    - Root cause is already widely known
    - Root cause is what the reader already suspects
    - Root cause is generic ("stress causes health problems")

  RESULT:
    IF root cause is not counter-intuitive:
      STATUS: FAIL
      EVIDENCE: "Root cause '[text]' is not surprising to the target reader.
                [Reasoning for why it's expected/obvious]."
      REMEDIATION: "Sharpen the counter-intuitive element. Options:
                    1. Make it more specific (general → precise mechanism)
                    2. Invert the expectation ('healthy' food IS the cause)
                    3. Name an unexpected agent (not the usual suspect)"
    ELSE:
      STATUS: PASS
      EVIDENCE: "Root cause surprises because [counter-intuitive element]."
```

### Step 7: Check Rule 7 — WOVEN THROUGHOUT

```
RULE 7: Root cause language must appear in every section via threading guide.

CHECK PROCESS:
  FROM threading_guide:
    VERIFY: anchor_phrase defined and non-empty
    VERIFY: repetition_points cover all required downstream locations
    VERIFY: emphasis_level set

  FROM full_narrative_text:
    COUNT how many times anchor_phrase or root cause language appears
    VERIFY: Root cause referenced in at least 4 of 7 phases

  THREADING COMPLETENESS:
    IF emphasis_level == "foundation": ALL downstream locations must have instructions
    IF emphasis_level == "heavy": Lead, mechanism, close must have instructions
    IF emphasis_level == "moderate": Mechanism, close must have instructions

  RESULT:
    IF threading guide incomplete or anchor phrase missing:
      STATUS: FAIL
      EVIDENCE: "Threading guide missing [specific element]"
      REMEDIATION: "Complete threading guide. [Specific missing element] must be defined."
    IF root cause language appears in fewer than 4 of 7 phases:
      STATUS: FAIL
      EVIDENCE: "Root cause referenced in only [X] of 7 phases"
      REMEDIATION: "Add root cause references to: [missing phases]"
    ELSE:
      STATUS: PASS
      EVIDENCE: "Anchor phrase: '[phrase]'. Referenced in [X] of 7 phases.
                Threading guide complete for [emphasis_level] level."
```

### Step 8: Check Rule 8 — ANCHOR PHRASE

```
RULE 8: A memorable, repeatable anchor phrase must exist.

CHECK PROCESS:
  FROM threading_guide.anchor_phrase:
    VERIFY: non-empty
    VERIFY: <= 7 words
    VERIFY: specific (names root cause or evokes it)
    VERIFY: counter-intuitive or provocative

  MEMORABILITY TEST:
    □ Could a reader repeat this phrase from memory? YES/NO
    □ Does it work as a headline? YES/NO
    □ Is it distinct from competitor language? YES/NO

  RESULT:
    IF anchor phrase missing or generic:
      STATUS: FAIL
      EVIDENCE: "Anchor phrase '[phrase]' is [missing/generic/too long/not memorable]"
      REMEDIATION: "Create anchor phrase that is specific, under 7 words,
                    counter-intuitive, and memorable."
    ELSE:
      STATUS: PASS
      EVIDENCE: "Anchor phrase: '[phrase]' — [word count] words, specific, memorable."
```

### Step 9: Check Rule 9 — AUTHORITY FIRST

```
RULE 9: Credibility must be established BEFORE the counter-intuitive claim.

CHECK PROCESS:
  FROM full_narrative_text:
    LOCATE: Where does the authority/credential stack appear?
    LOCATE: Where does the root cause reveal appear?
    VERIFY: Authority PRECEDES reveal in the text

  ORDERING CHECK:
    Phase 3 (Authority Bridge) must come BEFORE Phase 4 (Root Cause Reveal)
    No counter-intuitive claim should appear before credentials are established

  RESULT:
    IF reveal appears before authority:
      STATUS: FAIL
      EVIDENCE: "Root cause claim at [position] precedes authority establishment at [position]"
      REMEDIATION: "Restructure: move credential stack before the root cause reveal.
                    The reader must think 'this person is credible' BEFORE hearing
                    the counter-intuitive claim."
    ELSE:
      STATUS: PASS
      EVIDENCE: "Authority established in Phase 3. Reveal delivered in Phase 4.
                Correct ordering confirmed."
```

### Step 10: Check Rule 10 — KILLS COMPETITORS

```
RULE 10: Countersells must eliminate competing solutions through root cause logic.

CHECK PROCESS:
  FROM countersells:
    VERIFY: At least 2 countersells present
    FOR each countersell:
      VERIFY: Specific competitor named (not generic)
      VERIFY: Root cause explicitly referenced as the weapon
      VERIFY: Clear verdict (treats_symptoms, makes_worse, wrong_enemy, or dangerous)

  ROOT CAUSE WEAPON CHECK:
    Each countersell argument must contain logic: "Because the REAL problem is
    [root cause], [competitor] fails because it only addresses [symptom/false belief]."

  RESULT:
    IF countersells < 2:
      STATUS: FAIL
      EVIDENCE: "Only [count] countersells present. Minimum is 2."
      REMEDIATION: "Add [2 - count] more countersells targeting specific competitors."
    IF any countersell doesn't reference root cause:
      STATUS: FAIL
      EVIDENCE: "Countersell for '[target]' doesn't use root cause as weapon"
      REMEDIATION: "Rewrite countersell to explicitly connect competitor failure
                    to the root cause."
    ELSE:
      STATUS: PASS
      EVIDENCE: "[count] countersells. Each uses root cause as weapon.
                Competitors killed: [list of targets]."
```


## QUALITY CRITERIA

1. **ALL 10 rules checked** — no rule can be skipped or assumed to pass
2. **Zero tolerance** — ALL 10 must pass; a single failure means the narrative does not meet TIER1 standards
3. **Evidence documented for every rule** — both pass and fail must include specific textual evidence
4. **Remediation is specific** — every failure must include actionable instructions for fixing the violation
5. **Internalized blame detection is thorough** (Rule 1) — scan must catch subtle blame as well as overt
6. **Specificity comparison is honest** (Rule 3) — the root cause must be genuinely more specific, not just different
7. **Counter-intuitive assessment is fair** (Rule 6) — assessed from the target reader's perspective, not the writer's

---

## FAILURE MODES

| Failure | Severity | Detection | Handling |
|---------|----------|-----------|----------|
| Rule 1 violation (blame internalized) | CRITICAL | Internalized blame language found | Return to 2.1-2.3 writers for removal |
| Rule 3 violation (not specific enough) | HIGH | Root cause equally vague as false belief | Return to Skill 02 for specificity enhancement; or human checkpoint |
| Rule 9 violation (reveal before authority) | HIGH | Structural ordering wrong | Return to 2.2 for restructuring |
| Rule 10 violation (weak countersells) | MODERATE | Countersells don't use root cause | Return to 2.4 for rewriting |
| Multiple rules fail simultaneously | CRITICAL | 3+ rules violated | Escalate to human checkpoint with failure log |
| Rule appears to pass but fails on close reading | LOW | False positive in automated check | Human review recommended for borderline cases |
