# Microskill 1.4: Proof Element Selector

**Layer:** 1 (Architecture)
**Purpose:** Select which specific proof elements from inventory for which specific placements

---

## CORE PRINCIPLE

Not all proof elements are equal. This microskill matches the RIGHT proof elements to the RIGHT placements based on:
1. Element strength (score)
2. Element type (matching section needs)
3. Element variety (avoiding redundancy)
4. Element freshness (not overusing any element)

---

## SELECTION CRITERIA

### Primary Selection Factors

| Factor | Weight | Description |
|--------|--------|-------------|
| Score | 30% | stage_adjusted_score from inventory |
| Type Match | 30% | Does type fit section need? |
| Specificity | 20% | Does element have specific numbers/outcomes? |
| Freshness | 10% | Has element been used elsewhere? |
| Variety | 10% | Does element add demographic/situational variety? |

### Score-Based Tiers

```
TIER 1 (Score 8.5+): Use for high-impact placements
  → Lead credibility hooks
  → Proof section openers
  → Close callbacks

TIER 2 (Score 7.0-8.4): Use for standard placements
  → Story section proof
  → Mechanism validation
  → Product introduction proof

TIER 3 (Score 5.5-6.9): Use for supporting placements
  → Additional testimonial cascade entries
  → Secondary authority mentions
  → Background data points

TIER 4 (Score <5.5): Reserve or exclude
  → Only use if inventory is thin
  → Consider reformulating for strength
```

---

## SECTION → ELEMENT MATCHING

### Lead Section

**Needs:** Authority anchor, credibility hook, scale hint

**Selection Priority:**
1. Highest-scoring authority proof
2. Most impressive scale proof (if available)
3. Most recognizable third-party validation

**Selection Query:**
```
SELECT elements WHERE:
  type IN (authority, social_proof, demonstration)
  AND score >= 7.5
  AND specificity_level = HIGH
ORDER BY score DESC
LIMIT 2
```

---

### Story Section

**Needs:** Transformation proof, suffering validation, expert introduction

**Selection Priority:**
1. Most dramatic transformation testimonial
2. Most relatable before state
3. Most credible expert mention

**Selection Query:**
```
SELECT elements WHERE:
  type IN (testimonial, demonstration, authority)
  AND has_transformation = true
  AND emotional_resonance = HIGH
ORDER BY transformation_contrast DESC
LIMIT 3
```

---

### Root Cause Section

**Needs:** Logic proof, failure explanation, villain evidence

**Selection Priority:**
1. Most damning data against villain
2. Clearest logic proof for root cause
3. Most surprising "why nothing worked" evidence

**Selection Query:**
```
SELECT elements WHERE:
  type IN (logic, data, study)
  AND supports_root_cause = true
ORDER BY counter_intuitive_score DESC
LIMIT 2
```

---

### Mechanism Section

**Needs:** Study validation, demonstration, scientific backing

**Selection Priority:**
1. Most rigorous study citation
2. Most impressive demonstration (numbers)
3. Most credible institutional backing

**Selection Query:**
```
SELECT elements WHERE:
  type IN (study, demonstration, logic)
  AND has_quantified_outcome = true
ORDER BY specificity_score DESC
LIMIT 4
```

---

### Proof Section (Dedicated)

**Needs:** Testimonial cascade, scale proof, variety demonstration

**Selection Priority:**
1. Build cascade from strongest to scale
2. Ensure demographic variety
3. Include transformation + satisfaction + scale

**Selection Query:**
```
SELECT elements WHERE:
  type IN (testimonial, social_proof, demonstration)
ORDER BY:
  CASE
    WHEN is_transformation_story THEN 1
    WHEN is_scale_proof THEN 2
    WHEN is_satisfaction_testimonial THEN 3
  END,
  score DESC
LIMIT 8
```

---

### Product Introduction Section

**Needs:** Formula validation, development proof, single strong testimonial

**Selection Priority:**
1. Most impressive formula/component proof
2. Strongest development/expertise proof
3. Single most compelling testimonial (if not used earlier)

**Selection Query:**
```
SELECT elements WHERE:
  type IN (demonstration, authority, testimonial)
  AND NOT used_in_earlier_section
ORDER BY product_relevance DESC
LIMIT 3
```

---

### Offer Section

**Needs:** Value testimonial, risk removal proof

**Selection Priority:**
1. "Worth every penny" type testimonial
2. Satisfaction/guarantee proof
3. Value justification data

**Selection Query:**
```
SELECT elements WHERE:
  type IN (testimonial, social_proof)
  AND sentiment = 'value_affirmation'
  AND NOT used_in_earlier_section
LIMIT 2
```

---

### Close Section

**Needs:** Callbacks to earlier proof, transformation reminder

**Selection Priority:**
1. Reference strongest lead/story proof
2. Scale callback
3. Future pacing based on real results

**Selection Query:**
```
SELECT elements WHERE:
  used_in_lead = true OR used_in_story = true
  -- For callback language creation
```

## VARIETY ENFORCEMENT

### Demographic Variety Check

```
FOR testimonials selected:
  CHECK: At least 2 different demographics represented
  CHECK: At least 2 different situations/conditions
  CHECK: At least 2 different outcome types

IF variety < threshold:
  → Swap lower-scoring elements for variety-adding elements
```

### Type Variety Check

```
FOR all elements selected:
  CHECK: At least 4 different proof types used
  CHECK: No single type > 40% of total

IF imbalanced:
  → Swap redundant elements for missing types
```
