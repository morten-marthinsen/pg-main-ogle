# 4.2-selection-processor.md

> **Layer:** 4 (Selection & Packaging)
> **Skill:** 10-headlines
> **Purpose:** Process human selection and any modifications
> **Inputs:** Human selection from 4.1 presentation
> **Outputs:** Confirmed selected headline with any modifications applied

---

## FUNCTION

Receive the human's headline selection decision, process any modifications they requested, and confirm the final selected headline. This skill handles all selection scenarios: direct selection, modified selection, and human-provided alternatives.

**Key Principle:** Human decision is FINAL. This skill processes their choice, not overrides it. If they pick option #5 when we thought #1 was best, we execute their choice.

---

## SELECTION SCENARIOS

### Scenario A: Direct Selection
Human selects one of the 5 options as-is.

```
INPUT: "Option 2"
PROCESS: Confirm selection, no modifications
OUTPUT: Candidate 2 becomes selected_headline
```

### Scenario B: Selection with Modifications
Human selects an option but requests changes.

```
INPUT: "Option 1, but change 'SHOCKING' to 'surprising'"
PROCESS: Apply modification, re-score, confirm
OUTPUT: Modified version becomes selected_headline
```

### Scenario C: Human-Provided Alternative
Human provides their own headline.

```
INPUT: "Use this instead: 'Why Diets Fail (And What Actually Works)'"
PROCESS: Score human headline on same rubric, confirm
OUTPUT: Human headline becomes selected_headline
```

### Scenario D: Request More Options
Human rejects all 5 and wants different direction.

```
INPUT: "None of these. Try more benefit-focused approach"
PROCESS: Return to Layer 2 with new direction
OUTPUT: New generation cycle
```

## EXECUTION PROTOCOL

### Step 1: Parse Human Input

```
RECEIVE human selection response

CLASSIFY response type:
  IF contains "Option [1-5]" alone:
    TYPE = DIRECT_SELECTION

  IF contains "Option [1-5]" AND modification request:
    TYPE = MODIFIED_SELECTION

  IF contains original headline text:
    TYPE = HUMAN_PROVIDED

  IF contains "none" OR "different" OR "try again":
    TYPE = REGENERATION_REQUEST
```

### Step 2: Process by Type

```
IF TYPE = DIRECT_SELECTION:
  EXTRACT option number
  RETRIEVE corresponding candidate
  CONFIRM with user: "Confirming selection of Option [X]: '[headline]'"
  PROCEED to Step 4

IF TYPE = MODIFIED_SELECTION:
  EXTRACT option number
  EXTRACT modification instructions
  APPLY modifications to headline/subheadline
  RE-SCORE modified version on 6-dimension rubric
  PRESENT modified version with new score
  CONFIRM: "Modified headline scores [X]. Confirming selection?"
  IF confirmed, PROCEED to Step 4

IF TYPE = HUMAN_PROVIDED:
  EXTRACT human's headline text
  SCORE on same 6-dimension rubric
  PRESENT score: "Your headline scores [X]/10"
  IF score < 7.0, WARN: "This scores below threshold. Proceed anyway?"
  IF confirmed, PROCEED to Step 4

IF TYPE = REGENERATION_REQUEST:
  EXTRACT new direction guidance
  RETURN to Layer 2 with direction override
  DO NOT proceed to Step 4
```

### Step 3: Modification Processing

```
FOR modified selections:

  PARSE modification type:
    - Word replacement ("change X to Y")
    - Removal ("remove X")
    - Addition ("add X")
    - Restructure ("make it shorter" / "add subheadline")

  APPLY modification to headline unit

  RE-SCORE the modified version:
    - Use same 6-dimension rubric
    - Note any score changes from original
    - Flag if modification drops below threshold

  PRESENT for final confirmation:
    "ORIGINAL: [old headline] (Score: 8.2)
     MODIFIED: [new headline] (Score: 8.0)

     Modification reduced curiosity score by 0.5.
     Confirm modified version?"
```

### Step 4: Finalize Selection

```
ONCE selection is confirmed:

  CREATE selected_headline object:
    - headline_text
    - subheadline_text
    - selection_source (generated | modified | human_provided)
    - final_score
    - modification_history (if any)
    - confirmation_timestamp

  LOCK selection (no further changes without new selection process)
```

## MODIFICATION HANDLING EXAMPLES

### Example 1: Word Replacement

**Human Input:** "Option 1, but use 'strange' instead of 'SHOCKING'"

**Processing:**
```
ORIGINAL: "The SHOCKING Reason Your Diet Keeps Failing"
MODIFIED: "The STRANGE Reason Your Diet Keeps Failing"

SCORE CHANGE:
  - Emotional Impact: 9 → 8 (less intense)
  - Curiosity: 9 → 9 (maintained)
  - Schema Distance: 7 → 7 (maintained)

OVERALL: 8.65 → 8.35
```

**Confirmation:** "Modified headline scores 8.35/10 (down 0.3). Emotional impact reduced slightly but still strong. Confirm?"

---

### Example 2: Human-Provided Alternative

**Human Input:** "Use this: 'The Diet Lie Nobody Talks About'"

**Processing:**
```
SCORING HUMAN HEADLINE:
  - Clarity: 7 (what lie? unclear)
  - Curiosity: 8 (creates good gap)
  - Specificity: 4 (no specific elements)
  - Emotional Impact: 7 (intrigue, not strong emotion)
  - Schema Distance: 6 (mild violation)
  - Lead Connection: 7 (can work with revelation lead)

OVERALL: 6.55/10 (BELOW THRESHOLD)
```

**Warning:** "Your headline scores 6.55/10, below the 7.5 threshold. Main weakness: specificity (no numbers, names, or timeframes). Would you like to:
1. Proceed anyway
2. Modify to add specificity
3. Select from the generated options"

---

### Example 3: Restructure Request

**Human Input:** "Option 2, but make it shorter for email subject line"

**Processing:**
```
ORIGINAL (Print format):
"Shield Your Wealth From Certain Annihilation And Grow Up To 10 Times
Richer In The Coming SLAUGHTER of The Stock Market"

COMPRESSED (Email format):
"The Coming Stock Market SLAUGHTER (and how to profit)"

SCORE CHANGE:
  - Clarity: 8 → 9 (more direct)
  - Specificity: 8 → 6 (lost "10 times" number)
  - Emotional Impact: 9 → 8 (less intense)

OVERALL: 8.40 → 7.85 (still above threshold)
```

**Confirmation:** "Compressed for email. Score dropped to 7.85 due to lost specificity. Confirm?"

## EDGE CASES

### Below-Threshold Human Headline
```
IF human_headline_score < 7.0:
  WARN but don't block
  "Your headline scores [X], below our 7.5 threshold.
   This doesn't mean it won't work — thresholds are guidelines.
   Proceed with your headline, or would you like to refine it?"
```

### Modification Worsens Significantly
```
IF modification_delta < -1.0:
  WARN strongly
  "This modification drops the score by more than 1 point.
   Consider keeping the original or trying different modification."
```

### Conflicting Modifications
```
IF human requests incompatible changes:
  "You asked to make it shorter AND add more specificity.
   These conflict. Which is the priority?"
```

---

## ANTI-PATTERNS (FORBIDDEN)

1. **Overriding Human Choice**
   - If they pick option #5, use option #5
   - Don't substitute "better" option

2. **Silent Modifications**
   - Every modification must be shown and confirmed
   - Human sees exactly what changed

3. **Blocking Low Scores**
   - Warn, don't block
   - Human can proceed with any score if they choose

4. **Rushing Confirmation**
   - Always show before/after for modifications
   - Always wait for explicit confirmation
