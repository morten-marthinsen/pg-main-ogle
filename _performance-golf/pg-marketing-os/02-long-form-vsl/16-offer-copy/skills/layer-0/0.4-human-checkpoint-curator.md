# Microskill 0.4 — Human Checkpoint Curator

**Skill:** 16-offer-copy
**Microskill:** 0.4-human-checkpoint-curator
**Version:** 1.0
**Layer:** 0 — Foundation & Loading
**Position:** Fourth microskill in Layer 0 (after 0.3)
**Persona:** Alex Rivera — Strategic Integration (Primary), Dr. James Liu — Research Director (Secondary)

---

## PURPOSE

The Human Checkpoint Curator is the mandatory human-in-the-loop gate between data loading and offer copy generation. It presents all confirmed and missing offer details to the human operator in a clear, actionable format and collects their response. This checkpoint is BLOCKING — the pipeline cannot proceed to Layer 1 until the human confirms, modifies, provides, or partially fills the offer data.

This microskill is especially critical because Skill 06 (offer architecture) may be incomplete or a placeholder. In many real-world executions, the human operator is the authoritative source for deliverable descriptions, bonus specifics, exact pricing, guarantee terms, and scarcity details. Even when Skill 06 data is complete, the human must confirm that the loaded data is accurate and campaign-current — offer details change frequently, and stale data produces inaccurate copy.

The curator presents information in a structured display that mirrors the final offer copy structure: product identity, deliverables, bonuses, pricing, guarantee, and scarcity/urgency. This format helps the human verify offer details in the same sequence they will appear in the final copy. The curator accepts four response types: CONFIRM (proceed as-is), MODIFY (adjust specific fields), PROVIDE (supply complete offer data), and PARTIAL (confirm existing data, fill gaps).

---

## IDENTITY

**This microskill IS:**
- The mandatory human checkpoint for offer data verification before copy generation
- A structured offer detail presenter — showing all data in the offer copy sequence
- A response collector that handles four input modes: CONFIRM, MODIFY, PROVIDE, PARTIAL
- A data merger that integrates human input with upstream data
- A schema enforcer that validates human-provided data against required formats
- The final authority gate — no offer copy is written until the human approves the data
- A gap-filling facilitator that guides the human through missing fields

**This microskill is NOT:**
- A validator — validation was completed by 0.3; this presents results and collects input
- A data loader — loading was completed by 0.1; this presents loaded data
- A copy writer — it collects data FOR copy writing, not writes copy
- An offer strategist — it does not suggest pricing, bonuses, or guarantee terms
- An optional step — this checkpoint is ALWAYS executed, even with complete data

## PROCESS

### Step 1: Construct the Checkpoint Display

1. Build the structured checkpoint display following this exact format:

```
+------------------------------------------------------+
| OFFER DETAILS CONFIRMATION                           |
|                                                      |
| Product:                                             |
|   Name: [product_name]                               |
|   Format: [product_format]                           |
|   Core Offer: [positioning or core description]      |
|   Promise: [primary_promise_text]                    |
|                                                      |
| Deliverables: [count] items                          |
|   1. [name] — [brief description]                    |
|   2. [name] — [brief description]                    |
|   ...                                                |
|   [or: "NO DELIVERABLES LOADED — must be provided"]  |
|                                                      |
| Bonuses: [count] bonuses totaling $[total_value]     |
|   1. [name] — $[value]                               |
|   2. [name] — $[value]                               |
|   ...                                                |
|   [or: "NO BONUSES LOADED — please provide if any"]  |
|                                                      |
| Pricing:                                             |
|   Retail Value: $[retail_value]                       |
|   Actual Price: $[actual_price]                       |
|   Terms: [payment terms]                             |
|   Savings: $[amount] ([percentage]%)                 |
|   [or: "PRICING NOT SET — must be provided"]         |
|                                                      |
| Guarantee:                                           |
|   Branded Name: [name or "NEEDS BRANDING"]           |
|   Duration: [days or "NOT SET"]                      |
|   Type: [type or "NOT SET"]                          |
|   Terms: [specifics or "NOT SET"]                    |
|                                                      |
| Scarcity/Urgency:                                    |
|   Type: [type or "NOT SET"]                          |
|   Justification: [reason or "NEEDS JUSTIFICATION"]   |
|                                                      |
| DATA STATUS: [completeness_percentage]% complete     |
| GAPS: [count] critical, [count] non-critical         |
|                                                      |
| OPTIONS:                                             |
|   [1] CONFIRM — proceed with these details           |
|   [2] MODIFY — adjust specific fields                |
|   [3] PROVIDE — supply complete offer data           |
|   [4] PARTIAL — confirm existing, fill gaps          |
|                                                      |
| DEFAULT: [CONFIRM or PARTIAL] ([reason])             |
+------------------------------------------------------+
```

2. For each section, visually distinguish between:
   - Confirmed data (from upstream): displayed normally
   - Missing data: displayed with "NOT SET" or "NEEDS [action]" markers
   - Warning data: displayed with a note (e.g., "generic — needs branding")
3. Calculate and display savings amount and percentage if both retail_value and actual_price are present

### Step 2: Present Critical Gaps Prominently

4. If critical gaps exist, display them prominently ABOVE the options:

```
| CRITICAL GAPS (must be resolved):                    |
|   - [critical gap 1]                                 |
|   - [critical gap 2]                                 |
```

5. If warnings exist, display below the main data:

```
| WARNINGS:                                            |
|   - [warning 1]                                      |
|   - [warning 2]                                      |
```

### Step 3: Set Default and Present to Human

6. Set the default option based on validation_report:
   - completeness >= 95% and no critical gaps: DEFAULT = [1] CONFIRM
   - completeness 60-94% or critical gaps present: DEFAULT = [4] PARTIAL
   - completeness < 60%: DEFAULT = [4] PARTIAL with prominent "offer data needed" banner
7. Present the complete display to the human operator
8. Wait for human response — this is BLOCKING; no timeout

### Step 4: Process Human Response

9. Based on the human's response:

**[1] CONFIRM:**
- Accept all loaded data as-is
- No modifications needed
- Proceed immediately to Layer 1
- Log: `{ response: "CONFIRM", modifications: none, data_source: "upstream" }`

**[2] MODIFY:**
- Human specifies which fields to change
- Accept field-level modifications
- Validate modified fields against schema requirements
- Re-display updated data for final confirmation (mini checkpoint)
- Log: `{ response: "MODIFY", fields_modified: [list], data_source: "merged" }`

**[3] PROVIDE:**
- Human supplies complete offer data in structured or freeform format
- Parse human input into the offer data schema
- Validate all provided fields
- Replace upstream data with human-provided data entirely
- Re-display for final confirmation
- Log: `{ response: "PROVIDE", fields_provided: [list], data_source: "human_provided" }`

**[4] PARTIAL:**
- Human confirms existing data as correct
- Human fills in specific gaps identified by the validation report
- Merge human input with existing upstream data
- Validate merged data meets minimum viable offer requirements
- Re-display merged data for final confirmation
- Log: `{ response: "PARTIAL", fields_filled: [list], data_source: "merged" }`

### Step 5: Validate Human Input

10. For any human-provided data, validate:
    - Product name: non-empty string, no placeholders
    - Deliverables: each has name and description at minimum
    - Bonus values: positive numbers
    - Pricing: retail_value > actual_price, both > 0
    - Guarantee: has type and duration at minimum
    - Scarcity: has type and justification (not generic)
11. If validation fails on human input:
    - Identify specific issues
    - Present issues to human
    - Request correction (loop back to Step 4)
12. Maximum correction loops: 3 (then accept with warnings)

### Step 6: Merge and Finalize Offer Data

13. Construct the finalized `confirmed_offer_data` object:
    - Merge upstream data with human modifications/additions
    - Mark each field's source: `upstream`, `human_provided`, or `merged`
    - Compute final completeness percentage
14. Verify minimum viable offer is now met:
    - Product name: present
    - At least 1 deliverable: present with description
    - Actual price: present and > 0
    - Guarantee: type present
15. If minimum viable STILL not met after human input: log BLOCKING error with specific missing fields

### Step 7: Construct Checkpoint Output

16. Compile the `human_checkpoint_result` with:
    - Confirmed offer data (all fields with sources)
    - Human response type
    - Fields modified/provided by human
    - Final completeness percentage
    - Minimum viable offer status
    - Ready-for-Layer-1 boolean
17. Log the checkpoint interaction:
    - What was presented
    - What the human responded
    - What data changed
    - Time spent at checkpoint
    - Number of correction loops (if any)

## FAILURE MODES

### Failure Mode 1: Human Provides Incomplete Data in PROVIDE Mode
- **Symptom:** Human selects PROVIDE but supplies only partial data, leaving gaps
- **Cause:** Human interprets PROVIDE as "I'll give you what I have" instead of "I'll give you everything"
- **Prevention:** PROVIDE mode prompt clarifies: "Supply COMPLETE offer data — all deliverables, bonuses, pricing, and guarantee"
- **Recovery:** Validate provided data, identify remaining gaps, switch to PARTIAL mode logic — merge what human provided with any remaining upstream data

### Failure Mode 2: Human Confirms Generic Guarantee
- **Symptom:** Human selects CONFIRM but guarantee data contains "money-back guarantee" (generic)
- **Cause:** Human does not recognize that generic guarantee language will produce weak offer copy
- **Prevention:** Display guarantee with explicit "generic — needs branding" warning; if guarantee is generic, prompt human specifically for a branded name before accepting CONFIRM
- **Recovery:** At Layer 2, guarantee writer (2.4) will generate a branded name from product name + guarantee type, but quality will be lower than human-branded

### Failure Mode 3: Pricing Math Inconsistency After Human Modification
- **Symptom:** Human modifies retail_value or actual_price but the savings amount/percentage displayed is stale
- **Cause:** Savings not recalculated after MODIFY response
- **Prevention:** Recalculate ALL derived values (savings_amount, savings_percentage, anchor_ratio) after any pricing modification
- **Recovery:** Re-display updated data with corrected math before final confirmation

### Failure Mode 4: Minimum Viable Still Not Met After Human Input
- **Symptom:** Pipeline blocked even after human interaction because minimum data requirements not met
- **Cause:** Human did not provide all critical gaps, or provided data that failed validation
- **Prevention:** After human response, explicitly check minimum viable and present remaining blocking issues
- **Recovery:** Loop back to human with specific blocking issues identified. Maximum 3 loops before escalation to error state.

### Failure Mode 5: Human Input Schema Mismatch
- **Symptom:** Human provides freeform text that does not parse into structured offer data fields
- **Cause:** PROVIDE mode accepts freeform input but parsing logic does not handle all formats
- **Prevention:** Provide a structured template for human input in PROVIDE mode; for MODIFY mode, specify which field is being changed and accept the new value directly
- **Recovery:** If parsing fails, present the structured template and ask human to re-format their input. Accept partial parsing results and flag unparseable sections.
