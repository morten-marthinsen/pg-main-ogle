**Position:** Second microskill in Layer 3 (after 3.1)

---

## PURPOSE

The Promise Repetition Calibrator ensures that the primary promise is restated throughout the offer section with varied language — never copy-pasted, never identical, but always recognizable as the same core transformation. The SIN methodology intentionally repeats the promise at multiple strategic points to reinforce the prospect's desire. But repetition without variation creates copy that sounds robotic. Variation without consistency creates copy that feels unfocused.

The calibrator maps exactly where the promise appears in the offer copy (product summary, value demonstration, consequence amplification, CTAs, and optionally in the guarantee), verifies that each instance uses different language, and ensures all instances convey the same core transformation. It also checks that the promise is not OVER-repeated (diminishing returns) or UNDER-repeated (missed reinforcement opportunities).

This is Offer Copy Principle #2: "Promise restated, never repeated." The calibrator is the enforcement mechanism for this principle. It treats the offer copy as a complete text and examines every instance of the promise to ensure the pattern of varied restatement is properly executed.

---

## IDENTITY

**This microskill IS:**
- A promise restatement mapper — locating every instance of the promise in the offer copy
- A variation verifier — confirming each restatement uses different words and phrasing
- An over-repetition detector — flagging excessive promise mentions that dilute impact
- An under-repetition detector — flagging missed opportunities to reinforce the promise
- A consistency enforcer — ensuring all variations convey the same core transformation
- A language diversity analyst — measuring how different each restatement is from the others

**This microskill is NOT:**
- A promise writer — the promise comes from Skill 04
- A restatement generator — it calibrates existing restatements, Layer 2 writers generate them
- A content rewriter — it identifies issues and provides direction; it does not rewrite entire sections
- A transition optimizer — that was 3.1
- A structural reorganizer — promise placement is fixed by the SIN methodology

## PROCESS

### Step 1: Extract the Primary Promise

1. Load the primary promise text from the promise package
2. Identify the core transformation elements:
   - Before state: what the prospect currently experiences
   - After state: what the prospect will experience
   - Core verb: the action the product enables (restore, double, add, eliminate, unlock)
   - Core object: what changes (sleep, revenue, distance, weight, confidence)
3. Document the "promise DNA" — the irreducible elements that must be present in every restatement

### Step 2: Map All Promise Instances in the Offer Copy

4. Scan the complete offer copy text and identify every instance where the promise is stated or restated
5. Expected promise restatement points (per SIN methodology):

| Location | SIN Chunk | Purpose |
|----------|-----------|---------|
| Product Summary | Chunk 3 | "You have everything you need to [promise]" |
| Value Demonstration | Chunk 7 | Benefits reference the promise transformation |
| Consequence Amplification | Chunk 14 | Current pain = before state of promise |
| CTA 1 | Chunk 13 | "Start [promise] today" |
| CTA 2 | Chunk 15 | "Stop [before state], start [after state]" |
| CTA 3 | Chunk 17 | "Grab [product] and [promise]" |
| Guarantee | Chunk 12 | "I'm confident [product] will [promise]" (optional) |
| Contrast Enhancement | Chunk 18 | "[Before state] vs. [after state]" (optional) |

6. For each identified instance, extract:
   - Exact text containing the promise
   - Location (chunk number and section)
   - Language used (specific words and phrasing)
   - Transformation elements present (before state, after state, both, or partial)

### Step 3: Verify Variation — No Copy-Paste

7. Compare every pair of promise instances:
   - Calculate language overlap: what percentage of words are shared between two instances?
   - Flag any pair with > 50% word overlap as "potentially copy-pasted"
   - Flag any pair with identical phrasing as "copy-paste violation"
8. Create a variation matrix:

```
         | Summary | Val Demo | Conseq. | CTA 1 | CTA 2 | CTA 3 |
---------|---------|----------|---------|-------|-------|-------|
Summary  |   --    | [score]  | [score] |[score]|[score]|[score]|
Val Demo | [score] |   --     | [score] |[score]|[score]|[score]|
Conseq.  | [score] | [score]  |   --    |[score]|[score]|[score]|
CTA 1    | [score] | [score]  | [score] |  --   |[score]|[score]|
CTA 2    | [score] | [score]  | [score] |[score]|  --   |[score]|
CTA 3    | [score] | [score]  | [score] |[score]|[score]|  --   |
```

Where score = 10 means completely different language, 1 means nearly identical.

9. Target: every pair should score >= 6 (meaningful variation)

### Step 4: Verify Consistency — Same Core Transformation

10. For each promise instance, verify:
    - Does it convey the same transformation? (before → after)
    - Does it reference the same core object? (sleep, revenue, distance, etc.)
    - Would a reader recognize this as the same promise stated differently?
11. Flag any instance that:
    - Introduces a NEW promise not in the original promise package
    - Changes the scope of the transformation (bigger or smaller than the primary promise)
    - Contradicts another instance (different timeframe, different outcome)
12. The promise must be RECOGNIZABLE across all instances — a reader should instinctively feel "yes, that's what this product does" at every restatement

### Step 5: Check Repetition Frequency

13. Count total promise instances:
    - Minimum: 3 (product summary + at least 2 CTAs)
    - Optimal: 5-7 (summary + value demo references + consequence + CTAs)
    - Maximum: 8 (every possible location — may feel over-repeated)
14. If under minimum (< 3):
    - Identify missing restatement points
    - Recommend adding promise reference at: product summary (highest priority), CTA 1, consequence amplification
15. If over maximum (> 8):
    - Identify lowest-impact instances
    - Recommend softening or removing excess instances (usually in minor transitions)
16. Assess spacing:
    - Promise instances should be roughly evenly distributed through the offer copy
    - No more than 2 instances within a single paragraph
    - No gap of more than 200 words between promise references in the deliverable-to-CTA section

### Step 6: Assess Restatement Quality

17. Rate each restatement on quality dimensions:

**Vividness (1-10):** How emotionally vivid is this restatement?
- 10: "Wake up every morning feeling ten years younger, looking in the mirror with genuine pride"
- 5: "Improve your sleep and feel better"
- 1: "Get the results you want"

**Specificity (1-10):** How specific is this restatement?
- 10: "Add 30 yards to every drive and break 80 for the first time in your life"
- 5: "Improve your golf game significantly"
- 1: "Get better at golf"

**Emotional Resonance (1-10):** How strongly does this restatement connect to the prospect's desires/fears?
- 10: Directly references a specific emotional pain or desire the prospect has
- 5: References a general desire
- 1: States a fact with no emotional connection

18. Flag any restatement scoring below 6 on any dimension for revision
19. The best restatement should appear at the most critical point (usually CTA 2 or consequence amplification)

### Step 7: Generate Revision Recommendations

20. For any flagged issues, generate specific revision recommendations:
    - Copy-paste violations: "Restatement at [location] uses identical language to [location]. Rewrite using: [alternative phrasing from restatement bank or new variation]"
    - Low variation: "Restatements at [location] and [location] share too much language. Differentiate by [specific suggestion]"
    - Missing restatements: "No promise reference found at [expected location]. Add: [suggested restatement]"
    - Low quality: "Restatement at [location] scores [X] on vividness. Improve by adding: [specific emotional detail]"
21. Provide alternative phrasings from the restatement bank (promise package) for each flagged instance

### Step 8: Compile Calibration Report

22. Package the complete calibration output with:
    - Map of all promise instances with exact text and locations
    - Variation matrix with all pairwise scores
    - Consistency verification results
    - Frequency assessment (count, distribution, spacing)
    - Quality scores per instance
    - Revision recommendations for flagged instances
    - Overall promise repetition calibration score

## QUALITY CRITERIA

| Criterion | Pass Condition |
|-----------|---------------|
| All instances mapped | Every promise restatement in the offer copy is identified |
| Minimum 3 instances | At least 3 promise restatements found |
| No copy-paste | Zero pairs with identical phrasing |
| Variation above threshold | All pairwise variation scores >= 6 |
| Consistent transformation | All instances convey the same core transformation |
| No scope changes | No instance expands or shrinks the promise beyond original |
| Optimal frequency | 3-8 total instances |
| Even distribution | No gap > 200 words between instances in main offer body |
| Quality above threshold | All instances score >= 6 average on vividness, specificity, resonance |
| Revision recommendations provided | All flagged issues have specific, actionable recommendations |
| Best restatement at critical point | Highest-quality restatement appears at consequence amplification or CTA 2 |

## FAILURE MODES

### Failure Mode 1: Copy-Paste Promise
- **Symptom:** Product summary and CTA 1 use the exact same words: "restore deep sleep and reverse aging"
- **Cause:** Layer 2 writers used the primary promise text verbatim at multiple points
- **Prevention:** Variation matrix catches identical phrasing; restatement bank provides alternatives
- **Recovery:** Rewrite one instance using a different angle: instead of "restore deep sleep" → "finally sleep through the night like you did twenty years ago"

### Failure Mode 2: Promise Drift
- **Symptom:** CTA 2 promises "lose 30 pounds" but the primary promise is "restore deep sleep and reverse aging"
- **Cause:** Writer introduced a supporting promise as if it were the primary promise
- **Prevention:** Consistency check verifies all instances reference the same core transformation
- **Recovery:** Rewrite drifted instance to align with primary promise; supporting promises can be referenced but must not replace the primary

### Failure Mode 3: Under-Repetition
- **Symptom:** Promise appears only in the product summary — nowhere else in the offer
- **Cause:** Layer 2 writers focused on deliverable details and forgot to weave the promise throughout
- **Prevention:** Frequency check requires minimum 3 instances
- **Recovery:** Add promise restatements at CTA 1, consequence amplification, and CTA 2 at minimum

### Failure Mode 4: Over-Repetition
- **Symptom:** Promise restated 12 times — every paragraph mentions "deep sleep and reverse aging"
- **Cause:** Writer used the promise as a filler or crutch in every section
- **Prevention:** Maximum 8 instances; flag clusters of 2+ in one paragraph
- **Recovery:** Remove or soften the lowest-impact instances; keep the strongest restatements
