**Skill:** 17-close

**Layer:** 0 — Foundation & Loading
**Position:** First microskill in Layer 0
**Persona:** Dr. Liu (Primary)

## PURPOSE

Loads all upstream data packages required for close generation. The close is the FINAL copy execution skill in the pipeline — it draws from nearly every preceding skill to assemble the last persuasive push. This loader retrieves five upstream packages: offer-copy-package.json (Skill 16), product-introduction-package.json (Skill 15), promise-package.json (Skill 11), structure-package.json (Skill 11), and proof-inventory-output.json (Skill 11).

From these packages, this microskill extracts the specific fields needed for close writing: guarantee details (branded name, terms, duration), pricing and value information, main benefits for the "You get" summary, product name and format, the emotional state at the handoff point from Skill 16, CTA language already used in prior sections (to avoid repetition), and proof assets available for P.S. and testimonial integration. Without complete upstream data, the close cannot achieve the Makepeace standard of repeating benefits, presenting the guarantee as confidence, and asking for the sale with specific supporting reasons.

This microskill runs first in Layer 0 and feeds into both the source teaching loader (0.2) and the input validator (0.3). It ensures that every data dependency the close needs is available before any strategic or creative work begins.

---

## IDENTITY

**This microskill IS:**
- The upstream data retrieval system for the close skill
- A multi-package loader pulling from 5 separate upstream skills (01, 04, 07, 12, 13)
- The guarantee detail extractor — branded name, terms, duration, risk reversal framing
- The pricing and value extractor — what the prospect pays and what they receive
- The benefit inventory builder — collecting all benefits for the "You get" summary
- The emotional state reader — capturing the prospect's emotional position at close entry
- The CTA language auditor — tracking what CTA phrases were already used upstream

**This microskill is NOT:**
- A source teaching loader (that is 0.2 — Makepeace methodology and Week 13 patterns)
- A data validator (that is 0.3 — validation happens after loading)
- A close writer or strategist — this is purely a data retrieval operation
- A guarantee architect — guarantee type and terms are already decided upstream

## PROCESS

### Step 1: Load Offer Copy Package (Skill 16)

1. Retrieve `offer-copy-package.json` from Skill 16 outputs directory
2. Extract guarantee section:
   - `branded_guarantee_name` — the named guarantee (e.g., "Triple-Back-Guarantee," "Double Your Money Back")
   - `guarantee_type` — contract or personal promise
   - `guarantee_duration` — number of days (30, 60, 90, 365)
   - `guarantee_terms` — specific terms and conditions
   - `guarantee_full_text` — the complete guarantee as written in the offer section
3. Extract pricing data:
   - `main_price` — the actual price the prospect pays
   - `price_anchors` — value comparisons used in the offer
   - `value_stack_total` — total perceived value of all deliverables
   - `savings_amount` — difference between value stack and price
   - `price_per_day` — daily cost breakdown if applicable
4. Extract bonus inventory:
   - All bonus items with names, descriptions, and stated values
   - Total bonus value for the "You get" summary
5. Extract downstream handoff data specifically prepared for the close:
   - `cta_language_used` — all CTA phrases used in Skill 13 (to avoid repetition)
   - `emotional_state_at_handoff` — the prospect's emotional position entering the close
   - `urgency_reference` — any urgency elements already established
   - `latest_promise_restatement` — the most recent version of the primary promise
6. Log any missing fields with severity level (critical vs. optional)

### Step 2: Load Product Introduction Package (Skill 15)

7. Retrieve `product-introduction-package.json` from Skill 15 outputs directory
8. Extract product identity data:
   - `product_name` — confirmed product name for consistent use throughout close
   - `product_format` — supplement, digital product, physical product, info product
   - `positioning_statement` — the market position sentence
   - `uniqueness_claim` — the "first" / "only" / "never before" claim
9. Extract mechanism context:
   - `mechanism_name` — the named mechanism for reference in benefit summary
10. Extract credibility signals for potential use in P.S. or guarantee sections
11. Cross-reference product name with Skill 13 to ensure consistency

### Step 3: Load Promise Package (Skill 11)

12. Retrieve `promise-package.json` from Skill 11 outputs directory
13. Extract primary promise:
    - `promise_text` — the core promise statement
    - `promise_type` — the type of promise (transformation, relief, gain, protection)
14. Extract all benefit claims:
    - `main_benefits` — the primary benefits to feature in the "You get" summary
    - `supporting_benefits` — secondary benefits for additional summary bullets
15. Compile master benefit list by merging main and supporting benefits
16. Note any benefits that have been proven (cross-reference with proof inventory in Step 5)

### Step 4: Load Structure Package (Skill 11)

17. Retrieve `structure-package.json` from Skill 11 outputs directory
18. Extract campaign format — determines close format constraints:
    - VSL: longer close, more CTAs, staccato tempo, checkout process description
    - Magalog: close with order form reference, sidebars possible, P.S. prominent
    - Email: shorter close, fewer CTAs, direct link focus
    - Advertorial: editorial-style close, softer CTA language
19. Extract argument arc — understand what logical chain the close must cap
20. Extract close position — where the close sits in the overall structure
21. Note total planned sections for context on how much has come before the close

### Step 5: Load Proof Inventory (Skill 11)

22. Retrieve `proof-inventory-output.json` from Skill 11 outputs directory
23. Extract strongest testimonials:
    - Select top 3-5 testimonials for potential P.S. use (Makepeace P.S. technique: most powerful testimonials)
    - Note testimonial details: name, result, specificity, emotional impact
24. Extract clinical proof for potential guarantee confidence statements
25. Extract statistical proof for potential "reasons why" close theme
26. Extract authority proof for potential "don't go it alone" close theme
27. Extract social proof for potential urgency/demand claims
28. Compile proof assets inventory specifically for close use

### Step 6: Load Campaign Brief

28B. Attempt to load `09-campaign-brief/outputs/campaign-brief-package.json`
28C. If file FOUND and VALID:
     - Extract `campaign_thesis`, `tone_direction`, `avatar_summary`
     - Extract `market_sophistication`, `proof_approach`, `promise_ceiling`
     - Set campaign_brief_status to LOADED
28D. If file NOT FOUND:
     - Set campaign_brief_status to MISSING
     - Log WARNING: "Campaign brief not found — close may not match campaign tone"
     - Continue — non-blocking

### Step 6B: Load Offer Copy Assembled Prose (for continuity)

28E. Attempt to load `16-offer-copy/outputs/offer-copy-assembled-prose.md`
28F. If file FOUND:
     - Load full text of offer copy prose
     - Extract for continuity:
       - How the offer was presented (to continue momentum, not restart)
       - All CTA language used (to AVOID repeating in close CTAs)
       - The rhythm and voice of the offer section
       - What promises/benefits were emphasized in the offer
       - The last paragraph (for seamless transition into close)
     - Set offer_copy_prose_status to LOADED
28G. If file NOT FOUND:
     - Set offer_copy_prose_status to NOT_AVAILABLE
     - Log WARNING: "Offer copy prose not available. Close will rely on handoff metadata."
     - Continue — non-blocking

### Step 6C: Load Context Reservoir (if available)

28H. Attempt to load `~outputs/[project-code]/context-reservoir.md`
28I. If file FOUND:
     - Load relevant sections: proof_elements_detailed (for P.S. testimonials), voc_quotes
     - Set context_reservoir_status to LOADED
28J. If file NOT FOUND:
     - Set context_reservoir_status to NOT_AVAILABLE
     - Continue — non-blocking

### Step 7: Build Close Data Manifest

29. Assemble all extracted data into a unified close data manifest:
    - Guarantee details (branded name, type, duration, terms, full text)
    - Pricing details (price, value, savings, per-day, anchors)
    - Product identity (name, format, positioning, uniqueness, mechanism)
    - Benefits inventory (main + supporting, ordered by impact)
    - Emotional state at close entry
    - CTA language already used (exclusion list)
    - Proof assets for close (testimonials, clinical, statistical, authority, social)
    - Campaign format and structural context
    - Promise restatement (latest version)
30. Tag each data element with source package and confidence level
31. Flag any missing critical elements for validator (0.3)

### Step 7: Generate Load Report

32. Create load summary documenting:
    - Total packages loaded (target: 5/5)
    - Total fields extracted per package
    - Any missing or incomplete fields
    - Data quality notes (e.g., guarantee lacks branded name, no testimonials available)
    - Readiness assessment: READY / PARTIAL / BLOCKED
33. Pass close data manifest to downstream microskills (0.2, 0.3)

## FAILURE MODES

### Failure Mode 1: Missing Offer Copy Package
- **Symptom:** No offer-copy-package.json found or file is empty
- **Cause:** Skill 13 did not complete or did not produce output
- **Prevention:** Verify Skill 13 completion before triggering Skill 14
- **Recovery:** Request Skill 13 re-run via feedback bus; block close execution until offer data available

### Failure Mode 2: Guarantee Data Incomplete
- **Symptom:** Guarantee lacks branded name or specific terms
- **Cause:** Skill 06 or Skill 13 did not fully specify guarantee details
- **Prevention:** Input validator (0.3) catches this; human checkpoint (0.4) can fill gaps
- **Recovery:** Flag for human checkpoint — request branded guarantee name and terms

### Failure Mode 3: Benefits Inventory Too Thin
- **Symptom:** Fewer than 5 benefits extracted from promise package
- **Cause:** Skill 04 produced a narrow promise with few supporting benefits
- **Prevention:** Cross-reference with proof inventory to find additional benefit claims from testimonials
- **Recovery:** Expand benefit list using proof inventory results and product introduction positioning

### Failure Mode 4: Conflicting Data Between Packages
- **Symptom:** Product name in Skill 12 differs from Skill 16, or price inconsistencies
- **Cause:** Upstream skills ran at different times or with different inputs
- **Prevention:** Cross-reference check in Step 7 catches discrepancies
- **Recovery:** Flag conflict for human resolution; use most recent package data as default

### Failure Mode 5: No Emotional State at Handoff
- **Symptom:** Skill 13 handoff does not specify emotional state for close entry
- **Cause:** Skill 13 downstream_handoffs.for_close field incomplete
- **Prevention:** Skill 13 agent mandates this field in its output contract
- **Recovery:** Infer emotional state from offer copy tone and pricing strategy (premium = confidence, discount = urgency)
