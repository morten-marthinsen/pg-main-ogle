## MANDATORY OUTPUT: CHANGES.MD (Learning #47)

**CRITICAL:** Every editorial execution MUST produce a `changes.md` file showing line-by-line changes. This is NOT optional.

### changes.md Structure

```yaml
changes_md_structure:
  format_key:
    strikethrough: "~~Original text~~ = deleted/replaced"
    new_marker: "[NEW] = New text added"
    changed_marker: "[CHANGED] = Modified text"
    rejected_marker: "[REJECTED] = Change rejected by client"
    approval_required: "[APPROVAL REQUIRED] = Needs human confirmation"

  change_entry:
    location: "[Line number or section reference]"
    original_text: "~~strikethrough format~~"
    new_text: "[CHANGED] new text here"
    reason: "[Why this change was made]"
    classification: "[AUTO-APPLY | APPLY-WITH-NOTE | APPROVAL-REQUIRED]"

  sections:
    - change_log: "All changes in document order"
    - changes_to_revert: "Client-rejected changes"
    - approval_pending: "Changes awaiting human approval"
    - summary_table: "Category counts and status"
```

### Change Classification System (Learning #49)

```yaml
change_classifications:
  AUTO_APPLY:
    description: "Safe changes applied directly"
    examples:
      - "Typo corrections"
      - "Formatting errors"
      - "Price formatting ($24999 → $249.99)"
      - "Clear grammatical errors"
    in_edited_copy: "Applied directly"
    in_changes_md: "Listed with [CHANGED] marker"

  APPLY_WITH_NOTE:
    description: "Applied but flagged for awareness"
    examples:
      - "Slop word removal"
      - "Word substitutions"
      - "Structural tweaks"
      - "CTA variations"
    in_edited_copy: "Applied directly"
    in_changes_md: "Listed with reason and note"

  APPROVAL_REQUIRED:
    description: "NEVER apply without human approval"
    examples:
      - "Root cause changes"
      - "Mechanism modifications"
      - "Big idea alterations"
      - "Promise rewording"
      - "Anchor phrase changes"
      - "Major positioning shifts"
    in_edited_copy: "KEEP ORIGINAL until approved"
    in_changes_md: "Listed as [APPROVAL REQUIRED] suggestion only"
```

## MANDATORY OUTPUT: TRACKED-CHANGES.MD (Learning #50)

**CRITICAL:** Every editorial execution MUST produce a `tracked-changes.md` file showing the COMPLETE copy with inline track changes. This is NOT optional. This is the standard professional editing format (like Word Track Changes or Google Docs editing mode).

### tracked-changes.md Structure

```yaml
tracked_changes_structure:
  format_key:
    location: "Top of document"
    contents:
      strikethrough: "~~Original text~~ = deleted/replaced"
      new_marker: "**[NEW]** = New text added"
      changed_marker: "**[CHANGED]** = Modified text"
      rejected_marker: "**[REJECTED]** = Change rejected by client"

  body:
    content: "COMPLETE copy with all changes shown inline"
    requirement: "Every change visible exactly where it occurs"
    note: "NOT abbreviated excerpts — full document"

  change_visibility:
    deleted_text: "Shown with ~~strikethrough~~"
    new_text: "Shown with **[CHANGED]** or **[NEW]** prefix"
    unchanged_text: "Shown normally (no markup)"

  footer:
    summary_table: "Quick reference of all changes by category"
    counts: "Number of changes by type"
```

### Key Distinction from changes.md

| File | Purpose | Format |
|------|---------|--------|
| `changes.md` | Change LOG | List of changes with locations, reasons, classifications |
| `tracked-changes.md` | Track Changes COPY | Full document with inline markup |

**Both are mandatory.** They serve complementary purposes:
- `changes.md` → Quick review of what changed and why
- `tracked-changes.md` → Review changes in context while reading

### tracked-changes.md Quality Checks

```yaml
tracked_changes_verification:
  - check: "Complete copy (not excerpts)"
    status: "[pass | fail]"
  - check: "Format key at top"
    status: "[pass | fail]"
  - check: "All changes shown inline with markup"
    status: "[pass | fail]"
  - check: "Strikethrough for deleted text"
    status: "[pass | fail]"
  - check: "Markers for new/changed text"
    status: "[pass | fail]"
  - check: "Summary table at end"
    status: "[pass | fail]"
```

## CROSS-VERIFICATION STEP (Learning #52 — MANDATORY)

**CRITICAL:** Every change must appear in BOTH `changes.md` AND `tracked-changes.md`. A change appearing in only ONE file is a PROTOCOL VIOLATION.

```yaml
cross_verification:
  purpose: "Ensure zero orphaned changes between the two change-tracking files"

  verification_process:
    step_1:
      action: "Extract all changes from changes.md"
      output: "List of change IDs/locations"

    step_2:
      action: "Extract all inline changes from tracked-changes.md"
      output: "List of ~~strikethrough~~ and [CHANGED]/[NEW]/[INSERTED] markers"

    step_3:
      action: "Cross-reference both lists"
      check: "Every item in changes.md has corresponding markup in tracked-changes.md"
      check: "Every markup in tracked-changes.md has corresponding entry in changes.md"

    step_4:
      action: "Flag any orphaned changes"
      orphan_types:
        - "In changes.md but NOT in tracked-changes.md"
        - "In tracked-changes.md but NOT in changes.md"

  orphan_resolution:
    if_orphan_found:
      - "HALT output assembly"
      - "Add missing entry to whichever file is incomplete"
      - "Re-run cross-verification"
      - "Only proceed when zero orphans"

  quality_gate: "Zero orphaned changes (all changes in both files)"

  verification_output:
    total_changes_in_changes_md: "[count]"
    total_changes_in_tracked_changes_md: "[count]"
    matched: "[count]"
    orphans_found: "[count]"
    orphan_details: "[list if any]"
    verification_status: "[PASS | FAIL]"
```

### Cross-Verification Checklist

```yaml
cross_verification_checklist:
  - check: "All deletions in changes.md appear as ~~strikethrough~~ in tracked-changes.md"
    status: "[pass | fail]"
  - check: "All [CHANGED] entries in changes.md appear with [CHANGED] marker in tracked-changes.md"
    status: "[pass | fail]"
  - check: "All [NEW] entries in changes.md appear with [NEW] marker in tracked-changes.md"
    status: "[pass | fail]"
  - check: "All [INSERTED] entries in changes.md appear with [INSERTED] marker in tracked-changes.md"
    status: "[pass | fail]"
  - check: "All inline markup in tracked-changes.md has corresponding changes.md entry"
    status: "[pass | fail]"
  - check: "Change counts match between files"
    status: "[pass | fail]"
```
