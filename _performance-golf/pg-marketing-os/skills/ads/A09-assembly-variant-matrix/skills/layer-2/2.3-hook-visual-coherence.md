# 2.3: Hook-Visual Coherence

## Purpose

For EVERY valid combination, evaluate whether the visual MATCHES the hook's register. A testimonial hook needs a UGC or talking-head visual, not motion graphics. A data hook needs a chart or text-overlay visual, not lifestyle footage. A demonstration hook needs a product-in-use visual. Copy register and visual register must be aligned.

## Inputs

- **RAW-COMBINATIONS.yaml:** From 1.1 -- full combination details (hook text, visual description)
- **INCOMPATIBILITY-LOG.yaml:** From 1.2 -- to identify which combinations are VALID
- **PLATFORM-MAPPING.yaml:** From 1.3 -- to exclude BLOCKED combinations
- **COPY-CATALOG.yaml:** From 0.1 -- full hook text and hook type for evaluation
- **ASSET-CATALOG.yaml:** From 0.2 -- full visual description and visual type for evaluation

## Process

```
STEP 1: BUILD EVALUATION SET
  - Same as 2.1 and 2.2: include all VALID, FLAGGED, and NEEDS_ADAPTATION combinations
  - Load full hook text and hook type from COPY-CATALOG.yaml
  - Load visual description and visual type from ASSET-CATALOG.yaml
  - Count total combinations to evaluate

STEP 2: FOR EACH COMBINATION, EVALUATE HOOK-VISUAL COHERENCE
  Apply the hook type to visual style compatibility rules:

  TESTIMONIAL / UGC HOOKS:
    - Compatible visual styles: UGC, talking head, before/after
    - Incompatible visual styles: Polished studio, motion graphics
    - Reasoning: Testimonial hooks create authenticity expectation.
      Polished visuals break the authenticity frame.

  DATA / STATISTICS HOOKS:
    - Compatible visual styles: Text overlay, charts, split-screen, data visualization
    - Incompatible visual styles: Pure lifestyle, pure product demo
    - Reasoning: Data hooks set an analytical frame.
      Lifestyle visuals without data overlays create cognitive dissonance.

  DEMONSTRATION HOOKS:
    - Compatible visual styles: Product demo, screen recording, before/after
    - Incompatible visual styles: Pure text overlay, abstract motion graphics
    - Reasoning: Demonstration hooks promise visual proof.
      The viewer expects to SEE the product in action.

  AUTHORITY HOOKS:
    - Compatible visual styles: Talking head, credentialed setting, expert interview
    - Incompatible visual styles: UGC casual, humor/absurd
    - Reasoning: Authority hooks establish expertise.
      Casual or humorous visuals undermine the authority frame.

  STORY / NARRATIVE HOOKS:
    - Compatible visual styles: Lifestyle, talking head, mixed media
    - Incompatible visual styles: Pure text overlay, pure charts, data visualization
    - Reasoning: Story hooks promise narrative engagement.
      Data visuals break the story immersion.

  PATTERN INTERRUPT HOOKS:
    - Compatible visual styles: ANY -- BUT visual must match the specific interrupt
    - Reasoning: Pattern interrupts are diverse. The key is that the visual
      supports the specific interrupt, not that it matches a category.
    - Example: An interrupt about "this ugly fruit" needs to show the fruit.

  QUESTION HOOKS:
    - Compatible visual styles: Text overlay, talking head (most visuals work)
    - Incompatible visual styles: N/A -- questions are flexible
    - Reasoning: Questions work with most visual treatments because
      the question itself carries the engagement.

STEP 3: ASSESS REGISTER ALIGNMENT
  Beyond type-matching, assess REGISTER alignment:
    - Is the visual tone (serious/playful/dramatic/casual) aligned with the hook tone?
    - Is the visual production quality aligned with the hook's implied brand quality?
    - Does the visual reinforce or contradict the hook's emotional register?

STEP 4: ASSIGN VERDICT
  For each combination:
    - PASS: Visual style matches the hook's type and register.
      A viewer seeing this hook with this visual would feel visual coherence.
    - FAIL: Visual style contradicts the hook's type or register.
      A viewer would experience a jarring disconnect between what they hear/read
      and what they see. Record specific mismatch.
    - FLAG: Visual is not the ideal match but doesn't actively contradict.
      Record concern and what visual type would be optimal.

  Each verdict MUST include:
    - Hook type (from 32-type taxonomy or A-J category)
    - Visual type (from asset catalog)
    - Why the visual does/doesn't match the hook's register
    - What visual type would be optimal (for FLAG/FAIL)

STEP 5: DO NOT BATCH-EVALUATE
  CRITICAL: Each combination gets INDIVIDUAL evaluation.
  "All Hook Type A combinations work with Visual V01" is FORBIDDEN.
  Each specific hook text with each specific visual gets its own assessment.

STEP 6: COMPUTE COHERENCE STATISTICS
  - Total evaluated / PASS / FAIL / FLAG
  - Pass rate (%)
  - Most common mismatch patterns (which hook types fail with which visual types)
  - Visual types with highest fail rate
  - Hook categories with highest fail rate

STEP 7: WRITE HOOK-VISUAL-COHERENCE.yaml
```
