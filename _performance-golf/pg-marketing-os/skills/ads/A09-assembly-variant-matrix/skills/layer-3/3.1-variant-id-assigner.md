# 3.1: Variant ID Assigner

## Purpose

Assign standardized, permanent variant IDs to every PASS combination using the binding naming convention: `[CONCEPT]-[HOOK]-[BODY]-[CTA]-[VISUAL]-[PLATFORM]`. Verify all IDs are unique, follow the exact format specification, and concept numbering matches Arena results. This creates the definitive variant-to-component mapping that all downstream processes rely on.

## Inputs

- **FULL-UNIT-COHERENCE.yaml:** From 2.4 -- all combinations with PASS/FLAG/FAIL verdicts
- **COHERENCE-VALIDATION-REPORT.md:** From 2.5 -- promoted FLAGs if any
- **ARENA-SCORES.yaml:** From 0.4 -- concept ID numbering (C1/C2/C3)
- **COPY-CATALOG.yaml:** From 0.1 -- copy element IDs (hook, body, CTA IDs)
- **ASSET-CATALOG.yaml:** From 0.2 -- asset IDs (visual IDs)

## Process

```
STEP 1: BUILD PASS LIST
  - From FULL-UNIT-COHERENCE.yaml, extract all combinations with verdict = PASS
  - From COHERENCE-VALIDATION-REPORT.md, include any promoted FLAGs
  - This is the definitive list of variants for the matrix

STEP 2: ASSIGN VARIANT IDS
  For each PASS combination:
    Construct variant ID using binding convention:

    FORMAT: [CONCEPT]-[HOOK]-[BODY]-[CTA]-[VISUAL]-[PLATFORM]

    RULES (NON-NEGOTIABLE):
    a. CONCEPT: C1, C2, C3 -- numbered from Arena results (ARENA-SCORES.yaml)
    b. HOOK: H01-H99 -- zero-padded to 2 digits, unique within concept
    c. BODY: B1, B2, B3 -- body variant number
    d. CTA: CTA1, CTA2, CTA3 -- CTA variant number
    e. VISUAL: V01-V99 -- zero-padded to 2 digits
    f. PLATFORM: META, TIKTOK, YT, GOOG, PIN, SNAP -- deployment platform LAST

    Examples:
      C1-H03-B1-CTA2-V01-META
      C2-H01-B2-CTA1-V03-TIKTOK
      C3-H07-B1-CTA3-V02-YT

STEP 3: VERIFY ID UNIQUENESS
  - Check: every variant ID is unique across the entire matrix
  - IF duplicates found: HALT -- resolve before proceeding
  - Two combinations with the same hook, body, CTA, visual, AND platform
    would be true duplicates -- remove one

STEP 4: VERIFY FORMAT COMPLIANCE
  For each variant ID, verify:
    a. Contains exactly 6 components separated by hyphens
    b. Concept component matches C[n] format
    c. Hook component matches H[nn] format (zero-padded)
    d. Body component matches B[n] format
    e. CTA component matches CTA[n] format
    f. Visual component matches V[nn] format (zero-padded)
    g. Platform component is one of: META, TIKTOK, YT, GOOG, PIN, SNAP
    h. No spaces, no special characters, only alphanumeric + hyphens
    i. Platform is the LAST component

  IF any ID fails format check: fix immediately before proceeding

STEP 5: CROSS-REFERENCE CONCEPT NUMBERING
  - Verify C1 in variant IDs = C1 in ARENA-SCORES.yaml
  - Verify C2 in variant IDs = C2 in ARENA-SCORES.yaml
  - Verify C3 in variant IDs = C3 in ARENA-SCORES.yaml
  - IF numbering mismatch: HALT -- concept IDs must be consistent

STEP 6: BUILD VARIANT-ID-MAP
  For each variant ID, record the mapping:
    variant_id -> {
      concept_id, concept_name,
      hook_id, hook_source_file,
      body_id, body_source_file,
      cta_id, cta_source_file,
      visual_id, visual_source_file,
      platform
    }

STEP 7: WRITE VARIANT-ID-MAP.yaml
```

## Quality Checks

- [ ] Every PASS combination (and promoted FLAGs) has a variant ID assigned
- [ ] All variant IDs follow binding naming convention EXACTLY
- [ ] All hook and visual numbers are zero-padded to 2 digits (H01, V03)
- [ ] Platform is the LAST component of every ID
- [ ] All IDs are unique (zero duplicates)
- [ ] Concept numbering matches Arena results (C1/C2/C3 consistent)
- [ ] All 6 components present in every ID (no shortcuts like C1-H3-META)
- [ ] No spaces or special characters in any ID
- [ ] Source file paths recorded for every component of every variant
- [ ] Promoted FLAGs are marked as such
