# 0.5: Soul.md Loader

## Purpose
Load project Soul.md if it exists. Extract voice constraints that affect script tone and language register. Note: A04 designs architecture, not copy -- but voice constraints inform module content brief direction (e.g., educational vs. conversational setup, authoritative vs. casual CTA style).

## Inputs
- Project Soul.md (optional but recommended)

## Process

### Step 1: Check for Soul.md
```
CHECK if file exists: [project]/SOUL.md
IF does not exist:
  LOG: "Soul.md not found -- voice constraints unavailable"
  CREATE empty voice_context object with status: "not_loaded"
  PROCEED (this is not a blocker -- Soul.md is optional)
ELSE:
  PROCEED to loading
```

### Step 2: Read and Extract Voice Constraints
```
READ Soul.md file
EXTRACT:
  - voice_register (formal | conversational | educational | casual | authoritative)
  - emotional_ceiling (maximum emotional intensity allowed)
  - emotional_floor (minimum emotional intensity required)
  - energy_signature (high | medium | low | varies)
  - anti_voice (phrases/styles to NEVER use)
  - pacing_signature (rapid-fire | steady | methodical | varied)
```

### Step 3: Interpret for Script Architecture
```
MAP voice constraints to script architecture implications:

IF voice_register == "educational":
  → Prefer Edutainment or AIDA frameworks
  → Setup modules should teach, not just state problems
  → CTAs should be invitation-style, not command-style

IF voice_register == "conversational":
  → Prefer UGC-DR or Hook-Body-CTA frameworks
  → Setup modules should use "you" language
  → CTAs should be peer-recommendation style

IF energy_signature == "high":
  → Fast-Paced Viral framework may be suitable
  → Shorter modules with more transitions
  → CTAs can be urgent/energetic

IF energy_signature == "low":
  → Story Narrative or Edutainment frameworks preferred
  → Longer modules with deeper development
  → CTAs should be thoughtful, not rushed

IF anti_voice contains hype language:
  → Avoid Fast-Paced Viral
  → CTAs must be understated
  → Proof modules should be data-focused, not testimonial-heavy
```

### Step 4: Document Voice-to-Architecture Mappings
```
CREATE voice_architecture_guide:
  recommended_frameworks: [list based on voice register]
  discouraged_frameworks: [list based on anti-voice]
  cta_style_direction: string
  setup_module_tone: string
  proof_module_style: string
```

## Output Schema
```yaml
soul_voice_context:
  soul_md_exists: boolean
  voice_register: string or null
  emotional_range:
    ceiling: string or null
    floor: string or null
  energy_signature: string or null
  anti_voice_patterns: [list or empty]
  pacing_signature: string or null
  voice_architecture_guide:
    recommended_frameworks: [list or empty]
    discouraged_frameworks: [list or empty]
    cta_style_direction: string or null
    setup_module_tone: string or null
    proof_module_style: string or null
  loading_status:
    loaded: boolean
    warnings: [list or empty]
```

## Output File
- **Path:** `layer-0-outputs/0.5-soul-md-loader.md`
- **Minimum size:** 1KB

## Quality Checks
1. If Soul.md exists, it was successfully read
2. If loaded, voice_register is extracted and valid
3. If loaded, voice-to-architecture mappings are documented
4. If not loaded, status clearly indicates "not_loaded" (not an error)
5. Anti-voice patterns (if present) are specific and actionable

## Dependencies
- Project Soul.md (optional dependency -- skill proceeds without it)
- No upstream skill dependencies
- Soul.md is typically created during Skills 00-01 but may not exist for all projects

## Note on Optional Nature
**Soul.md is recommended but not required.** A04 can execute without it -- frameworks will be selected purely on functional criteria (hook type, platform, length, vertical, awareness). However, when Soul.md is present, it adds an additional constraint layer that improves voice consistency across all ad creative.
