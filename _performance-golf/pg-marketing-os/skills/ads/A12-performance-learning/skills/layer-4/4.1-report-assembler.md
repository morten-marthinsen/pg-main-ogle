# 4.1: Report Assembler

**Version:** 1.0
**Layer:** 4 (Output Packaging)
**Type:** Assembly
**Model:** sonnet
**Dependencies:** All Layer 1-3.5 outputs, GATE_3.5 must be PASS
**Output:** PERFORMANCE-LEARNING-REPORT.md (Post-Campaign, 50KB+) OR MID-FLIGHT-LEARNING.md (Mid-Flight, 15KB+)

---

## Purpose

Assemble all Layer 1-3.5 outputs into the primary deliverable. This is an ASSEMBLY operation -- it combines pre-validated artifacts. It does NOT generate new analysis. POST-CAMPAIGN mode produces PERFORMANCE-LEARNING-REPORT.md (50KB+, all 8 required sections). MID-FLIGHT mode produces MID-FLIGHT-LEARNING.md (15KB+, focused on early signals and kill recommendations). Chunked assembly is MANDATORY because LLMs have ~30KB output limits per response. Use 5-10 write operations across 3 phases with size checkpoints at each phase.

## Process

### Step 1: Verify Gate 3.5 and Determine Report Type

```
READ checkpoints/LAYER_3.5_COMPLETE.yaml
IF NOT EXISTS:
  --> HALT: "GATE_3.5 not passed. Cannot begin Layer 4."

DETERMINE report type:
  IF mode == POST_CAMPAIGN:
    --> Target: PERFORMANCE-LEARNING-REPORT.md
    --> Minimum size: 50KB
    --> Required sections: ALL 8
    --> Chunked assembly: 3 phases, 5-10 write operations
  IF mode == MID_FLIGHT:
    --> Target: MID-FLIGHT-LEARNING.md
    --> Minimum size: 15KB
    --> Required sections: Focused subset (early signals, kill recs, initial learnings)
    --> Chunked assembly: 2 phases, 3-5 write operations
```

### Step 2: Chunked Assembly -- Post-Campaign Mode (50KB+)

```
PHASE 1: SKELETON + EXECUTIVE SUMMARY + CLASSIFICATION (>= 10KB)

  WRITE file structure with all 8 section headers.

  ## Metadata
  - Project: [name]
  - Mode: Post-Campaign Analysis
  - Campaign Duration: [start_date to end_date]
  - Total Spend: $[amount]
  - Total Variants Tested: [integer]
  - Variants with Sufficient Data: [integer]
  - Platforms: [list]
  - Vertical: [vertical]
  - Report Date: [ISO 8601]
  - Previous Report Date: [ISO 8601 if incremental, N/A if first]

  ## Section 1: Executive Summary
  ASSEMBLE FROM:
    - 1.4 classification: Campaign performance overview, win rate, top/bottom 3 variants
    - 2.1 prediction comparison: A10 prediction accuracy percentage
    - 1.3 time series: Creative fatigue summary (average half-life, refresh timing)
    - 3.6 learning validator: Top 3 actionable learnings (most impactful)
    - 3.5.x propagation outputs: Engine files updated list
  NOTE: Executive summary should be 2-3KB. Enough detail to stand alone.

  ## Section 2: Variant Classification
  ASSEMBLE FROM 1.4 classification:
    - Full VARIANT CLASSIFICATION TABLE (every variant)
    - Winners detail: each winner with variant ID, hook type, visual style, CPA, ROAS, hook rate, why it won
    - Performers detail: each performer with optimization recommendations
    - Losers detail: each loser with failure hypothesis
    - Insufficient Data variants: flagged for continued monitoring
    - Win rate vs benchmark analysis
    - Spend distribution analysis (budget allocation to winners vs losers)

  SIZE CHECKPOINT (BLOCKING):
    Measure file size after Phase 1.
    IF file < 10KB --> HALT: "Phase 1 assembly insufficient. Expand Section 1 and 2."

PHASE 2: PREDICTION ANALYSIS + ELEMENT PERFORMANCE (>= 25KB cumulative)

  ## Section 3: Prediction vs Reality Analysis
  ASSEMBLE FROM 2.1 prediction comparison + 2.2 divergence analysis + 2.3 scoring model audit:
    - Full PREDICTION ACCURACY TABLE
    - Divergence analysis for each non-ACCURATE prediction:
      - Variant ID, predicted rank vs actual rank
      - Divergence type (OPTIMISTIC / PESSIMISTIC / WILDLY_WRONG)
      - Hypothesis for why prediction diverged
    - Scoring model audit results:
      - Criteria that correlated with actual performance
      - Criteria that did NOT correlate
      - Recommended scoring weight adjustments
    - A10 systematic biases identified

  ## Section 4: Hook Type Performance
  ASSEMBLE FROM 2.4 hook type performance:
    - Performance by hook type table:
      | Hook Type | Count | Avg Hook Rate | Avg CPA | Avg ROAS | Win Rate | vs Benchmark |
    - Hook types that outperformed taxonomy benchmarks (with hypotheses)
    - Hook types that underperformed (with hypotheses)
    - Hook type x platform cross-tabulation
    - Hook category (A-J) performance ranking
    - Comparison with A01 competitive intelligence

  ## Section 5: Element Attribution Analysis
  ASSEMBLE FROM 2.5 element attribution + 1.3 time series:
    - Per-element performance ranking:
      - HOOKS ranked by isolated performance contribution
      - VISUALS ranked by isolated performance contribution
      - CTAs ranked by isolated performance contribution
      - BODIES ranked by isolated performance contribution
    - Winning element combinations (synergy detection)
    - Losing element combinations (conflict detection)
    - Creative fatigue analysis:
      - Per-variant fatigue curves
      - Average creative half-life
      - Fatigue patterns by element type
      - Frequency at fatigue onset
      - Refresh cadence recommendations

  SIZE CHECKPOINT (BLOCKING):
    Measure file size after Phase 2.
    IF file < 25KB --> HALT: "Phase 2 assembly insufficient. Expand Sections 3-5."

PHASE 3: LEARNINGS + PROPAGATION + RECOMMENDATIONS (>= 50KB cumulative)

  ## Section 6: Extracted Learnings
  ASSEMBLE FROM 3.1-3.5 learning outputs + 3.6 learning validator:
    - Full list of all validated learnings (minimum 10)
    - Each learning in WHAT/WHY/HOW/CONFIDENCE/SCOPE format
    - Learnings organized by domain:
      - Hook learnings (from 3.1)
      - Visual learnings (from 3.2)
      - Platform learnings (from 3.3)
      - Fatigue learnings (from 3.4)
      - Prediction model learnings (from 3.5)
    - Learning priority ranking (highest impact first)
    NOTE: This section should be the LARGEST section (15-20KB minimum).
    Each learning requires detailed data, hypothesis, and actionable recommendation.
    Do NOT summarize learnings. Include FULL detail from each learning output.

  ## Section 7: Engine Propagation Status
  ASSEMBLE FROM 3.5.1-3.5.5 propagation outputs:
    - Full PROPAGATION VERIFICATION TABLE:
      | Engine File | Patch File | Changes Proposed | Human Approval | Applied (Y/N) | Verified |
    - For each engine file: changes proposed, approval status, applied status, verification
    - Engine files NOT updated and why (rejected patches with reasons)
    - Cumulative engine improvement tracking

  ## Section 8: Next Campaign Recommendations
  SYNTHESIZE FROM all layer outputs:
    - Creative strategy recommendations:
      - Prioritized hook types for next campaign (ranked with rationale)
      - Recommended visual styles (with platform specificity)
      - Recommended ad lengths and formats
      - Recommended testing approach (which elements to test first)
    - Budget allocation recommendations:
      - Recommended initial spend distribution across variants
      - Recommended kill threshold (based on actual campaign data)
      - Recommended scale threshold for winners
    - Creative refresh strategy:
      - When to refresh creatives (based on fatigue analysis)
      - How many new variants to produce per refresh cycle
      - Which elements to refresh first (hooks, visuals, CTAs)
    - Platform strategy:
      - Platform prioritization for next campaign
      - Platform-specific creative recommendations
    - Testing volume recommendations:
      - Recommended variant count for next campaign
      - Recommended testing cadence
      - Minimum budget per variant for statistical significance

  SIZE CHECKPOINT (BLOCKING):
    Measure file size after Phase 3.
    IF file < 50KB --> HALT: "Phase 3 assembly insufficient. Expand Section 6 (learnings) first."
```

### Step 3: Chunked Assembly -- Mid-Flight Mode (15KB+)

```
IF mode == MID_FLIGHT:

  PHASE 1: SKELETON + EARLY SIGNALS (>= 5KB)
    ## Metadata (same format, mode = Mid-Flight Analysis)
    ## Section 1: Executive Summary (focused on early signals)
    ## Section 2: Variant Classification (current winners/losers with caveat about data maturity)

  PHASE 2: LEARNINGS + RECOMMENDATIONS (>= 15KB cumulative)
    ## Section 3: Early Signal Analysis (from available Layer 2 outputs)
    ## Section 4: Initial Learnings (from Layer 3, noting confidence level)
    ## Section 5: Kill Recommendations (variants to pause immediately)
    ## Section 6: Scale Recommendations (variants to increase budget)
    ## Section 7: Next Steps (what to monitor, when to run Post-Campaign)

  NOTE: Mid-Flight report does NOT include:
    - Engine propagation status (deferred to Post-Campaign)
    - Full hook type performance analysis (insufficient data)
    - Element attribution (requires more data points)
    - Fatigue analysis (too early to measure fatigue curves)

  SIZE CHECKPOINT (BLOCKING):
    IF file < 15KB --> HALT: "Mid-Flight report insufficient. Expand early signal analysis."
```

### Step 4: Post-Assembly Validation

```
AFTER assembly is complete:

  1. VERIFY all required sections exist and are populated (not empty headers)
  2. VERIFY file size meets minimum (50KB Post-Campaign, 15KB Mid-Flight)
  3. VERIFY no placeholder text remains (no "[TODO]" or "[PLACEHOLDER]")
  4. VERIFY all tables are populated with actual data
  5. VERIFY all learnings include WHAT/WHY/HOW (not just WHAT)
  6. VERIFY cross-references to layer output files are correct paths
  7. VERIFY metadata is complete and accurate
```

## Quality Checks

1. Chunked assembly used (5-10 writes for Post-Campaign, 3-5 for Mid-Flight) -- NOT single write
2. Size checkpoints passed at each phase (10KB, 25KB, 50KB for Post-Campaign)
3. All 8 sections populated for Post-Campaign (7 for Mid-Flight)
4. Section 6 (Extracted Learnings) is the LARGEST section (15-20KB minimum for Post-Campaign)
5. No placeholder text, no empty table cells, no "[TODO]" markers
6. All learnings include full WHAT/WHY/HOW format (not just WHAT)
7. Propagation status table complete with approval and application status
8. Next campaign recommendations are specific and data-backed (not generic advice)
9. Assembly is pure COMBINATION of pre-validated artifacts -- no new analysis generated
10. File size verified after final write operation
