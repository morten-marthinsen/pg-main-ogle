# 1.1: Platform Normalizer

**Version:** 1.0
**Layer:** 1 (Performance Data Ingestion & Metric Calculation)
**Type:** Data Processing
**Model:** sonnet
**Dependencies:** Layer 0 outputs (GATE_0 must be PASS)
**Output:** layer-1-outputs/1.1-platform-normalizer.md

---

## Purpose

Normalize platform-specific data into a unified schema. Handle platform naming differences (e.g., Meta "Link Clicks" = Google "Clicks"). Handle attribution differences (Meta 7-day click, Google 30-day). Produce per-platform data quality reports. Flag any anomalies (sudden spend spikes, zero-impression periods, negative metrics). This is the data cleansing step that ensures all downstream analysis operates on consistent, comparable data.

## Process

### Step 1: Verify Gate 0 Passed

```
READ checkpoints/LAYER_0_COMPLETE.yaml
IF NOT EXISTS:
  --> HALT: "GATE_0 not passed. Cannot begin Layer 1."
  --> Return to Layer 0

VERIFY gate_0.status == "PASS"
IF NOT PASS:
  --> HALT: "GATE_0 did not pass. Status: [status]"
```

### Step 2: Platform-Specific Field Normalization

```
FOR EACH platform in the dataset:

  META NORMALIZATION:
    "Link Clicks" --> link_clicks
    "Results" / "Website Purchases" --> conversions
    "Amount Spent" --> spend
    "Impressions" --> impressions
    "Reach" --> reach
    "3-Second Video Views" / "ThruPlays" --> video_views_3s
    "Video Plays at 100%" --> video_completions
    "Post Reactions" + "Post Comments" + "Post Shares" --> engagement_actions
    Attribution: Default 7-day click / 1-day view
    Currency: Normalize to USD if other currency detected

  GOOGLE ADS NORMALIZATION:
    "Clicks" --> link_clicks
    "Conversions" --> conversions
    "Cost" --> spend
    "Impressions" --> impressions
    "Reach" --> reach (may not be available at ad level)
    "Video views" --> video_views_3s (note: Google counts differ from Meta)
    "Video played to 100%" --> video_completions
    "Interactions" --> engagement_actions
    Attribution: Default 30-day click / conversion window varies
    Currency: Normalize to USD

  TIKTOK NORMALIZATION:
    "Clicks" --> link_clicks
    "Conversions" --> conversions
    "Spend" --> spend
    "Impressions" --> impressions
    "Reach" --> reach
    "Video Views" --> video_views_3s (TikTok auto-play, different baseline)
    "Video Watched to End" --> video_completions
    "Engagement" --> engagement_actions
    Attribution: Default 7-day click / 1-day view
    Currency: Normalize to USD

  YOUTUBE NORMALIZATION:
    "Clicks" --> link_clicks
    "Conversions" --> conversions
    "Cost" --> spend
    "Impressions" --> impressions
    "Unique Reach" --> reach
    "Views" --> video_views_3s (YouTube counts at 30s, different from Meta 3s)
    "Video Played to 100%" --> video_completions
    "Engagements" --> engagement_actions
    Attribution: Varies by campaign type
    Currency: Normalize to USD

DOCUMENT all mapping decisions with platform-specific notes on metric definitions.
```

### Step 3: Attribution Window Documentation

```
DOCUMENT attribution differences across platforms:

  CRITICAL: Attribution windows affect conversion counts.
    Meta 7-day click = conversion within 7 days of clicking
    Google 30-day click = conversion within 30 days of clicking
    TikTok 7-day click = conversion within 7 days of clicking

  IF multiple platforms present:
    --> NOTE: "Cross-platform conversion comparison is affected by attribution windows."
    --> "Meta/TikTok 7-day click vs Google 30-day click means Google may report
        more conversions for the same traffic volume."
    --> Document this for Layer 2 and Layer 3 analysts

  Do NOT attempt to adjust for attribution differences --
  document them and let analysts account for them in hypotheses.
```

### Step 4: Anomaly Detection and Flagging

```
FOR EACH variant-day combination in normalized data:

  DETECT ANOMALIES:
    spend_spike:
      IF daily_spend > 3x rolling_7_day_average_spend:
        FLAG: "Spend spike on [date] for [variant_id]: $[X] vs $[avg] average"
        Severity: MEDIUM

    zero_impression_with_spend:
      IF spend > 0 AND impressions == 0:
        FLAG: "Spend without impressions on [date] for [variant_id]: $[X] spent, 0 impressions"
        Severity: HIGH (possible data quality issue)

    impossible_values:
      IF conversions > link_clicks:
        FLAG: "More conversions than clicks for [variant_id] on [date]"
        Severity: HIGH (likely view-through conversions or data issue)
      IF link_clicks > impressions:
        FLAG: "More clicks than impressions for [variant_id] on [date]"
        Severity: HIGH (data error)

    negative_values:
      IF any metric < 0:
        FLAG: "Negative value in [metric] for [variant_id] on [date]"
        Severity: HIGH (refunds, adjustments, or data error)

    missing_days:
      IF a variant has data for day 1, 2, 3, 5, 6 but NOT day 4:
        FLAG: "Missing data for [variant_id] on [date]"
        Severity: LOW (may be normal -- variant paused/unpaused)
```

### Step 5: Produce Per-Platform Data Quality Report

```
FOR EACH platform:
  PRODUCE:
    platform: [name]
    date_range: [start] to [end]
    variants_on_platform: [count]
    total_rows: [count]
    data_completeness:
      fields_present: [list]
      fields_missing: [list]
    anomalies:
      total_anomalies: [count]
      high_severity: [count]
      medium_severity: [count]
      low_severity: [count]
      anomaly_details: [list]
    attribution_window: [description]
    currency: USD (normalized)
    notes: [platform-specific observations]
```
