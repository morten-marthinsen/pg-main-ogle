# 3.1: Variant ID Assignment

## Purpose
Assign unique, structured IDs to every variant following the naming convention: `[ConceptID]-[Platform]-[Length]-[VariantType]-[Number]`. Produce the MASTER VARIANT ID TABLE that maps every variant to its formal ID. These IDs are used in all downstream files and by A08 and A09.

## Inputs
- Quality-checked variant inventory (from 2.4, with quality updates from Layer 2.5)
- All variant scripts (from Layers 1-2, with any Layer 2.5 revisions)

## Process
```
STEP 1: READ the variant inventory (2.4) and Layer 2.5 quality results.
  - Exclude any variants that were removed during quality check
  - Include only variants that passed all Layer 2.5 checks

STEP 2: APPLY the naming convention to each variant.

  NAMING CONVENTION:
    [ConceptID]-[Platform]-[Length]-[VariantType]-[Number]

  COMPONENTS:
    ConceptID: C001, C002, C003, etc.
    Platform: META, TIKTOK, YOUTUBE, GOOGLE, LINKEDIN, PINTEREST
    Length: 6s, 15s, 30s, 60s, 2min, 3min
    VariantType: BASE, HOOK01-HOOK10, CTA01-CTA03, PLAT
    Number: Sequential if needed for disambiguation

  EXAMPLES:
    C001-META-60s-BASE         -- Concept 1, Meta, 60s, base version
    C001-META-60s-HOOK03       -- Concept 1, Meta, 60s, hook swap #3
    C001-META-60s-CTA02        -- Concept 1, Meta, 60s, CTA variant #2
    C001-TIKTOK-15s-PLAT       -- Concept 1, TikTok, 15s, platform adaptation
    C002-YOUTUBE-2min-HOOK07   -- Concept 2, YouTube, 2min, hook swap #7

STEP 3: VERIFY all IDs are unique.
  - No duplicate IDs
  - No ambiguous IDs
  - Every variant has exactly one ID

STEP 4: PRODUCE the Master Variant ID Table.

STEP 5: VERIFY total count matches the quality-checked variant inventory.
```
