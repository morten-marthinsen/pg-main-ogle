# 1.5: Naming Key Generator

## Purpose

Generate NAMING-KEY.md -- the Rosetta Stone that maps every file name to its variant matrix coordinates, every in-platform ad name to its file name, and every variant ID to its A10 scoring rank. This is the cross-reference document that makes the entire package traceable. Without it, the media buyer cannot trace an in-platform ad back to the scoring data, the variant matrix, or the original creative components.

This document also feeds A12 (Performance Learning Loop), which needs to map performance data back to creative variables (hook type, body framework, CTA type, visual treatment).

## Inputs

- 1.1-meta-package-builder.md output (Meta file names and in-platform names)
- 1.2-tiktok-package-builder.md output (TikTok file names and in-platform names)
- 1.3-youtube-package-builder.md output (YouTube file names and in-platform names)
- 1.4-google-display-package-builder.md output (Google file names and in-platform names)
- 0.1-scoring-report-loader.md output (A10 scores and test priority ranks)
- 0.2-variant-matrix-loader.md output (full variant matrix with component IDs)

## Process

```
STEP 1: COLLECT ALL NAMING DATA FROM PLATFORM BUILDERS
  - From 1.1 (Meta): Collect all [new file name, original file name, variant ID, in-platform name]
  - From 1.2 (TikTok): Collect all [new file name, original file name, variant ID, in-platform name]
  - From 1.3 (YouTube): Collect all [new file name, original file name, variant ID, in-platform name]
  - From 1.4 (Google): Collect all [new file name, original file name, variant ID, in-platform name]
  - For any N/A platform builders: skip (no data to collect)

STEP 2: BUILD FILE NAME -> VARIANT MATRIX CROSS-REFERENCE
  FOR EACH file across all platforms:
    - File name: [PLATFORM-CONCEPT-HOOK-BODY-CTA-VISUAL-PLACEMENT.ext]
    - Maps to:
      - Variant ID: [from matrix]
      - Concept: [C1/C2/C3 + descriptive name]
      - Hook ID: [HK##] + Hook Type from AD-HOOK-TAXONOMY.md
      - Body ID: [BD##] + Body Framework (PAS, BAB, AIDA, etc.)
      - CTA ID: [CT##] + CTA Type (Urgency, RiskReversal, etc.)
      - Visual ID: [VIS-X] + Visual Type (UGC, Polished, etc.)
      - Platform: [Meta/TikTok/YouTube/Google]
      - Placement: [feed/stories/reels/pre-roll/etc.]

STEP 3: BUILD IN-PLATFORM NAME -> FILE NAME CROSS-REFERENCE
  FOR EACH file across all platforms:
    - In-platform name: [Date-Concept-HookType-Body-CTA-Visual-Notes]
    - Maps to file name: [PLATFORM-CONCEPT-HOOK-BODY-CTA-VISUAL-PLACEMENT.ext]
    - This is what the media buyer uses to find the source file for any ad in Ads Manager

STEP 4: BUILD VARIANT ID -> A10 SCORE CROSS-REFERENCE
  FOR EACH approved variant:
    - Variant ID: [from matrix]
    - A10 Overall Score: [X.X]
    - A10 Test Priority Rank: [integer]
    - A10 Compliance Flags: [none or list]
    - Files in package: [list of file names across all platforms]

STEP 5: BUILD COMPONENT QUICK-REFERENCE
  - List all unique hooks with: Hook ID, Hook Type, description
  - List all unique bodies with: Body ID, Framework, description
  - List all unique CTAs with: CTA ID, CTA Type, description
  - List all unique visuals with: Visual ID, Visual Type, description
  - This lets the media buyer understand what each component code means

STEP 6: VALIDATE COMPLETENESS
  - Every file in the package appears in the naming key (zero orphans)
  - Every approved variant appears in the naming key (zero missing)
  - Every in-platform name maps to exactly one file
  - Every file maps to exactly one variant matrix entry

STEP 7: WRITE NAMING-KEY.md
  - Write to LAUNCH-PACKAGE/documentation/NAMING-KEY.md
  - This is a DELIVERABLE document, not just a per-microskill output

STEP 8: WRITE PER-MICROSKILL OUTPUT
  - Produce per-microskill output file at designated path
```
