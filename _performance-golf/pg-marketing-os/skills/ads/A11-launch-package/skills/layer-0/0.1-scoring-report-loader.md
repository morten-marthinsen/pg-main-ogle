# 0.1: Scoring Report Loader

## Purpose

Load the A10 SCORING-REPORT.md and extract all data needed for launch packaging: which variants passed the scoring threshold and are approved for launch, their individual scores, any compliance flags raised during scoring, performance predictions, and recommended test priority ordering. This is the PRIMARY gating input -- only variants that passed A10 scoring get packaged for launch.

## Inputs

- A10 SCORING-REPORT.md (primary input)
- A10 score threshold definition (minimum score for launch approval)

## Process

```
STEP 1: LOCATE AND READ SCORING REPORT
  - Find A10 SCORING-REPORT.md in the project outputs
  - IF file does not exist: HALT -- cannot proceed without A10 scoring
  - IF file exists but is empty or malformed: HALT -- A10 must re-run

STEP 2: EXTRACT APPROVED VARIANTS
  - Parse scoring report for all variants with scores at or above the approval threshold
  - For EACH approved variant, extract:
    - variant_id: [string -- maps to A09 variant matrix]
    - overall_score: [float]
    - dimension_scores: [per-dimension breakdown if available]
    - compliance_flags: [list of any compliance concerns from A10]
    - performance_prediction: [predicted performance tier or metric]
    - test_priority_rank: [integer -- A10's recommended test order]
    - target_platforms: [list of platforms this variant is approved for]
    - hook_id: [from variant matrix]
    - body_id: [from variant matrix]
    - cta_id: [from variant matrix]
    - visual_id: [from variant matrix]

STEP 3: EXTRACT REJECTED VARIANTS (for reference)
  - Parse all variants below threshold
  - Record: variant_id, score, reason for rejection
  - These do NOT get packaged but may be referenced in backup planning

STEP 4: COMPILE PLATFORM DISTRIBUTION
  - Group approved variants by target platform
  - Count: how many approved variants per platform
  - IF any target platform has 0 approved variants: FLAG -- potential gap

STEP 5: VALIDATE EXTRACTION
  - Verify: total approved + total rejected = total scored
  - Verify: every approved variant has a variant_id that can be cross-referenced
  - Verify: at least 1 approved variant exists (otherwise nothing to package)
  - IF 0 approved variants: HALT -- return to A10 (nothing passed scoring)

STEP 6: WRITE OUTPUT
  - Produce per-microskill output file at designated path
```

- **Path:** `[project]/A11-launch-package/layer-0-outputs/0.1-scoring-report-loader.md`

## Quality Checks

1. Total approved + total rejected = total scored (no variants lost)
2. At least 1 approved variant exists (otherwise HALT)
3. Every approved variant has variant_id, score, hook/body/CTA/visual IDs, and target platforms
4. Platform distribution is complete -- every target platform appears
5. Compliance flags from A10 are captured (they feed into Layer 3 compliance review)
6. Output file meets minimum 2KB size

## Dependencies

- A10 SCORING-REPORT.md must exist and be complete
- No upstream A11 microskill dependencies (runs in parallel with 0.0.1)
