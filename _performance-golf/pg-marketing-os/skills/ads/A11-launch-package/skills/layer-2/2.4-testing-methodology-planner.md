# 2.4: Testing Methodology Planner

## Purpose

Define the A/B testing plan that determines WHAT gets tested, in WHAT order, with WHAT win criteria, and HOW the media buyer makes decisions. The testing priority order is critical: hooks are tested first (highest ROI per AD-ENGINE-CLAUDE.md), then bodies, then visuals, then CTAs. Without a testing plan, the media buyer either tests randomly (wasting budget) or doesn't test at all (missing optimization opportunities).

A11 RECOMMENDS testing methodology. The media buyer DECIDES.

## Inputs

- 2.1-campaign-hierarchy-architect.md output (campaign structure for test mapping)
- 0.1-scoring-report-loader.md output (A10 scores, test priority, performance predictions)
- 0.2-variant-matrix-loader.md output (variant components for test isolation)
- 2.3-budget-distribution-planner.md output (budget constraints for test sizing)
- Campaign Brief (business objectives, testing budget)
- AD-ENGINE-CLAUDE.md testing methodology reference (if available)

## Process

```
STEP 1: DEFINE TESTING PRIORITY ORDER
  Based on AD-ENGINE-CLAUDE.md testing hierarchy:

  PRIORITY 1 -- HOOK TESTS (Highest ROI)
    - What: Same body + same CTA + same visual, DIFFERENT hooks
    - Why: Hook determines if anyone watches/reads the ad at all.
      A great body with a weak hook = zero views. Hook testing has
      the highest leverage on overall performance.
    - Isolate: Hook variable only
    - Group variants that share the same body/CTA/visual but have different hooks
    - These go live FIRST in Phase 1

  PRIORITY 2 -- BODY TESTS
    - What: Same hook (winner from Priority 1) + same CTA + same visual, DIFFERENT bodies
    - Why: Once the winning hook is identified, test which body converts best
    - Isolate: Body variable only
    - These run in Phase 1 (parallel with hook tests if budget allows) or Phase 2

  PRIORITY 3 -- VISUAL TESTS
    - What: Same hook + same body + same CTA, DIFFERENT visual treatments
    - Why: Visual treatment affects engagement but less than hook or body
    - Isolate: Visual variable only
    - These typically run in Phase 2

  PRIORITY 4 -- CTA TESTS
    - What: Same hook + same body + same visual, DIFFERENT CTAs
    - Why: CTA changes have the smallest individual impact but can
      incrementally improve conversion rates
    - Isolate: CTA variable only
    - These typically run in Phase 3 (optimization phase)

STEP 2: MAP VARIANTS TO TEST GROUPS
  FOR EACH priority level:
    - Identify variant sets that isolate the target variable
    - Group variants into test groups:
      - Test Group A: Variants [X, Y, Z] -- testing hooks (same BD01, CT01, VIS-A)
      - Test Group B: Variants [A, B, C] -- testing bodies (same HK01, CT01, VIS-A)
      - etc.
    - Map test groups to A10 scoring predictions (which variants are expected to win?)
    - Map test groups to campaign/ad set positions from 2.1
    - IF variant matrix doesn't have clean isolation
      (every variant changes multiple variables):
      FLAG and recommend best available test structure

STEP 3: DEFINE WIN CRITERIA
  Statistical significance vs. practical significance:

  STATISTICAL SIGNIFICANCE (ideal):
    - Minimum sample size per variant: calculate based on expected conversion rate
    - Confidence level: 95% (standard)
    - Formula: N = (Z^2 * p * (1-p)) / E^2
      where Z=1.96, p=estimated conversion rate, E=margin of error
    - IF budget doesn't allow statistical significance: use practical significance

  PRACTICAL SIGNIFICANCE (minimum viable):
    - Minimum spend per variant: 3x AOV
    - Minimum time: 72 hours (algorithm learning period)
    - Minimum actions: 50 clicks (for CTR evaluation)
    - Winner determination: variant with best CPA/ROAS after minimum sample
    - Confidence note: results may not be statistically significant with small samples

  For this campaign:
    - Recommended approach: [statistical / practical] based on budget
    - Minimum sample size per variant: [calculated or budget-constrained]

STEP 4: DEFINE TEST DURATION AND CADENCE
  - Phase 1 test duration: 7 days (recommended)
  - Check-in cadence:
    - 24 hours: sanity check (ads delivering, no errors, basic metrics)
    - 48 hours: early signal check (hook rates, CTR trends)
    - 72 hours: first kill evaluation (kill clear losers per kill criteria from 2.3)
    - 7 days: final Phase 1 evaluation (winners identified, test groups resolved)
  - Phase 2 test duration: 7 days
  - Phase 3: ongoing weekly optimization cadence

STEP 5: MAP A10 PREDICTIONS TO TEST PRIORITY
  - A10 scored and ranked variants by predicted performance
  - Highest-ranked variants from A10 go into Phase 1 test groups
  - Next-highest become Phase 2 backup variants
  - Test plan should validate A10 predictions (did the #1 ranked variant actually win?)
  - Discrepancies between A10 prediction and real performance feed into A12

STEP 6: WRITE TEST-PLAN.md (Deliverable)
  - Write to LAUNCH-PACKAGE/documentation/TEST-PLAN.md

STEP 7: WRITE PER-MICROSKILL OUTPUT
  - Produce per-microskill output file at designated path
```
