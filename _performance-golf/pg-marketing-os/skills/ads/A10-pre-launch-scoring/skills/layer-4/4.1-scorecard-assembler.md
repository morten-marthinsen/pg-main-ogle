# 4.1: Scorecard Assembler

**Version:** 1.0
**Layer:** 4 (Output Packaging)
**Type:** Assembly / Formatting
**Model:** sonnet
**Dependencies:** All Layer 0-3 outputs
**Output:** PRE-LAUNCH-SCORECARD.md

---

## Purpose

Assemble PRE-LAUNCH-SCORECARD.md from all Layer 1-3 outputs. This is the PRIMARY DELIVERABLE of A10 — the document that a marketing manager picks up and knows exactly what to launch, in what order, with what budget. Minimum size: 30KB. Use chunked assembly if needed (3-5 write operations). This is an ASSEMBLY operation — it combines pre-validated artifacts, it does NOT generate new analysis.

## Process

### Step 1: Verify All Source Files Exist

```
FOR EACH input file listed above:
  VERIFY: file exists
  VERIFY: file size > 0

IF any source file missing:
  HALT: "Cannot assemble scorecard — missing [file]. Complete Layer [X] first."
```

### Step 2: Assemble Using Chunked Write Strategy

```
Due to LLM output limits (~30KB per response), use CHUNKED ASSEMBLY:

CHUNK 1: Metadata + Executive Summary + Composite Score Weights (Sections 1-2)
CHUNK 2: Individual Variant Scorecards (Section 3 — this is the largest section)
CHUNK 3: Force-Ranked List + Tier 1 + Tier 2 + Tier 3 (Sections 4-7)
CHUNK 4: Compliance Register + Fatigue Forecast (Sections 8-9)
CHUNK 5: Testing Strategy + Prediction Confidence + Human Approval (Sections 10-12)

EACH CHUNK is written as a separate file operation (append mode or staged writes)
VERIFY after each chunk: content was written correctly
```

### Step 3: Assemble All 12 Required Sections

```
The PRE-LAUNCH-SCORECARD.md must contain ALL 12 sections as defined in the
AGENT.md Output Schema. Here is the assembly source for each:

SECTION 1: EXECUTIVE SUMMARY
  Source: 0.4 (campaign numbers), 2.3 (tier distribution), 2.2 (top 3),
          1.4 (compliance risks), 3.1 (launch timeline)
  Content: Campaign overview, testing volume, tier distribution, top 3
    predicted performers, top 3 compliance risks, launch date, kill threshold

SECTION 2: COMPOSITE SCORE WEIGHTS
  Source: 2.1 (weights and confidence methodology)
  Content: The 5 weights (30/20/25/15/10), confidence multipliers (1.0/0.85/0.7),
    rationale connecting weights to campaign KPIs

SECTION 3: INDIVIDUAL VARIANT SCORECARDS
  Source: 1.1-1.5 (per-variant scores), 2.1 (composite), 2.2 (rank), 2.3 (tier)
  Content: FOR EACH variant (ALL variants, not just Tier 1):
    - Variant ID, hook text, platform, format, parent concept
    - 5-dimension score table with confidence ranges and key evidence
    - Composite score, adjusted composite, rank, tier
    - Predicted lifespan, compliance flags, strengths, risks, recommendation
  THIS IS THE LARGEST SECTION — may need 2 chunks for large variant sets

SECTION 4: FORCE-RANKED VARIANT LIST
  Source: 2.2 (forced ranking)
  Content: Complete ranked table from #1 to #N

SECTION 5: TIER 1 — LAUNCH FIRST
  Source: 2.3 (tier assignment)
  Content: Variant IDs, descriptions, rationale, combined profile,
    expected win rate (6.6-30%), platform distribution, hook type diversity

SECTION 6: TIER 2 — BACKUP
  Source: 2.3 (tier assignment)
  Content: Variant IDs, activation triggers, expected promotion timeline

SECTION 7: TIER 3 — REVISE OR CUT
  Source: 2.3 (tier assignment)
  Content: 3A variants with revision directions, 3B variants with cut rationale

SECTION 8: COMPLIANCE RISK REGISTER
  Source: 1.4 (compliance scores)
  Content: All variants scoring <= 5 with risk type, specific issue, fix

SECTION 9: CREATIVE FATIGUE FORECAST
  Source: 1.5 (fatigue scores)
  Content: Per-variant fatigue prediction, campaign-level timeline,
    hook refresh recommendations

SECTION 10: TESTING STRATEGY
  Source: 3.1 (sequence), 3.2 (budget), 3.3 (kill), 3.4 (scale)
  Content: Testing sequence (week by week), budget allocation table,
    kill criteria per variant, scale criteria, success metrics

SECTION 11: PREDICTION CONFIDENCE DISCLOSURE
  Source: 0.5 (benchmarks — A12 availability), 1.6 (scoring validation)
  Content: Overall confidence, factors increasing/decreasing confidence,
    calibration note about probabilistic nature, historical accuracy if A12 exists

SECTION 12: HUMAN APPROVAL
  Content: Checklist for human sign-off (tier list, budget, kill/scale criteria,
    timeline, compliance risks), approver fields, notes field
```

### Step 4: Verify Minimum Size

```
AFTER assembly complete:
  CALCULATE file size of PRE-LAUNCH-SCORECARD.md

  IF size < 30KB:
    FAIL: "Scorecard is only [X]KB — minimum is 30KB."
    Likely causes:
      - Section 3 (Individual Scorecards) is summarized instead of complete
      - Section 10 (Testing Strategy) lacks specific dollar amounts
      - Multiple sections are abbreviated

    FIX: Expand sections that are too thin. Section 3 should be the largest
    section — every variant gets a full scorecard, not a summary.

  IF size >= 30KB:
    PASS — size threshold met
```

### Step 5: Verify All Sections Populated

```
FOR EACH of the 12 required sections:
  VERIFY: section header exists in assembled file
  VERIFY: section contains substantive content (not just header)
  VERIFY: section contains specific data (not placeholder text)

IF any section missing or empty: FAIL — complete the section
```

2. File size >= 30KB

5. Section 10 includes specific dollar amounts (not vague)

7. Section 12 includes approval checklist for human sign-off
8. All data traces to Layer 0-3 outputs (assembly, not generation)
9. No placeholder text remains (no "[fill in]" or "[TBD]")
10. Chunked assembly was used if file exceeds single-output limits
