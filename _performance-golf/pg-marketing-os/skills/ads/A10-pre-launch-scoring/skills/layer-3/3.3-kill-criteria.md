# 3.3: Kill Criteria

**Version:** 1.0
**Layer:** 3 (Testing Strategy)
**Type:** Strategy / Rules Definition
**Model:** opus
**Persona:** PERSONA_PERFORMANCE_STRATEGIST
**Dependencies:** 0.4 (Campaign Brief), 2.3 (Tier Assignment), 1.1 (Scroll-Stop Scores)
**Output:** kill_criteria.md

---

## Purpose

Define SPECIFIC, pre-committed kill criteria for every Tier 1 variant. Kill criteria must be defined BEFORE launch — deciding when to kill DURING the campaign leads to emotional attachment and overspending on losers. Each kill criterion must be expressed as a specific metric threshold, a specific dollar amount, or a specific timeline. "Let it run longer" is not a valid response to underperformance when pre-defined kill triggers are hit.

## Process

### Step 1: Define Default Kill Rule

```
DEFAULT KILL RULE (applies to ALL variants):
  Kill Trigger: $[kill_threshold] spend with ZERO conversions
  kill_threshold = 3 × AOV

  EXAMPLE:
    AOV = $100 → Kill at $300 spend with 0 conversions
    AOV = $50  → Kill at $150 spend with 0 conversions
    AOV = $200 → Kill at $600 spend with 0 conversions

  THIS IS EXACT. Not "around" $300. Not "roughly" 3x. The specific dollar amount.

  RATIONALE: If you've spent 3x what a single customer is worth and gotten
  zero customers, the ad is not working. Continuing to spend is throwing
  money away.
```

### Step 2: Define Secondary Kill Signals

```
SECONDARY KILL SIGNALS (supplement the primary kill rule):

SIGNAL 1: Hook Rate Failure
  Threshold: Hook rate below 15% after 1,000+ impressions
  What it means: The ad is not stopping the scroll
  Timeline: Can be assessed within 24-48 hours at moderate spend
  Action: If hook rate < 15% after 1K impressions, BEGIN monitoring closely.
    If still < 15% after 3K impressions, KILL (scroll-stop failure confirmed)

SIGNAL 2: CTR Below Platform Benchmark
  Threshold: CTR below platform minimum after 3 days of data
  Platform benchmarks:
    Meta: 1.0% CTR minimum (feed ads)
    TikTok: 0.8% CTR minimum
    Google Search: 2.0% CTR minimum
    YouTube: 0.5% CTR minimum (view-based)
  Action: If CTR < benchmark after 3 days, KILL

SIGNAL 3: CPA Above Threshold
  Threshold: CPA > 2× target CPA after 100+ clicks
  What it means: Getting clicks but not converting at acceptable cost
  Timeline: Requires sufficient click volume (100+)
  Action: If CPA > 2× target after 100 clicks, KILL
  NOTE: Only applicable if target CPA is defined in campaign brief

SIGNAL 4: Engagement Rate Failure
  Threshold: Video completion rate < 10% (for video ads) OR average watch time
    < 3 seconds (for 15s+ ads)
  What it means: Stopping the scroll but not holding attention
  Action: If engagement < threshold after 3 days, KILL
```

### Step 3: Define Kill Timeline Constraints

```
MINIMUM DATA WINDOW:
  No kill decisions before 72 HOURS of live data
  RATIONALE: Platform algorithms need time to optimize delivery.
    First 24-48 hours data is unreliable (learning phase).
    72 hours provides minimum sample for meaningful signal.

MAXIMUM DECISION WINDOW:
  Maximum 7 days before kill decision on any variant
  RATIONALE: Waiting longer delays Tier 2 activation and wastes budget.
    If a variant hasn't shown signal in 7 days, it's not going to.

DECISION WINDOW:
  72 hours (minimum) → 7 days (maximum)
  Day 3: Earliest possible kill (only for extreme failures like 0 hook rate)
  Day 5: Standard first kill checkpoint
  Day 7: Final kill decision — every variant must have a verdict by Day 7
```

### Step 4: Build Per-Variant Kill Trigger Table

```
FOR EACH Tier 1 variant:
  DOCUMENT the specific kill trigger in advance:

| Variant ID | Platform | Kill Trigger | Kill $ | Kill Timeline | Signal Priority |
|------------|----------|-------------|--------|--------------|-----------------|
| V-007      | Meta     | $300 spend, 0 CVR | $300 | Day 5-7 | Primary: spend |
| V-003      | TikTok   | $300 spend, 0 CVR OR hook rate <15% after 3K imp | $300 | Day 5-7 | Primary: spend, Secondary: hook rate |
| V-005      | Meta     | $300 spend, 0 CVR OR CTR <1.0% after 3 days | $300 | Day 3-7 | Primary: spend, Secondary: CTR |

FOR EACH variant, also note:
  - predicted_kill_day: estimated day when kill threshold is reached (from 3.2 spend-to-kill)
  - reallocation_target: where budget goes if killed (from 3.2)
```

### Step 5: Define Kill Decision Process

```
KILL DECISION PROCESS:

1. CHECK: Has minimum data window passed (72 hours)?
   IF NO: Do NOT make kill decision yet. Wait.

2. CHECK: Has any kill trigger been hit?
   IF YES: Proceed to kill confirmation.
   IF NO: Continue monitoring.

3. KILL CONFIRMATION:
   - Document which kill trigger was hit
   - Document the specific metrics at time of kill
   - Execute reallocation (per 3.2 protocol)
   - Activate next Tier 2 variant (per 3.1 sequence)
   - Record kill in performance log for A12

4. POST-KILL:
   - Budget reallocation completes within 24 hours
   - Tier 2 variant enters platform review queue
   - No "second chances" — a killed variant stays dead

FORBIDDEN:
  - "Let it run a few more days" after kill trigger is hit
  - Increasing spend on a variant that hit the kill threshold
  - Resurrecting a killed variant without complete revision (A07)
  - Emotional attachment overriding pre-defined kill rules
```

## Quality Checks

1. Default kill threshold correctly calculated (exactly 3x AOV)
2. Kill threshold expressed as exact dollar amount (not "approximately")
3. All secondary kill signals have specific metric thresholds
4. Platform-specific CTR benchmarks included for each platform
5. Minimum 72-hour data window enforced (no premature kills)
6. Maximum 7-day decision window enforced (no indefinite runs)
7. Per-variant kill triggers documented for every Tier 1 variant
8. Reallocation targets specified for every Tier 1 variant
9. Kill decision process is specific and sequential
10. Forbidden behaviors explicitly listed
