# 3.4: Scale Criteria

**Version:** 1.0
**Layer:** 3 (Testing Strategy)
**Type:** Strategy / Rules Definition
**Model:** opus
**Persona:** PERSONA_PERFORMANCE_STRATEGIST
**Dependencies:** 0.4 (Campaign Brief), 2.3 (Tier Assignment), 3.2 (Budget Allocation)
**Output:** scale_criteria.md

---

## Purpose

Define SPECIFIC scale criteria for winning variants — the triggers that move a variant from "testing" to "scaling" status. Scale criteria must include performance triggers, scaling protocol, watch metrics, and budget sources. Scaling too aggressively or too early wastes budget. Scaling too slowly leaves money on the table. The criteria must be pre-defined and connected to specific metrics.

## Process

### Step 1: Define Scale Trigger

```
SCALE TRIGGER (a variant must meet ALL of the following):

  1. PERFORMANCE: Achieves target CPA (or better)
     IF target_cpa is defined: CPA <= target_cpa
     IF target_cpa NOT defined: CPA <= AOV × 0.33 (conservative default)
     RATIONALE: Profitable at 3:1 ROAS minimum

  2. VOLUME: 3+ conversions
     RATIONALE: A single conversion could be anomaly. 3+ shows pattern.
     NOTE: For high-AOV products ($500+), 2 conversions may be sufficient.

  3. DURATION: 5+ days of data
     RATIONALE: Performance must be sustained, not a single-day spike.
     Platform algorithms need time to optimize delivery.
     Short-term performance often regresses to mean.

  4. STABILITY: CPA variance within acceptable range
     CPA must not swing more than 50% day-to-day over the 5-day window.
     Wild CPA swings suggest platform is still learning, not stable performance.

ALL 4 conditions must be met simultaneously before scaling begins.
```

### Step 2: Define Scaling Protocol

```
SCALING PROTOCOL:

STEP 1: VERIFY all 4 scale trigger conditions are met
  Document: current CPA, conversion count, days live, CPA variance

STEP 2: INCREASE daily spend by 20-30%
  Conservative: +20% (recommended for first scale event)
  Aggressive: +30% (only after 2+ successful scale events)

  EXAMPLE:
    Current daily spend: $42
    First scale: $42 × 1.20 = $50.40 (round to $50)
    Second scale: $50 × 1.25 = $62.50 (round to $63)
    Third scale: $63 × 1.30 = $81.90 (round to $82)

STEP 3: WAIT minimum 48 hours before next scale event
  RATIONALE: Platform algorithms need time to optimize at new spend level.
  Scaling too frequently causes delivery instability.

STEP 4: MONITOR watch metrics (see Step 3 below)

SCALING BUDGET SOURCE:
  1. Reserve budget (20-30% of weekly spend)
  2. Killed variant reallocation
  3. Tier 2 variant budget that hasn't been activated yet

  NOTE: Do NOT reduce other performing variants to fund scaling.
  Winners should not cannibalize each other's budget.
```

### Step 3: Define Watch Metrics During Scaling

```
AFTER EACH SCALE EVENT, MONITOR:

METRIC 1: CPA Stability
  Threshold: CPA must stay within 120% of pre-scale CPA
  CHECK after 48 hours post-scale

  IF CPA rises to 120-150% of pre-scale:
    → PAUSE scaling. Hold at current spend level.
    → Re-evaluate after 72 hours.
    → If CPA normalizes: resume scaling.
    → If CPA stays elevated: reduce to previous spend level.

  IF CPA rises > 150% of pre-scale:
    → IMMEDIATELY reduce to previous spend level.
    → Wait 5 days before attempting another scale.
    → This variant may have a spend ceiling.

METRIC 2: Hook Rate Stability
  Threshold: Hook rate must not drop more than 10% from pre-scale baseline
  CHECK daily post-scale

  IF hook rate drops 10-20%:
    → FATIGUE WARNING. This may be early fatigue signal.
    → Continue monitoring but begin planning hook refresh.

  IF hook rate drops > 20%:
    → FATIGUE CONFIRMED. Creative fatigue has set in.
    → Pause scaling. Begin hook refresh cycle.
    → This variant's scaling ceiling has been reached.

METRIC 3: Frequency
  Threshold: Average frequency should stay below 3.0 (Meta) or 2.5 (TikTok)
  CHECK weekly

  IF frequency exceeds threshold:
    → Audience saturation. Expand audience or reduce spend.
    → High frequency accelerates fatigue.

METRIC 4: Conversion Rate Stability
  Threshold: CVR should not drop more than 25% from pre-scale
  CHECK after 72 hours post-scale

  IF CVR drops > 25%:
    → The scale has degraded conversion quality.
    → Reduce to previous spend level.
    → Investigate: audience expansion may be bringing lower-quality traffic.
```

### Step 4: Build Per-Variant Scale Trigger Table

```
FOR EACH Tier 1 variant:
  DOCUMENT the specific scale trigger in advance:

| Variant ID | Scale CPA Target | Min Conversions | Min Days | Current Daily | Max Daily (3 scales) |
|------------|-----------------|-----------------|----------|---------------|---------------------|
| V-007      | <= $[target]     | 3               | 5        | $[X]          | $[X × 1.2³]         |
| V-003      | <= $[target]     | 3               | 5        | $[X]          | $[X × 1.2³]         |

FOR EACH variant, also calculate:
  scale_ceiling: maximum daily spend before budget is exhausted
    = (active_budget - other_variants_spend - minimum_reserve)
  theoretical_max_scales: how many 20% increases before hitting ceiling
```

### Step 5: Define Scale Decision Process

```
SCALE DECISION PROCESS:

1. CHECK: Have all 4 scale trigger conditions been met?
   IF NO: Continue monitoring. Do not scale.

2. CHECK: Has minimum 48 hours passed since last scale event for this variant?
   IF NO: Wait. Scaling too frequently destabilizes performance.

3. SCALE:
   - Increase daily spend by 20% (first scale) or 25-30% (subsequent)
   - Document: new daily spend, CPA at time of scale, conversion count
   - Set 48-hour monitoring window

4. MONITOR:
   - Check CPA after 48 hours (must stay within 120% of pre-scale)
   - Check hook rate daily (must not drop > 10%)
   - Check frequency weekly

5. CONTINUE OR PAUSE:
   - If all watch metrics pass: schedule next scale event (earliest: 48 hours)
   - If any watch metric fails: pause scaling, follow metric-specific protocol

SCALE CEILING:
  Stop scaling when:
    - Daily spend reaches [X]% of total active budget
    - Watch metrics degrade despite stable performance
    - Frequency exceeds platform threshold
    - Creative fatigue signals appear (hook rate decline > 20%)
```

## Quality Checks

1. Scale trigger has all 4 conditions explicitly defined
2. CPA target connected to actual campaign numbers (target CPA or AOV-based default)
3. Scaling increment is specific (20-30%, not vague)
4. 48-hour minimum between scale events enforced
5. All 4 watch metrics have specific thresholds and breach protocols
6. Per-variant scale trigger table complete for all Tier 1 variants
7. Budget source for scaling documented (reserve + killed variant reallocation)
8. Scale ceiling calculated for each variant
9. Forbidden behaviors listed (no cannibalizing other winners)
10. Decision process is specific and sequential
