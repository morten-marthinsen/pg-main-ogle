# 1.4: Compliance Scorer

**Version:** 1.0
**Layer:** 1 (Individual Variant Scoring)
**Type:** Judgment / Audit
**Model:** opus
**Persona:** PERSONA_COMPLIANCE_AUDITOR
**Dependencies:** 0.1 (Variant Matrix), 0.2 (Competitive Intelligence), 0.4 (Campaign Brief)
**Output:** compliance_scores.md

---

## Purpose

Score EVERY variant on Dimension 4: Compliance Risk. This predicts the probability that a variant will be REJECTED, RESTRICTED, or FLAGGED by platform ad review systems (Meta, TikTok, Google). A variant that gets rejected has an effective performance score of zero, regardless of creative quality. Compliance scoring is INVERTED — higher score = LOWER risk. Any variant scoring 3 or below is AUTOMATICALLY flagged for revision before launch, regardless of other dimension scores. Compliance is a GATE, not a nice-to-have.

## Process

### Step 1: Establish Compliance Framework

```
LOAD compliance baseline from 0.4:
  - vertical: string
  - compliance_risk_level: LOW / MODERATE / HIGH

LOAD vertical-specific compliance constraints:
  HEALTH vertical:
    FORBIDDEN: disease cure claims, before/after weight loss imagery (Meta),
    specific medical outcomes, "clinically proven" without published study,
    testimonials implying medical results
    RESTRICTED: supplement benefit claims (must be qualified),
    transformation imagery (TikTok sensitive)

  FINANCE vertical:
    FORBIDDEN: guaranteed returns, specific income promises,
    "risk-free" investment claims, unrealistic earnings screenshots
    RESTRICTED: income claims (must include disclaimers),
    trading results (must show typical results)

  PERSONAL DEVELOPMENT vertical:
    FORBIDDEN: income guarantees, unrealistic transformation claims
    RESTRICTED: success claims (must be qualified), before/after imagery

  GOLF/SPORTS vertical:
    LOWEST risk — few specific restrictions
    RESTRICTED: avoid unsubstantiated performance claims

  TECHNOLOGY vertical:
    FORBIDDEN: false feature claims, fake reviews
    RESTRICTED: comparison claims (must be substantiated)

LOAD platform-specific policies:
  META:
    - Before/after imagery restricted for weight loss
    - Personal attributes policy (don't imply user has a condition)
    - Misleading claims policy
    - Social issues, elections, politics restrictions
  TIKTOK:
    - Health claims restricted
    - Financial services advertising requires disclaimers
    - Content authenticity requirements (UGC must be genuine)
  GOOGLE:
    - Healthcare advertising policies (certification required for some)
    - Financial services restrictions
    - Gambling and games policies
```

### Step 2: Audit Each Variant

```
FOR EACH variant:

  CHECK 1: CLAIM ANALYSIS
    SCAN hook_text + body_text + cta_text for:
    - Disease claims ("cure", "treat", "heal", "reverse [disease]")
    - Income claims ("earn $X", "make $X/month", "guaranteed income")
    - Unrealistic transformation claims ("lose 30lbs in a week")
    - Absolute claims ("100% effective", "works for everyone", "guaranteed")
    - Unsubstantiated scientific claims ("clinically proven" without study)

    FOR EACH claim found:
      - CLASSIFY: forbidden / restricted / acceptable
      - NOTE: specific text containing the claim
      - RECOMMEND: revision language if restricted

  CHECK 2: VISUAL COMPLIANCE
    SCAN visual_treatment for:
    - Before/after imagery (Meta-restricted for weight loss)
    - Shocking/graphic content
    - Misleading visual representations
    - Fake social proof (fabricated screenshots, fake comments)

  CHECK 3: PLATFORM-SPECIFIC POLICY
    FOR the variant's target platform:
      - CHECK against platform-specific policies
      - IDENTIFY any content that triggers special review categories
      - NOTE if the variant requires platform-specific disclaimers

  CHECK 4: COMPETITIVE TOLERANCE PROXY
    FROM 0.2 competitive context:
      - ARE similar claims being successfully run by competitors?
      - IF YES: platform is currently tolerating this claim type
        (DOES NOT guarantee approval, but reduces risk)
      - IF NO: claim type may be novel for reviewers — higher risk

  CHECK 5: A03 COMPLIANCE NOTES
    IF A03 Format Strategy identified compliance constraints:
      - VERIFY variant follows those constraints
      - NOTE any violations
```

### Step 3: Score Each Variant

```
SCORING SCALE (INVERTED — higher = LOWER risk):

9-10: MINIMAL compliance risk — no claims, no restricted content, fully compliant
      No claims that could trigger review. Standard ad content.

8:    LOW risk — standard claims within platform guidelines
      Minor benefit claims that are well-supported. Standard for vertical.

6-7:  MODERATE risk — borderline claims that may trigger review but likely pass
      Claims that are common in the vertical but could flag automated review.
      Competitors successfully running similar claims.

4-5:  ELEVATED risk — claims that frequently trigger rejection in this vertical
      Health benefit claims without qualification. Income claims without
      disclaimers. Before/after imagery on restricted platforms.

2-3:  HIGH risk — variant contains restricted content or unsubstantiatable claims
      Multiple restricted claims. Visual compliance issues. Platform-specific
      policy violations.
      *** AUTOMATIC REVISION FLAG — cannot launch as-is ***

1:    CRITICAL risk — near-certain rejection
      Disease cure claims, guaranteed returns, extreme transformations.
      *** AUTOMATIC REVISION FLAG — may need complete rewrite ***

FOR EACH variant scoring 5 or below:
  PRODUCE specific recommended fixes:
    - IDENTIFY which claims/elements cause the risk
    - SUGGEST specific revision language
    - NOTE if the issue is fixable (claim rewrite) or fundamental (concept problem)
```

### Step 4: Generate Compliance Flags

```
FOR EACH variant with score <= 3:
  CREATE compliance_flag:
    variant_id: string
    compliance_score: float
    risk_type: string  # e.g., "unsubstantiated health claim"
    specific_issue: string  # exact text causing the issue
    platform_affected: string
    recommended_fix: string
    severity: enum[CRITICAL, HIGH]
    launch_blocked: true  # CANNOT launch until resolved

FOR EACH variant with score 4-5:
  CREATE compliance_warning:
    variant_id: string
    compliance_score: float
    risk_type: string
    specific_issue: string
    recommended_fix: string
    severity: ELEVATED
    launch_blocked: false  # can launch but human should review
```

## Quality Checks

1. ALL variants scored (count must match total from 0.1)
2. Vertical compliance constraints loaded and applied
3. Platform-specific policies checked for each variant's target platform
4. Every claim in every variant identified and classified
5. Variants scoring <= 3 have specific recommended fixes
6. Compliance flags list is complete (all score <= 3 variants)
7. Compliance warnings list is complete (all score 4-5 variants)
8. No variant with forbidden claims scores above 3
9. Competitive tolerance proxy checked against A01 data
10. "Compliance isn't an issue for this vertical" rationalization NOT present
