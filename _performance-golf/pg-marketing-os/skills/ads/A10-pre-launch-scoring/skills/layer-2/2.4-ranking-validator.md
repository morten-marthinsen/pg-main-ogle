# 2.4: Ranking Validator

**Version:** 1.0
**Layer:** 2 (Comparative Analysis & Tier Assignment)
**Type:** Validation
**Model:** opus
**Dependencies:** 2.1 (Composite Scores), 2.2 (Forced Ranking), 2.3 (Tier Assignment)
**Output:** ranking_validation.md

---

## Purpose

Validate the complete ranking and tier assignment for logical consistency, threshold compliance, budget alignment, and distribution health. This is the quality gate before Layer 3 (Testing Strategy). If the ranking has errors, inconsistencies, or pathological distributions, the testing strategy built on top will be flawed.

## Process

### Step 1: Completeness Check

```
VERIFY:
  a. All variants are ranked (no missing from forced ranking)
  b. All variants are assigned to exactly one tier
  c. Tier counts sum to total variant count

IF any variant missing from ranking: FAIL
IF any variant missing from tier assignment: FAIL
IF tier counts don't sum correctly: FAIL
```

### Step 2: Tie Resolution Check

```
VERIFY: No ties exist at tier boundaries
  - Check the LAST Tier 1 variant and FIRST Tier 2 variant: different composites?
  - Check the LAST Tier 2 variant and FIRST Tier 3 variant: different composites?

IF ties at boundaries: FAIL — tiebreakers not applied correctly
```

### Step 3: Threshold Consistency Check

```
VERIFY tier assignments are consistent with composite scores:

FOR EACH Tier 1 variant:
  - adjusted_composite >= 7.0? (must be true)
  - compliance_score >= 6? (must be true)

FOR EACH Tier 2 variant:
  - adjusted_composite >= 5.5? (must be true)
  - compliance_score >= 5? (must be true)
  - NOT in Tier 1 for legitimate reason? (budget cap OR doesn't meet Tier 1 threshold)

CHECK for inversions:
  - No variant with score 8.0 in Tier 2 while a variant with 6.5 is in Tier 1
    (higher scores must rank higher)
  - No variant with score 6.0 in Tier 3 while a variant with 5.5 is in Tier 2
    (unless compliance flag is the differentiator)

IF any inversion found: FAIL — fix tier assignment logic
```

### Step 4: Budget Alignment Check

```
VERIFY:
  a. Tier 1 count <= min_creatives_per_week
  b. If Tier 1 count < min_creatives_per_week AND Tier 2 has qualifying variants:
     note this as "room for additional Tier 1" (not an error, but worth documenting)
  c. If Tier 1 count == 0: FLAG for human attention

RECORD:
  tier_1_count: integer
  min_creatives_per_week: integer
  budget_utilization: tier_1_count / min_creatives_per_week
```

### Step 5: Compliance Gate Check

```
VERIFY:
  a. ZERO compliance-flagged variants (compliance <= 3) in Tier 1
  b. ZERO compliance-flagged variants (compliance <= 3) in Tier 2
  c. All compliance-flagged variants are in Tier 3

IF any compliance-flagged variant in Tier 1 or 2: FAIL — CRITICAL
This is a structural violation that CANNOT be overridden.
```

### Step 6: Distribution Health Check

```
CALCULATE tier percentages:
  tier_1_pct = tier_1_count / total_variants * 100
  tier_2_pct = tier_2_count / total_variants * 100
  tier_3_pct = (tier_3a_count + tier_3b_count) / total_variants * 100

ASSESS distribution health:
  HEALTHY: Tier 1 = 20-40%, Tier 2 = 30-40%, Tier 3 = 20-40%
  ACCEPTABLE: Some skew but meaningful differentiation across tiers

  RED FLAGS:
    - Tier 1 > 60% (scoring not critical enough)
    - Tier 3 > 70% (upstream skills may need re-execution)
    - Tier 1 = 0% (no variants ready — human decision needed)
    - All variants in one tier (scoring completely undifferentiated)

IF any red flag: FAIL — re-examine scoring differentiation
```

### Step 7: Tier 3 Documentation Check

```
FOR EACH Tier 3A variant:
  VERIFY: specific_reason is populated (not empty, not generic)
  VERIFY: revision_direction is populated (not empty, not generic)
  VERIFY: priority is assigned (HIGH / MEDIUM / LOW)

FOR EACH Tier 3B variant:
  VERIFY: specific_reason is populated (not empty, not generic)

"Generally weak" is NOT an acceptable reason.
"Compliance score 2 (disease claims), scroll-stop 3 (oversaturated)" IS acceptable.

IF any Tier 3 variant lacks documentation: FAIL
```

### Step 8: Produce Tier Distribution Check

```
BUILD the MANDATORY TIER DISTRIBUTION CHECK table:

┌─────────────────────────────────────────────────────────────────────────────┐
│  TIER DISTRIBUTION CHECK - [timestamp]                                       │
│                                                                              │
│  ┌────────────────────────────────────────────────────────────────────────┐ │
│  │ TIER       │ COUNT │ % OF TOTAL │ SCORE RANGE │ STATUS                 │ │
│  ├────────────────────────────────────────────────────────────────────────┤ │
│  │ Tier 1     │ [X]   │ [X]%       │ [X]-[X]     │ <= min creatives/wk?  │ │
│  │ Tier 2     │ [X]   │ [X]%       │ [X]-[X]     │ Backup ready?         │ │
│  │ Tier 3A    │ [X]   │ [X]%       │ [X]-[X]     │ Revision flagged?     │ │
│  │ Tier 3B    │ [X]   │ [X]%       │ [X]-[X]     │ Cut documented?       │ │
│  └────────────────────────────────────────────────────────────────────────┘ │
│                                                                              │
│  COMPLIANCE FLAGS IN TIER 1: [count — MUST be 0]                            │
│  TIER 1 COUNT VS MIN CREATIVES/WEEK: [X] vs [Y]                            │
│  ALL VARIANTS RANKED: [Y/N]                                                 │
│  DIFFERENTIATION: Are scores meaningfully spread? [Y/N]                     │
│                                                                              │
│  OVERALL: [PASS - proceed to Layer 3] or [FAIL - re-evaluate]               │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Quality Checks

1. All 8 validation steps executed
2. Completeness verified (all variants ranked and tiered)
3. No ties at tier boundaries
4. No threshold inversions
5. Tier 1 within budget cap
6. Zero compliance flags in Tier 1 or Tier 2
7. Distribution health assessed
8. All Tier 3 variants have specific (not generic) documentation
9. Overall verdict is PASS or FAIL only
10. If FAIL: specific items_to_fix listed
