# 1.6 Propagation Verification

## Purpose
VERIFY that all approved updates were actually written to their target files. Check file modification timestamps. If any update wasn't propagated → flag as CRITICAL.

## Assigned Model
`claude-opus-4` (Layer 1 intelligence)

## Input
- 1.2-teaching-updates.yaml (teaching file updates)
- 1.3-specimen-promotions.json (specimen additions)
- 1.5-gate-threshold-review.json (gate adjustments)
- File system access (to check timestamps and content)

## Process

### MUST
1. **For each teaching update from 1.2:**
   - Check target file exists at path
   - Verify file modification timestamp > update timestamp
   - Spot-check content: search for specific YAML key/value added
   - Verify file still parses correctly (no corruption)
   - Confirm version number incremented
2. **For each specimen promotion from 1.3:**
   - Check specimen file exists in library
   - Verify specimen index updated with new entry
   - Confirm performance data included in specimen file
3. **For each gate threshold adjustment from 1.5:**
   - Check gate config file modified
   - Verify new threshold value present
   - Confirm old threshold overwritten (not duplicated)
4. **Handle propagation failures:**
   - If file timestamp didn't change → RETRY ONCE
   - If retry fails → Mark propagation_status: FAILED
   - Log failure details (file path, error message)
   - Escalate to human (CRITICAL)
5. **Output verification results:**
   - Binary per update: COMPLETE or FAILED
   - Overall propagation status: COMPLETE (all passed) or FAILED (any failed)

### NEVER
- Skip timestamp verification
- Assume updates worked without checking
- Mark "PENDING" (binary only: COMPLETE or FAILED)
- Proceed if any propagation failed
- Skip retry on first failure

## Quality Gates
1. All updates verified
2. File timestamps checked
3. Content spot-checked
4. Failures logged
5. Retries attempted
6. Escalation triggered if needed

## Handoff
