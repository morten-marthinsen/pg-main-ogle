# Layer 1.1 — Timing Planner

## Purpose

Set the send schedule and frequency for the assembled email sequence based on the campaign type. Assign each email a specific day, time slot, and send order. The timing plan drives volume escalation for launch sequences and ensures appropriate pacing for all campaign types.

---

## Input Requirements

```yaml
from_layer_0:
  blueprint_context:
    campaign_type: string  # PL, AL, AR, DR, CP, BP, DU
    total_emails: integer
    duration_days: integer
    email_plan:
      - position: integer
        day: integer  # blueprint's day assignment
        urgency_level: integer
    urgency_plan:
      mechanism: string
      escalation: [object]
```

---

## Execution Protocol

### Step 1: Select Timing Template

Based on `campaign_type`, load the appropriate timing template:

```
PL (Product Launch):
  Day -1: 1 email (pre-launch warning)      → send_time: evening
  Day 1:  1-2 emails (launch open)           → send_time: morning + afternoon
  Day 2:  2 emails (proof/education)          → send_time: morning + evening
  Day 3:  2 emails (objections/social proof)  → send_time: morning + evening
  Day 4:  2-3 emails (urgency begins)         → send_time: morning + afternoon + evening
  Day 5:  5-8 emails (close day blitz)        → send_time: early_morning through late_evening

AL (Affiliate Launch):
  Day 1: 1 email → morning
  Day 2: 1 email → morning
  Day 3: 1-2 emails → morning + evening
  Day 4: 1-2 emails → morning + evening
  Day 5: 1-2 emails → morning + evening (last call)

AR (Autoresponder):
  Day 1-30: 1 email/day → morning
  No volume escalation
  All emails evergreen (no date references)

DR (Daily Relationship):
  Every day: 1 email → morning
  No volume escalation
  Ongoing — no end date

CP (Content Series):
  Day 1-7: 1 email/day → morning
  No volume escalation
  All content-forward

BP (Blatant Pitch):
  Day 1: 1 email → morning
  Single email — no timing logic needed

DU (Deadline Urgency):
  Day 1: 1 email → morning
  Single email — no timing logic needed
```

### Step 2: Map Emails to Timing Slots

Assign each email position to a specific day and time slot using the blueprint's day assignments as the primary guide and the timing template as the structural framework:

```
FOR each email in blueprint.email_plan:
  timing_assignment = {
    position: email.position,
    day: email.day,  # from blueprint
    send_time: determine_send_time(email.position, email.day, campaign_type),
    send_order: calculate_send_order(email.position, email.day)
  }
```

### Step 3: Apply Volume Escalation (Launch Sequences Only)

For PL and AL campaign types, verify that the close day has appropriate volume:

```
IF campaign_type == "PL":
  close_day = max(email.day for email in email_plan)
  close_day_emails = count(emails where day == close_day)
  IF close_day_emails < 5:
    flag_warning: "Close day has only {close_day_emails} emails. Launch best practice is 5+."

IF campaign_type == "AL":
  close_day = max(email.day for email in email_plan)
  close_day_emails = count(emails where day == close_day)
  IF close_day_emails < 1:
    flag_warning: "Close day has no emails. At minimum 1 last-call email needed."
```

### Step 4: Validate No Timing Conflicts

```
FOR each day in timing_plan:
  emails_this_day = [emails where day == this_day]
  IF len(emails_this_day) > 8:
    flag_error: "Day {day} has {count} emails. Maximum 8 per day."

  send_times = [email.send_time for email in emails_this_day]
  IF duplicates in send_times AND campaign_type NOT IN [PL, DU]:
    flag_warning: "Multiple emails at same send_time on day {day}."
```

### Step 5: Calculate Intervals

For ongoing/daily sequences, calculate the interval between emails:

```
IF campaign_type IN [AR, DR]:
  interval = "24 hours"
IF campaign_type IN [CP]:
  interval = "24 hours"
IF campaign_type IN [PL, AL]:
  interval = "varies by day — see timing plan"
```

## Output

```yaml
timing_plan:
  campaign_type: string
  total_emails: integer
  total_days: integer
  template_used: string

  schedule:
    - day: 1
      emails_count: integer
      email_assignments:
        - position: integer
          send_time: string  # early_morning, morning, afternoon, evening, late_evening
          send_order: integer  # 1st email of day, 2nd, etc.

    - day: 2
      emails_count: integer
      email_assignments:
        - position: integer
          send_time: string
          send_order: integer
    # ... all days

  volume_escalation:
    applies: [true|false]
    close_day: integer
    close_day_count: integer
    escalation_pattern: string  # description

  timing_validation:
    no_timing_conflicts: [true|false]
    max_emails_per_day: integer
    warnings: []

  interval:
    default: string  # "24 hours" or "varies"
    notes: string
```

---

## Failure Modes

| Failure | Remediation |
|---------|-------------|
| Campaign type not recognized | Default to DR (daily) timing. Log warning. |
| Blueprint day assignments conflict with timing template | Blueprint is authoritative. Use blueprint days. Log deviation from template. |
| Close day has too few emails (PL) | Log warning — not a HALT, but flag for human review. |
| More than 8 emails assigned to one day | HALT — Maximum 8 emails per day. Flag to human for blueprint adjustment. |
| No timing template for campaign type | Use blueprint day assignments directly. Log that no template was applied. |
