# Layer 0.2 — Subject Line Loader

## Purpose

Load ALL selected subject lines from E2 (Subject Line Engine) output into working memory. Verify that every email position has at least one selected subject line. Extract the subject line text, formula category, and selection metadata.

---

## Input Requirements

```yaml
from_blueprint:
  total_emails: integer
  emails:
    - position: integer
      body_type: string

from_e2_output:
  subject_lines_directory: "[project]/E2-subject-line-engine/outputs/"
  expected_files:
    - "email-[position]-subject-lines.md"  # or consolidated SL output file
```

---

## Execution Protocol

### Step 1: Identify Expected Subject Lines

For each email position in the blueprint, expect at least one selected subject line from E2.

```
FOR each email in blueprint.emails:
  expected_sls.append({
    position: email.position,
    body_type: email.body_type
  })
```

### Step 2: Load Subject Lines

Load from E2 output — either per-email SL files or a consolidated subject line output:

1. READ E2 output file(s)
2. FOR each email position, EXTRACT:
   - `selected_subject_line` — the human-selected or top-ranked SL
   - `formula_category` — which of the 18 SL formula categories was used
   - `word_count` — word count of the subject line
   - `emotional_trigger` — primary emotional trigger
   - `alternatives` — 2-3 backup SLs (if available)
3. VERIFY at least one selected SL exists per email position

### Step 3: Check Subject Line Diversity

Before loading is complete, do a preliminary diversity check:

```
formula_categories_used = set()
FOR each sl in loaded_subject_lines:
  formula_categories_used.add(sl.formula_category)

sl_diversity = len(formula_categories_used)
# C5 criterion requires 3+ different formula categories
preliminary_diversity_check = sl_diversity >= 3
```

### Step 4: Track Loading Status

For each position, record:
- Subject line found: Y/N
- Formula category extracted: Y/N
- Alternatives available: Y/N

### Step 5: Build Subject Lines Index

Merge all loaded subject lines into a single `sl_index` object, ordered by position.

---

## Output

```yaml
sl_index:
  total_expected: [integer]
  total_loaded: [integer]
  loading_complete: [true|false]
  missing_positions: [integer]  # empty if complete

  subject_lines:
    - position: 1
      selected_subject_line: "[text]"
      formula_category: "[category name]"
      word_count: [integer]
      emotional_trigger: "[trigger]"
      alternatives:
        - text: "[alt SL 1]"
          formula_category: "[category]"
        - text: "[alt SL 2]"
          formula_category: "[category]"
      load_status: "SUCCESS"

    - position: 2
      # ... same structure

  diversity_check:
    unique_formula_categories: [integer]
    categories_used: [list of category names]
    preliminary_c5_passes: [true|false]  # >= 3 different categories

  warnings: []  # any non-blocking issues
```

---

## Failure Modes

| Failure | Remediation |
|---------|-------------|
| No subject line found for a position | Log missing position. Set loading_complete = false. Gate 0 will FAIL. |
| Subject line file found but empty | Log error. Mark position as not loaded. Gate 0 will FAIL. |
| Formula category not specified | Log as warning. Load SL anyway — validator can assess later. |
| Only 1-2 formula categories used across all SLs | Log as warning. C5 will flag this in Layer 3 validation. |
| Alternative SLs not available | Non-blocking. Note that only primary SL is available per position. |
