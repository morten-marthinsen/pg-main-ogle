# Layer 2.3 — Body Type Assignment

## Purpose

Assign a specific body type to each email in the sequence. Map available assets (from 1.3) to email positions (from 2.2) using strategic placement rules. The highest-richness assets go to the most important positions. Body type assignments determine the structural DNA of each email that E1 will generate.

---

## Input Requirements

```yaml
from_layer_1:
  asset_inventory:
    assets: [object]  # all tagged assets with body_type_fit + content_richness
    per_type_summary: object  # counts, avg richness, weights per type
    body_type_gaps: [string]

from_layer_2:
  sequence_blueprint:
    emails: [object]  # sequence with positions, functions, content_focus
    total_emails: integer

  confirmed_campaign:
    campaign_type: string
```

---

## Execution Protocol

### Step 1: Rank Assets by Content Richness

Sort all assets descending by content_richness score. The richest assets get placed in the most impactful email positions.

```
POSITION IMPORTANCE RANKING:
  1. First email (anchor — sets the tone, must hook)
  2. Last email (closer — final push, must convert)
  3. Email after first pitch (proof of claims)
  4. Midpoint email (re-engagement moment)
  5. All other emails (standard positions)
```

### Step 2: Apply Body Type Assignment Rules

For each email position, select the body type using these rules in priority order:

```
RULE 1: Anchor/Opener Emails (Position 1, first of each phase)
  → Preferred: Story (ST)
  → Rationale: Stories hook attention. The first email must pull the reader in.
  → Fallback: Quote-Opener (QO) if no strong story asset exists.

RULE 2: Social Proof Emails (after any pitch email)
  → Preferred: Testimonial (TM)
  → Rationale: After making a claim, immediately prove it with social proof.
  → Fallback: List-Based (LB) with data points.

RULE 3: Paradigm-Shift Emails (middle of sequence, education phase)
  → Preferred: Contrarian (CT)
  → Rationale: Challenge conventional thinking to create openness to new mechanism.
  → Fallback: Story (ST) with a "myth-busting" angle.

RULE 4: Objection-Handling Emails (before final pitch sequence)
  → Preferred: Q&A (QA)
  → Rationale: Address remaining doubts before the close.
  → Fallback: Testimonial (TM) that implicitly handles objections.

RULE 5: Feature/Benefit Emails (product education phase)
  → Preferred: List-Based (LB)
  → Rationale: Structured information delivery for mechanism/ingredient education.
  → Fallback: Quote-Opener (QO) with expert backing each point.

RULE 6: Authority-Building Emails (early sequence, trust phase)
  → Preferred: Quote-Opener (QO)
  → Rationale: External authority builds trust faster than self-claims.
  → Fallback: Story (ST) with credential weaving.

RULE 7: Personality/Entertainment Emails (daily relationship, scattered)
  → Preferred: Negative Response (NR)
  → Rationale: Addressing haters builds personality and engagement.
  → HARD LIMIT: Maximum 1 NR per 7 emails.
  → Fallback: Story (ST) with personality-forward angle.
```

### Step 3: Map Assets to Positions

For each email position with an assigned body type, find the best-fit asset:

```
FOR EACH email position:
  1. FILTER assets where body_type_fit == assigned body type
  2. SORT filtered assets by content_richness (descending)
  3. SELECT the highest-richness asset NOT already assigned to another position
  4. IF no unassigned assets of this type remain:
     a. CHECK secondary_fit matches
     b. IF secondary match found → use it
     c. IF no match → mark as "content_source: to_be_written"
  5. RECORD the assignment: position → asset_id → body_type → content_richness
```

### Step 4: Validate Variety Rules Post-Assignment

After all positions are assigned, re-validate:

```
CHECK 1: No consecutive same body type
  → Scan pairs: (email_1, email_2), (email_2, email_3), ...
  → IF same type adjacent:
    → SWAP the lower-importance position with the nearest different type

CHECK 2: Minimum 3 types per 5-email window
  → Slide window: positions [1-5], [2-6], [3-7], ...
  → COUNT unique types in each window
  → IF any window has < 3 unique types:
    → SWAP the most repeated type in that window with a different type

CHECK 3: NR count
  → COUNT NR assignments
  → IF count > floor(total_emails / 7):
    → REMOVE excess NR assignments, replace with next-best type
```

### Step 5: Generate Assignment Justification

For each email, document WHY this body type was assigned:

```
"Position 3 assigned Testimonial (TM) because it follows the first pitch email (Position 2).
 Asset TM-004 (richness: 8) selected — strongest testimonial with specific outcome numbers.
 This placement maximizes social proof impact immediately after the offer mention."
```

```yaml
    swaps_performed: integer  # how many swaps were needed to comply
    swap_details: [string]  # what was swapped and why

  unassigned_positions: [integer]  # positions marked "to_be_written"
  unassigned_count: integer
```

---

## Failure Modes

| Failure | Remediation |
|---------|-------------|
| Not enough assets for all positions | Mark positions as "to_be_written". E1 must generate these from scratch using upstream context. |
| Variety rules impossible to satisfy | Reduce email count. Remove emails until variety rules can be met with available types. |
| All high-richness assets are the same type | Distribute across positions regardless of rule priority. Variety > optimal placement. |
| No Story (ST) assets for anchor position | Use Quote-Opener (QO) for anchor. Flag: "No story available for opener — using authority hook instead." |
| Multiple positions competing for same asset | Higher-importance position wins. Lower position gets next-best asset or secondary fit. |
