# Layer 0.2 — Brief Parser (Mode B)

## Purpose

Parse a user-provided brief into a standardized brief_context object for standalone email campaign design. Extract product/offer description, target audience, mechanism/story (optional), available proof (optional), and campaign goal. Validate that minimum required fields are present.

**Mode:** B only. If operating in Mode A (downstream from CopywritingEngine), this microskill is SKIPPED and 0.1-upstream-loader.md runs instead.

---

## Input Requirements

```yaml
user_brief:
  format: "freeform text OR structured YAML/markdown"
  source: "user-provided directly in conversation or as file"
  minimum_fields:
    - product_or_offer_description  # REQUIRED
    - target_audience               # REQUIRED
    - campaign_goal                 # REQUIRED
  optional_fields:
    - mechanism_or_story
    - available_proof_or_testimonials
    - tone_or_voice_preferences
    - urgency_or_deadline_info
    - competitor_context
    - existing_email_list_size
    - price_point
```

---

## Execution Protocol

### Step 1: Detect Brief Format

Determine whether the brief is:
- **Freeform text** — requires extraction and structuring
- **Structured YAML/markdown** — requires parsing and validation
- **Conversation context** — requires compilation from multiple messages

If freeform, proceed to Step 2 (extraction). If structured, skip to Step 3 (validation).

### Step 2: Extract Fields from Freeform Brief

Scan the brief for each required and optional field. Use these extraction patterns:

| Field | Look For |
|-------|----------|
| `product_description` | Product name, what it is, what it does, format (supplement, course, service, etc.) |
| `target_audience` | Who it's for — demographics, psychographics, pain points, desires |
| `campaign_goal` | Launch, nurture, promote, educate, sell — the primary objective |
| `mechanism_story` | How/why the product works, unique angle, discovery story |
| `available_proof` | Testimonials, data, studies, expert endorsements, case studies |
| `tone_preferences` | Conversational, authoritative, casual, urgent, educational |
| `urgency_info` | Deadlines, limited inventory, price increases, bonus expiration |
| `competitor_context` | Who else sells to this audience, what they promise |
| `list_size` | Email list size — affects volume and segmentation strategy |
| `price_point` | Product price — affects urgency and value demonstration needs |

For each field, extract the EXACT language from the brief — do not summarize or rewrite. Preserve the user's phrasing.

### Step 3: Validate Minimum Fields Present

```
REQUIRED FIELD CHECK:
  product_description: [present/missing]
  target_audience: [present/missing]
  campaign_goal: [present/missing]

IF any REQUIRED field is missing:
  → Set validation_status = "INCOMPLETE"
  → List missing fields with specific questions to ask the user
  → DO NOT proceed to 0.3

IF all REQUIRED fields present:
  → Set validation_status = "COMPLETE"
  → Proceed to 0.3-input-validator.md
```

### Step 4: Assess Context Richness

Score the brief's context richness on a 1-10 scale based on how many optional fields are present and how detailed the required fields are:

| Score | Criteria |
|-------|----------|
| 1-3 | Only required fields, minimal detail |
| 4-5 | Required fields with moderate detail, 1-2 optional fields |
| 6-7 | Detailed required fields, 3-4 optional fields |
| 8-10 | Comprehensive brief with most optional fields populated |

Context richness directly influences downstream campaign complexity:
- **1-3:** Recommend shorter sequences (Content Series or Blatant Pitch)
- **4-5:** Standard sequences viable (Autoresponder, Affiliate Promotion)
- **6-7:** Full sequences viable (Product Launch, Daily Relationship)
- **8-10:** Complex sequences with high variety (Product Launch with full body type range)

### Step 5: Structure into brief_context Object

Compile all extracted/parsed fields into the standardized brief_context format.

## Output

```yaml
brief_context:
  mode: "B"
  validation_status: enum[COMPLETE, INCOMPLETE]
  missing_fields: [string]  # empty if COMPLETE
  missing_field_questions: [string]  # specific questions to ask user

  product:
    name: string
    description: string
    format: enum[supplement, course, ebook, service, software, physical, bundle, other]
    price: string  # "unknown" if not provided

  audience:
    description: string
    pain_points: [string]
    desires: [string]
    awareness_level: string  # estimated from brief if not explicit
    relationship_stage: string  # estimated: cold if new list, warm if existing

  goal:
    primary_goal: string
    campaign_type_hint: string  # preliminary mapping to campaign type code
    urgency_info: string  # "none" if not provided

  assets:
    mechanism_story: string  # "none" if not provided
    proof_elements: [string]  # empty array if none provided
    testimonials: [string]  # empty array if none provided
    competitor_context: string  # "unknown" if not provided

  preferences:
    tone: string  # "not specified" if not provided
    list_size: string  # "unknown" if not provided

  context_richness:
    score: integer  # 1-10
    detail: string  # justification for score
    recommended_complexity: enum[minimal, standard, full, complex]
```

---

## Failure Modes

| Failure | Remediation |
|---------|-------------|
| Brief is too vague (no clear product) | Return INCOMPLETE with specific question: "What product or offer will this email campaign promote?" |
| No audience defined | Return INCOMPLETE with specific question: "Who is the target audience for this campaign?" |
| No goal defined | Return INCOMPLETE with specific question: "What is the primary goal? (launch, nurture, promote, educate, etc.)" |
| Brief contains contradictory information | Flag contradictions in output, ask user to clarify |
| Brief is a single sentence | Extract what is present, return INCOMPLETE with all missing field questions |
