# 0.1 — Input Loader

**Version:** 1.0
**Skill:** U5 — Upsell Editorial
**Layer:** 0
**Microskill:** 0.1 — Input Loader
**Model:** haiku
**Purpose:** Load the complete assembled upsell sequence from U4, carry forward U4 validation flags, verify the GATE sequence (U4 -> U5), and detect operating mode (A vs B).

---

## PATH RESOLUTION

All upstream packages use the standardized output folder:

```
PRIMARY:  outputs/[project-code]/upsell/[filename]
FALLBACK: outputs/[project-code]/[filename]
```

Where `[project-code]` is the 2-4 letter campaign identifier (e.g., RS1, SPD, ULV).

## INPUT

- `upsell-sequence-assembled.md` (from U4 — the assembled sequence) — REQUIRED
- `validation-report.md` (from U4 — flags for U5 to carry forward) — REQUIRED
- `context-reservoir.md` (human-curated) — RECOMMENDED — provides the original VOC quotes and proof elements for verifying congruence accuracy during editorial review
- U0 handoff spec (to confirm Mode A vs B context) — REFERENCED
- UPSELL-ENGINE-CLAUDE.md (5 Laws, structural rules) — REFERENCED

## EXECUTION STEPS

1. **Verify GATE sequence.** Check for U4 completion checkpoint (`GATE_U4_VERIFIED.yaml` or equivalent). U5 cannot execute without a completed U4 assembly. If no U4 checkpoint exists, HALT — request U4 execution first.

2. **Load the assembled sequence file.** Read `upsell-sequence-assembled.md` in full. This file contains the complete sequence: order bump, upsell page(s), and downsell page(s) assembled by U4.

3. **Parse piece inventory.** Extract and catalog every piece in the assembled sequence:
   - Order Bump(s): count, product name(s), word count per bump
   - Upsell Page(s): count, product name(s), word count per page, format (text/video)
   - Downsell Page(s): count, product name(s), word count per page
   - Total word count across all pieces
   - Estimated total reading time (at 200 wpm)

4. **Detect operating mode.** Determine whether the upstream pipeline ran in Mode A (downstream from U0 with full engine context) or Mode B (standalone brief). Check for:
   - Mode A indicators: U0 handoff spec present, mechanism-package.json referenced, soul.md loaded
   - Mode B indicators: standalone brief as source, "Mode B — limited upstream context" flag in U4 output
   - Log the detected mode. Mode B sequences may have weaker congruence data — flag this for the Congruence Auditor lens in Layer 1.

5. **Load U4 validation flags.** Read `validation-report.md` from U4. Extract ALL flagged issues:
   - Congruence warnings (mechanism name inconsistencies across pieces)
   - Word count warnings (any piece near or over limits)
   - Structure warnings (missing CAIRO/ARO sections)
   - Pricing warnings (cascade ratio violations)
   - Tone warnings (selling language detected)
   - For each flag: record piece affected, severity estimate, and U4's description.

6. **Extract congruence thread data.** From the assembled sequence, identify:
   - Mechanism name (as it appears across all pieces — exact string)
   - Root cause language (as it appears across all pieces — verbatim)
   - Promise phrasing (FE promise and its extensions)
   - Note any inconsistencies across pieces (e.g., mechanism name in bump differs from upsell page)

7. **Extract pricing data.** From the assembled sequence, identify:
   - FE price (referenced in the sequence)
   - Order bump price
   - Upsell price(s)
   - Downsell price(s)
   - Any ratios or anchoring already present

8. **Write output file** with full inventory, mode detection, U4 flags, congruence thread, and pricing data.

## CONSTRAINTS

1. U4 completion is a HARD prerequisite. No U4 checkpoint = no U5 execution. Do not proceed on partial U4 output.
2. Piece inventory must be EXACT — every piece in the assembled sequence must be cataloged with accurate word counts. Do not estimate. Count.
3. U4 flags must be carried forward IN FULL. Do not filter, summarize, or discard any U4 flag. Every flag transfers to U5's issue identification in Layer 1.
4. Congruence thread data is extracted from the ASSEMBLED sequence as-is, not from upstream packages. This captures what actually appears in the copy, not what was intended.
5. Mode detection is informational — it does not change U5's execution. Both modes get full editorial review. Mode B simply gets flagged for potentially weaker upstream data.
6. This microskill does NOT score anything. It loads and inventories. Scoring happens in 1.1.

## FAILURE MODES

| Failure | Detection | Response |
|---------|-----------|----------|
| Missing U4 assembled sequence | File not found or empty | HALT — request U4 execution. Cannot proceed without assembled sequence. |
| Missing U4 validation report | File not found | WARN — proceed but note that no upstream flags are available. All issues must be discovered fresh in Layer 1. |
| Unparseable sequence | Pieces cannot be distinguished (no headers, no structure) | HALT — request U4 re-assembly with proper section delineation. |
| Mode detection ambiguous | Neither Mode A nor Mode B indicators clear | Default to Mode B (conservative). Flag for human confirmation. |
| Congruence thread inconsistent | Mechanism name differs across pieces | Flag as pre-identified issue (likely P1) for Layer 1 to confirm and score. |
