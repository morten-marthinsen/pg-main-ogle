# 4.3 — Output Packager

**Version:** 1.0
**Skill:** U5 — Upsell Editorial
**Layer:** 4
**Microskill:** 4.3 — Output Packager
**Model:** sonnet
**Purpose:** Package the final revised upsell sequence (`upsell-sequence-final.md`) and generate the comprehensive editorial report (`EDITORIAL-REPORT.md`). Present both to human for BLOCKING review. Gate 3 does not pass without explicit human approval.

---

## INPUT

- The REVISED upsell sequence (post-Layer 2 — the current polished copy)
- `layer-1/1.1-baseline-scores.md` (baseline scores)
- `layer-1/1.2-issue-list.md` (all issues identified)
- `layer-1/1.3-issue-clusters.md` (fix clusters and revision plan)
- `layer-2/2.1-revisions.md` (all revisions applied, before/after, Arena selections)
- `layer-4/4.1-rescores.md` (final scores, deltas, regressions)
- `layer-4/4.2-sequence-criteria.md` (S1-S5 validation results)
- `layer-0/0.1-input-context.md` (project metadata, pricing, congruence thread)

## EXECUTION STEPS

1. **Verify prerequisites.** Before packaging:
   - 4.1 rescores show ALL pieces >= 7.5 weighted average? If not, HALT.
   - 4.2 shows ALL S1-S5 criteria PASS? If not, HALT.
   - If either check fails, this microskill cannot execute. Return to Layer 2 for additional revision.

2. **Assemble `upsell-sequence-final.md`.** This is the DELIVERABLE — the polished, final copy:

   a. **Write YAML metadata header** per the output schema in U5-UPSELL-EDITORIAL-AGENT.md:
      - Project name, FE product, FE mechanism
      - Editorial summary: total issues, per-severity counts, issues resolved, Arena rounds
      - Scores per piece: baseline, final, delta
      - S1-S5 results
      - Human review status: PENDING (will be updated after review)
      - Creation date

   b. **Write the final copy for each piece** in sequence order:
      - ORDER BUMP — with product name header
      - UPSELL PAGE — with product name header
      - DOWNSELL PAGE — with product name header
      - Use clean formatting. No editorial annotations in the final copy. No score references. No issue tags. This is the CLEAN deliverable.
      - Separate pieces with horizontal rules.

   c. **Verify completeness.** Cross-check: does every piece from the 0.1 inventory appear in the final sequence? Any missing piece = HALT.

3. **Generate `EDITORIAL-REPORT.md`.** This is the EVIDENCE TRAIL — comprehensive documentation of the editorial process:

   a. **Executive Summary:**
      - Pieces reviewed (count and names)
      - Total issues found (count, per-severity breakdown)
      - Arena rounds executed (count)
      - Overall improvement: baseline average -> final average (+delta)
      - S-criteria status: X/5 passing

   b. **Baseline Scores:** Per piece x per lens table from 1.1. Include weighted averages.

   c. **Issues Found:** Per issue from 1.2 — severity, piece, lens, passage, problem, target fix. Preserve the full detail.

   d. **Fix Clusters:** From 1.3 — cluster composition, routing (Arena/Direct Fix), execution order.

   e. **Revisions Applied:** From 2.1 — per cluster: before text, after text, method (Arena with winner name and score / Direct Fix), preservation check results. This is the most important section for the human reviewer — they can see exactly what changed and why.

   f. **Final Scores:** Per piece x per lens table from 4.1. Include deltas from baseline. Flag any regressions.

   g. **Sequence-Level Criteria (S1-S5):** From 4.2 — per criterion: PASS/FAIL with evidence. Include baseline comparison.

   h. **Improvement Summary:**
      - Per piece: baseline -> final -> delta
      - Per lens: average improvement
      - Most improved area
      - Regressions (if any, with justification)

   i. **Recommendations:** Any remaining suggestions that are BEYOND the editorial scope:
      - Upstream issues that would benefit from U2/U3 re-execution
      - Testing recommendations (e.g., "Consider A/B testing the downsell reframe")
      - Human judgment calls (e.g., "The upsell Intrigue section scored 7.6 — meets threshold but could be stronger with additional proof")
      - These are SUGGESTIONS, not requirements. The sequence passes editorial if all gates pass.

4. **Present for HUMAN REVIEW.** This is a BLOCKING gate:
   - Notify the human that `upsell-sequence-final.md` and `EDITORIAL-REPORT.md` are ready for review
   - The human must explicitly respond with one of:
     - **APPROVED** — sequence is final. Update metadata `human_review: APPROVED`.
     - **REVISION_REQUESTED** — human provides feedback. Return to Layer 1 (re-score with new direction) or Layer 2 (targeted revision per human feedback).
   - Do NOT auto-approve. Do NOT assume approval. Do NOT proceed without explicit human response.
   - Do NOT mark the sequence as complete while review is PENDING.

5. **Write both output files.**

## CONSTRAINTS

1. **Human review is BLOCKING.** This is non-negotiable. The sequence is NOT complete until a human explicitly approves. PENDING is not APPROVED. Silence is not approval. "Looks good" in conversation is not approval — the human must provide a clear APPROVED or REVISION_REQUESTED decision.
2. **`upsell-sequence-final.md` is CLEAN copy.** No editorial annotations, no score references, no issue tags, no strikethrough/tracked changes. This is the deliverable the human reads as a buyer would see it.
3. **`EDITORIAL-REPORT.md` is COMPREHENSIVE evidence.** Include ALL data from all upstream microskills. The human should be able to trace any decision from issue identification through clustering through revision through rescore.
4. **Both files must be written.** One without the other is incomplete. The final copy without the report lacks evidence. The report without the final copy lacks the deliverable.
5. **Prerequisites are HARD.** If 4.1 shows any piece < 7.5 or 4.2 shows any S-criterion failing, this microskill CANNOT execute. Do not package a failing sequence.
6. **Metadata must be accurate.** Counts, scores, and statuses in the YAML header must match the actual data. Do not estimate or round. Cross-check against upstream files.
7. **Every piece from inventory must appear in final.** If 0.1 inventoried 3 pieces, all 3 must appear in `upsell-sequence-final.md`. A missing piece means the sequence is incomplete.
8. **On REVISION_REQUESTED:** Capture the human's specific feedback. Determine whether to return to Layer 1 (re-baseline with new direction) or Layer 2 (targeted fix). Do not restart from scratch unless the human explicitly requests it.

## FAILURE MODES

| Failure | Detection | Response |
|---------|-----------|----------|
| Auto-approved without human | `human_review` set to APPROVED without human response | Reset to PENDING. Present report to human. Wait. |
| Failing piece packaged | 4.1 shows a piece < 7.5 but final sequence includes it | HALT. Return to Layer 2. Do not package failing copy. |
| S-criterion failure packaged | 4.2 shows S-criterion FAIL but final sequence proceeds | HALT. Return to Layer 2. Do not package a sequence that fails sequence-level criteria. |
| Missing piece in final | Piece count in final != piece count in 0.1 inventory | Add the missing piece. All pieces must appear. |
| Annotations in clean copy | Editorial notes, score references, or issue tags in `upsell-sequence-final.md` | Remove all annotations. The final copy is clean — for buyer consumption. |
| Incomplete editorial report | Report missing sections (baseline, issues, revisions, final scores, S-criteria) | Complete all sections. The report is the evidence trail. Incomplete evidence = incomplete review. |
| Metadata mismatch | YAML header counts differ from actual data | Correct metadata. Cross-check every number against source files. |
