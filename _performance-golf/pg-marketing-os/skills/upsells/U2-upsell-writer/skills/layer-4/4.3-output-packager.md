# 4.3 — Output Packager

**Version:** 1.0
**Skill:** U2 — 1-Click Upsell Writer
**Layer:** 4
**Microskill:** 4.3 — Output Packager
**Purpose:** Package the validated, scored, Arena-selected upsell page into the final deliverable with YAML metadata, clean CAIRO body, downstream handoff data for U3, and implementation notes.

---

## INPUT

- Arena-selected upsell page (validated by 4.1, scored by 4.2)
- `layer-4/4.1-validation-results.md` (ALL PASS confirmation)
- `layer-4/4.2-congruence-score.md` (congruence score >= 7.0 confirmation)
- `layer-0/0.1-input-context.md` (all metadata: product names, prices, mechanism, position, format, mode)
- `layer-1/1.1-congruence-map.md` (congruence anchors for downstream handoff)
- `layer-2/2.2-bonus-stack.md` (bonuses list for downstream handoff)
- Arena selection data (which competitor/hybrid was selected, weighted score)

**PREREQUISITES:**
- 4.1 validation: ALL 7 gates PASS
- 4.2 congruence score: >= 7.0
- If either prerequisite is not met, do NOT package. Return to the appropriate layer for revision.

## PROCESS

1. **Verify prerequisites.**
   - Confirm 4.1 validation results show ALL PASS
   - Confirm 4.2 congruence score >= 7.0
   - If either is not met → HALT. Do not produce a final deliverable from unvalidated or incongruent output.

2. **Build YAML metadata header.** Populate every field from upstream sources:
   ```yaml
   ---
   metadata:
     skill: "U2 — 1-Click Upsell Writer"
     version: "1.0"
     project: "[from input context]"
     position: "[upsell_1 | upsell_2]"
     format: "[video_script | text_page]"
     product: "[upsell product name]"
     price: "[upsell price]"
     fe_product: "[FE product name]"
     fe_price: "[FE price]"
     fe_mechanism: "[mechanism name — EXACT]"
     word_count: "[actual word count from validated output]"
     congruence_score: "[score from 4.2]"
     arena_selection:
       persona: "[selected competitor/hybrid name]"
       type: "[pure | hybrid]"
       weighted_score: "[Arena weighted score]"
     mode: "[A | B]"
     created: "[date]"
     validation_status: "ALL PASS (7/7 gates)"
   ---
   ```
   - Every field is REQUIRED. No "TBD" or placeholder values. If a field cannot be populated, something went wrong upstream — HALT and diagnose.

3. **Clean and format the CAIRO body.** Take the Arena-selected output and:
   - Ensure CAIRO section headers are standardized: `## CONGRATULATE`, `## AMPLIFY`, `## INTRIGUE`, `## REASON`, `## OFFER`
   - Remove any Arena-specific annotations, scoring notes, or generation metadata from the copy
   - Preserve all copy exactly as selected — do NOT edit, rewrite, or "improve" the Arena winner. Packaging is assembly, not editing.
   - Verify bonus stack and price presentation are integrated (from 2.2 and 2.3). If placeholders remain, integrate them now.
   - If video_script format: ensure stage directions, speaker attribution, visual cues, text overlays, and CTA screen direction are present
   - If text_page format: ensure headline hierarchy, CTA button formatting, and layout notes are present

4. **Build downstream handoff for U3 (Downsell Writer).** The downsell needs to know exactly what the buyer declined so it can REFRAME (not just discount). Populate:
   - **upsell_product:** exact product name the buyer declined
   - **upsell_price:** the price they said no to
   - **upsell_angle:** one-sentence summary of the angle used (e.g., "positioned as an accelerator that makes the FE mechanism work 47% faster")
   - **bonuses_included:** list of all bonuses offered in the upsell
   - **guarantee_type:** "mirrored" (same as FE) or "upsell-specific" (different terms)
   - **mechanism_name:** exact mechanism name (same thread continues into downsell)
   - **root_cause_language:** exact root cause language (same thread continues)
   - **promise_extension:** how the upsell extended the FE promise (downsell will need to maintain or scale down this extension)
   - **reframe_suggestions:** 2 specific suggestions for how the downsell could reframe the declined upsell:
     - Suggestion 1: typically a "stripped down" version — same core value, fewer bonuses, lower price
     - Suggestion 2: typically an "alternative angle" — same product, different emphasis (e.g., if upsell emphasized speed, downsell emphasizes simplicity)

5. **Build implementation notes.** Format-specific guidance for whoever implements the page:

   **If video_script:**
   - Total estimated runtime (word count / 150 words per minute)
   - Presenter tone notes per CAIRO section
   - Key visual moments (product reveals, text overlays, CTA screen)
   - Recommended video style (talking head, mixed with B-roll, screencast)
   - Audio/music suggestions (upbeat for Congratulate/Amplify, focused for Reason, clean for Offer)

   **If text_page:**
   - Recommended page layout (single column, centered, max 700px width)
   - CTA button styling (primary color, large, above fold for mobile)
   - Image placement suggestions (product image in Offer section, benefit image in Amplify)
   - Mobile responsiveness notes (all text readable without zoom, buttons full-width on mobile)
   - Above-fold content (what the buyer sees before scrolling — should include at minimum the Congratulate section)

6. **Write the final deliverable.** The output file goes in the PROJECT ROOT, not in layer-4/. This is the deliverable that downstream skills (U3, U4) consume.

7. **Write the packaging metadata file** in layer-4/ for audit trail purposes.

## CONSTRAINTS

1. **Do NOT edit the Arena-selected copy.** Packaging is assembly and formatting, not editing. The copy is exactly as the human selected it from the Arena. If there are quality concerns, they should have been caught in 4.1/4.2 or in the Arena itself.
2. **Every metadata field must be populated.** No TBD, no placeholders, no "see above." If a field cannot be populated, something is wrong — HALT and diagnose rather than shipping an incomplete deliverable.
3. **The primary deliverable (`upsell-page-draft.md`) goes in the PROJECT ROOT**, not in a layer subdirectory. This is the file that U3, U4, and U5 will consume. It must be easily findable.
4. **Downstream handoff for U3 is MANDATORY.** The downsell writer needs to know what was offered and declined. Without this data, U3 cannot write an effective reframe. Missing handoff data = GATE_3 FAIL.
5. **Reframe suggestions must be SPECIFIC, not generic.** "Offer a discount" is not a reframe suggestion. "Strip the two bonuses, keep the core product, reduce price to $27, and reposition as 'just the essentials'" IS a reframe suggestion.
6. **Implementation notes are for the HUMAN implementing the page**, not for other skills. Write them in plain language, not engine jargon. A web developer or video producer should be able to act on them directly.
7. **This is the LAST microskill in the U2 pipeline.** After this file is written, U2 is COMPLETE for this project. Verify all per-microskill output files exist before declaring completion.
