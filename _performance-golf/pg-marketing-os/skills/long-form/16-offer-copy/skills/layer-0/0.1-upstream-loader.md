## PURPOSE

The Upstream Loader is the first microskill in the offer copy pipeline. It loads all upstream data packages that feed the offer presentation system: the product introduction package (from Skill 15), the offer architecture package (from Skill 11), the structure package (from Skill 11), the promise package (from Skill 11), and the proof inventory output (from Skill 11). These five upstream packages collectively provide everything the offer copy skill needs to write a complete, accurate, and strategically aligned offer presentation.

This loader is especially important because Skill 06 (offer) may be incomplete or a placeholder. The loader must gracefully handle partial or missing offer data, flagging gaps without failing outright. The downstream input validator (0.3) and human checkpoint (0.4) rely on the loader's completeness report to determine which data needs human intervention.

Every field consumed from each package is explicitly mapped so that no downstream microskill ever references a field that was not loaded. The loader creates a unified upstream_data object that becomes the single source of truth for the entire pipeline.

---

## IDENTITY

**This microskill IS:**
- The data ingestion gateway for all five upstream packages required by Skill 13
- A completeness auditor that reports which packages are present, partial, or missing
- A field mapper that indexes every consumed field from every upstream package
- A graceful handler of missing Skill 06 data (offer-package may be incomplete)
- The single-source-of-truth constructor for all upstream data in the offer copy pipeline
- A dependency verifier ensuring Skill 12 handoff continuity data is present
- A schema validator that confirms each loaded package matches expected structure

**This microskill is NOT:**
- A data transformer — it loads and indexes, it does not alter upstream data
- A gap filler — it identifies gaps but does not attempt to fill them (that is 0.3 and 0.4)
- A source teaching loader — SIN methodology and TIER1 patterns are loaded by 0.2
- A validator of offer completeness — that is 0.3's responsibility
- A human-facing presenter — that is 0.4's responsibility

## PROCESS

### Step 1: Locate and Attempt Load of All Five Packages

1. Attempt to load `product-introduction-package.json` from Skill 15 outputs
2. Attempt to load `offer-package.json` from Skill 11 outputs
3. Attempt to load `structure-package.json` from Skill 11 outputs
4. Attempt to load `promise-package.json` from Skill 11 outputs
5. Attempt to load `proof-inventory-output.json` from Skill 11 outputs
6. For each package, record: `loaded: true/false`, `path`, `timestamp`, `field_count`

### Step 2: Validate Package Schema Integrity

7. For each successfully loaded package, verify the expected schema structure:
   - Confirm top-level keys exist
   - Confirm nested objects contain required fields
   - Flag any unexpected schema deviations as `schema_warning`
8. For product-introduction-package specifically, verify the `downstream_handoffs.for_offer_copy` section exists — this is the primary handoff and is CRITICAL
9. For offer-package specifically, determine completeness level:
   - `complete`: All core_offer, enhancement_stack, price_architecture, naming fields present
   - `partial`: Some fields present, gaps identified
   - `missing`: Package not found or empty
   - `placeholder`: Package exists but contains template/placeholder values

### Step 3: Extract and Index All Consumed Fields

10. From product-introduction-package, extract:
    - `product_name` (from handoff or narrative_sections)
    - `positioning_statement`
    - `components_established` (list of component names for proof reference)
    - `belief_level_at_handoff` and `emotional_state_at_handoff`
    - `value_stack_foundation` signals
11. From offer-package (if available), extract:
    - `deliverables` array with names, descriptions, features
    - `bonuses` array with names, values, descriptions
    - `guarantee` object with type, duration, branded_name, terms
    - `scarcity` and `urgency` objects
    - `price_architecture` with retail_value, actual_price, anchor_points, terms
12. From structure-package, extract:
    - `offer_position` within the argument flow
    - `word_count_target` for the offer section
    - `emotional_state_at_entry` for continuity
13. From promise-package, extract:
    - `primary_promise` text and type
    - `transformation` object (before/after states)
    - `restatement_bank` for promise repetition calibration (Layer 3)
14. From proof-inventory, extract:
    - Available proof assets by category
    - `proof_strength_index` for D-F-W-B-P proof element selection

### Step 4: Build Completeness Report

15. Generate a completeness report with the following structure:
    - Package-level status: each of the 5 packages marked as `loaded`, `partial`, `missing`
    - Field-level status: each critical field marked as `present`, `partial`, `absent`
    - Critical gaps: list of fields that are required for offer copy but missing
    - Offer data source determination:
      - `offer_package` if 07-offer data is complete
      - `human_required` if 07-offer is missing or placeholder
      - `merge_required` if 07-offer is partial
16. Count total fields consumed vs. total fields available
17. Calculate completeness percentage: `(fields_present / fields_required) * 100`

### Step 5: Determine Handoff Continuity Status

18. Verify the product introduction to offer copy handoff is intact:
    - `product_name` matches across packages (product-intro, offer, structure)
    - `emotional_state_at_handoff` from Skill 15 is present and actionable
    - `belief_level_at_handoff` provides a clear starting emotional state for offer entry
    - The last line of Skill 12's output can be identified for seamless transition
19. Flag any discontinuity: name mismatch, missing emotional state, unclear entry point

### Step 6: Compile Unified Upstream Data Object

20. Assemble all extracted data into a single `upstream_data` object organized by domain:
    - `product`: name, format, positioning, components, mechanism context
    - `offer`: deliverables, bonuses, guarantee, scarcity, urgency (may be partial)
    - `pricing`: retail_value, actual_price, anchor_points, terms, savings
    - `promise`: primary promise, supporting promises, restatement bank
    - `proof`: available proof assets indexed by category and strength
    - `structure`: offer position, word count target, entry emotional state
    - `handoff`: belief level, emotional state, entry point, continuity status
21. Attach the completeness report to the unified object
22. Mark the unified object as the authoritative data source for all downstream microskills

## FAILURE MODES

### Failure Mode 1: Product Introduction Package Missing
- **Symptom:** No product-introduction-package.json found or file is empty
- **Cause:** Skill 12 did not complete or output was not saved to expected path
- **Prevention:** Verify Skill 12 completion status before triggering Skill 13
- **Recovery:** BLOCKING failure — cannot proceed. Log error and request Skill 12 completion via feedback bus: `{ skill: "15-product-introduction", request: "handoff_alignment", issue: "package_missing" }`

### Failure Mode 2: Offer Package is Placeholder
- **Symptom:** offer-package.json exists but contains template values like "[INSERT DELIVERABLE]"
- **Cause:** Skill 06 is in placeholder state and was never populated for this campaign
- **Prevention:** Check for placeholder markers (`[INSERT`, `TBD`, `TODO`) in all string fields
- **Recovery:** Set offer_data_source to `human_required`, set checkpoint_default to `PARTIAL`, log all placeholder fields as critical gaps. Pipeline continues — human checkpoint (0.4) will collect missing data.

### Failure Mode 3: Product Name Mismatch Across Packages
- **Symptom:** Product name in product-introduction-package does not match offer-package or structure-package
- **Cause:** Packages were generated from different campaign runs or a rename occurred mid-pipeline
- **Prevention:** Compare product_name across all packages that contain it
- **Recovery:** Flag discontinuity, use product-introduction-package name as authoritative (most recent upstream), note mismatch in completeness report for human review at checkpoint

### Failure Mode 4: Promise Package Missing Restatement Bank
- **Symptom:** promise-package loads but restatement_bank is empty or absent
- **Cause:** Skill 04 generated a primary promise but did not build restatement variations
- **Prevention:** Check for restatement_bank existence and minimum entry count (3+)
- **Recovery:** Flag as non-critical gap. Layer 3 (promise repetition calibrator) can generate restatements from the primary promise if the bank is empty, but quality will be lower.

### Failure Mode 5: Proof Inventory Has No Relevant Assets
- **Symptom:** proof-inventory-output.json loads but contains no proof assets relevant to offer deliverables
- **Cause:** Proof inventory was built for mechanism/story but not mapped to individual deliverables
- **Prevention:** Check proof asset count and category coverage
- **Recovery:** Flag as gap. D-F-W-B-P proof elements (Layer 2) will need to use general authority proof or mechanism proof rather than deliverable-specific proof. Log request via feedback bus: `{ skill: "02-proof-inventory", request: "proof_gap", deliverable: "all", needed: "deliverable_specific_proof" }`
