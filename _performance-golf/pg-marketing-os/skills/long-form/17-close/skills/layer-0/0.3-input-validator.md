## PURPOSE

Validates that all data required for close generation is present, complete, and internally consistent. The close is the FINAL copy section — it draws from benefits (Skill 11), proof (Skill 11), structure (Skill 11), product identity (Skill 15), and offer presentation (Skill 16). If any critical data is missing, the close will be incomplete — a guarantee without a branded name, a benefit summary without enough benefits, a CTA sequence without knowledge of what CTAs already appeared upstream.

This validator checks five categories: (1) main benefits for the "You get" summary drawn from promise and proof inventory, (2) guarantee details including branded name, terms, and type, (3) pricing and value data from offer-copy-package, (4) product identity and campaign format, and (5) emotional state at the handoff point from Skill 16. Each category is validated against minimum thresholds — the close requires at least 5 benefits for the summary, a named guarantee with specific terms, a confirmed price, and a known campaign format.

Any missing data is flagged with severity level (critical vs. advisory) and passed to the human checkpoint (0.4) for resolution. A critical missing field blocks close execution. An advisory missing field allows the close to proceed with noted gaps.

---

## IDENTITY

**This microskill IS:**
- The completeness gatekeeper for close execution data
- A five-category validation engine (benefits, guarantee, pricing, product, emotional state)
- A missing-data severity classifier (critical vs. advisory)
- A cross-package consistency checker (names, prices, claims match across packages)
- The quality assurance step between data loading and human checkpoint
- A readiness assessor that produces a clear GO / CONDITIONAL / BLOCKED verdict

**This microskill is NOT:**
- A data loader (that is 0.1 — this validates what was loaded)
- A source teaching validator (teachings are validated in 0.2)
- A close quality validator (that is Layer 4 — this validates inputs, not outputs)
- A human-facing checkpoint (that is 0.4 — this validates data, 0.4 presents to human)
- A data creator — it identifies gaps, never fabricates data to fill them

## PROCESS

### Step 1: Validate Benefit Data for "You Get" Summary

1. Count total benefits available: main_benefits + supporting_benefits
2. Validate minimum threshold: at least 5 benefits required for the "You get" summary
   - **CRITICAL** if benefit_count < 3 — close cannot produce a compelling benefit summary
   - **ADVISORY** if benefit_count is 3-4 — close can proceed but benefit summary will be thin
   - **PASS** if benefit_count >= 5 — sufficient for strong "You get" summary
3. Check benefit quality:
   - Each benefit should be specific (not generic like "improved health")
   - Each benefit should be outcome-oriented (what the prospect GETS, not what the product DOES)
   - Flag any benefit that is too vague for the "You get" format
4. Cross-reference benefits with proof inventory:
   - Which benefits have testimonial support? (for P.S. technique: strongest testimonials)
   - Which benefits have clinical backing? (for guarantee confidence)
   - Tag proven vs. unproven benefits
5. Check that primary promise restatement is available (for closing theme integration)

### Step 2: Validate Guarantee Data

6. Check guarantee branded name:
   - **CRITICAL** if branded_name is null AND guarantee is the primary risk reversal — close needs a named guarantee for Makepeace and Week 13 patterns
   - **ADVISORY** if branded_name is null but guarantee has terms — close can proceed but guarantee section will lack naming impact
   - **PASS** if branded_name is present (e.g., "Triple-Back-Guarantee," "Double-Eagle Guarantee")
7. Check guarantee type:
   - **CRITICAL** if type is null — close must know whether to frame as "contract" or "personal promise"
   - **PASS** if type is "contract" or "personal_promise"
8. Check guarantee duration:
   - **ADVISORY** if duration is null — close can proceed but cannot specify timeframe in guarantee prose
   - **PASS** if duration is present (30, 60, 90, 180, 365 days)
9. Check guarantee terms:
   - **ADVISORY** if terms are null — close can write a general guarantee but cannot detail specific protections
   - **PASS** if terms present with specific conditions (keep supply, full refund, bonus gift, etc.)
10. Check guarantee full text from offer section:
    - **ADVISORY** if null — close will write guarantee from scratch rather than building on offer version
    - **PASS** if present — close can extend and enhance the offer guarantee presentation
11. Verify guarantee does NOT contain "if not satisfied, return for refund" language:
    - **CRITICAL** if this phrasing appears — violates Makepeace methodology. Flag for rewrite.

### Step 3: Validate Pricing and Value Data

12. Check main price:
    - **CRITICAL** if main_price is null — close cannot reference the price or calculate value
    - **PASS** if main_price is present
13. Check value stack total:
    - **ADVISORY** if value_stack_total is null — close can proceed without explicit value comparison
    - **PASS** if present — enables "total value of $[X]... but you pay only $[Y]" patterns
14. Check savings amount:
    - **ADVISORY** if null — derivable from price and value if both are present
    - **PASS** if present
15. Check bonus inventory:
    - **ADVISORY** if bonus items are empty — limits "But WAIT There's More" theme and P.S. bonus reveal
    - **PASS** if at least 1 bonus item present
16. Validate pricing consistency:
    - If price and value both present, verify savings = value - price
    - If price_per_day present, verify it matches price / duration calculation
    - Flag any arithmetic inconsistencies

### Step 4: Validate Product Identity and Campaign Format

17. Check product name:
    - **CRITICAL** if product name is null — close must reference the product by name
    - **PASS** if present
18. Check product format:
    - **CRITICAL** if format is null — close format constraints depend on this (supplement, digital, physical, info)
    - **PASS** if present
19. Check campaign format:
    - **CRITICAL** if campaign format is null — determines close length, CTA style, sidebar support, P.S. prominence
    - **PASS** if present (VSL, magalog, email, advertorial)
20. Cross-reference product name consistency:
    - Compare product name from Skill 15 with product name from Skill 16
    - **CRITICAL** if names conflict — the close must use ONE consistent name
    - **PASS** if names match or only one source provides the name
21. Check positioning statement and mechanism name:
    - **ADVISORY** if null — close can proceed but cannot integrate these into the closing theme
    - **PASS** if present — enables richer closing theme and Cialdini integration

### Step 5: Validate Emotional State and Handoff Data

22. Check emotional state at close entry:
    - **ADVISORY** if null — close can infer emotional state from pricing strategy and format
    - **PASS** if present (e.g., "High desire, price justified, seeking final reassurance")
23. Check urgency already established:
    - **ADVISORY** if null — close will establish urgency from scratch
    - **PASS** if present — close builds on existing urgency without contradicting it
24. Check CTA exclusion list:
    - **ADVISORY** if empty — close will not know what CTA phrases to avoid
    - **PASS** if present — ensures CTA variety across the full promotion
25. Check proof assets for close use:
    - **ADVISORY** if no testimonials available — limits P.S. technique options (cannot use "strongest testimonials")
    - **PASS** if at least 2 testimonials available for P.S. consideration
26. Check promise restatement:
    - **ADVISORY** if null — close cannot echo the latest promise version
    - **PASS** if present — close can weave promise into benefit summary and closing theme

### Step 6: Validate Source Teaching Completeness

27. Check Makepeace teaching load:
    - **CRITICAL** if elements_loaded < 6 — all 6 elements are mandatory
    - **CRITICAL** if themes_loaded < 7 — all 7 themes must be available for selection
    - **PASS** if 6 elements and 7 themes loaded
28. Check Week 13 teaching load:
    - **ADVISORY** if patterns_loaded < 10 — close can proceed with partial Week 13 patterns
    - **PASS** if all 10 patterns loaded
29. Check TIER1 teaching load:
    - **ADVISORY** if patterns_extracted < 5 — close will rely more heavily on Makepeace + Week 13
    - **PASS** if 5+ patterns extracted

### Step 7: Generate Validation Report

30. Compile validation report with:
    - Total fields validated
    - CRITICAL issues count (must be 0 to proceed)
    - ADVISORY issues count (noted but non-blocking)
    - PASS count
    - Overall readiness: GO (0 critical), CONDITIONAL (0 critical but 3+ advisory), BLOCKED (1+ critical)
31. For each issue, include:
    - Field name
    - Severity (CRITICAL / ADVISORY)
    - Description of what is missing or invalid
    - Recommended resolution action
    - Which downstream microskill is affected
32. Pass validation report to human checkpoint (0.4)

## FAILURE MODES

### Failure Mode 1: Critical Fields Missing — BLOCKED
- **Symptom:** Product name, main price, or campaign format is null
- **Cause:** Upstream skill did not complete or produced incomplete output
- **Prevention:** Skill pipeline enforces output validation at each upstream skill
- **Recovery:** Flag as BLOCKED; request upstream re-run or human provision at checkpoint (0.4)

### Failure Mode 2: Guarantee Contains Prohibited Language
- **Symptom:** Offer copy guarantee text includes "if not satisfied, return for refund"
- **Cause:** Skill 13 did not apply Makepeace methodology to guarantee presentation
- **Prevention:** Skill 13 should have its own guarantee writing standards
- **Recovery:** Flag as CRITICAL; the close skill will rewrite the guarantee entirely using Makepeace framing

### Failure Mode 3: Benefit Count Below Minimum
- **Symptom:** Fewer than 3 benefits available for the "You get" summary
- **Cause:** Skill 04 promise was narrowly scoped or proof inventory is thin
- **Prevention:** Skill 04 mandates minimum benefit claims; Skill 01 captures testimonial benefits
- **Recovery:** Extract additional benefits from product introduction (Skill 15) positioning and proof inventory testimonials; flag for human to confirm expanded benefit list

### Failure Mode 4: Price-Value Inconsistency
- **Symptom:** Savings amount does not equal value_stack_total minus main_price
- **Cause:** Arithmetic error in Skill 13 output or manual data entry error
- **Prevention:** Automated arithmetic check in Step 3
- **Recovery:** Recalculate savings; flag inconsistency for human review; use recalculated value in close

### Failure Mode 5: False PASS on Thin Data
- **Symptom:** Validation passes but close quality suffers due to minimal data
- **Cause:** All critical fields present but at absolute minimums (5 benefits, unnamed guarantee, no testimonials)
- **Prevention:** CONDITIONAL readiness designation for minimum-threshold passes (3+ advisory issues)
- **Recovery:** Human checkpoint (0.4) reviews CONDITIONAL verdicts and may provide additional data
