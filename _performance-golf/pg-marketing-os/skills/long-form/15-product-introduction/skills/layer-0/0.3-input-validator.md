**Skill:** 15-product-introduction

**Version:** 1.0
**Layer:** 0 — Foundation & Loading
**Position:** Third microskill in execution chain
**Persona:** Dr. James Liu — Research Director (Primary), Sarah A. Conco — Client Protection (Secondary)

---

## PURPOSE

Validates all loaded inputs against schema requirements and determines the completeness level of the offer package data. This microskill is the gatekeeper between raw loaded data and the human checkpoint — it determines what the human operator needs to see, confirm, modify, or provide from scratch. The critical output is the offer-completeness classification (COMPLETE, PARTIAL, ABSENT) and the minimum-data assessment that determines whether execution can proceed.

This microskill prevents two catastrophic failures: (1) proceeding without essential product data, and (2) asking the human to provide data that already exists upstream.

---

## IDENTITY

**This microskill IS:**
- A schema validation engine for all upstream data
- An offer-completeness classifier (COMPLETE / PARTIAL / ABSENT)
- A minimum-data-requirement enforcer (product name, core offer, price, guarantee)
- A checkpoint-default recommender based on data completeness
- A gap-to-human translator that converts technical gaps into human-readable questions

**This microskill is NOT:**
- A human interaction handler (that is 0.4)
- A data loader (that is 0.1)
- A strategic decision-maker about offer architecture (that is 07-offer)
- A data creator — it validates and classifies, never fabricates
- A quality judge of content — it checks presence, not quality

## PROCESS

### Step 1: Verify Upstream Loading State

1. Check upstream_data_environment.loading_report.overall_loading_state
2. If BLOCKED:
   - HALT — cannot validate data that failed to load
   - Surface blocking error from 0.1
3. If DEGRADED, OPERATIONAL, or COMPLETE:
   - Continue with validation
   - Note degraded state for downstream flagging

### Step 2: Verify TIER1 Intelligence State

4. Check tier1_intelligence_index.indexing_report.index_readiness
5. If INSUFFICIENT:
   - HALT — skill cannot function without adequate TIER1 intelligence
6. If PARTIAL:
   - Log WARNING: "TIER1 index is partial — some downstream patterns may be limited"
   - Continue
7. If COMPLETE:
   - Log confirmation
   - Continue

### Step 3: Validate Mechanism Narrative Data (REQUIRED)

8. Check mechanism_narrative_data.status
9. If status is not "loaded":
   - CRITICAL ERROR: Cannot write product introduction without mechanism narrative handoff
   - HALT execution
10. Validate each required field:
    - bridge_text: MUST be present (string, length > 20 characters)
    - mechanism_context: MUST be present (string, length > 10 characters)
    - mechanism_name: MUST be present (string, non-empty)
    - proof_elements_established: SHOULD be present (array, length > 0)
    - emotional_state_at_handoff: SHOULD be present
    - belief_level_at_handoff: SHOULD be present
11. For each MUST field missing:
    - Add to critical_gaps list
    - Flag as BLOCKING
12. For each SHOULD field missing:
    - Add to recommended_gaps list
    - Flag as NON-BLOCKING but DEGRADED

### Step 4: Validate Mechanism Data (REQUIRED)

13. Check mechanism_data.status
14. If status is not "loaded":
    - CRITICAL ERROR: Cannot maintain mechanism consistency without mechanism package
    - HALT execution
15. Validate fields:
    - mechanism_name: MUST match mechanism_narrative_data.mechanism_name (or conflict flagged)
    - villain_name: SHOULD be present for future pacing negative future
    - root_cause_name: SHOULD be present for future pacing

### Step 5: Deep Offer-Package Validation

16. This is the CRITICAL validation step. Perform field-by-field assessment:

**5A: Core Offer Validation**
17. Check offer_data.core_offer:
    - product_name: MINIMUM REQUIRED — string, non-empty, length 2-50
    - product_format: MINIMUM REQUIRED — must be one of [supplement, digital, physical, info_product, hybrid]
    - core_description: IMPORTANT — string, length > 20 characters
    - positioning_statement: IMPORTANT — string describing market position
    - uniqueness_claim: IMPORTANT — string with "first" / "only" / "never before" language
    - format_details: OPTIONAL — additional format-specific details

**5B: Enhancement Stack Validation**
18. Check offer_data.enhancement_stack:
    - bonuses: IMPORTANT — array of objects, each with name, value, description
      - Validate each bonus has all 3 required fields
      - Count total bonuses
      - Sum total stated value
    - guarantee: MINIMUM REQUIRED
      - type: must be one of [extreme, strong, moderate, standard]
      - duration: string describing time period
      - branded_name: CRITICAL — must not be generic (flag if "money-back guarantee" without branding)
      - description: string
    - scarcity: IMPORTANT
      - type: must be one of 5 scarcity types
      - justification: CRITICAL — must provide REASON (not just claim)
      - specific_claim: the actual scarcity statement
    - urgency: OPTIONAL — triggers and deadline

**5C: Price Architecture Validation**
19. Check offer_data.price_architecture:
    - actual_price: MINIMUM REQUIRED — number > 0
    - anchor_price: IMPORTANT — number > actual_price
    - per_day_price: IMPORTANT — number (actual_price / days of supply)
    - multi_buy_tiers: OPTIONAL — array of tier objects
    - anchor_points: IMPORTANT — array of descending numbers for cascade

**5D: Naming Validation**
20. Check offer_data.naming:
    - product_name: Cross-validate with core_offer.product_name (must match)
    - branded_elements: OPTIONAL — list of branded sub-names
    - naming_timing: IMPORTANT — immediate or late

**5E: Presentation Script Validation**
21. Check offer_data.presentation_script:
    - sin_methodology: OPTIONAL — D-F-W-B-P sequence
    - presentation_order: OPTIONAL — ordered list of sections

### Step 6: Classify Offer Completeness

22. Based on Step 5 field-by-field assessment, classify:

**COMPLETE** — All of the following are true:
- product_name present
- product_format present
- core_description present
- positioning_statement present
- uniqueness_claim present
- actual_price present
- anchor_price present
- guarantee type + duration + branded_name present
- scarcity type + justification present
- At least product_name and core_offer descriptions populated

**PARTIAL** — Some but not all:
- At least 2 of 4 MINIMUM REQUIRED fields present (product_name, product_format, actual_price, guarantee)
- But one or more IMPORTANT fields missing

**ABSENT** — Either:
- offer_data.status is "absent" or "error"
- Fewer than 2 MINIMUM REQUIRED fields present
- No usable offer data whatsoever

### Step 7: Validate Minimum Required Data Across ALL Sources

23. The 4 MINIMUM REQUIRED fields must exist from SOME source before the skill can proceed (even with human input). Check all sources:

**Product Name:**
- Sources: offer_data.core_offer.product_name, offer_data.naming.product_name
- If found in ANY source: mark as AVAILABLE
- If found in NO source: mark as MUST_PROVIDE (human must supply)

**Core Offer Description:**
- Sources: offer_data.core_offer.core_description
- If found: mark as AVAILABLE
- If not found: mark as MUST_PROVIDE

**Price (at least actual_price):**
- Sources: offer_data.price_architecture.actual_price
- If found: mark as AVAILABLE
- If not found: mark as MUST_PROVIDE

**Guarantee (type + duration):**
- Sources: offer_data.enhancement_stack.guarantee
- If found with at least type + duration: mark as AVAILABLE
- If not found or incomplete: mark as MUST_PROVIDE

24. Compile minimum_data_assessment:
    - fields_available: count and list
    - fields_must_provide: count and list
    - minimum_met: true if all 4 either AVAILABLE or will be requested from human

### Step 8: Determine Checkpoint Default

25. Based on offer_completeness:

| Offer Completeness | Checkpoint Default | Rationale |
|--------------------|--------------------|-----------|
| COMPLETE | CONFIRM | All data present — human just needs to verify |
| PARTIAL | PARTIAL | Some data present — human fills gaps, confirms existing |
| ABSENT | PROVIDE | No offer data — human must provide all product details |

26. The checkpoint default tells microskill 0.4 how to present the interaction to the human.

### Step 9: Generate Human-Readable Gap Summary

27. For each gap in the gap_registry (from 0.1) and validation gaps (from Steps 3-5), generate a human-readable description:

```
Gap: product_name
Question: "What is the product name?"
Source attempted: offer-package.json (Skill 11)
Status: NOT FOUND

Gap: guarantee.branded_name
Question: "What is the branded name for the guarantee? (e.g., 'Empty Jar Guarantee', 'Triple-Back-Guarantee')"
Source attempted: offer-package.json (Skill 11)
Status: GENERIC — currently 'money-back guarantee' which is unacceptable per Principle 6
```

28. Group gaps by priority:
    - MUST_PROVIDE: Cannot proceed without these (4 minimum required)
    - SHOULD_PROVIDE: Significantly improves output quality
    - NICE_TO_HAVE: Optional enrichment

### Step 10: Compile Validation Report

29. Build validation_report:

```yaml
validation_report:
  timestamp: string
  overall_validation_state: enum[valid, valid_with_gaps, invalid_needs_human, blocked]
  offer_completeness: enum[complete, partial, absent]
  checkpoint_default: enum[confirm, partial, provide]
  mechanism_narrative_valid: boolean
  mechanism_data_valid: boolean
  structure_data_valid: boolean
  proof_inventory_valid: boolean
  tier1_intelligence_valid: boolean
  minimum_data_assessment:
    product_name: enum[available, must_provide]
    core_offer: enum[available, must_provide]
    price: enum[available, must_provide]
    guarantee: enum[available, must_provide]
    all_minimum_met: boolean
  gap_summary:
    must_provide: [object]
    should_provide: [object]
    nice_to_have: [object]
    total_gaps: integer
  conflict_summary:
    conflicts: [object]
    total_conflicts: integer
  field_completeness_score: float  # 0.0 to 1.0 — percentage of all fields populated
```

### Step 11: Pass to Human Checkpoint

30. Package validation_report with upstream_data_environment and tier1_intelligence_index
31. Pass to microskill 0.4 (Human Checkpoint Curator)

## QUALITY CRITERIA

| Criterion | Pass Condition |
|-----------|---------------|
| Required packages validated | mechanism_narrative and mechanism packages confirmed present |
| Offer completeness accurate | Classification matches actual field presence |
| Minimum data assessed | All 4 minimum required fields checked across all sources |
| Checkpoint default correct | Default matches offer completeness level |
| Gaps human-readable | Every gap has a clear question the human can answer |
| No false completeness | COMPLETE status only when ALL fields genuinely populated |
| No false absence | ABSENT only when genuinely no usable data exists |
| Conflicts surfaced | All cross-package conflicts identified and documented |
| Validation non-destructive | No upstream data modified — only annotated |
| Blocking conditions enforced | HALT raised for genuinely missing required data |

---

## FAILURE MODES

### Failure Mode 1: Partial Classified as Complete
- **Symptom:** Human checkpoint shows CONFIRM but crucial fields are actually missing
- **Cause:** Completeness check only verified top-level field presence, not sub-fields
- **Prevention:** Field-level validation in Step 5 (not just object-level)
- **Recovery:** Re-validate at sub-field granularity

### Failure Mode 2: Available Data Classified as Must-Provide
- **Symptom:** Human asked to provide data that already exists upstream
- **Cause:** Validator checked wrong path or missed alternative source
- **Prevention:** Check ALL possible sources for each minimum field
- **Recovery:** Re-check all source paths

### Failure Mode 3: Generic Guarantee Not Flagged
- **Symptom:** "Money-back guarantee" passes validation despite violating Principle 6
- **Cause:** Guarantee validation only checked field presence, not branding quality
- **Prevention:** Specific check for generic guarantee language in Step 5B
- **Recovery:** Add branded_name quality check to guarantee validation

### Failure Mode 4: Mechanism Name Conflict Reaches Downstream
- **Symptom:** Product introduction uses inconsistent mechanism naming
- **Cause:** Conflict detected but not resolved or flagged
- **Prevention:** Explicit conflict resolution in validation (default to Skill 11 value)
- **Recovery:** Force re-resolution, flag for human in checkpoint

### Example 2: PARTIAL Offer — Missing Guarantee Branding

```

  offer_completeness: PARTIAL
  checkpoint_default: PARTIAL
  minimum_data_assessment:
    product_name: AVAILABLE ("Resurge")
    core_offer: AVAILABLE ("Anti-aging nutritional protocol")
    price: AVAILABLE ($49)
    guarantee: AVAILABLE but DEGRADED
      - type: moderate (60-day)

    must_provide: 0
    should_provide: 3

      - scarcity.justification: "What justifies the scarcity? (ingredient cost, manufacturing, supply constraints)"
```

### Example 3: ABSENT Offer — New Campaign

```
Validation Report:
  offer_completeness: ABSENT
  checkpoint_default: PROVIDE
  minimum_data_assessment:
    product_name: MUST_PROVIDE
    core_offer: MUST_PROVIDE
    price: MUST_PROVIDE
    guarantee: MUST_PROVIDE

    must_provide: 4

      - product_format: "What format? (supplement, digital, physical, info, hybrid)"
      - positioning_statement: "One-sentence market position"
      - uniqueness_claim: "What makes this first/only/never-before?"
      - anchor_price: "What comparison or anchor price?"
      - bonuses: "List any bonuses with names and values"
      - scarcity_type: "What scarcity mechanism? What justifies it?"
      - per_day_price: "Price broken down per day?"
      - naming_timing: "Reveal product name early or late in copy?"
    nice_to_have: 5
    total_gaps: 17
  field_completeness_score: 0.0

  NOTE: Human checkpoint will default to PROVIDE — operator must supply all product data
```
