## PURPOSE

Presents the validated product details to the human operator for confirmation, modification, or provision. This is a BLOCKING microskill — execution cannot proceed past Layer 0 without a human response confirming the minimum required product data. The checkpoint adapts its presentation based on offer-completeness level: showing all data for confirmation (COMPLETE), showing existing data plus gaps for partial merge (PARTIAL), or presenting a blank form for full provision (ABSENT).

This microskill is the final safeguard against writing product introduction copy with incorrect, incomplete, or hallucinated product details. Every fact about the product — name, price, guarantee, bonuses, scarcity — must be human-confirmed before any copy is generated.

---

## IDENTITY

**This microskill IS:**
- A human interaction orchestrator that presents product data for confirmation
- A data merger that combines upstream data with human input
- A minimum-data enforcer that will not release execution without 4 core fields
- An adaptive presenter that adjusts display format to data completeness level
- The final gate before creative execution begins

**This microskill is NOT:**
- An offer architect (that is Skill 11)
- A data validator (that is 0.3 — this microskill presents what 0.3 validated)
- A copywriter — it does not generate any narrative text
- A suggestion engine — it presents what exists and asks for what is missing
- Optional or skippable — this checkpoint is MANDATORY in all runs

## PROCESS

### Step 1: Determine Presentation Mode

1. Read validation_report.checkpoint_default
2. Set presentation_mode:

| Checkpoint Default | Presentation Mode | Display Behavior |
|--------------------|--------------------|-----------------|
| CONFIRM | full_review | Show ALL product data. Ask: "Confirm, or modify any field?" |
| PARTIAL | gap_fill | Show existing data + highlight gaps. Ask: "Fill gaps and confirm existing." |
| PROVIDE | blank_form | Show mechanism context only. Ask: "Provide all product details." |

### Step 3: Present to Human Operator

7. Display the constructed document
8. Present four response options:

```
--- RESPONSE OPTIONS ---
[1] CONFIRM — All data is correct. Proceed with execution.
[2] MODIFY — Data exists but needs changes. Specify which fields to modify.
[3] PROVIDE — I will provide all product details from scratch.
[4] PARTIAL — Some data is correct, I will fill in the gaps.

Default: [checkpoint_default from 0.3]
```

9. WAIT for human response. This is a BLOCKING operation.
10. Do NOT proceed until response is received.
11. Do NOT timeout — the human may need time to gather product details.

### Step 4: Process Human Response

12. Parse human response:

**If CONFIRM (Option 1):**
- Verify all 4 minimum required fields are present in existing data
- If all present: Accept all existing data as confirmed
- If any missing: REJECT CONFIRM — inform human that minimum data is incomplete
  - "Cannot confirm: [field] is missing. Please use PARTIAL or PROVIDE to supply missing data."
  - Return to Step 3

**If MODIFY (Option 2):**
- Parse the modifications provided
- Apply modifications to existing data
- Re-validate modified data against minimum requirements
- If minimum met: Accept modified data as confirmed
- If minimum not met: Ask for remaining fields

**If PROVIDE (Option 3):**
- Accept all human-provided data
- Map to standard schema:
  - product_name → core_offer.product_name
  - core_description → core_offer.core_description
  - actual_price → price_architecture.actual_price
  - guarantee fields → enhancement_stack.guarantee
  - bonuses → enhancement_stack.bonuses
  - scarcity → enhancement_stack.scarcity
- Validate against minimum requirements
- If minimum met: Accept provided data as confirmed
- If minimum not met: Ask for remaining fields

**If PARTIAL (Option 4):**
- Identify which existing fields the human confirms
- Identify which new fields the human provides
- Merge: confirmed existing + new provided
- Re-validate merged data against minimum requirements
- If minimum met: Accept merged data as confirmed
- If minimum not met: Ask for remaining fields

### Step 5: Validate Minimum Data Requirement

13. After processing response, verify ALL 4 minimum required fields are present:
    - Product Name: non-empty string
    - Core Offer Description: non-empty string, length > 10
    - Price (actual_price): number > 0
    - Guarantee (type + duration): both present

14. If ANY of the 4 are still missing:
    - Display: "MINIMUM DATA NOT MET. The following fields are required to proceed:"
    - List missing fields with questions
    - Return to Step 3 (do NOT proceed)

15. If ALL 4 present:
    - Proceed to Step 6

### Step 6: Enrich Confirmed Data with Mechanism Context

16. Attach mechanism context to confirmed product data:
    - mechanism_name (for consistency in product introduction copy)
    - villain_name (for future pacing negative future)
    - root_cause_name (for future pacing)
    - bridge_text (for bridge moment writing)
    - emotional_state_at_handoff (for emotional architecture)
    - belief_level_at_handoff (for introduction type selection)

17. If the human provided a guarantee without a branded name:
    - Flag: "Guarantee branded name not provided. Skill will generate one based on TIER1 patterns."
    - Set guarantee.branded_name_source to "skill_generated"

18. If the human provided scarcity without justification:
    - Flag: "Scarcity justification not provided. Principle 7 requires justification."
    - Set scarcity.justification_source to "needs_generation"

## QUALITY CRITERIA

| Criterion | Pass Condition |
|-----------|---------------|
| Human response received | Checkpoint is BLOCKING — response must exist |
| Minimum data present | All 4 fields (name, offer, price, guarantee) confirmed |
| No hallucinated data | Every field traced to either upstream source or human input |
| Display appropriate to mode | CONFIRM shows all data; GAP_FILL highlights missing; PROVIDE shows blank form |
| Merge integrity | If PARTIAL, merged data is consistent (no conflicts between human and upstream) |
| Mechanism context attached | Confirmed package includes mechanism name, villain, bridge text |
| Provenance tracked | Every field shows its source (upstream / human / skill-generated) |
| Flags accurate | Missing branded guarantee name flagged; missing scarcity justification flagged |
| No silent gaps | Any gap not filled by human is explicitly flagged for skill generation |
| Blocking enforced | No passage to Layer 1 until minimum data confirmed |

## FAILURE MODES

### Failure Mode 1: Proceeding Without Human Response
- **Symptom:** Layer 1 executes with unconfirmed data
- **Cause:** Blocking mechanism bypassed or timeout implemented
- **Prevention:** NO timeout on human checkpoint. Explicit BLOCKING flag.
- **Recovery:** HALT and re-present checkpoint

### Failure Mode 2: Human Provides Incomplete Minimum Data
- **Symptom:** Copy generated without product name or price
- **Cause:** Minimum data check not enforced after human response
- **Prevention:** Post-response validation loop (Step 5) with re-presentation on failure
- **Recovery:** Return to Step 3, highlight specific missing fields

### Failure Mode 3: Merge Conflict in PARTIAL Mode
- **Symptom:** Human modifies a field, but upstream value persists alongside
- **Cause:** Merge logic applies additions but not overrides
- **Prevention:** Human input ALWAYS overrides upstream for the same field
- **Recovery:** Re-merge with override priority: human > upstream

### Failure Mode 4: Generic Guarantee Accepted Without Flag
- **Symptom:** "Money-back guarantee" passes checkpoint without branded_name flag
- **Cause:** Branded name check not performed post-human-input
- **Prevention:** Step 6 specifically checks guarantee branding regardless of source
- **Recovery:** Flag for skill generation in Layer 2

### Failure Mode 5: Data Provenance Lost
- **Symptom:** Downstream cannot determine which fields are human-confirmed vs. upstream
- **Cause:** Provenance tracking not implemented
- **Prevention:** Every field tagged with source in data_provenance
- **Recovery:** Re-trace each field to its source
