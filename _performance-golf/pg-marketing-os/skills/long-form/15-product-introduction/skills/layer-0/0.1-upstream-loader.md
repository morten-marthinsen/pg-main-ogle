# Microskill 0.1 — Upstream Loader

**Skill:** 15-product-introduction
**Microskill:** 0.1-upstream-loader
**Version:** 1.0
**Layer:** 0 — Foundation & Loading
**Position:** First microskill in execution chain
**Persona:** Dr. James Liu — Research Director

---

## PURPOSE

Loads all upstream dependency packages required by the Product Introduction skill: the offer package (Skill 11), mechanism narrative package (Skill 17), mechanism package (Skill 11), structure package (Skill 11), and proof inventory output (Skill 11). This microskill establishes the complete data environment from which all downstream microskills draw their inputs.

This is the FIRST microskill to execute. Nothing proceeds until upstream data is loaded, inventoried, and gap-assessed. Because Skill 06 (offer) is currently a placeholder skill, the offer-package may be incomplete or absent — this microskill must detect that condition and flag it for the human checkpoint in microskill 0.4.

---

## IDENTITY

**This microskill IS:**
- A data ingestion engine that loads 5 upstream JSON packages
- A schema validator that checks each package against expected structure
- A gap detector that identifies missing fields across all packages
- An offer-completeness assessor that determines the state of Skill 11 output
- The foundation layer that all subsequent microskills depend on

**This microskill is NOT:**
- A data transformer or interpreter (that happens in downstream layers)
- A human interaction point (that is 0.4)
- A TIER1 intelligence loader (that is 0.2)
- A validator of data quality or strategic correctness (that is 0.3)
- A decision-maker about which data to use when conflicts exist

## PROCESS

### Step 1: Initialize Loading Environment

1. Create the loading manifest with timestamp, run_id, and expected packages list
2. Set loading status for each of 5 packages to PENDING
3. Initialize the gap registry as empty
4. Initialize the conflict registry as empty

### Step 2: Load Offer Package (Skill 11)

5. Attempt to load `07-offer/outputs/offer-package.json`
6. If file NOT FOUND:
   - Set offer_package_status to ABSENT
   - Log: "Offer package not found — Skill 06 may not have been executed"
   - Set offer_completeness to ABSENT
   - Add ALL offer fields to gap registry
   - Continue to next package (do NOT halt)
7. If file FOUND but EMPTY or MALFORMED:
   - Set offer_package_status to ERROR
   - Log error details
   - Set offer_completeness to ABSENT
   - Add ALL offer fields to gap registry
8. If file FOUND and VALID JSON:
   - Parse JSON structure
   - Check for core_offer field presence
   - Check for enhancement_stack field presence
   - Check for price_architecture field presence
   - Check for naming field presence
   - Check for presentation_script field presence
   - For each PRESENT field, extract and store value
   - For each ABSENT field, add to gap registry
   - Determine offer_completeness:
     - COMPLETE: All 5 top-level fields present with required sub-fields populated
     - PARTIAL: Some fields present, some missing
     - ABSENT: File found but no usable data
   - Set offer_package_status to LOADED

### Step 3: Load Mechanism Narrative Package (Skill 17)

9. Attempt to load `14-mechanism-narrative/outputs/mechanism-narrative-package.json`
10. If file NOT FOUND:
    - Set mechanism_narrative_status to MISSING
    - Log CRITICAL error: "Mechanism narrative package is REQUIRED and not found"
    - Flag as BLOCKING dependency
11. If file FOUND and VALID:
    - Parse JSON
    - Extract downstream_handoffs.for_product_introduction
    - Verify presence of:
      - bridge_text (the narrative bridge from mechanism to product)
      - mechanism_context (summary of mechanism for product section continuity)
      - mechanism_name (for consistent naming)
      - proof_elements_established (what has already been proven upstream)
      - emotional_state_at_handoff (where the reader IS emotionally)
      - belief_level_at_handoff (how much they believe the mechanism)
    - For each missing sub-field, add to gap registry
    - Set mechanism_narrative_status to LOADED

### Step 4: Load Mechanism Package (Skill 11)

12. Attempt to load `04-mechanism/outputs/mechanism-package.json`
13. If file NOT FOUND:
    - Set mechanism_package_status to MISSING
    - Log CRITICAL error: "Mechanism package is REQUIRED and not found"
    - Flag as BLOCKING dependency
14. If file FOUND and VALID:
    - Parse JSON
    - Extract winner mechanism details:
      - mechanism_name
      - mechanism_type
      - mechanism_description
      - villain_name
      - root_cause_name
    - CONSISTENCY CHECK: Compare mechanism_name from Skill 03 with mechanism_name from Skill 17 handoff
      - If MATCH: Log confirmation
      - If MISMATCH: Add to conflict registry with both values
    - Set mechanism_package_status to LOADED

### Step 5: Load Structure Package (Skill 11)

15. Attempt to load `08-structure/outputs/structure-package.json`
16. If file NOT FOUND:
    - Set structure_package_status to MISSING
    - Log WARNING: "Structure package not found — product section placement unknown"
    - Flag as NON-BLOCKING but DEGRADED
17. If file FOUND and VALID:
    - Parse JSON
    - Extract campaign_thesis
    - Extract flow_architecture fields:
      - sin_segue (how the narrative transitions)
      - product_introduction_placement (where in the overall flow)
      - estimated_word_count_range (target length for product section)
    - Extract downstream_handoffs.for_product_introduction
    - Set structure_package_status to LOADED

### Step 6: Load Proof Inventory (Skill 11)

18. Attempt to load `02-proof-inventory/outputs/proof-inventory-output.json`
19. If file NOT FOUND:
    - Set proof_inventory_status to MISSING
    - Log WARNING: "Proof inventory not found — component proof packages may be limited"
    - Flag as NON-BLOCKING but DEGRADED
20. If file FOUND and VALID:
    - Parse JSON
    - Extract proof_assets by category:
      - clinical_trials (for component reveal proof packages)
      - testimonials (for social proof in product section)
      - authority_endorsements (for credibility signals)
      - statistics (for specific claims)
      - demonstrations (for usage demonstration pattern)
    - Extract proof_density_score
    - Extract available_for_product_section subset
    - Set proof_inventory_status to LOADED

### Step 7: Cross-Package Consistency Validation

21. If mechanism_name exists in BOTH Skill 03 and Skill 11 outputs:
    - Compare for exact match
    - If mismatch, flag conflict and use Skill 11 value (most recent upstream)
22. If product_name exists in BOTH Skill 06 and any other source:
    - Verify consistency
    - Flag any discrepancies
23. If proof_elements_established (Skill 17) references proofs not in proof_inventory (Skill 11):
    - Log WARNING: "Proof inventory may be incomplete — mechanism narrative references proofs not in inventory"
24. Compile all conflicts into conflict_registry

### Step 8: Assess Overall Loading State

25. Count packages by status:
    - LOADED: Package found, parsed, data extracted
    - MISSING: Package not found (check if BLOCKING or NON-BLOCKING)
    - ERROR: Package found but corrupted/malformed
    - ABSENT: Specifically for offer-package when Skill 06 not run
26. Determine overall_loading_state:
    - COMPLETE: All 5 packages loaded with no gaps
    - OPERATIONAL: Required packages (11, 03) loaded; offer may be partial/absent
    - DEGRADED: Required packages loaded but structure and/or proof missing
    - BLOCKED: One or more REQUIRED packages (11, 03) missing — HALT execution
27. If BLOCKED:
    - Generate blocking_error with specific missing package(s)
    - Recommend: Run missing upstream skill(s) before proceeding
    - HALT — do not pass to next microskill

### Step 9: Load Campaign Brief

34. Attempt to load `09-campaign-brief/outputs/campaign-brief-package.json`
35. If file FOUND and VALID:
    - Extract `campaign_thesis`, `tone_direction`, `avatar_summary`
    - Extract `market_sophistication` (Schwartz stage 1-5)
    - Extract `proof_approach`, `promise_ceiling`
    - Set campaign_brief_status to LOADED
36. If file NOT FOUND:
    - Set campaign_brief_status to MISSING
    - Log WARNING: "Campaign brief not found — product positioning may drift"
    - Continue — non-blocking

### Step 10: Load Mechanism Narrative Assembled Prose (for continuity)

37. Attempt to load `14-mechanism-narrative/outputs/mechanism-narrative-assembled-prose.md`
38. If file FOUND:
    - Load full text of mechanism narrative prose
    - Extract for continuity:
      - How the mechanism was explained (to NOT re-explain in product intro)
      - Metaphors and analogies used (to AVOID repeating)
      - Proof elements cited (to AVOID re-citing)
      - The last paragraph/bridge text (for seamless transition)
      - What the reader now understands about the mechanism (to ASSUME, not repeat)
    - Set mechanism_prose_status to LOADED
39. If file NOT FOUND:
    - Set mechanism_prose_status to NOT_AVAILABLE
    - Log WARNING: "Mechanism narrative prose not available. Product introduction will rely on bridge_text only."
    - Continue — non-blocking but degraded

**Why this matters:** The product introduction must ASSUME the reader understands the mechanism (from the previous section). Without reading the mechanism prose, the product intro may re-explain the mechanism, which is the Pattern 4 duplication issue identified in the deduplication audit.

### Step 11: Load Context Reservoir (if available)

40. Attempt to load `outputs/[project-code]/context-reservoir.md`
41. If file FOUND:
    - Load relevant sections for product introduction writing
    - Extract: voc_quotes, proof_elements_detailed, expression_anchoring
    - Set context_reservoir_status to LOADED
42. If file NOT FOUND:
    - Set context_reservoir_status to NOT_AVAILABLE
    - Continue — non-blocking

### Step 12: Compile Loading Report

28. Build loading_report with:
    - timestamp
    - run_id
    - packages_loaded: count and list
    - packages_missing: count and list
    - packages_errored: count and list
    - offer_completeness: COMPLETE / PARTIAL / ABSENT
    - gap_registry: all missing fields by package
    - conflict_registry: all cross-package conflicts
    - overall_loading_state: COMPLETE / OPERATIONAL / DEGRADED / BLOCKED
    - blocking_issues: list of issues preventing progress (if any)
    - recommended_checkpoint_default: CONFIRM / PARTIAL / PROVIDE (based on offer_completeness)

### Step 10: Pass to Next Microskill

29. Package all loaded data into upstream_data_environment
30. Attach loading_report as metadata
31. Pass to microskill 0.2 (TIER1 Intelligence Loader)

## FAILURE MODES

### Failure Mode 1: Offer Package Assumed Present When Absent
- **Symptom:** Downstream microskills attempt to read offer fields that are null
- **Cause:** offer_completeness not properly set to ABSENT
- **Prevention:** Explicit file-existence check before any parsing attempt
- **Recovery:** Re-run loader with explicit ABSENT flag

### Failure Mode 2: Required Package Missing Not Detected
- **Symptom:** Execution proceeds without mechanism narrative or mechanism data
- **Cause:** Missing package not flagged as BLOCKING
- **Prevention:** Hardcoded REQUIRED status for packages 11 and 03
- **Recovery:** HALT execution, surface blocking error

### Failure Mode 3: Mechanism Name Conflict Ignored
- **Symptom:** Product introduction uses different mechanism name than mechanism narrative
- **Cause:** Conflict between Skill 03 and Skill 11 names not detected
- **Prevention:** Explicit cross-package consistency check in Step 7
- **Recovery:** Flag conflict, default to Skill 11 value (most recent)

### Failure Mode 4: Partial Offer Treated as Complete
- **Symptom:** Human checkpoint skipped or defaulted to CONFIRM when data is actually incomplete
- **Cause:** Completeness check not granular enough
- **Prevention:** Check each of 5 top-level fields AND their required sub-fields
- **Recovery:** Re-assess completeness with field-level granularity

### Failure Mode 5: Proof Inventory Disconnect
- **Symptom:** Mechanism narrative references proofs not available in proof inventory
- **Cause:** Proof inventory outdated or mechanism narrative used ad-hoc proofs
- **Prevention:** Cross-reference proof_elements_established with proof_assets
- **Recovery:** Log warning, flag for human review during checkpoint
