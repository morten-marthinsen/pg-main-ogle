## PURPOSE

Selects the component reveal pattern that will structure how the product's ingredients, features, steps, or elements are presented. The critical insight from TIER1 analysis is that components are PROOF, not features (Principle 4). This microskill selects the pattern that transforms a simple ingredient/feature list into a parade of evidence, where each component carries its own mini-proof package demonstrating that the mechanism works.

The selection is based on product format, component count, proof availability per component, and TIER1 pattern matching. The output drives microskill 2.3 (Component Reveal Writer).

---

## IDENTITY

**This microskill IS:**
- A component reveal pattern selector choosing from 5 TIER1-proven patterns
- A proof-per-component assessor that evaluates whether each component has supporting evidence
- A "features-to-proof" transformer that ensures components serve as evidence, not descriptions
- A structure selector that determines the architecture of the component section

**This microskill is NOT:**
- A component copywriter (that is 2.3)
- A proof gatherer (that is Skill 11)
- A product ingredient researcher — it works with confirmed components from 0.4
- A feature-benefit mapper — it maps components to PROOF, not benefits
- A mechanism explainer — that was done upstream in Skills 10-11

## PROCESS

### Step 1: Inventory Components and Their Proof

1. Extract the product's components from confirmed_product_package:
   - Supplements: ingredients with dosages
   - Digital: modules, tracks, sessions, tools
   - Physical: features, technologies, design elements
   - Info products: lessons, steps, drills, techniques
   - Hybrid: combination of above

2. For EACH component, assess available proof:

```yaml
component_proof_inventory:
  - component_name: string
    component_type: enum[ingredient, feature, step, module, technology]
    proof_available:
      clinical_trial: boolean
      authority_endorsement: boolean
      statistic: boolean
      demonstration: boolean
      discovery_story: boolean
      comparison_data: boolean
    proof_density: enum[rich, moderate, sparse, none]
    proof_details: [string]  # Specific proof items
```

3. Classify overall proof density:
   - RICH: 70%+ of components have at least 1 proof element
   - MODERATE: 40-69% of components have proof
   - SPARSE: 10-39% of components have proof
   - INSUFFICIENT: <10% of components have proof

4. If INSUFFICIENT:
   - Flag: "Component proof density is insufficient — component reveal will be thin"
   - Consider feedback request to Skill 01 for additional proof gathering

### Step 2: Score Each Component Reveal Pattern

5. Score each of the 5 patterns on a 0-10 compatibility scale:

**PATTERN 1: THREE-PRONG DEFENSE**
"Components organized into named framework with named prongs/categories."
```
Score inputs:
  + Components naturally group into 3-4 categories (+3)
  + Product has a framework name or can be named (+2)
  + Product format is supplement or physical with distinct subsystems (+2)
  + Each category has at least one proof element (+2)
  + Introduction type is manufacturing_quest or formula_development (+1)
  - Components don't naturally cluster (-4)
  - Fewer than 6 total components (can't split meaningfully) (-2)
  - No framework naming possible (-2)
Best fit: Supplements with 8+ ingredients that cluster into functions; physical products with distinct technology layers
Controls: Gundry "Three-Prong Defense" (Repair + Absorb + Rebalance), 357 "Tri-Fusion Technology" (Face + Cup + Crown)
```

**PATTERN 2: INGREDIENT DEEP DIVE**
"Each ingredient/component gets its own mini-section with its own proof."
```
Score inputs:
  + Each component has its own unique proof element (+3)
  + Proof is diverse (different type per component) (+2)
  + Component count is 3-8 (enough for depth, not too many) (+2)
  + Authority endorsements available per component (+2)
  + Product format is supplement (+1)
  - Components lack individual proof elements (-4)
  - Component count exceeds 10 (too many for deep dive) (-3)
  - Proof is repetitive across components (-2)
Best fit: Supplements with 4-8 key ingredients, each backed by unique authority or study
Controls: Montezuma's Secret (Cacao → Maca → Ginseng → Epimedium, each with authority), Gundry (Pepsin GI spotlight, L-glutamine dosage, Cloves antioxidant score)
```

**PATTERN 3: STEP-BY-STEP SYSTEM**
"Components revealed as sequential steps building on each other."
```
Score inputs:
  + Product is a method, system, or training (+4)
  + Components are sequential (step 1 before step 2) (+3)
  + Each step has a named drill, technique, or action (+2)
  + Product format is info_product or digital (+1)
  - Components are parallel, not sequential (-3)
  - Product is supplement (no "steps" to taking a pill) (-4)
  - Steps are generic without proprietary naming (-2)
Best fit: Info products, training programs, method-based digital products
Controls: PSS (Towel Turn → Tee to Toe → Reverse Release), SSS (No-Turn Backswing → Low Point Control), Superbrain (30-day stacking)
```

**PATTERN 4: USAGE DEMONSTRATION**
"Components revealed through demonstration of HOW TO USE the product."
```
Score inputs:
  + Product has a clear usage process (+3)
  + Usage is simple (3-5 steps max) (+2)
  + Product format is physical or digital with device/app (+2)
  + Simplicity of use is a selling point (+2)
  + Demonstration creates phantom ownership (+1)
  - Product is supplement (usage is "take pill") (-2)
  - Usage is complex or requires explanation (-2)
  - No visual or tactile experience to demonstrate (-2)
Best fit: Physical devices, apps, audio programs, beauty/health devices
Controls: no!no! PRO (Choose Tip → Glide → Buff), BBW (Headphones, 7 min, morning coffee)
```

**PATTERN 5: NUTRIENT PROTOCOL**
"Precise clinical protocol with exact dosages and scientific backing."
```
Score inputs:
  + Product is supplement with specific dosages (+4)
  + Clinical trial data supports specific amounts (+3)
  + Component count is 4-16 (protocol scope) (+1)
  + Scientific credibility is primary selling angle (+2)
  + Introduction type is formula_development (+1)
  - No specific dosages available (-4)
  - No clinical backing for amounts (-3)
  - Product is not supplement (-5)
Best fit: Supplements with clinically backed formulas and specific dosage data
Controls: Resurge (8 nutrients exact amounts, arginine + lysine = 695% HGH), Gundry (L-glutamine 213mg, 16 ingredients with dosages)
```

### Step 3: Assess Pattern-Component Count Alignment

6. Cross-reference selected pattern with actual component count:

| Pattern | Ideal Component Count | Too Few | Too Many |
|---|---|---|---|
| three_prong_defense | 8-16 (grouped into 3-4 prongs) | <6 | >20 |
| ingredient_deep_dive | 3-8 (each gets spotlight) | <3 | >10 |
| step_by_step_system | 2-5 steps | <2 | >7 |
| usage_demonstration | 2-5 usage steps | <2 | >5 |
| nutrient_protocol | 4-16 nutrients | <4 | >20 |

7. If component count falls outside ideal range:
   - If "too few": Consider combining with another pattern or expanding component descriptions
   - If "too many": Consider grouping (shift to three_prong_defense) or selecting top components for spotlight

### Step 4: Map Per-Component Proof Packages

8. For each component, define the proof package it will carry:

```yaml
component_proof_map:
  - component: string
    proof_package:
      proof_type: enum[clinical, authority, statistic, story, comparison, dosage]
      proof_content: string
      proof_source: string
    sequence_position: integer  # Order in reveal
    word_count_estimate: integer  # 50-150 words per component
    proof_strength: enum[strong, moderate, weak]
```

9. Sequence components by proof strength:
   - STRONGEST proof component goes FIRST (sets credibility expectation)
   - WEAKEST proof component goes in the MIDDLE (protected by surrounding strength)
   - SECOND STRONGEST goes LAST (ends on high note)
   - Exception: step_by_step_system must maintain logical step order

10. If any component has NO proof:
    - Flag: "Component [X] lacks proof — treat as mechanism-connection point, not proof point"
    - Connect to mechanism instead: "This ingredient activates [mechanism element] by..."

### Step 5: Determine Framework Naming (If Applicable)

11. If selected pattern is three_prong_defense:
    - Generate framework name candidates based on:
      - Number of prongs (3 = "Triple," "Three-Prong," "Tri-")
      - Product positioning (defense, fusion, system, protocol)
      - Mechanism language (terms from mechanism name)
    - Examples from TIER1:
      - "Three-Prong Defense" (Gundry — Repair, Absorb, Rebalance)
      - "Tri-Fusion Technology" (357 — Face, Cup, Crown)
    - Select or recommend framework name
    - Name each prong with a descriptive label

12. If selected pattern is step_by_step_system:
    - Each step needs a NAMED technique/drill
    - Examples from TIER1:
      - "Towel Turn" (PSS Step 1)
      - "Tee to Toe Drill" (PSS Step 2)
      - "No-Turn Backswing" (SSS Component 1)
    - Confirm each step has a proprietary name

### Step 6: Generate Component Reveal Selection

13. Compile component_reveal_selection:


## FAILURE MODES

### Failure Mode 1: Components Listed as Features, Not Proof
- **Symptom:** Component section reads like a spec sheet or ingredient label
- **Cause:** Proof packages not mapped to each component
- **Prevention:** Step 4 requires proof_package for every component
- **Recovery:** Re-map with proof elements; flag components lacking proof

### Failure Mode 2: Pattern Selected Without Component Count Check
- **Symptom:** Ingredient deep dive attempted with 16 ingredients (too many for individual spotlight)
- **Cause:** Component count not cross-referenced with pattern ideal range
- **Prevention:** Step 3 alignment check with "too few" / "too many" handling
- **Recovery:** Shift to three_prong_defense (group 16 into 3-4 categories)

### Failure Mode 3: Step-by-Step System for Non-Sequential Product
- **Symptom:** Supplement ingredients presented as "Step 1: Take L-glutamine, Step 2: Take zinc"
- **Cause:** Pattern selected based on format default without checking component nature
- **Prevention:** Sequential nature check in scoring (+3 for sequential, -3 for parallel)
- **Recovery:** Shift to nutrient_protocol or ingredient_deep_dive

### Failure Mode 4: No Proof Available for Any Component
- **Symptom:** Component section is entirely feature-based
- **Cause:** Proof inventory empty or disconnected from components
- **Prevention:** Proof density check in Step 1 with INSUFFICIENT flag
- **Recovery:** Request feedback to Skill 01 for proof gathering; fall back to mechanism-connection approach
