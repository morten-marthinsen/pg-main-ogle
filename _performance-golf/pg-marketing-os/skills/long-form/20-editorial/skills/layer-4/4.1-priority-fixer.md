# Microskill 4.1: Priority Fixer

**Layer:** 4 (Fix Application)
**Purpose:** Execute fixes in priority order, starting with critical issues

---

## FIX APPLICATION PROTOCOL

### Fix Order of Operations

```
1. CRITICAL fixes first (copy-breaking issues)
2. MAJOR fixes second (effectiveness issues)
3. MINOR fixes if time permits (polish issues)
4. STYLISTIC fixes last (preference-level)
```

**IMPORTANT:** Each fix must be applied one at a time with verification before moving to the next.

---

## FIX CLASSIFICATION SYSTEM (Learning #49 — MANDATORY)

**CRITICAL:** Before applying ANY fix, classify it into one of three categories:

```yaml
fix_classification:
  AUTO_APPLY:
    description: "Safe changes that can be applied without approval"
    criteria:
      - "Obvious typos (e.g., 'weWe' → 'we')"
      - "Formatting errors (e.g., '$24999' → '$249.99')"
      - "Clear grammatical errors"
      - "Punctuation fixes"
    action: "Apply directly to edited copy"
    tracking: "List in changes.md with [CHANGED] marker"

  APPLY_WITH_NOTE:
    description: "Applied but flagged for client awareness"
    criteria:
      - "Slop word removal (e.g., 'absolutely', 'incredible')"
      - "Word substitutions for clarity"
      - "Structural tweaks (paragraph reordering)"
      - "CTA variations"
      - "P.S. additions"
      - "Duplicate content removal"
    action: "Apply to edited copy"
    tracking: "List in changes.md with reason explained"

  APPROVAL_REQUIRED:
    description: "NEVER apply without explicit human approval"
    criteria:
      - "Root cause terminology changes"
      - "Mechanism naming modifications"
      - "Big idea alterations"
      - "Promise rewording"
      - "Anchor phrase changes (e.g., 'toe lag' → 'Hidden Slice Trigger')"
      - "Major positioning shifts"
      - "Villain identification changes"
      - "Core narrative modifications"
    action: "DO NOT apply to edited copy — keep original text"
    tracking: "List in changes.md as [APPROVAL REQUIRED] suggestion only"

classification_gate:
  before_each_fix:
    - "Identify fix classification"
    - "If APPROVAL_REQUIRED → present as suggestion, keep original"
    - "If AUTO_APPLY or APPLY_WITH_NOTE → apply and document"
```

### Major Element Protection (Learning #46)

```yaml
major_elements_protected:
  elements:
    - root_cause: "The underlying problem being addressed"
    - mechanism: "How the solution works"
    - big_idea: "The core marketing concept"
    - promise: "What's being promised to the prospect"
    - anchor_phrases: "Memorable terminology (e.g., 'toe lag', 'SliceFix')"
    - villain: "Who/what is being blamed"
    - positioning: "How the product is positioned vs. alternatives"

  rule: |
    NEVER change any major element without explicit human approval.
    Even if the change seems like an improvement, it may conflict with:
    - Strategic positioning decisions made upstream
    - Market testing results
    - Client preferences we don't have visibility into
    - Legal/compliance requirements

  violation_consequence: |
    Client rejection, wasted revision cycles, trust erosion.
    The 'Hidden Slice Trigger' change in SF2 VSL is a cautionary example.
```

## FIX VERIFICATION CHECKLIST

After each fix, verify:

```yaml
fix_verification:
  issue_id: "[ID]"

  checklist:
    - check: "Does the fix address the identified issue?"
      result: "[yes | partially | no]"

    - check: "Does the surrounding text still flow?"
      result: "[yes | needs adjustment | no]"

    - check: "Are there any grammar/spelling issues in the new text?"
      result: "[clean | issues found]"

    - check: "Does the fix maintain brand voice?"
      result: "[yes | needs adjustment | no]"

    - check: "Does the fix maintain section balance?"
      result: "[yes | now too long/short]"

    - check: "Are all callbacks/threads still intact?"
      result: "[yes | broken | needs repair]"

  overall_verdict: "[APPROVED | NEEDS REVISION | ROLLBACK]"
```

## FIX QUALITY STANDARDS

### Minimum Quality Requirements

Each fix must:

1. **Resolve the issue** - The identified problem must be fixed
2. **Maintain flow** - Surrounding text must still read naturally
3. **Preserve voice** - Brand/writer voice must remain consistent
4. **Not create new issues** - Fix shouldn't introduce new problems
5. **Be verifiable** - We can confirm the fix worked

### Quality Failure Response

```yaml
quality_failure_response:
  issue_id: "[ID]"
  failure_type: "[which quality standard failed]"

  options:
    option_1: "Revise fix approach"

    option_4: "Flag for human review"

  selected_option: "[which option]"
  rationale: "[why this option]"
  new_approach: "[if revising, what's the new approach]"
```
