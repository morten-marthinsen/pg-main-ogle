# Skill 14 — Microskill 0.3: Input Validator

**Skill:** 11 — Mechanism Narrative
**Microskill:** 0.3 — Input Validator
**Version:** 1.0
**Layer:** 0 (Foundation & Loading)
**Position:** Third microskill in pipeline
**Type:** Schema Validation & Gap Analysis
**Dependencies:** 0.1-upstream-loader output, 0.2-tier1-intelligence-loader output
**Output:** validation-report.json

---

## PURPOSE

Validates all loaded inputs against the schema requirements for mechanism narrative generation. This microskill performs comprehensive completeness checks, cross-validates data consistency between upstream sources, identifies gaps that need human resolution, and determines whether the pipeline can proceed automatically or requires the human checkpoint (0.4) to intervene. It is the quality gate between loading and execution.

---

## IDENTITY

**What this IS:**
- A schema enforcement layer that verifies data completeness and consistency
- A gap analyzer that identifies what is missing and how critical each gap is
- The decision point for whether human checkpoint activation is needed

**What this is NOT:**
- Not a data loader — all data is already loaded by 0.1 and 0.2
- Not a content evaluator — does not judge quality of mechanism strategy, only completeness
- Not a corrector — flags problems but does not fix them
- Not the human checkpoint — it decides IF checkpoint is needed, but 0.4 executes it

---

## PROCESS

### Step 1: Validate Mechanism Winner — Core Fields

```
CHECK mechanism.name:
  - Is non-empty string: PASS/FAIL
  - Length >= 2 characters: PASS/FAIL
  - Contains no placeholder text ("TBD", "TODO", "[name]"): PASS/FAIL
  SEVERITY if fail: CRITICAL — cannot generate narrative without mechanism name

CHECK mechanism.type:
  - Is valid enum value: PASS/FAIL
  - Valid values: [biological, compound, technology, system, behavioral, frequency]
  SEVERITY if fail: HIGH — affects narrative type classification in 1.1

CHECK mechanism.explanation:
  CHECK mechanism.explanation.core_logic:
    - Is non-empty: PASS/FAIL
    - Length >= 50 characters: PASS/FAIL
    - Contains cause-effect language: PASS/WARN
    SEVERITY if fail: CRITICAL — core logic is foundation of explanation

  CHECK mechanism.explanation.analogy:
    - Is non-empty: PASS/FAIL
    - Contains concrete imagery (not abstract): PASS/WARN
    SEVERITY if fail: HIGH — analogy needed for simplification

  CHECK mechanism.explanation.step_by_step:
    - Array length >= 2: PASS/FAIL
    - Each step is non-empty: PASS/FAIL
    SEVERITY if fail: MEDIUM — can construct from core_logic if missing

  CHECK mechanism.explanation.scientific_backing:
    - Is non-empty: PASS/FAIL
    - References at least one institution, study, or researcher: PASS/WARN
    SEVERITY if fail: MEDIUM — affects proof integration quality
```

### Step 2: Validate Mechanism Winner — Strategic Fields

```
CHECK mechanism.proof_elements:
  - Array length >= 1: PASS/FAIL
  - Each element has: claim, source, strength_score: PASS/WARN
  - At least 3 elements for adequate proof integration: PASS/WARN
  SEVERITY if fail: HIGH — proof integration phase will be thin

CHECK mechanism.e5_strategy:
  CHECK mechanism.e5_strategy.e_level:
    - Is valid enum [E2, E3, E4, E5]: PASS/FAIL
    - Is not null or empty: PASS/FAIL
    SEVERITY if fail: HIGH — e_level drives narrative type, frame, depth decisions

  CHECK mechanism.e5_strategy.strategy_name:
    - Is non-empty: PASS/FAIL
    - Matches e_level (E2→New Application, E3→Reframe, E4→Proprietary, E5→Paradigm Shift): PASS/WARN
    SEVERITY if fail: MEDIUM — can infer from e_level

  CHECK mechanism.e5_strategy.sophistication_context:
    - Is non-empty: PASS/WARN
    SEVERITY if fail: LOW — helpful but not blocking

CHECK mechanism.copy_integration_map:
  - Has lead_teaser: PASS/WARN
  - Has body_reveal: PASS/WARN
  - Has offer_connection: PASS/WARN
  - Has close_callback: PASS/WARN
  SEVERITY if fail: MEDIUM — can construct default integration map

CHECK mechanism.scorecard:
  - Has average: PASS/FAIL
  - Average is between 1.0 and 10.0: PASS/FAIL
  - At minimum visual_metaphor and simplicity scores present: PASS/WARN
  SEVERITY if fail: MEDIUM — scorecard informs calibration decisions
```

### Step 3: Validate Root Cause Narrative Handoff

```
CHECK root_cause_handoff.bridge_text:
  - Is non-empty: PASS/FAIL
  - Length >= 50 characters: PASS/FAIL
  - Ends with complete sentence (period, question mark, or ellipsis): PASS/WARN
  - Does NOT introduce the mechanism (that is Skill 11's job): PASS/FAIL
  SEVERITY if fail: CRITICAL — bridge_text is the literal transition point

CHECK root_cause_handoff.root_cause_context:
  - Is non-empty: PASS/FAIL
  - Contains identifiable root cause statement: PASS/WARN
  SEVERITY if fail: HIGH — mechanism must address THIS root cause

CHECK root_cause_handoff.villain_established:
  - Object is non-null: PASS/FAIL
  - Has villain name: PASS/FAIL
  - Has villain type: PASS/WARN
  - Has villain characteristics or description: PASS/WARN
  SEVERITY if fail: HIGH — villain-mechanism pairing in 3.2 requires villain

CHECK root_cause_handoff.emotional_state_at_handoff:
  - Is non-empty: PASS/FAIL
  - Maps to recognized emotional state:
    Valid states: [fear, anxiety, shame, inadequacy, frustration,
      exhaustion, curiosity, intrigue, empathy, connection,
      anger, indignation, confusion, hopelessness]
  - Can be mapped to one of 5 emotional arcs: PASS/WARN
  SEVERITY if fail: HIGH — emotional arc design in 1.3 requires starting state

CHECK root_cause_handoff.belief_state:
  - Is non-empty: PASS/WARN
  SEVERITY if fail: MEDIUM — helps calibrate explanation depth

CHECK root_cause_handoff.questions_opened:
  - Array length >= 1: PASS/WARN
  - Each question is a genuine unanswered question: PASS/WARN
  SEVERITY if fail: LOW — helpful for problem amplification but not blocking
```

### Step 4: Validate Structure Data

```
CHECK structure.campaign_thesis:
  - Is non-empty: PASS/FAIL
  - Length >= 20 characters: PASS/WARN
  SEVERITY if fail: MEDIUM — mechanism narrative should align with thesis

CHECK structure.mechanism_chunks:
  - Array length >= 1: PASS/FAIL
  - Each chunk has identifier and word_budget: PASS/WARN
  SEVERITY if fail: MEDIUM — can use default word budgets

CHECK structure.format:
  - Is valid enum [vsl, magalog, advertorial, sales_letter, email_series]: PASS/FAIL
  SEVERITY if fail: MEDIUM — affects pacing and formatting decisions

CHECK structure.word_budget:
  - Mechanism section has allocated words: PASS/WARN
  - Allocated words >= 400 (minimum for adequate mechanism narrative): PASS/WARN
  SEVERITY if fail: LOW — can use default budgets
```

### Step 5: Validate Proof Inventory

```
CHECK proof_inventory.mechanism_relevant:
  - Array length >= 3: PASS/WARN
  - Each element has: claim, source_citation, category: PASS/WARN
  SEVERITY if fail: MEDIUM — proof integration will be thinner

CHECK proof_inventory.by_category:
  - At least 2 categories have elements: PASS/WARN
  - institutional category has elements: PASS/WARN (needed for stacking)
  SEVERITY if fail: LOW — single-category proof can still work

CHECK proof_inventory.total_count:
  - >= 3: PASS/WARN
  - >= 5: optimal
  SEVERITY if fail: LOW — can proceed with fewer proof elements
```

### Step 6: Validate TIER1 Intelligence Index

```
CHECK tier1_intelligence_index.metadata.parts_indexed:
  - == 11: PASS/FAIL
  SEVERITY if fail: CRITICAL — TIER1 index IS the teaching foundation

CHECK each index is populated:
  - narrative_types: has 7 entries: PASS/FAIL
  - explanation_sequence: has 5 entries: PASS/FAIL
  - framing_patterns: has 8 entries: PASS/FAIL
  - proof_strategies: has 6 entries: PASS/FAIL
  - naming_architectures: has 5 entries: PASS/FAIL
  - simplification_techniques: has 6 entries: PASS/FAIL
  - bridge_patterns: has 4 entries: PASS/FAIL
  - emotional_arcs: has 5 entries: PASS/FAIL
  - e_level_matrix: has 4 entries: PASS/FAIL
  - villain_mechanism_pairings: has 4 entries: PASS/FAIL
  - decision_matrix: is populated: PASS/FAIL
  SEVERITY if any fail: HIGH — missing patterns degrade downstream decisions

CHECK cross_references are built:
  - narrative_type_to_frame exists: PASS/WARN
  - e_level_to_narrative_type exists: PASS/WARN
  - arc_to_handoff_state exists: PASS/WARN
  SEVERITY if fail: MEDIUM — downstream skills can still function without cross-refs
```

### Step 7: Cross-Validation Checks

```
CROSS-CHECK 1: Mechanism E-level matches TIER1 matrix recommendations
  - Lookup e_level in e_level_matrix
  - Verify mechanism.type is compatible with recommended narrative_types
  - IF mismatch: WARN — may indicate unusual combination requiring human judgment

CROSS-CHECK 2: Root cause villain is compatible with mechanism
  - Verify villain_established.type can pair with mechanism.type
  - Using villain_mechanism_pairings, check at least one architecture fits
  - IF no pairing fits: WARN — 3.2 will need creative solution

CROSS-CHECK 3: Emotional handoff state maps to available arc
  - Lookup emotional_state_at_handoff in arc_to_handoff_state
  - Verify at least one arc starts from this state
  - IF no arc maps: WARN — 1.3 will need to construct custom arc

CROSS-CHECK 4: Proof elements sufficient for E-level requirements
  - E2 needs most proof (>= 5 elements recommended)
  - E5 needs least proof (>= 2 elements acceptable)
  - Compare available count to E-level requirement
  - IF insufficient: WARN — proof integration will be lighter than optimal

CROSS-CHECK 5: Word budget sufficient for E-level depth
  - E5 paradigm shift needs most words (>= 2000 for mechanism)
  - E2 needs fewest (>= 400)
  - Compare allocated budget to E-level minimum
  - IF insufficient: WARN — may need to compress explanation phases
```

### Step 8: Calculate Completeness Score

```
CALCULATE completeness_score:
  CRITICAL fields (weight 3x each):
    - mechanism.name
    - mechanism.explanation.core_logic
    - mechanism.e5_strategy.e_level
    - root_cause_handoff.bridge_text
    - root_cause_handoff.villain_established
    - root_cause_handoff.emotional_state_at_handoff
    - tier1_intelligence_index (all 11 parts)

  HIGH fields (weight 2x each):
    - mechanism.type
    - mechanism.explanation.analogy
    - mechanism.proof_elements (>= 1)
    - root_cause_handoff.root_cause_context
    - structure.format

  MEDIUM fields (weight 1x each):
    - mechanism.explanation.step_by_step
    - mechanism.explanation.scientific_backing
    - mechanism.copy_integration_map
    - mechanism.scorecard
    - structure.campaign_thesis
    - structure.mechanism_chunks
    - proof_inventory.mechanism_relevant

  LOW fields (weight 0.5x each):
    - mechanism.e5_strategy.sophistication_context
    - root_cause_handoff.belief_state
    - root_cause_handoff.questions_opened
    - structure.word_budget

  SCORE = (sum of present field weights) / (sum of all field weights) * 100

  THRESHOLD: 85% minimum to proceed
```

### Step 9: Determine Checkpoint Default

```
IF completeness_score >= 95%
   AND zero CRITICAL failures
   AND zero HIGH failures:
  SET checkpoint_default = "CONFIRM"
  SET checkpoint_urgency = "LOW"
  LOG "All data present and consistent — auto-confirm recommended"

ELIF completeness_score >= 85%
   AND zero CRITICAL failures:
  SET checkpoint_default = "CONFIRM"
  SET checkpoint_urgency = "MEDIUM"
  LOG "Data adequate with minor gaps — confirm recommended with review"

ELIF completeness_score >= 70%
   AND zero CRITICAL failures:
  SET checkpoint_default = "MODIFY"
  SET checkpoint_urgency = "HIGH"
  LOG "Significant gaps detected — modification recommended"

ELIF completeness_score < 70%
   OR any CRITICAL failure:
  SET checkpoint_default = "PROVIDE"
  SET checkpoint_urgency = "CRITICAL"
  LOG "Insufficient data — human must provide missing elements"
```

### Step 10: Compile Validation Report

```
CONSTRUCT validation_report:
  metadata:
    validated_at: timestamp
    microskill: "0.3-input-validator"
    version: "1.0"

  completeness:
    score: number (0-100)
    threshold: 85
    passes_threshold: boolean

  field_results:
    critical: [{ field, status, severity, detail }]
    high: [{ field, status, severity, detail }]
    medium: [{ field, status, severity, detail }]
    low: [{ field, status, severity, detail }]

  cross_validation:
    checks_run: 5
    checks_passed: integer
    warnings: [string]

  checkpoint_decision:
    default_action: enum[CONFIRM, MODIFY, SWITCH, PROVIDE]
    urgency: enum[LOW, MEDIUM, HIGH, CRITICAL]
    rationale: string

  gaps_requiring_resolution:
    critical_gaps: [{ field, description, impact }]
    recommended_gaps: [{ field, description, impact }]

WRITE validation_report to working memory
PASS to 0.4-human-checkpoint-curator
```

---

## QUALITY CRITERIA

1. **Every required field checked** — no field in input contract skipped
2. **Severity correctly assigned** — CRITICAL only for truly blocking items, not inflated
3. **Cross-validation performed** — all 5 cross-checks executed
4. **Completeness score accurate** — weighted calculation correct, not inflated or deflated
5. **Checkpoint default appropriate** — matches the severity of gaps found
6. **Gaps actionable** — each gap description tells the human exactly what is needed
7. **No false passes** — empty strings, placeholder text, and null values caught

---

## FAILURE MODES

| Failure | Symptom | Resolution |
|---------|---------|------------|
| Upstream data cache not received | 0.1 output missing | HALT — re-run 0.1 upstream loader |
| TIER1 index not received | 0.2 output missing | HALT — re-run 0.2 tier1 loader |
| Completeness score calculation error | Score > 100 or < 0 | Re-calculate with logging of each weight |
| All fields marked CRITICAL | Everything fails | Check if upstream data cache is empty/corrupt |
| Cross-validation contradictions | Check 1 passes but check 2 contradicts | Log both results, escalate to human checkpoint |
| Checkpoint default is PROVIDE but score >= 85 | Scoring inconsistency | Re-verify CRITICAL field results — one CRITICAL fail overrides score |
