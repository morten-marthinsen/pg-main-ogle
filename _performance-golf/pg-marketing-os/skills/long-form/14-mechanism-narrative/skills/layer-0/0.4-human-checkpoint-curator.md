# Skill 14 — Microskill 0.4: Human Checkpoint Curator

**Skill:** 11 — Mechanism Narrative
**Microskill:** 0.4 — Human Checkpoint Curator
**Version:** 1.0
**Layer:** 0 (Foundation & Loading)
**Position:** Fourth microskill in pipeline (final Layer 0 gate)
**Type:** Human-in-the-Loop Decision Gate
**Dependencies:** 0.1-upstream-loader, 0.2-tier1-intelligence-loader, 0.3-input-validator
**Output:** checkpoint-response.json
**BLOCKING:** Cannot proceed to Layer 1 without human response

---

## PURPOSE

Presents the mechanism selection and all relevant context to the human operator for confirmation before narrative generation begins. This is the last gate before execution — once the human confirms, the pipeline proceeds through Layers 1-4 with the confirmed mechanism and strategy. The checkpoint prevents wasted computation on a mechanism the human might reject and gives the operator a chance to modify, switch, or provide missing information.

---

## IDENTITY

**What this IS:**
- A human decision gate that requires explicit response before proceeding
- A clear, structured presentation of what will be used to generate the narrative
- An opportunity for the human to course-correct before execution

**What this is NOT:**
- Not an automated pass-through — even when default is CONFIRM, human must respond
- Not a creative step — presents data, does not generate content
- Not a validation step — validation happened in 0.3, this presents validated results
- Not optional — this microskill is BLOCKING in all cases

---

### Step 2: Present Four Response Options

```
DISPLAY options:

  [1] CONFIRM — Proceed with this mechanism as presented
      Use when: All data looks correct, mechanism is the right choice,
      strategy alignment feels right
      Effect: Pipeline proceeds to Layer 1 with current data unchanged

  [2] MODIFY — Adjust specific fields before proceeding
      Use when: Mechanism is right but needs tweaks to explanation,
      analogy, or strategy
      Effect: Opens modification interface, then re-validates

  [3] SWITCH — Select a different mechanism entirely
      Use when: Wrong mechanism was selected, or a better option exists
      Effect: Returns to Skill 03 mechanism selection, restarts pipeline

  [4] PROVIDE — Supply missing data manually
      Use when: Critical gaps exist that only the human can fill
      Effect: Opens data entry for missing fields, then re-validates

  DEFAULT HIGHLIGHTED: [checkpoint_decision.default_action]
```

### Step 3: Await Human Response

```
SET timeout: NONE — this is a blocking checkpoint with no timeout
AWAIT response from human operator

CAPTURE:
  response_action: enum[CONFIRM, MODIFY, SWITCH, PROVIDE]
  response_timestamp: timestamp
  response_notes: string (optional — human can add context)
```

### Step 4: Process CONFIRM Response

```
IF response_action == "CONFIRM":

  LOCK mechanism selection — no further changes to upstream data
  SET confirmed_mechanism:
    name: mechanism.name
    type: mechanism.type
    e_level: mechanism.e5_strategy.e_level
    strategy: mechanism.e5_strategy.strategy_name
    explanation: mechanism.explanation
    scorecard_average: mechanism.scorecard.average

  SET confirmed_context:
    villain: root_cause_handoff.villain_established
    emotional_state: root_cause_handoff.emotional_state_at_handoff
    bridge_text: root_cause_handoff.bridge_text
    proof_count: proof_inventory.total_count
    format: structure.format
    word_budget: structure.word_budget

  LOG: "Human confirmed mechanism: [name] at E[level]. Proceeding to Layer 1."
  PROCEED to Layer 1
```

### Step 5: Process MODIFY Response

```
IF response_action == "MODIFY":

  DISPLAY modifiable fields:
    [ ] mechanism.name — Current: "[value]"
    [ ] mechanism.type — Current: "[value]"
    [ ] mechanism.explanation.core_logic — Current: "[value]"
    [ ] mechanism.explanation.analogy — Current: "[value]"
    [ ] mechanism.explanation.step_by_step — Current: "[value]"
    [ ] mechanism.explanation.scientific_backing — Current: "[value]"
    [ ] mechanism.e5_strategy.e_level — Current: "[value]"
    [ ] mechanism.e5_strategy.strategy_name — Current: "[value]"

  PROMPT: "Select field(s) to modify and provide new values."

  AWAIT field modifications

  FOR EACH modified field:
    1. Validate new value against schema
    2. IF valid: UPDATE field in upstream_data_cache
    3. IF invalid: REJECT with explanation, re-prompt

  AFTER all modifications applied:
    RE-RUN validation (Step 7 of 0.3)
    IF validation passes: RE-PRESENT checkpoint with updated data
    IF validation fails: DISPLAY new gaps, return to options

  LOG: "Human modified [N] fields. Re-validating."
```

### Step 6: Process SWITCH Response

```
IF response_action == "SWITCH":

  DISPLAY: "Switching mechanism requires returning to Skill 03.
    Current mechanism narrative pipeline will be reset."

  CAPTURE switch_reason: string
    Prompt: "Brief reason for switching (helps Skill 03 refine selection):"

  OPTIONS:
    [A] Return to Skill 03 with switch_reason
        — Restarts mechanism selection, then re-enters Skill 11
    [B] Select from Skill 03's runner-up mechanisms (if available)
        — Displays runner-up list, human picks, pipeline continues
    [C] Cancel — return to checkpoint options

  IF option A:
    PACKAGE switch_request:
      from_skill: "14-mechanism-narrative"
      reason: switch_reason
      current_mechanism: mechanism.name
      request: "RE-SELECT"
    EMIT switch_request to Skill 03
    HALT Skill 11 pipeline pending Skill 03 re-execution

  IF option B:
    LOAD runner_up_mechanisms from mechanism-package.json (if available)
    DISPLAY runner-up list with names, types, scorecard averages
    AWAIT selection
    UPDATE mechanism in upstream_data_cache
    RE-RUN validation
    RE-PRESENT checkpoint

  IF option C:
    RETURN to Step 2 (options display)

  LOG: "Human requested mechanism switch. Reason: [switch_reason]"
```

### Step 7: Process PROVIDE Response

```
IF response_action == "PROVIDE":

  DISPLAY gaps requiring resolution:
    CRITICAL GAPS (must resolve):
    FOR EACH critical_gap:
      Field: [gap.field]
      Description: [gap.description]
      Impact: [gap.impact]
      Current Value: [current value or "MISSING"]
      Required Format: [schema requirement]
      >>> Enter value: _______________

    RECOMMENDED GAPS (optional but improve quality):
    FOR EACH recommended_gap:
      Field: [gap.field]
      Description: [gap.description]
      Current Value: [current value or "MISSING"]
      >>> Enter value (or SKIP): _______________

  AWAIT human input for each gap

  FOR EACH provided value:
    1. Validate against schema
    2. IF valid: UPDATE field in upstream_data_cache
    3. IF invalid: DISPLAY format requirement, re-prompt
    4. IF SKIP: Accept gap, note in checkpoint_response

  AFTER all gaps addressed:
    RE-RUN validation (full 0.3 re-execution)
    IF completeness_score >= 85%: RE-PRESENT checkpoint (likely default CONFIRM)
    IF completeness_score < 85%: RE-DISPLAY remaining gaps

  LOG: "Human provided [N] values. [M] gaps resolved, [K] gaps remaining."
```

### Step 8: Compile Checkpoint Response

```
CONSTRUCT checkpoint_response:
  metadata:
    checkpoint_at: timestamp
    microskill: "0.4-human-checkpoint-curator"
    version: "1.0"
    response_time: duration (from display to response)

  human_decision:
    action: enum[CONFIRM, MODIFY, SWITCH, PROVIDE]
    notes: string (if provided)
    modifications_made: [{ field, old_value, new_value }] (if MODIFY)
    values_provided: [{ field, value }] (if PROVIDE)
    switch_reason: string (if SWITCH)

  confirmed_state:
    mechanism_name: string
    mechanism_type: string
    e_level: string
    strategy_name: string
    scorecard_average: number
    villain_name: string
    emotional_state: string
    proof_count: integer
    format: string
    completeness_score: number

  pipeline_status:
    proceed: boolean
    proceed_to: "layer-1" or "skill-03" or "re-checkpoint"
    locked_fields: [string]          # Fields that cannot change after confirmation

WRITE checkpoint_response to working memory
IF proceed == true AND proceed_to == "layer-1":
  PASS confirmed state to Layer 1 microskills
```

## QUALITY CRITERIA

1. **Display is complete** — all 7 sections rendered with current data
2. **Options are clear** — each option describes when to use it and what happens
3. **Default is highlighted** — human sees the recommended action prominently
4. **BLOCKING enforced** — pipeline does not advance without explicit human response
5. **Modifications re-validated** — any changes trigger full re-validation before proceeding
6. **Response captured completely** — action, notes, modifications, and timestamps all recorded
7. **Locked fields enforced** — after CONFIRM, mechanism name, type, and e_level cannot change

---

## FAILURE MODES

| Failure | Symptom | Resolution |
|---------|---------|------------|
| Human does not respond | Pipeline stalls indefinitely | Correct behavior — BLOCKING checkpoint, wait until response |
| Human selects MODIFY but provides no changes | Empty modification set | Re-present options, note that no changes were made |
| Human provides invalid data in PROVIDE | Value fails schema validation | Display format requirements, re-prompt for that field |
| SWITCH requested but no runner-ups available | Option B unavailable | Route to Option A only (return to Skill 11) |
| Human confirms despite CRITICAL gaps | default was PROVIDE but human chose CONFIRM | WARN strongly that narrative quality will suffer, require explicit "I understand" acknowledgment |
| Re-validation fails after MODIFY | Modification created new inconsistency | Display new validation results, return to options |
