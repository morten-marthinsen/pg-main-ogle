# Layer 0.4 — Input Validator

## Purpose

Validate that all loaded upstream packages, vault intelligence, and teaching indexes meet minimum thresholds for story engineering. Determine story strategy viability, identify the specific story types that are feasible given the available data, flag blockers that would prevent story construction, and produce a Gate 0 recommendation before Layer 1 begins.

Unlike the lead validator (which checks for elevator pitch viability), the story validator must assess NARRATIVE viability — does the loaded data contain enough raw material to construct a compelling, believable, emotionally transporting story that makes the mechanism FEEL real? A story without a mechanism to prove, a root cause to reveal, or a lead-established tease to fulfill is a story without purpose.

---

## Input Requirements

```yaml
inputs_to_validate:
  story_context: object          # from 0.1-upstream-loader.md
  vault_story_intelligence: object  # from 0.2-vault-intelligence-loader.md
  teaching_index: object         # from 0.3-teachings-loader.md
```

## Execution Protocol

### Step 1: Validate Required Upstream Package Completeness

These are HARD REQUIREMENTS. Story engineering CANNOT proceed without them.

**1a. Mechanism Package — REQUIRED**
- [ ] `mechanism.name` present and non-empty (the story must NAME the mechanism)
- [ ] `mechanism.type` present (determines which story types are viable)
- [ ] `mechanism.uniqueness_claim` present (the "new and different" element the story dramatizes)
- [ ] `mechanism.explanation_text` present (raw material for mechanism revelation beat)
- [ ] At least one of: `mechanism.proof_elements` OR `mechanism.how_it_works` present (story needs substance to reveal)

**Failure Condition:** If mechanism package is missing or mechanism has no name, HALT. Story engineering requires a mechanism to emotionally prove.

**1b. Structure Package — REQUIRED**
- [ ] `campaign_thesis.statement` present (story must support this thesis)
- [ ] `argument_strategy.type` present (determines story tone and function)
- [ ] Story-relevant handoff data present: at least `story_function` OR `emotional_entry_point`

**Failure Condition:** If structure package is missing, HALT. Story must integrate with campaign argument flow.

**1c. Lead Package — REQUIRED**
- [ ] `lead_components.discovery_story_teased` status is determinate (true or false)
- [ ] `lead_components.open_loops` array is present (even if empty — we need to know)
- [ ] `lead_components.emotional_state_at_handoff` present (starting state for story's emotional arc)

**Failure Condition:** If lead package is missing, HALT. Story must expand what the lead teased. Without knowing what was teased, the story cannot fulfill its narrative obligations.

### Step 2: Validate Optional Upstream Package Completeness

These are STRONGLY RECOMMENDED but not blocking. Missing packages generate warnings and degrade specific capabilities.

**2a. Promise Package**
- [ ] `primary_promise.statement` present and concrete enough for story payoff
  - **Concrete Test:** Can the promise be dramatized as a specific result in the story's transformation/payoff beat? (e.g., "lose 30 pounds" = concrete; "feel better" = too vague)
  - If promise is too vague for story payoff: WARNING — story payoff will lack specificity
- [ ] `primary_promise.emotional_frame` present (shapes protagonist's starting state)
- [ ] `promise_ceiling` present (prevents story results from exceeding defensible claims)

**Impact if Missing:** Story payoff beat will lack a specific result to dramatize. Transformation/payoff quality degraded.

**2b. Root Cause Package**
- [ ] `root_cause.expression` present
- [ ] `root_cause.three_part_structure` present with all three parts:
  - [ ] `what_they_think` present (conventional wisdom the protagonist initially believes)
  - [ ] `what_it_really_is` present (the real cause the protagonist discovers)
  - [ ] `why_it_matters` present (significance that drives action)
- [ ] Counter-intuitive element present: `what_they_think` != `what_it_really_is` (if they're the same, there's no story twist)

**Counter-Intuitive Test:**
- Compare `what_they_think` and `what_it_really_is`
- If they are substantively different: PASS — story has a counter-intuitive twist (the ENGINE of story interest)
- If they are similar or identical: WARNING — story lacks a counter-intuitive revelation, which significantly weakens the discovery/revelation beats

**Impact if Missing:** Story loses its most compelling narrative element — the counter-intuitive twist. Discovery stories become flat ("he found a thing that works" vs. "what he thought was wrong was actually the opposite"). Root cause revelation beat will lack power.

**2c. Proof Inventory**
- [ ] `summary.overall_strength` present
- [ ] At least one proof type populated:
  - Authority proofs (for Yoda character credibility)
  - Social proofs (for ordinariness/beta-tester elements)
  - Statistical proofs (for data cascade in proof stories)
  - Demonstration proofs (for experiment/case study elements)
  - Specificity proofs (for believability details)

**Proof Strength Assessment for Story Type Viability:**
- `overall_strength` >= 80: Proof Story / Live Experiment type is VIABLE
- `overall_strength` >= 60: Discovery Story with proof weaving is optimal
- `overall_strength` >= 40: Discovery Story without heavy proof is viable
- `overall_strength` < 40: Story must rely primarily on emotional arc, not evidence

**Impact if Missing:** Story will lack proof elements for natural weaving. Proof Story type becomes non-viable. Verification beats in discovery stories will lack substance.

**2d. Headline**
- [ ] `main_headline` present
- [ ] Headline establishes emotional momentum the story continues

**Impact if Missing:** Story loses headline emotional continuity. Minor impact — story can proceed without headline context.

### Step 3: Validate Lead-Story Integration Prerequisites

These checks validate the specific NARRATIVE OBLIGATIONS the story must fulfill:

**3a. Discovery Story Tease Status**
- If `discovery_story_teased` = true:
  - [ ] Verify at least one discovery story format is viable given available data
  - [ ] Verify protagonist status is determined (new or continue existing)
  - [ ] Log: "OBLIGATION: Lead teased a discovery story. Story MUST deliver a discovery narrative."
- If `discovery_story_teased` = false:
  - [ ] Log: "No discovery story obligation from lead. Story type selection is unconstrained."
  - [ ] Note: proof stories, transformation stories, or warning stories may be more appropriate

**3b. Open Loop Obligations**
- Count total open loops from lead: `open_loop_count`
- Classify each loop by type:
  - `story_closeable`: loops the story should close (discovery teases, mechanism teases, counter-intuitive teases)
  - `downstream_closeable`: loops that close in mechanism explanation, offer, or close (tangible teases about the product itself)
- [ ] Verify story-closeable loops >= 0 (presence check only — it's valid for zero loops to need story closure)
- Log each story-closeable loop with its obligation

**3c. Emotional Handoff Validation**
- [ ] `emotional_state_at_handoff` is a recognizable emotional state (curiosity, hope, alarm, fascination, etc.)
- [ ] Verify this state is suitable as a STARTING POINT for story emotional arc design
- [ ] Flag if emotional state at handoff is "neutral" or "disengaged" — this would indicate a lead failure, not a story problem

**3d. Protagonist Continuity Check**
- If `protagonist_introduced` = true in lead:
  - [ ] Verify protagonist details are sufficient for story continuation (name, situation, emotional state)
  - [ ] Log: "CONTINUITY: Story must continue with protagonist [name] established in lead"
- If `protagonist_introduced` = false:
  - [ ] Log: "Story must introduce protagonist. Character architect has full creative latitude."

### Step 4: Validate Vault Intelligence

**4a. Pattern Count Threshold**
- [ ] `pattern_count` >= 5 (minimum story patterns from vault)
- If < 5: WARNING — limited vault reference for story type classification. Classification will rely more heavily on teaching frameworks.

**4b. Story Type Coverage**
- [ ] `story_type_count` >= 4 (at least 4 different story types in taxonomy)
- If < 4: WARNING — limited story type diversity. Some story types may lack vault exemplars.

**4c. Beat Catalog Population**
- [ ] At least 3 beat catalogs populated with full beat sequences
- If < 3: WARNING — beat mapping will have limited reference templates

**4d. Makepeace Pattern Loaded**
- [ ] Makepeace proof-story pattern extracted (all 8 sub-patterns)
- If incomplete: WARNING — proof story type will lack empirical pattern reference
- If campaign niche = finance/trading AND Makepeace pattern missing: ELEVATED WARNING — proof story is likely the optimal type for this niche

**4e. Emotional Progression Patterns**
- [ ] At least 4 emotional progression patterns documented
- If < 4: WARNING — emotional arc design will have limited reference

**4f. Character Archetypes**
- [ ] At least 3 protagonist archetypes cataloged
- [ ] At least 3 Yoda/guide archetypes cataloged
- If insufficient: WARNING — character architect will have limited archetype reference

### Step 5: Validate Teaching Index

**5a. Carlton Principles**
- [ ] All 10 Carlton principles indexed
- [ ] Each principle has: rule, anti_pattern, application_note
- [ ] Critical principles present: three_line_structure, transporting, breathless_prose, means_for_you, empathy, feel_something
- If fewer than 8 principles: WARNING — Carlton compliance checker (Layer 3) will be degraded
- If fewer than 6 principles: FAIL — insufficient Carlton reference for story refinement

**5b. Discovery Story Formats**
- [ ] All 3 formats indexed (standard, accidental, straight-to-expert)
- [ ] Each format has complete beat sequence with examples
- If fewer than 2 formats: WARNING — story type classification constrained
- If 0 formats: FAIL — no structural templates for story construction

**5c. Makepeace Proof Story Elements**
- [ ] At least 7 of 10 elements indexed
- [ ] Critical elements present: beta_tester_framing, data_cascade, ordinariness_proof, transparency_paradox
- If fewer than 5 elements: WARNING — proof story construction will be degraded
- If proof story likely needed (finance niche, high proof inventory) and < 7 elements: ELEVATED WARNING

**5d. Georgi Integration Rules**
- [ ] At least 3 of 5 rules indexed
- [ ] Critical rules present: lead_teases_story_tells, deliver_on_tease, open_loop_obligations
- If fewer than 3 rules: WARNING — lead-story integration guidance incomplete

**5e. Cross-Reference Index**
- [ ] Cross-reference index populated
- If missing: WARNING — downstream routing of teaching principles to microskills will be unguided

### Step 7: Compile Validation Report

Assemble the complete validation report with pass/fail status, warnings, blockers, story type feasibility, and Gate 0 recommendation.

**Gate 0 Decision Logic:**

- **PROCEED** if:
  - All 3 required packages loaded (mechanism, structure, lead)
  - At least 1 story type is feasible
  - Teaching index has at least 6 Carlton principles + 2 discovery formats
  - Vault intelligence loaded with at least 3 patterns

- **PROCEED WITH CAUTION** if:
  - All 3 required packages loaded
  - At least 1 story type is feasible
  - But: missing optional packages (promise, root cause, proof) OR vault intelligence < 5 patterns OR teaching index incomplete
  - Specific warnings documented for degraded capabilities

- **HALT** if:
  - ANY required package missing (mechanism, structure, or lead)
  - Zero story types are feasible
  - Teaching index has < 6 Carlton principles OR < 1 discovery format
  - Vault intelligence loading_status = empty

## Quality Gates

**PASS if:**
- [ ] All 3 required upstream packages present (mechanism + structure + lead)
- [ ] Mechanism has a specific name (not just a description)
- [ ] Lead package provides discovery_story_teased status and emotional_state_at_handoff
- [ ] At least 1 story type is feasible
- [ ] Teaching index has >= 6 Carlton principles
- [ ] Teaching index has >= 2 discovery story formats
- [ ] Vault intelligence has >= 3 story patterns
- [ ] Narrative obligations derived and documented

**PASS WITH WARNINGS if:**
- [ ] All PASS conditions met
- [ ] But one or more optional packages missing
- [ ] Or vault intelligence has < 5 patterns
- [ ] Or teaching index is partially incomplete
- [ ] Or promise is not concrete enough for story payoff

**FAIL if:**
- [ ] ANY required package missing (mechanism, structure, or lead)
- [ ] Mechanism package present but mechanism has no name
- [ ] Zero story types are feasible
- [ ] Teaching index has < 6 Carlton principles
- [ ] Teaching index has 0 discovery story formats
- [ ] Vault intelligence loading_status = empty

## Error Handling

### Gate 0 FAIL — Missing Mechanism
```
GATE 0 FAIL: Mechanism package absent or mechanism unnamed.
The story skill cannot engineer a narrative without a mechanism to prove.
The entire purpose of the story is to make the mechanism FEEL real.
REQUIRED ACTION: Run 04-mechanism skill before invoking 12-story.
```

### Gate 0 FAIL — Missing Lead Package
```
GATE 0 FAIL: Lead package absent.
The story must expand what the lead teased. Without knowing:
  - Whether a discovery story was teased
  - What open loops were created
  - What emotional state the reader is in
...the story cannot fulfill its narrative obligations.
REQUIRED ACTION: Run 11-lead skill before invoking 12-story.
```

### Gate 0 FAIL — Missing Structure Package
```
GATE 0 FAIL: Structure package absent.
The story must integrate with the campaign argument flow. Without knowing:
  - Where the story is placed in the campaign
  - What the campaign thesis is
  - What emotional state the story must deliver the reader to
...the story cannot serve its structural function.
REQUIRED ACTION: Run 08-structure skill before invoking 12-story.
```

### Gate 0 FAIL — No Feasible Story Types
```
GATE 0 FAIL: Zero story types are feasible given available data.
This typically means:
  - Mechanism exists but has no counter-intuitive element (discovery types blocked)
  - Proof inventory insufficient for proof story (proof type blocked)
  - No emotional frame for transformation story (transformation blocked)
REQUIRED ACTION: Review upstream packages for completeness. Ensure root cause
has counter-intuitive element and/or proof inventory has sufficient data.
```

### Gate 0 PROCEED WITH CAUTION — Promise Too Vague
```
GATE 0 WARNING: Promise statement is present but too vague for story payoff.
Current promise: "[promise_text]"
Story's transformation/payoff beat requires a SPECIFIC result to dramatize.
RECOMMENDED: Refine promise to be more specific before Layer 2 construction.
IMPACT: Story payoff will be emotionally weaker without a specific transformation to show.
```

### Gate 0 PROCEED WITH CAUTION — No Counter-Intuitive Element
```
GATE 0 WARNING: Root cause lacks counter-intuitive element.
what_they_think: "[text]"
what_it_really_is: "[text]"
These are substantively similar. The counter-intuitive twist is the ENGINE
of story interest — without it, the discovery/revelation beats will be flat.
RECOMMENDED: Revisit root cause skill to sharpen the counter-intuitive element.
IMPACT: Story will function but lack the "I never would have guessed" moment
that makes great discovery stories compelling.
```

## Handoff

Pass `validation_report` to:
- **STORY-AGENT.md** (for Gate 0 evaluation — determines whether to proceed to Layer 1)
- **1.1-story-type-classifier.md** (consumes `story_type_feasibility` to constrain classification to viable types only)
- **1.2-story-beat-mapper.md** (consumes `narrative_obligations_summary` to ensure required beats are mapped)
- **1.4-emotional-arc-designer.md** (consumes `lead_story_integration.handoff_emotional_state` as arc starting point)
