# Skill 13.0.4: Human Checkpoint Curator

**Version:** 1.0
**Skill:** 10 — Root Cause Narrative
**Layer:** 0 (Foundation & Loading)
**Position:** Fourth microskill — BLOCKING human checkpoint before narrative generation
**Type:** Human Interaction & Confirmation
**Dependencies:** upstream_data.json (from 0.1), validation_report.json (from 0.3)
**Output:** checkpoint_response.json

---

## PURPOSE

Present the loaded and validated root cause selection to the human operator for confirmation before any narrative writing begins. This is a BLOCKING checkpoint — no downstream processing can occur until the human responds. The checkpoint ensures the operator has final say over which root cause strategy gets translated into narrative prose, and provides four response options (CONFIRM, MODIFY, SWITCH, PROVIDE) to handle any scenario from full approval to complete data entry.

---

## IDENTITY

**This microskill IS:**
- A human-facing presentation layer that displays root cause data clearly and completely
- A BLOCKING gate that halts all processing until human response is received
- A response processor that handles four distinct response types with appropriate follow-up actions
- A re-validation trigger that ensures any modifications still pass input validation

**This microskill is NOT:**
- A decision-maker — the human makes all decisions; this skill presents and processes
- A data generator — it does not create root cause data, only presents what exists
- A validator — validation happened in 0.3; this skill presents results and handles responses
- An optional step — this checkpoint is MANDATORY regardless of validation score

## PROCESS

### Step 1: Compose Checkpoint Display

```
BUILD checkpoint_display:

  HEADER:
    "ROOT CAUSE NARRATIVE — HUMAN CHECKPOINT"
    "Skill 13-root-cause-narrative | Layer 0 Gate"
    "Status: [validation_status]"
    ""

  SECTION 1 — Root Cause Selection:
    "SELECTED ROOT CAUSE (from 03-root-cause):"
    ""
    "  Three-Part Structure:"
    "    WHAT THEY THINK:      [what_they_think]"
    "    WHAT IT REALLY IS:    [what_it_really_is]"
    "    WHY NOTHING WORKED:   [why_nothing_worked]"
    ""
    "  Expression Method:  [expression_method]"
    "  Reframe Technique:  [reframe_technique]"
    ""

  SECTION 2 — Villain:
    "  Villain:"
    "    Name: [villain.name]"
    "    Type: [villain.type]"
    "    Role: [villain.role]"
    ""

  SECTION 3 — Campaign Context:
    "  Niche:                  [niche]"
    "  Sub-niche:              [sub_niche]"
    "  Market Sophistication:  Stage [market_sophistication_stage]"
    ""

  SECTION 4 — Data Quality (from validation):
    "  Input Completeness:  [overall_completeness]%"
    IF critical_gaps not empty:
      "  CRITICAL GAPS:"
      FOR each gap: "    - [gap]"
    IF gaps not empty:
      "  GAPS:"
      FOR each gap: "    - [gap]"
    IF warnings not empty:
      "  WARNINGS:"
      FOR each warning: "    - [warning]"
    ""

  SECTION 5 — Countersells Summary:
    "  Countersells Defined: [count]"
    FOR each countersell:
      "    - [target] ([pattern]): [argument summary]"
    ""

  SECTION 6 — Alternatives (if available):
    IF alternative_root_causes not empty:
      "  ALTERNATIVE ROOT CAUSES AVAILABLE:"
      FOR each alternative:
        "    [index]. [what_it_really_is] (reframe: [reframe_technique])"
    ""

  SECTION 7 — Options:
    "  OPTIONS:"
    "    [1] CONFIRM  — Proceed with this root cause as shown"
    "    [2] MODIFY   — Adjust specific fields (three-part, villain, etc.)"
    "    [3] SWITCH   — Use an alternative root cause from the package"
    "    [4] PROVIDE  — Supply custom root cause data manually"
    ""
    "  DEFAULT: [[checkpoint_default]]"
    ""
    IF checkpoint_default == "CONFIRM":
      "  (All data validated. Press Enter or type '1' to proceed.)"
    IF checkpoint_default == "MODIFY":
      "  (Gaps detected. Review gaps above and modify as needed.)"
    IF checkpoint_default == "PROVIDE":
      "  (Insufficient data. Please provide missing root cause data.)"
```

### Step 2: Present Checkpoint to Human

```
DISPLAY checkpoint_display to human operator

SET state = AWAITING_RESPONSE
SET timeout = NONE  # Wait indefinitely — this is a blocking checkpoint

LOG: "Human checkpoint presented. Awaiting response."
LOG: "Default action: [checkpoint_default]"
```

### Step 3: Process Human Response

```
RECEIVE human_response

PARSE response:

  IF response is empty OR "yes" OR "proceed" OR "confirm" OR "1":
    SET action = "CONFIRM"
    LOG: "Human confirmed root cause selection."

  ELIF response contains "modify" OR "2" OR "change" OR "adjust":
    SET action = "MODIFY"
    PARSE modification_details from response:
      IDENTIFY which fields to modify
      EXTRACT new values for those fields
    LOG: "Human requested modifications: [modification_details]"

  ELIF response contains "switch" OR "3" OR "alternative":
    SET action = "SWITCH"
    IF alternative_root_causes available:
      PARSE which alternative from response
      IF specific alternative identified:
        SET selected_alternative = identified alternative
      ELSE:
        DISPLAY alternative options
        AWAIT selection
        SET selected_alternative = selected option
    ELSE:
      LOG: "No alternatives available. Treating as MODIFY."
      SET action = "MODIFY"
      PROMPT: "No alternative root causes in the package.
               Please provide modification details or custom data."
    LOG: "Human switched to alternative root cause: [selected_alternative]"

  ELIF response contains "provide" OR "4" OR "custom":
    SET action = "PROVIDE"
    PROMPT human for custom data:
      "Please provide the following root cause data:"
      "  1. WHAT THEY THINK: [current belief to reframe]"
      "  2. WHAT IT REALLY IS: [the actual root cause]"
      "  3. WHY NOTHING WORKED: [why past solutions failed]"
      "  4. Expression Method: [how the root cause is communicated]"
      "  5. Reframe Technique: [not_your_fault, conspiracy, hidden_truth,"
      "     opposite_of_truth, suppressed_science, conspiracy_expose]"
      "  6. Villain Name: [who/what to blame]"
      "  7. Villain Type: [pharma, medical_establishment, government,"
      "     food_industry, misinformation, biological_process, industry]"
      "  8. Villain Role: [suppressor, profiteer, deceiver, blocker, betrayer]"
    RECEIVE custom_data
    LOG: "Human provided custom root cause data."

  ELSE:
    ATTEMPT to interpret response as modification
    IF interpretable: SET action = "MODIFY" with parsed modifications
    ELSE:
      DISPLAY: "Response not recognized. Please choose:"
      "  [1] CONFIRM  [2] MODIFY  [3] SWITCH  [4] PROVIDE"
      AWAIT clarified response
      RE-PARSE
```

### Step 4: Apply Response Actions

```
IF action == "CONFIRM":
  SET confirmed_root_cause = upstream_data.root_cause  # unchanged
  SET modification_log = "None — confirmed as-is"

IF action == "MODIFY":
  FOR each modified field in modification_details:
    UPDATE upstream_data.root_cause.[field] = new_value
    ADD to modification_log: "[field]: [old_value] → [new_value]"
  SET confirmed_root_cause = updated upstream_data.root_cause

IF action == "SWITCH":
  LOAD selected_alternative from upstream_data.root_cause.alternative_root_causes
  REPLACE upstream_data.root_cause.three_part_structure with alternative's three-part
  REPLACE upstream_data.root_cause.expression_method with alternative's method
  REPLACE upstream_data.root_cause.reframe_technique with alternative's technique
  IF alternative has villain data:
    REPLACE upstream_data.root_cause.villain with alternative's villain
  ADD to modification_log: "SWITCHED to alternative root cause: [alternative summary]"
  SET confirmed_root_cause = updated upstream_data.root_cause

IF action == "PROVIDE":
  POPULATE upstream_data.root_cause with custom_data
  VALIDATE custom_data has minimum required fields:
    - three_part_structure (all 3 parts)
    - reframe_technique
    - villain.name (at minimum)
  IF minimum not met:
    PROMPT for missing fields
    REPEAT until minimum met
  ADD to modification_log: "CUSTOM data provided by human operator"
  SET confirmed_root_cause = custom upstream_data.root_cause
```

### Step 5: Re-Validate After Modification

```
IF action != "CONFIRM":
  RE-RUN validation checks from 0.3 on updated data:
    - Three-part structure completeness
    - Villain identification
    - Expression method and reframe technique
    - Overall completeness score

  IF re_validation.overall_completeness < 60%:
    DISPLAY: "Modified data still below minimum threshold (60%).
              Current completeness: [percentage]%.
              Gaps: [list]"
    PROMPT: "Would you like to [1] MODIFY further or [2] PROCEED anyway?"
    IF MODIFY: RETURN to Step 3
    IF PROCEED: LOG warning and continue

  IF re_validation.overall_completeness >= 60% AND < 85%:
    LOG: "Modified data passes with gaps. Completeness: [percentage]%"

  IF re_validation.overall_completeness >= 85%:
    LOG: "Modified data fully passes validation. Completeness: [percentage]%"
```

### Step 6: Compile Checkpoint Response

```
COMPILE checkpoint_response:
  action: enum[CONFIRM, MODIFY, SWITCH, PROVIDE]
  confirmed_root_cause:
    three_part_structure:
      what_they_think: string  # confirmed or updated
      what_it_really_is: string
      why_nothing_worked: string
    expression_method: string
    reframe_technique: string
    villain:
      name: string
      type: string
      role: string
    countersells: [object]
  niche_context:
    niche: string
    sub_niche: string
    market_sophistication_stage: integer
  modification_log: [string]
  post_modification_completeness: float
  human_notes: string  # any additional notes from the human

WRITE checkpoint_response to: layer-0-outputs/checkpoint_response.json

LOG for learning log:

  fields_modified: [list]

```

## QUALITY CRITERIA

1. **BLOCKING execution enforced** — no downstream processing can begin until this checkpoint completes with a human response
2. **Display is clear and complete** — all root cause data presented in readable format; human can make informed decision
3. **All four options functional** — CONFIRM, MODIFY, SWITCH, and PROVIDE all produce valid confirmed_root_cause objects
4. **Re-validation after modification** — any change to the data triggers re-validation to ensure minimum thresholds
5. **Default reflects validation status** — CONFIRM only when validator passed all checks; MODIFY when gaps exist; PROVIDE when data is critically insufficient
6. **Graceful response parsing** — handles natural language responses ("yes", "looks good", "change the villain") not just numeric codes
7. **Modification logged for learning** — every change is recorded so the learning log can track human modification patterns
8. **No timeout** — checkpoint waits indefinitely; a root cause narrative without human confirmation is unauthorized

---

## FAILURE MODES

| Failure | Severity | Detection | Handling |
|---------|----------|-----------|----------|
| Human response ambiguous | LOW | Response cannot be parsed into one of 4 actions | Re-prompt with clear options |
| SWITCH requested but no alternatives | MODERATE | alternative_root_causes is empty | Redirect to MODIFY or PROVIDE |
| PROVIDE data incomplete | MODERATE | Custom data missing required fields | Prompt for specific missing fields |
| Re-validation fails after modify | MODERATE | Post-modification completeness < 60% | Offer MODIFY again or PROCEED with warning |
| Human provides contradictory data | LOW | what_it_really_is same as what_they_think | Flag specificity issue, suggest refinement |
| Checkpoint bypassed programmatically | CRITICAL | Gate entered without human interaction | HALT — this is a system integrity violation |
