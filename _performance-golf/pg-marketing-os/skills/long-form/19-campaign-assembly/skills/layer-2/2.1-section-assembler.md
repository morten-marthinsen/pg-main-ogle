# Microskill 2.1: Section Assembler

**Layer:** 2 (Assembly)
**Purpose:** Assemble each section by loading drafted copy from upstream packages

---

## CORE FUNCTION

The Section Assembler pulls drafted copy from each upstream package and assembles it into the section structure defined by Layer 1. This is ASSEMBLY, not writing—the copy already exists in the packages.

---

## UPSTREAM PACKAGE MAPPING

| Section | Primary Package | Copy Location |
|---------|-----------------|---------------|
| Lead | lead-package | `lead_copy` or `drafted_lead` |
| Story | story-package | `story_copy` or `drafted_story` |
| Root Cause | root-cause-package | `root_cause_copy` or `drafted_content` |
| Mechanism | mechanism-package | `mechanism_copy` or `drafted_mechanism` |
| Proof Section | proof-weaving-package | `proof_blocks.proof_section` |
| Product | product-package | `product_copy` or `drafted_product` |
| Offer | offer-package | `offer_copy` or `drafted_offer` |
| Close | close-package | `close_copy` or `drafted_close` |
| P.S. | close-package | `ps_copy` or `ps_elements` |

---

## ASSEMBLY PROCESS

### Step 1: Load Section Copy

For each section in sequence:

```yaml
section_load:
  section_name: "[name]"
  source_package: "[package name]"
  copy_loaded: true | false
  word_count: "[count]"
  copy_status: COMPLETE | PARTIAL | MISSING
```

### Step 2: Validate Copy Completeness

Each section must have:

```
[ ] Opening hook/transition point
[ ] Core content body
[ ] Closing transition point
[ ] No placeholder text (e.g., "[INSERT HERE]")
[ ] No incomplete sentences
[ ] No "TBD" or "TODO" markers
```

### Step 3: Handle Missing or Incomplete Copy

```yaml
missing_copy_protocol:
  if: copy_status == MISSING
  then:
    - FLAG as blocking issue
    - IDENTIFY source package
    - REPORT: "Section [X] copy not found in [package]. Run [skill] to generate."
    - CANNOT proceed to assembly

  if: copy_status == PARTIAL
  then:
    - FLAG as warning
    - IDENTIFY missing components
    - ASSESS if assembly can continue with gaps noted
    - REPORT specific missing elements
```

### Step 4: Section Assembly Template

```yaml
assembled_section:
  section_id: "[SECTION_001]"
  section_name: "[name]"
  sequence_position: "[number]"

  content:
    opening_transition: "[copy or null if first section]"
    body: "[main section copy]"
    closing_transition: "[copy or null - written in 2.2]"

  metadata:
    source_package: "[package]"
    word_count: "[count]"
    proportion_of_total: "[percentage]"

  assembly_status: COMPLETE | NEEDS_TRANSITION | NEEDS_PROOF | INCOMPLETE

  integration_points:
    proof_insertion_markers: [list of positions]
    threading_requirements: [from golden thread map]
    callback_points: [from callback registry]
```

## SECTION-SPECIFIC ASSEMBLY NOTES

### Lead Assembly

- Must start with headline/hook (no transition in)
- Opening must immediately engage
- Check for mechanism name early mention
- Flag if promise not established

### Story Assembly

- Needs transition from lead
- Story arc must be complete (beginning → middle → transformation)
- Check for protagonist identification
- Flag if mechanism discovery moment missing

### Root Cause Assembly

- Strong transition from story ("But WHY...")
- Villain/enemy establishment
- Contrast with failed solutions
- Bridge to mechanism reveal

### Mechanism Assembly

- Transition that creates anticipation
- Complete explanation of HOW mechanism works
- Multiple proof integration points
- Clear differentiation from failed approaches

### Proof Section Assembly

- Load proof blocks from proof-weaving-package
- Maintain sequencing strategy from proof package
- Ensure proof cascade structure
- Check for variety of proof types

### Product Assembly

- Natural transition from mechanism to product
- Product delivers the mechanism
- Formula/component proof integration
- Bridge to offer

### Offer Assembly

- Value stack structure
- Guarantee placement
- Bonus structure
- Price reveal positioning

### Close Assembly

- Urgency escalation structure
- Callback integration points
- Final CTA positioning
- P.S. section attachment

## ASSEMBLY VALIDATION

For each assembled section:

```
[ ] Copy loaded from correct package
[ ] No placeholder or TODO text
[ ] Word count within proportion target (±15%)
[ ] Opening transition point present (except lead)
[ ] Closing transition point marked (except close)
[ ] Proof insertion points identified
[ ] Threading requirements noted
[ ] Callback points flagged
```
