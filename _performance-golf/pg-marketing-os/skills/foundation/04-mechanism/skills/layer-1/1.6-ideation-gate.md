# Skill 1.6: Ideation Gate

**Version:** 1.0
**Layer:** 1 (Mechanism Ideation)
**Type:** Quality Gate
**Dependencies:** type_candidates.json, naming_candidates.json, explanation_structures.json, mechanism_analogies.json, proof_maps.json
**Output:** gate_1_report.json + GATE_1 status

---

## Purpose

Binary PASS/FAIL checkpoint for Layer 1. Validates that mechanism ideation has produced sufficient viable candidates with complete packages (name + explanation + analogy + proof map) before proceeding to scorecard optimization.

---

## Input Schema

```yaml
input:
  type_candidates: from 1.1
  naming_candidates: from 1.2
  explanation_structures: from 1.3
  mechanism_analogies: from 1.4
  proof_maps: from 1.5
```

---

## Output Schema

```yaml
output:
  gate_1_report:
    gate_status: "PASS" | "FAIL"
    timestamp: string
    iteration: integer              # Which attempt (max 3)

    candidate_assessments:
      - candidate_name: string
```

## Gate Logic

### Check 1: Candidate Completeness

```
FOR each top-5 naming candidate:

  CHECK has_name:
    - naming_candidates contains this entry
    - Name is non-empty string
    - No anti-pattern flags

  CHECK has_explanation_all_levels:
    - explanation_structures contains entry for this candidate
    - level_1_elevator present and ≤ 25 words
    - level_2_paragraph present and 50-100 words
    - level_3_walkthrough present and 150-300 words

  CHECK has_analogy:
    - mechanism_analogies contains entry for this candidate
    - primary_analogy present
    - familiarity_score ≥ 7
    - accuracy_score ≥ 6

  CHECK has_proof_map:
    - proof_maps contains entry for this candidate
    - At least 1 critical claim has proof assigned
    - proof_coverage_ratio > 0

  is_complete = all 4 checks pass
```

### Check 2: Candidate Viability

```
FOR each complete candidate:

  is_viable IF:
    - proof_coverage_ratio ≥ 0.60 (critical claims mostly covered)
    - explanation clarity_score ≥ 7
    - analogy familiarity ≥ 7
    - No unresolved critical proof gaps
    - Name doesn't match known competitor mechanism

  IF NOT viable:
    LIST blocking_issues (specific reasons)
```

### Check 3: Diversity Requirements

```
ASSESS naming diversity:
  - Count distinct naming_patterns in viable candidates
  - PASS if ≥ 3 patterns represented
  - FAIL if < 3 (too homogeneous, limits creative options)

ASSESS type diversity:
  - Count distinct mechanism_types in viable candidates
  - PASS if ≥ 2 types represented
  - WARN if only 1 type (acceptable but narrow)
```

### Check 4: Gate Decision

```
MINIMUM VIABLE CANDIDATES: 5

IF viable_candidates ≥ 5 AND naming_diversity = true:
  gate_status = "PASS"
  reason = "Sufficient viable candidates with diversity"

ELSE IF viable_candidates ≥ 3 AND naming_diversity = true:
  gate_status = "PASS" (with warning)
  reason = "Minimum viable met but below optimal"

ELSE:
  gate_status = "FAIL"
  GENERATE remediation plan:

  IF insufficient candidates:
    → Return to 1.2-naming-generator for additional pass
    → Lower generation constraints slightly
    → Try different naming patterns

  IF incomplete packages:
    → Identify which skill produced gaps
    → Return to that skill for the specific candidates

  IF proof coverage too low:
    → Return to 1.5-proof-mapper with expanded search
    → Consider different mechanism types with better proof availability

  IF diversity too low:
    → Return to 1.2 with explicit pattern diversity instruction
    → Force at least 2 underrepresented patterns
```

### Check 5: Iteration Control

```
IF gate_status = "FAIL":
  INCREMENT iteration counter

  IF iteration ≥ 3:
    ESCALATE to human checkpoint
    MESSAGE: "After 3 iterations, unable to generate 5 viable mechanism
    candidates. Review inputs for feasibility issues:
    - Is the root cause too narrow for mechanism development?
    - Is proof availability too limited?
    - Is the Big Idea compatible with a strong mechanism?"

  ELSE:
    RETURN to specified remediation skill
    INCLUDE specific instructions from remediation plan
```

---

## Quality Criteria

1. Gate decision is binary — PASS or FAIL, no "partial pass"
2. Every FAIL has specific remediation instructions
3. Iteration limit prevents infinite loops
4. Viable count threshold is strict (5 minimum for optimal, 3 minimum to proceed)
5. Diversity check prevents mode collapse in naming
