# Skill 3.3: Vault Pattern Comparator

**Version:** 1.0
**Layer:** 3 (Validation & Selection)
**Type:** Comparative Analysis
**Dependencies:** threshold_gate_report.json (viable candidates), vault_patterns.json, explanation_structures.json, naming_candidates.json
**Output:** vault_comparison.json

---

## Purpose

Compare viable mechanism candidates against the top-25 vault exemplars. Are our candidates as strong as PROVEN mechanisms from the vault? Where do they exceed vault exemplars? Where do they fall short? This grounds the assessment in empirical reality — not just theoretical scoring.

---

## Input Schema

```yaml
input:
  viable_candidates: from 3.2 (only those that passed gate)
  vault_patterns: from 0.2 (exemplars section)
  explanation_structures: from 1.3
  naming_candidates: from 1.2
```

## Comparison Logic

### Step 1: Find Closest Vault Exemplar

```
FOR each viable candidate:

  FROM vault_patterns.exemplars.top_25:

  COMPARE on multiple dimensions:

  NAME SIMILARITY:
    - Same naming pattern? (clinical, branded_format, metaphor, origin, numbered, experiential, invented)
    - Same vocabulary family?
    - Similar phonetic structure?

  TYPE SIMILARITY:
    - Same mechanism type? (scientific, technical, natural, compound, behavioral, sequence, system)
    - Same proof structure?

  STRUCTURAL SIMILARITY:
    - Same explanation depth?
    - Same root cause approach?
    - Same niche/audience?

  SELECT closest exemplar by highest combined similarity
  NOTE: what makes them similar AND what makes them different
```

### Step 2: Benchmark Against Vault Averages

```
FROM vault_patterns:

  CALCULATE vault benchmarks:
    - Average mechanism score of top-25 exemplars
    - Most common naming patterns in top-25
    - Most common mechanism types in top-25
    - Average proof density of top-25

  FOR each viable candidate:
    COMPARE against these benchmarks:
    - Is candidate score above or below vault average?
    - Estimate percentile ranking if candidate were in vault
    - Identify where candidate exceeds vault AND where it falls short
```

### Step 3: Assess Innovation vs. Proven Patterns

```
FOR each viable candidate:

  CATEGORIZE each element:

  VAULT-VALIDATED (follows proven pattern):
    - Uses naming pattern common in top-25
    - Uses mechanism type with high vault scores
    - Uses proof structure seen in exemplars
    → SAFE — validated by vault evidence

  NOVEL (doesn't appear in vault):
    - New naming pattern not in vault
    - Unusual mechanism type for this niche
    - Unique structural approach
    → INNOVATIVE but unproven

  RISK ELEMENTS (novel + structurally important):
    - Core elements that are novel = higher risk
    - Supporting elements that are novel = lower risk
    - Flag anything novel AND critical to mechanism success
```

### Step 4: Generate Vault Confidence Rating

```
FOR each viable candidate:

  vault_confidence = weighted assessment:

  HIGH CONFIDENCE (follows proven patterns + exceeds benchmarks):
    - Uses vault-validated patterns for critical elements
    - Scores at or above vault top-25 average
    - Novel elements are in supporting positions only

  MEDIUM CONFIDENCE (mostly validated, some novel):
    - Most elements are vault-validated
    - 1-2 novel elements in important positions
    - Scores near vault average

  LOW CONFIDENCE (mostly novel, limited validation):
    - Multiple novel elements in critical positions
    - Below vault average on key dimensions
    - Risk elements in name, type, or core explanation

  NOTE: Low confidence ≠ bad mechanism
    → It means we're innovating beyond vault precedent
    → May be breakthrough OR may be untested failure
    → Flag for human review if low confidence
```

---

## Quality Criteria

1. Every viable candidate compared to specific vault exemplar
2. Percentile estimate grounded in vault score distribution
3. Innovation vs. validation clearly categorized
4. Risk elements explicitly identified
5. Vault confidence provides honest assessment of empirical grounding
