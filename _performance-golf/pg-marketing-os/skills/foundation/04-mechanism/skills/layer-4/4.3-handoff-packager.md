# Skill 4.3: Handoff Packager

**Version:** 1.0
**Layer:** 4 (Package Assembly)
**Type:** Output Formatting & Handoff
**Dependencies:** mechanism_brief.json, copy_integration_map.json, mechanism_selection.json
**Output:** mechanism_package.json (FINAL OUTPUT)

---

## Purpose

Package the complete mechanism output into the standardized format consumed by downstream skills (04-headlines, 05-leads, 06-vsl-structure, etc.). This is the TERMINAL skill — its output IS the mechanism skill's deliverable. The package must be self-contained: any downstream skill should be able to consume it without referencing mechanism internals.

---

## The Question

**"Can the next skill in the sequence (04-headlines) pick up this package and immediately begin work — with zero questions, zero ambiguity, and zero missing context?"**

---

input:

## Packaging Logic

### Step 1: Flatten Mechanism Brief Into Package Structure

```
FROM mechanism_brief:

  MAP each brief field to package field:

  mechanism_brief.identity → mechanism_package.mechanism (name, type, archetype)
  mechanism_brief.the_mechanism_in_brief → mechanism_package.mechanism.explanations
  mechanism_brief.proof_architecture → mechanism_package.mechanism.proof
  mechanism_brief.analogies → mechanism_package.mechanism.analogies
  mechanism_brief.emotional_architecture → mechanism_package.mechanism.emotional
  mechanism_brief.competitive_positioning → mechanism_package.mechanism.competitive
  mechanism_brief.scorecard_snapshot → mechanism_package.quality_scores

  SIMPLIFICATION RULE:
    The package is FLATTER than the brief
    Remove nesting where possible
    Downstream skills need quick access, not deep structure
    If a value requires traversing 3+ levels → flatten it
```

### Step 2: Compile Deployment Section

```
FROM copy_integration_map:

  EXTRACT key deployment guidance for each touchpoint
  CONDENSE into actionable summaries (not full map detail)

  FOR each touchpoint:
    KEEP: the decision/instruction
    REMOVE: the reasoning behind it (that's in the map if needed)

  EXAMPLE:
    copy_integration_map.headlines.form = "curiosity_gap"
    copy_integration_map.headlines.examples = [3 examples]
    copy_integration_map.headlines.caution = "never reveal full mechanism in headline"

    → package.deployment.headlines = {
        form: "curiosity_gap",
        examples: [3 examples],
        caution: "never reveal full mechanism in headline"
      }
```

### Step 3: Generate Downstream Instructions

```
FOR each downstream skill (04 through 09):

  GENERATE specific, actionable instructions:

  RULE: Instructions must be:
    1. SPECIFIC — not "use mechanism appropriately"
    2. ACTIONABLE — tells them WHAT to do, not just what to consider
    3. BOUNDED — tells them what NOT to do as well
    4. SELF-CONTAINED — doesn't require reading other sections

  FOR 04-headlines:
    - When to use mechanism name vs. tease
    - How much explanation is appropriate (none/hint)
    - Whether proof belongs in headlines for this mechanism
    - Name format in headlines (quotes? caps? descriptor?)

  FOR 05-leads:
    - Where mechanism first appears (paragraph number)
    - How much to reveal in lead vs. save for body
    - Emotional setup required before mechanism mention
    - Whether lead should BUILD TO mechanism or START WITH it

  FOR 06-vsl:
    - Timing of mechanism reveal (minutes in)
    - Visual/presentation requirements
    - Pacing and build-up structure
    - How mechanism connects to hook established earlier

  FOR 07-proof:
    - Which claims specifically need proving
    - What proof types are available
    - Integration style (narrative vs. section vs. sidebar)
    - Objection preemption language available

  FOR 08-offers:
    - How mechanism justifies the price point
    - Exclusivity language ready to deploy
    - How product components connect back to mechanism
    - Value anchoring using mechanism proof

  FOR 09-closes:
    - Final mechanism reference (what form)
    - How mechanism creates urgency
    - Future pacing language using mechanism
    - Decision framing using mechanism logic
```

### Step 4: Add Runner-Up Context

```
FROM mechanism_selection:

  INCLUDE runner-up summary for testing/pivot scenarios:

  runner_up:
    name: [the second-place mechanism name]
    score: [its average score]
    when_to_use: [conditions favoring runner-up over winner]
    key_difference: [the strategic difference between them]

  PURPOSE: If winner fails in copy testing (low conversion,
  audience confusion), the runner-up is immediately available
  without re-running the full mechanism pipeline.
```

### Step 5: Validate Package Completeness

```
COMPLETENESS CHECKS:

  CRITICAL (package CANNOT ship without):
    □ mechanism.name populated
    □ mechanism.explanations.level_1_sentence populated
    □ mechanism.explanations.level_2_paragraph populated
    □ mechanism.proof.primary populated with specific citation
    □ mechanism.emotional.before_state populated
    □ mechanism.emotional.after_state populated
    □ mechanism.competitive.inevitability populated
    □ deployment.reveal.structure_summary populated
    □ quality_scores.scorecard_average populated
    □ downstream_instructions.for_headlines populated
    □ downstream_instructions.for_leads populated

  IMPORTANT (should be populated):
    □ All analogy levels
    □ All downstream instructions (04-09)
    □ Runner-up context
    □ Consistency rules

  IF any CRITICAL field missing:
    status = "incomplete"
    FLAG which fields are missing
    TRACE back to source skill failure
    DO NOT ship package

  IF all CRITICAL fields present but some IMPORTANT missing:
    status = "complete_with_warnings"
    LOG warnings
    SHIP package (downstream can work without these)

  IF all fields populated:
    status = "complete"
    SHIP package
```

### Step 6: Format Verification

```
FINAL FORMAT CHECKS:

  1. All string values are actual content (not "TBD" or "[fill in]")
  2. All arrays have at least one element (no empty arrays for required fields)
  3. All scores are numbers (not strings of numbers)
  4. No internal references ("see section X") — package is self-contained
  5. No mechanism jargon from the BUILD process — only mechanism content
     (no references to "scorecard," "Layer 2," or "skill 2.7" in the package)
  6. Language register consistent throughout (not mixing formal/casual)
  7. Mechanism name spelled identically everywhere it appears

SELF-CONTAINMENT TEST:
  "If this package were the ONLY file a downstream skill received,
   could it do its job completely?"

  IF NO → identify what external knowledge is assumed and add it to the package
```

## Specimen Provenance Assembly

Collect and package provenance data from all upstream layers that used specimen-grounded generation or writing protocols.

```
ASSEMBLE specimen_provenance:

  COLLECT from upstream Layer 1 outputs (Study-Then-Generate):
    FOR each Layer 1 skill that tracked provenance:
      - skill_id: string
        specimens_studied: [source_file references]
        templates_applied: [template identifiers]
        techniques_used: [technique names]

  COLLECT from upstream Layer 2 outputs (Specimen-Grounded Writing):
    FOR each Layer 2 skill that tracked provenance:
      - skill_id: string
        power_elements_referenced: [source_file references]
        techniques_applied: [technique names]
        quality_calibration_source: [Gold specimen IDs]

  COMPILE provenance_summary:
    total_specimens_referenced: integer
    unique_source_files: [string]
    techniques_catalog_used: [string]
    gold_specimens_as_benchmarks: [string]

  APPEND specimen_provenance to output package

  NOTE: If upstream skills did not track provenance (e.g., project predates
  specimen integration), set specimen_provenance to null and log:
  "Specimen provenance not available — upstream skills did not track."
```

## Quality Criteria

1. Package is self-contained — no external references required
2. All critical fields populated with actual content
3. Downstream instructions are specific and actionable per skill
4. Runner-up available for testing/pivot scenarios
5. No build-process jargon leaks into package content
6. Mechanism name consistent across all fields
7. Package status accurately reflects completeness
8. Any downstream skill (04-09) can consume this without questions
