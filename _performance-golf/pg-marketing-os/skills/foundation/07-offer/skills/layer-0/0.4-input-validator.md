# Skill 0.4: Input Validator

**Version:** 1.0
**Layer:** 0 (Foundation)
**Type:** Validation Gate
**Dependencies:** upstream_data.json (0.1), vault_offer_patterns.json (0.2), teaching_frameworks.json (0.3)
**Output:** gate_0_report.json

---

## Purpose

Validate that all inputs from the three foundation loaders (0.1, 0.2, 0.3) are present, well-formed, and sufficient for offer construction to begin. This is the quality gate that determines whether the system has enough raw material to build a viable offer.

No offer construction proceeds until this gate passes. The cost of building on incomplete foundations is an offer that falls apart under scrutiny.

---

## The Principle

**"A gate that lets garbage through doesn't protect anything. A gate that blocks everything doesn't produce anything. The standard is: enough signal to build something real."**

## Validation Logic

### Tier 1: Existence Checks (Hard Requirements)

```
CRITICAL FIELDS — offer CANNOT be built without these:

CHECK upstream_data EXISTS and is non-empty
  → FAIL if missing: "No upstream data loaded"

CHECK the following fields are present and non-empty:

  FROM market_intelligence:
    ✓ market_psychology.pain_points      — Must know what hurts
    ✓ customer_language                  — Must speak their words
    ✓ objections_inventory               — Must know what stops them

  FROM proof_inventory:
    ✓ proof_assets                       — Must have evidence to cite

  FROM root_cause:
    ✓ root_cause_statement               — Must know WHY the problem exists

  FROM mechanism:
    ✓ mechanism_name                     — Must have a named mechanism
    ✓ mechanism_type                     — Must know what kind
    ✓ differentiation_claim              — Must know how it's different

  FROM promise:
    ✓ primary_promise                    — Must have the core transformation
    ✓ promise_specifics                  — Must have measurable outcomes

  FROM big_idea:
    ✓ big_idea_statement                 — Must have the overarching concept

EACH missing critical field = +1 CRITICAL_FAILURE count

IF CRITICAL_FAILURE count > 0:
  GATE_0 = FAIL
  REPORT which fields are missing
  RECOMMEND: "Return to upstream skill [X] and complete [field]"
```

### Tier 2: Completeness Checks (Soft Requirements)

```
IMPORTANT FIELDS — offer is BETTER with these but can proceed without:

  FROM market_intelligence:
    ○ prospect_awareness_level           — Helps calibrate offer sophistication
    ○ competitor_landscape               — Helps differentiation
    ○ demographic_profile                — Helps avatar targeting
    ○ buying_triggers                    — Helps urgency/scarcity design

  FROM proof_inventory:
    ○ proof_strength_ratings             — Helps prioritize proof usage
    ○ proof_gaps                         — Helps identify guarantee needs
    ○ credibility_anchors                — Helps authority positioning

  FROM root_cause:
    ○ conventional_failures              — Helps differentiation framing
    ○ insight_cascade                    — Helps value demonstration
    ○ emotional_root                     — Helps consequence amplification

  FROM mechanism:
    ○ mechanism_explanation              — Helps delivery vehicle design
    ○ mechanism_proof_map                — Helps proof integration
    ○ scorecard_scores                   — Helps quality benchmarking

  FROM promise:
    ○ promise_proof_alignment            — Helps guarantee design
    ○ promise_scope                      — Helps price architecture
    ○ sub_promises                       — Helps bonus design

  FROM big_idea:
    ○ intellectually_interesting         — Helps offer naming
    ○ emotionally_compelling             — Helps value demonstration
    ○ campaign_angle                     — Helps positioning

COUNT total_soft_fields = 19
COUNT present_soft_fields
CALCULATE completeness_percentage = present / total × 100

IF completeness_percentage < 50%:
  GATE_0 = WARNING
  NOTE: "Offer construction can proceed but quality may suffer"
  RECOMMEND: "Consider completing [missing fields] before proceeding"
```

### Tier 3: Vault Intelligence Validation

```
CHECK vault_offer_patterns EXISTS and is non-empty

VALIDATE:
  ✓ files_extracted ≥ 30         — Minimum empirical reference
  ✓ exemplars list is non-empty  — Must have models to reference
  ✓ frequency_analysis populated — Must have pattern data
  ✓ language_library populated   — Must have presentation language

IF vault data missing entirely:
  GATE_0 = WARNING (not FAIL — offer can be built from frameworks alone)
  NOTE: "Proceeding without vault intelligence. Offer will lack empirical grounding."

IF vault data present but thin (< 30 files extracted):
  GATE_0 = WARNING
  NOTE: "Limited vault intelligence. Offer patterns will rely more heavily on framework theory."
```

### Tier 4: Teaching Framework Validation

```
CHECK teaching_frameworks EXISTS and is non-empty

VALIDATE Hormozi Framework:
  ✓ value_equation present with all 4 levers
  ✓ grand_slam_construction present with all 5 steps
  ✓ enhancement_stack present (scarcity, urgency, bonuses, guarantees, naming)
  ✓ pricing_principles present

VALIDATE Todd Brown Framework:
  ✓ dfwbp_formula present with all 5 components
  ✓ presentation_sequence present with minimum 10 chunks
  ✓ value_demonstration technique present
  ✓ price_presentation sequence present
  ✓ three_cta_structure present

VALIDATE Intersection Map:
  ✓ framework_intersection_map present
  ✓ minimum 3 intersection points identified

IF either framework missing entirely:
  GATE_0 = FAIL
  NOTE: "Cannot build offer without theoretical framework"

IF intersection map missing:
  GATE_0 = WARNING
  NOTE: "Frameworks loaded but not cross-referenced. Manual coordination required."
```

### Tier 5: Cross-Input Coherence Check

```
VALIDATE that inputs tell a coherent story:

CHECK 1: Mechanism-Promise Alignment
  Does mechanism_name relate to primary_promise?
  Does mechanism_type support the transformation claimed?
  → If disconnect: FLAG "mechanism-promise incoherence"

CHECK 2: Market-Promise Alignment
  Does primary_promise address documented pain_points?
  Does promise language match customer_language?
  → If disconnect: FLAG "market-promise incoherence"

CHECK 3: Proof-Mechanism Coverage
  Do proof_assets cover mechanism's key claims?
  Are there major mechanism claims with zero proof?
  → If gap: FLAG "proof-mechanism gap" with specifics

CHECK 4: Big Idea-Campaign Coherence
  Does big_idea_statement connect to mechanism and promise?
  Does campaign_angle suggest a coherent offer direction?
  → If disconnect: FLAG "big-idea-campaign incoherence"

CHECK 5: Vault-Framework Compatibility
  Do vault patterns support the framework approaches being used?
  Are there vault patterns that conflict with framework recommendations?
  → If conflict: NOTE for Layer 1 consideration

COMPILE coherence_score:
  checks_passed / total_checks × 100

IF coherence_score < 60%:
  GATE_0 = WARNING
  NOTE: "Upstream inputs are internally inconsistent. Offer may need to choose which inputs to prioritize."
```

---

## Gate Decision Logic

```
GATE_0 DECISION MATRIX:

PASS:
  - Zero CRITICAL_FAILURE count (all hard requirements met)
  - completeness_percentage ≥ 50%
  - Teaching frameworks loaded
  - coherence_score ≥ 60%

PASS WITH WARNINGS:
  - Zero CRITICAL_FAILURE count
  - completeness_percentage 30-49%
  - OR vault intelligence thin/missing
  - OR coherence_score 40-59%
  → Proceed but flag limitations for downstream skills

FAIL:
  - Any CRITICAL_FAILURE (missing hard requirements)
  - OR completeness_percentage < 30%
  - OR both teaching frameworks missing
  - OR coherence_score < 40%
  → Do not proceed. Return to upstream skills.
```

## Quality Criteria

1. All 5 validation tiers executed completely
2. Every flag and warning includes specific field names and source skills
3. Recommendations are actionable ("Return to Skill 03 and complete mechanism_proof_map")
4. Gate decision is deterministic (same inputs always produce same decision)
5. No false passes — if critical data is missing, gate MUST fail
