# Skill 4.3: Handoff Packager

**Version:** 1.0
**Layer:** 4 (Package Assembly)
**Type:** Integration & Handoff
**Dependencies:** offer_brief.json (4.1), presentation_script.json (4.2), all Layer 3 validation outputs
**Output:** offer-package.json (FINAL OUTPUT)
**Persona:** Alex Rivera (Strategic Integration) + Dr. James Liu (Research Director)

---

## Purpose

Package the complete offer architecture for downstream consumption by copy production skills (VSL, email, landing page, ads). This is the final skill in the 07-offer pipeline. Its output — `offer-package.json` — becomes the input contract for every skill that needs to write copy about this offer.

The handoff packager ensures that downstream skills receive EXACTLY what they need: no more (which creates confusion), no less (which creates gaps).

---

## The Principle

**"The handoff is a contract. If the downstream skill needs it, it must be in the package. If the downstream skill doesn't need it, it shouldn't be. Precision in handoff prevents drift in execution."**

### Step 2: Build Downstream Handoff Contracts

```
EACH downstream skill needs SPECIFIC offer data.
Package exactly what each skill requires:

HANDOFF TO VSL (Skill 17):
  to_vsl:
    what_they_need:
      - Full presentation script (14 chunks)
      - Product name and positioning
      - Dream outcome statement
      - Mechanism name and one-liner
      - Value stack with all components
      - All bonus D-F-W-B-P presentations
      - "If all it did was..." script
      - Price presentation sequence
      - Guarantee conversational version
      - CTA sequence with urgency/scarcity
      - Proof elements for integration
      - Consequence amplification language
    format: "Full script ready for VSL adaptation"
    notes: "VSL will expand and adapt but the structure and
            components are locked from this package."

HANDOFF TO EMAIL (Skill 15):
  to_email:
    what_they_need:
      - Product name and offer name
      - Primary dream outcome (headline-ready)
      - Quick reference section
      - Guarantee short version
      - Urgency short version
      - Scarcity short version
      - Top 3 benefits (for subject lines)
      - Price with savings statement
      - 3 CTA variations
      - Consequence amplification points (for urgency emails)
      - Bonus highlights (for bonus-focused emails)
    format: "Modular components for email sequence assembly"
    notes: "Email skill will create multiple emails pulling different
            components. Each email doesn't use everything."

HANDOFF TO LANDING PAGE (Skill 16):
  to_landing_page:
    what_they_need:
      - Offer name + tagline
      - Hero section content (dream outcome + mechanism)
      - Value stack for visual presentation
      - Bonus section content
      - Guarantee badge content
      - Price presentation for order form area
      - CTA button copy (3 variations)
      - Urgency/scarcity bar content
      - Testimonial placement guidance (based on proof assets)
      - FAQ candidates (from objections + guarantee conditions)
    format: "Section-by-section content mapped to landing page layout"
    notes: "LP skill handles visual design and layout.
            Offer package provides the content to place."
```

### Step 3: Final Package Validation

```
BEFORE outputting the final package:

COMPLETENESS CHECK:
  FOR each section (1-7):
    Verify no empty or null fields
    Verify no placeholder text (brackets, TBD, etc.)
    Verify data matches source outputs exactly

CONSISTENCY CHECK:
  Product name is identical everywhere it appears
  Price is identical everywhere it appears
  Guarantee terms are identical everywhere they appear
  Dream outcome is thematically consistent (varied language, same meaning)

HANDOFF CHECK:
  FOR each downstream handoff:
    Verify all required fields are present
    Verify format matches what the downstream skill expects
    Verify no upstream-internal terminology leaked into handoffs

GATE CHECK:
  Confirm gate_1_result is PASS or CONDITIONAL_PASS
  Confirm gate_2_result is PASS or CONDITIONAL_PASS
  Confirm anti_slop_result is PASS
  IF any gate failed: PACKAGE SHOULD NOT BE ASSEMBLED
    (This should have been caught earlier, but double-check)
```

## Quality Criteria

1. All 7 sections of offer-package.json complete with no empty fields
2. All 3 downstream handoffs packaged with correct data
3. Completeness, consistency, and handoff checks all pass
4. Product name, price, guarantee, and dream outcome are consistent across all sections
5. No internal/framework terminology in prospect-facing sections
6. No placeholder text anywhere in the package
7. Gates confirmed (1 + 2 passed, anti-slop passed)
8. Package status is READY

## Success Statement

When this skill produces a READY package, the 06-Offer skill is COMPLETE. The offer has been:

1. **Constructed** (Layer 1) — Dream outcome crystallized, problems mapped to solutions, value equation optimized, delivery vehicles designed, value stack trimmed and assembled, core offer defined.

2. **Enhanced** (Layer 2) — Bonuses designed, guarantee architected, scarcity and urgency mechanisms created, offer named, price architected, value demonstration scripted.

3. **Validated** (Layer 3) — Value equation scored, vault patterns compared, competitor differentiation confirmed, promise-offer alignment verified, anti-slop check passed.

4. **Packaged** (Layer 4) — Offer brief compiled, presentation script assembled, downstream handoffs prepared.

The output `offer-package.json` is the single source of truth for everything about this offer.
