# Skill 3.4: Promise-Offer Alignment Validator

**Version:** 1.0
**Layer:** 3 (Validation & Selection)
**Type:** Coherence Validation
**Dependencies:** core_offer_definition.json (1.6), value_equation_scores.json (3.1), upstream_data.json (0.1)
**Output:** promise_alignment.json
**Persona:** Alex Rivera (Strategic Integration) + Dr. Richard Stern (Skeptic)

---

## Purpose

Validate that the constructed offer actually DELIVERS on the promise made by Skill 04 (Promise). This is the critical coherence check between what we SAID we'd do (promise) and what we BUILT to do it (offer). A misaligned offer-promise relationship creates one of two failures: broken promises (the offer can't deliver) or under-sold offers (the offer delivers more than the promise claims).

---

## The Principle

**"The offer is the receipt for the promise. Every claim in the promise must be redeemable through a specific component of the offer. No IOU's."**

---

## Alignment Process

### Step 1: Decompose the Promise

```
FROM upstream_data → promise (Skill 11 output):

BREAK the primary promise into checkable claims:

CLAIM TYPE 1: TRANSFORMATION CLAIM
  "You will [achieve/become/experience X]"
  Extract every transformation claimed
  List each as a testable statement

CLAIM TYPE 2: TIMEFRAME CLAIM
  "In [X days/weeks/months]"
  Extract every time commitment made
  List each as a verifiable deadline

CLAIM TYPE 3: METHOD CLAIM
  "Using [mechanism/method/system]"
  Extract every methodology reference
  List each as a deliverable requirement

CLAIM TYPE 4: EASE CLAIM
  "Without [sacrifice/effort/prerequisite]"
  Extract every ease/simplicity promise
  List each as an effort constraint

CLAIM TYPE 5: PROOF CLAIM
  "Proven by [evidence/results/testimonials]"
  Extract every evidence reference
  List each as a proof requirement

ALSO decompose sub_promises:
  FOR each sub-promise from Skill 04:
    Break into the same claim types
    List each as a testable statement

COMPILE: promise_claims_inventory
  Total claims: [count]
  By type: transformation [N], timeframe [N], method [N], ease [N], proof [N]
```

### Step 2: Map Claims to Offer Components

```
FOR each claim in promise_claims_inventory:

  FIND the offer component(s) that DELIVER this claim:

  SEARCH across:
    - Core product components (from value_stack)
    - Bonuses (from bonus_designs)
    - Guarantee (from guarantee_architecture)
    - Mechanism explanation (from mechanism_package)
    - Delivery vehicles (from delivery_vehicles)

  CLASSIFICATION:

  FULLY COVERED:
    A specific offer component directly delivers this claim
    The component's description explicitly addresses this promise
    Evidence: [which component, how it delivers]

  PARTIALLY COVERED:
    An offer component addresses this claim but not completely
    There's a gap between what's promised and what's delivered
    Evidence: [which component, what's covered, what's missing]

  NOT COVERED:
    No offer component addresses this claim
    The promise was made but the offer doesn't deliver it
    Risk: BROKEN PROMISE — must be remediated

  OVER-DELIVERED:
    The offer delivers MORE than what was promised for this claim
    Opportunity: Could strengthen the promise or highlight in marketing
    Evidence: [which component delivers more than promised]

MAP every claim to its coverage status
```

### Step 3: Coverage Analysis

```
CALCULATE coverage metrics:

  total_claims: count
  fully_covered: count
  partially_covered: count
  not_covered: count
  over_delivered: count

  coverage_percentage = (fully_covered + 0.5 × partially_covered) / total_claims × 100

  COVERAGE ASSESSMENT:
    ≥ 95%: EXCELLENT — offer fully supports the promise
    85-94%: GOOD — minor gaps to address
    70-84%: CONCERNING — notable gaps that could disappoint
    < 70%: CRITICAL — offer cannot deliver on the promise

  BROKEN PROMISE COUNT = not_covered claims
    ZERO TOLERANCE for transformation claims not covered
    WARNING for timeframe or ease claims not fully covered
    FLAG proof claims not covered for proof gap remediation
```

### Step 4: Reverse Check — Offer Components Without Promise Support

```
FOR each offer component (core + bonuses):

  DOES this component support a specific promise claim?

  IF YES: Document the connection
  IF NO: This component exists but has no promise justification

  COMPONENTS WITHOUT PROMISE CONNECTION:
    These aren't necessarily bad — they could be:
    - Bonuses that address objections (valid)
    - Accelerators that exceed the promise (valid)
    - Components that don't serve the promise at all (concern)

  FLAG any component that doesn't connect to ANY promise element
  EVALUATE: Is it valuable anyway, or is it filler?
```

### Step 5: Strength-of-Connection Assessment

```
FOR each fully covered claim:

  RATE the strength of connection (1-10):

  10: Component was DESIGNED to deliver this specific claim
      The connection is explicit, obvious, and unbreakable

  7-9: Component clearly delivers this claim
       The connection is evident even if not explicitly designed for it

  4-6: Component addresses this claim among other things
       The connection exists but isn't the primary purpose

  1-3: Component tangentially relates to this claim
       The connection is a stretch

  AVERAGE connection strength across all covered claims
  MINIMUM: Average ≥ 7.0 for a strong offer-promise alignment
```

### Step 6: Calculate Alignment Score

```
PROMISE-OFFER ALIGNMENT COMPOSITE:

  Base score = coverage_percentage / 10 (converts to 1-10 scale)

  ADJUSTMENTS:
    Broken promises (not covered):
      -1.0 per transformation claim not covered
      -0.5 per timeframe/ease claim not covered
      -0.3 per proof claim not covered

    Connection strength:
      IF average connection < 7.0: -0.5
      IF average connection < 5.0: -1.0

    Over-delivery:
      IF over_delivered > 0: +0.3 per claim (max +1.0)

  alignment_score = base + adjustments (capped at 10.0)

THRESHOLD: ≥ 8.0/10

IF < 8.0:
  Priority remediations:
  1. Cover all not_covered transformation claims
  2. Strengthen partially_covered claims
  3. Improve connection strength for weak connections
```

---

## Quality Criteria

1. Every promise claim decomposed and categorized by type
2. Every claim mapped to offer components with coverage status
3. Zero NOT_COVERED transformation claims (hard requirement)
4. Coverage percentage ≥ 85%
5. Average connection strength ≥ 7.0
6. Reverse check completed — filler components identified
7. Alignment score ≥ 8.0/10
8. Any broken promises flagged with specific remediation

---

## Anti-Patterns

- Do NOT assume the promise is wrong if the offer doesn't cover it — the offer may need to be expanded
- Do NOT ignore partially covered claims — these are future customer complaints
- Do NOT count a bonus as "covering" a core transformation promise — core must deliver core
- Do NOT inflate connection strength — weak connections create disappointed customers
- Do NOT modify the promise to match the offer without flagging this as a recommendation (promise was validated upstream)
