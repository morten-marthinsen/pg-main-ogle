# Skill 4.2: Downstream Mapper

**Version:** 1.0
**Layer:** 4 (Package)
**Type:** Mapping
**Dependencies:** layer3-output.json (from 3.6), BIG-IDEA-BRIEF.md (from 4.1)
**Output:** downstream_map.json

---

## Purpose

Map each approved Big Idea candidate's components to the specific downstream skills that will consume them. Each downstream skill (03-root-cause, 04-mechanisms, 04-headlines, 05-leads, 06-proof-demonstration, 07-offers, 08-closes) needs different data from the Big Idea output. This mapper creates an explicit extraction guide so no downstream skill needs to parse the entire brief.

---

## The Question

**"For each downstream skill, what exactly should it extract from this Big Idea output, where is it located, and in what format does it need it?"**

---

## Input Schema

```yaml
input:
  layer3_output: from 3.6
    compiled_output:
      big_idea_candidates: [5 candidate objects]
    approved_candidates: [5 entries]

  big_idea_brief: from 4.1 (BIG-IDEA-BRIEF.md)
```

## Mapping Logic

### Map 1: Root Cause Engineering (03-root-cause)

```
EXTRACTS:
  1. root_cause_reframe.what_they_think
     - Source: candidate.root_cause_reframe.what_they_think
     - Format: string (surface-level belief statement)
     - Per candidate: YES

  2. root_cause_reframe.what_it_really_is
     - Source: candidate.root_cause_reframe.what_it_really_is
     - Format: string (deep diagnosis)
     - Per candidate: YES

  3. root_cause_reframe.why_nothing_worked
     - Source: candidate.root_cause_reframe.why_nothing_worked
     - Format: string (previous solution failure explanation)
     - Per candidate: YES

  4. root_cause_reframe.technique_name
     - Source: candidate.root_cause_reframe.technique_name
     - Format: enum (false_solution_expose | enemy_reveal | scientific_discovery |
                      conspiracy_reveal | blame_removal | contrarian_truth)
     - Per candidate: YES

  5. schwartz_stage + emphasis_level
     - Source: header.schwartz_stage + candidate.root_cause_reframe.emphasis_level
     - Format: "Stage [X] → [heavy|medium|light] emphasis"
     - Per candidate: YES (emphasis may vary)

  6. enemy_identified
     - Source: candidate.enemy (full object)
     - Format: {name, type, how_caused_problem}
     - Per candidate: YES

CONSUMPTION NOTES:
  "Root Cause skill uses this to develop the full Root Cause argument —
   expanding the reframe into complete persuasion sequences, building
   the 'why nothing worked' narrative, and engineering the belief shift.
   The technique_name determines the rhetorical pattern. The enemy connects
   Root Cause to the mechanism reveal."
```

### Map 2: Mechanism Creation (04-mechanisms)

```
EXTRACTS:
  1. mechanism.mechanism_name
     - Source: candidate.mechanism.mechanism_name
     - Format: string (named mechanism)
     - Per candidate: YES

  2. mechanism.naming_pattern_used
     - Source: candidate.mechanism.naming_pattern_used
     - Format: string (pattern template)
     - Per candidate: YES

  3. mechanism.uniqueness_claim
     - Source: candidate.mechanism.uniqueness_claim
     - Format: string (differentiator statement)
     - Per candidate: YES

  4. mechanism.steps_or_components
     - Source: candidate.mechanism.steps_or_components
     - Format: [string] (3-5 named steps)
     - Per candidate: YES

  5. mechanism.swipe_sources
     - Source: candidate.mechanism.swipe_sources
     - Format: [string] (swipe IDs for reference)
     - Per candidate: YES

  6. root_cause_connection
     - Source: candidate.root_cause_reframe.what_it_really_is
     - Format: string (the problem the mechanism solves)
     - Per candidate: YES

  7. saturation_context
     - Source: layer1_output.saturation_mapping (matching mechanism zone)
     - Format: {zone, competitor_count, recommendation}
     - Per candidate: YES

CONSUMPTION NOTES:
  "Mechanism skill uses this as the seed for full mechanism development —
   expanding the named steps into detailed protocol descriptions, building
   the scientific/logical foundation, and engineering the 'why this works
   when nothing else did' argument. The saturation context ensures the
   mechanism maintains competitive differentiation."
```

### Map 3: Headline Engineering (04-headlines)

```
EXTRACTS:
  1. headlines (all)
     - Source: candidate.headlines[]
     - Format: [{text, formula_type, emotional_driver, swipe_source, usage_count}]
     - Per candidate: YES

  2. mechanism_name (for headline integration)
     - Source: candidate.mechanism.mechanism_name
     - Format: string
     - Per candidate: YES

  3. enemy_name (for headline integration)
     - Source: candidate.enemy.enemy_name
     - Format: string
     - Per candidate: YES

  4. promise_statement (for headline integration)
     - Source: candidate.promise.statement
     - Format: string
     - Per candidate: YES

  5. emotional_drivers
     - Source: candidate.emotional_driver_combination
     - Format: {primary, secondary, combined_score}
     - Per candidate: YES

  6. formula_performance_data
     - Source: layer1_output.pattern_analysis (headline formula usage/quality)
     - Format: [{formula_type, swipe_count, avg_quality}]
     - Shared: YES (same for all candidates)

CONSUMPTION NOTES:
  "Headline skill uses these as starting points for headline testing sets —
   expanding each seed headline into multiple variations, A/B test pairs,
   and format adaptations (email subject lines, ad copy, etc.). The formula
   performance data guides which formulas to prioritize for variation."
```

### Map 4: Lead Writing (05-leads)

```
EXTRACTS:
  1. lead_variations (all)
     - Source: candidate.lead_variations[]
     - Format: [{technique, narrative_structure, swipe_source,
                  opening_hook, root_cause_intro, mechanism_tease}]
     - Per candidate: YES

  2. root_cause_reframe (for lead expansion)
     - Source: candidate.root_cause_reframe (full object)
     - Format: {what_they_think, what_it_really_is, why_nothing_worked}
     - Per candidate: YES

  3. emotional_intensity
     - Source: candidate.emotional_driver_combination
     - Format: {primary_driver, pain_quote_count, hope_quote_count, intensity_rationale}
     - Per candidate: YES

  4. mechanism_tease_elements
     - Source: candidate.mechanism (name, uniqueness_claim, first_step)
     - Format: {name, hook, first_step_preview}
     - Per candidate: YES

  5. schwartz_stage
     - Source: header.schwartz_stage
     - Format: integer (1-5)
     - Shared: YES

CONSUMPTION NOTES:
  "Lead skill uses these as structural templates for full lead development —
   expanding the 3-section seeds (hook, intro, tease) into complete lead
   copy (1000-3000 words), adding transitional elements, proof previews,
   and belief-shifting sequences. The Schwartz stage determines how much
   awareness-building is needed before the mechanism reveal."
```

### Map 5: Proof Engineering (06-proof-demonstration)

```
EXTRACTS:
  1. proof_architecture (full)
     - Source: candidate.proof_architecture
     - Format: {primary_type, secondary_type, scientific_elements,
                transformation_elements, demonstration_plan,
                authority_elements, social_proof_elements, priority_order}
     - Per candidate: YES

  2. mechanism_to_prove
     - Source: candidate.mechanism (full object)
     - Format: {name, steps, uniqueness_claim}
     - Per candidate: YES

  3. promise_to_validate
     - Source: candidate.promise.statement
     - Format: string (what needs proof)
     - Per candidate: YES

  4. proof_swipe_sources
     - Source: candidate.proof_architecture.swipe_sources
     - Format: [string] (reference swipes for proof patterns)
     - Per candidate: YES

CONSUMPTION NOTES:
  "Proof skill uses this as the blueprint for creating actual proof assets —
   scripting demonstration sequences, structuring case studies, defining
   research citation requirements, and planning testimonial collection.
   The priority_order determines the sequence of proof presentation in
   the final sales piece."
```

### Map 6: Offer Construction (07-offers)

```
EXTRACTS:
  1. promise_statement
     - Source: candidate.promise (full object)
     - Format: {statement, specificity_markers}
     - Per candidate: YES

  2. mechanism_name
     - Source: candidate.mechanism.mechanism_name
     - Format: string (becomes product name basis)
     - Per candidate: YES

  3. transformation_elements
     - Source: candidate.proof_architecture.transformation_elements
     - Format: {before_after_plan, case_study_structure, volume_target}
     - Per candidate: YES

  4. emotional_drivers
     - Source: candidate.emotional_driver_combination
     - Format: {primary, secondary} (for offer framing)
     - Per candidate: YES

CONSUMPTION NOTES:
  "Offer skill uses the mechanism name as product naming seed, the promise
   as value proposition foundation, the transformation elements as guarantee
   backing, and the emotional drivers as urgency/desire framing. The Big Idea
   determines the positioning angle of the entire offer stack."
```

### Map 7: Close & CTA (08-closes)

```
EXTRACTS:
  1. enemy_identified
     - Source: candidate.enemy (full object)
     - Format: {name, type, how_caused_problem}
     - Per candidate: YES (for consequence framing)

  2. promise_statement
     - Source: candidate.promise.statement
     - Format: string (for CTA promise)
     - Per candidate: YES

  3. mechanism_name
     - Source: candidate.mechanism.mechanism_name
     - Format: string (for CTA product reference)
     - Per candidate: YES

  4. emotional_drivers
     - Source: candidate.emotional_driver_combination
     - Format: {primary, secondary, intensity_rationale}
     - Per candidate: YES (for urgency/fear framing)

CONSUMPTION NOTES:
  "Close skill uses the enemy for consequence framing (what happens if they
   don't act), the promise for CTA vision (what they get), the mechanism name
   for product reference, and the emotional drivers for urgency calibration.
   Higher intensity drivers warrant stronger close language."
```

### Cross-Skill Dependencies

```
BUILD dependency chain:
  06-big-idea → 03-root-cause (root_cause_reframe, enemy)
  06-big-idea → 04-mechanisms (mechanism seed, saturation context)
  06-big-idea → 04-headlines (headline seeds, formula data)
  06-big-idea → 05-leads (lead seeds, emotional intensity)
  06-big-idea → 06-proof-demonstration (proof architecture)
  06-big-idea → 07-offers (promise, mechanism name)
  06-big-idea → 08-closes (enemy, promise, emotional drivers)

  03-root-cause → 05-leads (expanded root cause argument)
  04-mechanisms → 05-leads (full mechanism description)
  04-mechanisms → 06-proof-demonstration (mechanism to prove)
  06-proof-demonstration → 07-offers (proof for guarantee)
  07-offers → 08-closes (offer to close on)
```

### Candidate Readiness Check

```
FOR each approved candidate:
  CHECK all required extracts are populated:
    root_cause_reframe: all 3 fields non-empty
    mechanism: name + 3+ steps + uniqueness_claim
    headlines: 10+ entries
    leads: 3+ entries with full copy
    proof_architecture: primary + secondary types + elements
    promise: statement non-empty
    enemy: name + type non-empty
    emotional_drivers: primary + secondary populated

  downstream_ready = ALL checks pass
  IF any check fails:
    missing_for_downstream = list of missing elements
```

---

## Constraints

- NEVER create mappings for skills that don't exist in the CopywritingEngine
- NEVER reference data locations that aren't in the validated output
- NEVER assume a candidate is selected (map all 5 unless user specifies)
- ALWAYS include format specifications for each extraction
- ALWAYS include consumption_notes explaining how the downstream skill uses the data
- ALWAYS verify cross-skill dependencies form a valid DAG (no circular references)
- ALWAYS run candidate readiness check before outputting

---

## Quality Criteria

1. All 7 downstream skills have explicit extraction maps
2. Every extraction includes: field name, source location, format, per-candidate flag
3. Consumption notes explain the "why" for each extraction
4. Cross-skill dependencies form valid directed graph
5. Candidate readiness check validates all required fields exist
6. No downstream skill requires parsing the full brief (direct field access)
7. Format specifications match the receiving skill's input schema
