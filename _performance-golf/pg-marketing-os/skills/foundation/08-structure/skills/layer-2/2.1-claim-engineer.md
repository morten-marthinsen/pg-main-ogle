# Layer 2.1 — Claim Engineer

## Purpose

Craft specific, factual, defensible claims for each CPB chunk. Each claim answers a prospect gap question with a statement that can be proven through evidence. Claims are the structural atoms of the campaign argument — everything else (proof, benefits) orbits around them.

## Execution Protocol

### Step 1: Review Claim Direction for Each Chunk

From the chunk assignments, each chunk has a `claim_direction` — the general thrust of what the claim should assert. Use this as the starting point, not a rigid template.

### Step 2: Apply the Three Claim Rules

For EVERY claim drafted, apply the E5 claim rules:

**Rule 1: No Opinion-Based Statements**

| Opinion (REJECT) | Fact/Specific (ACCEPT) |
|-------------------|----------------------|
| "This is the best approach" | "This approach produces 23% faster results than conventional methods" |
| "It works amazingly well" | "In clinical testing, 87% of participants showed improvement within 14 days" |
| "You'll love the results" | "Users report an average improvement of 12 strokes within 60 days" |
| "This is unlike anything you've seen" | "This is the only approach that addresses the root cause at the cellular level" |

**Rule 2: All Claims Must Be True**
- Every claim must be verifiable through the proof inventory
- If no proof exists for a claim, it cannot be made
- "True" means defensible — not necessarily proven beyond all doubt, but supported by available evidence

**Rule 3: All Claims Must Be Specific**
- Replace vague language with numbers, names, timeframes, percentages
- Specificity creates believability — "a few weeks" → "within 14 days"
- The more specific the claim, the more credible it sounds (even before proof is offered)

### Step 3: Draft Claims Using the Interrogation Method

For each chunk, use the Makepeace interrogation approach:

1. **State the gap question** the chunk must answer
2. **Answer it directly** in a single declarative sentence
3. **Ask "Why?"** — the answer becomes supporting detail
4. **Ask "How?"** — the answer becomes mechanism proof direction
5. **Ask "Says who?"** — the answer becomes proof requirement

Example interrogation for a "mechanism" gap:
- Gap: "How and why does this work?"
- Direct answer: "The [mechanism name] works by [specific action] on [specific target]"
- Why: "Because [root cause] creates [specific condition], and [mechanism] directly addresses that condition"
- How: "By [specific process] that [specific outcome]"
- Says who: "[Study/authority/demonstration]"

### Step 4: Classify Each Claim

```yaml
claim_classification:
  claim_type: enum[factual, specific, reframed, interrogation_derived]
  definitions:
    factual: "Verifiable through external evidence (studies, data, credentials)"
    specific: "Contains specific numbers, names, or timeframes"
    reframed: "Reframes a common belief using root cause insight"
    interrogation_derived: "Derived through mechanism interrogation (why/how/says who)"
```

### Step 5: Run the Opinion Test

For every claim, apply the opinion test:

> "If I removed all proof and evidence, would this statement sound like someone's opinion?"

- If YES → **FAIL** — rewrite with specifics
- If NO → **PASS** — the claim stands on its own as a factual assertion

### Step 6: Score Each Claim

| Dimension | Weight | Scale | Description |
|-----------|--------|-------|-------------|
| Specificity | 0.35 | 1-10 | How specific is the claim? Numbers, names, timeframes? |
| Defensibility | 0.35 | 1-10 | Can this be backed by available proof? |
| Thesis support | 0.15 | 1-10 | How directly does this support the campaign thesis? |
| Conversational tone | 0.15 | 1-10 | Does this sound like spoken language, not "copy"? |

**Minimum threshold:** Weighted average ≥ 7.0/10

## Quality Gates

- [ ] Every chunk has at least one claim drafted
- [ ] ALL claims pass the opinion test (zero failures)
- [ ] Every claim scored with weighted average ≥ 7.0
- [ ] Every claim has an interrogation trail documented
- [ ] No claim uses vague/opinion language ("best", "amazing", "incredible", "unique")
- [ ] Every claim contains at least one specific element (number, name, timeframe, percentage)
- [ ] Claims read as spoken language, not "copy" or marketing jargon
- [ ] Proof requirements identified for each claim
- [ ] Aggregate all_passing = true

## Anti-Patterns

- **Opinion masquerading as fact:** "This is the most powerful system" — opinion. "This system addresses 3 root causes that other approaches ignore" — fact.
- **Vague claims:** "You'll see results fast" — vague. "Users report measurable improvement within 14 days" — specific.
- **Unsupportable claims:** Making claims the proof inventory cannot back up — every claim must be traceable.
- **Copywriter voice:** "Picture this incredible breakthrough..." — sounds like an ad, not a prosecution. "Here's what the research shows..." — sounds like evidence.
- **Feature claims:** "The system has 7 modules" — that's a feature. "The system's 7-step process eliminates the #1 cause of failure" — that's a claim.
