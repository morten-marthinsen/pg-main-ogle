# Skill 3.6: Validation Gate

**Version:** 1.0
**Layer:** 3 (Validation)
**Type:** Quality Gate
**Dependencies:** volume_validation.json (from 3.1), quantification_validation.json (from 3.2), defensibility_validation.json (from 3.3), actionability_validation.json (from 3.4), anti_slop_validation.json (from 3.5)
**Output:** layer3-output.json

---

## Purpose

Final quality gate for Layer 3 — aggregates all 5 validation results, produces the overall PASS/FAIL determination, generates the approved candidates list (if PASS), and compiles the complete validated output ready for Layer 4 packaging. All 5 validators must independently PASS. No partial passes.

---

## The Question

**"Have all candidates passed all 5 validation gates, and is the complete output ready for brief generation?"**

---

## Input Schema

```yaml
input:
  volume_validation: from 3.1

  defensibility_validation: from 3.3
  actionability_validation: from 3.4

## Gate Logic

### Step 1: Aggregate Results

```
READ status from each validation file:
  volume_status = volume_validation.status
  quantification_status = quantification_validation.status
  defensibility_status = defensibility_validation.status
  actionability_status = actionability_validation.status
  anti_slop_status = anti_slop_validation.status

pass_count = COUNT of statuses == "PASS"
fail_count = COUNT of statuses == "FAIL"
```

### Step 2: Compile Summary Stats

```
FROM volume_validation:
  total_swipes_analyzed = total_swipes_analyzed.count
  candidates_validated = big_idea_examples.candidates_checked
  headlines_validated = headline_formulas.headlines_checked
  leads_validated = lead_techniques.leads_checked
  proof_plans_validated = proof_architectures.architectures_checked

FROM quantification_validation:
  vague_qualifiers_found = violations.total_found

FROM defensibility_validation:
  invalid_sources_found = swipe_source_verification.sources_invalid

FROM actionability_validation:
  incomplete_elements_found = SUM of (template_headlines + leads_with_descriptions +
                                       mechanisms_with_placeholders + vague_proof_plans)

FROM anti_slop_validation:
  ai_tell_violations = ai_tell_patterns.total_violations
  anti_slop_violations = total_violations
```

### Step 3: Determine Gate Status

```
gate_status = IF (pass_count == 5)
              THEN "PASS"
              ELSE "FAIL"
```

### Step 4: Build Approved Candidates (IF PASS)

```
IF gate_status == "PASS":
  FOR each candidate in layer2_output:
    BUILD approved entry:
      candidate_id = candidate.candidate_id
      validation_score = "5/5 gates passed"
      ready_to_brief = true
      swipe_evidence_count = candidate.swipe_evidence_count
      headlines_ready = candidate.headlines.length
      leads_ready = candidate.lead_variations.length
      proof_plan_ready = true
```

### Step 5: Build Failure Report (IF FAIL)

```
IF gate_status == "FAIL":
  COLLECT failed_gates (list of gate names with FAIL status)

  FOR each failed gate:
    DETERMINE routing:
      volume_validation FAIL → route to "Layer 0 (expand swipe analysis)"
      quantification_validation FAIL → route to "Layer 1/2 (replace vague qualifiers)"
      defensibility_validation FAIL → route to "Layer 2 (fix/add swipe sources)"
      actionability_validation FAIL → route to "Layer 2 (complete execution elements)"
      anti_slop_validation FAIL → route to "Layer 2 (rewrite AI-tell patterns)"

  COMPILE critical_issues:
    FROM each failed validation's issues array
    SORT by severity (critical first)
    INCLUDE candidate_id, element, issue, required_fix
```

### Step 6: Compile Full Output (IF PASS)

```
IF gate_status == "PASS":
  BUILD compiled_output:
    generation_metadata:
      candidates_generated = 5
      total_headlines = SUM of all candidate headline counts
      total_leads = SUM of all candidate lead counts
      swipes_analyzed = total_swipes_analyzed
      validation_date = current timestamp

    big_idea_candidates = COPY all candidate objects from layer2_output
      (including architectures, headlines, leads, proofs)
```

Fix failures in order listed, then re-run validation.
```

---

## Quality Criteria

1. All 5 gate results aggregated correctly (no missed gates)
2. Summary stats match actual values from individual validations
3. Gate status is strictly binary: 5/5 PASS required
4. Approved candidates list only populated on overall PASS
5. Failure report routes to correct upstream skill for each failure type
6. Compiled output contains complete candidate data (not references)
7. No advancement to Layer 4 unless gate_status == PASS
8. Failure report lists issues in severity order (critical first)
