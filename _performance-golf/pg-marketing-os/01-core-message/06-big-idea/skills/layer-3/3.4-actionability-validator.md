# Skill 3.4: Actionability Validator

**Version:** 1.0
**Layer:** 3 (Validation)
**Type:** Validation Gate
**Dependencies:** layer2-output.json (from 2.6)
**Output:** actionability_validation.json

---

## Purpose

Verify all creative components are immediately executable — headlines are complete copy (not templates), leads are written paragraphs (not descriptions), mechanism steps are actionable (not placeholders), and proof plans specify exact elements to create (not vague categories). A copywriter receiving this output should need zero follow-up questions.

---

## The Question

**"Could a copywriter take these outputs and start writing immediately — without asking a single clarification question?"**

---

## Input Schema

```yaml
input:
  layer2_output: from 2.6 (all candidates with headlines, leads, mechanisms, proofs)
```

## Process Logic

### Check 1: Headlines Are Complete Copy

```
FOR each candidate:
  FOR each headline:
    SCAN headline_text for bracket patterns: /\[.*?\]/

    IF brackets found:
      FAIL
      LOG: "Candidate [id] headline [index]: Contains placeholder '[matched_text]'"
      RECORD in failures

    ALSO CHECK:
      - Is headline a complete sentence or compelling fragment? (not just a formula name)
      - Does it contain candidate-specific content? (mechanism name, enemy, root cause)
      - Is it longer than 5 words? (rejects bare fragments)

    IF headline is just a formula label (e.g., "WARNING" or "QUESTION"):
      FAIL
      LOG: "Candidate [id] headline [index]: Formula label, not actual headline"
```

### Check 2: Leads Have Written Copy

```
FOR each candidate:
  FOR each lead_variation:
    FOR each section IN [opening_hook, root_cause_intro, mechanism_tease]:
      SCAN for description patterns (case-insensitive):
        "Opens with"
        "Introduces the"
        "Tells a story about"
        "Teases the mechanism"
        "Transitions to"
        "Builds to"
        "Establishes the"
        "Sets up"
        "Describes"
        "Mentions"
        "References"
        "Explains that"

      IF any description pattern found:
        FAIL
        LOG: "Candidate [id] lead [index] [section]: Description language found: '[matched_text]'"

      ALSO CHECK opening_hook specifically:
        COUNT sentences (split by ., !, ?)
        IF sentence_count < 2:
          FAIL
          LOG: "Candidate [id] lead [index] opening_hook: Only [count] sentence (needs 2+)"

      ALSO CHECK root_cause_intro:
        IF text length < 200 characters:
          FAIL
          LOG: "Candidate [id] lead [index] root_cause_intro: Too short ([length] chars, needs 200+)"

      ALSO CHECK mechanism_tease:
        IF text length < 100 characters:
          FAIL
          LOG: "Candidate [id] lead [index] mechanism_tease: Too short ([length] chars, needs 100+)"
```

### Check 3: Mechanisms Have Named Steps

```
FOR each candidate:
  FOR each step in mechanism.steps_or_components:
    CHECK for generic placeholder patterns:
      "First step"
      "Second step"
      "Third step"
      "Take the [generic]"
      "Wait for results"
      "See the difference"
      "Follow the protocol"
      "Apply the method"
      "Use the system"

    CHECK for specificity:
      Does step contain a specific VERB? (not just "do" or "use")
      Does step contain a specific OBJECT? (not just "supplement" or "method")
      Is step longer than 4 words?

    IF generic or non-specific:
      FAIL
      LOG: "Candidate [id] step [index]: Generic placeholder: '[step_text]'"
```

### Check 4: Proof Plans Specify Elements

```
FOR each candidate:
  FOR each proof section (scientific, transformation, demonstration, authority, social):
    CHECK for vague specifications:
      "Use testimonials" (no specifics)
      "Scientific proof needed" (no detail)
      "Get expert" (no credential type)
      "Show results" (no format)
      "Research" (no specific area)

    CHECK for specificity markers:
      Does section name SPECIFIC research areas/journals/fields?
      Does section specify EXACT counts/volumes/targets?
      Does section describe WHAT to create (not just what type)?
      Does section include FORMAT details?

    IF vague (no specificity markers):
      FAIL
      LOG: "Candidate [id] proof section [section]: Vague — '[first 50 chars]'. Needs specific elements."
```

### Final Determination

```
status = IF (template_headlines == 0 AND
             leads_with_descriptions == 0 AND
             mechanisms_with_placeholders == 0 AND
             vague_proof_plans == 0)
         THEN "PASS"
         ELSE "FAIL"
```

---

## Constraints

- NEVER pass headlines containing bracket placeholders (zero tolerance)
- NEVER pass leads containing description language (zero tolerance)
- NEVER pass mechanism steps that are generic placeholders
- NEVER pass proof plans that don't specify what to create
- ALWAYS scan ALL sections of ALL candidates (no sampling)
- ALWAYS report specific candidate_id, element index, and failing text
- ALWAYS include the actual failing text snippet in the failure report
- CHECK length minimums: hooks 2+ sentences, root_cause 200+ chars, tease 100+ chars

---

## Quality Criteria

1. All headlines scanned for bracket patterns using regex
2. All lead sections scanned for 12+ description language patterns
3. All mechanism steps checked for generic placeholder patterns
4. All proof sections checked for vague vs. specific content
5. Length minimums enforced for lead sections
6. Every failure includes candidate_id, element index, and failing text
7. Status is binary: any actionability failure = FAIL
8. Issues array consolidates all failures across all checks
