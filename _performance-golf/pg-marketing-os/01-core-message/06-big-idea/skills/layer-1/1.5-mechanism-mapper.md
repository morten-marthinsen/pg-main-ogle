# Skill 1.5: Mechanism Mapper

**Version:** 1.0
**Layer:** 1 (Intelligence)
**Type:** Analysis
**Dependencies:** layer0-output.json (from 0.6)
**Output:** mechanism_mapping.json

---

## Purpose

Inventory all mechanism components — Root Cause reframe techniques, naming patterns, and proof demonstration types — matched to the market's Schwartz sophistication stage. Produces a complete mechanism toolkit showing which approaches are stage-appropriate, how mechanisms are named in winning swipes, and which proof architectures perform highest.

---

## The Question

**"What mechanism components are available, and which ones match this market's sophistication stage?"**

---

input:

## Process Logic

### Step 1: Load Schwartz Stage Context

```
READ deep_research_summary.schwartz_stage (integer 1-5)
READ deep_research_summary.root_cause_emphasis

DETERMINE expected emphasis:
  IF schwartz_stage == 5: expected_emphasis = "heavy"
  IF schwartz_stage == 4: expected_emphasis = "heavy"
  IF schwartz_stage == 3: expected_emphasis = "light"
  IF schwartz_stage <= 2: expected_emphasis = "minimal"
```

### Step 2: Analyze Root Cause Inventory

```
LOAD component_pools.root_cause_techniques

FOR each root_cause_technique:
  READ reframe_technique
  READ emphasis_level_distribution (heavy/medium/light counts)
  READ usage_count
  READ example_reframes

  CALCULATE schwartz_stage_match:
    IF expected_emphasis == "heavy":
      IF technique's dominant_emphasis == "heavy": match = "perfect"
      IF technique's dominant_emphasis == "medium": match = "good"
      IF technique's dominant_emphasis == "light": match = "poor"

    IF expected_emphasis == "light":
      IF technique's dominant_emphasis == "light": match = "perfect"
      IF technique's dominant_emphasis == "medium": match = "good"
      IF technique's dominant_emphasis == "heavy": match = "poor"

    IF expected_emphasis == "minimal":
      IF technique's dominant_emphasis == "light": match = "good"
      ELSE: match = "poor"

  WHERE dominant_emphasis = key with highest count in emphasis_level_distribution

  DETERMINE recommended_emphasis:
    Based on schwartz_stage, recommend how strongly to deploy this technique

  BUILD recommendation:
    "[match] match for Stage [stage]. Used in [usage_count] swipes.
     Dominant emphasis: [dominant] ([count] of [total] swipes at this level).
     [IF perfect/good: Recommended for this market.]
     [IF poor: Use cautiously — emphasis mismatch for Stage [stage].]"
```

### Step 3: Analyze Mechanism Naming Patterns

```
LOAD component_pools.mechanisms

EXTRACT naming patterns by analyzing mechanism_name strings:

PATTERN TYPES:
  "[Number]-[Timeframe] [Benefit]"
    MATCH: names containing a number + time unit + benefit word
    EXAMPLES: "5-Minute Metabolism Fix", "7-Second Morning Ritual"

  "[Adjective] [Scientific Term]"
    MATCH: descriptive adjective + medical/scientific terminology
    EXAMPLES: "Ancient Himalayan Secret", "Cellular Rejuvenation Protocol"

  "[Authority] [Method]"
    MATCH: credibility marker + method/protocol/system
    EXAMPLES: "Doctor's Protocol", "Harvard-Developed System"

  "[Metaphor/Object] [Effect]"
    MATCH: concrete object/metaphor + benefit/transformation
    EXAMPLES: "Metabolic Switch", "Brain Fog Eraser"

  "[Discovery] [Location/Source]"
    MATCH: discovery/finding language + origin story
    EXAMPLES: "Okinawan Longevity Code", "Amazon Rainforest Compound"

FOR each pattern_type:
  COUNT usage_frequency (how many vault mechanisms match this pattern)
  COLLECT example mechanism names (actual text from vault)
  CALCULATE average_quality_score from matching swipes

  ASSIGN quality_tier:
    >= 85: "Elite"
    75-84: "Strong"
    70-74: "Good"
    < 70: "Weak"

SORT by usage_count DESC, then average_quality_score DESC
```

### Step 4: Analyze Proof Demonstration Types

```
LOAD component_pools.proof_architectures

FOR each proof_type:
  READ usage_count
  READ average_demonstration_score
  READ average_quality_score
  COLLECT swipe_sources (minimum 3)

  ASSIGN quality_tier:
    average_demonstration_score >= 8.5: "Elite"
    average_demonstration_score >= 7.5: "Strong"
    average_demonstration_score >= 6.5: "Good"
    average_demonstration_score < 6.5: "Weak"

SORT by average_demonstration_score DESC

BUILD recommendation:
  "[tier] PROOF TYPE — [usage_count] swipes use this at [demo_score]/10
   demonstration score ([quality_score] overall quality).
   [IF Elite: Primary proof architecture recommendation.]
   [IF Strong: Strong supporting proof option.]
   [IF Good/Weak: Lower priority — consider only if primary types unavailable.]"
```

### Step 5: Build Inventory Summary

```
CALCULATE:
  total_reframe_techniques = root_cause_inventory.length
  stage_appropriate_count = COUNT where schwartz_stage_match IN ["perfect", "good"]
  total_naming_patterns = mechanism_naming_patterns.length
  total_proof_types = proof_demonstration_types.length
  top_proof_type = proof_demonstration_types[0].proof_type (highest demo score)
  top_naming_pattern = mechanism_naming_patterns[0].pattern_type (highest usage)
```

---

## Constraints

- NEVER use vague qualifiers (many, often, most, some, several, usually, typically, around, approximately)
- NEVER estimate Schwartz stage match — calculate from actual emphasis distributions
- NEVER recommend a proof type without citing demonstration score and usage count
- ALWAYS match root cause techniques to the specific Schwartz stage
- ALWAYS cite 3+ swipe sources for proof types
- ALWAYS include 3+ example mechanism names for each naming pattern
- ALL scores and counts must be exact values from pool data (not estimated)

---

## Quality Criteria

1. Schwartz stage match calculated from actual emphasis_level_distribution counts
2. All naming patterns include 3+ real mechanism name examples from vault
3. All proof types sorted by demonstration score with exact float values
4. Root cause recommendations reference specific stage match quality
5. Inventory summary counts match actual array lengths
6. Recommendations contain only quantified claims (zero vague qualifiers)
7. All swipe_sources contain valid swipe_id strings from processed vault files
