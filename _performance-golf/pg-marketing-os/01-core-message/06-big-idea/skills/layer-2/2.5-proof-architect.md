# Skill 2.5: Proof Architect

**Version:** 1.0
**Layer:** 2 (Generation)
**Type:** Architecture Design
**Dependencies:** candidate_architectures.json (from 2.2), layer1-output.json (from 1.6), layer0-output.json (from 0.6)

---

## Purpose

Design the complete proof architecture for each of the 5 candidates — primary proof type, supporting proofs, demonstration plan, authority elements, social proof targets, and proof priority order. Matches proof types to Big Idea type, Schwartz stage, and available vault evidence. Every proof element is specific and actionable (not "studies show" or "research confirms").

---

## The Question

**"Does each candidate have a specific, actionable proof plan that a production team could execute without asking follow-up questions?"**

## Proof Type Matching Rules

### By Big Idea Type

```
ENEMY_BASED candidates:
  Primary: authority_proof (expert exposing the enemy)
  Secondary: scientific_proof (research proving enemy's harm)
  Demonstration: exposé-style content showing enemy's impact

ROOT_CAUSE_REFRAME candidates:
  Primary: scientific_proof (research proving root cause)
  Secondary: transformation_proof (before/after showing root cause fix)
  Demonstration: educational content explaining the root cause mechanism

NEW_MECHANISM candidates:
  Primary: demonstration_proof (showing how mechanism works)
  Secondary: transformation_proof (results from using mechanism)
  Demonstration: step-by-step walkthrough of mechanism

SCIENTIFIC_DISCOVERY candidates:
  Primary: scientific_proof (published research)
  Secondary: authority_proof (researcher credentials)
  Demonstration: data visualization of findings

PROPHECY_WARNING candidates:
  Primary: scientific_proof (trend data supporting prediction)
  Secondary: authority_proof (expert making the warning)
  Demonstration: timeline/projection showing consequences
```

### By Schwartz Stage

```
Stage 5 (Most Sophisticated):
  REQUIRES: Heavy scientific proof + expert authority
  AVOID: Testimonial-only proof, social proof alone
  EMPHASIS: "Why" explanations, mechanism of action

Stage 3-4 (Moderate):
  REQUIRES: Mix of scientific + transformation proof
  EFFECTIVE: Before/after + expert endorsement
  EMPHASIS: Results + credibility combination

Stage 1-2 (Least Sophisticated):
  REQUIRES: Transformation proof + social proof volume
  EFFECTIVE: Testimonials, user counts, success stories
  EMPHASIS: "It works" evidence, social validation
```

## Process Logic

### Step 1: Determine Proof Type Match

```
FOR each candidate:

  LOAD:
    big_idea_type from candidate_architectures
    schwartz_stage from deep_research_summary
    mechanism from candidate_architectures
    enemy from candidate_architectures

  APPLY Big Idea Type matching rules → primary_proof_type, secondary_proof_type
  CROSS-CHECK with Schwartz Stage rules:
    IF conflict: Schwartz Stage rules override (audience sophistication > idea type)

  VERIFY proof types are proven in vault:
    CHECK component_pools.proof_architectures for matching proof_type
    RECORD usage_count and average_demonstration_score
    REQUIRE usage_count >= 3 AND average_demonstration_score >= 6.5
```

### Step 2: Build Scientific Elements

```
FOR each candidate:

  BUILD research_needed:
    - Identify 2-3 SPECIFIC research areas that support the mechanism
    - Name the field of study (not just "studies show")
    - Reference the root cause connection
    - Example: "Journal of Hepatology study on bile production decline with age"
    - NOT: "Research showing the product works"

  BUILD expert_validation:
    - Specify credential type needed (not just "a doctor")
    - Match to mechanism domain
    - Example: "Board-certified hepatologist with published research on bile metabolism"
    - NOT: "Expert in the field"

  BUILD credibility_elements:
    - List 3-5 specific credibility markers
    - Examples: "Published in peer-reviewed journal", "20+ years clinical practice",
                "Board certification in relevant specialty"
    - NOT: "Impressive credentials", "Years of experience"
```

### Step 3: Build Transformation Elements

```
FOR each candidate:

  BUILD before_after_plan:
    - Specify what MEASUREMENTS to show (not just "results")
    - Match to promise_statement specifics
    - Example: "Visual transformations showing belly fat reduction with waist
               measurements in inches, body fat percentage, and timeline in weeks"
    - NOT: "Before and after pictures"

  BUILD case_study_structure:
    - Specify COUNT + FORMAT
    - Example: "3-5 detailed case studies: starting measurements, protocol followed
               (all 3 steps), weekly progress, final measurements, total timeframe"
    - NOT: "Some case studies showing it works"

  BUILD testimonial_requirements:
    - Specify what makes a GOOD testimonial for this candidate
    - Example: "Video testimonials with specific results: pounds lost (number),
               inches from waist (number), timeframe to first results, mention
               of previous failed approaches"
    - NOT: "Positive testimonials from happy customers"

  BUILD volume_target:
    - Specific number (not "lots of" or "many")
    - Example: "Minimum 50+ transformation stories across age groups 35-65"
    - NOT: "Many success stories"
```

### Step 4: Build Demonstration Plan

```
FOR each candidate:

  BUILD mechanism_visual:
    - Describe WHAT to show visually
    - Match to mechanism's steps_or_components
    - Example: "Animated diagram showing: 1) impaired bile duct, 2) bitter compound
               reaching liver, 3) bile production reactivating, 4) fat being dissolved"
    - NOT: "Visual showing how it works"

  BUILD protocol_walkthrough:
    - Specific format for demonstrating each step
    - Example: "Step-by-step video: Show the 3 compounds, explain timing (morning/evening),
               demonstrate the daily protocol, show what to expect at day 3, week 1, week 4"
    - NOT: "Demonstration of the protocol"

  BUILD educational_content:
    - Specify what the ROOT CAUSE education section teaches
    - Example: "Explanation of why low-fat diets impair bile production: the fat→bile
               feedback loop, what happens when bile drops below threshold, and why
               calorie restriction can't fix a bile production problem"
    - NOT: "Educational content about the problem"
```

### Step 5: Build Authority & Social Elements

```
FOR each candidate:

  BUILD authority_elements:
    expert_credentials: Specific credential type + experience level
    media_validation: Specific publication types or appearance formats
    professional_recognition: Named association types or award categories

  BUILD social_proof_elements:
    user_count: Specific number target (e.g., "10,000+ protocol users")
    success_rate: Specific percentage (e.g., "87% report measurable reduction")
    testimonial_volume: Specific count (e.g., "500+ verified video testimonials")
    community_validation: Specific format (e.g., "Private group with 5,000+ members")
```

### Step 6: Build Proof Priority Order

```
FOR each candidate:

  RANK proof elements in deployment order:
    1. First priority: What establishes ROOT CAUSE credibility
    2. Second priority: What builds trust in MECHANISM
    3. Third priority: What SHOWS it works (results)
    4. Fourth priority: What AIDS understanding (demonstration)
    5. Fifth priority: What REMOVES risk (social proof volume)

  FORMAT as numbered list with rationale:
    "1. Scientific validation (bile flow research) — establishes Root Cause credibility"
    "2. Expert authority (doctor credentials) — builds trust in mechanism"
    "3. Transformation results (before/after) — shows it works"
    "4. Demonstration (how it works visuals) — aids understanding"
    "5. Social proof (testimonial volume) — removes risk"
```

## Constraints

- NEVER use "studies show" or "research confirms" without specifying WHICH studies
- NEVER specify "a doctor" without defining credential type and domain
- NEVER set volume targets as "many" or "lots of" — use exact numbers
- NEVER leave demonstration plans as "show how it works" — specify what to show
- NEVER produce proof plans that a production team couldn't execute
- ALWAYS match primary proof type to Big Idea type AND Schwartz stage
- ALWAYS cite 3+ swipe sources proving the proof type works
- ALWAYS produce exactly 5 complete proof architectures
- ALL specifications must be actionable (team can execute without follow-up questions)

---

## Quality Criteria

1. All 5 candidates have complete proof architectures (no empty sections)
2. Primary proof types match both Big Idea type AND Schwartz stage
3. All scientific elements name specific research fields (not generic "studies")
4. All transformation elements specify exact measurement types and counts
5. All demonstration plans describe what to show visually (not "show how it works")
6. All authority elements specify credential types (not "an expert")
7. All social proof elements contain specific number targets
8. Proof priority order provides logical rationale for each position
9. All plans are actionable — production team needs zero follow-up questions
