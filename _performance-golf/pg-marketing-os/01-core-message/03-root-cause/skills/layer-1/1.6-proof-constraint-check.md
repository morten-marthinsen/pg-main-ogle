# 1.6 Proof Constraint Check

## Purpose
Validate that the derived root cause can be PROVEN to the audience. A root cause that's true but unprovable creates skepticism. This check ensures we can demonstrate the root cause credibly—either through evidence, logic, demonstration, or authority.

---

## The Proof Principle

**Root Cause must be BELIEVABLE and DEMONSTRABLE**

```
INVALID:
Root Cause: "Invisible toxins in your body cause all disease"
Problem: Abstract, unprovable, sounds like conspiracy

VALID:
Root Cause: "Lectins from certain foods damage your gut lining"
Provable: Named substance, specific mechanism, can show research/results
```

---

## Input Requirements

From 1.5 Mechanism Constraint Check:
- Validated root cause + mechanism pair

From project context:
- Available proof assets (research, testimonials, demonstrations, credentials)

---

## Execution Protocol

### Step 1: Identify Proof Requirements

What TYPE of proof does this root cause need?

| Root Cause Type | Proof Required | Example |
|-----------------|---------------|---------|
| **Physiological** | Scientific research, studies, expert citation | "Lectins damage gut lining" → needs research citation |
| **Psychological** | Relatable demonstration, universal experience | "You decelerate from 70 yards" → "Everyone does this—watch" |
| **Mechanical/Technical** | Visual demonstration, physics explanation | "Clubface must square to path" → show cause/effect |
| **Hidden/Systemic** | Pattern evidence, expose-style investigation | "Big Golf ignores slicers" → competitor analysis |
| **False Belief** | Contradiction demonstration, before/after | "You believe X, but watch what happens when..." |

---

### Step 2: Inventory Available Proof Assets

List what proof you have access to:

```
PROOF ASSET INVENTORY:
├── Research/Studies
│   └── [List available citations, studies, data]
├── Expert/Authority
│   └── [Credentials, endorsements, who backs this]
├── Demonstrations
```

### Step 3: Match Proof to Root Cause

For the validated root cause, identify the best proof strategy:

```
PROOF STRATEGY:
Root Cause: "[Statement]"

PRIMARY PROOF: [Type]
├── Asset: "[What we'll use]"
└── Why it works: "[How it proves the root cause]"

SECONDARY PROOF: [Type]
├── Asset: "[Backup proof]"
└── Role: "[Supporting, not primary]"

PROOF GAPS:
├── What we can't prove directly: "[Gap]"
└── How we'll address it: "[Strategy]"
```

---

### Step 4: Test Proof Strength

Run the proof through these tests:

```
PROOF STRENGTH TEST:
Root Cause: "[Statement]"
Primary Proof: "[Asset]"

1. CREDIBILITY TEST
   └── Would a skeptic find this proof credible? YES/NO
   └── Why/why not: "[Explanation]"

2. RELEVANCE TEST
   └── Does the proof DIRECTLY support the root cause? YES/NO
   └── Connection: "[How proof relates to root cause]"

3. ACCESSIBILITY TEST
   └── Can audience understand the proof easily? YES/NO
   └── Complexity level: "[Simple/Moderate/Complex]"

4. SUFFICIENCY TEST
   └── Is proof strong enough to overcome objections? YES/NO
   └── Likely objections: "[What skeptic might say]"

5. DEMONSTRATION TEST
   └── Can we SHOW rather than just TELL? YES/NO
   └── Demo method: "[How to demonstrate]"

OVERALL PROOF STRENGTH: Strong | Moderate | Weak | Insufficient
```

---

### Step 5: Handle Proof Gaps

If proof is weak in areas, use these strategies:

| Gap Type | Strategy | Example |
|----------|----------|---------|
| **No research** | Use credentials + logic | "David Leadbetter discovered this coaching 7 #1s" |
| **Abstract root cause** | Make it concrete through analogy | "Like fumbling the snap in football" |
| **Can't demonstrate directly** | Use before/after results | "Watch what happens after 10 shots" |
| **Skeptic-prone** | Stack multiple proof types | Research + credentials + demonstration |
| **Novel claim** | Anchor to known authority | "The same physics that keeps F1 cars on track" |

---

### Step 6: Document Proof Plan

```
PROOF PLAN:
Root Cause: "[Statement]"

PROOF STACK (in order of presentation):
1. [Proof Type]: "[Specific asset]"
   └── Purpose: "[What this establishes]"
2. [Proof Type]: "[Specific asset]"
   └── Purpose: "[What this adds]"
3. [Proof Type]: "[Specific asset]"
   └── Purpose: "[Final reinforcement]"

DEMONSTRATION APPROACH:
└── "[How we'll SHOW the root cause]"

OBJECTION HANDLING:
├── Objection: "[Likely skeptic response]"
│   └── Counter: "[How proof addresses it]"
└── Objection: "[Another skeptic response]"
    └── Counter: "[How proof addresses it]"

PROOF GAP MITIGATION:
└── "[How we're handling any remaining gaps]"
```

---

## Quality Gates

Before passing to derivation synthesis:

- [ ] Proof strategy identified for root cause
- [ ] At least 2 proof types available (not just one)
- [ ] Proof strength is at least "moderate"
- [ ] Major objections have proof-based counters
- [ ] Proof gaps have mitigation strategies

---

## Critical Principle

**Unprovable root cause = Copy that fails**

If you can't prove the root cause:
1. Adjust the root cause to something provable
2. Find different proof assets
3. Reduce the scope of the claim

Never proceed with a root cause you can't back up. The copy will collapse at the proof section.
