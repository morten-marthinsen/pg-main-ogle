## Purpose

Take the full inventory of delivery vehicles and architecturally TRIM (remove low-value, high-cost items) and STACK (arrange remaining items into a compelling value presentation). This is Hormozi Grand Slam Offer Construction Step 5 — the moment where raw materials become a structured offer.

The goal is not to include everything. The goal is to create a stack where the prospect thinks: **"I get ALL this for ONLY that?"**

---

## The Principle

**"A great offer isn't the one with the most stuff. It's the one where every component makes the prospect feel like they're stealing."**

---

## Trim & Stack Matrix

```
                    HIGH PERCEIVED VALUE
                           │
              ┌────────────┼────────────┐
              │  INCLUDE    │   INCLUDE  │
              │  (if cost   │   (always) │
              │  manageable)│            │
              │    ★★★      │    ★★★★    │
    HIGH ─────┼────────────┼────────────┤───── LOW
    COST      │  CUT       │   INCLUDE  │     COST
              │  (always)  │   (always) │
              │    ★        │    ★★★★★   │
              │            │            │
              └────────────┼────────────┘
                           │
                    LOW PERCEIVED VALUE
```

**The magic quadrant:** High perceived value + Low delivery cost = ALWAYS INCLUDE.
These are the components that make offers feel irresistible.

## Trim Process

### Step 1: Score Every Vehicle on the Trim Matrix

```
FROM delivery_vehicles.json:

FOR each vehicle:

  SCORE PERCEIVED VALUE (1-10):
    Factors:
    - Does this solve a HIGH priority problem? (+3 if yes)
    - Does this directly drive the dream outcome? (+2 if yes)
    - Is this tangible and easy to understand? (+2 if yes)
    - Does this LOOK valuable? (named well, substantial-sounding) (+1-2)
    - Would the prospect be disappointed if this was removed? (+1-2)

  SCORE DELIVERY COST (1-10):
    Factors:
    - Does this require ongoing labor? (+3 if yes)
    - Does this require custom creation per customer? (+3 if yes)
    - Does this require physical materials? (+2 if yes)
    - Does this require live time/attendance? (+1-2)
    - Can this be created once and delivered infinitely? (-3 if yes)
    - Is this a digital asset? (-2 if yes)

  CALCULATE: value_to_cost_ratio = perceived_value / delivery_cost

  CLASSIFY:
    ★★★★★  Ratio ≥ 4.0 — MUST INCLUDE (high value, low cost)
    ★★★★   Ratio 2.5-3.9 — INCLUDE (strong contributor)
    ★★★    Ratio 1.5-2.4 — CONSIDER (include if stack needs it)
    ★★     Ratio 0.8-1.4 — TRIM CANDIDATE (remove unless strategic)
    ★      Ratio < 0.8 — CUT (costs more than it's worth in perception)
```

### Step 2: Apply Trim Decisions

```
FOR each vehicle scored ★ or ★★:

  BEFORE cutting, CHECK strategic exceptions:

  EXCEPTION 1: Does this vehicle address a CRITICAL objection?
    If it's the only thing addressing a top-3 objection → KEEP

  EXCEPTION 2: Does this vehicle provide essential social proof?
    If removing it leaves a proof gap → KEEP

  EXCEPTION 3: Does this vehicle support the guarantee?
    If the guarantee depends on this component → KEEP

  IF no exception applies → CUT
  IF exception applies → KEEP but flag as "strategic inclusion"

LOG all trim decisions with reasoning
```

### Step 3: Validate Post-Trim Coverage

```
AFTER trimming:

CHECK 1: All 4 value equation dimensions still represented
  Every remaining vehicle should collectively address:
    ✓ Dream Outcome (something that delivers the result)
    ✓ Likelihood (something that proves it works)
    ✓ Time Delay (something that accelerates results)
    ✓ Effort (something that reduces work)

  IF any dimension uncovered → re-evaluate trim decisions

CHECK 2: Core transformation still deliverable
  Can the prospect still achieve the dream outcome with what remains?
  IF no → re-include cut vehicles until achievable

CHECK 3: Sufficient variety
  Are there at least 3 different delivery method types?
  (Not all PDFs, not all videos, not all calls)
  IF not → consider vehicle redesign for variety

CHECK 4: Minimum stack size
  Remaining core vehicles ≥ 3
  Remaining bonus candidates ≥ 2
  IF under minimums → re-evaluate
```

## Stack Process

### Step 4: Determine Stack Architecture

```
ARRANGE remaining vehicles into a VALUE STACK:

STACK ARCHITECTURE DECISIONS:

DECISION 1: What is the CORE PRODUCT?
  The main thing they're buying — the primary vehicle(s)
  Usually 1-3 components that deliver the core transformation
  This is what the offer is "about"

DECISION 2: What are the BONUSES?
  Additional components that enhance the core
  Each bonus should address a DIFFERENT concern/desire
  Bonuses are "added value" — the "and you ALSO get" items

DECISION 3: What is the PRESENTATION ORDER?
  Stack components from most foundational to most aspirational:
    Level 1: The core system/method (foundation)
    Level 2: Implementation support (makes core easier)
    Level 3: Acceleration tools (makes results faster)
    Level 4: Advanced/bonus resources (extra value)

  EACH level builds on the previous, creating a sense of ACCUMULATION
```

### Step 5: Design the Value Stack Presentation

```
CONSTRUCT the stack as it will be presented:

VALUE STACK FORMAT:

CORE PRODUCT: "[Product Name]"
  Component 1: "[Vehicle Name]" — [one-line benefit] — Value: $[amount]
  Component 2: "[Vehicle Name]" — [one-line benefit] — Value: $[amount]
  Component 3: "[Vehicle Name]" — [one-line benefit] — Value: $[amount]

  Core Product Total Value: $[sum]

BONUS 1: "[Bonus Name]"
  — [one-line description + benefit]
  — Value: $[amount]

BONUS 2: "[Bonus Name]"
  — [one-line description + benefit]
  — Value: $[amount]

BONUS 3: "[Bonus Name]"
  — [one-line description + benefit]
  — Value: $[amount]

  Total Bonus Value: $[sum]

────────────────────────────────
TOTAL STATED VALUE: $[grand total]
────────────────────────────────
```

### Step 6: Validate Value-to-Price Ratio

```
REFERENCE vault_offer_patterns → value_architecture_patterns:

  vault_average_value_to_price_ratio: [from vault data]

CALCULATE our ratio:
  total_stated_value / anticipated_actual_price

BENCHMARKS:
  ≥ 10x — Exceptional (prospect feels like robbery in their favor)
  5-9x  — Strong (clear value disparity)
  3-4x  — Acceptable (reasonable deal)
  < 3x  — Weak (may not feel like enough value)

IF ratio < 5x:
  OPTIONS:
  1. Add another high-value/low-cost bonus
  2. Increase stated values on existing components (within credibility)
  3. Re-include a trimmed vehicle that was borderline

IF ratio > 20x:
  WARNING: May stretch credibility
  Consider reducing some stated values to stay believable
```

### Step 7: Stack Coherence Check

```
REVIEW the assembled stack for COHERENCE:

CHECK 1: Does the stack tell a story?
  The components should progress LOGICALLY:
  "Here's the system → Here's how to implement it → Here's how to go faster →
   Here are extra tools to make it even easier"

  IF disjointed: REORDER components

CHECK 2: Does each bonus ADD something the core doesn't cover?
  No bonus should duplicate core content
  Each bonus should address a DISTINCT concern or desire

  IF overlap: CONSOLIDATE or RENAME to differentiate

CHECK 3: Are the stated values believable?
  Would a reasonable prospect accept these values?
  No single PDF should be valued at $997
  No digital course should be valued at $10,000 unless exceptional

  IF values strain credibility: ADJUST downward

CHECK 4: Is the total stated value impressive WITHOUT being absurd?
  Aim for impressive but defensible
  The prospect should think "that's a lot of value" not "that's obviously inflated"
```

## Quality Criteria

1. Every trim decision documented with reasoning
2. Post-trim coverage validates all 4 value equation dimensions
3. Minimum 3 core product components + minimum 2 bonuses
4. Value-to-price ratio ≥ 5x
5. Stack components are logically ordered (foundation → implementation → acceleration)
6. No bonus duplicates core content
7. All stated values are defensible (could charge separately at that price)
8. Stack coherence checks all pass

---

## Anti-Patterns

- Do NOT include everything — trimming is the skill, not accumulation
- Do NOT value a PDF at $997 because "it would save them $997 in mistakes" — that's the BENEFIT, not the price
- Do NOT create bonuses that are just leftovers from trimming — bonuses should feel intentionally created
- Do NOT order the stack randomly — deliberate progression creates perceived completeness
- Do NOT ignore the vault comparison — if every successful offer in the niche has 5+ bonuses and you have 2, investigate why
