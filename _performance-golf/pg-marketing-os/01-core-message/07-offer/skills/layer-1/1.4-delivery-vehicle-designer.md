## Purpose

Design HOW each solution is delivered to the prospect. The delivery vehicle is the container that transforms an abstract solution into a tangible thing the prospect can see, touch, access, and use. This is Hormozi Grand Slam Offer Construction Step 4.

A brilliant solution delivered poorly is worthless. A good solution delivered brilliantly feels premium. The delivery vehicle is what makes an offer feel like a PRODUCT rather than a vague promise.

---

## The Principle

**"The prospect doesn't buy the solution. They buy the delivery vehicle that contains the solution. 'Accountability' is invisible. 'Weekly 1-on-1 coaching calls with a certified mentor' is a product."**

---

## Four Design Dimensions (Hormozi)

Every delivery vehicle is designed across 4 dimensions:

```
DIMENSION 1: ATTENTION LEVEL
  How much personalized attention does the prospect receive?
  │
  ├── 1-on-1        → Maximum attention, maximum perceived value
  ├── Small Group    → High attention, peer learning benefit
  ├── Large Group    → Moderate attention, community energy
  └── Self-Paced     → Zero live attention, maximum convenience

DIMENSION 2: CLIENT EFFORT
  How much work does the prospect do?
  │
  ├── Done-For-You (DFY)    → They do nothing; we handle it
  ├── Done-With-You (DWY)   → We collaborate; they participate
  ├── Do-It-Yourself (DIY)  → They execute with our tools/instructions
  └── Figure-It-Out (FIO)   → They get raw information (lowest value)

DIMENSION 3: DELIVERY METHOD
  How does the prospect access/receive this?
  │
  ├── In-Person     → Physical presence, highest engagement
  ├── Live Virtual  → Real-time digital, high engagement
  ├── Recorded      → On-demand, maximum convenience
  ├── Written       → Text-based (guides, checklists, reports)
  └── Software      → Interactive tool, app, or platform

DIMENSION 4: COMMUNICATION FREQUENCY
  How often does the prospect interact with this vehicle?
  │
  ├── Real-Time     → Always available (chat, app notifications)
  ├── Daily         → Daily touchpoints (emails, check-ins)
  ├── Weekly        → Weekly sessions or updates
  ├── Monthly       → Monthly reviews or materials
  └── One-Time      → Single delivery, no ongoing interaction
```

## Design Process

### Step 1: Pull Solutions Requiring Vehicles

```
FROM problem_solution_map.json:
  EXTRACT all HIGH and MEDIUM priority solutions
  EXCLUDE LOW priority (these were trimmed by priority scoring)
  EXCLUDE solutions of type "guarantee" (handled separately in Layer 2)

FOR each solution:
  NOTE: solution_type (education/tool/system/done_for_you/social_proof)
  NOTE: solution_delivery (preliminary delivery idea from 1.2)
  NOTE: mechanism_connection (how this relates to core mechanism)
```

### Step 2: Design Vehicle for Each Solution

```
FOR each solution:

  DESIGN across all 4 dimensions:

  ATTENTION LEVEL:
    EVALUATE: What level of attention does this solution REQUIRE?
    EVALUATE: What level creates the highest PERCEIVED value?
    EVALUATE: What level is DELIVERABLE at scale?

    SELECT: attention_level with justification

  CLIENT EFFORT:
    REFERENCE: value_equation_optimization.json → lever_4_effort
    APPLY: hierarchy_upgrades (move UP where possible)
    EVALUATE: What's the minimum effort for maximum result?

    SELECT: effort_level with justification

  DELIVERY METHOD:
    EVALUATE: What method fits the content type?
      - Teaching → recorded video or live virtual
      - Tool → software, PDF template, or calculator
      - System → platform with workflow
      - Social proof → written case studies or video testimonials

    EVALUATE: What method matches prospect behavior?
      FROM upstream_data → demographic_profile and market_psychology
      Do they prefer video? Written? Audio? Interactive?

    SELECT: delivery_method with justification

  COMMUNICATION FREQUENCY:
    EVALUATE: How often does the prospect need to engage for results?
    EVALUATE: What frequency is sustainable to deliver?
    EVALUATE: What frequency creates ongoing perceived value?

    SELECT: frequency with justification
```

### Step 3: Name Each Delivery Vehicle

```
FOR each designed vehicle:

  TRANSFORM from abstract solution → tangible product name

  NAMING PRINCIPLES:
    1. The name should make the vehicle feel like a REAL PRODUCT
    2. The name should hint at the BENEFIT
    3. The name should feel PROPRIETARY (not generic)

  NAMING FORMULAS:
    - "The [Adjective] [Benefit] [Container]"
      Example: "The Rapid Results Accelerator"

    - "[Outcome] [Tool Type]"
      Example: "Fat Loss Command Center"

    - "[Number]-[Unit] [Action] [Container]"
      Example: "7-Day Quick Start Blueprint"

    - "[Proprietary Name] [Container]"
      Example: "MindShift Protocol Workbook"

  AVOID:
    - Generic names ("Module 3", "Bonus Video", "Extra PDF")
    - Names that describe format instead of benefit ("60-Minute Video")
    - Names that sound like homework ("Assessment Worksheet")

  GENERATE 2-3 name options per vehicle
  SELECT strongest name based on:
    - Benefit clarity
    - Perceived value
    - Memorability
    - Proprietary feel
```

### Step 4: Assign Perceived Value

```
FOR each delivery vehicle:

  ESTIMATE perceived dollar value to the prospect

  VALUE ANCHORING METHODOLOGY:
    1. What would this cost if purchased separately?
       - 1-on-1 coaching call = $200-500/session
       - Course module = $97-297
       - Template/checklist = $27-97
       - Software tool = $47-297/month
       - Community access = $47-97/month
       - Done-for-you service = $500-5000

    2. Reference vault_offer_patterns for comparable valuations
       FROM vault_offer_patterns → value_architecture_patterns
       What do similar offers assign as value for comparable vehicles?

    3. ASSIGN value that is:
       - Defensible (could actually charge this separately)
       - Impressive (contributes to the value stack total)
       - Not absurd (doesn't destroy credibility)

  LOG: stated_value with justification
```

### Step 5: Map Vehicle → Offer Component

```
FOR each delivery vehicle:

  CLASSIFY as:

  CORE PRODUCT COMPONENT:
    Part of the main product — essential for the primary transformation
    These form the value stack presented in the offer

  POTENTIAL BONUS:
    Valuable but not essential for the primary transformation
    Addresses a secondary concern or accelerates results
    Could be positioned as "bonus" in the offer stack

  POTENTIAL GUARANTEE SUPPORT:
    Component that supports or delivers the guarantee
    Example: "If you don't see results in 30 days, you get [this bonus + refund]"

  TRIM CANDIDATE:
    Low perceived value relative to delivery cost
    Could be cut without weakening the offer

  TAG each vehicle with its classification
  This classification feeds into 1.5 (Trim & Stack)
```

### Step 6: Validate Against Vault Patterns

```
FROM vault_offer_patterns:

  COMPARE designed vehicles to vault exemplars:

  WHAT'S COMMON:
    - Do our vehicle types match what successful offers use?
    - Are we using attention/effort levels consistent with the market?

  WHAT'S DIFFERENT:
    - Do we have any vehicle types NOT seen in the vault?
    - Are we innovating on delivery (good) or just being weird (bad)?

  WHAT'S MISSING:
    - Do vault exemplars include vehicle types we haven't designed?
    - Are there delivery innovations we should adopt?

  FLAG insights for consideration but don't slavishly copy
```

## Quality Criteria

1. Every HIGH priority solution has a designed delivery vehicle
2. All 4 design dimensions specified for every vehicle
3. Every vehicle has a proprietary name (no generics like "Module 1")
4. Every vehicle has a stated value with justification
5. Classification is complete — every vehicle tagged as core/bonus/guarantee/trim
6. At least one vehicle moves UP the effort hierarchy (DFY or DWY element)
7. Vehicle designs reference prospect behavior data from upstream research

---

## Anti-Patterns

- Do NOT design vehicles that the product can't actually deliver
- Do NOT name vehicles with marketer jargon ("Proprietary Synergy Matrix")
- Do NOT assign values that insult the prospect's intelligence ($10,000 for a PDF)
- Do NOT design all vehicles at the same attention/effort level — diversity creates perceived comprehensiveness
- Do NOT skip the vault comparison — empirical patterns reveal blind spots
