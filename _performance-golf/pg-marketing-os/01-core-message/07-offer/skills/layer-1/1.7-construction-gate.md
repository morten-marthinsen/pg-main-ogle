# Skill 1.7: Construction Gate

**Version:** 1.0
**Layer:** 1 (Offer Construction)
**Type:** Quality Gate
**Dependencies:** All Layer 1 outputs (1.1-1.6)
**Output:** gate_1_report.json
**Persona:** Dr. Richard Stern (Skeptic/Devil's Advocate)

---

## Purpose

Quality checkpoint that validates the core offer construction before enhancement begins. This gate ensures the structural foundation is sound — because enhancing a broken offer with bonuses, guarantees, and urgency just makes it a well-decorated broken offer.

Dr. Stern's job is simple: try to break it. If it survives, it proceeds.

---

## The Principle

**"Enhancement cannot fix construction. A flawed foundation with beautiful trim is still a flawed building. Gate 1 exists to prevent building on sand."**

---

## Gate Evaluation

### Checkpoint 1: Dream Outcome Quality (from 1.1)

```
EVALUATE dream_outcome.json:

✓ Primary statement is specific and vivid (not vague)
  TEST: Can you SEE the outcome as a film scene?
  FAIL TRIGGER: "Feel better" / "Achieve success" / "Improve your life"

✓ All 5 layers populated (functional → status)
  TEST: Are all layers non-empty and distinct?
  FAIL TRIGGER: Missing layers or layers that repeat the same thing

✓ Composite validation score ≥ 7.0/10
  TEST: Check score from 1.1 output
  FAIL TRIGGER: Score < 7.0

✓ Uses prospect language (not marketer language)
  TEST: Do phrases match customer_language from research?
  FAIL TRIGGER: Corporate/marketing speak dominating

SCORE: [1-10] with PASS/FAIL determination
```

### Checkpoint 2: Problem-Solution Coverage (from 1.2)

```
EVALUATE problem_solution_map.json:

✓ Minimum 15 unique problems identified
  TEST: Count total problems
  FAIL TRIGGER: < 15 problems (insufficient problem exploration)

✓ All 3 phases covered (before, during, after)
  TEST: Count per phase — minimum 5/4/3
  FAIL TRIGGER: Any phase below minimum

✓ All 4 value equation dimensions represented
  TEST: Count per dimension — minimum 2 each
  FAIL TRIGGER: Any dimension with < 2 problems

✓ All documented objections addressed
  TEST: Cross-reference objections_inventory → mapped solutions
  FAIL TRIGGER: Unmapped objections from research

✓ Minimum 10 HIGH priority problem-solution pairs
  TEST: Count high-priority items
  FAIL TRIGGER: < 10 high-priority pairs

SCORE: [1-10] with PASS/FAIL determination
```

### Checkpoint 3: Value Equation Optimization (from 1.3)

```
EVALUATE value_equation_optimization.json:

✓ All 4 levers have optimization strategies applied
  TEST: Check each lever has ≥ 2 strategies with specific changes
  FAIL TRIGGER: Any lever with zero or vague optimizations

✓ Quick wins identified (results within 7 days)
  TEST: Minimum 2 quick wins documented
  FAIL TRIGGER: No quick wins or vague timeframes

✓ Effort hierarchy has at least one upgrade
  TEST: At least one component moved up DIY → DWY or DWY → DFY
  FAIL TRIGGER: Everything stays at same effort level

✓ Cross-lever tensions identified and resolved
  TEST: Analysis exists (even if no tensions found)
  FAIL TRIGGER: Analysis section empty/missing

SCORE: [1-10] with PASS/FAIL determination
```

### Checkpoint 4: Delivery Vehicle Design (from 1.4)

```
EVALUATE delivery_vehicles.json:

✓ Every HIGH priority solution has a delivery vehicle
  TEST: Cross-reference problem_solution_map HIGH priority → vehicles
  FAIL TRIGGER: HIGH priority solution without vehicle

✓ All 4 design dimensions specified for every vehicle
  TEST: No empty fields across attention/effort/method/frequency
  FAIL TRIGGER: Missing dimensions

✓ Every vehicle has a proprietary name
  TEST: No generics ("Module 1", "Bonus PDF", "Video Training")
  FAIL TRIGGER: Any generic name

✓ Every vehicle has a stated value with justification
  TEST: Values present and justified (not just assigned)
  FAIL TRIGGER: Unjustified values or missing values

✓ Mix of delivery methods (not all same type)
  TEST: Minimum 3 different delivery methods across all vehicles
  FAIL TRIGGER: Homogeneous delivery

SCORE: [1-10] with PASS/FAIL determination
```

### Checkpoint 5: Value Stack Architecture (from 1.5)

```
EVALUATE value_stack.json:

✓ Trim decisions documented with reasoning
  TEST: Every trimmed item has explanation
  FAIL TRIGGER: Arbitrary trimming without justification

✓ Minimum 3 core components + minimum 2 bonuses
  TEST: Count core and bonus items
  FAIL TRIGGER: Below minimums

✓ Value-to-price ratio ≥ 5x
  TEST: total_stated_value / anticipated_price
  FAIL TRIGGER: Ratio < 5x

✓ Stack tells a logical story (progressive, not random)
  TEST: Review stack order for logical progression
  FAIL TRIGGER: Random ordering with no narrative flow

✓ All 4 value equation dimensions covered post-trim
  TEST: Remaining vehicles collectively address all dimensions
  FAIL TRIGGER: Any dimension uncovered after trimming

SCORE: [1-10] with PASS/FAIL determination
```

### Checkpoint 6: Core Offer Definition (from 1.6)

```
EVALUATE core_offer_definition.json:

✓ All 7 identity questions answered substantively
  TEST: No placeholder or generic answers
  FAIL TRIGGER: Template language or empty answers

✓ Offer positioning has upstream data justification
  TEST: Selection traced to market_psychology, mechanism_type, big_idea_type
  FAIL TRIGGER: Arbitrary selection without evidence

✓ Offer thesis ≤ 4 sentences with all required elements
  TEST: Count sentences; check for product name, avatar, mechanism, outcome, proof, guarantee
  FAIL TRIGGER: Missing elements or over 4 sentences

✓ All 5 coherence checks pass
  TEST: Zero MISALIGNED ratings
  FAIL TRIGGER: Any MISALIGNED coherence check

✓ Offer is differentiated from competitors
  TEST: Can distinguish this offer from top 3 competitor offers
  FAIL TRIGGER: Sounds identical to existing market offers

SCORE: [1-10] with PASS/FAIL determination
```

## Gate Decision Logic

```
COMPOSITE SCORING:

  Weight each checkpoint equally:
    dream_outcome_score × 1.0
    problem_coverage_score × 1.0
    value_equation_score × 1.0
    delivery_vehicle_score × 1.0
    value_stack_score × 1.0
    core_definition_score × 1.0

  gate_1_composite = average of all 6 scores

GATE DECISION:

  PASS: gate_1_composite ≥ 7.0 AND no individual checkpoint < 6.0
    → Proceed to Layer 2 (Enhancement)

  CONDITIONAL PASS: gate_1_composite ≥ 6.0 AND no checkpoint < 5.0
    → Proceed to Layer 2 with documented warnings
    → Layer 2 skills must account for weaknesses

  FAIL: gate_1_composite < 6.0 OR any checkpoint < 5.0
    → Return to failing Layer 1 skill(s) for remediation
    → Specify exactly what needs to change
    → Re-run gate after fixes
```

## Remediation Guidance

```
IF GATE FAILS:

FOR each failing checkpoint:

  GENERATE specific remediation instructions:
    failing_skill: which skill produced the inadequate output
    failing_criteria: which specific criteria were not met
    required_action: what specifically needs to change
    examples: what "good" would look like for this criterion

  EXAMPLE REMEDIATION:
    failing_skill: "1.1 Dream Outcome Crystallizer"
    failing_criteria: "Composite score 5.8 — below 7.0 threshold"
    required_action: "Specificity and mechanism connection are weak.
                      Add concrete numbers to the functional layer.
                      Connect the dream outcome explicitly to what
                      the mechanism enables."
    examples: "Instead of 'improve your golf game,' write
              'shave 5 strokes off your handicap in 30 days using
              the Neural Groove Protocol'"

PRIORITIZE remediations by:
  1. Checkpoints that scored lowest
  2. Checkpoints with FAIL triggers (not just low scores)
  3. Issues that cascade into multiple downstream problems
```

## Quality Criteria

1. All 6 checkpoints evaluated with specific evidence (not generic assessments)
2. Every FAIL trigger identified with the specific output that triggered it
3. Composite score calculated correctly (average of 6 checkpoint scores)
4. Remediation instructions are actionable (not "make it better" but "add X to Y")
5. Gate decision follows the logic precisely (no subjective overrides)
6. Strengths documented alongside weaknesses (balanced assessment)
