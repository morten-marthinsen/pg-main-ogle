# Skill 4.1: Offer Brief Compiler

**Version:** 1.0
**Layer:** 4 (Package Assembly)
**Type:** Documentation Synthesis
**Dependencies:** All Layer 1-3 outputs
**Output:** offer_brief.json
**Persona:** Alex Rivera (Strategic Integration) + Dr. James Liu (Research Director)

---

## Purpose

Compile the complete offer brief — the single source of truth for everything the offer IS, CONTAINS, COSTS, GUARANTEES, and PROMISES. This document is not a presentation script. It's the reference document that downstream skills (VSL, email, landing page) use to write accurate, aligned copy.

Think of it as the "spec sheet" for the offer. Every downstream copywriter, designer, or funnel builder should be able to read this and know exactly what to produce.

---

## The Principle

**"The offer brief is the contract between architecture and copy. Architecture decides what the offer IS. Copy decides how to PRESENT it. The brief is where they meet."**

---

## Compilation Process

### Step 1: Identity Section

```
FROM core_offer_definition.json:

COMPILE:

identity:
  product_name: string         # Final product name
  offer_name: string           # Full offer name (from 2.5)
  tagline: string              # One-line hook
  positioning: string          # System/secret/method/tool/experience
  offer_thesis: string         # The one-paragraph synthesis

  target:
    primary_avatar: string
    avatar_situation: string
    avatar_desire: string
    awareness_level: string    # From upstream research

  transformation:
    primary_dream_outcome: string
    supporting_outcomes: [string]
    timeframe: string
    mechanism_name: string
    mechanism_type: string
    mechanism_one_liner: string
```

### Step 2: Core Offer Section

```
FROM value_stack.json + delivery_vehicles.json:

COMPILE:

core_offer:
  total_core_components: integer
  total_core_value: string

  components:
    - name: string
      description: string       # What it IS
      benefit: string           # What it DOES for them
      delivery_method: string   # How they receive it
      stated_value: string      # Dollar value

  INCLUDE: Only core product components (not bonuses)
```

### Step 3: Enhancement Stack Section

```
FROM bonus_designs.json + guarantee_architecture.json + scarcity_strategy.json + urgency_strategy.json:

COMPILE:

enhancement_stack:
  bonuses:
    total_bonuses: integer
    total_bonus_value: string
    items:
      - name: string
        description: string
        benefit: string
        stated_value: string
        type: string              # Tool/checklist/video/etc.
        objection_addressed: string

  guarantee:
    type: string                  # Unconditional/conditional/stacked/etc.
    duration: string
    conditions: [string] | null
    outcome_promise: string
    refund_mechanism: string
    bonus_retention: boolean
    formal_language: string
    conversational_language: string
    short_language: string

  scarcity:
    type: string
    mechanism: string
    quantity: string | null
    primary_statement: string
    short_statement: string

  urgency:
    type: string
    deadline: string | null
    consequence_offer: string
    consequence_problem: string
    primary_statement: string
    short_statement: string
```

### Step 4: Price Architecture Section

```
FROM price_architecture.json:

COMPILE:

price:
  anchor_value: string           # Total stated value
  actual_price: string           # What they pay
  discount_percentage: string
  dollar_savings: string
  payment_options:
    - option_name: string
      terms: string
      total_collected: string
  recommended_default: string
  price_justification: string
  comparison_anchors: [string]
```

### Step 5: Naming Section

```
FROM offer_names.json:

COMPILE:

naming:
  primary_name: string
  formula_components:
    magnetic_reason: string
    avatar: string
    goal: string
    time_interval: string
    container_word: string
  alternates: [string]
  presentation_formats:
    sales_page: string
    email_subject: string
    ad_headline: string
    spoken: string
```

### Step 6: Proof Summary Section

```
FROM upstream_data → proof_inventory:

COMPILE:

proof_summary:
  strongest_elements:
    - type: string              # Testimonial/study/data/expert/demo
      content: string           # What the proof says
      source: string            # Where it comes from
      strength: integer         # 1-10
  credibility_anchors: [string]
  proof_gaps: [string]          # Known weaknesses in proof
```

### Step 7: Validation Scores Section

```
FROM all Layer 3 outputs:

COMPILE:

validation:
  value_equation:
    dream_outcome: integer
    perceived_likelihood: integer
    perceived_time_delay: integer
    perceived_effort: integer
    composite: float

  differentiation_score: float
  promise_alignment_score: float
  anti_slop: "PASS" | "FAIL"

  gate_1_result: "PASS" | "CONDITIONAL_PASS" | "FAIL"
  gate_2_result: "PASS" | "CONDITIONAL_PASS" | "FAIL"

  known_weaknesses: [string]
  known_strengths: [string]
```

### Step 8: Quick Reference Section

```
COMPILE a rapid-access reference:

quick_reference:
  one_sentence: string           # Offer in one sentence
  who_for: string                # Avatar in 10 words
  what_they_get: string          # Stack in one paragraph
  price: string                  # Price in one line
  guarantee: string              # Guarantee in one line
  deadline: string               # Urgency in one line
  why_different: string          # Differentiator in one line
```

---

## Quality Criteria

1. All 8 sections complete with no empty fields
2. Every data point traceable to a specific upstream output
3. No interpretation or embellishment — this is documentation, not copy
4. Quick reference section is genuinely quick (one sentence per item)
5. Validation scores match Layer 3 outputs exactly (no rounding or adjusting)
6. Enhancement stack includes all components with exact language versions
7. Price section includes all payment options with exact terms
8. Brief is self-contained — a reader needs no other document to understand the full offer
