**Version:** 1.0

---

## Purpose

Assemble the complete offer presentation script following Todd Brown's SIN Offer Presentation template. This takes all the individual scripts, language, and presentation components from Layers 1-2 and assembles them into a single, sequential, ready-to-deploy presentation flow.

This is NOT copywriting. It's assembly. Every component already exists — this skill arranges them in the correct order with proper transitions.

---

## The Principle

**"The presentation script is the blueprint that downstream copy skills (VSL, sales page, email) adapt to their format. Get the sequence wrong here, and every downstream deliverable inherits the error."**

---

## Todd Brown SIN Offer Presentation Template (14 Chunks)

```
The offer presentation follows this exact sequence:

 1. Core Product Introduction
 2. Deliverable Presentations (D-F-W-B-P × each)
 3. Product Summary (restate promise)
 4. Bonus Transition
 5. Bonus Presentations (D-F-W-B-P × each)
 6. Total Value Statement
 7. Value Demonstration ("if all it did was..." × 3)
 8. Price Presentation (anchor → discount → actual → terms → savings)
 9. Risk Reversal (guarantee presentation)
10. CTA 1 (direct call to action)
11. Consequence Amplification (current pain reminder)
12. After-State Contrast (desired future)
13. CTA 2 (urgency-enhanced call to action)
14. CTA 3 (final call to action)
```

## Assembly Process

### Chunk 1: Core Product Introduction

```
SOURCE: core_offer_definition.json → identity

ASSEMBLE:
  "Introducing [Product Name] — [tagline or one-liner]."

  Or, if following a marketing argument:
  "That's why I created [Product Name] — [mechanism one-liner that connects
  to the preceding marketing argument]."

TRANSITION TO CHUNK 2:
  "Here's everything you get when you [grab/enroll in/sign up for]
   [Product Name] today..."
```

### Chunk 2: Core Deliverable Presentations

```
SOURCE: value_demonstration_script.json → core_deliverable_presentations

ASSEMBLE:
  Each core component's D-F-W-B-P in stack order.
  Include transitions between components.

  "The first thing you get is [Component 1]..."
    [Full D-F-W-B-P]
  "The second thing you get is [Component 2]..."
    [Full D-F-W-B-P]
  "The third thing you get is [Component 3]..."
    [Full D-F-W-B-P]
```

### Chunk 3: Product Summary

```
SOURCE: value_demonstration_script.json → product_summary

ASSEMBLE:
  "With everything you're getting — [Component 1], [Component 2],
   [Component 3] — you have everything you need to [PRIMARY PROMISE
   in fresh language]."
```

### Chunk 4: Bonus Transition

```
SOURCE: value_demonstration_script.json → bonus_transition

ASSEMBLE:
  "But that's not all. When you [grab/enroll in] [Product Name] today,
   you also get..."
```

### Chunk 5: Bonus Presentations

```
SOURCE: value_demonstration_script.json → bonus_presentations

ASSEMBLE:
  Each bonus's D-F-W-B-P in presentation order (from 2.1).
  Include transitions between bonuses.

  "You also get [Bonus 1]..."
    [Full D-F-W-B-P]
  "And that's not all. You also get [Bonus 2]..."
    [Full D-F-W-B-P]
  "And to make this truly complete, you also get [Bonus 3]..."
    [Full D-F-W-B-P]
```

### Chunk 6: Total Value Statement

```
SOURCE: value_demonstration_script.json → total_value_statement

ASSEMBLE:
  "With everything that you get today — [list all components + bonuses] —
   [Product Name] has a total value of $[ANCHOR VALUE]."
```

### Chunk 7: Value Demonstration

```
SOURCE: value_demonstration_script.json → if_all_it_did_was

ASSEMBLE:
  Round 1: [full script from 2.7]
  Round 2: [full script from 2.7]
  Round 3: [full script from 2.7]
  Sequence Close: [from 2.7]
```

### Chunk 8: Price Presentation

```
SOURCE: price_architecture.json → price_presentation_script

ASSEMBLE:
  Step 1 (already delivered in Chunk 6): Total value
  Step 2: "But today, when you [grab] [Product Name], it isn't going to
          cost you $[anchor]."
  Step 3: [Intermediate anchor if applicable]
  Step 4: "Instead, today you get everything for just $[ACTUAL PRICE]."
  Step 5: "[Payment terms]"
  Step 6: "That's [X]% off — a savings of over $[savings]."
```

### Chunk 9: Risk Reversal

```
SOURCE: guarantee_architecture.json → guarantee_language → conversational_version

ASSEMBLE:
  The conversational guarantee statement.
  Include the "you don't have to say yes, just maybe" frame if applicable.
  Include bonus retention statement if applicable.
```

### Chunk 10: CTA 1

```
CONSTRUCT:
  "Go ahead, click the button [below], [grab/enroll in/sign up for]
   [Product Name] right now.

   You either [restate primary promise in new words] —
   or it costs you nothing."
```

### Chunk 11: Consequence Amplification

```
SOURCE: upstream_data → market_psychology → pain_points
        dream_outcome.json → anti_dream

CONSTRUCT:
  "There's no need for you to continue [current pain 1]...
   No need to continue [current frustration 2]...
   No need to continue [current struggle 3]...

   You can change all of that today with [Product Name]."

  Use 3-4 specific pain points from research, in prospect language.
```

### Chunk 12: After-State Contrast

```
SOURCE: dream_outcome.json → layer_stack, movie_scene_moment

CONSTRUCT:
  "[Positive outcome 1]. [Specific measurable improvement in timeframe].
   [Desired identity/status state]."

  Paint the contrast: what they're escaping → what they're gaining.
```

### Chunk 13: CTA 2 (Urgency-Enhanced)

```
SOURCE: urgency_strategy.json + scarcity_strategy.json

CONSTRUCT:
  "Go ahead, click the button below, [grab/enroll in] [Product Name]
   right now.

   [Urgency statement — deadline and consequence]
   [Scarcity statement — quantity and consequence]

   If you want it, now is the time to act."
```

### Chunk 14: CTA 3

```
CONSTRUCT:
  "Go ahead, click the button below."

  Simple. Direct. Final.
```

## Post-Assembly Quality Check

```
AFTER assembling all 14 chunks:

CHECK 1: Flow Continuity
  Read the entire script start to finish.
  Does it flow naturally from one chunk to the next?
  Are transitions smooth or jarring?
  FIX any jarring transitions.

CHECK 2: Promise Consistency
  How many times is the primary promise restated?
  Is it restated with varied language each time?
  FLAG if same exact phrasing used more than once.

CHECK 3: Tone Consistency
  Does the script maintain a consistent voice throughout?
  No sudden shifts from conversational to formal.
  No sudden shifts from confident to hedging.

CHECK 4: Length Assessment
  Estimate total word count.
  Flag if under 1,500 words (probably too thin).
  Flag if over 5,000 words (probably too long for initial presentation).
  NOTE: This is the TEMPLATE. VSL/sales page skills will expand.

CHECK 5: Missing Components
  Verify all 14 chunks present and populated.
  Verify no placeholder text remains (brackets, TBD, etc.).
```

## Quality Criteria

1. All 14 chunks present and populated with actual content (no placeholders)
2. Script flows naturally from chunk to chunk
3. Primary promise restated minimum 3 times with varied language
4. Tone is consistent throughout
5. Word count is within range (1,500 - 5,000 for template)
6. Every D-F-W-B-P is complete (no missing elements)
7. All 3 CTAs present with different emotional triggers between them
8. Price presentation follows exact Todd Brown 7-step sequence
