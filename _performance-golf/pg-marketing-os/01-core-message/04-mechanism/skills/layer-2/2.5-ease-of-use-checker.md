# Skill 2.5: Ease of Use Checker

**Version:** 1.0
**Layer:** 2 (Scorecard Optimization)
**Type:** Dimension Evaluation + Optimization
**Dimension:** 5 of 12 — EASE OF USE
**Dependencies:** gate_1_report.json, explanation_structures.json
**Output:** ease_of_use_scores.json

---

## Purpose

Evaluate and optimize each mechanism candidate on EASE OF USE: Does the mechanism make the product/solution seem EASY to use? The mechanism should reduce perceived effort, not add complexity. If the mechanism sounds hard to activate or maintain, conversion dies.

---

## The Question

**"Does this mechanism make the PRODUCT seem easy to use / the SOLUTION seem easy to achieve?"**

---

## Scoring Rubric

```
1-3 (HARD): Mechanism implies significant effort, complexity, or lifestyle change.
     - "You'll need to meditate 2 hours daily"
     - "This 47-step protocol requires..."
     - Sounds like work, discipline, or suffering

4-6 (MODERATE): Some effort implied but manageable.
     - "Just 15 minutes a day"
     - "A few simple adjustments"
     - Sounds doable but not effortless

7-8 (EASY): Mechanism implies minimal effort from the user.
     - "Take one capsule with breakfast"
     - "Just press play and listen"
     - The mechanism does the work, not the user

9-10 (EFFORTLESS): Mechanism works FOR you with zero perceived effort.
     - "It works while you sleep"
     - "Your body does this automatically once triggered"
     - "One activation and it runs in the background"
     - User's role is just to START it — mechanism handles the rest
```

---

## Evaluation Logic

### Step 1: Identify Effort Signals

```
FOR each candidate:

  FROM explanation_structures:

  SCAN for EFFORT-INCREASING language:
    - Time commitments ("daily practice", "weeks of training")
    - Action requirements ("you must", "you need to")
    - Discipline language ("consistency", "commitment", "dedication")
    - Complexity signals ("multiple steps", "careful calibration")
    - Lifestyle changes ("change your diet", "wake up earlier")

  SCAN for EFFORT-REDUCING language:
    - Automatic language ("works for you", "triggers itself")
    - Minimal action ("just one", "simply", "in seconds")
    - Passive language ("your body does", "it activates")
    - Once-and-done ("one-time setup", "permanent change")
    - While-you-X ("while you sleep", "in the background")

  RATIO: effort_reducing / (effort_reducing + effort_increasing)
  HIGH RATIO = high score, LOW RATIO = low score
```

### Step 2: Assess Who Does the Work

```
FOR each candidate:

  DETERMINE: Who is doing the work in this mechanism?

  USER does the work (score 1-4):
    "You practice the technique daily"
    "You follow the 12-step protocol"
    "You change your morning routine"

  USER + MECHANISM share work (score 5-7):
    "You take the capsule, then the mechanism activates"
    "You spend 5 minutes, then it works for hours"
    "You do one thing, the mechanism handles the rest"

  MECHANISM does the work (score 8-10):
    "Once activated, it runs automatically"
    "Your body's natural process takes over"
    "The compound works 24/7 without your attention"
    "It's already happening — you just need to stop blocking it"
```

### Step 3: Optimize If Below Threshold

```
IF current_score < 7:

  TECHNIQUE 1: Shift Work to Mechanism
    REFRAME user actions as "activation" not "effort"
    "You practice daily" → "You trigger it once, then it builds automatically"
    "You follow the protocol" → "The protocol runs itself after setup"

  TECHNIQUE 2: Minimize Perceived Steps
    REDUCE visible complexity
    "12-step process" → "One simple activation"
    "Daily practice" → "3 minutes to start your day"

  TECHNIQUE 3: Add Automatic Language
    INJECT mechanism autonomy:
    "Once [mechanism] is activated, your body's own [process]
     takes over and does the rest"

  TECHNIQUE 4: The "Already Happening" Frame
    IF applicable: Frame mechanism as REMOVING a blocker
    rather than ADDING an action:
    "Your body already knows how to [result] —
     [mechanism] just removes the thing stopping it"

  TECHNIQUE 5: Reduce Time Horizon
    IF results take time: Focus on FIRST noticeable change
    "Full results in 90 days" → "Feel the first shift within 48 hours"

  RE-SCORE after optimization
```

## Output Schema

```yaml
output:
  ease_of_use_scores:
    for_each_candidate:
      - candidate_name: string
        current_score: integer[1-10]
        effort_analysis:
          effort_increasing_signals: [string]
          effort_reducing_signals: [string]
          ratio: float
          who_does_work: enum[user, shared, mechanism]
          perceived_time_commitment: string
          perceived_complexity: enum[simple, moderate, complex]

        optimization_applied: boolean
        optimized_version:
          reframed_explanation: string
          effort_reduction_technique: string
          new_score: integer[1-10]
```

---

## Quality Criteria

1. Every candidate has explicit effort signal identification
2. "Who does the work" is clearly determined
3. Candidates scoring < 7 receive effort-reduction optimization
4. No optimized version contains discipline/commitment language
5. Top candidates frame mechanism as doing the work for the user
