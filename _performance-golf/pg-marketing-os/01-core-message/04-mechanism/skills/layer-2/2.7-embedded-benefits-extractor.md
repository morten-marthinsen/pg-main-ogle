# Skill 2.7: Embedded Benefits Extractor

**Version:** 1.0
**Layer:** 2 (Scorecard Optimization)
**Type:** Dimension Evaluation + Optimization
**Dimension:** 7 of 12 — EMBEDDED BENEFITS
**Dependencies:** gate_1_report.json, naming_candidates.json
**Output:** embedded_benefits_scores.json

---

## Purpose

Evaluate and optimize each mechanism candidate on EMBEDDED BENEFITS: Are the BENEFITS baked into the mechanism name and concept itself? The best mechanism names don't just label what it IS — they promise what it DOES. Hearing the name alone should create desire.

---

## The Question

**"Does the mechanism NAME itself communicate the BENEFIT?"**

---

## Scoring Rubric

```
1-3 (NO BENEFITS): Name is descriptive only, no benefit implied.
     - "The Alpha Protocol" — what benefit? No idea.
     - "Compound 47" — tells you nothing about the result.
     - Name requires explanation to create desire.

4-6 (PARTIAL BENEFITS): Name hints at benefit but doesn't deliver it.
     - "The Energy Method" — suggests energy but vaguely.
     - "The Clarity System" — points to clarity but doesn't promise it.
     - Name needs the explanation to become compelling.

7-8 (CLEAR BENEFITS): Name directly implies the primary benefit.
     - "The Cortisol Switch" — implies you can switch off stress.
     - "The Fat Burning Reset" — implies you reset your fat burning.
     - Hearing the name alone creates desire.

9-10 (MULTI-BENEFIT): Name implies MULTIPLE benefits simultaneously.
     - "The Inflammation Eraser" — erases inflammation (primary)
        + implies precision and completeness (secondary).
     - "The Neural Performance Optimizer" — brain performance +
        optimization + neural specificity — 3 benefits in one name.
     - Name alone is almost a headline.
```

---

## Evaluation Logic

### Step 1: Parse Name Components

```
FOR each candidate:

  DECOMPOSE mechanism name into parts:

  EXAMPLE: "The Cortisol Switch"
    Part 1: "Cortisol" — identifies the problem/target
    Part 2: "Switch" — implies controllable on/off action

  EXAMPLE: "Neural Trading Reset"
    Part 1: "Neural" — brain-based (credibility + specificity)
    Part 2: "Trading" — context/audience (relevance)
    Part 3: "Reset" — implies return to optimal state (benefit)

  FOR each part:
    CLASSIFY: Is this a benefit word, a credibility word,
    a specificity word, or a neutral descriptor?
```

### Step 2: Identify Implied Benefits

```
FOR each candidate:

  WHAT DOES THE NAME PROMISE without any explanation?

  TECHNIQUE: Read ONLY the name. What does a prospect WANT
  after hearing it?

  LIST all benefits the name implies:
    - Direct benefits (the name says it: "Fat Burning")
    - Implied benefits (the name suggests it: "Reset" → fresh start)
    - Aspirational benefits (the name evokes it: "Neural" → smart, advanced)

  COUNT total implied benefits
  ASSESS strength of each (clear vs. vague implication)
```

### Step 3: The Name-Only Desire Test

```
FOR each candidate:

  SIMULATE: Prospect hears ONLY the mechanism name.
  No explanation, no context, just the name.

  QUESTION: Does hearing this name create DESIRE to learn more?

  STRONG DESIRE (score 8-10):
    - "What is that? I want that."
    - Name alone is intriguing AND beneficial
    - Would click an ad with just this name

  MODERATE DESIRE (score 5-7):
    - "That sounds interesting."
    - Name suggests something good but needs context
    - Might click, might not

  NO DESIRE (score 1-4):
    - "What does that mean?"
    - Name is meaningless without explanation
    - Would scroll past this
```

### Step 4: Optimize If Below Threshold

```
IF current_score < 7:

  TECHNIQUE 1: Replace Neutral Words with Benefit Words
    "Protocol" → "Accelerator" (implies speed)
    "Method" → "Breakthrough" (implies barrier-breaking)
    "System" → "Engine" (implies automatic power)
    "Discovery" → "Solution" (implies problem-solving)

  TECHNIQUE 2: Add Outcome Word
    "The Cortisol [???]" → "The Cortisol Eraser" (implies elimination)
    "The Neural [???]" → "The Neural Upgrade" (implies improvement)

  TECHNIQUE 3: Stack Benefit Implications
    Single benefit: "The Fat Burner" (burns fat)
    Multi benefit: "The Metabolic Reset" (metabolism + fresh start + optimization)

  TECHNIQUE 4: Test Revised Names
    FOR each optimized name:
      RE-RUN name-only desire test
      KEEP only if desire level increases

  NOTE: Name changes here feed BACK into other Layer 2 skills
  (may affect simplicity, image strength, etc.)

  RE-SCORE after optimization
```

## Output Schema

```yaml
output:
  embedded_benefits_scores:
    for_each_candidate:
      - candidate_name: string
        current_score: integer[1-10]
        name_components:
          - part: string
            classification: enum[benefit, credibility, specificity, neutral]
        implied_benefits:
          - benefit: string
            strength: enum[direct, implied, aspirational]
        benefit_count: integer
        desire_test_level: enum[strong, moderate, none]

        optimization_applied: boolean
        optimized_name:
          new_name: string
          benefits_added: [string]
          new_score: integer[1-10]
          name_change_flag: boolean    # True if name was altered
```

---

## Quality Criteria

1. Every candidate's name decomposed into classified parts
2. All implied benefits identified (direct, implied, aspirational)
3. Desire test produces clear level assessment
4. Candidates with score < 7 receive benefit-word optimization
5. Any name changes flagged for cross-skill consistency check
