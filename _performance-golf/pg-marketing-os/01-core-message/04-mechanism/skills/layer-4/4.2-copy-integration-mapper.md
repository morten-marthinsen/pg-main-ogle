# Skill 4.2: Copy Integration Mapper

**Version:** 1.0
**Layer:** 4 (Package Assembly)
**Type:** Deployment Planning
**Dependencies:** mechanism_brief.json, upstream_context (big_idea, niche, audience)
**Output:** copy_integration_map.json

---

## Purpose

Map EXACTLY where and how the mechanism appears across all downstream copy skills. A mechanism isn't mentioned once — it's woven throughout headlines, leads, body copy, proof sections, offers, and closes. This mapper defines the mechanism's role at each touchpoint so downstream skills don't have to guess.

---

## The Question

**"At every point in the sales copy where the mechanism could appear — what form does it take, what job does it do, and how much detail is appropriate?"**

## Mapping Logic

### Step 1: Determine Copy Format Rules

```
BASED ON copy_format:

  VSL:
    - Mechanism reveal typically 8-15 minutes in
    - Build curiosity before reveal
    - Name repeated 7-12 times total
    - Proof woven into explanation (not separate)
    - Visual demonstrations amplify mechanism

  SALES PAGE:
    - Mechanism in subheadline or first 300 words (teaser)
    - Full reveal in middle third of page
    - Mechanism name in at least 2 section headers
    - Proof section explicitly ties back to mechanism
    - Offer section frames product AS delivery vehicle for mechanism

  EMAIL SEQUENCE:
    - Email 1-2: Problem + hint of mechanism (no name)
    - Email 3-4: Mechanism reveal + explanation
    - Email 5+: Proof + mechanism in offer context
    - Subject lines: 30% reference mechanism by name

  ADVERTORIAL:
    - Mechanism appears in first 200 words (editorial hook)
    - Treated as "discovery" or "breakthrough" language
    - Proof integrated as "journalism" not "selling"
    - Name used more formally (as if reporting on it)

  WEBINAR:
    - Mechanism is "Content Segment 2" (after problem, before offer)
    - 10-20 minute dedicated explanation
    - Live demonstration or walkthrough
    - Q&A handles mechanism objections
```

### Step 2: Map Headline Touchpoints

```
FROM mechanism_brief:

  DETERMINE headline form:

  IF mechanism name is inherently intriguing:
    form = "name_only"
    Example: "The [Mechanism Name]: Why Everything You've Tried Has Failed"

  IF mechanism name needs context:
    form = "benefit_hint"
    Example: "How a Hidden [Type] Explains Why You Can't [Problem]"

  IF audience is high-sophistication:
    form = "curiosity_gap"
    Example: "The [Niche] Secret That [Expert] Called '[Mechanism Name]'"

  IF mechanism has strong proof:
    form = "claim"
    Example: "[Mechanism Name]: The [Proof Point] Reason [Benefit]"

  CAUTION:
    - Never explain mechanism fully in headline
    - Never use mechanism name if it sounds generic out of context
    - Never promise what mechanism can't deliver
    - Name in headline ONLY if name itself creates curiosity
```

### Step 3: Map Reveal Section Structure

```
FROM mechanism_brief.deployment_notes:

  STRUCTURE the reveal section:

  SETUP (what comes before):
    - Final problem agitation
    - "But what if there was a reason..." bridge
    - Curiosity peak before reveal

  REVEAL MOMENT:
    IF name_first: "It's called [Mechanism Name]."
    IF concept_first: "Here's what's actually happening... [explain]...
                       Scientists call this [Mechanism Name]."

  EXPLANATION:
    USE explanation level based on audience sophistication:
      Low sophistication → L1 (one sentence) + analogy
      Medium sophistication → L2 (paragraph) + proof integration
      High sophistication → L3 (full walkthrough) + data

  PROOF WITHIN REVEAL:
    INTERWEAVE proof at claim moments:
      "[Claim statement]. In fact, [proof element].
       Which means [implication for prospect]."

  EMOTIONAL PEAK:
    Identify the moment of maximum emotional response:
      - The "aha" (understanding) → use for Explainer archetype
      - The "oh no" (doomsday) → use for Provocateur archetype
      - The "that's me" (recognition) → use for Scientist archetype
      - The "I want that" (desire) → use for Activator archetype

  TRANSITION OUT:
    Bridge to proof section OR offer:
      "And the evidence is overwhelming..."
      OR "Which is exactly why [Product] was built around this..."
```

### Step 4: Map Offer Connection

```
FROM mechanism_brief.root_cause_connection:

  FRAME product AS mechanism delivery vehicle:

  inevitability_language:
    "[Product] is the ONLY [format] built specifically around [Mechanism Name]"
    "Without [Mechanism Name], any [product type] is just guessing"
    "[Mechanism Name] is why [Product] works when [alternatives] don't"

  exclusivity_claim:
    FROM differentiation assessment:
    "You won't find [Mechanism Name] in [competitor type]"
    "This is the first time [Mechanism Name] has been available in [format]"

  value_anchor:
    FROM super_power + proof:
    "[Mechanism Name] alone is worth [anchor price] based on [proof]"
    "Consider: [proof element] — and that's just ONE result of [Mechanism Name]"
```

### Step 5: Define Consistency Rules

```
ACROSS all touchpoints:

  NAME CONSISTENCY:
    - Always capitalize mechanism name the same way
    - Define: use quotes first time, no quotes thereafter? Or always caps?
    - Never abbreviate unless abbreviation is part of the name
    - If mechanism has a "type" label (e.g., "protocol," "method," "system"),
      use it consistently — don't switch between "protocol" and "system"

  CLAIM CONSISTENCY:
    - The core claim stated in reveal = same claim referenced in proof/offer
    - Don't escalate claims downstream (if reveal says "helps," offer can't say "guarantees")
    - Proof must match claims EXACTLY (not "in the ballpark")

  TONE CONSISTENCY:
    - Mechanism discussed with same register throughout
    - If reveal is conversational, proof section can't suddenly become academic
    - Emotional register established at reveal carries through

  REPETITION RULES:
    - Name the mechanism by name: minimum 5 times, maximum 15 times across full copy
    - Casual callbacks: "remember the [Name]..." at least twice after reveal
    - Never go more than 500 words without a mechanism reference after reveal
```

---

## Quality Criteria

1. Every major copy section has explicit mechanism touchpoint defined
2. Reveal section has complete structure (setup → reveal → explain → proof → transition)
3. Offer section explicitly frames product as mechanism delivery vehicle
4. Consistency rules defined for name, claims, and tone
5. Format-specific notes provided for the target copy format
6. No section left with "use judgment" — every touchpoint has specific guidance
7. Headline form matches mechanism name characteristics
