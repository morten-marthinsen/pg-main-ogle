# Layer 1.3 — Gap-to-Chunk Converter

## Purpose

Convert each identified prospect gap question into a CPB chunk assignment — defining what each chunk must accomplish, its scope, and its relationship to the campaign thesis. This creates the structural blueprint that Layer 2 will fill with actual claims, proof, and benefits.

## Execution Protocol

### Step 1: Group Related Gaps

Some gap questions are closely related and can be addressed by a single CPB chunk. Others require their own dedicated chunk.

**Grouping rules:**
- **Standalone gaps:** Any gap classified as "critical" gets its own dedicated chunk
- **Combinable gaps:** Two "supporting" gaps can share a chunk IF they share the same proof domain
- **Sequential gaps:** Gaps that naturally flow into each other (e.g., "How does it work?" → "Why is it better?") can be adjacent chunks but should NOT be combined
- **Never combine:** "access" gap with any other gap — it always stands alone as the final chunk

### Step 2: Create Chunk Assignment for Each Gap/Group

For each chunk, define:

```yaml
chunk_assignment:
  chunk_id: "CPB-001"  # sequential identifier
  assigned_gaps: [string]  # gap_ids this chunk addresses
  primary_gap: string  # the dominant gap if multiple
  chunk_purpose: string  # what this chunk must accomplish
  thesis_connection: string  # how this chunk supports the overall thesis
  claim_direction: string  # what the claim should assert
  proof_requirements:
    required_proof_types: [string]  # which proof types are needed
    available_proof_ids: [string]  # from inventory
    proof_gap: boolean  # true if needed proof is not in inventory
  benefit_direction: string  # what benefit should the prospect feel
  upstream_element_focus: enum[mechanism, root_cause, promise, combined]
  estimated_complexity: enum[simple, moderate, complex]
```

### Step 3: Determine Chunk Count

The optimal number of CPB chunks depends on:

| Factor | Fewer Chunks (5-6) | More Chunks (7-9) |
|--------|--------------------|--------------------|
| Proof inventory density | Sparse (< 15 elements) | Dense (> 25 elements) |
| Mechanism complexity | Simple, intuitive | Complex, multi-step |
| Schwartz stage | Stage 1-2 (less skepticism) | Stage 4-5 (high skepticism) |
| Niche | Sports instruction, self-improvement | Health, wealth (regulated, skeptical) |
| Campaign format | Short VSL, email | Long-form sales page, magalog |

**Hard limits:**
- Minimum: 5 chunks (cannot build adequate argument with fewer)
- Maximum: 12 chunks (beyond this, argument loses focus)
- Target: 7-9 chunks (optimal depth for most campaigns)

### Step 4: Validate Chunk Coverage

Check that every identified gap (critical, important, supporting) is assigned to at least one chunk. No gap should be orphaned.

```python
def validate_coverage(gaps, chunk_assignments):
    assigned_gaps = set()
    for chunk in chunk_assignments:
        assigned_gaps.update(chunk["assigned_gaps"])

    unassigned = [g for g in gaps if g["gap_id"] not in assigned_gaps]
    if unassigned:
        return {"status": "fail", "unassigned_gaps": unassigned}
    return {"status": "pass"}
```

### Step 5: Verify Access Chunk Position

The "access" gap (How can I get started?) MUST be assigned to the final chunk. This is non-negotiable per the E5 framework.

### Step 6: Document Chunk Dependencies

Some chunks build on information established in earlier chunks. Map dependencies:
- Which chunks reference the mechanism explanation? (depends on a chunk that explains it)
- Which chunks reference the root cause? (depends on root cause being established)
- Which chunks reference results? (may depend on mechanism being understood first)

## Quality Gates

- [ ] Every identified gap assigned to at least one chunk
- [ ] No orphaned gaps (coverage_check.all_covered = true)
- [ ] Chunk count within range (5-12, target 7-9)
- [ ] Chunk count rationale documented
- [ ] Access gap assigned to final chunk position
- [ ] Each chunk has clear purpose and thesis connection
- [ ] Each chunk has claim direction defined
- [ ] Proof requirements documented per chunk
- [ ] Proof gaps flagged (for potential feedback bus)
- [ ] Dependencies mapped between chunks

---

## Anti-Patterns

- **Mega-chunks:** Combining 4+ gaps into a single chunk — too broad, loses focus
- **Micro-chunks:** Creating a separate chunk for every minor supporting gap — too fragmented
- **Orphan gaps:** Leaving any gap unassigned — leaves belief holes in the argument
- **Access in the middle:** Placing the "How to get started" chunk anywhere but last
- **Ignoring dependencies:** Referencing mechanism details before they've been explained
- **Uniform complexity:** All chunks marked "simple" or all "complex" — real arguments have variety
