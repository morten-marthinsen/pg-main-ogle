# Skill 2.1: Pattern Analysis

**Version:** 1.0
**Layer:** 2 (Analysis)
**Type:** Pattern Detection
**Dependencies:** quotes_validated.json
**Output:** patterns.md

---

## PURPOSE

Cluster quotes by theme within each bucket. Identify the top 3-5 patterns per bucket with frequency counts and representative quotes. This surfaces the dominant themes in the evidence base — what problems are most common, what hopes are most frequently expressed, etc.

---

## INPUT

```yaml
input:
  quotes_validated: quotes_validated.json
```

---

## PROCESS

### Step 1: Group Quotes by Bucket

```
PARTITION quotes_validated into 4 sets:
  - pain_quotes = all quotes where bucket == "pain"
  - hope_quotes = all quotes where bucket == "hope"
  - root_cause_quotes = all quotes where bucket == "root_cause"
  - solutions_tried_quotes = all quotes where bucket == "solutions_tried"
```

### Step 2: Identify Themes Within Each Bucket

```
FOR each bucket:

  SCAN quotes for recurring themes:

  THEME DETECTION methods:
  1. Keyword clustering — group by shared terminology
  2. Semantic similarity — group by meaning even with different words
  3. Problem/solution pairing — group by what they're describing

  EXAMPLE pain themes (golf market):
  - "Inconsistency" — quotes about unpredictable results
  - "Embarrassment" — quotes about social/public failure
  - "Wasted money" — quotes about failed purchases
  - "Information overload" — quotes about conflicting advice
  - "Physical limitations" — quotes about age, flexibility

  ASSIGN each quote to one theme
  COUNT quotes per theme
```

### Step 3: Rank and Select Top Patterns

```
FOR each bucket:

  RANK themes by:
  1. Quote count (primary)
  2. Emotional intensity (secondary — high intensity = more weight)
  3. Source diversity (tertiary — themes from 2+ sources rank higher)

  SELECT top 3-5 patterns per bucket
  (Fewer if bucket has low quote count)

  FOR each selected pattern:
    - Count: total quotes in this pattern
    - Sources: number of distinct sources
    - Select 2-3 representative quotes (highest engagement + clearest expression)
```

### Step 4: Compile Pattern Output

```markdown
# patterns.md

## Pattern Analysis

Generated: [date]
Total quotes analyzed: [count]

---

## Pain Patterns

### 1. [Pattern Name] (X quotes from Y sources)

**Theme:** [One-sentence description of this pattern]

**Representative Quotes:**

> "[Quote 1]"
> — [Author], [Platform], [Date]

> "[Quote 2]"
> — [Author], [Platform], [Date]

> "[Quote 3]"
> — [Author], [Platform], [Date]

### 2. [Pattern Name] (X quotes from Y sources)
...

---

## Hope Patterns
...

## Root Cause Patterns
...

## Solutions Tried Patterns
...
```

---

## OUTPUT

**File:** `patterns.md` saved to project folder

**Contents:**
- 3-5 patterns per bucket (12-20 total patterns)
- Quote counts and source diversity for each
- 2-3 representative quotes per pattern

---

## CONSTRAINTS

1. NEVER invent patterns not supported by quotes. Every pattern needs 3+ quotes minimum.
2. NEVER use the same quote as representative for multiple patterns.
3. ALWAYS include quote counts — patterns without quantification are speculation.
4. ALWAYS attribute representative quotes with source information.
5. NEVER exceed 5 patterns per bucket — focus on dominant themes.

---

## PATTERN NAMING GUIDELINES

```
GOOD pattern names:
- Short, descriptive nouns or noun phrases
- Capture the essence in 2-4 words
- Examples: "Inconsistency Frustration", "Wasted Money", "Social Embarrassment"

BAD pattern names:
- Long sentences
- Vague labels ("Various Issues", "Different Problems")
- Marketing-speak ("Transformational Opportunity")
```

---

## ERROR HANDLING

| Error | Response |
|-------|----------|
| Bucket has < 10 quotes | Report 1-2 patterns only, flag thin evidence |
| Theme has only 1-2 quotes | Do not elevate to pattern, leave as individual quotes |
| No clear patterns emerge | Report as "Diffuse themes — no dominant patterns" |
| Quotes split evenly | Report top 3 even if counts are similar |

---

## ANTI-DEGRADATION RULE REFERENCES

- **Rule 8:** Do not synthesize during assembly — this skill produces analysis, assembly just compiles it.
