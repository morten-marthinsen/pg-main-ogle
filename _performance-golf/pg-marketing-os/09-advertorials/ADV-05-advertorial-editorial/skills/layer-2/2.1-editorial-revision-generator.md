# 2.1-editorial-revision-generator.md

> **Layer:** 2 (Revision)
> **Skill:** ADV-05-advertorial-editorial
> **Purpose:** Generate revised versions addressing audit findings
> **Outputs:** `revision_candidates` array with revised advertorial versions

---

## FUNCTION

Generate revised versions of the assembled advertorial that address all findings from the editorial smell audit and hook strength review. Each revision targets specific issues while preserving (or improving) overall editorial quality. Revisions must not introduce new problems -- anti-degradation checks run on every revision.

---

## REVISION PROTOCOL

### Step 1: Prioritize Issues

```
FROM smell_audit.issues_for_revision + hook_review.revision_suggestions:
  RANK issues by:
    1. severity (severe > moderate > minor)
    2. section (body > bridge > lead > hook -- most text = most impact)
    3. type (ad-smell > voice drift > proof density > compliance)

  CREATE revision_agenda:
    Priority 1: [most severe issues]
    Priority 2: [moderate issues]
    Priority 3: [minor polish]
```

### Step 2: Generate Targeted Revisions

```
FOR each revision candidate (generate 3-5):

  APPROACH 1: Conservative Edit
    - Fix only Priority 1 issues
    - Minimal changes to text
    - Preserve original voice and structure
    - Best for pieces that are mostly editorial-ready

  APPROACH 2: Moderate Revision
    - Fix Priority 1 and 2 issues
    - Adjust phrasing, reframe problematic sections
    - May restructure 1-2 paragraphs
    - Best for pieces with several moderate issues

  APPROACH 3: Deep Editorial Polish
    - Address all priorities
    - Full editorial pass on voice, pacing, transitions
    - May rewrite problematic paragraphs entirely
    - Best for pieces that need significant editorial improvement

  FOR each approach:
    1. Apply revisions per priority agenda
    2. Run anti-slop check on revised text
    3. Compare before/after editorial score
    4. Verify no new ad-smell introduced
    5. Verify voice register maintained
    6. Verify proof density still compliant
    7. Verify CTA still singular and text-linked
```

### Step 3: Anti-Degradation Check on Revisions

```
FOR each revised version:
  COMPARE to original:
    promotional_language_before: [count]
    promotional_language_after: [count]
    IF after > before: REJECT revision

    editorial_authenticity_before: [score]
    editorial_authenticity_after: [score]
    IF after < before: REJECT revision

    slop_phrases_after: [list]
    IF any found: REJECT revision

  ONLY revisions that improve or maintain quality proceed
```

### Step 4: Score Revisions

```
FOR each surviving revision:
  SCORE on editorial criteria (1-10):
    - editorial_authenticity
    - revision_quality (net improvement)
    - hook_strength (maintained or improved)
    - voice_unity
    - proof_integration
    - bridge_naturalness
    - compliance_readiness

  RANK by weighted total
  PASS top 5 to Arena
```

## FAILURE MODES

| Mode | Trigger | Remediation |
|------|---------|-------------|
| All revisions degrade quality | Every revision introduces problems | Return to specific issue with targeted fix |
| Revision introduces slop | Anti-slop check catches new phrases | Remove slop, regenerate revision |
| CTA modified during revision | Revision changed CTA count or format | Revert CTA to original |
| Voice drift in revision | Revision shifted register | Realign to established register |
| Word count changed significantly | Revision expanded/contracted too much | Trim/expand to stay within bounds |
