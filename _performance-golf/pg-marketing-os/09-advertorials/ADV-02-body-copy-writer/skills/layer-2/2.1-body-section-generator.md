# 2.1-body-section-generator.md

> **Layer:** 2 (Generation)
> **Skill:** ADV-02-body-copy-writer
> **Purpose:** Generate body sections per type-specific structure template
> **Outputs:** `body_sections` array with complete section content

---

## FUNCTION

Generate the substantive body sections following the type-specific structure template. Each section must serve its designated purpose (context, mechanism, proof, narrative, transition), maintain the voice register established in the lead, and stay within word count targets. This microskill handles structure and narrative; proof embedding is handled separately in 2.2.

---

## GENERATION PROTOCOL

### Step 1: Section-by-Section Generation

```
FOR each section in body_structure_template.sections:

  INPUTS:
    - section.purpose: What this section accomplishes
    - section.word_target: Target word count
    - section.proof_allowed: Whether proof goes here (from weaving plan)
    - section.mechanism_mention: Whether mechanism is introduced here
    - lead_handoff.voice_register: Voice to maintain
    - lead_handoff.narrative_thread: Thread to continue

  GENERATE section content:
    1. Open with transition from previous section (or from lead for section 1)
    2. Deliver section's purpose content
    3. If proof_allowed: Leave proof insertion point marked [PROOF_SLOT]
    4. Close with transition to next section
    5. Maintain voice register throughout

  VALIDATE:
    [ ] Word count within 80-120% of target
    [ ] Voice register matches lead
    [ ] Purpose fulfilled
    [ ] Transition smooth
    [ ] No promotional language
    [ ] Product name absent (if before allowed section)
```

### Step 2: Narrative Thread Maintenance

```
ACROSS all sections:
  VERIFY narrative coherence:
    - Each section builds on previous
    - No information repeated verbatim
    - Tension/curiosity maintained or built
    - Reader always has reason to continue
    - Mechanism revelation paced per template

  THREAD elements:
    - Callback to hook/lead elements
    - Progressive mechanism disclosure
    - Escalating evidence weight
    - Building toward bridge transition
```

### Step 3: Mechanism Introduction (Type-Appropriate)

```
IN mechanism introduction section:
  FRAME mechanism editorially:

  EDITORIAL FRAMING (required):
    "What [researchers/experts] identified was [mechanism]..."
    "The [category] compound, known as [mechanism name], works by..."
    "The approach centers on [mechanism name], which [explanation]..."

  PROMOTIONAL FRAMING (forbidden):
    "Our product uses [mechanism name]..."
    "We developed [mechanism name] to..."
    "[Brand] contains [mechanism name]..."

  PRODUCT NAME:
    - Not before designated section in template
    - When introduced, frame as "available as [product name]" or similar
    - Never lead with product name -- always lead with mechanism
```

### Step 4: Quality Check Per Section

```
FOR each generated section:
  CHECK:
    [ ] Reads as article content, not ad copy
    [ ] Voice consistent with established register
    [ ] Appropriate information density (not too thin, not too dense)
    [ ] Transitions connect naturally
    [ ] No banned phrases from anti-slop lexicon
    [ ] Word count within bounds
```

## FAILURE MODES

| Mode | Trigger | Remediation |
|------|---------|-------------|
| Voice drift | Section deviates from register | Rewrite matching established voice |
| Section too thin | Word count < 60% of target | Expand with editorial detail |
| Promotional language | Ad-speak detected | Rewrite through editorial lens |
| Product name too early | Named before allowed section | Replace with mechanism/category language |
| Narrative break | Section doesn't connect to previous | Add transitional sentence |
