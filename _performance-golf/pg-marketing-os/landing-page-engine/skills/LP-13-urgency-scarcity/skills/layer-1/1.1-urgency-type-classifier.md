# 1.1 — Urgency Type Classifier

**Skill:** LP-13-urgency-scarcity
**Position:** Layer 1, Microskill 1
**Purpose:** Select 1-2 urgency types from the 6-type taxonomy based on available data, LP-06 recommendations, page type, and vertical. This is the gating decision that determines which Layer 2 microskills execute. Maximum 2 types. If no justified urgency exists, output the no-urgency determination and route to Layer 4.
**Effort Level:** medium
**Dependencies:** `urgency-data.md` (0.2), `packages-loaded.md` (0.1), `specimens-loaded.md` (0.3)
**Output:** `urgency-types-selected.md`

---

## Input Schema

```yaml
input:
  urgency_data_inventory:
    types_with_sufficient_data: "[list from urgency-data.md]"
    types_with_partial_data: "[list from urgency-data.md]"
```

## Execution Logic

1. **Review the urgency data inventory.** List every type rated SUFFICIENT or PARTIAL. These are the candidates for selection.

2. **Apply the page-type filter.** Some urgency types work better for specific page types:

   | Urgency Type | Type A Fit | Type B Fit | Hybrid Fit |
   |-------------|-----------|-----------|------------|
   | Supply Scarcity | Medium | High | High |
   | Time-Based | High | High | High |
   | Price-Based | High | High | High |
   | Availability Window | High | Low | Medium |
   | Bonus Deadline | High | Low | Medium |
   | Natural Urgency | High | Low | Medium |

   - Type B pages rarely use Availability Window, Bonus Deadline, or Natural Urgency (these require narrative space that Type B does not provide)
   - Type A pages can use any type effectively

3. **Apply the vertical lens.** Some urgency types are more natural for certain verticals:

   | Vertical | Natural Urgency Types |
   |---------|----------------------|
   | Health/Supplement | Supply Scarcity (batch manufacturing), Natural Urgency (progressive condition) |
   | Info Product | Time-Based (launch window), Availability Window (cohort), Bonus Deadline |
   | Finance | Time-Based (market window), Natural Urgency (compounding cost of delay) |
   | Personal Development | Availability Window (coaching capacity), Bonus Deadline |
   | Technology/SaaS | Price-Based (introductory pricing), Time-Based (launch) |
   | Ecomm/DTC | Supply Scarcity (limited edition), Price-Based (sale pricing) |

4. **Check LP-06 recommendations.** If LP-06 recommended specific types AND those types have SUFFICIENT data, prefer them. LP-06 made the strategic decision with full page context.

5. **Select 1-2 types using the priority algorithm:**

   ```
   Priority 1: Type with SUFFICIENT data + LP-06 recommendation + high page-type fit
   Priority 2: Type with SUFFICIENT data + high page-type fit + natural for vertical
   Priority 3: Type with SUFFICIENT data + LP-06 recommendation (any page-type fit)
   Priority 4: Type with PARTIAL data + LP-06 recommendation (flag for client input)
   ```

   - Select the highest-priority type as PRIMARY
   - If a second type is compatible (check compatibility matrix) and has SUFFICIENT data, select as SECONDARY
   - NEVER select more than 2 types

6. **Run compatibility check** on selected types against the compatibility matrix from LP-13-AGENT.md:
   - Supply + Time: Compatible if independent justifications
   - Supply + Price: Caution -- only if independently justified
   - Time + Price: Compatible (often same constraint)
   - Time + Bonus: Compatible (often same deadline)
   - Supply + Availability: INCOMPATIBLE
   - Natural + Any: Compatible
   - 3+ types: NEVER

7. **Determine no-urgency path.** If:
   - All types rated INSUFFICIENT, AND
   - LP-06 recommended no urgency, AND
   - Brief contained no urgency inputs
   - Then: Output `no_justified_urgency: true` and document the reason. LP-13 will skip to Layer 4 with `urgency_present: false`.

8. **Write `urgency-types-selected.md`** with selection rationale.

## Quality Criteria
- Maximum 2 types selected (hard limit)
- Every selected type has at least PARTIAL data (never selected from INSUFFICIENT)
- Compatibility matrix checked for multi-type selections
- LP-06 recommendations considered (not blindly followed -- data must support)
- Layer 2 routing explicitly documented (which microskills run)
- No-urgency path explicitly evaluated (not assumed impossible)

## Failure Modes
- Selecting 3+ types ("all three are justified!") -> Maximum is 2. Period. Pick the 2 strongest.
- Selecting a type with INSUFFICIENT data because "urgency is important" -> Insufficient data = cannot justify = cannot select. Fabricated urgency is worse than no urgency.
- Ignoring LP-06 recommendations entirely without stated reason -> LP-06 did strategic analysis. Override requires data-based justification.
- Selecting incompatible types -> Check the compatibility matrix BEFORE finalizing selection.
- Assuming every page needs urgency -> Some pages have no justified urgency. Setting `urgency_present: false` is a valid, correct output.
