# 4.2 — Summary Writer

**Skill:** LP-13-urgency-scarcity
**Position:** Layer 4, Microskill 2
**Purpose:** Write the human-readable URGENCY-SUMMARY.md file. This is the document the human reviewer reviews to understand what urgency/scarcity was selected, why it was selected, what copy was produced, and how it scored. Unlike urgency-package.json (which is machine-readable for downstream skills), this file is optimized for human review and decision-making.
**Effort Level:** low
**Dependencies:** `urgency-package.json` (4.1), all Layer 1-3 output files
**Output:** `URGENCY-SUMMARY.md`

---

## Input Schema

```yaml
input:
  urgency_package: "[from urgency-package.json]"
  urgency_types_selected: "[from 1.1]"
  justification_plan: "[from 1.2]"
  placement_strategy: "[from 1.3]"
  urgency_audit: "[from 3.1]"
  justification_audit: "[from 3.2]"
  page_type: "[type_a | type_b | hybrid]"
  project_name: "[from packages-loaded.md]"
```

---

## Execution Logic

1. **Read urgency-package.json** and extract all key data for the summary.

2. **Write the summary** following the template from LP-13-AGENT.md's URGENCY-SUMMARY.MD REQUIRED SECTIONS.

3. **Include decision rationale** — not just WHAT was selected, but WHY:
   - Why these specific urgency types were chosen
   - Why other types were rejected
   - What data supported the selection
   - What data gaps exist (flagged for client)

4. **Include copy excerpts** — not the full copy (that's in the package), but representative excerpts that show the approach:
   - Urgency section headline (Type A)
   - Master justification paragraph opening sentence
   - Key urgency elements (Type B badges/indicators)
   - P.S. callback

5. **Include validation results** with enough detail for the human reviewer to assess quality without reading the full audit files.

6. **Include downstream integration notes** — how LP-14 and LP-15 should use this package.

## Quality Criteria
- Summary is readable by a human without context about LP-13's internal execution
- Urgency types include justification summaries (not just type names)
- Copy excerpts give enough flavor to evaluate the approach without reading full files
- Data gaps clearly listed with impact on specific copy elements
- Validation results include enough detail to assess quality at a glance
- Placement map shows exactly where every urgency element goes on the page
- Downstream handoff instructions are specific and actionable

## Failure Modes
- Summary that only lists type names without justification -> the human reviewer needs to know WHY these types were selected and WHAT makes the urgency real.
- Missing data gap section -> If any claims are UNVERIFIABLE, the human reviewer needs to know so he can confirm with the client.
- Vague downstream handoffs ("urgency available") -> Must specify the urgency type, constraint, deadline, and which copy elements LP-14/LP-15 should integrate.
- Not including rejection rationale -> Knowing what was considered and rejected helps the human reviewer evaluate if the right decision was made.
- Copy excerpts that are too long (dumping full copy into summary) -> Summary should have representative excerpts. Full copy lives in the package.
