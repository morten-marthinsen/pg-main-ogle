# 0.1 — Brief Loader

**Skill:** LP-10-social-proof-writer
**Position:** Layer 0, Microskill 1
**Purpose:** Load and verify the upstream package files that LP-10 depends on: `page-brief.json` (LP-00), `section-sequence.json` (LP-04), `conversion-intelligence.json` (LP-01), and `hero-section-package.json` (LP-07). Extract the key fields that govern proof copy voice, audience targeting, and threading.
**Effort Level:** low
**Dependencies:** LP-00 output, LP-01 output, LP-04 output, LP-07 output
**Output:** `brief-loaded.md`

---

## Input Schema

```yaml
input:
  project_directory: "[path to active project output directory]"
  expected_files:
    - "page-brief.json"
    - "section-sequence.json"
    - "conversion-intelligence.json"
    - "hero-section-package.json"
```

---

## Execution Logic

1. **Locate each upstream file** in the project output directory. Check for existence of all four files.

2. **If `page-brief.json` or `section-sequence.json` is MISSING:** HALT immediately. These are required — LP-10 cannot determine page type, audience, or section context without them.

3. **If `conversion-intelligence.json` is MISSING:** Continue with WARNING. Proof copy can be written without benchmark data, but proof element impact data will be unavailable.

4. **If `hero-section-package.json` is MISSING:** Continue with WARNING. Threading anchor will be unavailable — proof copy may not align perfectly with the hero section's promise.

5. **Load `page-brief.json`** and extract:
   - `page_type` (type_a | type_b | hybrid)
   - `vertical` (health, finance, info_product, personal_dev, technology, etc.)
   - `awareness_stage` (1-5)
   - `audience_profile` (primary pain, failed solutions, primary desire, demographics)
   - `voice_direction` (tone, register, anti-voice)
   - `offer_summary` (product name, price point, guarantee)
   - `proof_inventory` (if included — available testimonials, studies, endorsements)

6. **Load `section-sequence.json`** and extract:
   - `total_sections` (count)
   - Section names and order (for transition context — what comes before/after each proof block)
   - Proof density assignments per section

7. **Load `conversion-intelligence.json`** and extract:
   - Proof element impact data (which proof types have highest conversion impact for this vertical)
   - Benchmark data relevant to proof architecture
   - Industry-specific proof norms

8. **Load `hero-section-package.json`** and extract:
   - `threading_anchors.primary_hook_phrase` — proof must substantiate this promise
   - `threading_anchors.promise_statement` — proof must connect back to this
   - `threading_anchors.audience_qualifier` — proof intros should reference this audience

9. **Write `brief-loaded.md`** with all extracted fields organized by source file.

## Quality Criteria
- Critical files (page-brief.json, section-sequence.json) loaded and confirmed present
- Key fields extracted with no gaps (missing fields marked `not_available`, not silently omitted)
- Page type confirmed — governs all proof copy decisions
- Section order context captured for transition planning
- Threading anchors extracted for proof-to-promise alignment

## Failure Modes
- page-brief.json or section-sequence.json missing -> HALT with specific file names listed. Do not proceed with partial data.
- Fields present but empty in upstream JSON -> Extract as `not_available` and flag — do not invent values.
- Proof inventory in brief does not match proof-architecture.json inventory -> Flag discrepancy in 0.2 (not this microskill's responsibility to resolve).
