# 0.2 — Proof Architecture Loader

**Skill:** LP-10-social-proof-writer
**Position:** Layer 0, Microskill 2
**Purpose:** Load `proof-architecture.json` (LP-05 output) — THE critical input for LP-10. Extract and inventory every `slot_id`, proof type, format, quality requirement, and content source. This file IS the writing assignment sheet.
**Effort Level:** low
**Dependencies:** LP-05 output (`proof-architecture.json`)
**Output:** `proof-slots-loaded.md`

---

## Input Schema

```yaml
input:
  project_directory: "[path to active project output directory]"
  required_file: "proof-architecture.json"
```

---

## Execution Logic

1. **Locate `proof-architecture.json`** in the project output directory.

2. **If MISSING:** HALT immediately. LP-10 CANNOT function without LP-05's proof architecture. There is no fallback — the entire skill depends on this file.

3. **Load the file** and extract the complete `section_proof_map` array.

4. **For each section in the proof map,** extract:
   - `section_number` and `section_name`
   - `density_level` (none | light | medium | heavy | maximum)
   - Each `proof_element` within the section:
     - `slot_id` — the unique identifier that LP-10 writes copy for
     - `type` — the proof type code (clinical_study, expert_endorsement, before_after, photo_testimonial, volume_signal, text_testimonial, media_mention, certification, platform_review, performance_guarantee, stat_bar, video_testimonial)
     - `count` — how many of this type in this slot
     - `format` — text, photo, video, widget, stat_bar, cascade, badge, certificate, comparison, before_after
     - `quality_requirement` — LP-05's quality spec for this slot
     - `content_source` — available (with path/description) or requires_sourcing
     - `placement_within_section` — opening, embedded, closing, standalone_block
   - `transition_in` and `transition_out` directions from LP-05
   - `wave_zone` — early, education, accumulation, close

5. **Extract proof strategy metadata:**
   - `proof_strategy.mode` (proof_saturation | authority_substitution)
   - `proof_strategy.wave_pattern` (type_a_wave | type_b_wave | hybrid_wave)
   - `volume_targets` — planned testimonial/study/endorsement counts

6. **Extract proof gaps:**
   - All entries from the `proof_gaps` array — these inform which slots will need placeholder copy

7. **Build the complete slot inventory** — a numbered list of every writing assignment.

8. **Write `proof-slots-loaded.md`** with the full slot inventory.

## Quality Criteria
- proof-architecture.json loaded — HALT if missing
- Every slot_id extracted and inventoried — none silently omitted
- Quality requirements captured verbatim from LP-05 — these are the writing specs
- Content source status captured for each slot — critical for determining placeholder vs. real copy
- Transition directions captured for transition planning in Layer 1
- Proof gaps documented — informs which microskills will need `requires_sourcing` handling

## Failure Modes
- proof-architecture.json missing -> HALT. No fallback. LP-10 cannot execute without LP-05's output.
- Slot_id format inconsistent across sections -> Flag but continue — use slot_ids as-is from LP-05.
- Quality requirements vague in LP-05 output -> Flag for Layer 1 analysis — 1.1 will need to interpret vague requirements.
- All slots marked `requires_sourcing` -> Continue but flag WARNING — LP-10 will produce a placeholder package, not a final copy package.
