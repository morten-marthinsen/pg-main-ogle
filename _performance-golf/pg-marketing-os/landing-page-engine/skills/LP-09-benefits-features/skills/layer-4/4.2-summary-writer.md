# 4.2 — Summary Writer

**Skill:** LP-09-benefits-features
**Position:** Layer 4, Microskill 2
**Purpose:** Write the human-readable BENEFITS-SECTION-SUMMARY.md that provides the human reviewer with a quick overview of all benefit section copy, quality scores, proof gaps, and key decisions. This is the document the human reviewer reads to decide whether to review the full package or request revisions.
**Effort Level:** low
**Dependencies:** `benefits-section-package.json` (4.1), `benefit-audit.md` (3.1), `dfwbp-audit.md` (3.2), `anti-slop-scan.md` (3.3)
**Output:** `BENEFITS-SECTION-SUMMARY.md`

---

## Input Schema

```yaml
input:
  benefits_package: "[benefits-section-package.json -- compiled output]"
  benefit_audit: "[benefit-audit.md -- score + details]"
  dfwbp_audit: "[dfwbp-audit.md -- completeness details]"
  anti_slop_scan: "[anti-slop-scan.md -- result]"
```

---

## Execution Logic

1. **Extract key information** from the compiled package:
   - Total sections written
   - Content types used
   - Total elements across all sections
   - Total word count
   - Proof gaps (count + list)
   - Threading anchor reference

2. **Extract quality scores:**
   - Benefit audit score (X.X/10)
   - D-F-W-B-P audit result (PASS/FAIL + completion rate)
   - Anti-slop scan result (PASS/FAIL + flag count)

3. **Summarize each section** with:
   - Section name and content type
   - Element count and word count
   - Key benefits listed (1-line summaries)
   - Proof status (all cited / gaps noted)

4. **Document proof gaps** that need human resolution:
   - Which elements lack proof
   - What type of proof would be ideal
   - Impact on the section (minor gap vs. critical — is this a key benefit without evidence?)

5. **Write BENEFITS-SECTION-SUMMARY.md** in the format below.

---

## Quality Criteria
- Summary is scannable in under 2 minutes
- Every section has a clear summary with key benefits listed
- Proof gaps are prominently displayed with impact assessment
- Quality scores are accurate (match the audit files)
- Recommendation is actionable — tells the human reviewer exactly what to do next
- Downstream handoff notes are complete

## Failure Modes
- Summary is longer than the actual package -> Keep it concise. This is a summary, not a reproduction.
- Proof gaps buried in text -> Proof gaps must be in a dedicated table, not mentioned in passing.
- Missing sections from the summary -> Every section in the package must appear in the section summaries.
- Recommendation says "looks good" when proof gaps exist on critical benefits -> Be honest. If a key benefit lacks proof, that needs human attention.
