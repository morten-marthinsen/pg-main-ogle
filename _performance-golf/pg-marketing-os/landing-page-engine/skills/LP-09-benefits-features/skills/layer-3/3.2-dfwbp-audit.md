# 3.2 — D-F-W-B-P Audit

**Skill:** LP-09-benefits-features
**Position:** Layer 3, Microskill 2
**Purpose:** Systematic completeness check on every D-F-W-B-P chain across all benefit section copy. This is the structural verification that ensures no element was shipped without all five components. Audit Point 1 in the 10-point rubric (3.1) checks this at a high level — this microskill does the detailed, element-by-element verification.
**Effort Level:** medium
**Dependencies:** All Layer 2 outputs, `dfwbp-structure-plan.md` (1.2)
**Output:** `dfwbp-audit.md`

---

## Input Schema

```yaml
input:
  dfwbp_structure_plan: "[from 1.2 -- planned chains for every element]"
  layer_2_outputs:
    benefit_bullets: "[benefit-bullets.md]"
    ingredient_deep_dives: "[ingredient-deep-dives.md]"
    feature_explanations: "[feature-explanations.md]"
    how_it_works: "[how-it-works.md]"
    comparison_benefits: "[comparison-benefits.md]"
```

---

## Execution Logic

1. **Load the D-F-W-B-P structure plan** from 1.2. This is the reference — every element planned in 1.2 must have a corresponding D-F-W-B-P chain in the Layer 2 output.

2. **For each planned element**, locate its D-F-W-B-P chain in the Layer 2 output and verify:

   **D (Deliverable):** Is the deliverable stated? Is it specific? Does it match the plan?
   - Present and specific = PASS
   - Missing or vague = FAIL

   **F (Feature):** Is the technical feature stated? Is it accurate per the brief?
   - Present, specific, matches brief data = PASS
   - Missing, vague, or contradicts brief = FAIL

   **W (Why):** Is the bridge present? Does it explain HOW the feature creates the benefit?
   - Present with mechanism/logic explanation = PASS
   - Missing, or just restates the benefit without explaining why = FAIL
   - "Because it's effective" or similar tautology = FAIL

   **B (Benefit):** Is the benefit in customer language? Is it specific and picturable?
   - Present, specific, in audience language = PASS
   - Missing, vague ("feel better"), or in clinical/technical language = FAIL

   **P (Proof):** Is evidence provided? Is it specific?
   - Named source + year + number = PASS
   - PROOF_GAP explicitly flagged = PASS (documented gap)
   - "Studies show" or "research proves" = FAIL (vague proof)
   - Missing entirely with no PROOF_GAP flag = FAIL (silent omission)

3. **Count results:**
   - Total elements audited
   - Complete chains (all 5 PASS)
   - Incomplete chains (any FAIL)
   - PROOF_GAP flags (documented gaps)
   - Missing elements (planned in 1.2 but not in Layer 2 output)

4. **Determine audit result:**
   - 100% of chains complete (with PROOF_GAP counting as documented) = PASS
   - Any chain with a FAIL on any component = FAIL with specific revision list

## Quality Criteria
- Every element from the 1.2 plan is accounted for (either found in output or flagged as missing)
- Each of the 5 components checked independently (not just "chain looks good")
- PROOF_GAP distinguished from missing proof (flagged gap = documented; missing proof = oversight)
- Incomplete chains have specific revision instructions per missing component
- Audit is verifiable — another agent could re-run this audit and get the same result

## Failure Modes
- Marking a vague Why as PASS ("because it's good for you" counted as a bridge) -> The Why must explain the mechanism or logic. Tautologies and platitudes fail.
- Marking "studies show" as PASS for Proof -> This is a Category 3 forbidden phrase. It fails the specificity check.
- Not catching missing elements -> Cross-reference the 1.2 plan element-by-element. Every planned element must appear.
- Counting PROOF_GAP as FAIL -> PROOF_GAP is an explicitly documented gap. It counts as "present" because the gap is identified and escalated. Silent omission (no proof and no flag) is what fails.
