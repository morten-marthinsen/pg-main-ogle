# 0.1 — Brief + Data Loader

**Skill:** LP-09-benefits-features
**Position:** Layer 0, Microskill 1
**Purpose:** Load and verify the upstream package files that LP-09 depends on: `page-brief.json` (LP-00), `conversion-intelligence.json` (LP-01), and `hero-section-package.json` (LP-07). Extract key fields that govern all downstream generation decisions.
**Effort Level:** low
**Dependencies:** LP-00 output (`page-brief.json`), LP-01 output (`conversion-intelligence.json`), LP-07 output (`hero-section-package.json`)
**Output:** `packages-loaded.md`

---

## Input Schema

```yaml
input:
  project_directory: "[path to active project output directory]"
  expected_files:
    - "page-brief.json"
    - "conversion-intelligence.json"
    - "hero-section-package.json"
  optional_files:
    - "proof-architecture.json"
```

---

## Execution Logic

1. **Locate each upstream file** in the project output directory. Check for existence of all three required files and the optional proof-architecture.json.

2. **If `page-brief.json` is MISSING:** HALT immediately. LP-09 cannot proceed without product data and audience profile.

3. **If `hero-section-package.json` is MISSING:** FLAG as warning but continue. Tone threading will use page-brief.json voice direction as fallback.

4. **Load `page-brief.json`** and extract:
   - `page_type` (type_a | type_b | hybrid)
   - `vertical` (health, finance, info_product, personal_dev, technology, etc.)
   - `awareness_stage` (1-5)
   - `audience_profile` (primary pain, failed solutions, primary desire, demographics)
   - `voice_direction` (tone, register, anti-voice)
   - `product_data`:
     - `product_name`
     - `ingredients` (if supplement/health — name, dose, form for each)
     - `features` (if SaaS/info — name, description for each)
     - `deliverables` (what the customer actually receives)
     - `mechanism_name` (if available)
   - `offer_summary` (price point, guarantee duration)
   - `copywriting_engine_mode` (downstream | standalone)

5. **Load `conversion-intelligence.json`** (LP-01) and extract:
   - `benchmark_conversion_rate` (for this vertical + page type)
   - `high_impact_elements` (which elements drive the most conversion lift)
   - `benefit_section_benchmarks` (avg word count, bullet count, proof density norms for vertical)

6. **Load `hero-section-package.json`** (LP-07) and extract:
   - `threading_anchors.primary_hook_phrase`
   - `threading_anchors.promise_statement`
   - `threading_anchors.audience_qualifier`
   - `hero_copy.headline.text` (for tone reference)
   - `validation.hero_audit_score` (for quality context)

7. **Load `proof-architecture.json`** (LP-05, optional) and extract:
   - `proof_assets` (clinical studies, statistics, testimonials available for embedding)
   - `proof_density_tier` (low | medium | high | extreme)
   - `proof_placement_plan` (which proof goes in which section type)

8. **Write `packages-loaded.md`** with all extracted fields organized by source.

## Quality Criteria
- page-brief.json loaded with product data extracted (ingredients, features, or deliverables)
- Hero threading anchor extracted for tone governance (or fallback documented)
- Proof assets identified from proof-architecture.json or flagged as "source from brief"
- Output is self-contained — downstream microskills can work from this file alone

## Failure Modes
- page-brief.json missing -> HALT. Cannot proceed without product data.
- Product data section empty (no ingredients, no features, no deliverables) -> HALT with message requesting product data.
- hero-section-package.json missing -> WARNING (continue with brief voice direction as tone fallback). Document the fallback.
- Ingredient doses missing from brief -> Extract what is available, flag missing doses as `needs_data` for each ingredient.
