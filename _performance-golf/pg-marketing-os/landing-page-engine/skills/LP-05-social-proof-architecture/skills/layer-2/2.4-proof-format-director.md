# 2.4 — Proof Format Director

**Skill:** LP-05-social-proof-architecture
**Position:** Layer 2, Microskill 4
**Purpose:** Specify the visual/structural format for every proof element. Format determines how proof LOOKS on the page — text block, photo card, video embed, widget, stat bar, cascading review feed, badge, or certificate. LP-10 and LP-15 need exact format direction to produce the right output.
**Effort Level:** medium
**Dependencies:** `proof-density-map.md` (2.2), `proof-type-sequence.md` (2.3), `specimen-proof-patterns.md` (0.3)
**Output:** `proof-format-direction.md`

---

## Format Definitions

| Format Code | Visual Description | Best For | LP-10 Output |
|------------|-------------------|----------|-------------|
| `text_block` | Quoted testimonial text with name, location, outcome below. No image. | Text testimonials, embedded proof in narrative sections | Formatted testimonial text |
| `photo_card` | Customer photo (left or top) + testimonial text + name/location/outcome | Photo testimonials, proof block entries | Photo placeholder + testimonial text |
| `video_embed` | Video player embed with pull-quote text overlay or below | Video testimonials, high-engagement proof blocks | Pull-quote text + video embed placeholder |
| `widget` | Third-party review widget (Trustpilot, Judge.me, Google Reviews) | Above-fold rating strips, trust bars | Widget integration spec |
| `stat_bar` | 3-number horizontal strip with icons ("50K+ Customers / 4.8 Stars / 2,847 Reviews") | Above fold, social proof strips | Stat bar copy with numbers |
| `cascade` | Scrollable review feed with star ratings, filtering, load-more | Type B review sections | Review template spec + sample reviews |
| `badge` | Visual badge/icon with label (GMP Certified, Inc. 5000) | Trust bar, guarantee proximity | Badge label text |
| `certificate` | Visual guarantee certificate with branded name, duration, terms | Guarantee section | Certificate copy |
| `comparison` | Table with checkmarks (us) and X marks (competitors) | Comparison table section | Comparison table content |
| `before_after` | Side-by-side layout with "Before" / "After" labels, timeframe, outcome | Before/after gallery, proof blocks | Before/after pairs with copy |
| `inline_mention` | Brief credibility mention woven into body copy (no visual block) | Story, root cause, FAQ answers | Sentence-level copy direction |
| `pull_quote` | Highlighted quote in larger/styled text, attributed below | Product intro authority, key testimonial highlight | Pull-quote text + attribution |

## Format Selection Rules

### By Section Type (Type A)

| Section | Recommended Formats | Avoid |
|---------|-------------------|-------|
| Above fold / Trust bar | `badge`, `widget`, `stat_bar` | `text_block` (too dense for above fold) |
| Lead / Hook | `inline_mention` | `photo_card` (breaks reading flow) |
| Story | `inline_mention` | All standalone formats (story is narrative) |
| Root cause | `inline_mention` | `cascade` (wrong section for review volume) |
| Mechanism | `inline_mention`, `pull_quote` | `video_embed` (distracts from mechanism explanation) |
| Proof Block 1 | `photo_card`, `text_block`, `pull_quote` | `cascade` (too early for volume proof) |
| Product Introduction | `pull_quote` | `cascade` |
| Ingredient deep-dives | `inline_mention`, `text_block` | `video_embed` per ingredient (too heavy) |
| Proof Block 2 | `photo_card`, `video_embed`, `text_block`, `before_after` | `inline_mention` (proof blocks are standalone) |
| Value stack / Offer | `text_block`, `pull_quote` | `cascade` |
| Guarantee | `certificate`, `badge` | `text_block` |
| Proof Block 3 | `before_after`, `photo_card`, `stat_bar` | `inline_mention` |
| FAQ | `inline_mention` | All standalone formats |
| Close / P.S. | `inline_mention`, `stat_bar` | `cascade` |

### By Section Type (Type B)

| Section | Recommended Formats | Avoid |
|---------|-------------------|-------|
| Above fold | `widget`, `stat_bar` | `text_block` |
| Key benefits | `badge` | `cascade` |
| Trust bar | `badge` | `text_block` |
| How it works | None (no proof) | All formats |
| Ingredient deep-dives | `inline_mention` | `cascade` |
| Social proof strip | `stat_bar` | `text_block` |
| Before/after gallery | `before_after` | `inline_mention` |
| Review cascade | `cascade` | `pull_quote` (cascade IS the format) |
| FAQ | `inline_mention` | All standalone formats |
| Guarantee | `certificate`, `badge` | `cascade` |
| Comparison table | `comparison` | All other formats |

## Execution Logic

**Step 1:** Load the density map (2.2) and type sequence (2.3).

**Step 2:** For each proof element in each section:
a) Check the section type against the format selection rules
b) Select the appropriate format code
c) Write a format direction specific enough for LP-10 to execute:
   - For `photo_card`: Specify photo position (left/top), approximate testimonial length (1-3 sentences), required info (name, location, outcome)
   - For `cascade`: Specify number of visible reviews, filter categories, sort order
   - For `stat_bar`: Specify exact 3 numbers to display and phrasing
   - For `inline_mention`: Specify where in the section text the mention should appear (opening, mid, closing)
   - For `certificate`: Specify guarantee name, duration, visual style direction

**Step 3:** Cross-reference with specimen format patterns (0.3). If a specimen in the same vertical uses a proven format for a similar proof type, note the specimen reference.

**Step 4:** Verify format variety — avoid using the same format for 5+ consecutive proof elements. Visual monotony is as damaging as proof type monotony.

## Quality Criteria
- Every proof element has a format code AND a specific execution direction
- Format selection follows the rules table (no cascade in FAQ, no inline_mention in proof blocks)
- Execution directions are specific enough for LP-10 (not "add a testimonial" but "photo card with headshot left, 2 sentences, specific outcome + timeframe, name + city")
- Format variety maintained (no 5+ consecutive same format)
- Specimen references included where applicable

## Failure Modes
- No format specified for an element -> HALT. Every element needs a format.
- Same format for 5+ consecutive elements -> Introduce format variation (swap a text_block for a pull_quote, or a photo_card for a video_embed)
- LP-10 execution direction is vague -> Rewrite with layout position, content length, required information fields
