# 1.4 — Proof Gap Analyzer

**Skill:** LP-05-social-proof-architecture
**Position:** Layer 1, Microskill 4
**Purpose:** Compare what proof the architecture REQUIRES against what proof actually EXISTS. Identify every gap by type, quantify severity, and provide specific sourcing recommendations. This is the document that tells the client or team exactly what proof assets must be gathered before launch.
**Effort Level:** medium
**Dependencies:** `proof-classification.md` (1.1), `volume-requirements.md` (1.2), `wave-pattern-selection.md` (1.3)
**Output:** `proof-gap-analysis.md`

---

## Input Schema

```yaml
input:
  proof_classification: "[from 1.1 — classified assets with counts per type]"
  volume_requirements: "[from 1.2 — required vs. available per type]"
  wave_pattern: "[from 1.3 — selected wave with zone assignments]"
  proof_mode: "[PROOF_SATURATION | AUTHORITY_SUBSTITUTION]"
  page_type: "[type_a | type_b | hybrid]"
```

---

## Gap Severity Classification

| Severity | Definition | Impact |
|----------|-----------|--------|
| **Critical** | Proof type is required for the selected wave pattern AND zero assets available | Architecture cannot be executed. Launch blocked until sourced. |
| **High** | Proof type is required AND available count < 50% of minimum threshold | Architecture can be partially executed but conversion impact is significant. |
| **Medium** | Proof type is recommended AND available count < recommended threshold but above minimum | Architecture executable but suboptimal. Source before launch if possible. |
| **Low** | Proof type is optional AND not available | Minimal conversion impact. Source opportunistically. |

---

## Execution Logic

**Step 1:** For each proof type in the hierarchy (ranks 1-6 + additional types), calculate:
- Required count (from volume requirements)
- Available count (from classification)
- Gap = required - available
- If gap > 0: assign severity based on classification above

**Step 2:** Identify CRITICAL gaps — these block the architecture:
- Type A: zero testimonials (need 15 minimum)
- Type A: zero studies AND health vertical (studies are rank 1 for health)
- Type B: fewer than 10 reviews (need 50 minimum; cannot even display rating strip below 10)
- Any page type: zero proof of any kind (cannot execute any wave pattern)

**Step 3:** For each gap, generate a specific sourcing recommendation:
- WHO should be asked (existing customers, medical advisory board, PR team)
- WHAT specifically to request (testimonial with specific outcome, study citation, endorsement quote)
- HOW MANY needed to close the gap
- PRIORITY of this gap relative to others

**Step 4:** Assess overall gap severity:
- `no_gaps`: All proof types at or above recommended thresholds
- `minor_gaps`: Some types below recommended but all above minimum
- `significant_gaps`: One or more types below minimum threshold
- `critical_gaps`: One or more critical gaps that block architecture execution

**Step 5:** If AUTHORITY_SUBSTITUTION mode was selected, verify that the authority-type proof (expert endorsements, performance guarantees, celebrity authority, media mentions) is actually available. If the mode was selected because testimonials are scarce but authority proof is ALSO scarce — flag as `architecture_blocked`.

---

## Quality Criteria
- Every proof type from the hierarchy checked against requirements
- Severity assigned based on defined criteria (not guessed)
- Sourcing recommendations are specific (who, what, how many)
- Critical gaps clearly separated from lower-severity gaps
- Overall severity assessment is honest

## Failure Modes
- All proof types at zero -> Output shows `critical_gaps` for every type. Assessment: "Architecture blocked — no proof assets available. Cannot proceed to Layer 2 until minimum proof is sourced. Recommend collecting at least 5 customer testimonials and 1 expert endorsement before continuing."
- AUTHORITY_SUBSTITUTION mode but zero authority proof -> "WARNING: No testimonials AND no authority proof. Proof architecture cannot be executed in either mode. Escalate to human."
