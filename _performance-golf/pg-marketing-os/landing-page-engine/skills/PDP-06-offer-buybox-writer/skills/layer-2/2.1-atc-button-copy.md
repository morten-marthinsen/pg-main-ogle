# 2.1 — ATC Button Copy

**Skill:** PDP-06-offer-buybox-writer
**Position:** Layer 2, Microskill 1
**Purpose:** Write the primary "Add to Cart" button copy across all five states: default, loading, success, out-of-stock, and disabled (no variant selected). The default label MUST be "Add to Cart" -- standard, not creative. This is the single most important conversion element on the PDP.
**Effort Level:** medium
**Dependencies:** 0.3 blueprint-extraction.md (ATC button spec from PDP-03), 0.2 pdp-reference-load.md (ATC standards), 1.2 variant-copy.md (variant type for disabled state)
**Output:** `atc-copy.md`

---

## Input Schema

```yaml
input:
  blueprint_extraction: "[path to blueprint-extraction.md]"
  pdp_reference_load: "[path to pdp-reference-load.md]"
  variant_copy: "[path to variant-copy.md or 'no variants']"
```

## Execution Logic

**Step 1:** Read `blueprint-extraction.md`. Extract PDP-03's ATC button spec:
- Full-width confirmation
- State change requirements
- Post-click behavior (slide-out mini-cart)

**Step 2:** Read `pdp-reference-load.md`. Confirm ATC label standards from Baymard.

**Step 3:** Read `variant-copy.md` (if exists). Extract the variant type label for the disabled state copy ("Select a Flavor" vs "Select a Size").

**Step 4:** Write the default state label:
```
Label: "Add to Cart"
```
- This is NON-NEGOTIABLE. Not "Buy Now". Not "Get Yours". Not "Add to Bag". Not "Yes, I Want This!". Just "Add to Cart."
- Baymard: standard labels require zero cognitive processing. Users tap "Add to Cart" on autopilot.

**Step 5:** Write the loading state label:
```
Label: "Adding..."
```
- Brief loading state while cart processes
- Shows a spinner icon (UX) alongside text

**Step 6:** Write the success state label:
```
Label: "Added!"
```
- Checkmark icon (UX) alongside text
- Auto-dismiss after 1.5s, then triggers slide-out mini-cart
- Button returns to default state after mini-cart appears

**Step 7:** Write the out-of-stock state label:
```
Label: "Sold Out -- Notify Me"
```
- Converts an out-of-stock visit into a lead capture
- Button click opens email notification signup (not a dead end)
- Button style changes: muted color, no accent

**Step 8:** Write the disabled state label (no variant selected):
```
Label: "Select a [Variant Type]"
```
- [Variant Type] = the actual variant type from variant-copy.md (e.g., "Flavor", "Size", "Color")
- If no variants exist, this state does not apply
- Button appears inactive (UX: grayed out, not clickable)
- Guides user to make the required selection before adding to cart

**Step 9:** Write the accessibility label:
```
aria-label: "Add [Product Name] to your shopping cart"
```

**Step 10:** Validate all labels against ATC standards:
- Default = "Add to Cart": Y/N
- Loading = "Adding...": Y/N
- Success = "Added!": Y/N
- OOS = "Sold Out -- Notify Me": Y/N
- Disabled = "Select a [type]": Y/N
- No creative labels in any state: Y/N

**Step 11:** Write `atc-copy.md`.

---

## Quality Criteria
- Default label is exactly "Add to Cart" (case-sensitive, no variations)
- All 5 states have explicit label copy
- Disabled state uses the actual variant type name from variant-copy.md
- Post-click is slide-out mini-cart (not cart redirect)
- Accessibility label includes product name
- No creative, clever, or marketing-driven labels in any state
- File exists after execution

## Failure Modes
- Default label written as anything other than "Add to Cart" -> Fix immediately. This is the #1 forbidden behavior.
- No variant type available for disabled state -> If no variants, omit disabled state entirely
- PDP-03 specified cart redirect instead of mini-cart -> Override to mini-cart per Baymard. Flag inconsistency with PDP-03.
- PDP-03 blueprint does not include ATC button (impossible, but defensive) -> HALT, every buy box has an ATC button
