# 1.5 — Short Description Copy

**Skill:** PDP-06-offer-buybox-writer
**Position:** Layer 1, Microskill 5
**Purpose:** Write the 3-5 bullet-point short description that appears in the buy box. Each bullet must be benefit-forward (what the product does for the user), not feature-forward (what the product contains). This is the only prose copy in the buy box and the primary persuasion text above the fold.
**Effort Level:** high
**Dependencies:** 0.1 input-verification.md (product data, pain points, ingredients), 0.3 blueprint-extraction.md (short description spec from PDP-03)
**Output:** `short-description-copy.md`

---

## Execution Logic

**Step 1:** Read `input-verification.md`. Extract:
- Key ingredients/features list
- Primary pain points (audience)
- Purchase motivations
- Product category/vertical
- Any LP-01 verbatim quotes (for language grounding)

**Step 2:** Read `blueprint-extraction.md`. Extract PDP-03's short description spec:
- Number of bullets specified (3-5)
- Any content direction PDP-03 provided
- Bullet ordering guidance

**Step 3:** Read `pdp-reference-load.md`. Confirm short description standards:
- Benefit-forward (not feature-forward)
- Specific (not generic)
- Scannable (short bullets, not paragraphs)

**Step 4:** Map each key ingredient/feature to its primary BENEFIT:
```
Feature -> Benefit Mapping:
  "500mg Ashwagandha" -> "Supports calm, focused energy without the jitters"
  "Clinically studied dose" -> "Doctor-recommended strength backed by research"
  "3rd party tested" -> "Independently verified for purity and potency"
  "60 capsules" -> "Full 30-day supply in every bottle"
  "No artificial fillers" -> "Clean formula -- no artificial colors, flavors, or fillers"
```
- The benefit is WHAT IT DOES FOR THE USER
- The feature is WHAT IT IS
- Lead with the benefit. Feature can appear as supporting detail.

**Step 5:** Select the top 3-5 benefits based on:
1. Pain point alignment (does this benefit address a top audience pain point?)
2. Differentiation (does this benefit set the product apart from competitors?)
3. Specificity (is this benefit specific enough to be credible?)
4. Variety (no two bullets addressing the same benefit angle)

**Step 6:** Write each bullet:
- Start with the benefit verb or outcome (not the ingredient/feature)
- Keep to one line (10-15 words max per bullet)
- Use specific language, not generic claims
- Use checkmark or bullet prefix, not numbered list

**Step 7:** Validate each bullet against the anti-slop filter:
- No "revolutionary", "game-changing", "unlock", "transform", "journey"
- No "harness the power", "elevate", "supercharge"
- No vague superlatives ("premium quality", "best in class", "cutting-edge")

**Step 8:** Validate benefit-forward orientation:
- For each bullet, ask: "Does this lead with what the product DOES, not what it IS?"
- If any bullet leads with a feature name or ingredient dosage -> rewrite

**Step 9:** Write `short-description-copy.md`.

## Quality Criteria
- Exactly 3-5 bullets (not 2, not 6+)
- Every bullet leads with a benefit (what it does), not a feature (what it is)
- Each bullet is 10-15 words max (scannable)
- No two bullets address the same benefit angle
- Language is specific, not generic ("Supports deep, uninterrupted sleep" not "Improves sleep quality")
- Zero AI telltale words (revolutionary, game-changing, transform, unlock, journey, elevate)
- Bullets address documented audience pain points from the brief
- File exists after execution

## Failure Modes
- No ingredient/feature data in brief -> Write benefit bullets based on product category norms; flag as data gap for human review
- All bullets end up feature-forward despite rewriting -> Revisit audience pain points and rewrite from pain -> benefit, not ingredient -> benefit
- Fewer than 3 compelling benefits -> If product truly has fewer than 3 distinct benefits, write 3 bullets minimum using different angles on the same core benefits
- Anti-slop check fails -> Rewrite offending bullets with specific, grounded language
- PDP-03 blueprint does not include short description -> Skip this microskill; document in execution log
