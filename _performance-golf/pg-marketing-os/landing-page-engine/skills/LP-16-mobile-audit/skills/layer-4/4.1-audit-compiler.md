# 4.1 — Audit Compiler

> **Skill:** LP-16-mobile-audit
> **Layer:** 4 (Package Assembly)
> **Model:** haiku
> **Input:** All Layer 2 outputs (`viewport-analysis.md`, `content-density-audit.md`, `cta-mobile-audit.md`, `media-performance-audit.md`, `speed-estimation.md`) + Layer 3 outputs (`mobile-score.md`, `mobile-fixes.md`)
> **Output:** `mobile-audit.json`

---

## PURPOSE

Compile all audit data from Layer 2 and Layer 3 into the final `mobile-audit.json` output file. This is a mechanical assembly task — no judgment, no scoring, no re-evaluation. Read the data from intermediate files and structure it into the JSON schema defined in LP-16-AGENT.md.

---

## PROCEDURE

1. **Read** all 7 intermediate files (5 from Layer 2 + 2 from Layer 3).
2. **Map** data from intermediate files to the JSON schema fields.
3. **Validate** that all required fields are populated.
4. **Write** `mobile-audit.json` to the project output directory.

### Field Mapping

| JSON Field | Source File | Source Section |
|------------|-----------|---------------|
| `schema_version` | Static: "1.0" | — |
| `created` | Generate ISO timestamp | — |
| `project_name` | `brief-load.md` or `mobile-score.md` | Page header |
| `page_type` | `mobile-criteria.md` | Page Type field |
| `primary_device_context` | `device-context.md` | Primary Device Context |
| `mobile_readiness_score` | `mobile-score.md` | Mobile Readiness Score |
| `verdict` | `mobile-score.md` | Verdict |
| `dimension_scores.viewport_fit` | `viewport-analysis.md` + `mobile-score.md` | Dimension scores + findings |
| `dimension_scores.content_density` | `content-density-audit.md` + `mobile-score.md` | Dimension scores + findings |
| `dimension_scores.cta_accessibility` | `cta-mobile-audit.md` + `mobile-score.md` | Dimension scores + findings |
| `dimension_scores.media_optimization` | `media-performance-audit.md` + `mobile-score.md` | Dimension scores + findings |
| `dimension_scores.page_speed` | `speed-estimation.md` + `mobile-score.md` | Dimension scores + speed data |
| `dimension_scores.typography` | `viewport-analysis.md` + `mobile-score.md` | Typography assessment |
| `dimension_scores.navigation` | Multiple sources + `mobile-score.md` | Navigation synthesis |
| `priority_fixes` | `mobile-fixes.md` | Ranked Fix List |
| `speed_inventory` | `speed-estimation.md` | Content Inventory + Weight Estimation |
| `metadata` | All source paths | File paths used |

### Validation Checklist

Before writing the file, verify:

- [ ] `mobile_readiness_score` is a number between 0.0 and 10.0
- [ ] `verdict` is one of: MOBILE_READY, MINOR_MOBILE_FIXES, SIGNIFICANT_MOBILE_GAPS, MOBILE_REBUILD
- [ ] All 7 dimensions have score, weight, weighted_score, and findings array
- [ ] All dimension weights sum to 1.00
- [ ] `priority_fixes` is sorted by priority_score descending
- [ ] `speed_inventory` has all component counts
- [ ] Every finding with status FLAG or FAIL has non-null fix and expected_impact
- [ ] `metadata` includes all source file paths

---

## OUTPUT

Write `mobile-audit.json` following the exact schema from LP-16-AGENT.md Output Schema section.

---

## CRITICAL RULES

1. **Mechanical assembly only.** Do not change any scores, fix recommendations, or findings. Compile exactly what the intermediate files contain.
2. **Schema compliance is mandatory.** The output must match the JSON schema in LP-16-AGENT.md exactly. No extra fields, no missing fields.
3. **Validate before writing.** Run through the validation checklist before writing the file. A malformed JSON file is worse than no file.
4. **If data is missing from intermediate files:** Note it in the findings as "DATA_MISSING — [source file] did not contain [expected field]." Do NOT fabricate data to fill the gap.
