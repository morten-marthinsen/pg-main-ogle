# 0.2 — Brief Loader

**Skill:** LP-00-brief-classifier
**Position:** Layer 0, Microskill 2
**Purpose:** Load and parse the product brief document. Extract structural metadata (product name, format, word count, sections present/missing) so downstream microskills can operate on confirmed brief content rather than raw input.
**Effort Level:** low
**Dependencies:** `mode-declaration.md` (0.1 output), product brief path from user
**Output:** `brief-load.md`

---

## Input Schema

```yaml
input:
  brief_path: "[path to product brief document]"
  product_name: "[product name from user trigger or mode-declaration.md]"
  operating_mode: "[downstream | standalone — from mode-declaration.md]"
```

---

## Execution Logic

**Step 1:** Locate the product brief document. Check in order:
```
Priority 1: Path provided directly by user
Priority 2: ./outputs/[product-name]/brief.*
Priority 3: ./clients/[client-name]/[project]/brief.*
Priority 4: Any .md or .txt file in project folder containing "brief" in filename
```

**Step 2:** Load the brief content. Record:
- File path (absolute)
- File format (md, txt, pdf, docx)
- Word count
- Character count

**Step 3:** Scan for standard brief sections and mark each as `present` or `missing`:
- Product name / description
- Target audience / demographics
- Price point / pricing structure
- Offer details (bonuses, guarantee)
- Traffic source / campaign context
- Competitive landscape / positioning
- Claims / benefits / mechanism
- Voice / tone direction
- Visual assets / brand guidelines

**Step 4:** Extract the product name explicitly. If multiple names appear (brand name vs. product name), record both.

**Step 5:** Flag any critical gaps. A brief missing BOTH audience AND pricing is flagged as `thin_brief` — downstream microskills will rely heavily on inference and must flag outputs as `brief_inferred`.

---

## Quality Criteria
- Brief file located and loaded successfully
- All 9 section categories scanned — none skipped
- Word count and format recorded accurately
- Product name extracted (not left as placeholder)
- Brief quality classification assigned (comprehensive / adequate / thin_brief)
- Full brief content included in output for downstream access

## Failure Modes
- No brief file found at any priority path → HALT with message: "No product brief found. Provide a brief document path."
- Brief file exists but is empty or unreadable → HALT with message: "Brief file found but empty/unreadable at [path]."
- Brief is extremely short (<100 words) → Flag as `thin_brief`, proceed with warning
