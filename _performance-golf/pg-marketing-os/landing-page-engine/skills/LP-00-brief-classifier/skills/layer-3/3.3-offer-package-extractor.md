# 3.3 — Offer Package Extractor

**Skill:** LP-00-brief-classifier
**Position:** Layer 3, Microskill 3
**Purpose:** Pull full offer details from the CopywritingEngine offer package (Skill 07) and enrich/overwrite the Layer 2 offer summary (from 2.2) with validated data. Where the offer package provides confirmed pricing, bonuses, or guarantee details, those replace brief-inferred values. This microskill ONLY executes in Downstream Mode.
**Effort Level:** medium
**Dependencies:** `package-inventory.md` (0.3 — confirms offer package exists), `offer-summary.md` (2.2 — Layer 2 baseline), CopywritingEngine offer output files
**Output:** `offer-extract.md`

---

## Input Schema

```yaml
input:
  operating_mode: "downstream"  # This microskill ONLY runs in Downstream Mode
  offer_package_path: "[path to 07-offer/outputs/offer-package.json or: not_found]"
  campaign_brief_path: "[path to 09-campaign-brief/outputs/campaign-brief.json or: not_found]"
  layer_2_offer_summary: "[from offer-summary.md — the baseline to enrich]"
```

---

## Execution Logic

**Step 1: Confirm Downstream Mode.**
If operating mode is `standalone`, skip this microskill entirely. Log: "3.3 skipped — Standalone Mode. No offer package to extract."

**Step 2: Load the offer package.**
Primary source: `07-offer/outputs/offer-package.json`
Fallback source: `09-campaign-brief/outputs/campaign-brief.json` (offer section)

If neither exists, log: "3.3 — no offer package found. Layer 2 offer-summary.md remains as baseline (brief_inferred)."

**Step 3: Extract validated offer details.**
For each field in the offer summary schema, compare the offer package value against the Layer 2 baseline:

- **Product name:** Confirm or correct
- **Pricing:** Extract exact prices for all tiers (single, multi-pack, subscription, trial)
- **Bonuses:** Extract complete bonus list with values and descriptions
- **Guarantee:** Extract exact duration, type, and branded name
- **Urgency levers:** Extract any scarcity/deadline mechanisms from the offer architecture

**Step 4: Enrichment merge.**
For each field:
- If offer package provides data → Use it, mark as `offer_package_validated`
- If offer package is silent on a field → Retain Layer 2 value, keep original source tag
- If offer package contradicts Layer 2 → Use offer package (it's validated), note the correction

**Step 5: Identify offer architecture details** not captured in Layer 2:
- Offer stack structure (what's bundled at each tier)
- Pricing psychology cues (charm pricing, anchoring, strikethrough)
- Guarantee framing language (how the guarantee is presented in copy)
- Order form configuration hints (if present)

**Step 6: Tag the enriched offer as `offer_package_validated` for fields that came from the offer package. Fields still from brief remain `brief_inferred`.**

## Quality Criteria
- Every field populated or marked `not_available` — zero empty fields
- Source tags accurate: `offer_package_validated` only for fields that came from the offer package
- Layer 2 values retained when offer package is silent (not overwritten with empty)
- Enrichment delta accurately shows what changed between Layer 2 and Layer 3
- Pricing extracted with exact dollar amounts — no rounding or approximation

## Failure Modes
- Offer package not found → Log skip, Layer 2 offer-summary.md remains as baseline
- Offer package JSON is malformed → Log parse error, extract what's readable, note failures
- Offer package conflicts with brief on pricing → Use offer package value (validated source), note the discrepancy
- Offer package missing key fields (no guarantee mentioned) → Retain Layer 2 value for that field, keep `brief_inferred` tag
