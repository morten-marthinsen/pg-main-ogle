# 1.2 — Hybrid Detector

**Skill:** LP-00-brief-classifier
**Position:** Layer 1, Microskill 2
**Purpose:** Test for Hybrid conditions before final type declaration. Hybrid is a valid first-class type that must be explicitly detected — it cannot be left to chance.
**Effort Level:** low
**Dependencies:** `classification-signals.md` (1.1 output)
**Output:** `hybrid-test.md`

---

## Input Schema

```yaml
input:
  type_a_signal_count: "[from classification-signals.md]"
  type_b_signal_count: "[from classification-signals.md]"
  brief_summary: "[key product and market facts]"
```

---

## Hybrid Trigger Conditions

Test ALL four conditions. If ANY is true, the result is `hybrid_detected`.

**Condition 1: Signal Balance**
`type_a_signals >= 3 AND type_b_signals >= 3`

**Condition 2: Explicit Brief Language**
Brief uses any of these phrases or close variants:
- "extended PDP"
- "long-form product page"
- "below the fold content"
- "hybrid landing page"
- "ecomm page with story"
- "sales page with buy button above fold"

**Condition 3: Known Hybrid Pattern Examples**
Product closely resembles one of these named hybrid archetypes:
- Magnesium Breakthrough model (ecomm above fold + extensive below-fold education)
- Kachava model (DTC ecomm + long educational content module)
- Earth Echo / Cacao Bliss model (DTC + story + mechanism below fold)
- Happy Mammoth model (ecomm + extended below-fold content)
- Athletic Greens / AG1 model (DTC + editorial content)

**Condition 4: Price + Traffic Mismatch**
Product is priced in Type A territory ($50–$150+ single unit) BUT primary traffic source is warm/hot (organic search, existing email list, brand awareness). This usually produces a hybrid — strong above-fold for ready-to-buy + educational content below for skeptics.

## Output Schema

`hybrid-test.md`:

```markdown
# Hybrid Detection Test — [Product Name]

## Condition Tests

### Condition 1: Signal Balance
- Type A signals: [count] / Type B signals: [count]
- Balance test (3+ each): [TRIGGERED | not triggered]

### Condition 2: Explicit Brief Language
- Hybrid language found: [Y/N]
- Quote: "[relevant phrase if found]"
- Test: [TRIGGERED | not triggered]

### Condition 3: Known Hybrid Archetype
- Closest archetype match: [name | none]
- Match strength: [strong | moderate | weak | none]
- Test: [TRIGGERED | not triggered]

### Condition 4: Price + Traffic Mismatch
- Price point: $[X] — [Type A territory | Type B territory | neutral]
- Traffic source: [warm/hot | cold | unknown]
- Mismatch: [Y/N]
- Test: [TRIGGERED | not triggered]

## Hybrid Detection Result
[HYBRID DETECTED — proceed to hybrid type declaration]
[NOT HYBRID — proceed to type_a or type_b declaration per signal tally]

## Hybrid Zone Split (if HYBRID DETECTED)
[Proposed split, e.g.: "Zone 1 (Sections 1–3): Type B above-fold architecture. Zone 2 (Sections 4+): Type A educational content."]
[or: N/A]
```

---

## Quality Criteria
- All 4 conditions explicitly tested (not just checked mentally)
- If hybrid detected, zone split proposed
- Result is one of: `hybrid_detected` or `not_hybrid`

## Failure Modes
- Brief has no traffic source information → Condition 4 defaults to "unknown" — not triggered
- Product resembles hybrid archetype but brief says otherwise → Flag as note, defer to explicit brief language
