# 2.8 — Conversion Flow Audit

> **Skill:** PDP-16-mobile-conversion-audit
> **Layer:** 2 (Audit Execution)
> **Position:** 8 of 8
> **Model:** sonnet
> **Purpose:** Audit the conversion flow from ATC through checkout: cart path, express payments, error states, and dead ends. A PDP can have perfect copy, perfect UX, and perfect social proof, but if the conversion flow has a dead end or an unhandled error state, the user bounces at the moment of maximum intent.
> **Dependencies:** `assembled-page-load.md`, `pdp-reference-load.md`, `pdp-audit-criteria.md`
> **Output:** `conversion-flow-audit.md`

## Execution Logic

**POSITIONAL REINFORCEMENT:** You are auditing the conversion flow — the path from "Add to Cart" tap to checkout completion. You are NOT auditing the ATC button design (2.2 covers that) or the sticky ATC (2.6 covers that). Focus on what happens AFTER the user taps ATC and the path to purchase completion.

**Step 1:** From `assembled-page-load.md`, extract conversion flow specifications:
- Post-ATC behavior (mini-cart content and actions)
- Mini-cart to checkout path
- Express payment specification
- Error state handling (OOS, quantity limits, payment failures)
- Cross-sell/upsell in mini-cart (if specified)

**Step 2:** Evaluate each criterion:

### Criterion 8.1: Cart Path Specified
- **Conversion flow principle:** ATC -> mini-cart -> checkout must be a continuous, frictionless path.
- **PASS:** Post-ATC flow specified: (1) ATC tap -> slide-out mini-cart with item summary, quantity adjuster, subtotal, "Proceed to Checkout" button. (2) "Proceed to Checkout" goes to checkout page. (3) No dead ends — every screen has a clear next action.
- **FAIL:** Post-ATC flow not specified, mini-cart content not defined, or path to checkout unclear.
- **Evidence required:** Trace the complete path from ATC tap to checkout entry.

### Criterion 8.2: Express Payments
- **Baymard (6.6):** Express payments should be below ATC with "OR" divider, one row of logos.
- **PASS:** Express payment specification: positioned below main ATC, OR divider between ATC and express options, one row of payment method logos (Apple Pay, Google Pay, Shop Pay, etc.).
- **FAIL:** Express payments above ATC, no divider, multiple rows, or not specified.
- **Evidence required:** Quote express payment layout specification.

### Criterion 8.3: Error State Handling
- **UX principle:** Every error the user can encounter must have a clear, helpful response.
- **PASS:** Assembled page specifies handling for at least 3 common error states: (1) Out-of-stock variant selected -> visual indication + alternative suggestion, (2) Quantity exceeds available stock -> error message + available quantity shown, (3) ATC while no variant selected (if multi-variant) -> prompt to select variant.
- **FAIL:** Fewer than 3 error states addressed, or error handling not specified.
- **Evidence required:** List error states addressed in assembled page.

### Criterion 8.4: No Dead Ends
- **Conversion flow principle:** Every page state must have a clear next action. A user should never reach a state where the only option is to hit the browser back button.
- **PASS:** All specifiable page states have next actions: (1) Main page -> ATC or scroll, (2) Mini-cart -> checkout or continue shopping, (3) Gallery overlay -> close to return to page, (4) Empty cart state -> "Continue Shopping" link.
- **FAIL:** Any page state without a specified next action.
- **Evidence required:** Map all page states and their exit paths.

**Step 3:** Write finding for each criterion.

**Step 4:** Write `conversion-flow-audit.md`.

## Quality Criteria

- All 4 criteria evaluated
- Cart path traced from ATC through checkout (not just "mini-cart exists")
- Error states evaluated for HELPFULNESS, not just presence (an error message that says "Error" is not helpful)
- Dead ends mapped across all page states, not just the happy path
- UNSPECIFIED = FAIL
- File exists after execution

## Failure Modes

- Conversion flow not specified in assembled page -> Score all 4 as FAIL. Note that LP-15 assembly should include post-ATC flow specification from PDP-06.
- Platform handles conversion flow (Shopify, etc.) -> Still FAIL if not specified. The audit evaluates what was SPECIFIED, not what the platform might do by default. Relying on platform defaults without explicit specification is a risk.
- Mini-cart specified but checkout path not -> FAIL on 8.1. The flow is incomplete.
