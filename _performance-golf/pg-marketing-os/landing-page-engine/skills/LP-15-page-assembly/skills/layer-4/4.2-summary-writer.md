# 4.2 — Summary Writer

> **Skill:** LP-15-page-assembly
> **Layer:** 4
> **Position:** 2 of 3
> **Purpose:** Write ASSEMBLY-SUMMARY.md — a human-readable overview of the assembled page, including section map, Arena verification status, threading summary, and quality scores. This is the file the human reviewer reads to understand what LP-15 produced.
> **Effort Level:** sonnet
> **Dependencies:** `assembled-page-package.json`, `assembly-audit.md`
> **Output:** `ASSEMBLY-SUMMARY.md`

---

## INPUT

```yaml
input:
  assembled_package: "assembled-page-package.json"
  assembly_audit: "assembly-audit.md"
  arena_verification: "arena-copy-verification.md"
  threading_validation: "threading-validation.md"
  completeness_audit: "completeness-audit.md"
  brief_load: "brief-load.md"
```

---

## EXECUTION

### Step 1: Extract Key Data

From the assembled page package and audit files, extract:
- Project name, page type, vertical
- Assembly mode (full/partial)
- Total sections, total word count
- Section map with source skills
- Arena verification results
- Threading summary
- Quality scores
- Missing elements (if any)

### Step 2: Write Summary Following Template

Follow the ASSEMBLY-SUMMARY.md template defined in LP-15-AGENT.md. The summary must be:
- **Scannable:** Headers, tables, bullet points. No walls of text.
- **Honest:** If there are gaps, state them. If threading is weak, say so.
- **Actionable:** Downstream routing tells the human reviewer what to run next.

### Step 3: Include the Headline and CTA

Always include the actual Arena-selected headline and CTA text in the summary. This lets the human reviewer immediately see what will be on the page without opening the JSON.

### Step 4: Write ASSEMBLY-SUMMARY.md

## QUALITY CRITERIA

- Summary includes actual headline and CTA text (not just "Arena-selected variant")
- Section map covers every section with source skill
- Quality scores match assembly-audit.md
- Arena verification status clearly stated
- Threading summary is meaningful (not just "threading is good")
- Downstream routing accurately reflects assembly state
- File exists after execution

---

## FAILURE MODES

- Summary says "Arena Verified" without including the actual text -> Include the text. The summary's value is letting the human reviewer see the page at a glance.
- Summary inflates quality scores -> Copy scores directly from assembly-audit.md. Do not editorialize.
- Summary omits missing elements -> If completeness-audit.md flagged gaps, the summary must include them. Do not present a partial assembly as complete.
- Summary is a wall of prose -> Use tables, bullets, and headers. This is a scannable document.
