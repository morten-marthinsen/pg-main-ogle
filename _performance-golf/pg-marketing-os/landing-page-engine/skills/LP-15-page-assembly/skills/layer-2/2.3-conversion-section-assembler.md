# 2.3 — Conversion Section Assembler

> **Skill:** LP-15-page-assembly
> **Layer:** 2
> **Position:** 3 of 4
> **Purpose:** Assemble all conversion sections — value stack, pricing, guarantee, urgency, FAQ, CTA placements, close, and P.S. These are the sections that directly drive the conversion action.
> **Effort Level:** opus
> **Dependencies:** `section-order-plan.md`, `content-load.md`, `transition-strategy.md`, `arena-verification-load.md`
> **Output:** `conversion-section-assembly.md`

  transition_strategy: "transition-strategy.md"
  arena_verification: "arena-verification-load.md"

## EXECUTION

**POSITIONAL REINFORCEMENT:** You assemble conversion sections and write transitions. The Arena-selected CTA text goes in at every CTA placement — verbatim from arena-verification-load.md. You do NOT rewrite pricing copy, guarantee copy, FAQ answers, or urgency copy. Your creative contribution is TRANSITIONS between these sections.

### Step 1: Identify Conversion Sections

From `section-order-plan.md`, extract all conversion sections. These typically include:

**Type A conversion sections:**
- Value Stack / Offer Block
- Pricing Block (with CTA Block 2)
- Guarantee
- Urgency/Scarcity
- Proof Block 3 (transformation proof)
- FAQ / Objection Handler
- CTA Block 3 / Close (benefit summary, binary choice, final CTA)
- P.S.

**Type B conversion sections:**
- FAQ (accordion format)
- Guarantee (branded block)
- Comparison Table
- Upsell / Bundle
- Sticky Mobile Bar (technical spec, not copy)

### Step 2: Retrieve Arena-Selected CTA Text

From `arena-verification-load.md`, retrieve the authoritative CTA text for each placement position. The Arena-selected CTA goes at EVERY CTA placement specified in the section order plan.

**CRITICAL:** If the LP-06 CTA architecture specifies multiple CTA placements (e.g., after pricing, after guarantee, at close), the Arena-selected CTA text is used at each one. CTA text variations per position (if LP-14 provided position-specific variants) should be drawn from the Arena-selected set.

### Step 3: Assemble Each Conversion Section

For each conversion section in LP-04 order:

#### Value Stack (LP-11)
- Place the value stack content from pricing-section.json
- Include: individual item values, bonus descriptions, total value statement
- Write transition-in from previous section (typically proof or education)
- Threading anchor: this is where the promise meets the offer

#### Pricing Block (LP-11)
- Place price display: sale price, original price, savings
- Place bundle options (if any)
- Place subscription options (if any)
- Place per-day calculation (if any)
- Include CTA at this position — Arena-selected text
- Write transition from value stack to pricing ("All of this would cost $X separately...")

#### Guarantee (LP-11 or LP-13)
- Place the full guarantee copy: branded name, duration, terms
- Include visual treatment notes (badge, certificate)
- Write transition from pricing to guarantee (risk reversal bridge)

#### Urgency/Scarcity (LP-13)
- Place urgency copy: deadline, countdown, limited stock, justification
- Write transition from guarantee to urgency (escalation)
- Verify scarcity justification is present — unjustified scarcity destroys trust

#### FAQ (LP-12)
- Place all FAQ Q&A pairs in priority order
- Note format specification (accordion for Type B, inline for Type A)
- Write transition into FAQ ("Still Have Questions?" or "You Might Be Wondering...")

#### Close / Final CTA (LP-14 + assembled)
- Write benefit summary ("Here's everything you get...")
- Include binary choice / crossroads moment (for Type A)
- Place FINAL CTA — Arena-selected text
- Include checkout instructions (if applicable)

#### P.S. (Type A only)
- Place P.S. content: restate guarantee, add urgency, or bonus reveal
- This is the last thing on the page — it must reinforce the strongest motivation

### Step 4: Place All CTA Instances

From `section-order-plan.md` and LP-06 architecture, place CTA buttons at every specified position:

|-------|-------------------|----------|----------------|
| 1 | Above Fold | [from 2.1] | YES |
| 2 | After Pricing | [from arena-verification-load.md] | YES |
| 3 | Pre-Close | [from arena-verification-load.md] | YES |
| [additional] | [position] | [text] | [YES/NO] |

**Rule:** The Arena-selected CTA variant is used for ALL primary CTA placements. If LP-14 provided position-specific text variations (e.g., "Start My Transformation" for post-story, "Get My Supply" for post-pricing), use the Arena-selected variant for the primary positions and position-specific variants for secondary positions — but ONLY if the position-specific variants were part of the Arena-selected set.

### Step 5: Handle Missing Conversion Content

If any conversion section has no content package (PARTIAL mode):
- Insert `[CONTENT NEEDED — LP-XX: Section Name]` placeholder
- Still write the transition structure around the gap
- Note in the assembly that LP-17 will flag this as a failure

## QUALITY CRITERIA

- Every conversion section from `section-order-plan.md` is present (or has explicit placeholder)
- Arena-selected CTA text appears verbatim at all primary CTA placements
- Every section boundary has a written transition
- Guarantee has branded name, duration, and specifics (not just "money-back guarantee")
- Urgency has justification (not fabricated scarcity)
- FAQ Q&As are complete (not truncated)
- 3+ CTA placements across the page
- P.S. present for Type A pages
- File exists after execution

---

## FAILURE MODES

- CTA text modified from Arena selection -> STOP. Use the exact text from arena-verification-load.md. No synonyms, no "improvements."
- Guarantee placed without branded name -> Check LP-11 content. If the guarantee copy lacks a branded name, assemble it as-is and note the gap — LP-17 Point 14 will flag it.
- Urgency section uses fabricated scarcity -> Assemble the urgency copy from LP-13 as-is. LP-15 does not judge. But note if no justification is provided.
- FAQ truncated due to context pressure -> Place ALL FAQ Q&As. If context window is limited, split into chunks. Do not truncate.
- Close section invents new copy -> The benefit summary in the close should reference benefits already stated earlier in the page, not introduce new ones. Assemble from existing content.
