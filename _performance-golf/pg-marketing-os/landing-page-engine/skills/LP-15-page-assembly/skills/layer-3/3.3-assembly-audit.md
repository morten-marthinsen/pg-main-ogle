# 3.3 — Assembly Audit

> **Skill:** LP-15-page-assembly
> **Layer:** 3
> **Position:** 3 of 3
> **Purpose:** Score the assembled page on four dimensions: section ordering, transition quality, threading integrity, and element completeness. Minimum score of 7.0/10 required to proceed.
> **Effort Level:** sonnet
> **Dependencies:** All Layer 2 outputs, `completeness-audit.md`, `arena-copy-verification.md`, `threading-validation.md`
> **Output:** `assembly-audit.md`

---

## INPUT

```yaml
input:
  above_fold: "above-fold-assembly.md"
  body_sections: "body-section-assembly.md"
  conversion_sections: "conversion-section-assembly.md"
  completeness_audit: "completeness-audit.md"
  arena_verification: "arena-copy-verification.md"
  threading_validation: "threading-validation.md"
  section_order_plan: "section-order-plan.md"
  page_type: "[type_a | type_b | hybrid]"
```

---

## EXECUTION

**POSITIONAL REINFORCEMENT:** You are scoring the assembly — not the copy quality. Copy quality is LP-17's job. You evaluate: Is the page assembled correctly? Are transitions present and functional? Does it thread? Is it complete?

### Step 1: Score Section Ordering (0-10)

Compare the assembled section order against LP-04's section-sequence.json (via `section-order-plan.md`):

| Score | Criteria |
|-------|---------|
| 10 | Assembled order matches LP-04 exactly. Zero deviations. |
| 8-9 | 1 minor deviation (section moved by 1 position) with documented justification. |
| 6-7 | 2-3 minor deviations, all justified. Or 1 major deviation (section moved by 3+ positions) justified. |
| 4-5 | Multiple deviations, some unjustified. |
| 1-3 | Assembled order bears little resemblance to LP-04. |

### Step 2: Score Transition Quality (0-10)

Evaluate every transition between sections:

| Score | Criteria |
|-------|---------|
| 10 | Every boundary has a transition. All transitions reference threading anchors. All transitions match page type (written for Type A, header for Type B). |
| 8-9 | Every boundary has a transition. 80%+ reference threading anchors. Type-appropriate. |
| 6-7 | 1-2 boundaries lack transitions. Most reference threading anchors. |
| 4-5 | 3+ boundaries lack transitions. Transitions are generic ("and now..."). |
| 1-3 | Sections concatenated without transitions. Bare assembly. |

For each transition, check:
- Does it exist? (Y/N)
- Does it reference a threading anchor (promise/mechanism/proof)? (Y/N)
- Is it type-appropriate? (written for Type A, header for Type B) (Y/N)
- Does it create forward momentum? (Y/N)

### Step 3: Score Threading Integrity (0-10)

Pull the threading score directly from `threading-validation.md`. This score was already calculated in microskill 2.4.

### Step 4: Score Element Completeness (0-10)

Pull from `completeness-audit.md`:

| Score | Criteria |
|-------|---------|
| 10 | 100% of required elements present. All recommended elements present. |
| 8-9 | 100% of required elements present. Some recommended elements missing. |
| 6-7 | 90-99% of required elements present. Minor gaps. |
| 4-5 | 80-89% of required elements present. Noticeable gaps. |
| 1-3 | Below 80% of required elements. Major gaps. |

### Step 5: Calculate Composite Score

```
Assembly Score = (Section Ordering x 0.20) + (Transition Quality x 0.30) + (Threading Integrity x 0.30) + (Element Completeness x 0.20)
```

**Weighting rationale:**
- Transition quality (30%): Transitions are LP-15's primary creative output. If they are weak, the assembly failed.
- Threading integrity (30%): A page that does not thread from headline to close is fundamentally broken.
- Section ordering (20%): LP-04 set the order with conversion architecture in mind. Deviations matter.
- Element completeness (20%): Missing elements will be caught by LP-17, but LP-15 should catch them first.

### Step 6: Determine Verdict

| Score | Verdict | Action |
|-------|---------|--------|
| 8.0+ | EXCELLENT | Proceed to Layer 4. |
| 7.0-7.9 | GOOD | Proceed to Layer 4. Note areas for improvement. |
| 5.0-6.9 | NEEDS IMPROVEMENT | HALT. Fix the lowest-scoring dimension before proceeding. |
| Below 5.0 | ASSEMBLY FAILURE | HALT. Return to Layer 2 and reassemble. |

### Step 7: Check Arena Verification Gate

From `arena-copy-verification.md`, confirm the gate result:
- PASS: Include in audit as "Arena Verification: PASS"
- FAIL: Overall assembly audit FAILS regardless of composite score. Arena verification is a hard gate.

### Step 8: Write assembly-audit.md

## QUALITY CRITERIA

- All four dimensions scored with evidence
- Composite score correctly calculated using weighted formula
- Arena verification gate checked (hard requirement)
- Verdict accurately reflects the scores (7.0+ = PROCEED, below 7.0 = HALT)
- If HALT, specific fix instructions provided
- File exists after execution

---

## FAILURE MODES

- Inflating scores to hit the 7.0 threshold -> Score honestly. If transitions are bare concatenation, that is a 1-3, not a 6-7. If threading has 2 high-severity breaks, that is a 3-4, not a 7-8.
- Ignoring Arena verification failure -> Arena verification is a HARD gate. If it fails, the overall verdict is HALT regardless of composite score. Even a 10.0 composite with a failed Arena gate = HALT.
- Weighting dimensions incorrectly -> Transitions and Threading are each 30%. Ordering and Completeness are each 20%. Use the formula.
- Providing vague fix instructions on HALT -> "Improve transitions" is not actionable. "Write transitions for boundaries 3, 7, and 11 which currently have bare concatenation. Reference the promise anchor at boundary 3 and the mechanism anchor at boundary 7." That is actionable.
