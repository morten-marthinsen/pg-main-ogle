# 3.4 — PDP Assembly Validator

> **Skill:** LP-15-page-assembly
> **Layer:** 3
> **Position:** 4 of 4
> **Purpose:** Validate the PDP page assembly against PDP-specific requirements. Verifies carousel completeness, buy box completeness, BTF coverage, UX compliance, and confirms no Type A patterns leaked into the PDP assembly. Runs INSTEAD of 3.1 (Completeness Audit) + 3.2 (Arena Copy Verification) for PDP pages.
> **Effort Level:** sonnet
> **Dependencies:** `pdp-page-assembly.md` (from 2.5), `pdp-packages-loaded.md` (from 0.5), `threading-validation.md` (from 2.4 — adapted for PDP)
> **Output:** `pdp-assembly-validation.md`

---

## INPUT

```yaml
input:
  pdp_assembly: "pdp-page-assembly.md"
  pdp_packages: "pdp-packages-loaded.md"
  threading_validation: "threading-validation.md"
  brief_load: "brief-load.md"
  page_type: "[type_b | hybrid]"
```

---

## WHEN TO RUN

This microskill runs ONLY when `page_type` is `type_b` or `hybrid`. For `type_a` pages, use 3.1 + 3.2 instead. **Never run both paths.**

---

## EXECUTION

**POSITIONAL REINFORCEMENT:** You are validating the PDP assembly — checking completeness, correctness, and compliance against PDP-specific requirements. You are NOT evaluating copy quality (that is LP-17's job). You check: is everything there, does it match the source packages, and did any Type A patterns leak in?

### Step 1: Verify Carousel Completeness (0-10)

Cross-reference `pdp-page-assembly.md` carousel against `pdp-packages-loaded.md` PDP-03 data:

| Check | Criteria | Points |
|-------|---------|--------|
| Slide count | All 10 slides from PDP-03 present in assembly | 4 |
| Slide order | Slides assembled in PDP-03's specified order | 2 |
| Lazy-loading | Slide 1 = eager, Slides 2-10 = lazy | 1 |
| Alt text | All slides have alt text specified | 1 |
| Navigation | Swipe + arrows + indicators specified | 1 |
| Zoom | Zoom capability specified | 1 |

**Scoring:**
- 10: All 10 slides, correct order, all technical specs
- 8-9: All slides present, 1-2 minor spec gaps
- 6-7: 8-9 slides present, or major spec gaps
- 4-5: 6-7 slides present
- Below 4: Fewer than 6 slides or carousel not assembled

**If PDP-03 provided fewer than 10 slides:** Score against what PDP-03 provided, not against 10. Flag the upstream gap.

### Step 2: Verify Buy Box Completeness (0-10)

Cross-reference `pdp-page-assembly.md` buy box against PDP-03 architecture + PDP-06 copy:

| # | Component | Check | Source |
|---|-----------|-------|--------|
| 1 | Product title | Present with PDP-06 copy? | PDP-03 + PDP-06 |
| 2 | Rating strip | Present with count + average? | PDP-03 + PDP-05 |
| 3 | Price block | Sale, original, savings all present? | PDP-03 + PDP-06 |
| 4 | Variant selector | Options listed with pricing? | PDP-03 + PDP-06 |
| 5 | Quantity selector | Specified? | PDP-03 |
| 6 | ATC button | Present with VERBATIM PDP-06 text? | PDP-03 + PDP-06 |
| 7 | Shipping info | Present? | PDP-06 |
| 8 | Payment badges | Present? | LP-08 |
| 9 | Short description | Present with PDP-06 copy? | PDP-03 + PDP-06 |
| 10 | Stock indicator | Present? | LP-13 |

**ATC button verbatim check:** Extract the ATC button text from `pdp-page-assembly.md` and compare against PDP-06's ATC text from `pdp-packages-loaded.md`. Must be exact match (minor formatting differences acceptable). If modified -> FAIL this check.

**Scoring:**
- 10: All 10 components present, ATC text verbatim, all copy sourced correctly
- 8-9: 9-10 components, all critical components present (title, price, ATC)
- 6-7: 8-9 components, or ATC text not verbatim
- 4-5: 6-7 components, or missing a critical component
- Below 4: Missing 4+ components or no buy box assembled

**Buy box component cross-reference:**
- PDP-06 buy box copy MUST match PDP-03 architecture. If PDP-06 has copy for a component that PDP-03 did not architect, flag as a drift issue. If PDP-03 architected a component that PDP-06 did not provide copy for, flag as missing copy.

### Step 3: Verify BTF Section Coverage (0-10)

Cross-reference `pdp-page-assembly.md` BTF sections against PDP-04 section sequence:

1. List all sections from PDP-04's `pdp-section-sequence.json` (via `pdp-packages-loaded.md`)
2. List all BTF sections assembled in `pdp-page-assembly.md`
3. Check each PDP-04 section: is it assembled? Is it in the correct position?
4. Check each assembled section: does it have content from PDP-07 (or shared LP skill)?

| Check | Criteria | Points |
|-------|---------|--------|
| Section count | All PDP-04 sections present in assembly | 4 |
| Section order | Assembled order matches PDP-04 sequence | 2 |
| Content populated | Every section has actual content (not placeholder) | 2 |
| Source attribution | Every section's content source identified | 1 |
| Mid-page CTA | Mid-page CTA present after benefits section | 1 |

**Scoring:**
- 10: All sections present, correct order, all content populated, mid-page CTA present
- 8-9: All sections present, 1-2 minor order deviations (documented), content populated
- 6-7: 1-2 sections missing, or 3+ order deviations, or some placeholder content
- 4-5: 3+ sections missing, or significant order deviation
- Below 4: Fewer than 50% of PDP-04 sections assembled

### Step 4: Verify UX Compliance (0-10)

Check PDP-specific UX requirements:

| Check | Criteria | Points |
|-------|---------|--------|
| Sticky ATC bar | Specification present with trigger, contents, style, touch target | 2.5 |
| FAQ accordion | FAQ assembled in accordion format (not inline) | 1.5 |
| Reviews integration | PDP-05 review system assembled (histogram, filters, UGC) | 2 |
| Mobile-first specs | Touch targets 44px+, font minimums, no horizontal scroll | 1.5 |
| No Type A pattern leaks | No story transitions, no P.S., no long-form close, no narrative bridges | 2.5 |

**Type A leak detection — check for these FORBIDDEN patterns:**
- Story-style transitions ("Now that you understand...", "But that's not all...", "Let me tell you...")
- P.S. section anywhere in the assembly
- Long-form close section (Makepeace-style close with P.S. + P.P.S.)
- Narrative bridges between sections (paragraphs connecting sections via story)
- Hero image + headline above-fold pattern (instead of carousel + buy box)
- Value stack presentation (Type A conversion block pattern)

If ANY Type A pattern detected -> Deduct 2.5 points from UX score AND flag as a critical issue.

**Scoring:**
- 10: Sticky ATC present, FAQ accordion, reviews integrated, mobile-first, zero Type A leaks
- 8-9: All present, 1 minor gap (e.g., touch target not specified for one element)
- 6-7: 1-2 UX elements missing, or 1 Type A pattern detected
- 4-5: Sticky ATC missing, or reviews not integrated, or 2+ Type A patterns
- Below 4: Multiple critical UX elements missing, or pervasive Type A contamination

### Step 5: Cross-Reference PDP-03 GATE_3 Results

If PDP-03 output includes a GATE_3 checkpoint (confirming all PDP-03 audits passed before handoff):
- Verify GATE_3 status is PASS
- If GATE_3 is FAIL or not found: flag as WARNING — the upstream PDP-03 package may be incomplete

This is an advisory check, not a scoring dimension. It provides confidence that the upstream packages are production-ready.

### Step 6: Calculate Composite PDP Assembly Score

```
PDP Assembly Score = (Carousel Completeness x 0.25) + (Buy Box Completeness x 0.25) + (BTF Coverage x 0.25) + (UX Compliance x 0.25)
```

**Equal weighting rationale:** All four dimensions are equally critical for a PDP page. A PDP with a complete carousel but no buy box is useless. A PDP with perfect content but no sticky ATC bar loses mobile conversions. A PDP with Type A patterns leaked in confuses the scan-optimized reading path.

### Step 7: Determine Verdict

| Score | Verdict | Action |
|-------|---------|--------|
| 8.0+ | EXCELLENT | Proceed to Layer 4. |
| 7.0-7.9 | GOOD | Proceed to Layer 4. Note areas for improvement. |
| 5.0-6.9 | NEEDS IMPROVEMENT | HALT. Fix the lowest-scoring dimension. Return to 2.5 to reassemble the failing section. |
| Below 5.0 | ASSEMBLY FAILURE | HALT. Return to 2.5 and reassemble. If upstream packages are incomplete, return to PDP skills. |

**Hard gates (override composite score):**
- ATC button text not verbatim -> HALT regardless of score
- Sticky ATC bar missing -> HALT regardless of score
- Type A pattern leak detected (story transition, P.S., long-form close) -> HALT regardless of score

### Step 8: Write pdp-assembly-validation.md

## QUALITY CRITERIA

- Carousel verified slide-by-slide against PDP-03 specification
- Buy box verified component-by-component (all 10) against PDP-03 + PDP-06
- ATC button text verbatim comparison performed with exact text shown
- BTF sections verified against PDP-04 sequence — every section accounted for
- PDP-05 review system integration verified (histogram, filters, UGC)
- Sticky ATC bar specification verified as present
- FAQ accordion format verified (not inline)
- Type A pattern leak scan performed — all 6 forbidden patterns checked
- Buy box architecture drift check performed (PDP-03 vs PDP-06 alignment)
- PDP-03 GATE_3 cross-referenced (advisory)
- Composite score correctly calculated using equal weighting (4 x 25%)
- Hard gates checked independently of composite score
- Verdict accurately reflects scores (7.0+ = PROCEED) AND hard gates
- File exists after execution

---

## FAILURE MODES

- Skipping the ATC verbatim check -> The ATC button is the primary conversion element. Its text must be verbatim from PDP-06. Not checking this is equivalent to LP-15's Arena verification skip for Type A pages. Always compare exact text.
- Not scanning for Type A pattern leaks -> Type A contamination is the PDP equivalent of the Arena bypass failure. A PDP page with story transitions reads wrong to scan-optimized shoppers. Always run the 6-pattern leak scan.
- Scoring carousel as 10/10 when slides are missing -> If PDP-03 specified 10 slides and only 8 are assembled, that is NOT a 10. Score honestly against the specification.
- Treating the sticky ATC bar as optional -> It is MANDATORY. Missing sticky ATC bar = HALT, even if the composite score is 10.0 across other dimensions. This is a hard gate.
- Inflating UX compliance to avoid HALT -> Score honestly. If story transitions leaked in, deduct the full 2.5 points. Do not rationalize ("it's just one transition").
- Ignoring PDP-03/PDP-06 drift -> If PDP-03 architected a component that PDP-06 did not provide copy for, that is a gap that LP-15 cannot fix. Flag it so the human can decide whether to return to PDP-06 or accept the gap.
