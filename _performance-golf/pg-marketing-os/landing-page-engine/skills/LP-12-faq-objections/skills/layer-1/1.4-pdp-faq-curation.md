# 1.4 — PDP FAQ Curation

**Skill:** LP-12-faq-objections
**Position:** Layer 1, Microskill 4
**Purpose:** Curate the top 5-7 FAQ items for PDP pages following Baymard's "curated, not exhaustive" rule. PDP FAQ is fundamentally different from Type A FAQ: fewer items, sourced from real customer questions (reviews, support tickets, competitor FAQs), sequenced hardest-last, and positioned AFTER the reviews section. This microskill runs INSTEAD of 1.2 (Sequence Planner) for Type B and Hybrid pages — it both curates AND sequences in a single pass.
**Effort Level:** medium
**Dependencies:** `objection-classification.md` (1.1), `packages-loaded.md` (0.1), `specimens-loaded.md` (0.3)
**Output:** `pdp-faq-curation.md`

## Execution Logic

1. **Verify page type.** This microskill runs ONLY for Type B and Hybrid pages. If page type is Type A, SKIP this microskill and use 1.2 (Sequence Planner) instead.

2. **Gather FAQ candidate sources** in priority order:

   | Priority | Source | Why |
   |----------|--------|-----|
   | 1 (highest) | Actual customer reviews | Real questions from real buyers — highest signal |
   | 2 | Support ticket themes | Recurring pre-purchase questions that block conversion |
   | 3 | Competitor FAQ analysis | What competitors answer reveals what customers universally ask |
   | 4 | Research quotes | Implicit questions embedded in audience language |
   | 5 (lowest) | Objection classification (1.1) | Engine-generated objections — use only to fill gaps |

   **Critical rule:** If PDP-05 review system data is available, extract ALL implicit questions from review text before looking at other sources. Phrases like "I wish I knew before buying..." "My only concern was..." "I was worried about..." are gold-standard FAQ sources.

3. **Apply the 3-criteria filter** to every candidate question:

   | Criterion | Weight | Definition | Threshold |
   |-----------|--------|------------|-----------|
   | Frequency | 40% | How often is this question asked across sources? | Must appear in >= 2 sources OR be a universal product-type question |
   | Purchase-blocking | 35% | Does this question prevent buying if unanswered? | Score 1-10: must be >= 5 to qualify |
   | Specificity | 25% | Is this question specific to THIS product (not generic)? | Must reference product category, ingredient, feature, or use case |

   **Disqualify:** Generic questions that apply to any product and are not purchase-blocking. "What makes your company different?" is generic. "Why does this contain both glycinate AND threonate?" is specific.

4. **Select the top 5-7 items.** Hard maximum 7. Hard minimum 5.

   Selection rules:
   - All 3 mandatory objection categories (price/value, efficacy, trust/guarantee) must each have at least 1 item in the final 5-7. If the 3-criteria filter drops a mandatory category, force-include the highest-scoring item from that category.
   - After mandatory categories are secured, fill remaining slots with the highest-scoring items across all sources.
   - If exactly 5 items qualify, use 5. Do NOT pad to 7 with low-quality questions.
   - If more than 7 qualify, cut the lowest-scoring items.
   - Prefer items sourced from reviews/support tickets over engine-generated items.

5. **Sequence the selected items** using the hardest-last pattern:

   ```
   Position 1-2: Easy technical/logistics questions (low purchase-blocking score)
   Position 3-4: Moderate questions (specificity-driven, mid purchase-blocking)
   Position 5-6: Hard questions (efficacy, comparison)
   Position 6-7: Hardest questions (price/value, trust/guarantee)
   Final position: The single most purchase-blocking question — answered with maximum proof
   ```

   **No two adjacent questions from the same category.** Varies the rhythm.

6. **Source attribution.** For every selected item, record WHERE the question came from (review, support ticket, competitor FAQ, research quote, engine-generated). This is non-negotiable — it proves the questions are real, not invented.

7. **Write `pdp-faq-curation.md`.**

## Quality Criteria
- Exactly 5-7 items selected (not 4, not 8, not 10)
- All 3 mandatory objection categories represented
- Every item has source attribution with a specific reference (not "common question")
- Real-source items (reviews, support tickets, competitor FAQs) outnumber engine-generated items
- Hardest-last sequencing applied with hard questions in the final positions
- No two adjacent questions from the same category
- 3-criteria filter scores documented for every selected AND rejected item
- PDP-05 review data used if available (not ignored)

## Failure Modes
- Selecting 8+ items for a PDP FAQ -> Baymard rule: 5-7 maximum. More items signal that the FAQ is exhaustive rather than curated. Cut the lowest-scoring items.
- All items from engine-generated classification with no real-source items -> The entire point of PDP FAQ curation is that the questions come from REAL customers. If no review/support/competitor data is available, flag as DEGRADED_SOURCE_QUALITY and proceed with best available — but document the gap.
- Skipping mandatory categories to stay within 7 items -> Mandatory categories override the cap. If all 3 mandatory items plus the 4 best non-mandatory items equal 7, that is the correct selection. Never drop a mandatory category to include a "more interesting" non-mandatory question.
- Missing source attribution -> Every item must trace to a specific source. "This is a common question" is not source attribution. "Appeared in 3 of 5 competitor FAQs" or "Review by 'Sarah M.' on product page" is.
- Using this microskill for Type A pages -> STOP. Type A pages use 1.2 (Sequence Planner) with the full 10-12 item set. This microskill is for PDP pages only.
