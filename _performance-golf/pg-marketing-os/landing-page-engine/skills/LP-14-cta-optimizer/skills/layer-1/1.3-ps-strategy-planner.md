# 1.3 — P.S. Strategy Planner

> Version: 1.0
> Layer: 1
> Model: sonnet
> Input: packages-loaded.md, cta-architecture-loaded.md
> Output: ps-strategy.md

**Skill:** LP-14-cta-optimizer
**Position:** Layer 1, Microskill 3
**Purpose:** Decide the P.S. copy strategy: how many P.S. lines (1-3), which specific benefit to restate in P.S. #1, which urgency/scarcity to reference in P.S. #2 (if applicable), and which proof point for P.S. #3 (if applicable). This prevents the P.S. writer (2.3) from improvising and ensures P.S. copy strictly restates — never introduces.
**Effort Level:** medium
**Dependencies:** `packages-loaded.md` (0.1), `cta-architecture-loaded.md` (0.2)
**Output:** `ps-strategy.md`

## Execution Logic

1. **Determine P.S. count based on page type:**
   - Type A (Long-Form): 2-3 P.S. lines (reader has invested significant reading time — P.S. is the final recency-bias anchor)
   - Type B (Ecomm/PDP): 0-1 P.S. lines (P.S. is less conventional on product pages — only include if the page has a below-fold content area)
   - Hybrid: 1-2 P.S. lines

2. **Select P.S. #1 benefit to restate:**
   - Read the promise statement from LP-07's threading anchors
   - Read the primary desire from the audience profile
   - The P.S. #1 benefit must be ONE of:
     - The core promise from the headline (threading anchor)
     - The primary audience desire (from research/brief)
     - The guarantee (risk removal)
   - Select the strongest one — the benefit that would make a scanner who reads ONLY the P.S. want to scroll back up

3. **Select P.S. #2 content (if applicable):**
   - Only include P.S. #2 if urgency is PERMITTED (LP-13 data exists and is justified)
   - P.S. #2 restates the urgency/scarcity from LP-13 — it does NOT create new urgency
   - Extract the specific urgency copy from LP-13 that P.S. #2 will reference
   - If urgency is FORBIDDEN: Skip P.S. #2 entirely. Do not substitute with a different P.S. type.

4. **Select P.S. #3 content (if applicable):**
   - Only include P.S. #3 for Type A pages with strong proof assets
   - P.S. #3 is a final proof point — authority endorsement, clinical study result, or social proof stat
   - The proof must already be stated on the page. P.S. #3 reminds — it does not introduce.
   - Select the single most impressive proof point already present on the page

5. **For each P.S. line, create a content brief:**
   - Exact benefit/claim to restate (verbatim from the page, not rewritten)
   - Source section (where on the page this was originally stated)
   - CTA reference (which CTA button text the P.S. should direct to)
   - Sentence target (2-4 sentences)

6. **Write `ps-strategy.md`** with the complete P.S. plan.

## Quality Criteria
- P.S. count appropriate for page type
- Each P.S. line has a specific, named benefit/claim to restate — not a vague direction
- Source section identified for each restated claim (proves it already exists on the page)
- Urgency P.S. only planned if LP-13 data is PERMITTED
- New Claim Check explicitly verifies no P.S. line introduces new information

## Failure Modes
- P.S. #2 planned without LP-13 permission -> Remove P.S. #2. No urgency without data.
- P.S. benefit not traceable to a specific page section -> This may be a new claim in disguise. Trace every P.S. claim to its source.
- All 3 P.S. lines restating the same benefit -> Diversify: benefit + urgency + proof, not benefit + benefit + benefit.
- P.S. count excessive for Type B -> Type B rarely uses P.S. If including, limit to 1.
- "Content brief" left vague ("restate the main benefit") -> Must be specific: "[exact text to restate]" with "[source section]".
