# 3.1 — CTA Copy Validator

> Version: 1.0
> Layer: 3
> Model: sonnet
> Input: CTA_SELECTION.yaml, packages-loaded.md
> Output: cta-validation.md

**Skill:** LP-14-cta-optimizer
**Position:** Layer 3, Microskill 1
**Purpose:** Validate the human-selected CTA copy against three critical dimensions: threading (connection to LP-07's promise), length compliance (button text 3-8 words, context max 2 sentences), and emotional appeal coverage (>= 5 of 6 types across all positions). This is the structural validation that catches misalignment before the audit.
**Effort Level:** medium
**Dependencies:** `CTA_SELECTION.yaml` (human-created), `packages-loaded.md` (0.1)
**Output:** `cta-validation.md`

---

## Input Schema

```yaml
input:
  cta_selection: "CTA_SELECTION.yaml"
  packages: "packages-loaded.md"
  required_fields:
    - primary_cta (button_text, context_text, emotional_appeal)
    - secondary_ctas (all positions)
    - ps_copy (all lines)
    - final_close (text)
    - threading_anchors (from packages-loaded.md)
    - modification_notes (from CTA_SELECTION.yaml)
```

## Execution Logic

### Check 1: Threading Validation

For each CTA element (primary, secondary CTAs, final close), check for threading to LP-07:

1. Load LP-07 threading anchors:
   - `primary_hook_phrase`: "[phrase]"
   - `promise_statement`: "[statement]"

2. Check primary CTA:
   - Does button text or context text echo the hook phrase? [Y/N]
   - Does button text or context text reference the promise? [Y/N]
   - Quote the specific threading reference.

3. Check final close paragraph:
   - Does it explicitly reference the hook phrase? [Y/N]
   - Does it restate or echo the promise statement? [Y/N]
   - Quote the specific references.

4. Check secondary CTAs:
   - Do any secondary CTAs reference the promise? (Desirable but not required for every position)

5. **Threading verdict:**
   - Primary CTA threads to LP-07: REQUIRED
   - Final close threads to LP-07: REQUIRED
   - At least 1 secondary CTA threads: RECOMMENDED

### Check 2: Length Compliance

For every CTA element, verify word counts:

1. **Primary CTA button text:** Count words. Must be 3-8. Flag if outside range.
2. **All secondary CTA button texts:** Count words. Must be 3-8. Flag if outside range.
3. **All context texts:** Count sentences. Must be 0-2 sentences. Flag if > 2.
4. **P.S. lines:** Count sentences per P.S. Must be 2-4. Flag if outside range.
5. **Final close:** Count sentences. Must be 3-5. Flag if outside range.

### Check 3: Emotional Appeal Coverage

Map all selected CTAs to their emotional appeal types:

1. List every CTA (primary + all secondary positions)
2. Tag each with its emotional appeal type (1-6)
3. Count unique appeal types represented
4. Verify >= 5 of 6 types covered (urgency may be blocked)
5. If < 5 covered: flag which types are missing

### Check 4: Modification Application

If human provided modification_notes in CTA_SELECTION.yaml:
1. List each modification request
2. Verify modifications have been applied to the selected copy
3. If modifications not yet applied: flag for application before assembly

## Quality Criteria
- All four checks completed — no checks skipped
- Threading check quotes specific text (not just Y/N — show the evidence)
- Length compliance checked for every element individually
- Emotional appeal coverage mapped to specific positions
- Modification notes verified as applied

## Failure Modes
- Marking threading as PASS without quoting the specific echo -> Must quote the exact words that constitute threading. "PASS" without evidence is not verifiable.
- Counting compound words incorrectly for button text length -> Count actual words separated by spaces. Hyphenated words count as one.
- Missing a CTA position in length check -> Every position from CTA_SELECTION.yaml must be checked. Do not skip optional positions.
- Urgency appeal counted when BLOCKED -> If LP-13 data is unavailable, urgency cannot count toward coverage. Adjust coverage threshold to 4/5 (excluding urgency).
