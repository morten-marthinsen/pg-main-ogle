# 2.4 — Offer Test Generator

> **Skill:** LP-18-ab-test-plan
> **Layer:** 2
> **Position:** 4 of 5
> **Purpose:** Generate offer/pricing A/B test hypotheses covering price anchoring approach, guarantee prominence, value stack ordering, urgency treatment, and offer type alternatives.
> **Effort Level:** opus
> **Dependencies:** `prioritized-opportunities.md`, `assembled-page-load.md`, `benchmark-load.md`
> **Output:** `offer-tests.md`

---

## INPUT

```yaml
input:
  prioritized_opportunities: "prioritized-opportunities.md"
  assembled_page: "assembled-page-load.md"
  benchmarks: "benchmark-load.md"
  page_type: "[from assembled-page-load.md]"
```

---

## EXECUTION

**POSITIONAL REINFORCEMENT:** You are generating SPECIFIC offer test hypotheses. Every hypothesis must cite the exact current offer/pricing element and propose a specific, implementable change. "Test a better offer" is forbidden. Remember: offer testing typically outperforms headline testing (conversion-data-reference.md).

### Step 1: Check if Offer Tests Are Warranted

Read `prioritized-opportunities.md`. Only proceed if offer-category opportunities appear. If none, write `offer-tests.md` with: "No offer test opportunities prioritized for this page." and exit.

### Step 2: Extract Current Offer State

From `assembled-page-load.md`, extract:
- Price display: exact format, text, and position
- Anchoring method: what technique is currently used (crossed-out original, per-unit breakdown, comparison, savings display, none)
- Guarantee: exact text, branded name (if any), duration, placement
- Urgency elements: countdown, limited stock, limited-time, deadline, none
- Value stack: structure and order (bonuses listed before or after main product)
- Subscription options: present/absent, savings displayed
- Bundle options: present/absent, tier structure

### Step 3: Generate Offer Test Hypotheses

For each offer opportunity, generate 1-2 specific hypotheses from these test dimensions:

**Dimension A: Price Anchoring**
| Current Approach | Test Against | Reasoning |
|-----------------|-------------|-----------|
| No anchoring (price displayed flat) | Crossed-out higher price | Anchoring makes the current price feel like a deal |
| Crossed-out price only | Per-day breakdown ("Just $1.13/day") | Micro-pricing reduces perceived cost |
| Single price point | Tiered bundles (1x / 3x / 6x) | BIOptimizers and LMNT both use tiered bundles — ascending commitment |
| Per-unit price | Total savings display ("Save $47.40") | Dollar savings are more tangible than percentages for some audiences |
| No comparison | "If bought separately: $X" value stack | Value-before-price establishes perceived worth |

**Dimension B: Guarantee Treatment**
From conversion-data-reference.md — longer guarantees reduce return rates AND increase conversions:
| Current Guarantee | Test Against | Reasoning |
|------------------|-------------|-----------|
| 30-day | 60-day or 90-day | Longer guarantee = lower returns (counterintuitive). 60-day: 8-12% vs 30-day: 10-15% |
| Generic "money-back guarantee" | Branded guarantee with specific name | BIOptimizers: "365-Day No Questions Asked." Branded feels more committed |
| Guarantee in footer only | Guarantee prominently near pricing AND near CTA | Visibility increases risk reversal effectiveness |
| No guarantee displayed | Add prominent guarantee section | Guarantee visibility: "Significant" lift (conversion-data-reference.md) |

**Dimension C: Urgency/Scarcity Treatment**
| Current State | Test Against | Reasoning |
|--------------|-------------|-----------|
| No urgency | Justified deadline ("Introductory pricing ends [date]") | +15-25% for limited-time offers (conversion-data-reference.md). Must be justified, not fake. |
| Generic urgency ("Act Now!") | Specific scarcity ("Only 247 units at this price") | Specific numbers are more credible than generic urgency language |
| Countdown timer | No countdown (trust-based) | Some audiences distrust countdown timers — test removal vs keeping |
| No scarcity | Stock-level indicator ("Only X left at this price") | Real scarcity (when genuine) creates urgency without manipulation |

**Dimension D: Value Stack Ordering**
| Current State | Test Against | Reasoning |
|--------------|-------------|-----------|
| Product listed first, bonuses after | Bonuses first, product as "plus you get" | Leading with bonuses can increase perceived total value |
| No value stack | Add explicit value stack with individual prices | Breaking value into components increases perceived worth |
| Value stack without subtotals | Add "Total value: $X" above actual price | Anchoring the savings gap |

**Dimension E: Offer Type**
From conversion-data-reference.md — offer type hierarchy:
| Current Offer | Test Against | Reasoning |
|-------------|-------------|-----------|
| Hard offer (standard price) | Soft offer (installments, try first) | +20-50% vs hard offer |
| Single purchase | Subscribe-and-save option | Subscription revenue + lower perceived commitment. LMNT: 13% savings. |
| Standard price | Free trial / $1 trial | MAMG and Graziosi pattern — trial entry with auto-billing. Highest trial conversion. |

### Step 4: Write Each Hypothesis Using the Schema

```yaml
test_id: "T-[N]"
category: "offer"
hypothesis: "If we change [exact offer element] from [control] to [variant], conversion rate will increase by [X-Y%] because [reasoning]."
control:
  description: "[current offer state]"
  specific_text_or_element: "[exact current pricing/guarantee/urgency element]"
variant:
  description: "[proposed change]"
  specific_text_or_element: "[exact proposed change — specific format, text, placement]"
expected_lift: "[X-Y%]"
lift_source: "[citation]"
implementation_difficulty: "[low | medium | high]"
```

### Step 5: Write Offer Tests

Compile all offer test hypotheses into `offer-tests.md`.

---

## QUALITY CRITERIA

- Only generates tests if offer opportunities were prioritized
- Current offer state fully documented from assembled page
- Each hypothesis specifies exact control and variant
- Guarantee tests reference the counterintuitive longer-guarantee data
- Urgency tests only propose justified scarcity (not fake countdown timers)
- Offer type changes marked as high implementation difficulty (they change the business model)
- Lift estimates cite conversion-data-reference.md offer type data
- File exists after execution

---

## FAILURE MODES

- Offer details not extractable from assembled page -> Flag as INCOMPLETE CONTROL. Provide hypotheses with available data.
- Proposing offer type changes that require business decisions (e.g., adding a trial) -> Mark as HIGH difficulty and note: "Requires business decision — pricing model change."
- Proposing fake urgency -> NEVER propose artificial scarcity or countdown timers without a genuine reason. If urgency is recommended, specify that the justification must be real.
- Single-offer violation -> All offer test variants must maintain the single-offer rule. Never propose adding a second competing offer to the page.
