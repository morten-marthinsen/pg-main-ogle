# 1.2 — Test Prioritizer

> **Skill:** LP-18-ab-test-plan
> **Layer:** 1
> **Position:** 2 of 2
> **Purpose:** Rank test opportunities by: (a) expected conversion lift from benchmark data, (b) ease of implementation (copy change vs structural change), (c) traffic requirements (low-traffic-friendly vs needs high volume). Select top 5-10 for hypothesis generation.
> **Effort Level:** sonnet
> **Dependencies:** `test-opportunities.md`, `benchmark-load.md`
> **Output:** `prioritized-opportunities.md`

---

## INPUT

```yaml
input:
  opportunities: "test-opportunities.md"
  benchmarks: "benchmark-load.md"
  estimated_traffic: "[from assembled-page-load.md page brief data, or 'unknown']"
```

---

## EXECUTION

**POSITIONAL REINFORCEMENT:** You are RANKING opportunities, not generating hypotheses. Output is a prioritized list with scores. Do not write test hypotheses — that is Layer 2's job.

### Step 1: Score Each Opportunity on Three Axes

For each opportunity in `test-opportunities.md`, assign scores on three axes:

**Axis 1: Expected Lift (1-10)**
Use the element-level impact data from `benchmark-load.md`:

| Expected Lift | Score |
|--------------|-------|
| Very High (>50% documented) | 9-10 |
| High (20-50% documented) | 7-8 |
| Medium (10-20% documented) | 5-6 |
| Low-Medium (5-10% documented) | 3-4 |
| Low (<5% documented) | 1-2 |

**Axis 2: Ease of Implementation (1-10)**

| Change Type | Score | Examples |
|------------|-------|---------|
| Copy-only change (text swap) | 9-10 | Headline rewrite, CTA text change, guarantee text edit |
| Minor layout change | 7-8 | Move proof block higher, add sticky CTA bar, reorder two sections |
| Moderate layout change | 5-6 | Add new section, restructure above-fold, add video embed |
| Structural change | 3-4 | Redesign section architecture, add interactive elements |
| Full page restructure | 1-2 | Change page type, rebuild layout, new technology required |

**Axis 3: Traffic Friendliness (1-10)**

| Characteristic | Score |
|---------------|-------|
| Small MDE works (even 5% relative change is valuable) | 9-10 |
| Moderate MDE needed (10-20% relative change) | 6-8 |
| Large MDE needed (30%+ relative change to be meaningful) | 3-5 |
| Very subtle change — needs massive sample to detect | 1-2 |

### Step 2: Calculate Priority Score

**Priority Score = (Lift Score x 0.50) + (Ease Score x 0.30) + (Traffic Score x 0.20)**

Weight rationale:
- Lift is most important — test the thing most likely to move the needle
- Ease is second — low-effort tests can be run faster and cheaper
- Traffic is third — traffic constraints are real but solvable with time

### Step 3: Rank by Priority Score

Sort all opportunities by priority score descending.

### Step 4: Apply Selection Rules

From the ranked list, select the top 5-10 opportunities for hypothesis generation:

**Selection rules:**
1. Include at least 1 opportunity from each category that has FAIL points from LP-17
2. Do not include more than 3 opportunities from any single category (avoid over-testing one area)
3. If estimated traffic is <500/month, limit to top 3-5 opportunities only
4. If LP-17 score was 18-20 (LAUNCH_READY), tests will be optimization-focused — still include 5 minimum
5. If LP-17 score was 15-17 (MINOR_OPTIMIZATION), prioritize FAIL-point tests first

### Step 5: Flag Potential Conflicts

Scan the selected opportunities for potential conflicts:
- Two opportunities targeting the same element -> Flag as conflicting
- Two opportunities in the same above-fold zone -> Flag as potentially interacting
- An opportunity that changes the page structure + another that tests within that structure -> Flag as dependent

### Step 6: Write Prioritized Opportunities

Write the ranked, selected list to `prioritized-opportunities.md`.

## QUALITY CRITERIA

- All opportunities scored on three axes with documented reasoning
- Priority scores calculated correctly (weighted formula)
- Top 5-10 selected following selection rules
- Category distribution balanced (no more than 3 from any single category)
- Conflicts identified and flagged
- Traffic advisory included
- File exists after execution

---

## FAILURE MODES

- Fewer than 5 opportunities from Layer 1.1 -> Select all. If 3-4, proceed but note the limited test plan. If <3, flag for human review.
- All opportunities cluster in one category (e.g., 8 headline opportunities) -> Select top 3 from the dominant category, then look for lower-ranked opportunities in other categories to ensure breadth.
- Traffic estimate unknown -> Proceed with all 5-10 selected. Layer 3 will calculate sample sizes. The human reviewer will need to assess feasibility based on actual traffic data.
- All opportunities have similar priority scores -> When scores are within 0.5 of each other, prioritize by category diversity (spread across categories rather than clustering).
