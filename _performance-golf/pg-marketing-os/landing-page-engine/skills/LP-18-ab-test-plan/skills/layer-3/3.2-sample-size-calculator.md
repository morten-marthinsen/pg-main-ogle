# 3.2 — Sample Size Calculator

> **Skill:** LP-18-ab-test-plan
> **Layer:** 3
> **Position:** 2 of 2
> **Purpose:** For each validated hypothesis, estimate the required sample size based on baseline conversion rate, minimum detectable effect, and standard 95% confidence / 80% power. Flag tests that require >10K visitors. Recommend testing sequence for low-traffic pages.
> **Effort Level:** sonnet
> **Dependencies:** `validated-hypotheses.md`, `benchmark-load.md`
> **Output:** `sample-sizes.md`

---

## INPUT

```yaml
input:
  validated_hypotheses: "validated-hypotheses.md"
  benchmarks: "benchmark-load.md"
  baseline_cvr: "[from page brief or industry benchmark]"
  estimated_daily_traffic: "[from page brief or 'unknown']"
```

## EXECUTION

**POSITIONAL REINFORCEMENT:** You are CALCULATING sample sizes, not evaluating hypotheses. Use the standard formula. Show your work. Flag high-traffic tests. Recommend a testing sequence.

### Step 1: Determine Baseline Conversion Rate

The baseline CVR (p) is needed for every calculation. Source hierarchy:
1. Actual page conversion rate from page-brief.json (if available)
2. Industry benchmark from conversion-data-reference.md matched to page vertical
3. Median CVR all industries: 6.6% (Unbounce Q4 2024) as fallback

Document which source is used. The accuracy of sample size estimates depends heavily on the baseline CVR accuracy.

### Step 2: Determine MDE for Each Test

The Minimum Detectable Effect (MDE) is the smallest lift worth detecting. For each test:

1. Start with the hypothesis's expected lift range (e.g., "+15-30%")
2. Convert the LOW end of the relative range to an absolute effect:
   - If baseline CVR = 3% and expected relative lift = 15%, then absolute MDE = 0.45% (delta = 0.0045)
3. Sanity check: if the absolute MDE is <0.5%, the test will require very high traffic. Consider using a larger MDE (the high end of the expected range) for practical sizing.
4. Document the chosen MDE and reasoning.

### Step 3: Calculate Sample Size for Each Test

**Formula:**

```
n (per variant) = (Z_alpha/2 + Z_beta)^2 * p * (1 - p) / delta^2
```

Where:
- Z_alpha/2 = 1.96 (for 95% confidence, two-tailed)
- Z_beta = 0.84 (for 80% power)
- (Z_alpha/2 + Z_beta)^2 = (2.80)^2 = 7.84
- p = baseline conversion rate (as decimal)
- delta = minimum detectable effect (as decimal, absolute)

**Total visitors needed = 2 * n** (control + variant)

**Worked examples at common baselines:**

| Baseline CVR | MDE (absolute) | MDE (relative) | n per variant | Total visitors |
|-------------|---------------|----------------|---------------|---------------|
| 3% | 0.6% | 20% | 6,335 | 12,670 |
| 3% | 0.9% | 30% | 2,816 | 5,632 |
| 3% | 1.5% | 50% | 1,014 | 2,028 |
| 5% | 1.0% | 20% | 3,727 | 7,454 |
| 5% | 1.5% | 30% | 1,657 | 3,314 |
| 5% | 2.5% | 50% | 597 | 1,194 |
| 10% | 2.0% | 20% | 1,764 | 3,528 |
| 10% | 3.0% | 30% | 784 | 1,568 |
| 10% | 5.0% | 50% | 282 | 564 |

### Step 4: Estimate Test Duration

If estimated daily traffic is available:
- Duration (days) = Total visitors needed / estimated daily visitors
- Duration (weeks) = Duration (days) / 7

If estimated daily traffic is unknown:
- Provide sample size only
- Include a duration lookup table:

| Daily Traffic | Duration for 5,000 total visitors | Duration for 10,000 | Duration for 20,000 |
|-------------|----------------------------------|---------------------|---------------------|
| 100/day | 50 days | 100 days | 200 days |
| 250/day | 20 days | 40 days | 80 days |
| 500/day | 10 days | 20 days | 40 days |
| 1,000/day | 5 days | 10 days | 20 days |
| 2,500/day | 2 days | 4 days | 8 days |

### Step 5: Flag High-Traffic Tests

Mark any test requiring >10,000 total visitors as **HIGH_TRAFFIC**.
Mark any test requiring >25,000 total visitors as **VERY_HIGH_TRAFFIC**.

For HIGH_TRAFFIC tests, recommend:
- Consider increasing the MDE (testing for a larger effect) to reduce sample size
- Consider running this test only on high-traffic pages
- Flag as Phase 2 or Phase 3 test

### Step 6: Build Testing Sequence Recommendation

Based on priority ranks, conflict map, and traffic requirements:

**Phase 1 (Run First):**
- Top 2-3 highest-priority tests that do NOT conflict with each other
- Prefer tests with lower sample size requirements (faster results)

**Phase 2 (Run After Phase 1):**
- Next-priority tests, including any that conflict with Phase 1 tests
- Tests that may need to be adjusted based on Phase 1 results

**Phase 3 (Run Last):**
- Lower-priority tests
- HIGH_TRAFFIC tests that require extended run time
- Tests dependent on Phase 1 or Phase 2 outcomes

**Low-traffic advisory (if estimated traffic <500/month):**
- Recommend running ONLY Phase 1 (top 2-3 tests)
- Recommend sequential testing (one at a time)
- Estimate total testing timeline (may be 6-12 months)
- Suggest considering a broader MDE to reduce required sample size

### Step 7: Write Sample Sizes

Compile results into `sample-sizes.md`.

---

## FAILURE MODES

- Baseline CVR unknown and no industry benchmark matches -> Use median 6.6% (Unbounce) as fallback. Note the uncertainty: "Sample sizes are estimates based on the industry median CVR. Actual required sample sizes may differ significantly based on the page's true conversion rate."
- Daily traffic unknown -> Provide sample sizes without duration estimates. Include the lookup table. Note: "Human reviewer must estimate traffic and calculate duration."
- All tests require HIGH_TRAFFIC -> Recommend running only the top 1-2 tests with the largest expected effect (larger MDE = smaller sample). Flag for human review: "All tests require significant traffic. Consider running one test at a time with relaxed MDE."
- Calculated sample size seems unreasonably large (>100K) -> Re-check the MDE. A very small MDE requires enormous samples. Recommend increasing the MDE to a practical level and document the trade-off.
