# 3.2 — Specificity Audit

**Skill:** LP-08-trust-elements
**Position:** Layer 3, Microskill 2
**Purpose:** Test every individual trust signal for specificity using a binary pass/fail per signal. This is the dedicated specificity gate — the 10-point audit in 3.1 checks specificity at the aggregate level, but this microskill tests each signal independently. A single generic signal = FAIL for this audit.
**Effort Level:** medium
**Dependencies:** All Layer 2 outputs, `trust-validation.md` (3.1)
**Output:** `specificity-audit.md`

---

## Execution Logic

1. **For each individual trust signal, apply the Specificity Test:**

   **THE TEST:** Can a reader distinguish this trust signal from a generic competitor's trust signal? If you could swap the brand name and the signal would still apply unchanged to any product in the category, it FAILS.

2. **Specificity criteria (signal must contain at least ONE):**
   - **A specific number:** "365-Day", "1,270,268+", "4.8/5", "83,502", "256-bit"
   - **A specific name:** "Dr. James DiNicolantonio", "Forbes", "NSF International", "Stripe"
   - **A specific timeframe:** "since 2019", "365 days", "within 24 hours"
   - **A specific certification name:** "GMP", "NSF", "Prop 65", "USDA Organic"
   - **A specific standard/protocol:** "pharmaceutical-grade", "peer-reviewed", "third-party"

3. **Signals that ALWAYS fail specificity (regardless of other content):**
   - "100% Satisfaction Guaranteed"
   - "Premium Quality"
   - "Trusted Brand"
   - "Best Seller" (without ranked source)
   - "Doctor Recommended" (without named doctor)
   - "Highest Quality"
   - "Your satisfaction is our priority"
   - Any signal from the LP-08 forbidden word list

4. **Test each signal independently.** Record:
   - Element ID
   - Element type (trust_bar, badge, certification, media, authority, volume, security)
   - Exact copy text
   - Specific detail present (Y/N)
   - What type of specific detail (number, name, timeframe, certification, standard)
   - Result: PASS or FAIL

5. **Calculate totals:**
   - Total signals tested
   - Total passed
   - Total failed
   - List of failed signal IDs

6. **Determine audit result:**
   - ZERO failed signals → PASS
   - ONE OR MORE failed signals → FAIL (list every failure with rewrite instruction)

7. **For each failed signal, provide a specific rewrite suggestion** that adds the missing specificity.

## Quality Criteria
- Every trust signal in the output tested individually (not assessed as a group)
- Binary result per signal: PASS or FAIL (no "mostly specific")
- Failed signals have specific rewrite suggestions (not just "make it more specific")
- Competitor swap test applied to trust bar signals
- Audit result is binary: any failure = FAIL for the audit overall

## Failure Modes
- Passing a signal that contains no specific detail → Apply the test strictly. "Premium Quality" has no number, name, timeframe, certification, or standard. It fails.
- Assessing specificity at the aggregate level only → Each signal must pass individually. A trust bar with 4 specific signals and 1 generic signal still fails on the generic one.
- Generic rewrite suggestions → "Make it more specific" is not a rewrite suggestion. "Change '100% Satisfaction Guaranteed' to '365-Day Money-Back Guarantee — No Questions Asked'" is a rewrite suggestion.
- Skipping the competitor swap test → This test catches signals that SEEM specific but are actually category-generic (e.g., "Third-Party Tested" without naming the lab).
