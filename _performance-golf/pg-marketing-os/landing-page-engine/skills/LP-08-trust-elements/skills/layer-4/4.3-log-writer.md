# 4.3 — Log Writer

**Skill:** LP-08-trust-elements
**Position:** Layer 4, Microskill 3
**Purpose:** Write `execution-log.md` — a complete record of LP-08's execution including every microskill that ran, spec files read, gate results, timing, and any issues encountered. This file is used for verification and debugging.
**Effort Level:** low
**Dependencies:** All previous microskill outputs
**Output:** `execution-log.md`

---

## Input Schema

```yaml
input:
  all_outputs: "[list of all files produced during LP-08 execution]"
  gate_results: "[GATE_0, GATE_1, GATE_2, GATE_3 results]"
  validation_results: "[from trust-validation.md and specificity-audit.md]"
  issues_encountered: "[any halts, revisions, or issues during execution]"
```

---

## Execution Logic

1. **Compile the execution record** by walking through each layer and microskill.

2. **For each microskill, document:**
   - Microskill ID and name
   - Spec file read before execution (Y/N)
   - Input dependencies met (Y/N)
   - Output file produced
   - Any issues or notes

3. **For each gate, document:**
   - Gate ID
   - All conditions checked
   - Result (PASS/FAIL)
   - If FAIL: what action was taken

4. **Document final output files** with existence confirmation.

5. **Write `execution-log.md`.**

## Quality Criteria
- Every microskill documented (even those that produced `not_applicable` output)
- Every gate result documented with all conditions
- Final output file existence confirmed
- Issues and revisions tracked (if any)
- Spec file reading documented per microskill

## Failure Modes
- Skipping microskills in the log → Every microskill must appear, even `not_applicable` ones.
- Not documenting gate failures → If a gate failed and required revision, both the failure and the revision must be logged.
- Writing the log before execution is complete → This is the LAST file written.
