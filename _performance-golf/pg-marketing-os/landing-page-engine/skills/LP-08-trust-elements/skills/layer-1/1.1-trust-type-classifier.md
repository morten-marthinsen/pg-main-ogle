# 1.1 — Trust Type Classifier

**Skill:** LP-08-trust-elements
**Position:** Layer 1, Microskill 1
**Purpose:** Classify all available trust signals from the trust inventory into the 7 trust element types. Determine which types are available, which are not applicable, and flag any gaps. This classification drives all downstream decisions about what trust copy gets written.
**Effort Level:** medium
**Dependencies:** `trust-inventory.md` (0.2), `packages-loaded.md` (0.1)
**Output:** `trust-classification.md`

---

## Input Schema

```yaml
input:
  trust_inventory: "[from trust-inventory.md — complete inventory]"
  page_type: "[from packages-loaded.md — type_a | type_b | hybrid]"
  vertical: "[from packages-loaded.md]"
  proof_strategy_mode: "[from packages-loaded.md — proof_saturation | authority_substitution]"
```

---

## Execution Logic

1. **Read `trust-inventory.md`** and classify each available signal into one of 7 trust element types:

   | Type | Code | Signals That Belong |
   |------|------|-------------------|
   | Trust Bar | `trust_bar` | Guarantee, certifications (badge form), volume (short form), shipping |
   | Media Mentions | `media_mention` | Publication features, press coverage, podcast appearances |
   | Certifications | `certification` | GMP, NSF, Prop 65, FDA-registered, ISO, SOC 2, COA |
   | Trust Badges | `trust_badge` | Guarantee badge, quality badge, shipping badge, security badge |
   | Security Signals | `security` | SSL, payment processor, encryption, privacy |
   | Authority Signals | `authority` | Named expert endorsements, advisory board, doctor endorsements |
   | Volume Signals | `volume` | Customer count, review count, units sold, subscriber count |

2. **Note signals that serve MULTIPLE types:**
   - Guarantee → trust_bar (short form) AND trust_badge (badge form) AND security (checkout)
   - Certifications → trust_bar (short form) AND certification (explanation form)
   - Review count → volume (number) AND rating strip context (Type B)

3. **Apply page type filter:**
   - **Type A:** Trust bar, media mentions, certifications (explanation form), authority signals, volume signals. Security signals minimal (only near order form).
   - **Type B:** Trust bar (aggressive), trust badges, rating strip, security signals, certifications (badge form), volume signals. Media mentions minimal (logo strip only).
   - **Hybrid:** Both Type A and Type B elements, weighted toward the dominant zone.

4. **Determine status for each type:**
   - `included` — signals available, will be written
   - `not_applicable` — no signals available for this type, skip in Layer 2
   - `requires_sourcing` — LP-05 identified a gap; signals needed but not yet available

5. **Flag gaps:** Any trust type that LP-05 architecture expects but the inventory cannot supply.

6. **Write `trust-classification.md`.**

## Quality Criteria
- Every signal from trust inventory classified into at least one type
- Multi-type signals identified (not forced into single classification)
- Status determination is accurate: `included` only when real data exists, `not_applicable` only when genuinely absent
- Gaps flagged with reference to LP-05 architecture expectations
- Page type filter applied (Type A vs Type B have different trust element priorities)

## Failure Modes
- Classifying a signal as `included` when the underlying data is vague or unverifiable → Only classify as `included` if specific, verifiable data exists.
- Missing multi-type signals → Guarantee data serves trust bar AND badges AND security. Missing this means incomplete downstream coverage.
- Ignoring LP-05 architecture expectations → If LP-05 planned a trust-type element in a section, LP-08 must either supply it or document the gap.
- Classifying everything as `not_applicable` to reduce work → Each classification must be justified against the actual inventory data.
