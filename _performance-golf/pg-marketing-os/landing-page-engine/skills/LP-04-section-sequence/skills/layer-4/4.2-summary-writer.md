# 4.2 — Summary Writer

**Skill:** LP-04-section-sequence
**Position:** Layer 4, Microskill 2
**Purpose:** Write `SECTION-SEQUENCE-SUMMARY.md` — a human-readable summary of the section sequence designed for quick review. This is the document the human reviewer reads to understand the page architecture at a glance before Phase 3 writing begins. It should communicate the "what and why" of the sequence in plain language, not just repeat the JSON structure.
**Effort Level:** low
**Dependencies:** `section-sequence.json` (4.1)
**Output:** `SECTION-SEQUENCE-SUMMARY.md`

---

## Input Schema

```yaml
input:
  section_sequence_json: "[from 4.1 — compiled section-sequence.json]"
  sequence_audit_score: "[from 3.3]"
  page_type: "[type_a | type_b | hybrid]"
  product_name: "[from brief]"
  awareness_stage: "[1-5]"
```

---

## Execution Logic

**Step 1:** Load `section-sequence.json`.

**Step 2:** Write the summary document with the following sections:
- **Page overview:** Product name, page type, awareness stage, total sections, target word count
- **Section sequence table:** Compact table showing section order, names, word counts, and key specs
- **Persuasion flow narrative:** 3–5 sentences describing the page's persuasion logic from above-fold to close — the "story" of how the page moves the visitor from awareness to purchase
- **Key architecture decisions:** Why optional sections were included or excluded, awareness-stage adjustments, any unusual structural choices
- **Proof wave summary:** Where proof is light, where it's heavy, and why
- **CTA placement summary:** Where CTAs appear and what drives each
- **Validation results:** Sequence score, logic test results, any noted issues
- **Downstream skill instructions:** Brief notes for Phase 3 writers about key constraints or emphases

**Step 3:** Keep the summary under 150 lines. This is a review document, not a specification. The specification is `section-sequence.json`.

**Step 4:** Write to `SECTION-SEQUENCE-SUMMARY.md` in the project output directory.

---

## Quality Criteria
- Summary is readable in under 2 minutes — compact, no redundancy
- Persuasion flow narrative communicates the "why" of the sequence, not just the "what"
- Key decisions documented — someone reviewing this should understand the architectural reasoning
- Under 150 lines total
- All data matches `section-sequence.json` — no discrepancies

## Failure Modes
- Summary exceeds 150 lines → Edit for conciseness. This is a summary, not a specification.
- Persuasion flow narrative is generic ("the page builds trust and then sells") → Rewrite with product-specific logic.
- Data in summary contradicts JSON → Fix the summary. JSON is the source of truth.
