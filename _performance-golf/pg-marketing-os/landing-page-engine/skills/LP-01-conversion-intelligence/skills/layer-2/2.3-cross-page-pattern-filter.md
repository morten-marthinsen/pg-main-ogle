# 2.3: Cross-Page Pattern Filter

> **Skill:** LP-01-conversion-intelligence
> **Layer:** 2
> **Position:** 3 of 4
> **Purpose:** Filter cross-page-pattern-analysis.md to extract only the patterns relevant to this specific page type + vertical build, discarding patterns that apply to other page types or verticals.
> **Effort Level:** sonnet
> **Dependencies:** `brief-extract.md` (from 0.1), cross-page-pattern-analysis.md
> **Output:** `filtered-patterns.md`

---

## INPUT

```yaml
input:
  brief_extract: "brief-extract.md"
  cross_page_analysis: "LandingPageEngine/specimens/cross-page-pattern-analysis.md"
  page_type: "[type_a | type_b | hybrid]"
  primary_vertical: "[vertical_id]"
  traffic_temperature: "[cold | warm | hot]"
  awareness_stage: "[1-5]"
```

---

## EXECUTION

1. **Read cross-page-pattern-analysis.md** in full. This contains 9 major analysis sections across 11 specimens.

2. **Filter Section 1: Section Sequence Patterns** by page type.
   - For Type A builds: extract the Type A Universal Skeleton + Type A detailed sequence mapping + Type A key observations
   - For Type B builds: extract the Type B Universal Skeleton + Type B detailed sequence mapping + Type B key observations
   - For Hybrid builds: extract both skeletons, note which applies to which zone
   - Extract the Universal Sections table (elements appearing in 8+/11 pages) and Optional Sections table

3. **Filter Section 2: Proof Architecture Patterns** by build context.
   - Extract the proof density distribution — identify where this build should fall
   - Extract first proof position data relevant to this page type
   - Extract proof type frequency table
   - Note the "Bimodal Proof Distribution" finding — the build must commit to proof-saturated OR authority-substitution, not middle ground (unless it's a mid-ticket subscription)

4. **Filter Section 3: CTA Architecture Patterns** by page type.
   - Extract CTA count distribution for similar page types
   - Extract CTA text consistency patterns
   - Extract risk reversal at CTA patterns

5. **Filter Section 4: Offer Structure Patterns** by build context.
   - Extract relevant offer models (based on the build's pricing model)
   - Extract pricing psychology techniques applicable to this build
   - Extract guarantee spectrum data

6. **Filter Section 5: Headline Patterns** by page type + traffic source.
   - Extract headline type distribution
   - Note that curiosity/secret headlines are absent from the dataset (all specimens are direct/outcome-first, consistent with Facebook ad traffic)

7. **Filter Section 7: Vertical Distribution** for vertical-specific patterns.
   - Extract vertical-specific section requirements (e.g., health/supplements need Ingredient Breakdown, Scientific Citations, Mechanism Explanation)
   - Extract dominant patterns for this vertical

8. **Filter Section 8: Key Strategic Insights** for build-relevant insights.
   - Include insights relevant to this page type + vertical
   - Exclude insights about page types or verticals not matching this build

9. **For each filtered pattern, assess applicability:**
   - DIRECTLY APPLICABLE: Pattern confirmed across 2+ specimens matching this build
   - LIKELY APPLICABLE: Pattern observed in related specimens, reasonable inference
   - REFERENCE ONLY: Pattern from a different context, included for awareness

10. **Write filtered-patterns.md** with only the relevant patterns.

## QUALITY CRITERIA

- Every pattern traced to a specific section in cross-page-pattern-analysis.md
- No patterns invented — all must appear in the source file
- Filtering is honest — patterns excluded are documented with reasons
- Applicability levels (DIRECTLY APPLICABLE, LIKELY APPLICABLE, REFERENCE ONLY) are consistently applied
- Vertical-specific section requirements accurately extracted from Section 7
- Strategic insights filtered to this build's context, not a raw dump

---

## FAILURE MODES

- **Page type not covered in cross-page-pattern-analysis.md:** The analysis covers Type A and Type B. If the build is Hybrid, combine patterns from both and note zone applicability. If it's a page type not in the analysis, document the gap.
- **Vertical not represented in Section 7:** Not all verticals have specimens. If this vertical has no dedicated analysis, note the gap and use the closest vertical's patterns as LIKELY APPLICABLE.
- **Over-filtering:** Do not exclude patterns just because they come from a different vertical. If a pattern is universal (e.g., FAQ placement), include it with DIRECTLY APPLICABLE. Only exclude patterns that are specifically tied to a non-matching context.
- **Under-filtering:** Do not include every pattern from the analysis. The purpose is filtering. If 60%+ of the original analysis makes it through, the filtering was too loose.
