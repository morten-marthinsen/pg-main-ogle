# 2.1: Hook-Body Coherence

## Purpose

For EVERY valid combination, evaluate whether the body DELIVERS on what the hook PROMISES. The hook creates an expectation (curiosity gap, pain identification, result promise, question). The body must fulfill that expectation. This is the first of four coherence dimensions and runs in parallel with 2.2 and 2.3.

**This is the core value of A09.** Assembly without coherence validation produces broken ads where the hook promises one thing and the body delivers another. The model uses opus for this task because it requires deep semantic reasoning about promise-fulfillment alignment.

## Inputs

- **RAW-COMBINATIONS.yaml:** From 1.1 -- full combination details (hook text, body summary)
- **INCOMPATIBILITY-LOG.yaml:** From 1.2 -- to identify which combinations are VALID (not removed)
- **PLATFORM-MAPPING.yaml:** From 1.3 -- to exclude BLOCKED combinations
- **COPY-CATALOG.yaml:** From 0.1 -- full hook text and body content for evaluation

## Process

```
STEP 1: BUILD EVALUATION SET
  - From INCOMPATIBILITY-LOG.yaml and PLATFORM-MAPPING.yaml:
    - Include all combinations with status VALID (not removed, not blocked)
    - Include FLAGGED combinations (they proceed to coherence)
    - Include NEEDS_ADAPTATION combinations (adaptation is production, not coherence)
  - Load full hook text and body content from COPY-CATALOG.yaml for each combination
  - Count total combinations to evaluate

STEP 2: FOR EACH COMBINATION, EVALUATE HOOK-BODY COHERENCE
  Apply the hook type category rules:

  CATEGORY A (Curiosity/Information Gap):
    - Hook creates a curiosity gap or information promise
    - Body MUST answer the question or close the gap
    - FAIL if body discusses an unrelated topic

  CATEGORY B (Authority/Social Proof):
    - Hook presents an authority figure or social proof
    - Body MUST present the authority's claim or proof
    - FAIL if body ignores the authority entirely

  CATEGORY C (Problem/Pain):
    - Hook names a specific problem or pain point
    - Body MUST address the specific problem named in hook
    - FAIL if body pivots to a different problem

  CATEGORY D (Transformation/Results):
    - Hook promises a transformation or specific results
    - Body MUST show or explain the transformation
    - FAIL if body discusses theory without results

  CATEGORY E (Identity/Belonging):
    - Hook speaks to a specific identity group
    - Body MUST speak to the identified group
    - FAIL if body addresses a different audience

  CATEGORY F (Pattern Interrupt):
    - Hook interrupts with something unexpected
    - Body MUST resolve the interrupt with relevance to product/offer
    - FAIL if body stays absurd without connecting to product

  CATEGORY G (Platform-Native):
    - Hook uses platform-native format conventions
    - Body MUST match the native format's expectations
    - FAIL if body breaks platform conventions

  CATEGORY H (Scarcity/Urgency):
    - Hook creates urgency or scarcity
    - Body MUST justify the urgency with a real reason
    - FAIL if body provides no urgency justification

  CATEGORY I (Value/Education):
    - Hook promises value or educational content
    - Body MUST deliver the promised value/education
    - FAIL if body withholds value behind a hard sell

  CATEGORY J (Story/Narrative):
    - Hook opens a story arc
    - Body MUST continue the story arc
    - FAIL if body abandons the narrative

STEP 3: ASSIGN VERDICT
  For each combination:
    - PASS: Body clearly fulfills the hook's promise. The viewer who clicks
      after the hook will receive what was implied.
    - FAIL: Body delivers something different from what the hook promises.
      The viewer will feel misled or confused. Record specific disconnect.
    - FLAG: Borderline case -- the body partially fulfills the hook's promise
      or the connection is indirect. Needs human judgment.
      Record specific concern and what would make it PASS.

  Each verdict MUST include:
    - The hook type category (A-J)
    - What the hook promises (specific text)
    - What the body delivers (specific text)
    - Why the verdict is PASS/FAIL/FLAG (specific reasoning)

STEP 4: DO NOT BATCH-EVALUATE
  CRITICAL: Each combination gets INDIVIDUAL evaluation with specific reasoning.
  "These 40 are all coherent" is FORBIDDEN.
  "Probably coherent" is not a valid assessment.
  Every combination must have its own verdict with its own reasoning.

STEP 5: COMPUTE COHERENCE STATISTICS
  - Total evaluated
  - Total PASS
  - Total FAIL
  - Total FLAG
  - Pass rate (%)
  - Most common failure pattern
  - Most common hook category failing

STEP 6: WRITE HOOK-BODY-COHERENCE.yaml
  - Write per-combination verdicts with full reasoning
  - Write statistics
```
