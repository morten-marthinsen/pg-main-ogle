# 2.2-variation-expander.md

> **Layer:** 2 (Generation)
> **Skill:** E2-subject-line-engine
> **Purpose:** Expand top 10 candidates into 2-3 variants each to maximize the quality ceiling
> **Outputs:** `expanded_pool` object with original candidates plus variants

---

## FUNCTION

Take the top 10 candidates from the volume generation pool (2.1) and create 2-3 carefully crafted variants of each. Variations explore different word choices, structural tweaks, and tonal shifts while preserving the core idea. This expansion phase increases the probability of finding the optimal expression of each strong concept.

**Key Insight:** The best subject line is often a VARIANT of a good concept, not the original phrasing. A strong SL-HOW concept might have 3 expressions -- one casual, one authoritative, one edgy -- and the best version only emerges through exploration.

---

## VARIATION TECHNIQUES

| Technique | Definition | Example |
|-----------|-----------|---------|
| **Word Swap** | Replace 1-2 words while keeping structure | "The email assassin" --> "The email butcher" |
| **Tonal Shift** | Same content, different emotional register | "How to write emails" --> "How a lazy man writes emails" |
| **Compression** | Same meaning in fewer words | "The secret way to double open rates" --> "The open rate doubler" |
| **Expansion** | Add one specificity element | "The email trick" --> "The 3-second email trick" |
| **Reframe** | Same concept, different angle of entry | "Why emails fail" --> "The email graveyard" |
| **Formula Shift** | Same concept, different formula category | SL-HOW to SL-THE: "How to write better" --> "The better email" |

## EXECUTION PROTOCOL

```
EXECUTE 2.2-variation-expander:
  INPUT: candidate_pool (from 2.1), email_context, niche_calibration
  OUTPUT: expanded_pool object

  1. RANK candidates from 2.1 by combined preliminary scores
     (alignment + originality + word count status)

  2. SELECT top 10 candidates for expansion
     CRITERIA:
       - Preliminary alignment >= "medium"
       - Not rejected for slop or length
       - Represent at least 4 different formula categories

  3. FOR each of the top 10 candidates:
     a. ANALYZE the core concept (what is this SL really about?)
     b. IDENTIFY the strongest element (what makes this SL work?)
     c. IDENTIFY the weakest element (what could be better?)

     d. GENERATE Variant A: Word Swap or Compression
        - Keep the core concept and strongest element
        - Replace the weakest element
        - Target sweet spot word count (6-7)

     e. GENERATE Variant B: Tonal Shift
        - Same concept, different emotional register
        - If original is casual, make it authoritative (or vice versa)
        - If original is subtle, make it bold (or vice versa)

     f. GENERATE Variant C (optional): Reframe or Formula Shift
        - Same concept, completely different structural approach
        - Only generate if a clear alternative angle exists
        - Do not force -- quality over quantity

     g. FOR each variant:
        - TAG with parent SL ID (e.g., "SL-003-A", "SL-003-B")
        - TAG with variation technique used
        - COUNT words
        - PRELIMINARY alignment check
        - PRELIMINARY originality check

  4. QUALITY FILTER expanded variants:
     a. REJECT any variant worse than its parent
     b. REJECT any variant that is essentially identical to parent (< 2 word changes)
     c. REJECT any variant that violates anti-slop rules
     d. REJECT any variant over 10 words

  5. MERGE expanded variants into candidate pool:
     - Original 20-50 candidates from 2.1
     - + 20-30 expansion variants from this step
     - = Total pool of 40-80 candidates for Arena

  6. FINAL CANDIDATE COUNT CHECK:
     IF total < 30: Generate additional variants from candidates ranked 11-15
     IF total > 80: Trim lowest-quality candidates

  7. ASSEMBLE expanded_pool object
  8. RETURN expanded_pool
```

---

## EXPANSION RULES

1. **Never make variants worse** -- Every variant must be at LEAST as strong as its parent
2. **2 minimum, 3 maximum** -- Don't over-expand. Diminishing returns after 3 variants.
3. **Preserve the core insight** -- The variant changes expression, not the underlying concept
4. **Maintain formula integrity** -- A variant of a SL-HOW should still be recognizable as SL-HOW
5. **Compression over expansion** -- When in doubt, make it shorter, not longer
6. **Don't repeat the pool** -- Each variant must be meaningfully different from ALL other candidates

## FAILURE MODES

| Mode | Trigger | Remediation |
|------|---------|-------------|
| CRITICAL | Fewer than 10 candidates to expand | Expand from all available candidates |
| CRITICAL | Total pool < 30 after expansion | Expand candidates ranked 11-20 |
| DEGRADED | Most variants worse than parents | Reduce to 1 variant per parent |
| DEGRADED | All variants same technique | Force technique diversity |
