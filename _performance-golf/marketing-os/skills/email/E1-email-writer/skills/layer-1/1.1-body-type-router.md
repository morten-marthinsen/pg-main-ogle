# 1.1-body-type-router.md

> **Layer:** 1 (Type-Specific Architecture)
> **Skill:** E1-email-writer
> **Purpose:** Route to correct structural template based on assigned body type (CT/QO/TM/QA/LB/ST/NR)
> **Inputs:** `email_context.assignment.body_type` from 0.1, `structural_specimens` from 0.2
> **Outputs:** `routing_decision` with body type confirmation, structural requirements, and generation constraints

---

## FUNCTION

Confirm the assigned body type, load the structural requirements for that type, and establish the generation constraints that all downstream microskills (1.2, 1.3, 2.1-2.4) must follow. This is a ROUTING skill, not a DECISION skill -- the body type is assigned by the E0 blueprint and CANNOT be changed here.

**Key Principle:** Body type is assigned, not selected. This microskill routes execution to the correct structural path. If the body type feels wrong for the content, the feedback goes to E0 (strategist), not to this skill.

## EXECUTION PROTOCOL

```
EXECUTE 1.1-body-type-router:

  STEP 1: Confirm Body Type
    1. READ email_context.assignment.body_type
    2. VERIFY body type is one of [CT, QO, TM, QA, LB, ST, NR]
    3. IF body type invalid: FAIL -- "Invalid body type. Check blueprint."

  STEP 2: Load Structural Requirements
    1. LOAD routing table entry for assigned body type (from above)
    2. EXTRACT all structural requirements
    3. VERIFY structural specimens from 0.2 match this body type

  STEP 3: Check Frequency Constraints
    IF body_type = NR:
      1. CHECK sequence context for other NR emails within 7 positions
      2. IF another NR within 7 positions: WARN -- "NR frequency constraint may be violated"

  STEP 4: Set Generation Constraints
    1. COMPILE constraints from routing table:
       - Opening type and max lines
       - Body structure and section count
       - Bridge type and purpose
       - CTA pattern (from blueprint)
       - Word count range
       - Tone
       - Paragraph max
    2. These constraints are BINDING for microskills 1.2, 1.3, 2.1-2.4

  STEP 5: Assemble routing_decision
  STEP 6: Return routing_decision
```

## FAILURE MODES

| Mode | Trigger | Remediation |
|------|---------|-------------|
| CRITICAL | Invalid body type code | HALT -- Verify E0 blueprint |
| CRITICAL | Body type mismatch with specimens | HALT -- Return to 0.2 and reload |
| WARNING | NR frequency constraint risk | LOG -- Flag for E0 strategist review |
| PROTOCOL VIOLATION | Attempting to change body type | HALT -- Body type is assigned by E0 |

---

## ANTI-PATTERNS (FORBIDDEN)

1. **Changing body type** -- The body type is ASSIGNED by E0. This skill routes, it does not decide.
2. **Softening structural requirements** -- "Close enough" body type structure is not acceptable. CT must follow CT structure.
3. **Skipping frequency check for NR** -- NR emails have a 1-per-7 constraint that must be verified.
4. **Ignoring specimen alignment** -- If specimens don't match body type, something went wrong in 0.2.
