# Layer 1.1 — Goal Analyzer

## Purpose

Determine the campaign's primary goal and map it to one of 7 campaign types using a structured decision tree. The campaign type selection drives everything downstream — template loading, sequence length, body type distribution, emotional arc shape, and urgency calibration.

---

## Input Requirements

```yaml
from_layer_0:
  mode: enum[A, B]
  context_richness:
    score: integer
    complexity_level: enum[minimal, standard, full, complex]
    viable_campaign_types: [string]

  # Mode A inputs:
  upstream_context:
    offer:
      product_name: string
      positioning: string
    structure:
      campaign_thesis: string
    research:
      audience_intelligence: object

  # Mode B inputs:
  brief_context:
    goal:
      primary_goal: string
      campaign_type_hint: string
      urgency_info: string
    audience:
      relationship_stage: string
```

---

## Execution Protocol

### Step 1: Extract Goal Signal

**Mode A:** Infer the campaign goal from upstream packages:
- If offer is a NEW product (no existing customer base) → likely LAUNCH
- If offer is positioned as a partner/affiliate → likely AFFILIATE
- If structure emphasizes education → likely CONTENT SERIES
- If research indicates cold audience → likely AUTORESPONDER or CONTENT SERIES
- If campaign thesis is urgency-driven → likely DEADLINE or LAUNCH

**Mode B:** Use the explicit `primary_goal` field from the brief:
- Match user's stated goal language to campaign type codes
- Use `campaign_type_hint` from 0.2-brief-parser as starting point

### Step 2: Apply Campaign Type Decision Tree

```
WHAT IS THE PRIMARY GOAL?

├── "Launch a new product" OR "introduce a new offer"
│   └── PRODUCT LAUNCH (PL)
│       Template: launch-sequence.md
│       Duration: 5-7 days, 12-17 emails
│       Prerequisite: context_richness >= 6
│
├── "Promote a partner's product" OR "affiliate promotion"
│   └── AFFILIATE LAUNCH (AL)
│       Template: affiliate-promotion.md
│       Duration: 3-5 days, 5-7 emails
│       Prerequisite: context_richness >= 4
│
├── "Nurture new subscribers" OR "welcome sequence"
│   └── AUTORESPONDER (AR)
│       Template: autoresponder-series.md
│       Duration: 14-30 days, 14-30 emails (evergreen)
│       Prerequisite: context_richness >= 5
│
├── "Build ongoing relationship" OR "daily emails"
│   └── DAILY RELATIONSHIP (DR)
│       Template: daily-relationship.md
│       Duration: Ongoing, 1 email/day
│       Prerequisite: context_richness >= 5
│
├── "Educate / build trust" OR "content series"
│   └── CONTENT SERIES (CP)
│       Template: content-series.md
│       Duration: 5-7 days, 5-7 emails
│       Prerequisite: context_richness >= 3
│
├── "Single promotional push" OR "one-off promo"
│   └── BLATANT PITCH (BP)
│       Template: one-off-promotional.md
│       Duration: 1 day, 1 email
│       Prerequisite: context_richness >= 1
│
└── "Monthly deadline reminder" OR "deadline/expiration"
    └── DEADLINE URGENCY (DU)
        Template: none (single email)
        Duration: 1 day, 1 email
        Prerequisite: context_richness >= 1
```

### Step 3: Validate Against Context Richness

Cross-reference selected campaign type against `viable_campaign_types` from the validation report:

```
IF selected_type NOT IN viable_campaign_types:
  → Log warning: "Selected campaign type [X] requires context_richness >= [Y],
    but current score is [Z]."
  → Recommend downgrade: suggest the most complex viable type instead
  → DO NOT auto-override — flag for human decision at Gate 1
```

### Step 4: Generate Justification

Document the selection rationale:
- What goal signals were detected
- Why this campaign type (not alternatives) matches the goal
- Any context richness concerns
- Any audience stage considerations (cold audience → lean toward CP/AR)

### Step 5: Identify the Template File

Map campaign type to its template file path:

```yaml
template_mapping:
  PL: "templates/launch-sequence.md"
  AL: "templates/affiliate-promotion.md"
  AR: "templates/autoresponder-series.md"
  DR: "templates/daily-relationship.md"
  CP: "templates/content-series.md"
  BP: "templates/one-off-promotional.md"
  DU: null  # No template — single email
```

## Output

```yaml
goal_analysis:
  detected_goal: string  # the goal as understood from inputs
  campaign_type: enum[PL, AL, AR, DR, CP, BP, DU]
  campaign_type_name: string  # full name
  template_file: string  # path to template, or null for DU
  justification: string  # why this type was selected
  context_richness_check:
    required_minimum: integer
    actual_score: integer
    passes: boolean
    downgrade_recommendation: string  # null if passes

  duration:
    days: integer  # or "ongoing" for DR
    email_count_range: string  # e.g., "12-17"

  confidence: enum[high, medium, low]
  confidence_notes: string  # what would increase confidence

  alternatives_considered:
    - type: string
      why_rejected: string
```

---

## Failure Modes

| Failure | Remediation |
|---------|-------------|
| Goal is ambiguous (could map to multiple types) | Present top 2 candidates with trade-offs. Flag for human decision at Gate 1. |
| Goal doesn't match any campaign type | Default to Content Series (CP) — safest option. Flag for human review. |
| Context richness too low for selected type | Recommend downgrade with explanation. Do not auto-select. |
| User explicitly requests a type that context can't support | Warn about quality ceiling. "A Product Launch with context_richness 3 will produce a thin campaign. Recommend Content Series instead, or provide additional brief detail." |
