# Layer 2.2 -- Revision Executor

## Purpose

Execute the revision plan from 2.1-issue-clustering.md. For P1/P2 issue clusters, run full Arena (3 rounds, 7 competitors, adversarial critique). For P3/P4 issue clusters, apply direct fixes. Track every revision applied with before/after text for the editorial report.

**CRITICAL:** This is the highest-effort microskill in E4. Arena runs for P1/P2 issues require `effort: max`. Every revision must preserve voice, flow, and threading.

## Execution Protocol

### Step 1: Validate Revision Plan

Before executing any revisions, verify the plan:

```
VALIDATION:
  [ ] All P1 clusters assigned to Arena
  [ ] All P2 clusters assigned to Arena
  [ ] No P1/P2 issues assigned to direct fix
  [ ] Execution order follows priority (P1 first → P4 last)
  [ ] Total clusters <= 15 (manageable scope)

IF validation fails:
  HALT -- Return to 2.1-issue-clustering for re-clustering.
```

### Step 2: Execute Arena for P1/P2 Clusters

For each cluster with `resolution_method = "arena"`:

```
ARENA EXECUTION PER CLUSTER:

  1. PREPARE ARENA CONTEXT:
     - Load affected emails (full text)
     - Load the specific issue(s) to fix
     - Load the email's body type template (for structural reference)
     - Load voice fingerprint (from 1.2 voice consistency)
     - Load email-quality-rubric.md criteria relevant to the issue

  2. ARENA BRIEF (sent to all 7 competitors):
     "You are revising [N] emails from an email campaign.
      The issue: [specific issue description]
      Affected emails: [position numbers]

      FOR EACH affected email:
        Current text: [full email body]
        Issue: [specific problem in this email]

      CONSTRAINTS:
        - Fix the specific issue identified
        - Preserve the persona voice throughout
        - Preserve the content-to-pitch ratio
        - Preserve the body type structure
        - Do NOT rewrite the entire email -- surgical fix only
        - The revised email must feel like it was always this way

      DELIVER: Revised version of each affected email with the issue fixed."

  3. RUN 3 ROUNDS:
     Round 1: 7 competitors generate revised emails
     → Critic identifies weakest fix per competitor
     → Targeted revision
     → Scoring against issue-specific criteria

     Round 2: Learning Brief distributed
     → 7 competitors re-generate with learnings
     → Critique → Revision → Scoring

     Round 3: Cumulative Learning Brief
     → 7 competitors generate FINAL versions
     → Final scoring

  4. POST-ARENA SYNTHESIS:
     Architect creates 2-3 hybrid revisions from Round 3 outputs.

  5. HUMAN SELECTION (if scope warrants):
     For P1 clusters: Present candidates to human.
     For P2 clusters: Auto-select highest-scoring if score >= 8.5.
     If no candidate >= 7.5: Flag for human intervention.

  6. APPLY WINNING REVISION:
     Update the email(s) in the working sequence.
     Record before/after text.
     Record Arena scores.
```

### Arena Scoring Criteria (Email-Specific)

| Criterion | Weight | What It Measures |
|-----------|--------|------------------|
| Issue Resolution | 25% | Does the revision actually fix the identified problem? |
| Voice Preservation | 20% | Does the fix maintain persona voice throughout? |
| Flow Preservation | 15% | Does the fix maintain reading momentum? |
| Ratio Preservation | 10% | Does the fix maintain content-to-pitch balance? |
| Bridge Smoothness | 10% | If bridge-related, is it now seamless? |
| Engagement Improvement | 10% | Is the revised version more engaging? |
| Niche Appropriateness | 10% | Does the fix stay niche-appropriate? |

### Step 3: Execute Direct Fixes for P3/P4 Clusters

For each cluster with `resolution_method = "direct_fix"`:

```
DIRECT FIX EXECUTION PER CLUSTER:

  FOR each affected email in the cluster:

    1. READ the current email text
    2. IDENTIFY the specific fix needed (from cluster fix_instructions)
    3. APPLY the fix:

       PARAGRAPH BREVITY (P3/P4):
         Split paragraphs > 3 sentences into shorter ones.
         Maintain sentence meaning. Do not reword.

       MINOR VOICE WOBBLE (P3):
         Identify 2-3 off-register phrases.
         Replace with voice-consistent alternatives.
         Keep the semantic meaning identical.

       CTA STRENGTHENING (P3):
         Rewrite CTA only (not the rest of the email).
         Make action clearer and emotional appeal stronger.

       SIGN-OFF CONSISTENCY (P3/P4):
         Standardize sign-off format across all emails.

       WORD COUNT ADJUSTMENT (P3):
         If too long: Trim redundant sentences (not key content).
         If too short: Expand thin sections with additional detail.

       FORMATTING (P4):
         Fix spacing, line breaks, formatting inconsistencies.

       MINOR WORD CHOICE (P4):
         Swap individual words for better alternatives.
         Do not change sentence structure.

    4. RECORD before/after text for each fix
    5. VERIFY fix doesn't introduce new issues:
       - Voice still consistent?
       - Ratio still in range?
       - Body type structure intact?
```

### Step 4: Cross-Check Revisions for Side Effects

After ALL revisions (Arena + direct fix) are applied:

```
SIDE EFFECT CHECK:

  1. VOICE DRIFT:
     Did Arena revisions to some emails create voice inconsistency
     with non-revised emails? If yes: flag for additional direct fix.

  2. RATIO SHIFT:
     Did adding/removing content to fix an issue change the ratio?
     If any email now violates ratio: flag for direct fix.

  3. CONTINUITY BREAK:
     Did revisions remove any cross-email callbacks or references?
     If yes: restore or recreate the callback.

  4. SEQUENCE FLOW:
     Read emails in sequence order. Do revised emails still flow
     naturally from previous email and into next email?
     If flow breaks: flag for direct fix.
```

## Failure Modes

| Failure | Remediation |
|---------|-------------|
| Arena fails to produce revision above 7.5 for P1 issue | Run one additional Arena round with explicit constraint. If still failing, flag for human intervention -- do not auto-pass. |
| Direct fix introduces a new P2 issue | Escalate the new issue to Arena treatment. |
| Side effects create more issues than revisions fixed | Flag for human: "Editorial revisions are creating instability. Recommend returning to E1 for regeneration of affected emails." |
| Arena context too large (>8 emails per cluster) | Clusters should have been split in 2.1. If missed, split now and run separate Arena instances. |
| Competitor revisions all feel like rewrites instead of fixes | Remind in Arena brief: "Surgical fix ONLY. Do not rewrite. The reader should not be able to tell which part was revised." |
