# Layer 4.2 — Threading Guide Generator

**Skill:** 09 — Campaign Brief
**Microskill:** 4.2 — Threading Guide Generator
**Layer:** 4
**Model:** sonnet
**Purpose:** Generate the section threading guide — open loops, recurring elements, and transition rules that create narrative continuity across all campaign sections.

---

## Input

| Source | File | Required |
|--------|------|----------|
| 4.1-campaign-brief-assembler | `campaign-brief.json` (assembled brief without threading) | Yes |
| 3.3-emotional-arc-designer | `3.3-emotional-arc.json` (emotional transitions inform threading) | Yes |

| File | Format | Min Size |

## Execution Steps

### Step 1: Design Open Loops (Minimum 3)

Identify 3-5 open loops to plant in early sections that resolve in later sections. Each loop creates an information gap the prospect needs closed:

```yaml
open_loops:
  - loop_id: string
    loop_description: string      # The specific information gap created
    loop_trigger: string          # The specific line or concept that opens the loop
    planted_in: string            # Section where loop is opened (e.g., "Lead", "Story")
    resolved_in: string           # Section where loop closes (e.g., "Mechanism Narrative", "Offer")
    purpose: string               # Why this loop exists — what attention it sustains
    specificity_check: boolean    # Is the loop specific enough? (see anti-patterns)
```

**Loop design rules:**
- Loops must create SPECIFIC information gaps, not vague curiosity
- Each loop must reference a concrete element from the campaign (mechanism name, root cause detail, proof data point)
- Loops should resolve in the section that naturally contains the answer
- Distribution: at least 1 loop planted in Lead, 1 in Story, 1 in Root Cause or Mechanism section

**Bad loop:** "We'll reveal the mechanism later" (vague)
**Good loop:** "What's the one compound that [Expert Name] discovered in [specific context] that reversed [specific condition] in [timeframe]?" (specific)

### Step 2: Design Recurring Elements (Minimum 2)

Identify 2-3 concepts, phrases, or images that should recur across sections to create narrative unity:

```yaml
recurring_elements:
  - element_id: string
    element: string               # A phrase, concept, or image that recurs
    element_type: string          # (phrase, concept, image, metaphor, data_point)
    first_appearance: string      # Section where it first shows up
    recurrences:
      - section: string
        variation: string         # How the element appears differently each time
        purpose: string           # Why it recurs here
    unity_function: string        # How this element makes the campaign feel connected
```

**Recurring element design rules:**
- Elements should evolve slightly each time (not exact repetition)
- At least one element should be tied to the Big Idea's core concept
- At least one element should be tied to the mechanism name
- Recurrences should feel like callbacks, not redundancy

### Step 3: Design Transition Rules

For each section-to-section boundary, specify the transition approach:

```yaml
transition_rules:
  - from_section: string
    to_section: string
    transition_type: string       # (bridge, callback, escalation, pivot, reveal)
    guidance: string              # Specific direction for how to execute this transition
    emotional_shift: string       # What emotional change happens at this boundary
```

**Transition types (use only these):**
- **Bridge:** Smooth continuation — "And that's why..."
- **Callback:** Reference to earlier element — "Remember when I said..."
- **Escalation:** Raising stakes — "But it gets worse..." / "And not just that..."
- **Pivot:** Changing direction — "But here's what nobody tells you..."
- **Reveal:** Payoff of earlier setup — "And that's when I discovered..."

**Required transitions:**
1. Lead -> Story
2. Story -> Root Cause Narrative
3. Root Cause Narrative -> Mechanism Narrative
4. Mechanism Narrative -> Product Introduction
5. Product Introduction -> Offer Copy
6. Offer Copy -> Close

### Step 4: Validate Threading Completeness

```
CHECK 1: open_loops.count >= 3
  IF FAIL: Generate additional loops until minimum met

CHECK 2: recurring_elements.count >= 2
  IF FAIL: Generate additional elements until minimum met

CHECK 3: All 6 section transitions have rules
  IF FAIL: Generate missing transition rules

CHECK 4: No vague loops (each loop has specific information gap)
  IF FAIL: Revise vague loops to be specific

CHECK 5: Transition types are from approved list only
  IF FAIL: Reclassify to approved types
```

### Step 5: Integrate Threading into Campaign Brief

Add the threading guide to campaign-brief.json:

```yaml
threading_guide:
  open_loops: [object]         # from step 1
  recurring_elements: [object] # from step 2
  transition_rules: [object]   # from step 3
```

Also update campaign-brief-summary.md with a readable threading section.

## Quality Gate

- [ ] Minimum 3 open loops designed with specific information gaps
- [ ] Each open loop references concrete campaign elements (mechanism name, root cause detail, proof)
- [ ] Open loops distributed: at least 1 in Lead, 1 in Story, 1 in later section
- [ ] Minimum 2 recurring elements designed with first appearance and recurrences
- [ ] At least 1 recurring element tied to Big Idea, 1 tied to mechanism
- [ ] All 6 section-to-section transitions have rules
- [ ] Transition types are from approved list only (bridge, callback, escalation, pivot, reveal)
- [ ] No vague loops — all loops are specific enough to create real information gaps
- [ ] Threading guide integrated into campaign-brief.json
- [ ] campaign-brief-summary.md updated with threading section

---

## Anti-Patterns

- **Vague loops:** "We'll reveal the mechanism later" — REJECT. Must be specific: "What's the [specific thing] that [specific person] discovered?"
- **Exact repetition:** Using the same phrase verbatim everywhere — this is redundancy, not threading. Elements should EVOLVE.
- **Missing transitions:** Any section boundary without a transition rule — the campaign will feel disjointed
- **Wrong transition types:** Using types not in the approved list — keep to the 5 types for consistency
