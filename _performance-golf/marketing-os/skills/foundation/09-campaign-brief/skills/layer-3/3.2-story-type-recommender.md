# Layer 3.2 — Story Type Recommender

**Skill:** 09 — Campaign Brief
**Microskill:** 3.2 — Story Type Recommender
**Layer:** 3
**Model:** opus
**Purpose:** Recommend story type based on proof asset strength, mechanism type, narrative needs, and Big Idea — with vault precedent and alternatives for human selection.

## Execution Steps

### Step 1: Assess Proof Asset Strength Profile

Analyze the proof inventory to determine what proof types are available and their strength:

```
PROOF ASSET STRENGTH drives primary recommendation:
  Strong clinical/scientific proof    -> Discovery Story or Revelation Story
  Strong testimonial proof            -> Transformation Story or Proof Story
  Strong logical/demonstrable proof   -> Live Experiment Story
  Limited proof assets                -> Origin Story or Warning Story
```

Extract proof_inventory.by_category and proof_inventory.overall_strength from upstream context. Map the strongest proof category to the corresponding story type.

### Step 2: Determine Narrative Needs

Analyze what the campaign's strategic elements require from the story:

```
NARRATIVE NEEDS modify recommendation:
  Root cause requires blame externalization    -> Conspiracy Story or Revelation Story
  Mechanism requires credibility building      -> Doctor-Discovery Story or Authority Story
  Big Idea requires emotional buy-in           -> Vulnerability Story or Transformation Story
  Market is skeptical                          -> Proof Story with live experiment framing
  Product has compelling origin                -> Origin Story
```

Cross-reference:
- `root_cause.villain_profile` — does the villain framing demand a conspiracy/revelation narrative?
- `mechanism.type` and `mechanism.e5_type` — does the mechanism need authority establishment?
- `big_idea.emotional_core` — does the Big Idea's emotion need a story to carry it?
- `research.market_snapshot` — is the market skeptical, hopeful, frustrated?

### Step 3: Classify Story Function

Determine what function the story must serve in this specific campaign:

```
STORY FUNCTION determines position and type:
  Discovery Story:       Shows HOW the mechanism was found
  Proof Story:          Shows the mechanism WORKING
  Origin Story:         Shows WHY the product exists
  Transformation Story: Shows WHAT happened when someone used it
  Revelation Story:     Shows an expert uncovering a hidden truth
  Warning Story:        Shows what happens if prospect does NOT act
  Vulnerability Story:  Shame-to-dignity arc mirroring the prospect
```

The story function should align with:
- The campaign thesis (from structure-package)
- The emotional arc needed (what emotion must the story section produce?)
- The proof assets available (the story must be SUPPORTABLE)

### Step 4: Check Vault Precedent

Query vault_intelligence.story_type_correlations:
- What story types pair with this proof profile in elite controls?
- What story types work with this mechanism type in the same niche?
- What story types appear in the highest-quality extractions?

### Step 5: Produce Recommendation

```yaml
story_type_recommendation:
  primary:
    type: string
    function: string  # discovery, proof, origin, transformation, revelation, warning, vulnerability
    rationale: string  # 3-5 sentences connecting proof assets, narrative needs, mechanism type
    vault_precedent: string  # example from vault intelligence if available
    confidence: number  # 1-10
    proof_basis: string  # which proof assets drive this choice
    narrative_need: string  # which narrative requirement this serves
  alternatives:
    - type: string
      function: string
      rationale: string
      trade_off: string  # what you gain and lose vs. primary
      confidence: number
    - type: string
      function: string
      rationale: string
      trade_off: string
      confidence: number
  anti_recommendations:
    - type: string
      why_not: string  # why this story type would fail for this campaign
    - type: string
      why_not: string
```

### Step 6: Validate Recommendation

- Does the primary recommendation leverage the campaign's strongest proof assets?
- Is the story function aligned with the campaign thesis?
- Can the recommended story type be executed with available proof? (A Proof Story with no testimonials = problem)
- Does the recommendation avoid proof gaps? (Don't recommend a story type that requires proof you don't have)

## Quality Gate

- [ ] Primary story type recommendation produced with 3-5 sentence rationale
- [ ] Story function classified (discovery, proof, origin, transformation, revelation, warning, vulnerability)
- [ ] Rationale connects proof asset strength, mechanism type, and narrative needs
- [ ] Vault precedent cited if available
- [ ] Confidence score is 1-10 and justified
- [ ] At least 2 alternatives provided with trade-offs
- [ ] At least 2 anti-recommendations with specific reasons
- [ ] Story type leverages available proof (not proof the campaign doesn't have)
- [ ] Low-confidence recommendations (<5) marked as "exploratory"
- [ ] No anti-slop violations in rationale text

---

## Anti-Patterns

- **Proof-blind recommendation:** Recommending Transformation Story when there are no testimonials — REJECT
- **Function mismatch:** Recommending a Discovery Story when the mechanism doesn't have a compelling discovery narrative — REJECT
- **No proof connection:** Rationale doesn't cite which specific proof assets the story type leverages — incomplete
- **Generic alternatives:** Alternatives that don't meaningfully differ from the primary — useless
