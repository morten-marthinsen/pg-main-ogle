# 1.2 Content Type Timing Matrix

## Purpose
Map each piece of content from S14 to its optimal posting time based on platform, format, and content function.

---

## Input Requirements

From Layer 0:
- `0.3-upstream-data.yaml` (content inventory)

From Layer 1:
- `1.1-platform-timing.yaml` (platform timing matrix)

---

## Execution Protocol

### Step 1: Assign Initial Posting Times

For EVERY content piece from S14:

```yaml
content_id: "[ID]"
content_title: "[Title]"
platform: "[Primary platform]"
content_type: "[Format]"
function: "[Awareness/Engagement/Conversion/Community]"

recommended_time:
  date: "[Calculated based on campaign calendar]"
  time: "[From platform timing matrix for this function]"
  timezone: "[From campaign]"
  rationale: "[Why this time]"

alternative_times:
  option_1: "[Backup time]"
  option_2: "[Backup time]"
```

### Step 2: Apply Posting Frequency Constraints

- Check no platform exceeds maximum posts per day
- Enforce minimum gaps between posts on same platform
- Distribute content across campaign date range
- Balance weekly rhythm (don't overload one day)

### Step 3: Resolve Conflicts

If multiple pieces want same time slot:
1. Priority to hero content (highest virality score)
2. Shift lower-priority content to alternative times
3. Document all shifts
