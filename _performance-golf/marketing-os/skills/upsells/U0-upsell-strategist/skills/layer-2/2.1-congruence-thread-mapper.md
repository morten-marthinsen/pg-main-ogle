# 2.1 — Congruence Thread Mapper

**Layer:** 2
**Model:** opus
**Depends on:** GATE_1 PASS
**Produces:** `layer-2/2.1-congruence-map.md`

---

## Purpose

Map the congruence thread that connects the front-end offer to every position in the upsell sequence. This ensures the buyer experiences one continuous journey, not a series of disconnected sales pages.

## The 4 Congruence Threads

### Thread 1: Mechanism Name
The exact mechanism name from the front-end MUST appear in every upsell position. Not a paraphrase. Not a synonym. The EXACT name.

- Front-end: "The Metabolic Switch"
- Bump: "...activates your Metabolic Switch through food"
- Upsell: "...supercharge your Metabolic Switch"
- Downsell: "...the core Metabolic Switch protocol"

### Thread 2: Root Cause Language
The root cause framing from the front-end must be maintained. Don't introduce a new problem — reference the same one.

### Thread 3: Promise Continuity
Each upsell EXTENDS the promise — makes it bigger, faster, more complete. It does NOT replace the promise with a new one.

### Thread 4: Buyer Validation
Every page opening references the purchase decision: "You just secured X" / "Your decision to X" / "Because you chose X."

## Execution Steps

1. Read mechanism name from upstream context (Layer 0)
2. Read root cause language from upstream context
3. Read promise statement from upstream context
4. For each position in the sequence (from Layer 1):
   a. Write the exact mechanism reference for this position
   b. Write the root cause callback for this position
   c. Write the promise extension for this position
   d. Write the buyer validation opener for this position
5. Validate: read all position threads in sequence — does it feel like ONE story?
