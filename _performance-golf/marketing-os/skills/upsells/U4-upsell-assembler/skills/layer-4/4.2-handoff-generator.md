# 4.2 — Handoff Generator

**Skill:** U4 — Upsell Sequence Assembler
**Microskill:** 4.2 — Handoff Generator
**Layer:** 4
**Model:** sonnet
**Input:** `upsell-sequence-assembled.md` (from 4.1), `validation-report.md` (from 4.1), `layer-0/0.1-input-context.md`
**Output:** `e0-handoff.yaml`

---

## PURPOSE

Generate two downstream handoffs from the assembled upsell sequence: (1) the U5 editorial handoff (which is the assembled sequence document itself — no additional file needed), and (2) the E0 email strategy handoff as a structured YAML file containing all the data E0 needs to craft post-purchase email sequences.

---

## EXECUTION STEPS

1. **Read `upsell-sequence-assembled.md`** (the primary deliverable from 4.1). This is the complete assembled sequence with YAML metadata, sequence map, and full copy.

2. **Read `validation-report.md`** for any issues that need to be communicated to U5.

3. **Read `layer-0/0.1-input-context.md`** for metadata fields that may not be in the assembled document (guarantee terms, bonus details, angles).

4. **Confirm U5 handoff.**
   - The U5 handoff IS the assembled sequence document (`upsell-sequence-assembled.md`)
   - U5 also receives the validation report — which contains the fix recommendations organized by priority (Critical / Important / Advisory)
   - No additional file needed for U5
   - Log this confirmation in the output

5. **Build the E0 handoff YAML.** E0 (Email Strategist) needs structured data to create post-purchase email sequences. The email strategy depends on what was offered in the upsell sequence and what the buyer's response might be. Populate:

   **Project + FE context:**
   - Project name
   - FE product name
   - FE price
   - FE mechanism name

   **Offers in sequence** — for each position (bump, upsell, downsell):
   - Position in funnel
   - Product name
   - Price
   - Angle (one sentence describing how this piece was positioned)
   - Format (for upsell: text_page or video_script)
   - Reframe type (for downsell only)

   **Congruence thread data:**
   - Mechanism name (exact — this threads into email copy)
   - Root cause language (exact — emails must use the same root cause)
   - Core promise (so emails extend it consistently)

   **Guarantee terms:**
   - Full guarantee description (type, duration, terms)
   - This appears in confirmation emails and abandoned-upsell follow-ups

   **Email strategy inputs** — three buyer scenarios E0 must plan for:

   a. **abandoned_upsell:** Buyer purchased FE (and possibly bump) but declined ALL upsells.
      - E0 needs: what was offered, at what price, what angle
      - Strategy guidance: follow up with value content that seeds future upsell. Do NOT immediately re-pitch the declined offer. Nurture with content that demonstrates the mechanism working, build desire over 3-5 emails, then present the upsell again with a different angle or time-limited incentive.

   b. **upsell_accepted:** Buyer accepted the upsell (possibly after bump).
      - E0 needs: what they bought, at what price, bonuses included
      - Strategy guidance: welcome sequence should reference the upsell product specifically, provide onboarding/getting-started content, identify cross-sell opportunities based on what they already own. The congruence thread continues into email — mechanism name and root cause language should appear in emails.

   c. **downsell_accepted:** Buyer declined the upsell but accepted the downsell.
      - E0 needs: what they declined (full upsell), what they accepted (downsell), the reframe angle used
      - Strategy guidance: nurture toward the full product upgrade. The buyer showed interest (accepted downsell) but not at the full price. Future emails can demonstrate the value gap between what they have (downsell) and what they could have (full upsell). Do NOT make them feel bad about their choice — validate it, then build desire over time.

6. **Validate the handoff YAML.** Before writing:
   - Every field populated (no empty strings, no placeholders)
   - Mechanism name matches exactly across all sections
   - Prices are consistent with the assembled document
   - All three email scenarios covered
   - Angle summaries are SPECIFIC, not generic ("positioned as an accelerator that doubles the mechanism's speed" not "positioned as a good product")

7. **Write the output file.**

**Contents:**

```yaml
    - position: "order_bump"

      price: "[bump price]"

    - position: "upsell_1"
      product: "[upsell product name]"
      price: "[upsell price]"

      format: "[text_page | video_script]"

    - position: "downsell_1"
      product: "[downsell product name]"
      price: "[downsell price]"
      reframe_type: "[core_extract | payment_plan | lite_version | different_format | different_emphasis]"
      angle: "[one sentence — how the downsell was reframed from the declined upsell]"

  congruence_thread:
    mechanism_name: "[exact mechanism name — this threads into email copy]"

    promise: "[core promise as extended through the sequence]"

    abandoned_upsell:
      scenario: "Buyer purchased FE but declined all upsells"
      products_owned: ["[FE product]"]
      products_declined: ["[upsell product]", "[downsell product]"]
      strategy: "Follow up with value content demonstrating the mechanism in action. Seed desire over 3-5 emails. Re-present upsell with different angle or time-limited incentive. Do NOT immediately re-pitch."
      congruence_note: "Emails must use mechanism name '[mechanism name]' and root cause language to maintain thread continuity."

    upsell_accepted:
      scenario: "Buyer accepted upsell (and possibly bump)"
      products_owned: ["[FE product]", "[upsell product]"]
      strategy: "Welcome sequence references upsell product specifically. Onboarding content for combined FE + upsell. Cross-sell opportunities based on expanded ownership. Congruence thread continues — mechanism name and root cause in emails."
      bonuses_included: ["[bonus 1]", "[bonus 2]"]

    downsell_accepted:
      scenario: "Buyer declined upsell but accepted downsell"
      products_owned: ["[FE product]", "[downsell product]"]
      products_declined: ["[upsell product at $[price]]"]
      reframe_used: "[reframe type and angle]"
      strategy: "Validate the downsell choice. Nurture toward full product upgrade over time. Demonstrate value gap between downsell and full upsell through content and results. Do NOT create buyer's remorse."
      upgrade_path: "[how the downsell buyer can eventually upgrade to the full upsell — if applicable]"

# U5 Handoff Note:
# The U5 editorial handoff is upsell-sequence-assembled.md + validation-report.md
# No separate file needed — U5 reads the assembled sequence as its primary input
# and the validation report for editorial fix priorities.
```

## CONSTRAINTS

- **Both handoffs are required for GATE_2 to pass.** U5 handoff = assembled sequence document (already exists from 4.1). E0 handoff = this YAML file. Missing either = GATE_2 FAIL.
- **E0 handoff is STRUCTURED DATA, not prose.** E0 is a strategy tool that ingests structured inputs. Keep the YAML clean, well-organized, and machine-parseable. Angle summaries and strategy notes are the only prose elements.
- **Mechanism name must be EXACT in the handoff.** The E0 email strategy will use this name in email copy. If the handoff says "Metabolic Switch" but the sequence uses "The Metabolic Switch Protocol," emails will break the congruence thread.
- **Angle summaries must be SPECIFIC.** Generic angles like "a good add-on" or "a helpful product" give E0 nothing to work with. Specific angles like "positioned as the done-for-you implementation guide that eliminates the #1 barrier (not knowing where to start)" give E0 a clear direction for email content.
- **Email strategy inputs describe SCENARIOS, not email copy.** E0 will create the actual email strategy and sequences. U4's job is to give E0 the data it needs: what was offered, what was accepted/declined, what angles were used, and how the congruence thread should continue.
- **Do not generate email copy.** U4 generates handoff data. E0 generates email strategy. E1 generates email copy. The pipeline is clear.
- **Guarantee terms must be complete.** "Money-back guarantee" is not complete. "60-day no-questions-asked money-back guarantee on [product]" is complete. E0 uses this in confirmation and follow-up emails.

---

## FAILURE MODES

| Failure | Detection | Response |
|---------|-----------|----------|
| E0 handoff YAML missing | File not written to project root | Write the file. Both handoffs are required for GATE_2. |
| Mechanism name inconsistent between handoff and assembled sequence | Compare mechanism_name in YAML against fe_mechanism in assembled document metadata | Correct to match the canonical mechanism name from mechanism-package.json. The handoff must use the exact same string. |
| Angle summary is generic | Angle says "a helpful product" or similar vague language | Rewrite with specifics: product name, positioning relative to FE, the core value proposition in one sentence. Read the actual copy to extract the angle. |
| Email strategy scenario missing | Fewer than 3 scenarios in email_strategy_inputs | Add the missing scenario. All three (abandoned, accepted, downsell_accepted) are required — E0 needs to plan for all buyer paths. |
| Guarantee terms incomplete | guarantee_terms says only "money-back guarantee" | Read the actual copy for full guarantee details: type, duration, conditions, which products it covers. If guarantee details are not in the copy, check U0 strategy. |
| YAML syntax error | Indentation wrong, missing colons, broken quotes | Validate YAML structure before writing. Misformatted YAML will break E0's parsing. |
