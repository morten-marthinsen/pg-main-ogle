# 4.2 — Summary Writer

> Version: 1.0
> Layer: 4
> Model: haiku
> Input: cta-copy-package.json, CTA_SELECTION.yaml, cta-audit.md
> Output: CTA-COPY-SUMMARY.md

**Skill:** LP-14-cta-optimizer
**Position:** Layer 4, Microskill 2
**Purpose:** Write a human-readable summary of the complete CTA copy package. This file is for human review — it presents the selected CTAs, P.S. copy, final close, emotional appeal coverage, audit results, and all candidate options in a scannable format.
**Effort Level:** low
**Dependencies:** `cta-copy-package.json` (4.1), `CTA_SELECTION.yaml`, `cta-audit.md` (3.4)
**Output:** `CTA-COPY-SUMMARY.md`

---

## Input Schema

```yaml
input:
  cta_package: "cta-copy-package.json"
  cta_selection: "CTA_SELECTION.yaml"
  cta_audit: "cta-audit.md"
```

---

## Execution Logic

1. Read `cta-copy-package.json` for all final copy and validation data.
2. Read `cta-audit.md` for the 10-point scoring breakdown.
3. Assemble the summary following the required sections from LP-14-AGENT.md.

## Quality Criteria
- All sections from LP-14-AGENT.md's CTA-COPY-SUMMARY.MD REQUIRED SECTIONS are present
- Selected copy matches cta-copy-package.json exactly (no discrepancies between summary and package)
- All candidates table includes every candidate from generation (not just selected ones)
- Audit breakdown matches cta-audit.md scores

## Failure Modes
- Summary text differs from package JSON -> Summary must be generated FROM the package, not independently. Cross-check.
- Missing sections -> All required sections must be present even if content is "N/A" or "Skipped."
- Candidates table incomplete -> All generated candidates must be listed, not just the top scorers.
