# 3.2 — Objection Coverage Audit

**Skill:** LP-12-faq-objections
**Position:** Layer 3, Microskill 2
**Purpose:** Verify that the complete FAQ section covers all 3 mandatory objection categories and at least 4 of 6 total categories. This is a separate audit from the 10-point FAQ validator because coverage is a structural requirement — the FAQ cannot proceed to assembly with uncovered mandatory categories, regardless of how well-written the existing answers are.
**Effort Level:** low
**Dependencies:** `core-objection-answers.md` (2.2), `technical-faq-answers.md` (2.3), `logistics-faq-answers.md` (2.4), `objection-classification.md` (1.1)
**Output:** `coverage-audit.md`

---

## Input Schema

```yaml
input:
  all_faq_items: "[complete Q&A pairs from 2.2 + 2.3 + 2.4 with their category tags]"
  original_classification: "[from objection-classification.md -- what was planned]"
  mandatory_categories:
    - price_value
    - efficacy_results
    - trust_guarantee
  total_category_minimum: 4
```

---

## Execution Logic

1. **Map every FAQ item to its objection category.** Using the category tags from the answer files:

   ```
   Q1: [category]
   Q2: [category]
   Q3: [category]
   ...
   Q[N]: [category]
   ```

2. **Count items per category:**

   | Category | Item Count | Items |
   |----------|-----------|-------|
   | Price/Value | [count] | [Q#, Q#...] |
   | Efficacy/Results | [count] | [Q#, Q#...] |
   | Trust/Guarantee | [count] | [Q#, Q#...] |
   | Timing | [count] | [Q#, Q#...] |
   | Comparison | [count] | [Q#, Q#...] |
   | Technical/Logistics | [count] | [Q#, Q#...] |

3. **Verify mandatory categories:**
   - Price/Value: count >= 1? [Y/N]
   - Efficacy/Results: count >= 1? [Y/N]
   - Trust/Guarantee: count >= 1? [Y/N]
   - IF ANY = N: FAIL — must add the missing category before assembly

4. **Verify total category coverage:**
   - Categories with >= 1 item: [count] of 6
   - Minimum required: 4
   - IF count < 4: FAIL — must add items from underrepresented categories

5. **Compare against original plan:**
   - Check if the planned objections from objection-classification.md all made it into the final FAQ
   - Flag any planned objections that were dropped during writing
   - Flag any NEW objections added during writing that were not in the plan

6. **Identify coverage gaps:**
   - Which categories have 0 items?
   - Are any of those categories relevant for this product/vertical?
   - Recommend: should the gap be filled, or is the category genuinely irrelevant?

7. **Check for over-concentration:**
   - Does any single category have > 50% of the total items?
   - If Technical/Logistics has 7 of 10 items, the FAQ is too heavily skewed toward easy questions
   - Recommend rebalancing if any category dominates

8. **Determine audit result:**
   - PASS: All 3 mandatory categories present AND >= 4 of 6 total AND no over-concentration
   - FAIL: Any mandatory category missing OR < 4 total OR severe over-concentration

## Quality Criteria
- Every FAQ item accurately mapped to its category (no miscategorizations)
- Mandatory categories checked with zero tolerance — 0 items = FAIL, period
- Over-concentration flagged if any category exceeds 50% of total items
- Plan-vs-actual comparison surfaces any drift from the Layer 1 plan
- Gaps assessed for relevance (not every category MUST be present, but 4 of 6 must be)

## Failure Modes
- Miscategorizing a question to inflate a mandatory category -> "How do I take it?" is Technical/Logistics, not Efficacy. "What results can I expect?" is Efficacy, not Technical. Categorize by what the ANSWER addresses.
- Counting a question in 2 categories to inflate coverage -> Each question maps to exactly 1 category. Dual-mapping inflates counts.
- Passing with 3 categories total -> Minimum is 4 of 6. 3 categories is a FAIL even if all 3 mandatory are present.
- Ignoring over-concentration -> An FAQ with 8 technical questions and 2 core objections is structurally imbalanced. The reader's biggest concerns (price, efficacy, trust) are underserved.
