# 3.1 — Offer Architecture Validator

**Skill:** LP-06-offer-cta-architecture
**Position:** Layer 3, Microskill 1
**Purpose:** Check internal consistency across all Layer 1-2 outputs. Verify that pricing strategy matches offer type, CTA placements match section sequence, guarantee duration matches claims, urgency types match offer classification, and value stack math is correct. This is a structural integrity audit — it catches inconsistencies before the one-offer audit (3.2) and before package assembly (Layer 4).
**Effort Level:** medium
**Dependencies:** All Layer 1 and Layer 2 outputs
**Output:** `architecture-validation.md`

---

## Input Schema

```yaml
input:
  offer_type_classification: "[from 1.1]"
  pricing_display_strategy: "[from 1.2]"
  cta_strategy: "[from 1.3]"
  guarantee_architecture: "[from 1.4]"
  value_stack_architecture: "[from 2.1]"
  price_anchor_design: "[from 2.2]"
  cta_placement_map: "[from 2.3]"
  urgency_scarcity_plan: "[from 2.4]"
  cta_language_directions: "[from 2.5]"
  page_type: "[type_a | type_b | hybrid]"
```

---

## Validation Checks (8 Checks)

### Check 1: Pricing Strategy <-> Page Type Consistency
```
IF page_type = type_a:
  pricing_strategy MUST = price_after_value
  price_above_fold MUST = N
IF page_type = type_b:
  pricing_strategy MUST = price_above_fold
  price_above_fold MUST = Y
IF page_type = hybrid:
  pricing_strategy MUST = hybrid_both
```
**Result:** PASS or FAIL + specific inconsistency

### Check 2: Offer Type <-> Urgency Consistency
```
IF offer_type = limited:
  urgency MUST be included with quantity-based scarcity
IF offer_type = limited_time:
  urgency MUST be included with deadline-based scarcity
IF offer_type = oto:
  urgency MUST be included with page-level urgency
IF offer_type = hard AND urgency included:
  urgency justification MUST reference a real-world reason
```
**Result:** PASS or FAIL + specific inconsistency

### Check 3: CTA Count Verification
```
cta_placements from 2.3 >= 3
cta_placements match count declared in 1.3 strategy
every CTA in 2.3 has 3+ directions in 2.5
```
**Result:** PASS or FAIL + missing CTAs or directions

### Check 4: Guarantee Branding Check
```
guarantee.branded_name is NOT any of:
  - "money-back guarantee"
  - "satisfaction guaranteed"
  - "full refund guarantee"
  - "100% guarantee"
  - "risk-free guarantee"
guarantee.branded_name INCLUDES product name OR specific outcome
guarantee.duration is a specific number of days
guarantee.terms include claim process
```
**Result:** PASS or FAIL + specific violation

### Check 5: Value Stack Math Verification (Type A only)
```
IF pricing_strategy = price_after_value:
  sum of assigned values = total_value stated
  total_value / actual_price = value_to_price_ratio
  ratio between 2x and 5x
  per_day_cost = price / days_of_supply (calculated correctly)
```
**Result:** PASS or FAIL + math errors

### Check 6: CTA Section <-> Section Sequence Alignment
```
Every CTA section number in 2.3 exists in section-sequence.json
No CTA placed in a section that doesn't exist
CTA zones match section positions (early CTA not in close section, etc.)
```
**Result:** PASS or FAIL + misaligned placements

### Check 7: Urgency Credibility Verification
```
IF urgency included:
  every claim has a documented real-world justification
  no claim matches forbidden fabricated patterns
  claims are specific (name dates, quantities, or constraints)
IF urgency excluded:
  exclusion reason is documented
```
**Result:** PASS or FAIL + unjustified claims

### Check 8: CTA Language Personalization Verification
```
Every CTA direction in 2.5 includes at least ONE of:
  - product name
  - specific benefit
  - savings amount
  - audience identity
  - guarantee duration
No direction uses forbidden generic language (Buy Now, Get Started, Submit, Click Here, Act Now without reason)
```
**Result:** PASS or FAIL + generic directions

---

## Execution Logic

**Step 1:** Run all 8 checks sequentially. Document each result with evidence.

**Step 2:** Compile results. Calculate overall status:
- All 8 PASS -> Architecture Validation: PASS
- Any FAIL -> Architecture Validation: FAIL. List specific fixes required.

**Step 3:** For each FAIL, prescribe the exact fix:
- Which file needs updating
- What specific change is needed
- Which microskill should re-run (if applicable)

## Quality Criteria
- All 8 checks run with documented evidence
- Every FAIL has a specific prescribed fix (not "fix the issue")
- Math verification shows actual calculations
- Generic language check references specific CTA IDs
- Overall result is binary: PASS (all 8 pass) or FAIL (any fail)

## Failure Modes
- Upstream file missing (Layer 1 or 2 output not produced) -> Cannot run the relevant check. Mark as INCOMPLETE. HALT — return to the missing layer.
- Value stack math is off by rounding -> Recalculate. Minor rounding differences ($0.01-$0.99) are acceptable. Document the adjustment.
- Multiple checks fail -> List ALL fails and ALL fixes. Do not stop at the first failure.
