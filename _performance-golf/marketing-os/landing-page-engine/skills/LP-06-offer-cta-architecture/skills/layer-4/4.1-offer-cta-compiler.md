# 4.1 — Offer-CTA Compiler

**Skill:** LP-06-offer-cta-architecture
**Position:** Layer 4, Microskill 1
**Purpose:** Assemble the final `offer-cta-architecture.json` from all validated Layer 1-3 outputs. This is the single canonical output file that LP-11 (Offer/Pricing Writer), LP-13 (Urgency/Scarcity Stack), LP-14 (CTA Copy Optimizer), LP-15 (Page Assembly), and LP-17 (Conversion Audit) consume. The compiler does not make decisions — it compiles validated data into the schema defined in AGENT.md. Every field must be populated. No placeholders. No "TBD."
**Effort Level:** low
**Dependencies:** All Layer 1-3 outputs, GATE_3 status = PASS
**Output:** `offer-cta-architecture.json`

---

## Input Schema

```yaml
input:
  all_layer_outputs: "[all .md files from Layers 1-3]"
  architecture_validation_result: "[from 3.1 — must be PASS]"
  one_offer_audit_result: "[from 3.2 — must be PASS]"
  gate_3_status: "[PASS — required to proceed]"
```

---

## Execution Logic

**Step 1:** Verify GATE_3 status is PASS. Both `architecture-validation.md` (3.1) and `one-offer-audit.md` (3.2) must show PASS. If FAIL -> HALT. Do not compile an architecture that failed validation.

**Step 2:** Build the JSON structure following the exact schema from LP-06-AGENT.md `OFFER-CTA-ARCHITECTURE.JSON SCHEMA`:

- **Header fields:** schema_version, created (ISO timestamp), project_name, page_type
- **one_offer_rule:** single_offer_confirmed (Y), offer_description, competing_offers_detected (N)
- **offer_classification:** primary_type, secondary_types, rationale — from `offer-type-classification.md` (1.1)
- **pricing_display:** strategy, anchor_method, price_points, savings_display, bundle_tiers, value_stack — from `pricing-display-strategy.md` (1.2), `value-stack-architecture.md` (2.1), `price-anchor-design.md` (2.2)
- **guarantee:** branded_name, duration, type, terms, visual_treatment, placement_sections — from `guarantee-architecture.md` (1.4)
- **cta_architecture:** total_placements, personalization_approach, placements array, sticky_mobile_cta — from `cta-strategy.md` (1.3), `cta-placement-map.md` (2.3), `cta-language-directions.md` (2.5)
- **urgency_scarcity:** included (Y/N), types_used or exclusion_reason — from `urgency-scarcity-plan.md` (2.4)
- **downstream_handoffs:** skill-to-section mapping for LP-11, LP-13, LP-14, LP-15, LP-17
- **validation:** one_offer_audit, architecture_consistency, cta_count_minimum, guarantee_branded, urgency_justified_or_excluded — all PASS

**Step 3:** For each section of the JSON, pull data from the correct upstream file:
- Offer classification -> `offer-type-classification.md` (1.1)
- Pricing display -> `pricing-display-strategy.md` (1.2) + `value-stack-architecture.md` (2.1) + `price-anchor-design.md` (2.2)
- Guarantee -> `guarantee-architecture.md` (1.4)
- CTA architecture -> `cta-strategy.md` (1.3) + `cta-placement-map.md` (2.3) + `cta-language-directions.md` (2.5)
- Urgency/scarcity -> `urgency-scarcity-plan.md` (2.4)
- Validation -> `architecture-validation.md` (3.1) + `one-offer-audit.md` (3.2)

**Step 4:** Validate JSON structure. Confirm:
- No empty fields (no "", no null, no "TBD")
- All CTA placements have 3+ emotional appeal directions
- Guarantee branded_name is NOT generic
- One-offer audit = PASS
- All validation fields = PASS

**Step 5:** Write `offer-cta-architecture.json` to the project output directory.

---

## Output Schema

`offer-cta-architecture.json` — follows the exact JSON schema defined in LP-06-AGENT.md under `OFFER-CTA-ARCHITECTURE.JSON SCHEMA`.

Key structural requirements:
- `one_offer_rule.single_offer_confirmed`: "Y"
- `one_offer_rule.competing_offers_detected`: "N"
- `cta_architecture.total_placements`: >= 3
- `guarantee.branded_name`: NOT any generic guarantee name
- `validation`: all fields = "PASS"
- Every CTA in `placements` array has `emotional_appeals` array with 3+ entries

---

## Quality Criteria
- JSON is valid and parseable — no syntax errors
- Every field populated — zero empty strings, nulls, or placeholders
- One-offer rule fields are populated correctly
- CTA count matches total_placements field
- Validation block shows all PASS
- Downstream handoffs section populated with skill-specific directions

## Failure Modes
- GATE_3 not passed -> HALT. Cannot compile unvalidated architecture.
- Upstream file missing data for a required field -> Do NOT use placeholder. HALT and identify which upstream output is incomplete.
- JSON syntax error on write -> Validate JSON before finalizing. Re-format if needed.
- CTA placements array length does not match total_placements -> Fix before writing. This is a compilation error.
