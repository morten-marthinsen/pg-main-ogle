# 1.1 — Headline Type Confirmation

**Skill:** LP-07-hero-section
**Position:** Layer 1, Microskill 1
**Purpose:** Confirm or override LP-03's headline type recommendation (T1-T7) based on the full copy asset context now available. LP-03 made this decision with architectural information only — LP-07 now has audience pain quotes, mechanism names, and promise statements that may reveal a stronger angle.
**Effort Level:** medium
**Dependencies:** `packages-loaded.md` (0.1), `copy-assets.md` (0.2), `specimens-loaded.md` (0.3)
**Output:** `headline-type-confirmed.md`

## Execution Logic

1. **State LP-03's recommendation and rationale.** Write it out explicitly before evaluating.

2. **Evaluate confirmation criteria** — LP-03's recommendation STANDS if:
   - The headline type aligns with the awareness stage (see alignment table below)
   - Copy assets do not reveal a stronger angle that demands a different type
   - The vertical pattern matches (e.g., health supplement discovery stories favor T2/T6)
   - Specimen analysis supports the type selection

3. **Evaluate override criteria** — LP-03's recommendation is OVERRIDDEN if:
   - A specific mechanism name is available AND LP-03 selected a non-mechanism type (T5, T3) → Consider T2 (Curiosity/Revelation) or T6 (Discovery)
   - Strong audience pain quotes are available AND LP-03 selected a non-empathy type → Consider T3 (Problem-First) or T5 (Question)
   - A specific proof anchor exists (study, number, authority) AND LP-03 selected a soft type → Consider T7 (Number-Specific) or T1 (Direct Benefit)
   - The root cause anchor is genuinely counterintuitive → Consider T4 (Warning/Alert) regardless of LP-03's pick

4. **Apply the awareness-type alignment check:**

   | Awareness Stage | Strong Types | Weak Types |
   |----------------|-------------|------------|
   | 1-2 (Unaware) | T2, T4, T6 (curiosity-first) | T1, T7 (assume solution knowledge) |
   | 3 (Solution-Aware) | T1, T3, T5 (problem/benefit focus) | T4 (warning less effective when solutions known) |
   | 4-5 (Product-Aware) | T1, T7 (specifics matter) | T2 (curiosity unnecessary — they know the category) |

5. **Make the decision:** CONFIRM or OVERRIDE with documented rationale.

6. **If overriding:** State the new type, what copy asset evidence supports it, and why it will outperform LP-03's pick.

## Quality Criteria
- LP-03 recommendation explicitly stated before evaluation begins (no invisible starting point)
- Override decision includes specific copy asset evidence, not just "feels stronger"
- Awareness-type alignment verified regardless of confirm/override
- Implications section gives Layer 2 microskills actionable guidance
- If overriding, the rationale must be strong enough that LP-03 would agree given the new information

## Failure Modes
- Overriding without evidence ("T2 feels more interesting") → Revert to LP-03 recommendation. Overrides require specific data.
- Confirming without checking copy assets → Re-read copy-assets.md before confirming. Rubber-stamp confirmation is a protocol violation.
- Selecting a type misaligned with awareness stage → Check alignment table. If still choosing a "weak" type for the awareness stage, provide explicit rationale for why this case is an exception.
