# 4.2 — Summary Writer

**Skill:** LP-07-hero-section
**Position:** Layer 4, Microskill 2
**Purpose:** Write the HERO-SECTION-SUMMARY.md file using the required template from LP-07-ANTI-DEGRADATION.md. This is the human-readable executive summary of the hero section — what was selected, how it scored, and what threads downstream skills must maintain. It is the first file a human or downstream skill reads to understand the hero section output.
**Effort Level:** low
**Dependencies:** `hero-section-package.json` (4.1), `HERO_SELECTION.yaml`, `hero-audit.md` (3.6), `headline-candidates.md` (2.2), `loop-audit.md` (3.2), `anti-slop-scan.md` (3.4), `reading-level.md` (3.5)
**Output:** `HERO-SECTION-SUMMARY.md`

---

## Input Schema

```yaml
input:
  hero_package: "[compiled hero-section-package.json from 4.1]"
  hero_selection: "[HERO_SELECTION.yaml]"
  hero_audit: "[hero-audit.md — score and per-point results]"
  all_candidates: "[headline-candidates.md — full candidate table]"
  loop_audit: "[loop-audit.md — loop count and types]"
  anti_slop: "[anti-slop-scan.md — PASS result]"
  reading_level: "[reading-level.md — grade level]"
  page_type: "[type_a | type_b | hybrid]"
```

---

## Execution Logic

1. **Pull all required data** from input files. Do not summarize from memory — re-read each source file.

2. **Write the summary** following the EXACT template from LP-07-ANTI-DEGRADATION.md. Every section in the template is required. No sections may be omitted.

3. **Build the all-candidates table** from `headline-candidates.md`. Include every candidate that was generated (not just the top 5). Show ID, Lens, Headline Text, and Score.

4. **Populate threading anchors** from `hero-section-package.json`. These are critical — downstream skills LP-08, LP-09, LP-10, LP-14, and LP-15 all reference these anchors.

5. **State downstream handoffs explicitly.** Name each downstream skill and what it receives.

## Quality Criteria
- Every section from the ANTI-DEGRADATION.md template present — no omissions
- Selected headline, deck, and pre-head are EXACT text (not paraphrased)
- All candidates table includes EVERY candidate generated, not just top 5
- Threading anchors populated — these are the most critical downstream data
- Audit section accurately reflects the results from 3.2-3.6 validation files
- File is self-contained: a reader with no context about LP-07's execution should understand the hero section output completely

## Failure Modes
- Omitting a template section ("Above-Fold Copy" section skipped for Type A) → Include section with "N/A — Type A page" rather than omitting
- Paraphrasing the selected headline instead of quoting it exactly → Copy the EXACT text from HERO_SELECTION.yaml
- All-candidates table includes only the top 5 → Include all candidates. The full table is needed for reference and future A/B test planning.
- Threading anchors left empty → These MUST be populated. If not identifiable, escalate to human before writing the summary.
