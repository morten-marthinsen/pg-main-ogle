# 2.1 — Pre-Head Generator

**Skill:** LP-07-hero-section
**Position:** Layer 2, Microskill 1
**Purpose:** Write 3 pre-head variants — the micro-targeting hook that appears above the headline. The pre-head earns the reader's attention before the headline earns their interest. It must be specific to the problem or promise — NEVER generic "ATTENTION: [demographic]" framing.
**Effort Level:** medium
**Dependencies:** `headline-type-confirmed.md` (1.1), `hook-strategy.md` (1.3), `copy-assets.md` (0.2), `specimens-loaded.md` (0.3)
**Output:** `pre-heads.md`

---


## Execution Logic

1. **Read hook strategy coordination notes** from `hook-strategy.md`. The pre-head must SET UP or CONTRAST WITH the headline — never repeat it.

2. **Determine pre-head function** — choose one per variant:

   | Function | Pattern | Example |
   |----------|---------|---------|
   | **Pain qualifier** | Names the specific pain to self-select | "If your knees ache every time you climb stairs..." |
   | **Audience qualifier** | Names the exact person this is for | "For men over 50 who refuse to slow down" |
   | **Curiosity seed** | Plants a question before the headline answers it | "What 47 cardiologists know that your doctor doesn't" |
   | **Proof whisper** | States one credibility signal quietly | "Based on a 12-week clinical trial with 1,247 participants" |
   | **Urgency signal** | Creates time pressure without false scarcity | "This changes on March 15th" |
   | **Contrarian setup** | Contradicts common belief to make headline land harder | "Everything you've been told about joint pain is wrong" |

3. **Generate 3 pre-head variants** — each using a DIFFERENT function from the table above. Diversity is mandatory.

4. **Apply the anti-generic test** to each variant:
   - Would this pre-head work on a DIFFERENT product in the same vertical? If yes → too generic. Revise.
   - Does it contain a specific number, named condition, specific audience descriptor, or concrete claim? If no → too vague. Revise.
   - Does it use the word "ATTENTION" followed by a demographic? If yes → FORBIDDEN. Rewrite entirely.

5. **Score each variant** on two criteria:

   | Criterion | Weight | 1 (Poor) | 5 (Average) | 10 (Excellent) |
   |-----------|--------|----------|-------------|----------------|
   | **Relevance** | 60% | Could apply to any product | Partially targeted | Perfect self-selection for this specific audience + problem |
   | **Specificity** | 40% | No concrete details | Some specifics | Contains a number, name, condition, or timeframe |

6. **Identify the recommended variant** (highest score) and explain why it best sets up the confirmed headline type.

## Quality Criteria
- 3 variants generated, each using a different pre-head function
- Zero instances of "ATTENTION: [demographic]" pattern
- Each variant passes the anti-generic test (would NOT work on a different product)
- Each variant contains at least one specific element (number, named condition, timeframe, audience descriptor)
- Scoring uses both relevance and specificity with documented scores

## Failure Modes
- All 3 variants use the same function (e.g., all pain qualifiers) → Regenerate with mandatory function diversity
- Pre-head repeats the headline claim → Revise to either SET UP or CONTRAST WITH the headline, never echo it
- Pre-head is longer than 15 words → Trim. Pre-heads are micro-copy — brevity is the constraint.
- Generic pre-head passes anti-generic test by accident → Re-apply test strictly: "Would Competitor X use this exact pre-head?" If plausible → revise.
