# 3.1 — Lead Revision (Type A Only)

**Skill:** LP-07-hero-section
**Position:** Layer 3, Microskill 1
**Purpose:** Revise the lead draft based on the human's selection notes from HERO_SELECTION.yaml. The selected headline, deck, and any modification_notes now define the creative direction — the lead must be realigned to serve the specific headline that was chosen, not the general territory from Layer 2.
**Effort Level:** max
**Dependencies:** `HERO_SELECTION.yaml` (BLOCKING — must exist before this microskill executes), `lead-draft.md` (2.4), `headline-candidates.md` (2.2), `copy-assets.md` (0.2)
**Output:** `lead-final.md`

---

## Execution Logic

1. **If revision_direction = "Approved as-is":**
   - Copy `lead-draft.md` contents to `lead-final.md` with no changes.
   - Add a header noting: "Lead approved as-is by human selection. No revisions applied."
   - Skip to output schema.

2. **If revision_direction contains specific notes:**

   **Step 1: Parse modification notes.** List every specific instruction from the human. Number them.

   **Step 2: Verify headline-lead alignment.**
   - Read the selected headline text. What specific promise does it make?
   - Read the lead opening. Does the first 50 words of the lead serve this specific headline?
   - If the lead was written before the headline was selected (it was — in Layer 2.4), it may reference a different headline's angle. Identify and fix any misalignment.

   **Step 3: Apply each modification note sequentially.**
   - For each note: identify the specific passage in the lead that needs to change. Quote it. Write the revised version. Explain what changed and why.
   - If a note is ambiguous, apply the most conservative interpretation and flag for human review.

   **Step 4: Verify loop preservation.**
   - After all edits: confirm that every open loop from the original draft is still open.
   - If a revision closed a loop (answered a question the lead was supposed to leave unanswered): re-open it.
   - If the human's notes explicitly ask to close a loop: comply, but flag the loop count for the audit in 3.2.

   **Step 5: Check reading flow.**
   - Read the revised lead start to finish. Does it read as one continuous piece, or do the edits create seams?
   - Smooth any seams while preserving the human's requested changes.

   **Step 6: Run quick anti-slop check** (full scan in 3.4, but catch obvious violations now).

3. **Produce the final lead** with revision annotations.

## Quality Criteria
- HERO_SELECTION.yaml verified present and human-created before any work begins
- Every modification note from the human explicitly addressed (applied, or flagged if ambiguous)
- All open loops from the original draft preserved (or explicitly noted if revision closed one)
- Lead realigned to the SELECTED headline's specific promise, not the general headline territory
- Final copy reads as a seamless piece — no edit artifacts or tonal shifts at revision points

## Failure Modes
- Executing without HERO_SELECTION.yaml → ABSOLUTE VIOLATION. This microskill cannot run without the file.
- Ignoring a modification note ("I applied the major notes but this minor one seemed unnecessary") → Apply ALL notes. Minor notes reflect taste.
- Revision closes a loop without flagging it → Loop count must be re-verified after every revision. A revision that closes a loop changes the audit math.
- Lead serves a different headline than the one selected (because it was drafted before selection) → Realign the opening and promise references to the specific selected headline.
