# 2.3 — Feature Explanation Writer

**Skill:** LP-09-benefits-features
**Position:** Layer 2, Microskill 3
**Purpose:** Write feature explanation sections for SaaS, info product, B2B service, and non-supplement consumer pages. Each feature gets a dedicated sub-section explaining what it does, why it matters, the benefit it produces, and proof. This is the education layer for non-supplement products.
**Effort Level:** max
**Dependencies:** `benefit-section-classification.md` (1.1), `dfwbp-structure-plan.md` (1.2), `specimens-loaded.md` (0.3), `packages-loaded.md` (0.1)
**Output:** `feature-explanations.md`

## Pre-Generation Protocol

**Step 1: Load feature specimens.**
Study LP-SPEC-03 (Graziosi — 7 individual feature sections) and LP-SPEC-07 (Hyros — feature-to-benefit framing) for structural patterns.

**Step 2: Review D-F-W-B-P plans** for each feature from 1.2.

**Step 3: Calibrate for awareness stage.**
- Stage 1-2 (Unaware/Problem-Aware): Features must be framed as solutions to a problem the reader just realized they have. Lead with the problem, not the feature.
- Stage 3 (Solution-Aware): Features differentiate from alternatives. "Unlike X, this does Y."
- Stage 4-5 (Product/Most-Aware): Features confirm the decision. Specifics and proof dominate.

**Step 4: Read hero threading anchor.** Feature sections must feel like part of the same page as the hero.

## Generation Execution

### Per-Feature Explanation Structure

#### Type A Format (Long-Form):

```markdown
## [Feature Name]

[1 sentence: What it does — plain language, not marketing speak. If the reader has never heard this term, they understand it after this sentence.]

**Why this matters:**
[2-3 sentences: The "Why" bridge. Connect the feature to a BUSINESS or LIFE outcome. Not "because it's better technology" but "because without this, you're [specific consequence]. With it, you [specific improvement]."]

**The result:**
[1-2 sentences: Specific benefit + proof point. The benefit is the outcome the customer experiences. The proof is specific evidence — a customer result, a statistic, a named case.]
```

#### Type B Format (Condensed):

```markdown
### [Feature Name]
[1 sentence: What it does + why it matters + key benefit — all in one compressed statement.]
[Proof point if available — 1 sentence.]
```

#### Feature Grid Format (Type B Alternative):

```
| Feature | What It Does | Why It Matters |
|---------|-------------|---------------|
| [Name] | [1 sentence] | [1 sentence benefit] |
```

### Feature Presentation Order

1. **Hero feature** — the feature most directly connected to the headline promise (longest treatment, leads the section)
2. **Differentiating features** — features competitors lack or do differently (medium treatment)
3. **Expected features** — features the audience expects (briefest treatment — confirm, don't oversell)

### Section Opening and Closing

**Section opening:**
```markdown
## Everything Inside [Product Name]

[1-2 sentences: Frame the feature set. Not "here are the features" but "here's how [product] actually delivers on [the promise from the hero section]."]
```

**Section closing:**
```markdown
[2-3 sentences: Tie features together. Reinforce the cumulative value — the whole is greater than the parts. Reference the hero promise.]
```

## Quality Criteria
- Every feature has a complete explanation with D-F-W-B-P chain
- "What it does" is plain language (test: would a non-technical person understand?)
- "Why it matters" connects to a business/life outcome, not just technology
- Proof is specific (customer result, statistic, case study) or flagged as PROOF_GAP
- Feature order leads with hero feature, not alphabetical or random
- Section opening frames WHY, not WHAT
- Tone matches hero threading anchor
- Zero forbidden words from anti-slop list

## Failure Modes
- Feature explanation is just a restatement of the feature name -> The explanation must add value. "Real-time dashboard: This is a dashboard that updates in real time" adds nothing. "Real-time dashboard: See which ad creatives are burning money right now — not 3 hours from now when the budget is already gone" adds value.
- "Why this matters" is "because it's better" -> Must be specific. Better HOW? Better for WHOM? What happens if they DON'T have this?
- All features get equal treatment -> Hero feature deserves 2-3x the word count of secondary features. Equal treatment signals nothing is important.
- Proof is a general claim ("companies see great results") -> Must be specific: "Hyros customers report average 15-25% ROAS improvement in 30 days."
