# 1.3 — Mandatory Section Validator

**Skill:** PDP-04-btf-section-sequencer
**Position:** Layer 1, Microskill 3
**Purpose:** Confirm that ALL mandatory BTF sections for this product type are present in the ordered sequence. Resolve any PENDING sections from taxonomy selection. This is a blocking check — if mandatory sections are missing, PDP-04 cannot proceed. Also validates that no mandatory section has been accidentally excluded or lost during ordering.
**Effort Level:** low
**Dependencies:** `taxonomy-selection.md` (1.1), `section-ordering.md` (1.2), `brief-verified.md` (0.1)
**Output:** `mandatory-check.md`

---

## Input Schema

```yaml
input:
  taxonomy_selection: "[full taxonomy decisions from 1.1]"
  section_ordering: "[final sequence from 1.2]"
  product_type: "[from brief-verified.md]"
  proof_inventory: "[section eligibility signals from brief-verified.md]"
```

---

## Execution Logic

**Step 1:** Load the mandatory section checklist from the ANTI-DEGRADATION.md TAXONOMY-MC-CHECK block. For this product, identify which sections are:
- **Always mandatory** (6 sections): product_highlights, what_to_expect, the_solution, why_it_works, how_to_use, faq
- **Conditional mandatory** (5 sections — depends on proof inventory):
  - ugc_video_carousel: MANDATORY if ugc_video_count >= 3
  - the_expert: MANDATORY if has_expert_endorsement = Y
  - whats_in_it: MANDATORY if product is supplement or formulated
  - comparison_chart: MANDATORY if has_comparison_competitors = Y
  - reviews_ratings: MANDATORY if testimonials_count >= 25

**Step 2:** Confirm page type is type_b or hybrid. If not, HALT — redirect to LP-04.

**Step 3:** Cross-check every mandatory section against the ordered sequence from 1.2:
- For each always-mandatory section: verify it appears in the sequence
- For each conditional-mandatory section: check condition against proof inventory, then verify it appears if condition is met

**Step 4:** Resolve PENDING sections:
- If a conditional mandatory section was PENDING (proof data unknown), attempt resolution:
  - If proof inventory has been updated since 1.1, use new data
  - If still unknown, include the section with a NOTE that content may be placeholder pending proof verification
  - Document the resolution in the output

**Step 5:** Run the TAXONOMY-MC-CHECK from ANTI-DEGRADATION.md:

```yaml
TAXONOMY-MC-CHECK:
  page_type_confirmed: "[type_b | hybrid]"
  taxonomy_source: "NLS 16-Section PDP BTF Taxonomy from PDP-04-AGENT.md"

  mandatory_sections_present:
    product_highlights: "[Y/N]"
    what_to_expect: "[Y/N]"
    the_solution: "[Y/N]"
    why_it_works: "[Y/N]"
    how_to_use: "[Y/N]"
    faq: "[Y/N]"

  conditional_mandatory:
    ugc_video_carousel: "[Y/N/N-A]"
    the_expert: "[Y/N/N-A]"
    whats_in_it: "[Y/N/N-A]"
    comparison_chart: "[Y/N/N-A]"
    reviews_ratings: "[Y/N/N-A]"

  any_mandatory_missing: "[Y/N]"
```

**Step 6:** Write `mandatory-check.md` with results.

## Quality Criteria
- All 6 always-mandatory sections confirmed present in sequence with position numbers
- All 5 conditional-mandatory sections evaluated against conditions
- PENDING sections resolved with documented rationale
- TAXONOMY-MC-CHECK block completed with all fields filled
- No mandatory section missing — if any are, status is FAIL and action required is documented

## Failure Modes
- Any always-mandatory section missing from sequence -> FAIL. Cannot proceed. Must return to 1.1 to add the section.
- Conditional-mandatory section meets condition but is not in sequence -> FAIL. Must add the section.
- Page type is type_a -> HALT. Redirect to LP-04. PDP-04 does not handle Type A pages.
- Multiple PENDING sections unresolvable -> WARNING with list. Include in sequence with notes, but flag for human review.
- Section ordering from 1.2 has fewer sections than taxonomy decisions from 1.1 -> HALT. A section was dropped during ordering. Reconcile before proceeding.
