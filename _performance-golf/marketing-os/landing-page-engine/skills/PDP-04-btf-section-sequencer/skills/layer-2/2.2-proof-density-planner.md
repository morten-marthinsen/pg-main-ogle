# 2.2 — Proof Density Planner

**Skill:** PDP-04-btf-section-sequencer
**Position:** Layer 2, Microskill 2
**Purpose:** Specify the exact proof type AND count for every section in the BTF sequence. "Add proof" or "include testimonials" is NOT acceptable — every section must have a specific proof spec with type, count, and notes. PDP pages rely heavily on proof because visitors are in purchase mode — proof converts browsers to buyers.
**Effort Level:** medium
**Dependencies:** `optional-decisions.md` (1.4 — locked section list), `pdp-reference-loaded.md` (0.3 — proof density defaults), `brief-verified.md` (0.1 — proof inventory)
**Output:** `proof-density-plan.md`

---

## Input Schema

```yaml
input:
  final_sections_list: "[from optional-decisions.md]"
  proof_density_defaults: "[from pdp-reference-loaded.md — per-section density levels and types]"
  proof_inventory: "[from brief-verified.md — what proof actually exists]"
```

## Execution Logic

**Step 1:** Load proof density defaults from PDP-04-AGENT.md reference data:

| Section | Density | Primary Types |
|---------|---------|--------------|
| Product Highlights | Light | Badge/icon proof only |
| UGC Video Carousel | Heavy | Video testimonials (the section IS proof) |
| The Expert | Heavy | Credential strip, institution logos, video |
| What's in It | Medium per ingredient | 1 study or "clinically studied" badge per key ingredient |
| What to Expect | Light | Timeline icons, mechanism references |
| The Problem | None | Agitation, not proof |
| The Solution | Light | 1 differentiator claim supported |
| Why It Works | Medium | Mechanism diagram, 1-2 study references |
| Comparison Chart | Medium | Checkmarks = implicit proof of superiority |
| Cost Savings | None | Math, not proof |
| How It's Made | Medium | Certification badges, sourcing photos |
| How to Use | None | Instructional only |
| Reviews & Ratings | Maximum | 25-100+ reviews, histogram, photo strip |
| FAQ | Embedded | Proof woven into answers |
| Identity Matching | Light | Persona-matched testimonial snippets |
| The Proof (Data) | Maximum | Study circles, survey graphs, data visualizations |

**Step 2:** For each section in the locked sequence, create a proof density spec:
- **Density level:** none | light | medium | heavy | maximum | embedded
- **Proof elements:** Array of {type, count, notes}
- Each element MUST have a specific count — NOT "some" or "several"

**Step 3:** Map proof density to actual proof inventory:
- If proof inventory says 50 reviews exist, Reviews & Ratings can spec "display 15-20 featured reviews"
- If proof inventory says 0 clinical studies, any section speccing "study reference" must be adjusted
- If proof inventory has gaps (UNKNOWN), use conservative estimates and flag

**Step 4:** Check proof type diversity across all sections:
- Must have at least 3 different proof types across the full BTF sequence
- Proof types: testimonial, study, before_after, authority, volume, demonstration, badge, video, data_viz
- If fewer than 3 types, look for opportunities to diversify

**Step 5:** Calculate proof wave summary:
- Early zone (sections 1-3): What proof density exists?
- Mid zone (sections 4-8): What proof density exists?
- Late zone (sections 9+): What proof density exists?
- Early zone MUST have at least one heavy proof section (UGC, Expert, or Reviews)

**Step 6:** Write proof density plan.

## Quality Criteria
- Every section has a density level assignment — no sections skipped
- Every section with density > none has specific proof elements with TYPE and COUNT
- No section uses vague language ("add proof," "include testimonials," "use studies")
- Proof elements align with actual proof inventory — don't specify what doesn't exist
- At least 3 different proof types across the full sequence
- Early zone has at least 1 heavy proof section
- Sections with "none" density have a documented reason (instructional, math, agitation)

## Failure Modes
- Any section has density > none but no proof elements listed -> HALT. Density without specifics is the #3 failure mode.
- Proof element specifies type but no count -> HALT. "Testimonials" without a number is unusable downstream.
- Proof element references data that doesn't exist in inventory -> Adjust spec to match available proof. Flag the gap.
- Fewer than 3 proof types across all sections -> Add diversity. Look for underutilized proof types.
- No heavy proof section in early zone (sections 1-3) -> Reorder or add proof. Early authority is critical for PDP conversion.
- "Add proof here" or "include images" language found -> HALT. Rewrite with specific type + count.
