# 3.2 — Fix Generator

> **Skill:** LP-16-mobile-audit
> **Layer:** 3 (Scoring & Fixes)
> **Model:** sonnet
> **Input:** `mobile-score.md`, `viewport-analysis.md`, `content-density-audit.md`, `cta-mobile-audit.md`, `media-performance-audit.md`, `speed-estimation.md`, `benchmark-load.md`
> **Output:** `mobile-fixes.md`

---

## PURPOSE

Generate a ranked list of specific, actionable mobile fixes from all issues identified across the 5 Layer 2 audit files. Each fix must be concrete enough for a developer or designer to implement without ambiguity. Fixes are ranked by (expected conversion impact x ease of implementation) to help stakeholders prioritize.

---

## PROCEDURE

### Step 1: Collect All Issues

Read the Issues section from every Layer 2 audit file. Compile a master list of every finding that scored below the threshold or was flagged.

### Step 2: Score Each Fix

For each issue, assign:

**Expected Impact Score (1-5):**

| Score | Meaning | Data Source |
|-------|---------|------------|
| 5 | Major conversion impact (>20% lift potential) | Sticky ATC (+high, conversion-data-reference.md), mobile responsiveness (+25.2%, VWO) |
| 4 | Significant impact (10-20% lift potential) | Page speed (-7% per second, Google), reading optimization (+56%, VWO — indirectly via mobile readability) |
| 3 | Moderate impact (5-10% lift potential) | Image optimization (load time improvement), CTA touch targets (reduces friction) |
| 2 | Minor impact (2-5% lift potential) | Typography fixes, minor spacing issues |
| 1 | Low impact (<2% lift potential) | Cosmetic mobile improvements, minor navigation aids |

**Ease of Implementation Score (1-5):**

| Score | Meaning | Examples |
|-------|---------|---------|
| 5 | Trivial — CSS change or config toggle | Add lazy loading attribute, change font size |
| 4 | Easy — single element change | Add sticky bar (most platforms have built-in), convert images to WebP |
| 3 | Moderate — section-level change | Break text into accordions, restructure CTA placement |
| 2 | Hard — multi-section change | Restructure above-fold layout, add responsive image srcsets |
| 1 | Major — architectural change | Rebuild section sequence, redesign page for mobile-first |

**Priority Score = Impact x Ease** (maximum 25)

### Step 3: Rank Fixes by Priority Score

Sort all fixes by priority score descending. In case of tie, higher Impact wins (conversion impact > ease).

### Step 4: Write Fix Details

For EACH fix, provide:

| Field | Required Content |
|-------|-----------------|
| Element | Exact element name and location (e.g., "Section 5: Benefits/Features — paragraph 3") |
| Issue | What is wrong, with specific evidence (e.g., "92-word paragraph renders as 14 mobile lines") |
| Fix | Exact action to take (e.g., "Break into 3 bullet points of ~30 words each, add icon before each bullet") |
| Expected Impact | Conversion lift estimate with source (e.g., "+25.2% mobile conversions — VWO, mobile-responsive optimization") |
| Impact Score | [1-5] |
| Ease Score | [1-5] |
| Priority Score | [calculated] |
| Skill to Revise | Which upstream LP skill should make this change (e.g., "LP-09 Benefits/Features Writer" or "LP-15 Page Assembly") |

### Step 5: Group by Implementation Phase

Group fixes into phases for implementation:

| Phase | Description | Timeline |
|-------|------------|---------|
| Phase 1: Quick Wins | Ease 4-5, any impact | Implement immediately |
| Phase 2: High Impact | Impact 4-5, any ease | Implement before launch |
| Phase 3: Optimization | Impact 2-3, ease 3-5 | Implement in first optimization round |
| Phase 4: Nice to Have | Impact 1-2, ease 1-3 | Backlog for future optimization |

## CRITICAL RULES

1. **Every fix must pass the "developer test."** Could a developer read this fix and implement it without asking clarifying questions? If not, it is too vague. Add specifics.
2. **No fix under 25 words.** Generic fixes like "optimize images" or "add sticky bar" are insufficient. The fix must specify WHAT, WHERE, and HOW.
3. **Impact estimates must cite sources.** conversion-data-reference.md has data for: speed (-7%/sec), mobile responsiveness (+25.2%), CTA personalization (+202%), reading level (+56%). Use these. If no benchmark exists for a specific fix, state "significant impact expected (no specific benchmark)."
4. **Priority Score drives the ranking.** Not impact alone, not ease alone. The product of both. A high-impact fix that requires architectural rebuild (5 x 1 = 5) ranks below a moderate-impact easy fix (3 x 5 = 15).
5. **Group by implementation phase.** Stakeholders need to know what to do NOW (quick wins), what to do before launch (high impact), and what can wait (backlog). The phase grouping is not optional.
6. **Always specify the upstream skill to revise.** If a fix involves restructuring the hero section, route to LP-03 + LP-07. If it involves CTA changes, route to LP-06 + LP-14. If it involves implementation specs, route to LP-15. This enables efficient revision routing.
