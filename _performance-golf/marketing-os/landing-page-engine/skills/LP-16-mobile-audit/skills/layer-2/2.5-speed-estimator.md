# 2.5 — Speed Estimator

> **Skill:** LP-16-mobile-audit
> **Layer:** 2 (Mobile Audit)
> **Model:** opus
> **Input:** `assembled-page-load.md`, `mobile-criteria.md`, `benchmark-load.md`
> **Output:** `speed-estimation.md`
> **Covers Dimensions:** Page Speed (primary)

---

## PURPOSE

Estimate the page weight and load time based on the content inventory in the assembled page. LP-16 cannot run Lighthouse, WebPageTest, or any real performance testing tool. But it CAN — and MUST — estimate load time from the content inventory: sections, images, videos, scripts, and their optimization status. The estimate flags obvious problems before implementation.

**POSITIONAL REINFORCEMENT:** You are estimating page load time on a mobile device over a 4G connection. Every element adds weight. Every unoptimized asset adds MORE weight. The goal is not a precise measurement — it is a directional estimate that catches pages estimated to exceed the 2-second (Type B) or 3-second (Type A) target. An estimate that catches a 4-second page is valuable even if the actual load time turns out to be 3.5 or 4.5 seconds.

### Step 3: Load Time Estimation

Apply the load time formula:

```
Estimated load time = (total_page_weight / connection_speed) + overhead

Where:
- connection_speed = 1.5 MB/s (1,536 KB/s) for mobile 4G average
- overhead = 0.5 seconds (DNS resolution + TCP handshake + TLS)
- render_penalty = +0.3s per render-blocking script
```

**Calculate:**
- Base load: [total_weight_KB] / 1,536 = [N] seconds
- Overhead: 0.5 seconds
- Render-blocking scripts: [N] x 0.3 = [N] seconds
- **Total estimated load time:** [sum] seconds

### Step 4: Compare to Target

From `mobile-criteria.md`:

| Page Type | Target | Maximum | Estimated | Status |
|-----------|--------|---------|-----------|--------|
| Type A | < 3s | 4s | [N]s | [PASS / OVER TARGET / CRITICAL] |
| Type B | < 2s | 3s | [N]s | [PASS / OVER TARGET / CRITICAL] |
| Hybrid | < 2s | 3s | [N]s | [PASS / OVER TARGET / CRITICAL] |

**Status definitions:**
- **PASS:** Estimated load < target
- **OVER TARGET:** Estimated load between target and maximum acceptable
- **CRITICAL:** Estimated load > maximum acceptable

Apply conversion-data-reference.md speed data:

```
If estimated load = 3.0 seconds and target = 2.0 seconds:
```

## CRITICAL RULES

1. **Always calculate.** Never hand-wave speed estimation. The formula exists. Use it. Even an approximate answer is better than "it should be fast."
2. **"Unspecified" = worst case.** Unspecified image format = 300KB (PNG assumption). Unspecified lazy loading = loads immediately. The estimate must be conservative.
3. **Cite the source.** Every conversion impact estimate must reference conversion-data-reference.md. "-7% per second" comes from Google. "+7% sessions per 10% speed improvement" comes from Google. State the source.
4. **Show the optimization potential.** Do not just flag that the page is slow — show what happens if images are converted to WebP, lazy loading is added, and unnecessary scripts are removed. The "fully optimized estimate" gives stakeholders a clear target.
5. **Type A and Type B have different targets.** Type B: < 2 seconds. Type A: < 3 seconds. A 2.5-second page PASSES for Type A but is OVER TARGET for Type B. Always check the correct threshold from `mobile-criteria.md`.
