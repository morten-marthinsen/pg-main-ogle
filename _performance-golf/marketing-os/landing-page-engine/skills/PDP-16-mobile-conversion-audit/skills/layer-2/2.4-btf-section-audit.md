# 2.4 — BTF Section Audit

> **Skill:** PDP-16-mobile-conversion-audit
> **Layer:** 2 (Audit Execution)
> **Position:** 4 of 8
> **Model:** sonnet
> **Purpose:** Audit below-the-fold section quality against PDP scannability and module independence requirements. PDP visitors hunt-and-peck — they do not read linearly. Every BTF section must work as an independent module, be scannable at a glance, and serve a clear conversion purpose. Dead weight sections dilute the page.

## Execution Logic

**POSITIONAL REINFORCEMENT:** You are auditing BTF section QUALITY — scannability, independence, proof density, and purpose. You are NOT auditing social proof (2.5), navigation (2.6), or conversion flow (2.8). Focus on the structural quality of each BTF module.

**Step 1:** From `assembled-page-load.md`, extract all BTF sections:
- List each section by name and position
- Note section type (highlights, UGC, expert, ingredients, timeline, comparison, FAQ, reviews, etc.)
- Note section length/density

**Step 2:** Evaluate each criterion:

### Criterion 4.1: Section Scannability
- **PDP Law 2:** Scannability over storytelling. PDP visitors hunt-and-peck.
- **Baymard (6.7):** Dense paragraphs on mobile cause scanner's fatigue — users skip entirely. "Snackable" chunks required.
- **PASS:** Every BTF section uses headers, bullets, short paragraphs (3 lines max on mobile), or visual layouts. No dense text blocks.
- **FAIL:** Any BTF section has paragraphs longer than 3-4 lines on mobile, no headers, or wall-of-text formatting.
- **Evidence required:** Identify specific sections that pass/fail scannability. Quote problematic content.

### Criterion 4.2: Module Independence
- **PDP Law 3:** Hunt-and-peck trumps linear reading. Each section must be self-contained.
- **PASS:** Each BTF section makes sense without reading the section before it. No "as mentioned above" or dependency on prior section's explanation.
- **FAIL:** Any section requires context from a prior section to make sense.
- **Evidence required:** Identify sections with cross-dependencies.

### Criterion 4.3: Proof Density Met
- **PDP-04 spec:** Each section type has a minimum proof element requirement.
- **PASS:** Sections that claim benefits include supporting proof (stats, testimonials, expert endorsement, study reference).
- **FAIL:** Sections make claims without any proof element. Pure assertion without evidence.
- **Evidence required:** Identify claimless sections.

### Criterion 4.4: No Dead Weight
- **PDP principle:** Every section must serve a conversion purpose (resolve objection, build desire, provide proof, or remove friction).
- **PASS:** Every BTF section has a clear conversion purpose that can be articulated in one sentence.
- **FAIL:** Any section exists without a clear reason — filler content, redundant sections, sections that repeat information from above-fold.
- **Evidence required:** For each section, state its conversion purpose. Flag sections without one.

**Step 3:** Write per-section analysis for each BTF section, then aggregate to criterion-level verdicts.

**Step 4:** Write `btf-section-audit.md`.

## Quality Criteria

- Every BTF section individually analyzed
- All 4 criteria evaluated with evidence
- Scannability evaluated from a MOBILE perspective (3-line paragraph limit, not desktop)
- Module independence tested by asking "does this make sense if read first?"
- Dead weight identification is honest (sections without purpose are flagged, not rationalized)
- File exists after execution

## Failure Modes

- No BTF sections in assembled page -> Note "No BTF sections found — above-fold-only PDP." Score 4.1-4.4 as N/A. This is unusual but not impossible for minimal PDPs.
- All sections pass individually but feel redundant -> Flag redundancy in 4.4 even if each has a nominal purpose. Two "ingredients" sections or two "proof" sections may be dead weight.
- Cannot determine proof density requirements -> Default to "at least one proof element per benefit-claiming section."
