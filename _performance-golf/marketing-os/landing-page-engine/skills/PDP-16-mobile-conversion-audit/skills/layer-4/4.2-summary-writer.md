# 4.2 — Summary Writer

> **Skill:** PDP-16-mobile-conversion-audit
> **Layer:** 4 (Package Assembly)
> **Position:** 2 of 3
> **Model:** haiku
> **Purpose:** Write `PDP-AUDIT-SUMMARY.md` — the human-readable executive summary of the audit. This is what Anthony and the team read first. It must be scannable, direct, and actionable. No JSON, no code blocks in narrative sections.
> **Dependencies:** `composite-score.md`, `per-section-scores.md`, `priority-ranking.md`
> **Output:** `PDP-AUDIT-SUMMARY.md`

---

## Input Schema

```yaml
input:
  composite_score: "composite-score.md"
  per_section_scores: "per-section-scores.md"
  priority_ranking: "priority-ranking.md"
```

---

## Execution Logic

**Step 1:** Read all 3 input files.

**Step 2:** Write the executive summary using this structure:

### Section 1: Score + Verdict (3 lines max)
- Composite score as large heading
- Classification (Exceptional / Good / Needs Revision / Structural Issues)
- Recommendation (Proceed / Flag / Halt)

### Section 2: Section Scorecard (table)
- All 8 sections with scores and weights
- Color-code interpretation: 9+ green, 7-8.9 yellow, below 7 red

### Section 3: Top 5 Priority Fixes
- From priority-ranking.md, extract the top 5 items
- For each: criterion name, Baymard stat, specific fix action
- Keep each fix to 2-3 sentences maximum

### Section 4: Upstream Skill Impact
- Which PDP skills need revision
- Scope of revision (spec update, partial rerun, full rerun)

### Section 5: Proceed / Flag / Halt Decision
- Final recommendation with rationale
- If proceeding: what to carry forward as A/B test hypotheses (from priority ranking)
- If flagged: which sections must improve and by how much
- If halted: what structural rework is needed

**Step 3:** Write in direct, scannable style:
- Short paragraphs (2-3 sentences max)
- Tables for data
- Bold for key numbers and verdicts
- No filler, no pleasantries, no "overall the page is well-designed"

**Step 4:** Write `PDP-AUDIT-SUMMARY.md`.

## Quality Criteria

- Score and verdict immediately visible (first 3 lines)
- All 8 sections in scorecard
- Top 5 fixes are from priority ranking (not invented)
- Each fix includes the Baymard stat and specific action
- No JSON or code blocks in narrative sections
- No filler language ("overall well-designed", "great job")
- Scannable — a reader can get the key takeaways in 30 seconds
- File exists after execution

## Failure Modes

- Summary does not match report -> Cross-reference composite score, section scores, and priority items against the JSON report. They must match exactly.
- Top 5 fixes are vague -> Pull specific fix recommendations from priority-ranking.md. Each must name the exact action.
- Summary is too long (> 2 pages) -> Cut. Executive summaries are summaries. Detail lives in the JSON report.
- Missing decision recommendation -> Always end with a clear Proceed/Flag/Halt decision and rationale.
