# 4.2 — Summary Writer

> **Skill:** LP-18-ab-test-plan
> **Layer:** 4
> **Position:** 2 of 3
> **Purpose:** Write a human-readable `AB-TEST-PLAN-SUMMARY.md` from the compiled `ab-test-plan.json`. This is the primary deliverable for the human reviewer — it must be clear, actionable, and scannable.
> **Effort Level:** sonnet
> **Dependencies:** `ab-test-plan.json`
> **Output:** `AB-TEST-PLAN-SUMMARY.md`

---

## INPUT

```yaml
input:
  test_plan: "ab-test-plan.json"
```

---

## EXECUTION

### Step 1: Read ab-test-plan.json

Parse the complete test plan. Extract all data needed for the summary.

### Step 2: Write Page Overview Section

Summarize:
- Page type, audit score, verdict
- Baseline conversion rate and source
- Estimated monthly traffic (or "unknown")
- Total tests planned
- Test categories represented
- Estimated total test duration (sum of all phases)

### Step 3: Write Priority-Ranked Tests Section

For each test (in priority-rank order), write a scannable summary:
- Test ID and descriptive title
- Category
- Full hypothesis statement
- Control state (exact current element)
- Variant (exact proposed change)
- Expected lift with source
- Implementation difficulty
- Sample size and estimated duration

Use clear headers and formatting. Each test should be readable in 30 seconds.

### Step 4: Write Testing Sequence Section

Explain the phased approach:
- Phase 1: which tests, why first, total duration
- Phase 2: which tests, dependencies on Phase 1, total duration
- Phase 3: which tests, why last, total duration
- Total estimated timeline

### Step 5: Write Conflict Map

Table showing:
- Which tests conflict with each other
- The shared element causing the conflict
- The recommended sequence

### Step 6: Write Low-Traffic Advisory (if applicable)

If the page has low traffic (<500/month) or traffic is unknown:
- Specific guidance on which tests to run
- Sequential testing recommendation
- Realistic timeline expectations
- Suggestion to increase MDE for practical testing

### Step 7: Write Implementation Notes

For each test, brief notes on:
- What the implementation team needs to build
- Estimated level of effort (copy change vs design change vs structural change)
- Any dependencies on external resources (new testimonials, video production, etc.)

### Step 8: Compile and Write

Assemble all sections into `AB-TEST-PLAN-SUMMARY.md`.

## QUALITY CRITERIA

- Every test from ab-test-plan.json represented in the summary
- Tests in priority-rank order
- Testing sequence clearly explained with durations
- Conflict map complete
- Low-traffic advisory included when applicable
- Implementation notes practical and specific
- Document is scannable — a reviewer can understand the plan in 5 minutes
- File exists after execution

---

## FAILURE MODES

- ab-test-plan.json not available -> HALT. Cannot write summary without compiled plan.
- Formatting errors make the document hard to scan -> Use consistent header levels, tables, and bullet points. No dense paragraphs.
- Missing duration estimates (traffic unknown) -> Include sample sizes but note: "Duration estimates require traffic data. See duration lookup table in the test plan."
