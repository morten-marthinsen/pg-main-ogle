# 4.1 — Proof Architecture Compiler

**Skill:** LP-05-social-proof-architecture
**Position:** Layer 4, Microskill 1
**Purpose:** Assemble the validated proof architecture into the final proof-architecture.json output file. This is a compilation task — no new decisions. Pull data from Layer 2 outputs (validated by Layer 3) and format into the JSON schema defined in LP-05-AGENT.md.
**Effort Level:** medium
**Dependencies:** `proof-density-map.md` (2.2), `proof-type-sequence.md` (2.3), `proof-format-direction.md` (2.4), `architecture-validation.md` (3.1), `coverage-audit.md` (3.2), `wave-pattern-selection.md` (1.3), `proof-gap-analysis.md` (1.4), `volume-requirements.md` (1.2)
**Output:** `proof-architecture.json`

---

## Input Schema

```yaml
input:
  # Layer 1 outputs
  wave_pattern_selection: "[from 1.3 — mode, pattern, zone boundaries]"
  volume_requirements: "[from 1.2 — threshold check, totals]"
  proof_gap_analysis: "[from 1.4 — gaps with severity]"
  proof_inventory: "[from 0.2 — inventory summary]"

  # Layer 2 outputs
  proof_density_map: "[from 2.2 — per-section element specs]"
  proof_type_sequence: "[from 2.3 — within-section ordering]"
  proof_format_direction: "[from 2.4 — format codes and directions]"

  # Layer 3 outputs
  architecture_validation: "[from 3.1 — score and pass/fail per point]"
  coverage_audit: "[from 3.2 — 3-test results]"

  # Upstream
  page_type: "[type_a | type_b | hybrid]"
  project_name: "[from page-brief.json]"
```

---

## Execution Logic

**Step 1:** Verify GATE_3 passed:
- Architecture validation score >= 7.0/8
- Coverage audit overall = PASS
- If either failed, this microskill should not be running -> HALT

**Step 2:** Build the `proof_strategy` block:
- Pull `mode` and `wave_pattern` from 1.3
- Write `mode_rationale` from 1.3 decision rationale

**Step 3:** Build the `proof_inventory_summary` block:
- Pull all inventory counts from 0.2
- Set `inventory_sufficiency` from 0.2 assessment

**Step 4:** Build the `volume_targets` block:
- Pull planned counts by type from 2.2 volume verification table
- Pull threshold check result from 1.2

**Step 5:** Build the `section_proof_map` array:
For each section in the sequence:
- Pull section number, id, and name from section-sequence.json
- Pull density level from 2.2
- For each proof element in the section:
  - Assign a unique `slot_id` in format: `[section_id]_proof_[N]`
  - Pull type code, rank, count from 2.2
  - Pull format from 2.4
  - Pull quality requirement from 2.2
  - Pull content source (available/requires_sourcing) from 2.2
  - Pull placement_within_section from 2.4
- Pull within-section element order from 2.3
- Pull transition_in and transition_out from 2.1
- Assign wave_zone from 1.3 zone boundaries

**Step 6:** Build the `proof_wave_summary` block:
- Pull zone definitions from 1.3

**Step 7:** Build the `proof_gaps` array:
- Pull all gaps from 1.4 with severity and sourcing recommendations

**Step 8:** Set `architecture_score` from 3.1.

**Step 9:** Set `coverage_audit_result` from 3.2.

**Step 10:** Set `downstream_handoffs` with standard LP-05 handoff messages.

**Step 11:** Write the complete JSON to `proof-architecture.json`.

---

## Output Schema

The output follows the exact `proof-architecture.json` schema defined in LP-05-AGENT.md. Refer to that schema for the complete field list.

Key structural requirements:
- Every section from section-sequence.json must appear in `section_proof_map`
- Every proof element must have all required fields: slot_id, type, rank, count, format, quality_requirement, content_source, placement_within_section
- All numeric fields must be actual numbers (not strings)
- All proof gaps must be included with severity classifications
- `schema_version` = "1.0"
- `created` = ISO timestamp

---

## Quality Criteria
- JSON is valid and parseable
- All data pulled from validated Layer 2/3 outputs (no new data invented)
- Section proof map has entries for every section
- Proof elements have all required fields
- Volume targets are consistent with density map totals
- Gaps from 1.4 are fully included

## Failure Modes
- Layer 3 gate not passed -> HALT. Cannot compile until validation passes.
- Density map totals don't match volume targets -> Reconcile before writing. Update volume targets to match actual planned elements.
- Missing field in any proof element -> Fill from source output. If source output is also missing the field, return to Layer 2.
