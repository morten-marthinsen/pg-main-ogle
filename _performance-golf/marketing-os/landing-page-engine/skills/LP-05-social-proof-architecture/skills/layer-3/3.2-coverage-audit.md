# 3.2 — Coverage Audit

**Skill:** LP-05-social-proof-architecture
**Position:** Layer 3, Microskill 2
**Purpose:** Run three diagnostic tests on the completed proof architecture: (1) Proof Desert Detection, (2) Proof Monotony Detection, (3) Type A/B Pattern Integrity. These tests catch structural failures that the 8-point validator may miss because they require full-sequence analysis rather than individual-point scoring.
**Effort Level:** medium

## Test 1: Proof Desert Detection

**Definition:** A proof desert is 3 or more consecutive sections where the proof density is "none."

**Why it matters:** Visitors who scroll through 3+ sections without encountering any proof signal lose accumulated trust. By the time they reach the next proof block, they may have already bounced or formed negative impressions. A single light proof element (an inline mention, a badge, a quick stat) breaks the desert.

**Execution:**
1. List all sections in order with their density levels
2. Scan for consecutive "none" density sections
3. Count the longest run of consecutive "none" sections
4. If any run >= 3: FAIL

**Exception:** Type B "How It Works" section can be "none" density without triggering a desert warning IF the sections immediately before AND after it have proof. The "How It Works" section is a visual/educational section where proof is inappropriate.

```yaml
DESERT-TEST:
  sections_in_order:
    - section_1: "[density]"
    - section_2: "[density]"
    [all sections]

  consecutive_none_runs:
    - run_1: "Sections [N] through [N] — [count] consecutive none"
    [all runs]

  longest_run: "[count]"
  desert_found: "[Y/N — any run >= 3]"
  result: "[PASS | FAIL]"
  fix: "[If FAIL: Insert light proof (inline_mention or badge) in section [N] to break the desert]"
```

## Test 2: Proof Monotony Detection

**Definition:** Proof monotony occurs when a single proof type accounts for > 70% of all proof elements, OR when fewer than 3 unique proof types are used across the full page.

**Why it matters:** Different visitors need different evidence types to believe. A clinical-minded visitor needs studies. A social-proof-driven visitor needs testimonial volume. An authority-seeking visitor needs expert endorsements. Monotonous proof addresses one skepticism mode and ignores the others.

**Execution:**
1. Count total proof elements across all sections
2. Count elements per proof type code
3. Calculate each type's percentage of total
4. Identify the dominant type and its percentage
5. Count unique types used
6. If dominant type > 70% OR unique types < 3: FAIL

```yaml
MONOTONY-TEST:
  total_elements: "[count]"

  type_distribution:
    - type: "[code]"
      count: "[N]"
      percentage: "[N%]"
    [all types]

  dominant_type: "[code]"
  dominant_percentage: "[N%]"
  unique_types: "[count]"

  over_concentration: "[Y/N — dominant > 70%]"
  insufficient_variety: "[Y/N — unique < 3]"
  result: "[PASS | FAIL]"
  fix: "[If FAIL: Add [proof types] to sections [N] to diversify. Reduce [dominant type] in sections [N] or replace with alternative types.]"
```

## Test 3: Type A/B Pattern Integrity

**Definition:** The wave pattern's density rhythm must match the page type's decision process. Type A uses an oscillating wave (light -> medium -> heavy -> light -> heavy). Type B uses a front-loaded volume pattern (heavy -> light -> none -> heavy/maximum).

**Why it matters:** Type A visitors are on a persuasion journey — they need proof to build incrementally. Dumping maximum proof in the lead section overwhelms. Type B visitors want proof IMMEDIATELY — they're making a fast purchase decision. Delaying proof until mid-page loses them.

**Execution:**
1. Map each section's density to the expected wave pattern for the page type
2. Check for violations:

**Type A violations:**
- Maximum density in the first 2 sections (too heavy too early)
- No heavy density sections in the proof block positions
- No proof in close zone (no transformation proof before final CTA)

**Type B violations:**
- Light or none density above fold (rating strip/review count MUST be above fold)
- No maximum density section (review cascade MUST exist)
- Heavy density in education sections (how it works, ingredients — should be none to medium)

**Hybrid violations:**
- Zone 1 (Type B above fold) missing immediate proof
- Zone 2 (Type A education) having maximum density (should follow Type A wave)
- Zone 3 (Conversion) missing review cascade

```yaml
PATTERN-INTEGRITY-TEST:
  page_type: "[type_a | type_b | hybrid]"
  wave_pattern: "[code]"

  density_by_section:
    - section_1: "[density]" — expected: "[expected for this position in wave]"
    - section_2: "[density]" — expected: "[expected]"
    [all sections]

  violations:
    - "[description of violation — e.g., 'Section 1 (above fold) has density = none; Type B requires heavy (rating strip)']"
    [all violations]

  violation_count: "[count]"
  result: "[PASS (0 violations) | FAIL (1+ violations)]"
  fix: "[For each violation: specific density adjustment to make]"
```

## Execution Logic

**Step 1:** Run Test 1 (Proof Desert Detection).
**Step 2:** Run Test 2 (Proof Monotony Detection).
**Step 3:** Run Test 3 (Type A/B Pattern Integrity).
**Step 4:** Aggregate results. ALL THREE tests must PASS for the coverage audit to pass.
**Step 5:** If any test fails, document specific fixes.

## Quality Criteria
- All 3 tests run with full evidence documentation
- No test skipped because "it looks fine"
- Violations specifically identified with section numbers
- Fixes reference specific Layer 2 files to update
- Overall audit result is binary (PASS/FAIL — no "conditional pass")

## Failure Modes
- All 3 tests fail simultaneously -> Return to Layer 1 and reassess proof strategy. The architecture has fundamental problems.
- Test 3 fails (pattern integrity) -> This is the most critical failure. The wave pattern may need to be re-selected (1.3) before Layer 2 outputs can be fixed.
- Test 1 fails (desert) -> Usually fixable by adding a single light proof element. Return to 2.1 and 2.2 only.
