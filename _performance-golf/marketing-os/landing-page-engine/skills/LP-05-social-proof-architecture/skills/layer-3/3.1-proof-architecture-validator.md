# 3.1 — Proof Architecture Validator

**Skill:** LP-05-social-proof-architecture
**Position:** Layer 3, Microskill 1
**Purpose:** Score the complete proof architecture against the 8-point validation checklist. Every point is scored 0 (fail) or 1 (pass). Minimum 7.0/8 to proceed to Layer 4. If score is below threshold, identify specific failures and return to Layer 2 for revision.
**Effort Level:** medium
**Dependencies:** `proof-placement-plan.md` (2.1), `proof-density-map.md` (2.2), `proof-type-sequence.md` (2.3), `proof-format-direction.md` (2.4)
**Output:** `architecture-validation.md`

## The 8-Point Validation Checklist

### Point 1: Early Trust Anchoring (1 point)
**Question:** Does proof appear within the first 25% of the page?
**Check:** First proof element's section number <= first_25_percent_section
**Evidence required:** Section number and proof type of the earliest proof element
**PASS:** Proof present at or before first-25% boundary
**FAIL:** First proof appears after the 25% mark

### Point 2: Early Proof Quality (1 point)
**Question:** Is the early proof an authority-type or volume-type signal (not a generic testimonial)?
**Check:** Early proof type is one of: `expert_endorsement`, `clinical_study`, `media_mention`, `certification`, `volume_signal`, `stat_bar`, `widget`
**Evidence required:** The type code of the earliest proof element
**PASS:** Early proof is authority or volume type
**FAIL:** Early proof is a generic text testimonial (low-persuasiveness as a trust anchor)

### Point 3: Testimonial Volume Threshold — Type A (1 point)
**Question:** Are >= 15 testimonials planned across all proof blocks? (Type A and Hybrid only)
**Check:** Sum of all `text_testimonial` + `photo_testimonial` + `video_testimonial` + `before_after` elements across all sections
**Evidence required:** Total count with breakdown by section
**PASS:** Count >= 15 (Type A) or >= 10 (Hybrid Type A zone)
**FAIL:** Count below threshold
**N/A:** Type B pages — auto-PASS this point

### Point 4: Review Volume Threshold — Type B (1 point)
**Question:** Are >= 50 reviews planned in the cascade section? (Type B and Hybrid only)
**Check:** Sum of `platform_review` elements in cascade sections
**Evidence required:** Review count in cascade section
**PASS:** Count >= 50 (Type B) or >= 25 (Hybrid Type B zone)
**FAIL:** Count below threshold
**N/A:** Type A pages — auto-PASS this point

### Point 5: Specific Outcomes Required (1 point)
**Question:** Do all planned testimonials include specific outcomes/numbers in their quality requirements?
**Check:** Review quality_requirement field for every testimonial element in the density map
**Evidence required:** Count of testimonials with specific outcome requirements vs. total testimonials
**PASS:** 100% of testimonials have outcome-specific quality requirements
**FAIL:** Any testimonial has a generic quality requirement (no specific outcome/numbers)

### Point 6: Before/After Deployment (1 point)
**Question:** Is before/after format used where applicable?
**Check:** If product has measurable transformation outcomes (weight loss, energy, appearance, revenue, etc.) AND before/after proof is available or could be sourced — is `before_after` format used in at least one proof block?
**Evidence required:** Before/after element count + product applicability assessment
**PASS:** Before/after used where applicable, OR product has no measurable transformation outcomes (N/A = PASS)
**FAIL:** Product has applicable transformation outcomes but before/after format is absent from architecture

### Point 7: Wave Pattern Match (1 point)
**Question:** Does the selected wave pattern match the page type classification?
**Check:** `type_a` page uses `type_a_wave`, `type_b` uses `type_b_wave`, `hybrid` uses `hybrid_wave`
**Evidence required:** Page type + wave pattern code
**PASS:** Pattern matches type
**FAIL:** Pattern does not match type (Type A/B conflation)

### Point 8: Proof Type Diversity (1 point)
**Question:** Are at least 3 different proof types used across the page?
**Check:** Count of unique proof type codes across all sections
**Evidence required:** List of unique types used
**PASS:** 3 or more unique types
**FAIL:** Fewer than 3 unique types (proof monotony)

---

## Execution Logic

**Step 1:** Run each of the 8 checks in sequence. For each check:
- State the question
- Show the evidence
- State PASS or FAIL
- If FAIL: State what must change

**Step 2:** Calculate total score (sum of all PASS points out of 8).

**Step 3:** Determine action:
- 8/8: Architecture ready. Proceed to Layer 4.
- 7/8: Minor gap. Proceed to Layer 4 with the failed point noted for human review.
- 5-6/8: Structural issues. Return to Layer 2 and revise the failed points.
- Below 5/8: Architecture rebuild required. Return to Layer 1 and reassess strategy.

**Step 4:** If any points failed, write specific revision instructions referencing which Layer 2 output must be updated.

## Quality Criteria
- All 8 points evaluated with evidence
- No point skipped or assumed to pass
- Failed points have specific revision instructions
- Score calculated correctly
- Action recommendation matches scoring thresholds

## Failure Modes
- Attempted to proceed with score < 7 -> HALT. Revise Layer 2 outputs.
- Point 7 (wave pattern match) fails -> This is a CRITICAL failure. Return to 1.3 and re-select pattern.
- Multiple points fail simultaneously -> Address in priority order: Point 7 (pattern match) -> Point 1 (early trust) -> Points 3/4 (volume) -> Point 8 (diversity) -> Points 5/6 (quality) -> Point 2 (early quality)
