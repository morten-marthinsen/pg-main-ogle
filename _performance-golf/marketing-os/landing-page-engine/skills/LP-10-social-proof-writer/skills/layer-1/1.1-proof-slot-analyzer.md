# 1.1 — Proof Slot Analyzer

**Skill:** LP-10-social-proof-writer
**Position:** Layer 1, Microskill 1
**Purpose:** Categorize every `slot_id` from `proof-architecture.json` into one of the 9 proof copy types. Determine which Layer 2 microskills need to execute. This is the task assignment step — it converts the proof architecture into a writing plan.
**Effort Level:** medium
**Dependencies:** `proof-slots-loaded.md` (0.2), `brief-loaded.md` (0.1)
**Output:** `proof-slot-analysis.md`

---

## Input Schema

```yaml
input:
  proof_slots: "[from proof-slots-loaded.md — complete slot inventory]"
  page_type: "[from brief-loaded.md — type_a | type_b | hybrid]"
  proof_mode: "[from proof-slots-loaded.md — proof_saturation | authority_substitution]"
```

---

## Execution Logic

1. **Load the complete slot inventory** from `proof-slots-loaded.md`.

2. **For each slot_id,** classify it into one of the 9 proof copy types:

   | Proof Type Code (from LP-05) | Maps to LP-10 Copy Type | Layer 2 Microskill |
   |-------|------|------|
   | `text_testimonial` | testimonial_format | 2.2 |
   | `photo_testimonial` | testimonial_format | 2.2 |
   | `platform_review` | review_section | 2.3 |
   | `before_after` | before_after_caption | 2.4 |
   | `expert_endorsement` | expert_endorsement | 2.5 |
   | `clinical_study` | expert_endorsement (framing) | 2.5 |
   | `volume_signal` | volume_signal | 2.7 |
   | `stat_bar` | volume_signal | 2.7 |
   | `video_testimonial` | video_testimonial_cta | 2.2 (formatting) + special handling |
   | `media_mention` | expert_endorsement (framing) | 2.5 |
   | `certification` | N/A — handled by LP-08 Trust Elements | SKIP |
   | `performance_guarantee` | N/A — handled by LP-11 Offer Writer | SKIP |

3. **Identify proof blocks** — groups of proof slots within the same section that need:
   - A testimonial intro (2.1) — if the section contains 2+ testimonials/reviews
   - Transition IN copy (2.6) — if the section has `transition_in` direction from LP-05
   - Transition OUT copy (2.6) — if the section has `transition_out` direction from LP-05

4. **Identify case study slots** — any slot where LP-05's quality requirement mentions "detailed case study" or "success story with metrics." These get case study intro treatment (part of 2.1).

5. **Build the Layer 2 execution plan:**
   - Which microskills need to run (only those with assigned slots)
   - How many writing assignments per microskill
   - Execution order recommendation

6. **Flag any ambiguous slots** — where LP-05's type code doesn't cleanly map to an LP-10 copy type. Document interpretation and rationale.

7. **Write `proof-slot-analysis.md`.**

## Quality Criteria
- Every slot_id from proof-slots-loaded.md classified (100% coverage — no orphan slots)
- Classification uses the mapping table above — not ad-hoc interpretation
- Layer 2 execution plan clearly shows which microskills run and which are skipped
- Proof blocks identified for intro treatment (threshold: 2+ proof elements in a section)
- Transition pairs extracted from LP-05's transition directions
- Ambiguous slots documented with rationale (not silently interpreted)

## Failure Modes
- Slot_id not matching any proof type code -> Flag as ambiguous, assign to closest copy type, document rationale.
- Classifying `certification` or `performance_guarantee` slots as LP-10 writing assignments -> These belong to LP-08 and LP-11 respectively. Mark as SKIPPED.
- All slots classified to a single copy type -> Verify against proof-architecture.json. If correct (e.g., a page with only testimonials), proceed. If incorrect, re-examine classification.
- Missing transition directions from LP-05 -> Generate IN/OUT transitions anyway for sections with density >= medium. Document that direction was inferred, not specified.
