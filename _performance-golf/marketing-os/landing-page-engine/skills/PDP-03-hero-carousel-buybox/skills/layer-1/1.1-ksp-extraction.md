# 1.1 — KSP Extraction

**Skill:** PDP-03-hero-carousel-buybox
**Position:** Layer 1, Microskill 1
**Purpose:** Extract the top 10 Key Selling Points (KSPs) from the brief and research data. These KSPs map directly to carousel slide content — each slide must communicate at least one KSP. Without a ranked KSP inventory, carousel slides become generic category claims instead of product-specific persuasion.
**Effort Level:** medium
**Dependencies:** 0.1 input-verification.md, 0.2 pdp-reference-load.md
**Output:** `ksp-inventory.md`

---

## Input Schema

```yaml
input:
  input_verification: "[path to input-verification.md]"
  pdp_reference_load: "[path to pdp-reference-load.md]"
  page_brief: "[path to page-brief.json]"
  conversion_intelligence: "[path to LP-01 output, or 'not available']"
```

## Execution Logic

**Step 1:** Read `input-verification.md` to get product data, ingredient/feature list, audience pain points, and purchase motivations.

**Step 2:** Read LP-01 conversion intelligence (if available) to get audience verbatim quotes and competitor weak points.

**Step 3:** Generate initial KSP candidates from all sources. A KSP is a specific, provable claim about the product that matters to the buyer. Sources:
- Product ingredients/features → functional benefits
- Audience pain points → product solutions
- Competitor gaps → differentiation claims
- Research/testing data → credibility claims
- Manufacturing/sourcing → quality claims

**Step 4:** Score each KSP candidate on three dimensions:
- **Relevance (1-5):** How much does this matter to the target buyer?
- **Differentiability (1-5):** Can competitors make the same claim?
- **Provability (1-5):** Can this be backed with evidence?

**Step 5:** Rank KSPs by composite score (Relevance × 2 + Differentiability + Provability). Select top 10.

**Step 6:** Map each KSP to the NLS carousel story position where it fits best:
| Story Position | KSP Types That Fit |
|---------------|-------------------|
| 1. Outcome | Primary benefit, headline promise |
| 2. Problem Signs | Pain points the product solves |
| 3. Ingredients | Specific ingredients/features with benefits |
| 4. Enjoyment | Taste, texture, experience, ease of use |
| 5. How It Works | Mechanism, process, science |
| 6. What to Expect | Timeline, milestones, results progression |
| 7. How to Use | Dosage, application, simplicity |
| 8. Credibility | Studies, expert endorsement, certifications |
| 9. Comparison | Superiority claims, competitive advantages |
| 10. Testimonials | Social proof, real results |

**Step 7:** Verify coverage — every story position should have at least one KSP mapped. If a position has no KSP, flag it as a data gap.

**Step 8:** Write `ksp-inventory.md`.

## Quality Criteria
- Exactly 10 KSPs selected and ranked
- Each KSP is specific (not "high quality" but "third-party tested for 150+ contaminants")
- Each KSP scored on all three dimensions with justified ratings
- Every NLS story position has at least one KSP mapped (or gap flagged)
- Rejected candidates documented with reasons
- KSPs are product-specific, not generic category claims
- File exists after execution

## Failure Modes
- Fewer than 10 distinct KSPs from brief data → Flag gap, note which positions need human input
- All KSPs cluster in 2-3 story positions → Redistribute or flag coverage gaps
- KSPs are generic ("made with natural ingredients") → Rewrite to be specific ("500mg KSM-66 Ashwagandha per serving")
- input-verification.md not found → HALT, run Layer 0 first
- No differentiability in any KSP → Flag — product may lack competitive moat
