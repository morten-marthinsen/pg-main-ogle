# 1.3 — Transition Strategy

> **Skill:** LP-15-page-assembly
> **Layer:** 1
> **Position:** 3 of 3
> **Purpose:** Plan section-to-section transitions for the entire page. Identify the threading anchors that will maintain narrative coherence from headline to close. Produce a transition brief for each section boundary.
> **Effort Level:** sonnet
> **Dependencies:** `section-order-plan.md`, `content-load.md`, `brief-load.md`
> **Output:** `transition-strategy.md`

## EXECUTION

**POSITIONAL REINFORCEMENT:** You are PLANNING transitions, not writing them. Layer 2 writes the actual transition copy. You identify what each transition must accomplish and which threading anchors must be maintained.

### Step 1: Identify the Core Thread

From `brief-load.md` and `content-load.md`, identify the three threading anchors that must run through the entire page:

1. **The Promise** — the headline promise (from LP-07 Arena selection). What does the page commit to delivering?
2. **The Mechanism** — the mechanism/root cause that explains WHY the promise can be kept. What is the "aha" insight?
3. **The Proof** — the evidence chain that validates the mechanism and promise. What makes it credible?

These three anchors must be referenced or reinforced at transition points. A page where the promise is stated in the headline but never mentioned again until the close has a broken thread.

### Step 2: Plan Each Transition

For each section boundary identified in `section-order-plan.md`, create a transition brief:

**Transition Brief Template:**
```
Boundary [X]: [Section A] -> [Section B]
- Previous section's key takeaway: [what the reader now knows/feels]
- Next section's purpose: [what the reader needs to learn/feel next]
- Threading anchor to reinforce: [promise / mechanism / proof — which one?]
- Transition type: [bridge / pivot / escalation / callback / header-only]
- Transition direction: [emotional -> logical / logical -> emotional / emotional -> emotional / logical -> logical]
- Estimated transition length: [1 sentence / 1 paragraph / header only]
- Notes: [any special considerations]
```

### Step 3: Classify Transition Types

**Bridge:** Connects two related sections smoothly. "Now that you understand X, let's look at Y..."
**Pivot:** Shifts the reader's frame. "But here's what most people don't realize..."
**Escalation:** Raises stakes. "And it gets worse..."
**Callback:** References an earlier section. "Remember the [mechanism] we discussed? Here's the proof..."
**Header-Only:** For scan-optimized sections (Type B), the section header IS the transition.

### Step 4: Apply Page Type Rules

**Type A transitions:**
- Every boundary gets a written transition (1 sentence to 1 paragraph)
- Story-driven transitions use emotional language
- Education transitions use "but here's what..." pivots
- Proof transitions use "don't take my word for it" bridges
- Conversion transitions use urgency/consequence language

**Type B transitions:**
- Most boundaries use header-only transitions ("How It Works", "What Our Customers Say")
- Transitions between scan-optimized sections are visual/structural, not narrative
- Only the benefits-to-proof and proof-to-offer boundaries may need written transitions
- Keep everything scannable

**Hybrid transitions:**
- Type B above-fold with header-only transitions
- Type A body sections with written transitions
- The transition from Type B above-fold to Type A body is critical — must be smooth

### Step 5: Map Threading Anchor Reinforcements

Identify where in the page each threading anchor is reinforced:

| Threading Anchor | Introduced | Reinforced | Reinforced | Closed |
|-----------------|-----------|-----------|-----------|--------|
| Promise | Above Fold (headline) | [Section X transition] | [Section Y transition] | Close/CTA |
| Mechanism | Mechanism section | [Section X transition] | [Section Y transition] | Close/CTA |
| Proof chain | First proof block | [Section X transition] | [Section Y transition] | Final proof/close |

**Rule:** Each threading anchor must be reinforced at least twice between introduction and close. If it is introduced in the headline and not mentioned again until the close, the thread is broken.

### Step 6: Write transition-strategy.md

---

## QUALITY CRITERIA

- Every section boundary from `section-order-plan.md` has a transition brief
- Three threading anchors identified (promise, mechanism, proof)
- Each anchor has a reinforcement map showing where it is introduced, reinforced, and closed
- Transition types appropriate for page type (Type A = mostly written, Type B = mostly header-only)
- No boundary left unplanned
- File exists after execution

---

## FAILURE MODES

- Promise anchor cannot be identified (headline not available) -> Use the headline from `arena-verification-load.md` (authoritative text). If still not available, HALT — cannot plan transitions without knowing the promise.
- Mechanism anchor not clear from content -> Note "MECHANISM_UNCLEAR" and plan transitions that leave room for the mechanism to be identified during assembly. Flag for human review.
- Type B page getting full written transitions -> Type B pages are scan-optimized. Most boundaries should be header-only. If you are planning written paragraphs for every Type B boundary, you are treating it like Type A. Check the page type.
