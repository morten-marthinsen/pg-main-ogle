# 0.2 — Urgency Data Loader

**Skill:** LP-13-urgency-scarcity
**Position:** Layer 0, Microskill 2
**Purpose:** Extract and consolidate all urgency-relevant data from upstream packages and the project brief into a single urgency data file. This is the raw material that Layer 1 uses to decide which urgency types are justified and which are not. If the data does not support urgency, this microskill must say so explicitly.
**Effort Level:** low
**Dependencies:** `packages-loaded.md` (0.1)
**Output:** `urgency-data.md`

---

```yaml
input:
  packages_loaded: "[from packages-loaded.md]"

    deadline_data: "[from packages-loaded.md extraction]"

  lp06_urgency_strategy: "[from packages-loaded.md extraction]"
  lp06_urgency_types_recommended: "[from packages-loaded.md extraction]"
  lp11_offer_pricing_exists: "[Y/N -- has LP-11 already run?]"
  page_type: "[type_a | type_b | hybrid]"
  vertical: "[from page-brief.json]"
```

## Execution Logic

1. **Inventory all urgency data points** from the brief and upstream packages. For each of the 6 urgency types, document what data exists (or does not exist):

   **Type 1 -- Supply Scarcity:**
   - Batch manufacturing data? [Y/N + details]
   - Ingredient sourcing constraints? [Y/N + details]
   - Inventory numbers? [Y/N + details]
   - Manufacturing schedule? [Y/N + details]

   **Type 2 -- Time-Based:**
   - Launch window dates? [Y/N + details]
   - Enrollment period start/end? [Y/N + details]
   - Seasonal relevance? [Y/N + details]
   - Event-tied deadline? [Y/N + details]

   **Type 3 -- Price-Based:**
   - Introductory price documented? [Y/N + details]
   - Future price documented? [Y/N + details]
   - Price increase date? [Y/N + details]
   - Reason for price increase? [Y/N + details]

   **Type 4 -- Availability Window:**
   - Cohort size limit? [Y/N + details]
   - Reason for capacity limit? [Y/N + details]
   - Next availability date? [Y/N + details]
   - Waitlist mentioned? [Y/N + details]

   **Type 5 -- Bonus Deadline:**
   - Bonus list with expiration dates? [Y/N + details]
   - Reason bonuses expire? [Y/N + details]
   - Partner/licensing constraints? [Y/N + details]

   **Type 6 -- Natural Urgency:**
   - Progressive condition data? [Y/N + details]
   - Cost-of-delay evidence? [Y/N + details]
   - Competitive urgency data? [Y/N + details]
   - Research on delay consequences? [Y/N + details]

2. **Rate data sufficiency** for each type:
   - **SUFFICIENT:** Enough data to write specific, justified urgency copy
   - **PARTIAL:** Some data exists but not enough for full justification (flag for client input)
   - **INSUFFICIENT:** No data or only generic data -- cannot justify this urgency type

3. **Check LP-06 recommendations** against available data:
   - If LP-06 recommended Type X but data is INSUFFICIENT for Type X, flag the gap
   - If LP-06 recommended no urgency and data confirms no urgency, confirm alignment

4. **Identify the strongest justifiable urgency type(s)** based on data sufficiency -- this is a preliminary recommendation for Layer 1, not a final decision.

5. **Write `urgency-data.md`** with complete inventory.

## Quality Criteria
- All 6 urgency types inventoried with explicit data or "No data" for each field
- Data sufficiency rating is honest -- INSUFFICIENT means INSUFFICIENT, not "maybe we can make it work"
- LP-06 recommendations checked against available data
- Preliminary assessment is conservative -- when in doubt, rate as PARTIAL or INSUFFICIENT
- Output is self-contained -- Layer 1 can make classification decisions from this file alone

## Failure Modes
- Rating INSUFFICIENT data as SUFFICIENT to enable urgency copy -> This is the path to fabricated urgency. Rate honestly.
- Inventing urgency data not present in any upstream file -> Only extract what exists. Do not infer deadlines, batch sizes, or capacity limits.
- Ignoring LP-06 recommendations entirely -> LP-06 did the strategic thinking. Check its recommendations against data, do not override without data-based reason.
- Rating all types as INSUFFICIENT without checking for Natural Urgency -> Natural urgency (cost of delay) often exists even when structural urgency data is absent. Check the vertical and condition for progressive damage patterns.
