# 1.2 — Scarcity Justification Planner

**Skill:** LP-13-urgency-scarcity
**Position:** Layer 1, Microskill 2
**Purpose:** Document the real, verifiable reason behind each selected urgency/scarcity claim BEFORE any copy is written. This is the anti-fabrication gate. Every urgency claim that will appear on the page must have its justification written here first. Layer 2 writers cannot create claims that are not pre-justified in this plan.
**Effort Level:** medium
**Dependencies:** `urgency-types-selected.md` (1.1), `urgency-data.md` (0.2), `packages-loaded.md` (0.1)
**Output:** `justification-plan.md`

---

## Execution Logic

1. **For each selected urgency type, build a justification profile:**

   **Step 1: State the claim.** What specific urgency/scarcity statement will the copy make? Write it as a concrete claim, not a vague direction.
   - BAD: "We'll mention limited supply"
   - GOOD: "We will state that this batch is limited to 5,000 bottles and the next batch ships in 6-8 weeks"

   **Step 2: Identify the real constraint.** What physical, financial, logistical, or temporal constraint makes this claim true?
   - Manufacturing batch size
   - Ingredient sourcing timeline
   - Launch window calendar date
   - Price schedule from business plan
   - Coaching capacity per instructor
   - Partner agreement expiration
   - Progressive condition research

   **Step 3: Document the data source.** Where in the upstream files is this constraint documented?
   - page-brief.json field: [field name]
   - offer-cta-architecture.json field: [field name]
   - Client input: [quote/reference]
   - Product constraint: [documented where]

   **Step 4: Run the "Why?" x3 test.**
   - Why is this urgent? [Answer with specific constraint]
   - Why does that constraint exist? [Answer with real-world reason]
   - Why can't the constraint be removed? [Answer -- if it CAN be removed, the urgency is artificial]

   **Step 5: Rate justification strength.**
   - **STRONG:** All three "Why?" answers are specific and verifiable. Copy can be written confidently.
   - **MODERATE:** First two "Why?" answers are specific. Third answer is reasonable but not independently verifiable. Copy can be written but flagged for client confirmation.
   - **WEAK:** Only the first "Why?" has a specific answer. Others are inferred. Do NOT write urgency copy from WEAK justifications -- flag for client input.

2. **For PARTIAL data types,** document exactly what data is missing and what the client needs to provide before urgency copy is finalized.

3. **For Natural Urgency (Type 6),** the justification framework is different:
   - State the specific cost of delay (what gets worse each day/week/month)
   - Cite research or data supporting the progressive nature of the problem
   - Specify the threshold/tipping point (if known) -- when delay becomes irreversible
   - This is not "fake urgency about a deadline" -- it is honest communication about the consequences of inaction

4. **Define the specific claims that Layer 2 is authorized to write.** Layer 2 writers cannot create claims beyond what is justified here.

5. **Write `justification-plan.md`** with complete profiles.

## Quality Criteria
- Every claim that will appear in urgency copy is pre-justified in this document
- Each justification has a documented data source (not invented)
- "Why?" x3 test completed for every claim
- No WEAK justifications proceeding to Layer 2 -- these must be flagged for client input
- Authorized claims list explicitly bounds what Layer 2 can write
- Forbidden claims list explicitly prevents Layer 2 from overstepping data

## Failure Modes
- Writing vague justifications ("supply is limited because we make small batches") without specific numbers -> Justification must be as specific as the claim. If the claim says "5,000 bottles per batch," the justification must explain why 5,000.
- Proceeding with WEAK justifications to avoid the no-urgency path -> Weak = not justified = do not write copy. Flag for client input.
- Not defining authorized claims for Layer 2 -> Without explicit bounds, Layer 2 may generate claims that go beyond what data supports.
- Inventing justification data not in any upstream file -> Justification must trace to documented data. If the data does not exist, the justification does not exist.
- Skipping the "Why?" x3 test ("the first Why is enough") -> The third "Why?" is the real test. It catches urgency that sounds real but is actually artificial.
