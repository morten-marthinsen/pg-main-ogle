# Page Swiping Pre-Check — Run This BEFORE Writing Any Page

This checklist exists because of real mistakes made on the SPD Backend project. Every item below represents something that was wrong on a first draft and had to be fixed. Do not skip any step.

---

## PHASE 1: Information Gathering (Before Writing Anything)

### Pricing — Get EXACT numbers. NEVER guess or calculate.

- [ ] **Full/retail price** of the main product (the strikethrough price)
- [ ] **Offer price** of the main product (the actual price customer pays)
- [ ] **PG1 member pricing?** — Is there a PG1 toggle? If yes: PG1 price, non-PG1 price, savings amount
- [ ] **Upsell 1 full price** (the strikethrough)
- [ ] **Upsell 1 offer price** (on expanded page)
- [ ] **Upsell 1 downsell price** (the "double discount" price)
- [ ] **Upsell 2 full price** (the strikethrough)
- [ ] **Upsell 2 offer price** (on expanded page)
- [ ] **Upsell 2 downsell price** (the "double discount" price)
- [ ] **Upsell 2 downsell STRUCTURE** — Same product cheaper? Pick-1? Reduced quantity?

**Why this matters:** On SPD, I guessed $359 (was $299), $598 (was $499), and $189 (was $199). Every price was wrong.

### Bonuses — Don't invent them.

- [ ] **How many bonuses** does the main product have? List them by exact name.
- [ ] **Do the upsells have bonuses?** If yes, what are they? If no, confirm: "No bonuses on upsells, correct?"
- [ ] **Get exact bonus names** — including fun/marketing names (e.g., "Primal Speed Protocol" not just "Speed Protocol")

**Why this matters:** On SPD, I invented 5 bonuses across 3 pages. Zero of them were real.

### Product Details — Verify everything.

- [ ] **Exact product name** including branded/fun names (e.g., "Naughty 9-Wood" not "Super 9-Wood")
- [ ] **Feature counts** — Cross-reference VSL script AND micro-scripts doc (numbers may differ; use micro-scripts)
- [ ] **Presenter for each page** — Who presents the main VSL? Each upsell VSL? Who signs the "Our Promise" section?
- [ ] **Presenter ROLE** — Coach? Engineer? Speed specialist? Don't assume.
- [ ] **Year** for footer copyright — use current year

**Why this matters:** I used wrong product names, wrong feature counts (12 vs 15), wrong presenter roles ("engineer" instead of "coach"), and wrong year (2025 instead of 2026).

### Positioning & Terminology — Check for source docs.

- [ ] **Is there a micro-scripts or positioning document?** If yes, READ IT FIRST. It supersedes the VSL for page copy.
- [ ] **Check for KILLED/BANNED terms** — positioning docs may explicitly list terms to never use
- [ ] **Note the preferred terminology** — e.g., "face control" (correct) vs "clubface control" (wrong)
- [ ] **Note any binary frames** — e.g., "Fast AND straight, not fast OR straight"
- [ ] **Note any key taglines** — e.g., "Speed without direction just gets you lost quicker"

**Why this matters:** I used "Transfer Gap" (a killed term), "clubface control" (wrong), and missed the binary frame entirely.

### Page Structure

- [ ] **What pages are needed?** Confirm the exact list (delay, expanded, downsell, thank you, etc.)
- [ ] **Expanded page format?** Full page with delay header? Or offer-stack-only (append below delay)?
- [ ] **Delay page** — Does the user already have one written? If yes, create placeholder only.
- [ ] **Thank you page** — Model after existing PDF/template or custom?

**Why this matters:** I made the upsell 2 expanded as "offer stack only" when user wanted a full page.

## PHASE 2: Source Authority Hierarchy

When sources conflict, follow this order:

| Priority | Source | What It Governs |
|----------|--------|-----------------|
| 1 (highest) | **User's direct instructions** | Pricing, bonuses, structure, format |
| 2 | **Micro-scripts / positioning docs** | Language, terminology, frames, killed terms |
| 3 | **Existing PG pages and swipe templates** | Format patterns, PG1 pricing, badges, selectors |
| 4 | **VSL scripts** | Narrative, story arc, presenter info, product details |

The VSL is NOT the single source of truth for page copy. The micro-scripts doc often has cleaner, more refined language.

## PHASE 3: Writing Rules

### Copy Rules

- [ ] **Benefits before mechanisms** — Lead with WHAT it does, then HOW. ("Fixes your big miss" before "Y-Trac Weight System puts weight off-axis...")
- [ ] **Binary frames** — If positioning docs provide "AND X, not OR X" frames, use them prominently
- [ ] **Benefit-first CTAs** — "Yes, I Want Distance AND Accuracy" NOT "Yes, I Want The SpeedTrac"
- [ ] **Iron replacement framing** — "Replaces your 4 or 5 iron" NOT "for 175-220 yards"
- [ ] **Keep it simple** — "7 features all working together" > listing all 7 features by name
- [ ] **Two punchy paragraphs** > one long paragraph
- [ ] **Summary sentences** — End product descriptions with a bold summary like "That's how you gain distance AND accuracy."
- [ ] **Softer tone for senior golfers** — "heck of a deal" not "hell of a deal." "bad golfer" not "stink at golf."

### Golden Thread Rule

- [ ] **Every section must echo the same core concepts using the same LANGUAGE** — root cause name, solution mechanism, and binary frame should appear in problem, solution, bullets, testimonials, transitions, and CTAs
- [ ] **If a section could belong to a different product's page, it fails** — rewrite with this page's specific concepts

### Bullet Destination Rule

- [ ] **Bullets must tease the CORRECT downstream destination** — if CTA leads to a SpeedTrac VSL, bullets tease SpeedTrac content, not SRSW content
- [ ] **Read the destination's positioning docs/micro-scripts** before writing tease bullets
- [ ] **Include a counter-sell bullet** — "No gym, no speed sticks... no dropping $500 on a new driver" — eliminates alternatives with a price anchor

### Testimonial Rules

- [ ] **NEVER use placeholders** — `[TESTIMONIAL: Insert here]` is not acceptable
- [ ] **Testimonials must echo the page's golden thread** — each must contain 2+ identifiable echoes of page concepts (root cause name, binary frame, "What It's NOT" list, solution mechanism)
- [ ] **Write realistic testimonials** using:
  - Product's key language ("feel the face," "fast AND straight")
  - Specific numbers (mph gained, fairways hit, handicap, time using product)
  - Target demographic details (age 55-70, handicap 8-20)
  - Before/after contrast ("always sprayed it" → "dispersion got WAY tighter")

### Physical Product Rules

- [ ] **LH/RH selector** on EVERY page where the product is sold
- [ ] **Shaft flex selector** with the standard 5 options
- [ ] **"IN STOCK AND READY TO SHIP"** badge near CTA
- [ ] **"CUSTOMIZE YOUR [PRODUCT]"** header above selectors

### Design Directives

- [ ] **VSL thumbnail brief** — Include curiosity directive (e.g., "DOES NOT REVEAL the product")
- [ ] **Image references** — Be specific about what's shown (product only? product + bonus? product + box?)

## PHASE 4: Post-Writing QC

### Consistency Checks

- [ ] **Bonus count** matches EVERYWHERE — header, body, image brackets, CTA area, decline text
- [ ] **All prices** are consistent — strikethrough, offer, savings, percentage all calculated correctly
- [ ] **Product names** are consistent — check every instance (including decline text, guarantee section)
- [ ] **Presenter names and roles** are consistent throughout
- [ ] **Year in footer** is correct (current year)

### Killed Term Sweep

- [ ] Search all pages for any KILLED terms from the positioning doc
- [ ] Search for wrong terminology variants (e.g., "clubface control" if "face control" is correct)

### Template Compliance

- [ ] Section order matches the swipe template
- [ ] Formatting (bold, italic, caps) matches the template pattern
- [ ] All required sections are present (guarantee, contact info, decline text, etc.)
- [ ] Design notes in [brackets] are descriptive enough for the dev/design team

### Downsell-Specific Checks

- [ ] Price label says "DISCOUNTED PRICE" not "REGULAR PRICE" (it's already been discounted once)
- [ ] Downsell structure matches what user specified (pick-1, bundle, same-but-cheaper)
- [ ] "Our Promise" section features the right presenter (person customer bought from, not necessarily product creator)

## Quick Reference: Common First-Draft Mistakes

| Mistake | Frequency | Fix |
|---------|-----------|-----|
| Invented bonuses | Very Common | ASK. Never assume bonuses exist. |
| Wrong pricing | Very Common | GET EXACT NUMBERS before writing. |
| Used killed terminology | Common | Read positioning doc. Search for banned terms. |
| Placeholder testimonials | Common | Write real ones. Never placeholder. |
| Product name wrong | Common | Verify exact marketing names. |
| Feature count wrong | Common | Cross-reference ALL source docs. |
| CTA = product name | Common | CTA = benefit ("Distance AND Accuracy"). |
| Mechanism before benefit | Common | Flip it. Benefit first, mechanism second. |
| Bullets tease wrong destination | Common | Read destination positioning docs first. |
| Testimonials are generic | Common | Must echo 2+ page concepts by name. |
| Copy too long / throat-clearing | Common | Brevity pass: target 60-70% of draft length. |
| Section breaks golden thread | Occasional | Rewrite with root cause name + binary frame. |
| Wrong presenter/role | Occasional | Verify from VSL scripts. Coach ≠ Engineer. |
| Wrong year | Occasional | Just... use the current year. |

## PHASE 5: Headline & Scanner Story Quality (From SF2 Pre-Sale Page Project)

### Scanner Story Validation

Before approving a page, map ALL headlines in reading order and verify they tell a complete narrative arc WITHOUT body copy. If a scanner can't follow the argument from headlines alone, the page fails.

**Process:**
1. List every H1, H2, H3 on the page in order
2. Read them aloud as a sequence — does the story make sense?
3. Gaps in logic = missing headlines. Add them.
4. Redundant headlines = trim or differentiate.

### Headline Rules

| Rule | Bad Example | Good Example |
|------|-------------|--------------|
| No blind/vague transitions | "Until Now." / "But Here's The Thing" / "What Nobody Tells You" | "I Stopped Fixing *Swings* & Started Fixing *SLICES*" |
| Must reference concrete elements | "Something Changed Everything" | "What 182,000 Slices Taught Me About Yours" |
| Numbers + curiosity gaps | "Why I Was Wrong About Your Slice" | "I Asked 183,000 Amateur Golfers Why They Slice. 93% Guessed Wrong." |
| Pivot headlines must be ACTIVE | "But what if there was another way?" | "I Stopped Fixing Swings & Started Fixing SLICES" |

**The pivot headline (problem → solution) is the hardest and most important headline on any page.** It must be concrete and active, never passive or blind. Generate 10 options with constraints, then pick.

### Headline Iteration Process

1. Draft initial headlines
2. Map the full scanner story to check flow
3. Identify weak/vague/blind headlines
4. Generate 10 replacement options with explicit constraints (must include a number, named concept, or active verb)
5. User selects from options
6. Re-validate scanner story with new headlines

## PHASE 6: Copy Polish Rules (From SF2 Pre-Sale Page Project)

### Body Copy

- [ ] **Kill filler words** — "Now, I know" → "I know." "They often just aren't" → "They often aren't." "even one yard" → "one yard."
- [ ] **Credibility numbers must be believable** — "helping over one hundred thousand" sounds like hype. "helping thousands" sounds real. Simpler = more credible.
- [ ] **Split bold statements for visual impact** — One bold paragraph becomes two shorter bold lines with a line break. Scanners catch both.
- [ ] **Data visualization > assertions** — "Your slice costs you distance" is weak. A split-screen showing 243 yards (draw) vs. 216 yards (slice) at the same swing speed is unignorable.
- [ ] **Cut redundant pain paragraphs** — If a data visual or comparison already makes the point, the paragraph restating it is dead weight. Cut it.

### CTA Architecture

- [ ] **Double CTA with social proof sandwich** — CTA → Testimonials → CTA. Two conversion opportunities.
- [ ] **Micro-copy under buttons** — Small parenthetical line that reinforces emotional benefit: "(Because Good Golf Starts With Good Tee Shots)"
- [ ] **CTA button must include the specific promise** — "YES, I WANT TO FIX MY SLICE FOREVER AND HIT LONGER DRIVES!" not "YES, I WANT STRAIGHTER DRIVES!"

### Image & Media Directives

- [ ] **Every image placeholder = creative brief** — Include emotional direction and specific visual. Not "[IMAGE]" — instead "[IMAGE: golfer DTL on tee pointing right, make the person FEEL THE PAIN]"
- [ ] **GIF directives** — Reference specific existing content where possible: "[GIF: Use the 'Turn It' intro from top performing ads]"
- [ ] **Video sub-headlines must create curiosity gaps** — Include numbers and imply hidden knowledge.

### Testimonial Architecture

- [ ] **3 testimonials > 4 generic ones** — Quality over quantity. Cut the weakest.
- [ ] **Bold the key result line** within each testimonial — Reader scanning testimonials should see the outcome instantly.
- [ ] **Remove visible guidance blocks** — [TESTIMONIAL GUIDANCE: ...] should never appear on the final page.
- [ ] **Make language conversational** — "A lotta times I'd go for it and it would bite me in the you know what" beats "I often found that swinging aggressively resulted in poor outcomes."

### Discovery/Bullet Sections

- [ ] **4 focused bullets > 6 unfocused bullets** — Trim to the strongest. Every bullet must earn its spot.
- [ ] **Bold the benefit phrase inside each bullet** — "✔️ **A bulletproof way to fix your slice ON THE COURSE** — without a single lesson..."
- [ ] **Final bullet can be a tease** — "✔️ **A Whole Lot More!**" is acceptable as a closer if preceding bullets are strong.

## PHASE 7: Pre-Sale Page Architecture (From SPD Pre-Sale Page Retrospective)

### Layered Problem Architecture

Pre-sale pages need a BUILT problem section, not just a stated problem. The SPD page demonstrates a 5-layer architecture:

| Layer | Purpose | SPD Example |
|-------|---------|-------------|
| 1. Direct statement | Name what's happening | "Your highest swing speed is happening AFTER you hit the ball" |
| 2. Named concept | Give it a handle | "You Have A Speed Gap" |
| 3. Escalating consequences | Make it visceral | "1 inch = 10 yards lost. 3 inches = 30 yards. 6 inches = driver hits like a 5-iron" |
| 4. Reframe the cause | Shift from symptom to root | "The Speed Gap Trap — it's not age, it's the gap widening" |
| 5. Failed solutions | Eliminate alternatives | "Swing Tips Won't Save You" + ❌ list |

**Checklist:**
- [ ] **Layer 1:** State the problem directly and concretely — no metaphors yet, just the reality
- [ ] **Layer 2:** Name the concept so the reader has a label for their problem
- [ ] **Layer 3:** Use escalating consequences (small → medium → devastating) with specific numbers at each step
- [ ] **Layer 4:** Reframe so the reader sees it's NOT what they assumed (not age, not ability — it's a specific mechanical issue)
- [ ] **Layer 5:** List 3-6 commonly-tried alternatives with ❌ markers, each one explicitly connected back to why it doesn't address the root cause. End with a bold statement framing all alternatives as insufficient.

### Escalating Consequences Pattern

When building Layer 3, use this structure:

```
❌ [Small gap] → [modest consequence]
❌ [Medium gap] → [significant consequence]
❌ [Large gap] → [devastating comparison that makes reader feel the pain]
```

**The final step should use a comparison, not just a number.** "Driver goes the same distance as a 5-iron" hits harder than "you lose 50 yards." Comparisons make abstract numbers concrete.

- [ ] **At least 3 escalation steps** — each worse than the last
- [ ] **Final step uses a comparison** — not just a bigger number but a gut-punch analogy
- [ ] **Close with an emotional kicker** — "The only thing you're losing faster than golf balls is confidence"

### "Release What You Have" Solution Frame

Instead of "you need something new," frame the solution as unlocking what the reader already has:

> "You don't need MORE speed. You need to release the speed you ALREADY HAVE at the right moment."

**Pattern:** "You don't need MORE [thing they think they need]. You need to [unlock/release/sync] the [thing they already have] [at the right moment/in the right way]."

- [ ] **Solution removes effort barrier** — "you already have what you need"
- [ ] **Solution reframes the problem** — it's not that you're lacking, it's that something is blocking you
- [ ] **Solution flatters the reader** — you're not deficient, you're just misdirected

### Two-Part Numbered Solution Structure

When presenting the solution payoff, use numbered parts for scannability:

```
**#1:** Your fastest swing speed happens AT IMPACT... so all your power transfers into the back of the ball.

**#2:** Because you're releasing your speed at the ball instead of after it, your face is more square naturally... Eliminating slices and hooks.
```

- [ ] **Use "#1:" and "#2:" (or "Because first:" / "And second:")** — numbered logic is scannable
- [ ] **Each part addresses one half of the binary frame** — #1 = distance, #2 = accuracy (or whatever the binary is)
- [ ] **End with a bold summary sentence** that echoes the binary frame: "That's how you gain distance AND accuracy at the same time."

### Expert Introduction vs. Credibility Section Split

On pre-sale pages, introduce the expert TWICE — differently each time:

| Section | Purpose | What Goes Here |
|---------|---------|----------------|
| Expert Introduction (early) | Emotional connection + mission | "This is Andrew Rice. He's made it his life's mission to end the Senior Speed Sacrifice" |
| Meet Your Coach (later) | Concrete credentials | Volume → Recognition → Authority → Relationships → Peer |

- [ ] **Expert intro is MISSION-first** — the expert's crusade, what they refuse to accept, why they care
- [ ] **"Meet Your Coach" section is PROOF-first** — checkmark bullets with specific numbers and titles
- [ ] **Use "MEET YOUR COACH" not "About the Expert"** — warmer, more personal header
- [ ] **Expert as crusader framing** — Frame the expert's mission as fighting AGAINST something ("refuses to accept that," "made it his life's mission to end X")

### "What It's NOT" Pattern — Operational Checklist

The ❌ elimination list is a structural element, not just a writing technique. Use it in:

- [ ] **Problem section** (Layer 5) — eliminates failed alternatives
- [ ] **Video/content descriptions** — "This is NOT about: ⦿ dropping your right foot back ⦿ flaring your trail foot..."
- [ ] **Counter-sell bullets** — "No gym, no speed sticks, no flexibility program..."

**Structure:**
1. List 3-6 specific, commonly-tried alternatives
2. Use ❌ or ⦿ negative markers
3. Each one states WHY it fails (connects back to root cause)
4. End with bold statement framing all as insufficient
5. Transition immediately to the solution

### Blind Product Pre-Sale Language Rules

When the page builds desire but the product is revealed on the NEXT page (the VSL):

- [ ] **Product name NEVER appears** — not once, anywhere, including image alt text and design notes
- [ ] **Use approved substitute language:**

| Instead Of | Use |
|------------|-----|
| [Product Name] | "Andrew's free presentation" / "[Expert]'s method" |
| The product | "the method" / "the exact process" / "the approach" |
| Product features | "what [Expert] shows you next" |
| Buy now | "Watch now" / "See what happens when..." |
| Product testimonials | Testimonials about the EXPERT and their APPROACH (not a named product) |

- [ ] **Testimonials stay blind** — they reference the expert's teaching and approach, never a product name
- [ ] **Bullets tease the destination VSL content** — but never reveal what the product IS
- [ ] **"Masterclass" or "Presentation" as the thing they're clicking to** — not "product page" or "order form"

### Video/Content Series Narrative Arc

When a pre-sale page includes a series of free videos or content pieces, select them for a specific progression:

| Position | Purpose | SPD Example |
|----------|---------|-------------|
| Video 1 | Establish authority + frame the dual promise | "The Senior Speed Secret" — frames longer AND straighter |
| Video 2 | Get specific about WHERE the problem lives | "Effortless Efficiency" — setup, ball position, attack angle |
| Video 3 | Most engaging/shareable drill | "The Power Pause" — counter-intuitive, immediately usable |
| Video 4 | Bridge to the product (without naming it) | "Speed Funnels" — creates "OK, what's the complete system?" |

- [ ] **Video 1 = authority + frame** — establishes the expert and the core promise
- [ ] **Middle videos = specificity + engagement** — deepen understanding, create "aha" moments
- [ ] **Final video = bridge** — creates the desire for "what's the complete system?" without naming it
- [ ] **Video sub-headlines create curiosity gaps** — use specific numbers and imply hidden knowledge

### CTA Sub-Copy Differentiation

When a page has two CTAs (primary and secondary), differentiate the sub-copy:

- [ ] **Primary CTA sub-copy = functional/aspirational** — "(Because Great Drives Make Golf More Fun!)"
- [ ] **Secondary CTA sub-copy = emotional/identity** — "(So I Love My Game Every Time I Tee It Up!)"
- [ ] **Both CTA BUTTONS stay identical** — same text on the button itself, only sub-copy changes

### Section Transition Three-Question Test

Every transition between sections must answer THREE questions:

1. **What exists?** (an in-depth masterclass / a complete system)
2. **How is it different from what they just saw?** (goes way deeper / covers what the videos couldn't)
3. **What specific result will they get?** (dead-straight distance to the course next time you play)

- [ ] **If a transition only answers 1 of 3, rewrite** — generic transitions like "there's more" fail this test
- [ ] **The result (question 3) must be specific and time-anchored** — "starting the next time you play" beats "eventually"
