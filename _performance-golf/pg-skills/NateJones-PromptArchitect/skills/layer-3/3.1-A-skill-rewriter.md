# 3.1-A: Skill Rewriter
*Apply all validated prescriptions to produce optimized skill file*

---

## PURPOSE

Execute the validated prescriptions from Layer 2 against the target skill file to produce an optimized version. This is the assembly step — every prescription becomes a concrete change in the output file.

**Scope:** Apply all prescriptions in priority order. Produce complete rewritten skill file. Preserve all correct existing content.
**Non-Goals:** Do NOT add improvements beyond what's prescribed. Do NOT gold-plate. Execute the prescriptions, nothing more.

---

## INPUTS

- Original target skill file content
- Validated prescriptions package from 2.5-A (in priority order)
- Specifically:
  - Constraint rewrites (2.1-A)
  - Generated exemplars (2.1-B)
  - Guardrail injections (2.2-A) with positions
  - Output spec tightening (2.2-B)
  - Wake word extractions (2.3-A)
  - Context layer separation (2.3-B)
  - Chain-of-Refinement protocol (2.4-A)

---

## INSTRUCTIONS

### Step 1: Apply Structural Reorganization First

If Context Layer Separation (2.3-B) prescribed reorganization:
- Restructure into Layer 1 / Layer 2 sections first
- This sets the skeleton for all other changes
- All subsequent insertions reference this new structure

If no reorganization needed, maintain current structure.

### Step 2: Apply Constraint Rewrites

For each conversion in 2.1-A output:
- Find the original instruction text in the file
- Replace with the rewritten constraint
- Verify surrounding context still flows naturally
- Mark as applied

### Step 3: Insert Guardrail Patterns

For each guardrail in 2.2-A output:
- Insert at the specified position
- If position is "Layer 1 section" → place in deterministic section
- If position is "Layer 2 section" → place in adaptive section
- If positional reinforcement prescribed → move/echo critical rules to first and last positions
- Verify no duplication with existing content

### Step 4: Replace/Enhance Output Specification

From 2.2-B output:
- If output spec was ABSENT → Insert complete new section
- If MINIMAL/PARTIAL → Replace with tightened version
- If ALMOST → Add only the missing elements (validation checks, quality criteria)
- Preserve any existing correct elements

### Step 5: Insert Exemplars

From 2.1-B output:
- Insert exemplar block in Block 3 (Reference) position
- If no Reference section exists, create one
- Place positive exemplars first, then anti-exemplars
- Ensure exemplars are clearly delimited from instructions

### Step 6: Add Chain-of-Refinement

From 2.4-A output (if applicable):
- Insert protocol after INSTRUCTIONS, before OUTPUT SPECIFICATION
- Include critique dimensions, think hierarchy mapping, and triggers
- Add validation reference to output spec ("must pass Phase 2 critique")

### Step 7: Extract Wake Words (if applicable)

From 2.3-A output:
- If wake words extracted, add a WAKE WORDS section
- Keep definitions concise (per-word: trigger, behavior, duration)
- Reference wake words in relevant instruction sections ("When [wake word] active, also apply...")

### Step 8: Final Assembly Check

After all prescriptions applied, verify the assembled file:
- Reads coherently from top to bottom
- No orphaned sections or dangling references
- No duplicate content (unless intentional reinforcement)
- No contradictions between sections
- Formatting is consistent (heading levels, list styles, code blocks)
- Total length is reasonable for the skill type

### Step 9: Add Change Log

At the bottom of the rewritten file, add:
```markdown
---
## CHANGE LOG (Audit: [date])
- [Prescription 1]: [brief description of change]
- [Prescription 2]: [brief description]
[continue for all applied prescriptions]
```

---

## OUTPUT SPECIFICATION

Produce the complete rewritten skill file:

```markdown
# [Skill Name] (OPTIMIZED)
[Complete rewritten content with all prescriptions applied]

---
## CHANGE LOG (Audit: [date])
[List of all changes applied]
```

**File location:** Save to `projects/[target-name]/rewritten/[original-filename]`

**Additional metadata output:**
```yaml
rewrite_metadata:
  original_file: [path]
  rewritten_file: [output path]
  prescriptions_applied: [count]
  prescriptions_skipped: [count, with reasons]
  lines_before: [count]
  lines_after: [count]
  net_change: [+/- lines]
  sections_modified: [list]
  sections_added: [list]
  sections_removed: [list, should be empty unless consolidating]
```

---

## CONSTRAINTS

- NEVER add content not specified in a prescription (no gold-plating, no "while I'm here" improvements)
- NEVER remove existing correct content (only replace what prescriptions target)
- NEVER change the skill's fundamental purpose or scope
- MUST apply prescriptions in the priority order specified by 2.5-A
- MUST preserve all non-operational content (document info, version, metadata)
- MUST maintain the skill's existing naming convention and file format
- If two prescriptions target the same text: Apply the higher-priority one, note the conflict in change log
- If a prescription can't be applied cleanly (context changed by earlier prescription): Adapt the prescription intent to the new context, note the adaptation
- The rewritten file must be COMPLETE and READY TO USE — no placeholders, no TODOs

---

## FALLBACK

- If prescription insertion point no longer exists (restructuring moved it): Find the logical new position, apply there, note in change log
- If prescription would make the file exceed reasonable length (>500 lines for a leaf skill): Flag for review, apply anyway but note "consider splitting this skill"
- If prescriptions are contradictory despite Layer 2 validation: Defer to higher-priority prescription, skip lower-priority one, note in change log
