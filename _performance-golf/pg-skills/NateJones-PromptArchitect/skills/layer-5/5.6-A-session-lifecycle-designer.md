# 5.6-A — Session Lifecycle Designer

> **Layer:** 5 (Multi-Agent Architecture)
> **Phase:** 5 — ARCHITECT Mode
> **Type:** Prescription
> **PRD Reference:** §5.7 Rule 4 (Plan for Endings)

---

## PURPOSE

Design episodic operation with planned resets for the multi-agent system. Continuous operation leads to context drift, signal dilution, and quality degradation. This skill defines session boundaries, checkpoint protocols, and restart handoffs.

**This skill answers:** "How long does each agent session last, what gets checkpointed, and how does the next session pick up?"

---

## INSTRUCTIONS

### Step 1: Define Session Timebox

For each agent role, determine the bounded duration:

```yaml
session_timebox:
  agent_role: [planner | worker_type | judge]
  max_duration: [time limit]
  rationale: [why this duration — based on task complexity, context drift rate, etc.]
  hard_or_soft: [HARD — terminate at limit | SOFT — complete current sub-task then terminate]
  early_termination_triggers:
    - "Task completed before timebox"
    - "Error encountered that cannot be resolved within scope"
    - "Context capacity approaching limit (70-80%)"
```

**Design Principles:**
- Workers: Short sessions (single task → terminate)
- Planners: Medium sessions (decompose + assign batch → checkpoint → may continue or restart)
- Judges: Short sessions (evaluate batch of outputs → deliver verdicts → terminate)

### Step 2: Define Checkpoint Protocol

What gets saved, where, and in what format at the end of each session:

```yaml
checkpoint_protocol:
  agent_role: [role]
  checkpoint_location: [file path, database, external store]
  checkpoint_format: [YAML, JSON, markdown — must be machine-readable]
  checkpoint_contents:
    always_saved:
      - "Completion status (done, partial, failed)"
      - "Output produced (or reference to output location)"
      - "Errors encountered (if any)"
      - "Decisions made and rationale"
    conditionally_saved:
      - condition: [when]
        data: [what additional data]
  checkpoint_validation:
    - "Checkpoint is parseable by next session"
    - "Checkpoint contains enough info to resume without re-reading context"
    - "Checkpoint does not contain raw context (summarized only)"
```

### Step 3: Define Restart Handoff

How a fresh session picks up from where the previous left off:

```yaml
restart_handoff:
  agent_role: [role]
  restart_sequence:
    1: "Load checkpoint from [location]"
    2: "Parse completion status"
    3: "If complete: proceed to next phase"
    4: "If partial: resume from last completed sub-step"
    5: "If failed: assess failure type and either retry or escalate"
  context_loading:
    from_checkpoint: [what gets loaded from the checkpoint]
    from_external_state: [what gets loaded from external storage]
    NOT_loaded: [what is intentionally excluded — previous raw context, stale data]
  fresh_start_guarantee:
    - "New session has zero carryover context from previous session"
    - "All state comes from external storage, not internal memory"
    - "Previous session's reasoning is NOT available (only its outputs)"
```

### Step 4: Define External State Storage

Where work products live between sessions:

```yaml
external_state:
  storage_location: [file system, database, object store]
  data_stored:
    - type: [work product type]
      format: [file format]
      location: [path pattern]
      lifecycle: [how long it persists, when it's cleaned up]
  access_pattern:
    - "Workers write to assigned output locations"
    - "Planner reads worker outputs via checkpoint"
    - "Judge reads worker outputs for evaluation"
    - "No agent reads another agent's internal state"
  state_isolation:
    - "Each worker has its own output directory"
    - "No shared write access to any location"
    - "Merge happens in orchestration layer, not in agents"
```

### Step 5: Validate Against Rule 4

| Check | Pass Condition |
|-------|---------------|
| No continuous operation | All agents have defined session boundaries |
| External state only | All persistent state lives outside agent context |
| Sessions are ephemeral | Any session can be terminated and restarted without data loss |
| Checkpoints are sufficient | A fresh session can resume from checkpoint alone |
| No context pollution | Long-running sessions split at natural boundaries |

---

## CONSTRAINTS

- NEVER design sessions that run indefinitely
- NEVER store state inside agent context between sessions
- ALWAYS define explicit termination triggers
- Checkpoints MUST be self-contained (readable without prior context)
- Sessions are cattle, not pets — any session can die and be replaced
- External state MUST be the single source of truth (not agent memory)
