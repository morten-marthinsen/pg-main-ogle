# 5.4-A — Worker Boundary Definer

> **Layer:** 5 (Multi-Agent Architecture)
> **Phase:** 5 — ARCHITECT Mode
> **Type:** Prescription
> **PRD Reference:** §5.7 Rule 2 (Workers Stay Ignorant)

---

## PURPOSE

Define precise isolation boundaries for each worker type: maximum task size, allowed actions, forbidden actions, and context limits. Workers perform better with less — this skill enforces minimum viable context.

**This skill answers:** "Exactly what can each worker do, what can't it do, and what does it know?"

---

## INSTRUCTIONS

### Step 1: Define Maximum Task Size

For each worker type, define the size boundary:

```yaml
task_boundary:
  worker_type: [name]
  max_duration: [time limit — e.g., "30 minutes", "1 hour"]
  max_scope: [scope limit — e.g., "one function", "one file", "one section"]
  max_output_size: [output limit — e.g., "500 lines", "one document section"]
  decomposition_rule: "If task exceeds these limits, it must be decomposed by the Planner into sub-tasks"
```

### Step 2: Define Allowed Actions (Whitelist)

For each worker type, enumerate EXACTLY what it can do. This is a whitelist — anything not on this list is forbidden.

```yaml
allowed_actions:
  worker_type: [name]
  can_do:
    - [action 1 — specific and concrete]
    - [action 2]
    - [action 3]
    - [action 4 — maximum 5-7 allowed actions]
  examples:
    - "Read specified input files"
    - "Write output to assigned output location"
    - "Use tool X to perform Y"
    - "Apply transformation Z to input data"
```

### Step 3: Define Forbidden Actions (Blacklist)

For each worker type, enumerate what it MUST NEVER do. This catches scope creep that the whitelist might miss.

```yaml
forbidden_actions:
  worker_type: [name]
  must_never:
    - "Modify files outside assigned scope"
    - "Access shared state or databases"
    - "Communicate with other workers"
    - "Create sub-tasks or spawn sub-agents"
    - "Reinterpret or expand the assigned task"
    - "Access system-level configuration"
    - [additional domain-specific prohibitions]
```

### Step 4: Define Context Boundary

Apply the minimum viable context principle (Rule 2):

```yaml
context_boundary:
  worker_type: [name]
  receives:
    - "Task-specific assignment (from Planner)"
    - "Input data required for this specific task"
    - "Output format specification"
    - "Acceptance criteria for this task"
  does_NOT_receive:
    - "System-level goals or objectives"
    - "Other workers' assignments or outputs"
    - "Planner's decomposition strategy"
    - "Judge's evaluation criteria"
    - "Historical context from previous runs"
    - "Performance metrics or system status"
```

### Step 5: Validate Boundaries

Check each worker's boundaries against these tests:

| Test | Pass Condition |
|------|---------------|
| **Ignorance test** | Worker cannot infer the existence of other workers from its context |
| **Scope test** | Worker cannot perform actions outside its whitelist even if prompted to |
| **Context test** | Worker receives no information beyond what's needed for its specific task |
| **Independence test** | Worker can complete its task without any external coordination |
| **Termination test** | Worker knows exactly when its task is done and terminates |

---

## CONSTRAINTS

- NEVER define boundaries that allow workers to see the big picture
- NEVER allow workers to modify their own scope or boundaries
- ALWAYS enforce information hiding structurally (through what's provided), not through instruction ("don't look at X")
- The whitelist MUST be exhaustive — if it's not listed, it's not allowed
- ALWAYS validate with the 5 boundary tests

## OUTPUT

```yaml
worker_boundaries:
  system: [system name]

  workers:
    - type: [worker type name]
      task_boundary:
        max_duration: [time]
        max_scope: [scope]
        max_output_size: [size]
      allowed_actions: [whitelist]
      forbidden_actions: [blacklist]
      context_receives: [list]
      context_excluded: [list]
      boundary_tests:
        ignorance: [PASS | FAIL]
        scope: [PASS | FAIL]
        context: [PASS | FAIL]

        termination: [PASS | FAIL]

  rule_2_compliance:

    structural_information_hiding: [YES | NO]

```
