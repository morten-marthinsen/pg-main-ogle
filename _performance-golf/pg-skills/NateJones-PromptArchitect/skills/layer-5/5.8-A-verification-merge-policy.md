# 5.8-A — Verification & Merge Policy

> **Layer:** 5 (Multi-Agent Architecture)
> **Phase:** 5 — ARCHITECT Mode
> **Type:** Prescription
> **PRD Reference:** §5.8 (Where Complexity Belongs), §5.9 (Architecture Summary)

---

## PURPOSE

Define how Worker outputs get verified, merged, and assembled into the final system output. This is where orchestration complexity lives — the merge layer is the intelligence center that transforms isolated Worker outputs into a coherent whole.

**This skill answers:** "How do we verify Worker outputs, resolve conflicts between them, and merge everything into the final product?"

---

## INSTRUCTIONS

### Step 1: Define Verification Gates

Each Worker output passes through verification before entering the merge queue:

```yaml
verification_gates:
  - gate: [name]
    scope: [which worker types this applies to]
    check: [specific verification — format, completeness, size, etc.]
    automated: [YES | NO — can this be checked automatically?]
    failure_action: [reject_to_planner | flag_for_review | auto_fix]

  standard_gates: # Apply to ALL worker outputs
    - "Output format matches specification"
    - "Output size within defined limits"
    - "Output references only allowed scope"
    - "No forbidden patterns detected"
    - "Checkpoint data complete and parseable"
```

### Step 2: Define Merge Authority

Who/what has authority to merge Worker outputs into the final product:

```yaml
merge_authority:
  type: [automated | human | hybrid]

  automated_merge:
    conditions: "ALL of the following must be true:"
    requirements:
      - "Worker output passed all verification gates"
      - "Judge verdict is ACCEPT"
      - "No conflicts with other accepted outputs"
      - "Output fits cleanly into assigned slot in final product"
    mechanism: [how the merge actually happens — file operations, API calls, etc.]

  human_merge:
    triggers:
      - "Judge verdict is ESCALATE"
      - "Conflict detected between Worker outputs"
      - "Final assembly of all Worker outputs into complete product"
      - "[Domain-specific triggers]"
    interface: [how the human reviews and approves — tool, format, etc.]

  hybrid_merge:
    auto_portion: [what can be merged automatically]
    human_portion: [what requires human review]
    handoff: [how auto → human handoff works]
```

### Step 3: Define Conflict Resolution

When Worker outputs contradict each other or don't fit together:

```yaml
conflict_resolution:
  detection:
    - type: [conflict type]
      signal: [how it's detected]
      example: [concrete example]

  conflict_types:
    overlap:
      definition: "Two Workers produced output covering the same scope"
      resolution: "Judge evaluates both, selects higher quality, or merges best elements"
      authority: [judge | planner | human]

    contradiction:
      definition: "Two Workers produced outputs that contradict each other"
      resolution: "Escalate to Planner for re-decomposition or human for decision"
      authority: [planner | human]

    gap:
      definition: "No Worker output covers a required scope area"
      resolution: "Planner assigns new task to fill the gap"
      authority: [planner]

    incompatibility:
      definition: "Individual outputs are correct but don't fit together"
      resolution: "Merge layer adapts outputs to fit, or Planner re-assigns with interface specs"
      authority: [merge_layer | planner]
```

### Step 4: Define Final Assembly

How individual outputs become the complete product:

```yaml
final_assembly:
  assembly_order:
    1: [first component to assemble]
    2: [second component]
    3: [and so on — explicit ordering]

  assembly_mechanism:
    type: [concatenation | template_filling | synthesis | compilation]
    description: [how pieces come together]
    tool: [what tool/process performs the assembly]

  post_assembly_validation:
    - "Complete product reviewed against original requirements"
    - "No gaps in coverage"
    - "Coherence across Worker boundaries verified"
    - "Final quality check by Judge or human"

  output:
    format: [final product format]
    location: [where it's delivered]
    metadata: [provenance — which Workers produced which parts]
```

### Step 5: Design the Merge Infrastructure

Following the principle that complexity belongs in orchestration (not agents):

```yaml
merge_infrastructure:
  components:
    - name: [component name]
      purpose: [what it does]
      type: [queue | processor | validator | assembler | monitor]

  data_flow:
    1: "Worker output → Verification Gates"
    2: "Verified output → Judge Evaluation"
    3: "Accepted output → Merge Queue"
    4: "Merge Queue → Conflict Detection"
    5: "No conflicts → Assembly"
    6: "Conflicts → Resolution Protocol"
    7: "Assembly → Post-Assembly Validation"
    8: "Validation passed → Final Delivery"

  monitoring:
    tracks:
      - "Worker completion rates"
      - "Verification gate pass/fail rates"
      - "Judge accept/reject/revise ratios"
      - "Conflict frequency and types"
      - "Time from Worker completion to final assembly"
    alerts:
      - condition: [what triggers an alert]
        action: [what happens — human notification, auto-retry, etc.]
```

### Step 6: Validate Against Scaling Principles

| Check | Pass Condition |
|-------|---------------|
| Complexity in orchestration | Merge logic lives in infrastructure, not in agents |
| No agent-to-agent merge | Workers never merge each other's outputs |
| Conflict resolution defined | Every conflict type has an explicit resolution path |
| Provenance tracked | Final product traces every section to its source Worker |
| Monitoring present | System health observable without querying agents |

---

## CONSTRAINTS

- NEVER allow Workers to merge outputs (merge is an orchestration function)
- NEVER allow merge without prior verification and Judge evaluation
- ALWAYS define explicit conflict resolution for every conflict type
- ALWAYS track provenance (which Worker produced which output)
- The merge layer MUST be the most complex part of the system (by design)
- Monitoring MUST be external to agents (agents don't self-report system health)
