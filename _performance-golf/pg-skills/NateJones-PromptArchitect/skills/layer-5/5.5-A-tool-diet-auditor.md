# 5.5-A — Tool Diet Auditor

> **Layer:** 5 (Multi-Agent Architecture)
> **Phase:** 5 — ARCHITECT Mode
> **Type:** Diagnostic + Prescription
> **PRD Reference:** §5.7 Rule 3 (No Shared State)

---

## PURPOSE

Audit and constrain tool access for each worker type. Tool proliferation creates shared state, contention, and decision paralysis. This skill enforces the tool diet: 3-5 core tools always available, others on demand through progressive disclosure.

**This skill answers:** "Which tools does each worker need, and which tools are creating problems?"

---

## INSTRUCTIONS

### Step 1: Inventory Current Tool Access

For each agent/worker in the system, list all available tools:

```yaml
tool_inventory:
  agent: [name/role]
  tools_available: [complete list]
  total_count: [number]
  tools_actually_used: [list of tools observed in use]
  tools_never_used: [list of tools available but never invoked]
  tools_shared_with_other_agents: [list of tools accessible by multiple agents]
```

### Step 2: Classify Tools

For each tool, classify by necessity:

| Classification | Definition | Action |
|---------------|------------|--------|
| **Core** | Worker cannot complete any task without it | Always available (3-5 max) |
| **Specialized** | Needed for specific task types only | On-demand via progressive disclosure |
| **Redundant** | Overlaps with another tool's capability | Remove — pick one |
| **Contention-creating** | Shared with other agents, creates coordination overhead | Isolate or replace with agent-specific version |
| **Unused** | Available but never invoked | Remove immediately |

### Step 3: Design Tool Sets Per Worker

For each worker type, define:

```yaml
tool_diet:
  worker_type: [name]

  always_on: # 3-5 core tools
    - tool: [name]
      purpose: [why this worker needs it always]
    - tool: [name]
      purpose: [why]
    # Maximum 5

  on_demand: # Available through progressive disclosure
    - tool: [name]
      trigger: [when this tool becomes available]
      purpose: [for what specific task type]

  removed: # Previously available, now eliminated
    - tool: [name]
      reason: [redundant | contention | unused | out of scope]
      replacement: [what covers its function, if anything]
```

### Step 4: Check for Tool-Based Shared State

Tools become shared state when multiple agents access the same external resource through them:

| Check | Violation Signal |
|-------|-----------------|
| Same database tool across workers | Workers can read/write each other's data |
| Same file system tool across workers | Workers can modify each other's files |
| Same API tool across workers | Rate limits and state shared |
| Same queue tool across workers | Contention on queue operations |

For each violation: isolate (give each worker its own tool instance/scope) or redesign (move the shared resource to the orchestration layer).

---

## CONSTRAINTS

- NEVER allow more than 5 core (always-on) tools per worker
- NEVER allow workers to share tool instances that create state coupling
- ALWAYS remove unused tools — they add cognitive load and selection confusion
- Tool selection accuracy degrades past ~12 options — enforce hard limits
- Progressive disclosure MUST have explicit trigger conditions (not "when needed")
