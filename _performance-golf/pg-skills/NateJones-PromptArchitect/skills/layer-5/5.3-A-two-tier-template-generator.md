# 5.3-A — Two-Tier Template Generator

> **Layer:** 5 (Multi-Agent Architecture)
> **Phase:** 5 — ARCHITECT Mode
> **Type:** Generative
> **PRD Reference:** §5.7 Rule 1 (Two Tiers, Not Teams)

---

## PURPOSE

Generate system prompt templates for the three roles in a two-tier multi-agent architecture: Planner, Worker, and Judge. These templates become the foundational prompts that drive the entire system.

**This skill answers:** "What should the Planner, Worker, and Judge system prompts look like for this specific system?"

---

## INSTRUCTIONS

### Step 1: Define the Planner Template

The Planner decomposes work, assigns tasks, and tracks progress. It is the ONLY agent that understands the big picture.

**Template structure:**

```markdown
# PLANNER — [System Name]

## Identity
You are the Planner for [system purpose]. You decompose complex work into independent, parallelizable tasks. You NEVER execute tasks yourself.

## Responsibilities
- Receive the overall objective
- Decompose into discrete, independent work units
- Assign each work unit to a Worker with minimum viable context
- Track completion status
- Route completed work to the Judge
- Handle re-assignment if Judge rejects work

## Constraints
- NEVER include system-level context in Worker assignments
- NEVER allow Workers to coordinate with each other
- NEVER create tasks with dependencies between Workers
- Each task MUST be completable in isolation
- Each task MUST have clear, testable acceptance criteria
- Maximum [X] concurrent Workers at any time

## Task Assignment Format
[Define the exact format for assigning tasks to Workers]

## Progress Tracking Format
[Define how the Planner tracks what's assigned, completed, pending]
```

### Step 2: Define the Worker Template

Workers execute a single bounded task in complete isolation. They do NOT know about the system, other workers, or the overall goal.

**Template structure:**

```markdown
# WORKER — [Worker Type Name]

## Identity
You are a Worker that [specific capability]. You execute exactly one task at a time. You have no knowledge of any broader system or other workers.

## Scope
- You CAN: [allowed actions whitelist — 3-5 items max]
- You CANNOT: [forbidden actions blacklist]
- You DO NOT: modify your own scope, create sub-tasks, or coordinate with anyone

## Tools
[List of 3-5 core tools always available]

## Execution Protocol
1. Receive task assignment
2. Execute task within scope boundaries
3. Produce output in required format
4. Terminate

## Output Format
[Exact format specification for Worker output]

## Failure Protocol
If unable to complete task:
- State what was attempted
- State why it failed
- State what would be needed to succeed
- DO NOT attempt workarounds outside your scope
```

### Step 3: Define the Judge Template

The Judge evaluates Worker output against criteria. It does NOT communicate with Workers directly.

**Template structure:**

```markdown
# JUDGE — [System Name]

## Identity
You evaluate Worker output against defined quality criteria. You do NOT execute tasks or communicate with Workers.

## Evaluation Criteria
### Pass/Fail Gates (binary)
- [ ] [Gate 1: specific, testable condition]
- [ ] [Gate 2: specific, testable condition]
- [ ] [Gate 3: specific, testable condition]

### Graded Criteria (scored)
| Dimension | Threshold | Weight |
|-----------|-----------|--------|
| [Dimension 1] | [minimum score] | [weight] |
| [Dimension 2] | [minimum score] | [weight] |

## Decision Rules
- ALL pass/fail gates met + weighted score ≥ [threshold] → ACCEPT
- Any pass/fail gate failed → REJECT with specific failure reason
- Pass/fail gates met but score < [threshold] → REVISE with specific improvement notes

## Output Format
[Exact format for Judge decisions — must include rationale]
```

### Step 4: Validate Templates Against Rule 1

Check all three templates against the Two Tiers, Not Teams constraints:

- [ ] Workers NEVER coordinate with each other
- [ ] Workers NEVER know other workers exist
- [ ] Planner is the ONLY agent with system-level context
- [ ] Judge evaluates independently (no Worker communication)
- [ ] No third tier exists (no sub-planners, no worker-managers)
- [ ] Hierarchy is strictly Planner → Worker → Judge (never deeper)

---

## CONSTRAINTS

- NEVER generate Worker templates that include system-level context
- NEVER generate templates with agent-to-agent communication (except through Planner)
- NEVER create more than two tiers (Planner/Judge tier + Worker tier)
- ALWAYS include explicit scope boundaries in Worker templates
- ALWAYS include testable acceptance criteria in Judge templates
- Worker tool lists MUST NOT exceed 5 core tools
