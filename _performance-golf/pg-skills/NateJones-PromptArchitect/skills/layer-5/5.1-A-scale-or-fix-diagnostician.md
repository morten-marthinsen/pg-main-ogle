# 5.1-A — Scale-or-Fix Diagnostician

> **Layer:** 5 (Multi-Agent Architecture)
> **Phase:** 5 — ARCHITECT Mode
> **Type:** Diagnostic
> **PRD Reference:** §5.6 (The Multi-Agent Scaling Problem), §5.7 (Six Rules)

---

## PURPOSE

Determine whether the system should scale (add more agents) or fix first (improve single-agent quality). This is the critical first decision — scaling a broken system amplifies the breakage.

**This skill answers:** "Should we add more agents, or should we fix what we have?"

---

## INSTRUCTIONS

### Step 1: Capture Current System State

```yaml
system_snapshot:
  agent_count: [current number of agents]
  architecture_type: [flat | hierarchical | pipeline | hybrid | single]
  single_agent_accuracy: [estimated % — how well does one agent perform the core task?]
  known_failures: [list of failure modes observed]
  coordination_mechanism: [how agents currently coordinate]
  scale_target: [desired agent count / concurrency level]
```

### Step 2: Pattern Match Against Failure Modes

Check the system against these diagnostic patterns:

| Pattern | Signal | Verdict |
|---------|--------|---------|
| Single-agent accuracy < 45% | Core task quality is low | **FIX FIRST** — adding agents will make it worse (Google-MIT finding) |
| Coordination overhead > capability gain | Agents spend more time coordinating than working | **REDESIGN** — architecture has fundamental serial dependencies |
| Tasks are naturally parallelizable | Work can be divided into independent units | **SCALE** — proceed with multi-agent architecture |
| Shared state required | Tasks require reading/writing common data | **REDESIGN** — eliminate shared state before scaling |
| Single agent hits context/time limits | One agent can't hold enough context or runs too long | **SCALE** — but with episodic design (Rule 4) |

### Step 3: Identify First Bottleneck

The first bottleneck is the single most impactful constraint. Identify it:

- **Quality bottleneck:** Single agent produces poor output → Fix prompts, add exemplars, improve constraints
- **Throughput bottleneck:** Single agent is too slow → Scale with parallel workers
- **Context bottleneck:** Single agent runs out of context → Scale with episodic sessions
- **Coordination bottleneck:** Agents can't work together efficiently → Redesign architecture

---

## CONSTRAINTS

- NEVER recommend scaling when single-agent accuracy is below 45%
- NEVER recommend "add more agents" as a fix for coordination problems
- ALWAYS identify the specific bottleneck type before recommending action
- ALWAYS reference the Google-MIT finding when single-agent accuracy is the issue
