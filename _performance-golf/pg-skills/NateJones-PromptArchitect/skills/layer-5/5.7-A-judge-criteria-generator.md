# 5.7-A — Judge Criteria Generator

> **Layer:** 5 (Multi-Agent Architecture)
> **Phase:** 5 — ARCHITECT Mode
> **Type:** Generative
> **PRD Reference:** §5.7 Rule 6 (Tests as Architecture)

---

## PURPOSE

Define evaluation criteria for the Judge role in a two-tier multi-agent system. The Judge evaluates Worker outputs without communicating with Workers. Criteria must be testable, unambiguous, and produce consistent verdicts across different Judge sessions.

**This skill answers:** "How does the Judge decide if Worker output is acceptable, and what happens on pass/fail/partial?"

---

## INSTRUCTIONS

### Step 1: Define Pass/Fail Gates (Binary)

Pass/fail gates are non-negotiable requirements. Any failure = automatic rejection.

```yaml
pass_fail_gates:
  - gate: [name]
    test: [specific, testable condition — must be binary YES/NO]
    failure_response: "REJECT — [specific reason template]"
    examples:
      pass: [example of passing output]
      fail: [example of failing output]
```

**Gate Design Rules:**
- Each gate MUST be testable without subjective judgment
- Each gate MUST produce the same result regardless of which Judge session evaluates it
- Minimum 3 gates, maximum 7 (too few = quality leaks, too many = false rejects)
- Gates should catch structural/format/completeness failures (not quality nuance)

### Step 2: Define Graded Criteria (Scored)

Graded criteria capture quality dimensions that exist on a spectrum:

```yaml
graded_criteria:
  - dimension: [name]
    description: [what this measures]
    scale: [0-5 or 0-10]
    threshold: [minimum acceptable score]
    weight: [relative importance — weights must sum to 1.0]
    scoring_rubric:
      0: [description of score 0]
      1: [description of score 1]
      2: [description of score 2]
      3: [description of score 3]
      4: [description of score 4]
      5: [description of score 5]
```

**Graded Criteria Design Rules:**
- Each dimension MUST have an explicit scoring rubric (no "use judgment")
- Rubric descriptions MUST be specific enough that two different Judges would assign the same score ±1
- Maximum 5 graded dimensions (more creates scoring fatigue and inconsistency)
- Weights must reflect actual priority (don't weight everything equally unless they're truly equal)

### Step 3: Define Decision Rules

Combine pass/fail gates and graded criteria into clear verdict logic:

```yaml
decision_rules:
  accept:
    condition: "ALL pass/fail gates passed AND weighted score ≥ [threshold]"
    action: "Mark as accepted, forward to merge queue"

  reject:
    condition: "ANY pass/fail gate failed"
    action: "Mark as rejected, return to Planner with specific failure reason"
    planner_action: "Planner reassigns task to new Worker with failure context"

  revise:
    condition: "ALL pass/fail gates passed BUT weighted score < [threshold]"
    action: "Mark as needs revision, return to Planner with specific improvement notes"
    planner_action: "Planner assigns revision task to new Worker with original output + improvement notes"

  escalate:
    condition: "[specific conditions that require human review]"
    action: "Flag for human review with Judge's analysis"
    examples:
      - "Ambiguous quality — Judge confidence < 70%"
      - "Novel situation not covered by existing criteria"
      - "Third rejection of same task"
```

### Step 4: Define Test Suite Integration (Rule 6)

Where automated tests exist, integrate them into Judge criteria:

```yaml
test_integration:
  automated_tests:
    - test: [test name/suite]
      maps_to: [which gate or criterion it validates]
      run_by: [Judge directly | CI/CD | orchestration layer]
      failure_handling: [automatic reject | flag for review]
  test_as_coordination:
    - "Test suites provide shared ground truth without shared state"
    - "Multiple Workers can consult tests independently"
    - "Tests validate without creating dependencies between agents"
  no_tests_available:
    fallback: "Shorter feedback loops + more human oversight"
    mitigation: "Judge criteria must be extra-specific to compensate"
```

### Step 5: Validate Criteria Consistency

Run these checks on the generated criteria:

| Check | Pass Condition |
|-------|---------------|
| Determinism | Same output → same verdict across Judge sessions |
| Completeness | No common failure mode uncaught by gates or criteria |
| Independence | Judge can evaluate without Worker communication |
| Specificity | Every criterion has a rubric, not subjective language |
| Actionability | Every rejection/revision includes specific, actionable feedback |

---

## CONSTRAINTS

- NEVER allow subjective criteria without explicit rubrics
- NEVER allow Judge to communicate with Workers (evaluation is independent)
- ALWAYS provide specific failure reasons with rejections (not "didn't meet standards")
- Decision rules MUST be deterministic — no "use your judgment" clauses
- ALWAYS define escalation triggers for edge cases requiring human review
- If no automated tests exist, criteria must be MORE specific to compensate
