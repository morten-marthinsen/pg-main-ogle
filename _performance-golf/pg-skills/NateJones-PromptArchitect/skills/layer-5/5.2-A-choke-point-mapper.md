# 5.2-A — Choke Point Mapper

> **Layer:** 5 (Multi-Agent Architecture)
> **Phase:** 5 — ARCHITECT Mode
> **Type:** Diagnostic
> **PRD Reference:** §5.6 (Serial Dependency Problem)

---

## PURPOSE

Inventory all coordination points, shared resources, and wait points in the multi-agent system. Map where serial dependencies exist so they can be eliminated or isolated.

**This skill answers:** "Where are agents blocking each other, and what shared resources create contention?"

---

## INSTRUCTIONS

### Step 1: Inventory Shared Resources

For every resource accessed by more than one agent, document:

```yaml
shared_resource:
  name: [resource identifier]
  type: [file | database | queue | memory | context | tool | API]
  agents_accessing: [list of agent names/roles]
  access_pattern: [read-only | read-write | write-only]
  contention_risk: [HIGH | MEDIUM | LOW]
  # HIGH = multiple agents write simultaneously
  # MEDIUM = multiple agents read, one writes
  # LOW = multiple agents read-only
```

### Step 2: Map Wait Points

Identify every point where one agent must wait for another:

```yaml
wait_point:
  blocking_agent: [who must finish first]
  blocked_agent: [who is waiting]
  dependency_type: [data | approval | resource_lock | sequencing]
  avoidable: [YES | NO]
  elimination_strategy: [how to remove this dependency, if possible]
```

### Step 3: Detect Communication Channels

Identify every agent-to-agent communication path:

```yaml
communication_channel:
  from_agent: [sender]
  to_agent: [receiver]
  mechanism: [shared file | message queue | direct call | shared state | event]
  direction: [one-way | bidirectional]
  creates_dependency: [YES | NO]
```

### Step 4: Rank Choke Points

From all shared resources, wait points, and communication channels, identify the **top 3 choke points** — the coordination bottlenecks that most constrain parallelism.

For each choke point, provide:
- What it is
- Why it creates a serial dependency
- How to eliminate or isolate it
- What architecture change is required

---

## CONSTRAINTS

- NEVER skip the inventory — every shared resource must be documented
- NEVER assume a resource is "fine" because it works at small scale
- ALWAYS classify contention risk even for read-only resources (they can still create dependencies through stale reads)
- ALWAYS provide elimination strategies — diagnosis without prescription is incomplete

---

## OUTPUT

```yaml
choke_point_map:
  system: [system name]
  total_shared_resources: [count]
  total_wait_points: [count]
  total_communication_channels: [count]

  communication_channels: [list of communication_channel objects]

    - rank: 1

      ...
    - rank: 3
      ...

```
