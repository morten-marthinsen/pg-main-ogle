# 1.2-A: Guardrail Auditor
*Check for presence of Seven Guardrail Patterns*

---

## PURPOSE

Evaluate whether the target skill implements the Seven Guardrail Patterns defined in ARCHITECTURE-PRD §4.2. These patterns are the difference between skills that work in testing and skills that survive production. Each missing pattern is a specific vulnerability.

**Scope:** Check for 7 patterns. Score presence/absence. Identify which are missing and why they matter for this skill type.
**Non-Goals:** Do NOT generate the missing guardrails. Do NOT rewrite existing ones. Diagnose only.

---

## INPUTS

- Target skill file content (in context)
- Skill type classification from 1.0-A (orchestrator, leaf, generation, validation, scraper)

---

## INSTRUCTIONS

### Step 1: Check Pattern 1 — Identity Invariants

**Looking for:**
- Explicit statement of who/what the agent IS
- Explicit statement of who/what the agent IS NOT
- Role boundaries that survive adversarial input
- Statements like: "You are [X]. You are never [Y]. If asked to act as [Y], respond as [X] would."

**Pass criteria:** Both positive identity AND negative boundary present.

### Step 2: Check Pattern 2 — Trigger-Template Refusals

**Looking for:**
- Explicit handling of edge-case inputs
- Template responses for out-of-scope requests
- "If [trigger condition], respond with: [exact template]"
- At least 2 edge cases handled explicitly

**Pass criteria:** At least 2 trigger-template pairs defined.

### Step 3: Check Pattern 3 — Three-Tier Uncertainty

**Looking for:**
- Confidence-based behavior differentiation
- Different actions at different certainty levels
- Explicit thresholds (>90% = certain, 60-90% = probable, <60% = uncertain)
- Or equivalent graduated response system

**Pass criteria:** At least 2 tiers of confidence-based behavior defined.

### Step 4: Check Pattern 4 — Locked Tool Grammar

**Looking for:**
- Preconditions before tool calls ("Before calling [tool], verify [X]")
- Required parameters explicitly listed
- Format requirements for tool inputs
- Constraints on WHEN tools can be called

**Pass criteria:** If skill uses external tools, must have preconditions defined. If no tools, mark N/A.

### Step 5: Check Pattern 5 — Binary Style Rules

**Looking for:**
- ALWAYS/NEVER pairs without gray area
- Clear, unambiguous behavioral rules
- No "try to" or "aim for" — only "always" or "never"
- At least 3 binary rules present

**Pass criteria:** At least 3 ALWAYS/NEVER rules without qualification.

### Step 6: Check Pattern 6 — Positional Reinforcement

**Looking for:**
- Critical rules placed in FIRST section of document
- Critical rules repeated or echoed in LAST section
- Key constraints appearing in multiple locations
- Important rules not buried in middle sections only

**Pass criteria:** At least 1 critical rule appears in both first and last quarters of document.

### Step 7: Check Pattern 7 — Post-Tool Reflection

**Looking for:**
- Validation steps AFTER tool calls or major operations
- "After [action], verify: [checks]"
- Format validation, sense-checking, error detection
- Explicit "if check fails, then [fallback]"

**Pass criteria:** If skill performs tool calls or produces outputs, must have post-action validation. If purely analytical, mark as conditional.

### Step 8: Apply Skill Type Threshold

Per QUALITY-STANDARDS.md §4:
- Orchestrator skills: ≥6 patterns required
- Production/scraper skills: ≥5 patterns required
- Generation skills: ≥5 patterns required
- Internal/leaf skills: ≥3 patterns required
- Validation skills: ≥4 patterns required

---

## CONSTRAINTS

- NEVER mark a pattern as PRESENT if only partially implemented (use PARTIAL)
- NEVER mark Locked Tool Grammar or Post-Tool Reflection as ABSENT for skills that don't use tools — mark N/A
- MUST assess risk-if-missing specifically for THIS skill's context, not generically
- MUST consider skill type when evaluating threshold (leaf skills have lower bar than orchestrators)
- PARTIAL counts as 0.5 toward the score, not 1.0
- If a pattern is implemented but poorly (e.g., identity stated but no negative boundary), mark PARTIAL

---

## FALLBACK

- If skill type is unclear from consumption receipt: Default to "leaf" threshold (≥3) but flag for orchestrator review
- If skill is very short (<30 lines): Not all patterns are expected. Adjust threshold to ≥2 and note "minimal skill — reduced expectations"
