# 1.1-C: Specificity Scorer
*Evaluate Goldilocks zone compliance across all sections*

---

## PURPOSE

Measure whether the target skill's language sits in the "Goldilocks zone" — specific enough to constrain meaningfully, flexible enough to leverage model intelligence. Too vague produces generic output. Too rigid produces stilted output. The sweet spot produces focused, high-quality work.

**Scope:** Score each section's specificity. Calculate percentage in Goldilocks zone. Identify problem areas.
**Non-Goals:** Do NOT rewrite vague/rigid sections. Diagnose only.

---

## INPUTS

- Target skill file content (in context)
- Consumption receipt section map from 1.0-A

---

## INSTRUCTIONS

### Step 1: Identify Scorable Sections

Evaluate each H2 or H3 section that contains operational content (instructions, constraints, output specs). Skip pure metadata sections (version info, document info).

### Step 2: Score Each Section

For each section, evaluate on the specificity spectrum:

**Score 0 — TOO VAGUE:**
Indicators:
- Uses undefined qualifiers: "appropriate," "relevant," "as needed," "good quality," "suitable"
- Gives direction without measurable criteria: "write well," "be thorough," "analyze carefully"
- Lacks concrete examples of what success looks like
- Could apply to ANY skill (not specific to this one)
- Reader cannot determine if they've satisfied the requirement

**Score 1 — SLIGHTLY VAGUE:**
Indicators:
- Has general direction but missing one key specificity element
- Uses some measurable criteria but leaves key aspects undefined
- Partially concrete — you know the direction but not the destination

**Score 2 — GOLDILOCKS ZONE:**
Indicators:
- Constrains WHAT to produce (specific deliverable type, format, structure)
- Defines measurable criteria (counts, thresholds, presence/absence checks)
- Leaves HOW to the model (doesn't script exact words or rigid sequences)
- Specific to THIS skill's context (couldn't be copy-pasted to another skill)
- Reader can unambiguously determine if requirement is met
- Allows for intelligent execution within bounds

**Score 1 — SLIGHTLY RIGID:**
Indicators:
- Prescribes exact phrasing where flexible language would work
- Over-specifies sequences that could be adaptive
- Leaves no room for model to improve on the specification
- Micromanages execution steps that are self-evident

**Score 0 — TOO RIGID:**
Indicators:
- Word-for-word scripts that must be followed exactly
- No room for adaptation to context
- Treats the model as a template-filler, not an intelligent agent
- Would produce worse output than a more flexible specification

### Step 3: Identify Vague Language Instances

Catalog every instance of vague language:
- "appropriate" (appropriate for what?)
- "relevant" (relevant to whom?)
- "as needed" (needed by what criteria?)
- "good" / "high quality" (measured how?)
- "thorough" (how thorough? what counts?)
- "carefully" (what does careful look like?)
- "properly" (proper by what standard?)
- "ensure quality" (quality measured by what metric?)

### Step 4: Identify Rigid Language Instances

Catalog every instance of over-rigidity:
- Exact word requirements where meaning matters more than words
- Forced sequences where order doesn't actually matter
- Step-by-step scripts for tasks the model can figure out
- Format specifications that are tighter than necessary for the purpose

### Step 5: Calculate Score

```
goldilocks_percentage = (sections_scoring_2 / total_sections_scored) × 100%
```

```yaml
specificity_audit:

  sections_evaluated: [count]

  slightly_rigid: [count]

  vague_instances:
    - section: "[section name]"
      line: [approximate line number]
      text: "[quoted vague language]"
      issue: "[why this is too vague]"
    # [continue for all instances]

  rigid_instances:
    - section: "[section name]"
      line: [approximate line number]
      text: "[quoted rigid language]"
      issue: "[why this is too rigid]"
    # [continue for all instances]

  worst_sections: # Sections with score 0
    - name: "[section name]"
      score: [0]
      direction: [vague | rigid]
      key_issue: "[primary problem]"
```

---

## CONSTRAINTS

- NEVER score a section as Goldilocks (2) if it contains ANY of the vague indicators without definition
- NEVER score as TOO RIGID (0) if the rigidity serves a genuine purpose (e.g., API call formats MUST be exact)
- MUST distinguish between NECESSARY precision (tool call syntax) and UNNECESSARY rigidity (creative language scripted)
- MUST consider context: a scraper skill SHOULD have rigid tool grammar; a generation skill SHOULD NOT have rigid language specs
- If a section is very short (1-2 lines), evaluate but note as "limited sample"
- Template/format code blocks are NOT rigid — they're output specifications (evaluate under Block 4, not specificity)

---

## FALLBACK

- If target has fewer than 3 scorable sections: Report scores but note "insufficient sections for meaningful percentage — interpret with caution"
- If target is entirely code/templates: Report as N/A for specificity (code should be exact)
