# 1.3-A: Production Readiness Checker
*Evaluate the Six Production Principles*

---

## PURPOSE

Determine whether the target skill is built for production reliability or only works in ideal conditions. The Six Production Principles (ARCHITECTURE-PRD §4.1) separate skills that survive real-world usage from those that break on first edge case.

**Scope:** Check 6 principles. Score presence/implementation quality. Identify production vulnerabilities.
**Non-Goals:** Do NOT add production features. Diagnose gaps only.

---

## INPUTS

- Target skill file content (in context)
- Skill type classification from 1.0-A
- Consumption receipt section map

---

## INSTRUCTIONS

### Step 1: Check Principle 1 — Stateful Intelligence

**Question:** Does this skill track what's been done, what's in progress, and what's next?

**Looking for:**
- Progress tracking mechanism (checklist, status fields, step markers)
- Awareness of completed vs remaining work
- Ability to resume from interruption
- State that persists beyond the current turn

**Scoring:**
- 0: No state awareness — operates as if every invocation is the first
- 1: Implicit state (assumes sequential execution but doesn't track)
- 2: Explicit state tracking with persistence mechanism

**N/A conditions:** Single-shot skills that complete in one turn don't need state.

### Step 2: Check Principle 2 — Bounded Uncertainty

**Question:** Does this skill quantify confidence rather than presenting all outputs as equally certain?

**Looking for:**
- Confidence scores or tiers on outputs
- Different behavior at different certainty levels
- Explicit "I'm not sure" conditions vs "I'm certain" conditions
- Calibrated language (not just hedge words, but structured uncertainty)

**Scoring:**
- 0: All outputs presented with equal confidence regardless of certainty
- 1: Some uncertainty language but not structured
- 2: Explicit confidence tiers with different actions per tier

### Step 3: Check Principle 3 — Intelligent Failure Detection

**Question:** Does this skill distinguish between types of failures and respond appropriately?

**Looking for:**
- Transient vs permanent error distinction
- Recoverable vs fatal failure handling
- Specific error responses (not just "if error, stop")
- Retry logic for transient failures
- Escalation logic for permanent failures

**Scoring:**
- 0: No error handling, or only "if error → stop"
- 1: Generic error handling (catches errors but treats all the same)
- 2: Typed error handling with appropriate response per type

### Step 4: Check Principle 4 — Capability-Based Routing

**Question:** Does this skill match tasks to appropriate tools/models/approaches?

**Looking for:**
- Tool selection logic (when to use tool A vs tool B)
- Model routing (simple tasks → fast model, complex → deep model)
- Approach selection based on task characteristics
- Not using a sledgehammer for thumbtacks

**Scoring:**
- 0: Single approach for all cases (no routing)
- 1: Some routing but implicit or incomplete
- 2: Explicit routing with criteria for each path

**N/A conditions:** Skills that only use one tool/approach by design.

### Step 5: Check Principle 5 — Behavioral Consistency

**Question:** Would this skill produce the same output given the same input?

**Looking for:**
- Deterministic decision logic (not "use your judgment")
- Explicit criteria for choices (not mood-dependent)
- Reproducible processes (another agent could follow these steps)
- No random or arbitrary elements

**Scoring:**
- 0: Relies heavily on "judgment" or "discretion" without criteria
- 1: Mostly deterministic but some undefined decision points
- 2: Fully reproducible — same input always produces same output structure

### Step 6: Check Principle 6 — Continuous Validation

**Question:** Does this skill validate at each step, or only at the end (or not at all)?

**Looking for:**
- Per-step validation checks
- Intermediate quality gates
- Early detection of drift from requirements
- Not just a final "check your work" at the end

**Scoring:**
- 0: No validation anywhere, or only "review at end"
- 1: Final validation only (end-of-process check)
- 2: Validation at multiple points throughout execution

### Step 7: Apply Threshold

Per QUALITY-STANDARDS.md §4:
- Orchestrator skills: ≥4 principles scoring 2, remainder ≥1
- Leaf skills: ≥3 principles scoring ≥1
- Generation skills: ≥4 principles scoring ≥1
- Validation skills: ≥3 principles scoring 2

Total score = sum of all applicable principle scores (0-12 max)

---

## CONSTRAINTS

- NEVER score a principle as 2 (full) if implementation is incomplete or implicit
- NEVER mark a principle N/A unless the skill genuinely has no use for it (single-shot, no tools, etc.)
- MUST assess vulnerability specifically — what breaks if this principle is missing FROM THIS SKILL
- MUST consider skill type: orchestrators need more principles than leaf skills
- If a principle is implemented via the parent system (e.g., state handled by MASTER-AGENT), note as "delegated to orchestrator" — score as 1 (partial), not 2
- Behavioral Consistency score of 0 is a critical finding for any skill type — always flag

---

## FALLBACK

- If skill is a pure template/reference (no execution logic): Report all as N/A, note "non-executable reference document"
- If skill type is ambiguous: Apply leaf-skill thresholds (lowest bar) but flag for type clarification
