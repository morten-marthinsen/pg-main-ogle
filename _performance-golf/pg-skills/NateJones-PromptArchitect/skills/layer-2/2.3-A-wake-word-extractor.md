# 2.3-A: Wake Word Extractor
*Identify reusable behavior patterns that can become named wake words*

---

## PURPOSE

Scan the target skill for repeated behavior patterns, mode switches, and conditional standards that could be extracted as reusable wake words. Wake words (ARCHITECTURE-PRD §2.6) are named prompt fragments that activate specific behaviors without re-explaining them each time.

**Scope:** Identify candidate patterns. Name them. Define their activation behavior. Package for reuse.
**Non-Goals:** Do NOT restructure the skill around wake words. Do NOT force extraction where patterns don't naturally repeat. Extract what's organic.

---

## INPUTS

- Target skill file content
- Consumption receipt from 1.0-A (section map)
- Any other skills in the same system (to check for cross-skill patterns)

---

## INSTRUCTIONS

### Step 1: Identify Repeated Behavior Patterns

Scan for content that:
- Appears in multiple sections with slight variations
- Describes a "mode" the agent should enter (analysis mode, critique mode, etc.)
- Defines a quality standard that could apply elsewhere
- Sets up a conditional behavior ("when doing X, always Y")
- Represents a reusable protocol (validation routine, format switch, etc.)

### Step 2: Identify Mode Switches

Look for language that toggles between behaviors:
- "When producing [output type A], use [approach A]"
- "For [context X], apply [different standard]"
- "In [situation], switch to [behavior]"

These are natural wake word candidates — named modes that activate on command.

### Step 3: Identify Cross-Skill Patterns

If other skills in the same system are available:
- Look for identical or near-identical patterns across skills
- These are HIGH-VALUE wake words (reusable across the system)
- Single-skill patterns are useful but less impactful

### Step 4: Name Each Wake Word

Names should be:
- Short (1-3 words)
- Descriptive of the BEHAVIOR, not the content
- Verb-first when possible ("deep-scan", "validate-strict", "anti-slop")
- Memorable and distinct from each other

### Step 5: Define Activation Behavior

For each wake word, specify:
- Trigger phrase (what the user or orchestrator says)
- Behavior set (what activates)
- Constraints that activate
- Output format changes (if any)
- Duration (this turn only, or persistent until deactivated)

### Step 6: Assess Extraction Value

Score each candidate:
- **HIGH VALUE:** Appears 3+ times, or applies across multiple skills
- **MEDIUM VALUE:** Appears 2 times, or represents a useful mode switch
- **LOW VALUE:** Appears once but represents a complex behavior that's easier to name than re-explain

Only prescribe HIGH and MEDIUM value extractions. LOW value candidates are listed as "optional."

---

## OUTPUT SPECIFICATION

```markdown
## WAKE WORD CANDIDATES: [Target Skill / System Name]

### High-Value Extractions

#### WAKE WORD: "[name]"
**Trigger:** When "[trigger phrase]" is invoked
**Behavior:**
- [Behavior/constraint 1]
- [Behavior/constraint 2]
- [Behavior/constraint 3]
**Duration:** [per-turn | persistent | until-deactivated]
**Found in:** [List of sections/skills where this pattern appears]
**Reusability:** [this skill only | cross-skill | system-wide]

[Repeat for each high-value candidate]

### Medium-Value Extractions

#### WAKE WORD: "[name]"
**Trigger:** "[trigger phrase]"
**Behavior:**
- [Behavior 1]
- [Behavior 2]
**Found in:** [where]
**Reusability:** [scope]

[Repeat for each medium-value candidate]

### Optional Extractions (Low Value)
- "[name]" — [brief description] — [found in section X]
- "[name]" — [brief description] — [found in section Y]

### Composability Notes
These wake words can be STACKED:
- "[word A]" + "[word B]" = [combined behavior]
- "[word C]" + "[word A]" = [combined behavior]

### Integration Instructions
- Add wake word definitions to [CLAUDE.md | skill's REFERENCE section | system-wide wake word registry]
- Reference by name in skill instructions instead of re-explaining behavior
```

---

## CONSTRAINTS

- NEVER extract a pattern that only appears once and is simple enough to inline
- NEVER create wake words with ambiguous names (must be immediately clear what they activate)
- NEVER create wake words that conflict with each other (activating both shouldn't create contradictions)
- MUST verify that the extracted behavior is ACTUALLY repeated (not just similar-looking text)
- MUST ensure wake word definitions are SELF-CONTAINED (reading the definition alone is sufficient to execute the behavior)
- Maximum 7 wake words per skill (more than that means the skill needs restructuring, not more wake words)
- Wake word names MUST NOT collide with existing wake words in the system

---

## FALLBACK

- If no repeated patterns found: Report "No wake word candidates identified — skill behaviors are all unique/contextual"
- If only LOW value candidates found: Report them as optional but don't prescribe extraction
- If the skill is very short (<50 lines): Unlikely to have extractable patterns — report N/A
