# 2.1-B: Exemplar Generator
*Generate missing exemplars for Block 3 (Reference)*

---

## PURPOSE

When the Structure Auditor (1.1-A) identifies a low Block 3 (Reference) score, this skill generates the missing exemplars. Exemplars are the single most powerful prompt architecture element — they show the model what "good" looks like, bypassing the limitations of verbal description.

**Scope:** Generate 2-3 positive exemplars + 1-2 anti-exemplars for the target skill's output type.
**Non-Goals:** Do NOT generate full reference documentation. Do NOT change the skill's purpose. Only produce the exemplar content that's missing.

---

## INPUTS

- Structure Auditor output from 1.1-A (Block 3 score and gap description)
- Target skill file content (to understand what it produces)
- Output specification from Block 4 (if present — defines what exemplars should look like)

---

## INSTRUCTIONS

### Step 1: Identify Exemplar Type Needed

Based on the skill's purpose and output spec, determine what kind of exemplars are needed:

| Skill Type | Exemplar Type Needed |
|------------|---------------------|
| Generation skill | Examples of ideal output |
| Analysis skill | Examples of ideal analysis format |
| Validation skill | Examples of pass/fail scenarios |
| Scraper skill | Examples of expected data format |
| Orchestrator | Examples of coordination patterns |
| Copy/marketing | Examples of ideal copy in the target voice |

### Step 2: Generate Positive Exemplars (2-3)

For each exemplar:
- Must demonstrate the QUALITY BAR (not just format compliance)
- Must be realistic (could plausibly occur in production)
- Must be distinct from each other (different scenarios/inputs)
- Must show the skill's output at its BEST
- Must follow the output specification format exactly

**Structure per exemplar:**
```markdown
### Exemplar [N]: [Brief scenario description]

**Input context:** [What triggered this output]
**Output:**
[The ideal output this skill would produce]

**Why this is good:** [1-2 sentences explaining what makes this exemplary]
```

### Step 3: Generate Anti-Exemplars (1-2)

Anti-exemplars show what the skill should NEVER produce. They teach through contrast.

For each anti-exemplar:
- Must represent a COMMON failure mode (not an absurd edge case)
- Must be clearly distinguishable from the positive exemplars
- Must demonstrate specific quality violations
- Must include annotation of what's wrong

**Structure per anti-exemplar:**
```markdown
### Anti-Exemplar [N]: [Brief scenario description]

**Input context:** [Same or similar input as a positive exemplar]
**Bad output:**
[What a poorly-configured skill would produce]

**What's wrong:**
- [Specific violation 1]
- [Specific violation 2]
- [Specific violation 3]
```

### Step 4: Align with Existing Output Spec

If the skill has a Block 4 (Output) specification:
- Exemplars MUST match the specified format exactly
- Exemplars MUST satisfy the stated quality criteria
- Anti-exemplars MUST violate specific stated constraints

If no Block 4 exists:
- Generate exemplars in the most likely format for this skill type
- Note that output spec should also be tightened (flag for 2.2-B)

### Step 5: Apply Anti-Slop Standards

All generated exemplar content MUST pass anti-slop standards:
- No filler phrases
- No hedge words
- No generic modifiers
- Specific and concrete
- Every sentence earns its place

Anti-exemplars should DELIBERATELY include slop (as demonstration of what's wrong), clearly annotated.

## CONSTRAINTS

- NEVER generate exemplars that are generic enough to apply to ANY skill (must be specific to THIS skill's domain)
- NEVER generate exemplars that contradict the skill's existing constraints or purpose
- NEVER make anti-exemplars too absurd — they should represent REALISTIC failure modes
- MUST generate at least 2 positive exemplars (1 is insufficient for pattern recognition)
- MUST generate at least 1 anti-exemplar (contrast is essential for learning)
- Positive exemplars MUST be genuinely excellent — not just "acceptable"
- Anti-exemplars MUST contain specific, annotated violations (not just "this is bad")
- If the skill produces structured data (YAML, JSON), exemplars must be valid syntax
- Exemplar length should match expected output length (don't write a 500-word exemplar for a skill that produces 50-word outputs)

---

## FALLBACK

- If skill purpose is unclear (can't determine what output should look like): Flag for manual review, generate best-effort exemplars with "tentative" marker
- If skill produces highly variable outputs (depends entirely on input): Generate exemplars for 3 different input scenarios to show range
