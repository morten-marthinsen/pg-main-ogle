# 2.5-A: Layer 2 Validator
*Validate all prescriptions before passing to Layer 3*

---

## PURPOSE

Verify that all Layer 2 prescriptions are internally consistent, complete, implementable, and properly traced to Layer 1 findings. This gate prevents contradictory or incomplete prescriptions from reaching the Skill Rewriter (Layer 3).

**Scope:** Cross-check all prescriptions. Verify traceability. Detect conflicts. Confirm implementability.
**Non-Goals:** Do NOT implement prescriptions. Do NOT add new prescriptions. Validate only.

---

## INPUTS

All Layer 2 skill outputs:
- 2.1-A: Constraint rewrites
- 2.1-B: Generated exemplars
- 2.2-A: Guardrail injections
- 2.2-B: Output spec tightening
- 2.3-A: Wake word extractions
- 2.3-B: Context layer separation
- 2.4-A: Chain-of-Refinement protocol

Layer 1 Diagnostic Summary (for traceability check)

---

## INSTRUCTIONS

### Step 1: Traceability Check

For every Layer 2 prescription, verify it traces to a Layer 1 finding:

```
Layer 1 Finding → Layer 2 Prescription
[Finding X] → [Prescription Y that addresses Finding X]
```

Flag any:
- **Orphan prescriptions:** Prescriptions that don't trace to any finding (prescription without diagnosis)
- **Unaddressed findings:** Priority 1-2 findings with no corresponding prescription
- **Mismatched prescriptions:** Prescription addresses a different issue than the finding it claims to fix

### Step 2: Conflict Detection

Check for contradictions between prescriptions:

**Constraint Conflicts:**
- Does any rewritten constraint (2.1-A) contradict a guardrail injection (2.2-A)?
- Does any binary rule (guardrail Pattern 5) contradict a Chain-of-Refinement flexibility (2.4-A)?
- Does any output spec constraint (2.2-B) conflict with existing preserved content?

**Structural Conflicts:**
- Does the context layer separation (2.3-B) put content in Layer 1 that another prescription treats as flexible?
- Does positional reinforcement (moving rules to first/last) break the insertion points specified for guardrails?

**Behavioral Conflicts:**
- Do wake words (2.3-A) activate behaviors that contradict constraints?
- Does the Chain-of-Refinement critique criteria contradict the output spec quality criteria?

### Step 3: Implementability Check

For each prescription, verify:
- **Insertable:** The specified insertion point exists in the target file
- **Complete:** All template content is filled (no remaining brackets [])
- **Formatted:** Matches the target file's markdown style
- **Sized:** Won't bloat the skill beyond reasonable length (flag if total additions >200 lines)
- **Independent:** Can be applied without prerequisites from other prescriptions (or dependencies are noted)

### Step 4: Completeness Check

Verify all Priority 1 and Priority 2 findings have prescriptions:

| Finding | Priority | Prescription | Status |
|---------|----------|--------------|--------|
| [Finding 1] | P1 | [Which prescription addresses it] | [COVERED | UNCOVERED] |
| [Finding 2] | P1 | [Prescription] | [COVERED | UNCOVERED] |
| [Finding 3] | P2 | [Prescription] | [COVERED | UNCOVERED] |

**Rule:** All P1 findings MUST be covered. P2 findings SHOULD be covered. P3-P4 findings are optional.

### Step 5: Volume Assessment

Count total prescriptions and assess:
- Is this a reasonable number of changes? (>15 prescriptions suggests the skill needs REBUILD, not piecemeal fixes)
- Are prescriptions focused on the highest-impact issues?
- Would applying all prescriptions make the skill significantly better, or just different?

### Step 6: Gate Decision

```
IF all checks pass:
    → PROCEED to Layer 3 with validated prescriptions
    → Include priority ordering for implementation

IF conflicts detected:
    → RESOLVE conflicts (remove lower-priority prescription in each conflict pair)
    → Re-validate after resolution
    → Then PROCEED

IF unaddressed P1 findings:
    → FLAG for orchestrator attention
    → PROCEED with available prescriptions + explicit gap list

IF prescriptions are unimplementable:
    → REVISE insertion points / format
    → Re-validate
    → Then PROCEED
```

---

## CONSTRAINTS

- NEVER allow orphan prescriptions through (prescriptions must trace to findings)
- NEVER allow conflicting prescriptions through (resolve ALL conflicts before proceeding)
- NEVER allow P1 findings to go uncovered without explicit flagging
- MUST resolve conflicts by keeping the higher-priority/higher-impact prescription
- MUST order prescriptions by priority for Layer 3 (most impactful first)
- If total prescription volume suggests REBUILD: Flag to orchestrator, but still proceed with OPTIMIZE pipeline (let Layer 3 validator make the final call)
- The validated prescriptions package is the BINDING input to Layer 3 — nothing else

---

## FALLBACK

- If Layer 2 skills produced errors (missing outputs): Note gaps, validate what's available, flag missing prescriptions
- If no prescriptions were generated (all diagnostics passed): Report "No prescriptions needed — skill passes all checks" and recommend PASS instead of OPTIMIZE
