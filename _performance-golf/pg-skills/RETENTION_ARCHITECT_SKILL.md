**Statistical Testing Fundamentals**:

| Element | Standard | Notes |
|---------|----------|-------|
| **Sample Size** | Calculate based on baseline + MDE | Baseline conversion, minimum detectable effect (10-20%), statistical power (80%), significance (95%) |
| **Test Duration** | Minimum 2 weeks | Full business cycle |
| **Segmentation** | Always analyze by RFM | Avoid Simpson's Paradox |
| **Multiple Testing** | Bonferroni adjustment | When running multiple simultaneous tests |

**Backend A/B Testing Priorities** (ranked by impact):

1. **Email Subject Lines** → 20-40% open rate lifts possible

**SQL Query Templates for Backend Analysis**:

**Cohort Retention Query**:
```sql
WITH cohorts AS (
  SELECT 
    DATE_TRUNC('day', first_purchase_date) as cohort_date,
    customer_id,
    DATEDIFF('day', first_purchase_date, purchase_date) as days_since_first
  FROM orders
)
SELECT 
  cohort_date,
  days_since_first,
  COUNT(DISTINCT customer_id) as customers,
  SUM(revenue) as cohort_revenue
FROM cohorts
GROUP BY 1,2
```

**RFM Segmentation Query**:
```sql
WITH customer_metrics AS (
  SELECT 
    customer_id,
    DATEDIFF('day', MAX(purchase_date), CURRENT_DATE) as recency,
    COUNT(DISTINCT order_id) as frequency,
    SUM(revenue) as monetary
  FROM orders
  GROUP BY 1
),
rfm_scores AS (
  SELECT 
    customer_id,
    NTILE(5) OVER (ORDER BY recency DESC) as r_score,
    NTILE(5) OVER (ORDER BY frequency ASC) as f_score,
    NTILE(5) OVER (ORDER BY monetary ASC) as m_score
  FROM customer_metrics
)
SELECT 
  customer_id,
  CONCAT(r_score, f_score, m_score) as rfm_segment
FROM rfm_scores
```

**Flow Performance Query**:
```sql
SELECT 
  flow_name,
  COUNT(DISTINCT recipient) as recipients,
  SUM(CASE WHEN clicked THEN 1 ELSE 0 END) as clicks,
  SUM(attributed_revenue) as revenue,
  SUM(attributed_revenue)/COUNT(DISTINCT recipient) as revenue_per_recipient
FROM email_performance
WHERE sent_date >= CURRENT_DATE - 30
GROUP BY 1
ORDER BY revenue DESC
```

### CRO & SPLIT TESTING PRINCIPLES

**Split Testing Methodology**:

**Hierarchy of Impact** (test in this order):
1. **Headlines/Subject Lines** → Biggest impact
2. **Offers** → Value proposition clarity
3. **CTAs** → Action clarity and friction
4. **Body Copy** → Supporting arguments
5. **Design** → Visual hierarchy and flow

**Statistical Standards**:
- Minimum 95% confidence (p-value < 0.05)
- Adequate sample size (calculate before starting)
- Test velocity: One variable per test
- Duration: 2-4 week cycles minimum
- Documentation: Control, variations, results, insights

**Channel-Specific Optimization**:

**Email CRO Elements**:
- **Subject Line**: Length (6-10 words optimal), personalization, urgency, curiosity
- **Preview Text**: Complementary vs contradictory to subject
- **Above Fold**: Value prop clarity, visual hierarchy, product image
- **CTA Design**: Color contrast, size (48px minimum), copy clarity, placement
- **Social Proof**: Placement (near CTA), specificity ("437 sold"), relevance

**SMS CRO Elements**:
- **Hook Speed**: Get to value in first 20 characters
- **Link Placement**: After first sentence (not at end)
- **Urgency Level**: "Next 2 hours" vs "While supplies last"
- **Personalization**: Name, purchase history, weather, behavior
- **Compliance**: Clear opt-out, sender ID

**Direct Mail CRO Elements**:
- **Envelope**: Teaser copy, handwritten font, urgency
- **Johnson Box**: Offer summary at top (boxed)
- **Letter Length**: Long-form for complex offers, short for urgency
- **Response Device**: Pre-filled info, multiple options (QR, phone, card)
- **Package Elements**: Lift notes, testimonials, dimensional items

**Popup/On-site CRO Elements**:
- **Trigger Timing**: Exit intent vs time-based vs behavior-based
- **Offer Clarity**: Discount percentage vs value-add
- **Form Fields**: Minimal (email only) vs progressive disclosure
- **Mobile Optimization**: Thumb-friendly, 48px tap targets
- **Frequency Capping**: Don't annoy repeat visitors

**Backend-Specific CRO Metrics**:

| Channel | Minimum | Excellent | Optimization Potential |
|---------|---------|-----------|----------------------|
| **Email Click Rate** | 2-4% | 7%+ | Subject lines, preview text, above-fold |
| **SMS Click Rate** | 8-12% | 20%+ | Hook speed, personalization, urgency |

| **Cart Recovery** | 10% | 20%+ | Progressive incentives, urgency |

### NAMING CONVENTION SYSTEM

**Purpose**: Create scannable, searchable, analytically-powerful naming that instantly communicates campaign purpose and enables sophisticated reporting.

**Core Structure**:

```
CAMPAIGN LEVEL
├── FLOWS: [STAGE]_[TRIGGER]_[PURPOSE]_[CAMPAIGN]
│   └── Individual Emails: [STAGE]_[TRIGGER]_[PURPOSE]_[CAMPAIGN]_[NUMBER/SMS]
│       └── Split Tests: [STAGE]_[TRIGGER]_[PURPOSE]_[CAMPAIGN]_[NUMBER][A/B]
│
└── BROADCASTS: [STAGE]_BCAST_[PURPOSE]_[CAMPAIGN]_[VERSION]
    └── Split Tests: [STAGE]_BCAST_[PURPOSE]_[CAMPAIGN]_[A/B/SMS]
```

**Component Definitions**:

| Component | Options | Purpose |
|-----------|---------|---------|
| **STAGE** | PRE (pre-purchase), POST (post-purchase) | Customer status |
| **TRIGGER** | LIST, CART, ORDER, VIEW, TIME | What started this flow |
| **PURPOSE** | WELCOME, ABANDON, UPSELL, WINBACK, CONFIRM | Campaign goal |
| **CAMPAIGN** | TIGERBETA, PGTA, CIJ, ASSESS2024, etc. | Specific offer/campaign |
| **VERSION** | 1, 2, 3 (sequence), A/B (split test), SMS | Position or variant |

**Examples**:

**Flow Naming**:
- `PRE_CART_ABANDON_TIGERBETA` → Flow name
- `PRE_CART_ABANDON_TIGERBETA_1` → First email
- `PRE_CART_ABANDON_TIGERBETA_2A` → Second email, variant A
- `PRE_CART_ABANDON_TIGERBETA_SMS` → Text message

**Broadcast Naming**:
- `PRE_BCAST_PROMO_CIJ_A` → Promotional broadcast, version A
- `POST_BCAST_UPSELL_PGTA_B` → Upsell broadcast, version B
- `PRE_BCAST_PROMO_CIJ_SMS` → SMS version

**Decision Tree for Naming**:

1. **Flow or Broadcast?**
   - Flow → Use trigger (CART, ORDER, LIST, VIEW, TIME)
   - Broadcast → Use BCAST

2. **Customer Status?**
   - Haven't bought yet → PRE
   - Already customers → POST

3. **What's the purpose?**
   - WELCOME, ABANDON, UPSELL, WINBACK, CONFIRM

4. **Which campaign?**
   - TIGERBETA, PGTA, CIJ, PUTTERLAUNCH, etc.

5. **Position/Version?**
   - Flow emails: 1, 2, 3, SMS
   - Split tests: A, B, C
   - Broadcasts: A, B, SMS

**Dashboard Analysis Power**:

This naming convention enables powerful filtering:

**Strategic Analysis**:
- All Pre-Purchase: Filter by `PRE_*`
- All Post-Purchase: Filter by `POST_*`
- Campaign Performance: Filter by `*_TIGERBETA`, `*_PGTA`, etc.

**Tactical Optimization**:
- Trigger Effectiveness: Compare `*_CART_*` vs `*_ORDER_*` vs `*_TIME_*`
- Purpose Performance: Compare `*_ABANDON_*` vs `*_UPSELL_*` vs `*_WELCOME_*`
- Channel Analysis: Compare regular emails vs `*_SMS`

**Campaign Analysis**:
- Campaign Variants: Compare `*_TIGERBETA` vs `*_TIGERCASE`
- Site-wide vs Specific: Compare `*_CIJ` vs `*_TIGER*` performance
- A/B Winners: Compare `*_A` vs `*_B` performance

**UTM Naming Convention**:

Klaviyo campaigns use custom UTM parameters for precise tracking:

| Parameter | Value | Notes |
|-----------|-------|-------|
| `utm_source` | Klaviyo | Platform (locked, never changes) |
| `utm_medium` | email or sms | Channel (locked, never changes) |
| `utm_campaign` | Campaign name | Dynamic field from naming structure above (locked) |
| `utm_id` | Campaign ID | Unique Klaviyo ID (locked) |
| `utm_term` | OFFER_Backend | **MUST CHANGE EVERY SEND/BUILD** |

**Critical Rule for utm_term**:
- Changes per launch/flow to track offers specifically
- Example launch: `SSG2_Backend` (same across entire launch)
- Example flow: `AbandonedCart_Backend` (same across entire flow)
- Enables precise DOMO tracking by offer

### THE UNDERSTANDING FRAMEWORK

**Core Principle**: "To the degree you don't understand a person, place, thing or circumstance is the degree you will be effect to it."

**Why Understanding Matters**:
- Less understanding = More friction = Less power = Less income = Less impact
- Most entrepreneurs stop 50% short of gathering enough data
- Root cause: Fear of looking stupid/inadequate

**Keys to the Kingdom**:
1. Goal is NOT to look smart - goal is to UNDERSTAND THOROUGHLY at all costs
2. Everyone else is terrified of looking dumb - they'll thank you for clarifying
3. If you don't clarify, everyone assumes you're clear
4. Repeat back your understanding until 100% clear

**Clarifying Questions Library**:

**Binary Clarification**:
- "Do you mean A or B?"
- "Is it X or Y you're referring to?"

**Reframing Request**:
- "I'm not sure I'm understanding, can you explain it a different way?"
- "Could you give me an example of what you mean?"

**Specific Clarification**:
- "What did you mean when you said ____________?"
- "What specifically are you referring to when you said __________?"

**Permission-Based Questions**:
- "Is it ok if I ask a dumb question?" (then ask)
- "Can I pause you to clarify something?"

**Circumstance Exploration**:
- "Is there anyone (or anything) else involved in this (situation)?"
- "What seems to be the challenge for you, specifically?"

**Definition Requests**:
- "What does that word mean?"
- "How does (this) affect (that)?"
- "How exactly does that work?"

**Playback Verification**:
- "I'm not sure I'm tracking with you, is it ok if I pause you to clarify?"
- "Let me make sure I'm understanding you (explain back) ... is that right?"

**Application in Retention Architect**:
- Use these questions to clarify user requests before building solutions
- Never assume - always verify understanding
- Repeat back what you've understood before proceeding
- Ask for specific examples when concepts are vague

## SQL QUERIES TO RUN

### Query 1: Daily Cohort Retention

```sql
WITH cohorts AS (
  SELECT 
    DATE_TRUNC('day', first_purchase_date) as cohort_date,
    customer_id,
    DATEDIFF('day', first_purchase_date, purchase_date) as days_since_first
  FROM orders
)
SELECT 
  cohort_date,
  days_since_first,
  COUNT(DISTINCT customer_id) as customers,
  SUM(revenue) as cohort_revenue
FROM cohorts
GROUP BY 1,2
ORDER BY 1,2
```

### Query 2: Email Revenue Attribution

```sql
SELECT 
  flow_name,
  COUNT(DISTINCT recipient) as recipients,
  SUM(CASE WHEN clicked THEN 1 ELSE 0 END) as clicks,
  SUM(attributed_revenue) as revenue,
  SUM(attributed_revenue)/COUNT(DISTINCT recipient) as revenue_per_recipient
FROM email_performance
WHERE sent_date >= CURRENT_DATE - 30
GROUP BY 1
ORDER BY revenue DESC
```

### Query 3: Segment Performance Analysis

```sql
WITH customer_segments AS (
  -- Your RFM segmentation logic
  SELECT customer_id, rfm_segment
  FROM rfm_table
)
SELECT 
  cs.rfm_segment,
  COUNT(DISTINCT o.customer_id) as customers,
  COUNT(DISTINCT o.order_id) as orders,
  SUM(o.revenue) as revenue,
  AVG(o.revenue) as avg_order_value
FROM orders o
JOIN customer_segments cs ON o.customer_id = cs.customer_id
WHERE o.purchase_date >= CURRENT_DATE - 30
GROUP BY 1
ORDER BY revenue DESC
```

**Response:**

## CHALLENGE IDENTIFICATION

## TEST PARAMETERS

| Parameter | Value | Notes |

| **Significance Level** | 95% | Standard confidence |
| **Statistical Power** | 80% | Standard power |

**REQUIRED SAMPLE SIZE:**  
You need **6,200 recipients per variant** (12,400 total) to detect a 25% improvement with confidence.

## TEST STRUCTURE

### Variant A (Control): Current Flow
- Keep exactly as is
- No changes whatsoever
- Baseline measurement

### Variant B (Challenger): New Flow

**Key Changes to Test:**

1. **Subject Line:**  
   - **From:** "We miss you"  
   - **To:** "Your exclusive comeback offer expires in 48 hours"

2. **Timing:**  
   - **From:** 30-45-60 days  
   - **To:** 45-60-75 days

3. **Offer:**

- Purchase conversion rate

**Secondary Metrics:**

- Click rate

## TEST DURATION

- **Minimum:** 14 days
- **Recommended:** 21 days (full purchase cycle)
- **CRITICAL:** Do NOT peek at results before completion

## ANALYSIS APPROACH

### Statistical Significance Testing

```python
from scipy.stats import chi2_contingency

# Example calculation
control_conversions = 310  # 5% of 6,200
variant_conversions = 388  # 6.25% of 6,200
control_total = 6200
variant_total = 6200

# Create contingency table
data = [[control_conversions, control_total - control_conversions],
        [variant_conversions, variant_total - variant_conversions]]

# Run chi-squared test
chi2, p_value, dof, expected = chi2_contingency(data)

if p_value < 0.05:
    print("Result is statistically significant!")
else:
    print("Not significant - continue testing or iterate")
```
