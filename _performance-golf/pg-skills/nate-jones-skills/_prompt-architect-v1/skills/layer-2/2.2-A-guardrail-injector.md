# 2.2-A: Guardrail Injector
*Generate and position missing guardrail patterns*

---

## PURPOSE

For each guardrail pattern marked ABSENT or PARTIAL by the Guardrail Auditor (1.2-A), generate the specific guardrail text appropriate for this skill's context and specify where to position it in the document.

**Scope:** Generate missing guardrail pattern implementations. Specify insertion positions. Ensure no conflicts with existing content.
**Non-Goals:** Do NOT restructure the skill. Do NOT modify existing guardrails that are already present. Only add what's missing.

---

## INPUTS

- Guardrail Auditor output from 1.2-A (which patterns are missing, skill type)
- Target skill file content
- Skill type (determines which patterns are mandatory)

---

## INSTRUCTIONS

### Step 1: Generate Missing Patterns

For each ABSENT or PARTIAL pattern, generate implementation text:

**Pattern 1: Identity Invariants (if missing)**
```markdown
## IDENTITY

You are [skill name] — a [specific role description].
You are NOT [common misidentification 1].
You are NOT [common misidentification 2].

If asked to perform tasks outside your scope, respond:
"This is outside [skill name]'s scope. My role is [purpose]. For [requested task], use [appropriate skill/tool]."
```
*Customize: Replace bracketed content with skill-specific identity.*

**Pattern 2: Trigger-Template Refusals (if missing)**
```markdown
## EDGE CASE HANDLING

IF [edge case 1 — most likely out-of-scope request]:
→ Respond: "[Template refusal explaining why and what to do instead]"

IF [edge case 2 — common misuse]:
→ Respond: "[Template refusal]"

IF [edge case 3 — ambiguous input]:
→ Respond: "[Request clarification template]"
```
*Customize: Identify the 2-3 most likely edge cases for THIS skill.*

**Pattern 3: Three-Tier Uncertainty (if missing)**
```markdown
## CONFIDENCE PROTOCOL

CERTAIN (>90% confidence): State directly without qualification.
PROBABLE (60-90% confidence): State with explicit qualifier: "Based on [evidence], this is likely [conclusion]. Confidence: [X]%."
UNCERTAIN (<60% confidence): Flag explicitly: "UNCERTAIN: [statement]. Basis: [limited evidence]. Recommend: [verification step]."

NEVER present uncertain findings with the same authority as certain ones.
```
*Customize: Define what "evidence" means for this skill's domain.*

**Pattern 4: Locked Tool Grammar (if missing, and skill uses tools)**
```markdown
## TOOL USAGE RULES

Before calling [tool name]:
- VERIFY: [precondition 1]
- VERIFY: [precondition 2]
- FORMAT: Input must be [required format]

NEVER call [tool] without [precondition].
NEVER call [tool] with [invalid input type].

After [tool] returns:
- CHECK: Response format is [expected]
- CHECK: Data contains [required fields]
- IF check fails: [fallback action]
```
*Customize: Specific to the tools this skill uses.*

**Pattern 5: Binary Style Rules (if missing)**
```markdown
## STYLE RULES (BINARY — NO EXCEPTIONS)

ALWAYS:
- [Rule 1 — specific to this skill's output type]
- [Rule 2]
- [Rule 3]

NEVER:
- [Rule 1 — most common quality violation for this output type]
- [Rule 2]
- [Rule 3]
```
*Customize: Based on anti-slop findings and skill purpose.*

**Pattern 6: Positional Reinforcement (if missing)**
This pattern is structural, not content. Instructions:
- Move the single most important constraint to the FIRST section after PURPOSE
- Echo that constraint in the LAST section before DOCUMENT INFO
- If the skill has a CONSTRAINTS section, ensure it contains the top 3 rules from throughout

**Pattern 7: Post-Tool Reflection (if missing, and skill uses tools/produces outputs)**
```markdown
## POST-EXECUTION VALIDATION

After producing output, verify:
1. Format matches specification: [specific format check]
2. All required sections present: [section checklist]
3. Content passes quality bar: [specific quality check]
4. No constraint violations: [re-check top 3 constraints]

IF any check fails:
→ Do NOT deliver
→ Identify which check failed
→ Revise output to pass all checks
→ Re-validate before delivery
```
*Customize: Specific checks for this skill's output type.*

### Step 2: Determine Insertion Positions

Position guardrails according to the Two-Layer Context Architecture:

**Layer 1 (Deterministic — top of file, after PURPOSE):**
- Identity Invariants
- Binary Style Rules
- Locked Tool Grammar

**Layer 2 (Adaptive — after INSTRUCTIONS, before OUTPUT):**
- Three-Tier Uncertainty
- Trigger-Template Refusals
- Post-Tool Reflection

**Structural (applied to document organization):**
- Positional Reinforcement (reorganization, not insertion)

### Step 3: Check for Conflicts

Before finalizing:
- Does any generated guardrail contradict an existing constraint?
- Does any guardrail duplicate an existing rule? (Merge, don't double)
- Do the guardrails create impossible conditions? (e.g., "ALWAYS do X" + "NEVER do X")

If conflicts found: Resolve by deferring to existing content (it was there first and may be intentional).

## OUTPUT SPECIFICATION

```markdown
## GUARDRAIL PRESCRIPTIONS: [Target Skill Name]

### Guardrails to Add

#### Pattern [N]: [Pattern Name]
**Position:** [Where in the document — after which section]
**Content:**
```
[Generated guardrail text]
```
**Rationale:** [Why this pattern matters for this skill]
**PRD Reference:** ARCHITECTURE-PRD §4.2 Pattern [N]

[Repeat for each missing pattern]

### Positional Reinforcement Changes (if applicable)
- Move "[constraint text]" from [current position] to [first section]
- Echo "[constraint text]" in [final section]

### Conflict Check
- Conflicts detected: [None | List of conflicts with resolution]

### Expected Impact
- Guardrail score: [before] → [after]
- Patterns present: [before count] → [after count]
```

---

## CONSTRAINTS

- NEVER generate guardrails that are generic/boilerplate — every guardrail must be specific to THIS skill's context
- NEVER add Pattern 4 (Locked Tool Grammar) or Pattern 7 (Post-Tool Reflection) to skills that don't use tools
- NEVER create guardrails that conflict with existing constraints
- MUST customize all template content (no brackets remaining in output)
- MUST specify exact insertion position (not "somewhere in the middle")
- If the skill is a leaf skill with threshold ≥3, only generate the 3 most impactful patterns (don't over-guardrail simple skills)
- Guardrails should be CONCISE — each pattern should be 5-15 lines, not paragraphs

---

## FALLBACK

- If all 7 patterns are already present: Report "Guardrails complete — no injections needed"
- If skill type makes certain patterns irrelevant: Skip those patterns, note as N/A in output
