# 2.2-B: Output Spec Tightener
*Strengthen Block 4 (Output) specifications*

---

## PURPOSE

When the Structure Auditor (1.1-A) scores Block 4 below 5, this skill tightens the output specification to eliminate ambiguity. The output spec is where most skill failures originate — if the agent doesn't know exactly what "done" looks like, it guesses. Guessing produces slop.

**Scope:** Add format precision, length constraints, quality criteria, and validation checks to the output spec.
**Non-Goals:** Do NOT change what the skill produces (the deliverable type). Only tighten HOW it's specified.

---

## INPUTS

- Structure Auditor output from 1.1-A (Block 4 score and gap description)
- Anti-Slop Scanner output from 1.2-B (any output_spec_issues identified)
- Target skill file content

---

## INSTRUCTIONS

### Step 1: Assess Current Output Spec State

Classify the current Block 4:

| State | Description | Action |
|-------|-------------|--------|
| ABSENT | No output section at all | Generate complete spec from scratch |
| MINIMAL | "Produce a report" / one-liner | Expand to full specification |
| PARTIAL | Format mentioned, missing criteria | Add missing elements |
| ALMOST | Good spec, missing validation | Add validation checks only |

### Step 2: Add Format Specification (if missing)

Define the EXACT format:
```markdown
## OUTPUT FORMAT

**Type:** [markdown | yaml | json | structured text | code]
**Encoding:** [standard markdown with H2/H3 headers]
**Sections Required:** [explicit list of required sections in order]
**Section Order:** [fixed | flexible within groups]
```

### Step 3: Add Structure Specification (if missing)

Define the document skeleton:
```markdown
## OUTPUT STRUCTURE

```
[Exact template showing:
- All required headers
- Required fields per section
- Optional fields (marked)
- Nesting/hierarchy
- Code block format for structured data]
```
```

### Step 4: Add Length Constraints (if missing)

Define bounds:
```markdown
## LENGTH CONSTRAINTS

**Total output:** [min-max lines OR words]
**Per section:** [max lines/words per section]
**Tables:** [max rows if applicable]
**Lists:** [max items if applicable]
**Code blocks:** [max lines if applicable]

NEVER exceed [max]. If content requires more, summarize and reference full detail.
```

### Step 5: Add Quality Criteria (if missing)

Define what "good" means:
```markdown
## QUALITY CRITERIA

The output is ACCEPTABLE when:
- [ ] [Criterion 1 — specific, measurable]
- [ ] [Criterion 2 — specific, measurable]
- [ ] [Criterion 3 — specific, measurable]

The output is UNACCEPTABLE if:
- [X] [Violation 1 — specific anti-pattern]
- [X] [Violation 2 — specific anti-pattern]
```

### Step 6: Add Validation Checks (if missing)

Define the self-check:
```markdown
## VALIDATION (Before Delivery)

Run these checks before delivering output:
1. FORMAT: Does it match the template structure? [Y/N]
2. COMPLETENESS: Are all required sections present? [Y/N]
3. LENGTH: Within specified bounds? [Y/N]
4. QUALITY: Passes all quality criteria? [Y/N]
5. CONSTRAINTS: No constraint violations detected? [Y/N]

ALL must be Y. If any is N, revise before delivery.
```

### Step 7: Eliminate Vague Language

Replace any remaining vague output language:

| Vague | Tightened |
|-------|-----------|
| "comprehensive report" | "Report containing sections A, B, C with minimum 3 findings per section" |
| "detailed analysis" | "Analysis with per-dimension scores, evidence quotes, and gap identification" |
| "clear summary" | "3-5 bullet summary, each bullet ≤20 words, covering [specific aspects]" |
| "appropriate format" | "[exact format: markdown with H2 per section, tables for scores]" |
| "high-quality output" | "Passes anti-slop scan (≤2 instances/100 lines) + all quality criteria checked" |

---

## OUTPUT SPECIFICATION

```markdown
## OUTPUT SPEC PRESCRIPTION: [Target Skill Name]

### Current State
- Block 4 Score: [X]/5
- Classification: [ABSENT | MINIMAL | PARTIAL | ALMOST]
- Key gaps: [list what's missing]

### Prescribed Output Specification

[Complete, ready-to-insert output specification covering:
- Format
- Structure (with template)
- Length constraints
- Quality criteria
- Validation checks]

### Vague Language Replacements
| Line | Original | Replacement |
|------|----------|-------------|
| [N] | "[vague text]" | "[precise text]" |
[continue for all instances]

### Integration Instructions
- Replace existing output section at line [N] with prescribed specification
- OR insert new section after [section name] if no output section exists

### Expected Impact
- Block 4 score: [before] → [expected after]
- Validation presence: [absent] → [present]
- Agent self-check capability: [none] → [5-point checklist]
```

---

## CONSTRAINTS

- NEVER make the output spec so rigid that the model can't adapt to different input scenarios
- NEVER add length constraints that are obviously wrong for the content type (don't say "max 50 words" for a research report)
- MUST preserve any existing output spec elements that are already well-defined
- MUST make quality criteria MEASURABLE (not "be thorough" — instead "minimum 3 evidence citations per finding")
- MUST make validation checks BINARY (pass/fail, not subjective)
- The prescribed spec should be SELF-CONTAINED — agent shouldn't need to read other files to understand what to produce
- If the skill produces multiple output types (conditional), spec each type separately

---

## FALLBACK

- If Block 4 already scores 5: Report "Output spec already optimal — no changes needed"
- If the skill's output is genuinely variable (depends entirely on input): Define the STRUCTURAL requirements (format, validation) while leaving content flexible
