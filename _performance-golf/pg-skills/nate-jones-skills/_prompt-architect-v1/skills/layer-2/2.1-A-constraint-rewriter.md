# 2.1-A: Constraint Rewriter
*Convert instruction statements into constraint statements*

---

## PURPOSE

Take instruction-heavy language identified by 1.1-B (Constraint Ratio Checker) and rewrite each qualifying instruction as a constraint. The goal: shift the ratio toward 70% constraints / 30% instructions without changing the skill's intent.

**Scope:** Rewrite flagged instructions as constraints. Preserve original intent. Produce before/after mapping.
**Non-Goals:** Do NOT add new constraints that aren't already implied. Do NOT delete existing constraints. Convert, don't invent.

---

## INPUTS

- Constraint Ratio Checker output from 1.1-B (including `sample_instructions_needing_conversion`)
- Full target skill file content
- Target ratio: ≥0.70

---

## INSTRUCTIONS

### Step 1: Identify All Conversion Candidates

From the full skill, identify every instruction statement that COULD be a constraint. An instruction is convertible when:
- It describes a behavior that should be INVARIANT (not context-dependent)
- It can be expressed as a boundary (what NOT to do, or what ALWAYS to do)
- Violating it would be a BUG, not a style choice
- The negative framing is clearer than the positive framing

### Step 2: Apply Conversion Patterns

**Pattern A: Positive Instruction → Negative Constraint**
```
BEFORE: "Write responses in a concise manner"
AFTER:  "NEVER exceed 200 words per response section. NEVER use filler phrases to pad length."
```

**Pattern B: Vague Instruction → Quantified Constraint**
```
BEFORE: "Keep the analysis thorough but focused"
AFTER:  "Each analysis section MUST address exactly one dimension. Maximum 3 paragraphs per dimension. NEVER combine multiple dimensions in one section."
```

**Pattern C: Process Instruction → Boundary Constraint**
```
BEFORE: "Try to validate your outputs before delivering"
AFTER:  "NEVER deliver output without running the validation checklist. If any check fails, the output is NOT ready."
```

**Pattern D: Preference Instruction → Binary Rule**
```
BEFORE: "Aim for active voice when possible"
AFTER:  "ALWAYS use active voice. NEVER use passive voice unless the actor is genuinely unknown."
```

**Pattern E: Guidance → Invariant**
```
BEFORE: "Consider the user's context when responding"
AFTER:  "MUST reference the user's stated context in every response. NEVER provide generic responses that ignore stated context."
```

### Step 3: Verify Conversion Quality

For each rewrite, check:
- Does the constraint preserve the INTENT of the original instruction?
- Is the constraint BINARY (unambiguous pass/fail)?
- Is the constraint ENFORCEABLE (agent can verify compliance)?
- Does the constraint IMPROVE behavior (not just rephrase)?

If ANY check fails, revise the constraint or mark as "keep as instruction" (not everything should be a constraint).

### Step 4: Recalculate Ratio

After all conversions:
```
new_ratio = (original_constraints + converted_count) / (total_operational - removed_instructions + converted_count)
```

Verify new ratio ≥0.70. If not, identify additional candidates.

### Step 5: Identify Instructions to KEEP

Some instructions SHOULD remain instructions:
- Sequential process steps (do A, then B, then C)
- Creative/generative directives that need flexibility
- Tool usage instructions (how to call a tool)
- Context-dependent actions (only if X, then do Y)

Flag these as "intentionally retained as instruction" with rationale.

## CONSTRAINTS

- NEVER convert an instruction that genuinely needs to be flexible (creative directions, context-dependent actions)
- NEVER create constraints that are impossible to verify (no "ensure quality" — that's not binary)
- NEVER change the skill's intent — only change HOW it's expressed
- MUST preserve every conversion as a traceable before/after pair
- MUST verify the new constraint is STRICTER than the old instruction (conversion should tighten, not loosen)
- If converting would make the skill worse (too rigid for its purpose), keep the instruction
- Maximum 70% conversion rate — some instructions are necessary. Don't over-convert.

---

## FALLBACK

- If ratio is already ≥0.70: Report "Already optimal — no conversions needed"
- If very few candidates exist (file is mostly context/examples): Report what's available but note "limited conversion opportunity — ratio improvement constrained by content type"
