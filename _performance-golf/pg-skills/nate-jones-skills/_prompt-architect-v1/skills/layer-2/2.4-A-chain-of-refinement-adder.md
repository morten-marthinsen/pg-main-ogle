# 2.4-A: Chain-of-Refinement Adder
*Add Think → Critique → Refine protocol to generation skills*

---

## PURPOSE

When a skill produces outputs without self-critique (detected by Production Readiness Checker 1.3-A, Principle 6: Continuous Validation, or by absence of post-generation review), this skill adds the Chain-of-Refinement protocol (ARCHITECTURE-PRD §2.5). This transforms single-pass generation into multi-pass quality improvement.

**Scope:** Add refinement protocol appropriate to skill type. Map to think hierarchy. Define critique criteria. Specify refinement triggers.
**Non-Goals:** Do NOT add Chain-of-Refinement to skills that are purely analytical, purely validation, or single-shot lookups. Only generation/production skills benefit.

---

## INPUTS

- Target skill file content
- Production Readiness output from 1.3-A (Principle 6 score)
- Skill type classification
- Anti-Slop findings from 1.2-B (inform critique criteria)

---

## INSTRUCTIONS

### Step 1: Determine Applicability

Chain-of-Refinement is appropriate when:
- Skill GENERATES content (copy, reports, analyses, recommendations)
- Output quality is subjective or has multiple valid approaches
- A single pass may miss requirements or produce slop
- The skill's output will be consumed by humans or critical downstream processes

Chain-of-Refinement is NOT appropriate when:
- Skill is purely mechanical (format conversion, data extraction)
- Skill is a validator/gate (already exists to critique others)
- Output is binary (pass/fail, yes/no)
- Skill is a single-shot lookup

**If not appropriate:** Report N/A and exit.

### Step 2: Define the Three Phases

**Phase 1 — THINK (Generate):**
```markdown
### Phase 1: Generate
- Produce initial output following all instructions
- Full creative/analytical pass — don't self-censor
- Apply format and structure requirements
- THINK command: `think` (standard depth)
```

**Phase 2 — CRITIQUE (Evaluate):**
Define specific critique dimensions for this skill:
```markdown
### Phase 2: Critique
Evaluate Phase 1 output against:
1. [Dimension 1 — specific to this skill's quality bar]
2. [Dimension 2]
3. [Dimension 3]
4. Anti-slop compliance (9 principles)
5. Output spec compliance (format, length, structure)
6. Constraint compliance (all NEVER/ALWAYS rules honored?)

Score each dimension: PASS / NEEDS IMPROVEMENT / FAIL
THINK command: `think hard` (critical evaluation)
```

**Phase 3 — REFINE (Improve):**
```markdown
### Phase 3: Refine
For each dimension scoring NEEDS IMPROVEMENT or FAIL:
- Identify the specific weakness
- Revise the output to address it
- Verify the revision didn't break other dimensions
- Re-score the revised dimension

THINK command: `think harder` (deep refinement)
Trigger: Only if Phase 2 found ≥1 dimension below PASS
```

### Step 3: Define Critique Dimensions

Customize critique dimensions based on skill type:

**For Copy/Marketing Skills:**
1. Anti-slop compliance (all 9 principles)
2. Benefit dimensionalization (≥3 layers)
3. Voice/brand alignment
4. CTA clarity and friction
5. Forward momentum (each section drives to next)

**For Analysis/Research Skills:**
1. Evidence quality (specific quotes, not generalities)
2. Completeness (all required aspects covered)
3. Insight depth (beyond surface observations)
4. Actionability (findings lead to clear next steps)
5. Anti-slop (no filler in analysis language)

**For Report/Deliverable Skills:**
1. Structure compliance (all sections present, correct order)
2. Data accuracy (numbers, quotes, references verified)
3. Clarity (each point understandable without re-reading)
4. Completeness (no gaps, no TODOs, no placeholders)
5. Conciseness (no redundant sections or repetitive points)

### Step 4: Define Refinement Triggers

Specify when refinement is mandatory vs optional:

```markdown
## REFINEMENT TRIGGERS

MANDATORY REFINEMENT (always refine):
- Any dimension scores FAIL
- Anti-slop scan detects >0 instances
- Output spec check reveals missing sections
- Constraint violation detected

OPTIONAL REFINEMENT (refine if available context):
- Dimensions at NEEDS IMPROVEMENT (not FAIL)
- Output feels generic (taste test)
- Length exceeds spec by >10%

SKIP REFINEMENT (proceed directly):
- All dimensions PASS
- Anti-slop clean
- Output spec fully satisfied
```

### Step 5: Map to Think Hierarchy

| Phase | Think Level | When to Escalate |
|-------|-------------|-----------------|
| Generate | `think` | Default for first pass |
| Critique | `think hard` | Always — critique needs depth |
| Refine | `think harder` | When multiple FAIL dimensions |
| Final check | `think` | Quick verification pass |

**Escalation rule:** If after refinement the output still has FAIL dimensions, escalate to `ultrathink` for one final attempt. If still failing after ultrathink, deliver with explicit quality flags.

## OUTPUT SPECIFICATION

```markdown
## CHAIN-OF-REFINEMENT PRESCRIPTION: [Target Skill Name]

### Applicability: [APPLICABLE | N/A]
**Reason:** [Why this skill benefits from / doesn't need refinement]

### Prescribed Protocol

#### Phase 1: Generate
[Phase 1 specification — what to produce on first pass]
**Think level:** `think`

#### Phase 2: Critique
Evaluate against these dimensions:
1. [Dimension 1] — [what PASS looks like]
2. [Dimension 2] — [what PASS looks like]
3. [Dimension 3] — [what PASS looks like]
4. Anti-slop compliance — ≤2 instances/100 lines
5. Output spec compliance — all required sections present
**Think level:** `think hard`

#### Phase 3: Refine
For each dimension below PASS:
- [Specific refinement instruction per dimension]
**Think level:** `think harder`
**Trigger:** ≥1 dimension below PASS
**Max iterations:** 2 (after 2 refinement loops, deliver with quality flags)

### Integration Instructions
- Insert after the current INSTRUCTIONS section
- Before the OUTPUT SPECIFICATION section
- Add to output spec: "Phase 2 critique results must be included in delivery metadata"

### Expected Impact
- Output quality: Single-pass → Multi-pass with self-critique
- Slop risk: [current] → reduced by systematic checking
- Think hierarchy utilization: [current usage] → structured escalation
```

---

## CONSTRAINTS

- NEVER add Chain-of-Refinement to validation/gate skills (they ARE the critique layer)
- NEVER allow infinite refinement loops (max 2 iterations, then deliver with flags)
- NEVER make Phase 2 critique dimensions vague ("is it good enough?") — MUST be specific and scorable
- MUST map each phase to a specific think hierarchy level
- MUST define what PASS looks like for each critique dimension (measurable, not subjective)
- MUST include a "skip refinement" condition (don't refine what's already good)
- The refinement protocol should add <30 lines to the skill (keep it concise)
- If skill already has a review/validation step: Enhance it rather than adding a parallel system

---

## FALLBACK

- If skill is N/A for Chain-of-Refinement: Report clearly and skip
- If skill already has a refinement loop: Evaluate whether it's well-structured. If yes, report "existing refinement sufficient." If poorly structured, prescribe improvements to the existing loop rather than adding new.
