# 1.2-B: Anti-Slop Scanner
*Detect filler, hedge words, buzzwords, and generic language*

---

## PURPOSE

Scan the target skill for language that would produce "slop" — generic, filler-laden, corporate-sounding output. If a skill's own language is sloppy, its outputs will be sloppy. If its output specifications tolerate slop, its deliverables will contain slop. This scanner catches both.

**Scope:** Scan all text content. Count slop instances. Report density score. Catalog specific violations.
**Non-Goals:** Do NOT rewrite sloppy language. Do NOT judge the skill's purpose or logic. Scan for linguistic anti-patterns only.

---

## INPUTS

- Target skill file content (in context)
- Total line count from 1.0-A consumption receipt

---

## INSTRUCTIONS

### Step 1: Scan for Filler Phrases

Flag every instance of:
- "It's important to note"
- "It's worth mentioning"
- "In today's fast-paced world"
- "At the end of the day"
- "When all is said and done"
- "In this day and age"
- "As we all know"
- "The fact of the matter is"
- "It goes without saying"
- "Needless to say"
- "That being said"
- "With that in mind"
- "Moving forward"
- "Going forward"
- "In order to" (use "to" instead)
- "Due to the fact that" (use "because")
- "At this point in time" (use "now")
- "In the event that" (use "if")
- "For the purpose of" (use "to" or "for")
- "In terms of" (usually deletable)

### Step 2: Scan for Hedge Words (Without Genuine Uncertainty)

Flag every instance of:
- "perhaps"
- "it seems"
- "one might argue"
- "arguably"
- "it could be said"
- "in some ways"
- "to some extent"
- "more or less"
- "kind of"
- "sort of"
- "basically"
- "essentially"
- "generally speaking"
- "for the most part"

**Exception:** If the hedge word is inside a THREE-TIER UNCERTAINTY context (expressing genuine calibrated uncertainty), do NOT flag it. Only flag when used as filler.

### Step 3: Scan for Corporate Buzzwords

Flag every instance of:
- "synergy" / "synergize"
- "leverage" (as verb, not noun)
- "ecosystem" (when not referring to actual technical architecture)
- "paradigm shift"
- "holistic approach"
- "move the needle"
- "low-hanging fruit"
- "circle back"
- "deep dive" (unless technical)
- "unpack" (overused)
- "streamline" (unless specific to process optimization)
- "empower"
- "best-in-class"
- "cutting-edge"
- "next-level"
- "game-changing"
- "disruptive"
- "scalable" (unless technical architecture context)
- "actionable insights"
- "value-add"
- "thought leader"
- "best practices" (unless referencing specific documented practices)

### Step 4: Scan for Generic Modifiers

Flag every instance of:
- "very" (what specific degree?)
- "really" (same)
- "quite" (same)
- "extremely" (same)
- "incredibly" (same)
- "absolutely" (unless expressing binary certainty)
- "significantly" (significant by what measure?)
- "substantially" (same)
- "highly" (how high?)
- "greatly" (how great?)

### Step 5: Scan for Unnecessary Passive Voice

Flag instances where active voice would be clearer:
- "It was determined that..." → "We determined..."
- "The analysis was performed..." → "[Agent] analyzed..."
- "Results were obtained..." → "[Agent] obtained..."
- "Consideration should be given to..." → "Consider..."
- "It is recommended that..." → "Do [X]" or "[X] is required"

**Exception:** Passive voice is acceptable when the actor is genuinely unknown or irrelevant. Only flag when active would be clearer AND the actor is known.

### Step 6: Scan for Sloppy Output Specifications

In the OUTPUT section specifically, flag:
- Quality criteria defined only as "high quality" or "well-written"
- Format specs that say "appropriate format" without defining what's appropriate
- Length specs that say "concise" without a word/line count
- Validation criteria that say "ensure quality" without measurable checks

### Step 7: Calculate Density

```
slop_density = total_instances / (total_lines / 100)
```

Threshold: ≤2 instances per 100 lines.

---

## OUTPUT SPECIFICATION

```yaml
anti_slop_audit:
  target: [filename]
  total_lines: [count]
  total_instances: [count]
  slop_density: [instances per 100 lines]
  threshold: 2.0
  status: [PASS | FAIL]

  breakdown:
    filler_phrases: [count]
    hedge_words: [count]
    corporate_buzzwords: [count]
    generic_modifiers: [count]
    passive_voice: [count]
    sloppy_output_specs: [count]

  instances: # All found, with location
    - line: [approx line number]
      category: [filler | hedge | buzzword | modifier | passive | output_spec]
      text: "[quoted context showing the slop]"
      severity: [high | medium | low]
    # [continue for all instances]

  worst_offenders: # Top 5 most impactful instances
    - text: "[quoted]"
      why_bad: "[specific impact on output quality]"
      category: [type]
    # [top 5]

  output_spec_issues: # Specific to Block 4 problems
    - issue: "[what's vague in the output spec]"
      impact: "Agent cannot self-validate because [reason]"
    # [all output spec issues]
```

---

## CONSTRAINTS

- NEVER flag technical terminology as buzzwords (e.g., "scalable" in a genuine architecture context is fine)
- NEVER flag hedge words that express genuine, calibrated uncertainty (e.g., "approximately 70%" is not slop)
- NEVER flag passive voice in quotes/exemplars (the skill is showing examples, not writing sloppily)
- MUST distinguish between the skill's own language and example/template language within the skill
- MUST weight output specification slop higher — that's where it directly infects agent outputs
- Severity levels: HIGH = output specs with unmeasurable criteria; MEDIUM = filler/buzzwords in instructions; LOW = passive voice, generic modifiers in context sections
- If a word appears in a code block or YAML template, do NOT flag it (it's structural, not linguistic)

---

## FALLBACK

- If target has 0 instances: Report as PASS with "Clean — no slop detected"
- If target is entirely code/templates with no prose: Report as N/A
- If density is borderline (1.8-2.2): Report as PASS but note "borderline — monitor in future audits"
