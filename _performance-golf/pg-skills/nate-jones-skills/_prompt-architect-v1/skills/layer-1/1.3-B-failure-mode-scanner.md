# 1.3-B: Failure Mode Scanner
*Check for Seven MCP Failure Modes*

---

## PURPOSE

Identify whether the target skill exhibits any of the Seven MCP Failure Modes defined in ARCHITECTURE-PRD §3.5. These are architectural anti-patterns specific to skills that use external tools (MCP servers, APIs, web scrapers, databases). Each failure mode present is a ticking production bomb.

**Scope:** Check for 7 failure modes. Flag any detected. Assess severity.
**Non-Goals:** Do NOT fix the failure modes. Diagnose only.
**N/A Condition:** If the skill uses NO external tools, this audit returns N/A with a clean pass.

---

## INPUTS

- Target skill file content (in context)
- Tool references identified in 1.0-A consumption receipt

---

## INSTRUCTIONS

### Step 0: Tool Usage Check

First, determine if this skill uses external tools:
- MCP server calls
- API calls (Firecrawl, Apify, Perplexity, etc.)
- Web fetch/scrape operations
- Database read/write
- File system operations beyond basic read/write

**If NO external tools:** Report all modes as N/A, status PASS, and exit.
**If YES:** Continue to Step 1.

### Step 1: Check Failure Mode 1 — Universal API Router Trap

**What it looks like:**
- Skill treats MCP/tools as a general-purpose API gateway
- No scoping of what each tool is FOR
- Tool calls without specific purpose matching
- "Use [tool] to get whatever data we need"

**Detection signals:**
- Vague tool invocations ("use Firecrawl to scrape relevant pages")
- No specification of WHICH tool for WHICH purpose
- Tool used for purposes it's not designed for
- Missing tool-to-task matching logic

**Severity:** HIGH — leads to incorrect tool usage and wasted resources

### Step 2: Check Failure Mode 2 — Context = Data Confusion

**What it looks like:**
- Raw data dumps directly into context window
- No summarization or filtering of tool outputs
- Treating context as unlimited storage
- "Read the full output and process it"

**Detection signals:**
- No instructions for filtering/summarizing tool outputs
- Tool outputs consumed in raw form
- No size limits on what enters context
- Context treated as data storage rather than working memory

**Severity:** HIGH — context overflow, quality degradation, context window waste

### Step 3: Check Failure Mode 3 — Hot Path Disaster

**What it looks like:**
- Tool calls inside tight loops or high-frequency operations
- No batching, caching, or pre-fetching
- Every iteration requires a new tool call
- "For each item, call [tool]..."

**Detection signals:**
- Tool calls inside "for each" or loop constructs
- No batch operation instructions
- No caching mentioned for repeated queries
- Sequential tool calls that could be parallelized

**Severity:** MEDIUM — performance death, cost explosion, rate limiting

### Step 4: Check Failure Mode 4 — Security Theater

**What it looks like:**
- Tool outputs trusted without validation
- No sanity-checking of returned data
- External data used directly in decisions without verification
- "Use the tool's output to determine..."

**Detection signals:**
- No validation step after tool calls
- External data used in critical decisions without checks
- No format/type verification of tool responses
- Trust by default rather than verify by default

**Severity:** HIGH — garbage-in-garbage-out, cascading errors from bad data

### Step 5: Check Failure Mode 5 — Magical Performance Expectations

**What it looks like:**
- Assuming real-time responses from inherently slow tools
- No timeout handling
- No async/polling patterns for long operations
- "Immediately get the results from..."

**Detection signals:**
- No timeout specifications for tool calls
- No handling of slow/delayed responses
- No fallback for when tools take too long
- Expectations of instant results from web scraping or complex queries

**Severity:** MEDIUM — hangs, timeouts, session failures

### Step 6: Check Failure Mode 6 — Microservices Trap

**What it looks like:**
- One tool call per tiny function
- Multiple sequential tool calls that could be one
- Over-decomposition of tool usage
- "First call [tool A] for X, then call [tool B] for Y, then call [tool C] for Z" (when one tool could do all three)

**Detection signals:**
- 5+ sequential tool calls in one workflow step
- Each tool call does minimal work
- No consolidation of related operations
- Context consumed by tool call overhead rather than results

**Severity:** LOW-MEDIUM — inefficiency, context waste, latency accumulation

### Step 7: Check Failure Mode 7 — Real-Time Delusion

**What it looks like:**
- Treating cached/stale data as live
- No freshness requirements specified
- Assuming tool data is current without verification
- "Get the latest data from..."

**Detection signals:**
- No timestamp checking on tool outputs
- No freshness/staleness declarations
- "Latest" or "current" language without verification mechanism
- No handling of stale data scenarios

**Severity:** MEDIUM — decisions made on outdated information

---

## CONSTRAINTS

- NEVER flag a mode as DETECTED without specific evidence (quote or section reference)
- NEVER flag standard file Read/Write as "external tools" — these are basic operations, not MCP/API calls
- MUST distinguish between tools used WITHIN the skill vs tools the skill's OUTPUT will be used with
- If a skill has a proper tool fallback chain defined, reduce severity of related modes by one level
- If a skill explicitly handles timeouts/caching, mark corresponding modes as CLEAR even if other signals present
- Modes 1, 2, and 4 are the most critical — flag these prominently when detected

---

## FALLBACK

- If skill references tools ambiguously ("use available tools"): Flag as potential Mode 1 but mark confidence as LOW
- If skill is designed to be orchestrated (tool usage handled by parent): Note "tool management delegated" and check if parent handles it
