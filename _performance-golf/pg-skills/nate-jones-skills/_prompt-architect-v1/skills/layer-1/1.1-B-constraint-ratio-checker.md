# 1.1-B: Constraint Ratio Checker
*Measure the ratio of constraints to instructions*

---

## PURPOSE

Calculate the constraint-to-instruction ratio in the target skill. The ARCHITECTURE-PRD establishes that constraints outperform instructions (Axiom 2: 70% constraints, 30% instructions). This skill measures whether the target follows that principle.

**Scope:** Count constraints. Count instructions. Calculate ratio. Report.
**Non-Goals:** Do NOT rewrite anything. Do NOT judge content quality. Count and calculate only.

---

## INPUTS

- Target skill file content (in context from 1.0-A)
- Consumption receipt (for section map)

---

## INSTRUCTIONS

### Step 1: Identify Constraint Statements

A statement is a CONSTRAINT if it:
- Contains boundary keywords: NEVER, ALWAYS, MUST NOT, DO NOT, CANNOT, FORBIDDEN, PROHIBITED
- Contains limit phrases: "is not allowed," "under no circumstances," "is prohibited"
- Defines quantitative boundaries: maximum, minimum, "no more than," "at least"
- Lists exclusions: "does NOT include," "excludes," "except"
- Frames anti-patterns: "what NOT to do," "avoid," "anti-pattern"
- Sets binary rules: "[X] is always [Y]" (immutable declarations)

**Count each distinct constraint statement as 1.** A bullet point with a constraint = 1. A sentence with a constraint = 1. A table row with a constraint = 1.

### Step 2: Identify Instruction Statements

A statement is an INSTRUCTION if it:
- Directs action: "do," "write," "create," "generate," "analyze," "produce," "execute"
- Describes process steps: "Step 1:", numbered actions, sequential directions
- Frames positive behaviors: "aim for," "try to," "should," "ideally"
- Requests output: "provide," "include," "output," "deliver"
- Describes what to do WITHOUT bounding what not to do

**Count each distinct instruction statement as 1.**

### Step 3: Handle Ambiguous Cases

Some statements are BOTH constraint and instruction:
- "ALWAYS validate output before delivery" → CONSTRAINT (ALWAYS + boundary)
- "Write concise responses under 200 words" → CONSTRAINT (quantitative limit)
- "Generate a report covering themes" → INSTRUCTION (action without boundary)
- "Analyze the data carefully" → INSTRUCTION (vague action)

**Rule:** If the statement sets a boundary (even while directing action), count as CONSTRAINT. If it only directs action without bounding, count as INSTRUCTION.

### Step 4: Calculate Ratio

```
ratio = constraint_count / (constraint_count + instruction_count)
```

### Step 5: Classify Non-Operational Content

Some content is neither constraint nor instruction:
- Context/background information
- Exemplars/examples
- Document metadata
- Headers/titles
- Tables of reference data

**Do NOT count these.** They don't factor into the ratio.

---

## CONSTRAINTS

- NEVER count headers, titles, or structural elements as either type
- NEVER double-count a statement (each statement counted exactly once)
- NEVER count exemplar content (examples of output) — only count the skill's own rules
- MUST distinguish between the skill's instructions TO the agent vs. instructions the agent gives to users
- MUST handle edge cases: a single sentence can contain both a constraint and instruction — count as constraint
- If the file has 0 operational statements (all context/examples), report as N/A not 0

---

## FALLBACK

- If target is purely a template with no operational content: Report ratio as N/A, note "template file — no operational statements"
- If target is a PRD (requirements doc, not execution doc): Adapt — PRDs should be mostly constraints. Flag if instruction-heavy.
