# 3.1-B: PRD Generator
*Generate a requirements document for systems that lack one*

---

## PURPOSE

When auditing a full system (MASTER-AGENT + sub-skills) that has no formal PRD, generate one. A system without a PRD has no authority document — the MASTER-AGENT becomes both executor and judge, which violates separation of concerns. The PRD defines WHAT must be achieved; the MASTER-AGENT defines HOW.

**Scope:** Extract implicit requirements from the system. Formalize into PRD structure. Define quality gates.
**Non-Goals:** Do NOT invent requirements that aren't already implied by the system's behavior. Extract and formalize, don't extend.

---

## INPUTS

- Target system's MASTER-AGENT.md (or equivalent orchestrator)
- All sub-skill files in the system
- Any existing quality documents, templates, or standards
- Diagnostic Summary from Layer 1 (understanding of system's current quality)

---

## INSTRUCTIONS

### Step 1: Extract Implicit Requirements

Read through the MASTER-AGENT and all skills. For each operational statement, ask:
- "What REQUIREMENT does this imply?"
- "What QUALITY STANDARD is this trying to achieve?"
- "What would FAILURE look like if this step didn't exist?"

Catalog every implied requirement.

### Step 2: Classify Requirements

Organize into categories:

**Functional Requirements (what the system produces):**
- Input specifications (what it accepts)
- Output specifications (what it delivers)
- Processing requirements (what transformations it performs)
- Integration requirements (what it connects to)

**Quality Requirements (how well it produces):**
- Accuracy standards (how correct outputs must be)
- Completeness standards (what coverage is required)
- Consistency standards (how reproducible results must be)
- Performance standards (any time/resource constraints)

**Operational Requirements (how it behaves):**
- Error handling requirements
- State management requirements
- Tool usage requirements
- Session management requirements

### Step 3: Define Quality Gates

For each layer/phase in the system, define:
- What conditions MUST be true to proceed (hard blockers)
- What conditions SHOULD be true (soft warnings)
- How to measure each condition (quantified where possible)
- What happens if conditions aren't met (remediation or halt)

### Step 4: Define Hard Blockers

Identify conditions that are NEVER acceptable:
- Missing required outputs
- Quality scores below minimum thresholds
- Incomplete processing of required inputs
- Tool failures without fallback execution

### Step 5: Assemble PRD Document

---

## CONSTRAINTS

- NEVER invent requirements that aren't implied by the existing system (extract, don't extend)
- NEVER make the PRD so strict that the current system would fail against it (it should describe the intended bar, not an impossible one)
- MUST make every requirement MEASURABLE (no "produce good output" — quantify)
- MUST include hard blockers (these are the most valuable part of a PRD)
- MUST define conflict resolution (PRD > MASTER-AGENT)
- The generated PRD should be 200-500 lines (comprehensive but not overwhelming)
- Flag any requirements that seem inconsistent with the system's purpose (possible system design issues)
- Include a "Generated" notice — this PRD should be reviewed by the human before becoming authoritative

---

## FALLBACK

- If system is too simple for a full PRD (only 2-3 skills): Generate a lightweight "Requirements Summary" instead (50-100 lines covering just functional requirements and hard blockers)
- If system purpose is unclear from the files: Flag for human clarification, generate best-effort PRD with explicit "ASSUMPTION" markers
