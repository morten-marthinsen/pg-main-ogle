# 3.2-B: Before-After Diff
*Show exactly what changed between original and optimized versions*

---

## PURPOSE

Produce a clear, human-readable diff showing every change made to the skill file during optimization. This provides transparency — the skill owner can see exactly what was modified, added, or restructured, with rationale for each change.

**Scope:** Compare original and rewritten files. Catalog all differences. Annotate with rationale.
**Non-Goals:** Do NOT evaluate whether changes are good. Do NOT add new changes. Document what was done.

---

## INPUTS

- Original target skill file content
- Rewritten skill file from 3.1-A
- Prescriptions package from 2.5-A (for rationale annotations)
- Change log from 3.1-A rewrite metadata

---

## INSTRUCTIONS

### Step 1: Identify All Differences

Compare original and rewritten files section by section:
- Sections that were MODIFIED (content changed)
- Sections that were ADDED (new content)
- Sections that were REMOVED (content deleted)
- Sections that were MOVED (same content, different position)
- Content that was SPLIT (one section became two)
- Content that was MERGED (two sections became one)

### Step 2: Classify Each Change

For each difference:
- **Type:** Modified / Added / Removed / Moved / Split / Merged
- **Category:** Structural / Constraint / Guardrail / Output / Exemplar / Refinement / Wake Word
- **Prescription Source:** Which Layer 2 skill prescribed this change
- **PRD Reference:** Which ARCHITECTURE-PRD section this enforces
- **Impact:** Which quality dimension this improves

### Step 3: Annotate with Rationale

For each change, provide a 1-sentence explanation:
- WHY this change was made (not just what changed)
- What quality dimension it improves
- What risk it mitigates

### Step 4: Highlight Key Transformations

Identify the 3-5 most impactful changes and give them detailed treatment:
- Full before/after text comparison
- Detailed rationale
- Expected behavior difference

## CONSTRAINTS

- NEVER present a change without rationale (every change must be justified)
- NEVER misrepresent the scope of changes (if only 2 lines changed, don't make it look dramatic)
- MUST show ACTUAL before/after text for the key transformations (not just descriptions)
- MUST trace every change to its prescription source
- MUST be honest about net size increase (adding guardrails and exemplars WILL make files longer — that's acceptable)
- Keep the diff readable — not a line-by-line git diff, but a structured human-readable comparison
- If nothing was removed: State explicitly "No content was removed — all changes are additions or rewrites"
- If file grew significantly: Note this is expected (production hardening adds content)

---

## FALLBACK

- If no changes were made (PASS): Report "No diff — skill passed all standards as-is"
- If only minor changes: Produce abbreviated diff (skip the "Top 5 Key Transformations" section, just list changes)
