# 0.5 Chunking Manager

**Version:** 1.0
**Created:** January 18, 2026
**Type:** Core Infrastructure Skill
**Purpose:** Prevent context window exhaustion when processing 1,000+ quotes

---

## CRITICAL CONTEXT

**Problem Solved:** Processing 1,000+ quotes (200K+ tokens) exceeds Claude's practical working context. Without chunking, the system will truncate data, hallucinate, or fail silently.

**This Skill Prevents:**
- Data loss from context overflow
- Hallucinated quotes and analysis
- Validation failures from missing data
- The exact failure mode that caused the Ion Golf Ball incident

---

## Activation

This skill is called by ANY skill processing large datasets:
- `1.5-A Content Extractor` (when raw content exceeds threshold)
- `1.5-B Basic Tagger` (when quote count exceeds threshold)
- `1.5-C Quantifier` (when processing full quote set)
- `2.7-A Intelligence Synthesizer` (when consolidating analysis)
- `3.2-A Handoff Packager` (when generating final output)

## Thresholds

```yaml
chunking_thresholds:
  # When to chunk (any threshold exceeded triggers chunking)
  quote_count: 100          # Chunk if > 100 quotes in single operation
  token_estimate: 50000     # Chunk if estimated tokens > 50K
  file_size_kb: 200         # Chunk if input file > 200KB

  # Batch sizes for processing
  batch_sizes:
    quote_extraction: 50    # Extract 50 quotes per batch
    quote_tagging: 100      # Tag 100 quotes per batch
    quote_analysis: 75      # Analyze 75 quotes per batch
    synthesis: 50           # Synthesize 50 items per batch
```

## Chunking Protocol

### Step 1: Assess Data Size

```yaml
assessment:
  input: [data to process]

  calculate:
    total_items: COUNT(items)
    estimated_tokens: total_items * average_tokens_per_item
    file_size: SIZE(input_file) in KB

  decision:
    IF total_items <= quote_count_threshold AND
       estimated_tokens <= token_threshold AND
       file_size <= file_size_threshold:
      → PROCESS_NORMALLY (no chunking needed)
    ELSE:
      → ACTIVATE_CHUNKING
```

### Step 2: Create Batches

```yaml
batch_creation:
  input: [full dataset]
  batch_size: [from thresholds based on operation type]

  process:
    1. Sort data by source/timestamp for consistency
    2. Split into batches of batch_size
    3. Assign batch IDs: batch_001, batch_002, ...
    4. Create batch manifest file

  output:
    batch_manifest.json:
      total_items: [number]
      batch_count: [number]
      batch_size: [number]
      batches:
        - batch_id: "batch_001"
          item_range: "001-100"
          status: "pending"
        - batch_id: "batch_002"
          item_range: "101-200"
          status: "pending"
```

### Step 3: Process Each Batch

```yaml
batch_processing:
  for_each_batch:
    1. Load batch data ONLY (not full dataset)
    2. Process batch with calling skill's logic
    3. Write batch output to file:
       - layer-1-outputs/batches/batch_001_tagged.json
       - layer-1-outputs/batches/batch_002_tagged.json
    4. Update batch manifest status: "complete"
    5. Clear batch from context
    6. Load next batch

  state_persistence:
    after_each_batch:
      - Update checkpoint with completed batch
      - Write running totals to aggregation file
      - Log: "Batch {batch_id} complete: {items_processed}/{total_items}"

  on_interruption:
    - Last completed batch is saved
    - Resume from next incomplete batch
    - Never reprocess completed batches
```

### Step 4: Aggregate Results

```yaml
aggregation:
  after_all_batches_complete:
    1. Load batch manifest
    2. For each completed batch:
       - Load batch output file
       - Extract summary/counts (NOT full content)
       - Append to aggregation
    3. Generate consolidated summary
    4. Write final output file

  aggregation_types:
    quote_extraction:
      aggregate: quote_counts_by_bucket, source_list, quote_ids
      detail_in_files: full quote text stays in batch files

    quote_tagging:
      aggregate: bucket_totals, subcategory_counts
      detail_in_files: individual tagged quotes

    analysis:
      aggregate: pattern_summary, key_findings
      detail_in_files: supporting evidence
```

## Integration with Other Skills

### For 1.5-A Content Extractor

```yaml
chunking_integration:
  trigger: raw_content_size > 200KB OR source_count > 20

  protocol:
    1. Split sources into batches of 10
    2. Extract quotes from each batch
    3. Assign quote IDs within batch (P-001 to P-XXX)
    4. Write batch output with ID offset for next batch
    5. Aggregate quote counts and ID ranges
```

### For 1.5-C Quantifier

```yaml
chunking_integration:
  trigger: quote_count > 100

  protocol:
    1. Load quote ID manifest (not full quotes)
    2. Process quotes in batches of 100
    3. For each batch:
       - Count by bucket and subcategory
       - Identify top quotes (store IDs only)
       - Write batch tallies
    4. Aggregate all batch tallies
    5. Identify overall top 25 per subcategory (by ID)
    6. Final output references quote IDs, not full text
```

### For 3.2-A Handoff Packager

```yaml
chunking_integration:
  trigger: ALWAYS (final handoff is always large)

  protocol:
    1. Generate handoff section-by-section
    2. Never load all quotes into context simultaneously
    3. For each bucket section:
       - Load only that bucket's data
       - Generate section
       - Write to file
       - Clear from context
    4. Concatenate sections into final file
    5. Use summary + ID references, not full quotes repeated
```

## Error Handling

```yaml
error_handling:
  batch_fails:
    - Log error with batch_id
    - Mark batch as "failed" in manifest
    - Continue with next batch
    - Report failed batches at end
    - Human decides: retry failed batches or proceed without

  aggregation_fails:
    - Save partial aggregation
    - Log which batches were aggregated
    - Report incomplete aggregation
    - Human can manually complete or restart aggregation

  context_overflow_despite_chunking:
    - Reduce batch_size by 50%
    - Retry current batch
    - If still fails: batch_size = 25 (minimum)
    - If minimum fails: CRITICAL ERROR - skill cannot process
```

## Validation

```yaml
validation:
  after_chunking_complete:
    - Verify all batches processed
    - Verify aggregation totals match batch totals
    - Verify no data loss: SUM(batch_counts) == total_input_count
    - Verify quote IDs are unique and sequential

  validation_output:
    chunking_validation.json:
      input_count: [original item count]
      processed_count: [sum of batch processed counts]
      match: true/false
      missing_items: [list if any]
      duplicate_ids: [list if any]
```

```yaml
  total_quotes: 1247
  processing_time: "00:23:45"
  batches_processed: 12
  batches_failed: 0
```

## Anti-Hallucination Safeguards

```yaml
safeguards:
  never_generate_quotes:
    - Chunking manager moves data, never creates it
    - All quotes must trace to batch input files
    - ID assignment is mechanical, not creative

  verify_data_integrity:
    - Hash input files before chunking
    - Hash batch files after processing
    - Verify hashes on aggregation
    - Any hash mismatch = CRITICAL ERROR

  no_context_shortcuts:
    - Never "remember" data from previous batch
    - Always re-read from files
    - Context is temporary, files are permanent
```
